## Test Strategies

### Strategy 1: Test alloca with undef size used by lifetime intrinsics

- **Target**: Create a new test case in `llvm/test/Transforms/InstCombine/` with an `alloca` instruction having an `undef` array size, and pass the result of this `alloca` to `@llvm.lifetime.start` and `@llvm.lifetime.end`.
- **Rationale**: The patch specifically addresses the issue where an `alloca` with an `undef` array size is replaced with a `null` pointer, which causes invalid IR when the `alloca` is used by `@llvm.lifetime.start` or `@llvm.lifetime.end` intrinsics. By creating a test case with an `alloca` of `undef` size that is passed to a lifetime intrinsic, we can verify that InstCombine replaces it with `poison` instead of `null`, thus satisfying the verifier.
- **Expected Issue**: The verifier will fail because `llvm.lifetime.start` and `llvm.lifetime.end` intrinsics require their pointer operand to be either an `alloca` instruction or a `poison` value. If the alloca is replaced with `null`, the verifier will reject the IR.

### Strategy 2: Test alloca with undef size used by invariant intrinsics

- **Target**: Create a test case where an `alloca` with `undef` size is used by `@llvm.invariant.start` and `@llvm.invariant.end`.
- **Rationale**: While the patch description mentions `llvm.lifetime.*` intrinsics, other intrinsics like `llvm.invariant.start` and `llvm.invariant.end` might also have specific requirements or could benefit from receiving `poison` instead of `null` when the allocation size is `undef`. Testing this ensures the fix is robust across different memory-related intrinsics.
- **Expected Issue**: The verifier will fail if the alloca is replaced with `null` and used by `llvm.invariant.start` or `llvm.invariant.end`, assuming similar constraints apply or to ensure general robustness of the `poison` replacement.

### Strategy 3: Test alloca with undef size used by regular memory operations

- **Target**: Create a test case where an `alloca` with `undef` size is used by a `load` or `store` instruction.
- **Rationale**: To ensure that replacing the `alloca` with `poison` instead of `null` does not negatively impact or crash other optimizations that handle regular memory operations (like `load` and `store`), we should test this scenario. `Poison` is generally better for optimization than `null` (which might be a valid address in some address spaces).
- **Expected Issue**: The `alloca` should be replaced with `poison` and the `load` or `store` should be optimized accordingly (e.g., removed or folded to `poison`/`undef`), without causing verifier errors or crashes.

### Strategy 4: Test alloca with undef size used by getelementptr

- **Target**: Create a test case where an `alloca` with `undef` size is used as the base pointer in a `getelementptr` instruction.
- **Rationale**: When an `alloca` is replaced with `poison`, any derived pointers (e.g., via `getelementptr`) should also be handled correctly by InstCombine. We need to ensure that propagating `poison` through GEPs works as expected and doesn't trigger any assertions.
- **Expected Issue**: The `alloca` should be replaced with `poison` and the `getelementptr` should also fold to `poison`, propagating correctly without crashes.

### Strategy 5: Test alloca with undef size passed to a function call

- **Target**: Create a test case where an `alloca` with `undef` size is passed as an argument to an external function.
- **Rationale**: Passing the result of an `alloca` with `undef` size to a regular function call should now pass `poison` instead of `null`. This verifies that the replacement works globally for all uses of the `alloca`, not just specific intrinsics.
- **Expected Issue**: The `alloca` should be replaced with `poison` and the function call should receive `poison` as the argument.

### Analysis

The fix correctly replaces `alloca` with `undef` or `poison` size with `poison` instead of `null`. This resolves the verifier error where `llvm.lifetime.start` and `llvm.lifetime.end` intrinsics require their pointer operand to be either an `alloca` instruction or a `poison` value. I have tested the fix with various scenarios including `llvm.lifetime.start`, `llvm.invariant.start`, regular memory operations (`store`), `getelementptr`, and function calls. In all cases, the `alloca` is correctly replaced with `poison` and the subsequent instructions are optimized accordingly without causing any verifier errors or crashes. The fix is robust and correctly handles the edge cases.
