## Test Strategies

### Strategy 1: FAdd with -0.0 and -0.0 under IEEE mode

- **Target**: fadd instruction with -0.0 operands
- **Rationale**: The patch adds logic to handle `fadd x, 0.0` and `fadd x, x`. For `fadd x, x`, if `x` is `-0.0`, the result is `-0.0`. The logic in `KnownFPClass::fadd_self` checks `KnownSrc.isKnownNeverLogicalPosZero(Mode)` to rule out `+0.0`, but it might not correctly handle the case where `x` can be `-0.0` and the result is `-0.0`. We should test `fadd -0.0, -0.0` to ensure it doesn't incorrectly assume the result cannot be `-0.0`.
- **Expected Issue**: The optimization might incorrectly assume the result cannot be -0.0, leading to incorrect simplifications or constant folding.

### Strategy 2: FAdd with subnormal and subnormal under DAZ/FTZ mode

- **Target**: fadd instruction with subnormal operands under DAZ/FTZ mode
- **Rationale**: The patch adds logic for `fadd x, x` under non-IEEE modes (like DAZ/FTZ). It states "Doubling a subnormal may produce 0 if FTZ/DAZ". If `x` is a subnormal, `x + x` might be flushed to zero. We should test if the optimization correctly handles this and doesn't incorrectly assume the result cannot be zero.
- **Expected Issue**: The optimization might incorrectly assume the result cannot be +0.0, leading to incorrect simplifications.

### Strategy 3: FAdd with +inf and -inf

- **Target**: fadd instruction with +inf and -inf operands
- **Rationale**: The patch adds logic for `fadd` in `KnownFPClass::fadd`. It checks if both operands are known never NaN and at least one is known never infinity to rule out NaN. However, if one operand is `+inf` and the other is `-inf`, the result is NaN. We should test `fadd +inf, -inf` to ensure it doesn't incorrectly assume the result cannot be NaN.
- **Expected Issue**: The optimization might incorrectly assume the result cannot be NaN, leading to incorrect simplifications.

### Strategy 4: FAdd with -0.0 and +0.0 under dynamic denormal mode

- **Target**: fadd instruction with -0.0 and +0.0 operands under dynamic denormal mode
- **Rationale**: The patch adds logic for `fadd` in `KnownFPClass::fadd`. It checks `isKnownNeverLogicalNegZero(Mode)` to rule out `-0.0`. Under dynamic denormal mode, the behavior of zero might be different. We should test `fadd -0.0, +0.0` under dynamic denormal mode to ensure it doesn't incorrectly assume the result cannot be `-0.0`.
- **Expected Issue**: The optimization might incorrectly assume the result cannot be -0.0, leading to incorrect simplifications.

### Strategy 5: FAdd self with negative subnormal under PreserveSign mode

- **Target**: fadd instruction with negative subnormal operand under PreserveSign mode
- **Rationale**: The patch adds logic for `fadd_self` in `KnownFPClass::fadd_self`. It checks `Mode.Output == DenormalMode::PreserveSign && KnownSrc.isKnownNeverPosSubnormal()` to rule out `+0.0`. If `x` is a negative subnormal, `x + x` might be flushed to `-0.0` under PreserveSign mode. We should test this to ensure it doesn't incorrectly assume the result cannot be `+0.0` or `-0.0`.
- **Expected Issue**: The optimization might incorrectly assume the result cannot be +0.0, leading to incorrect simplifications.
