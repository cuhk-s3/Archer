## Test Strategies

### Strategy 1: Oscillation with freeze and phi

- **Target**: Create a function with a loop or conditional branch that feeds a `phi` node. The `phi` node should have at least one incoming value that is guaranteed not to be undef or poison (e.g., a constant like `0`). The result of the `phi` node should be used by a `freeze` instruction.
- **Rationale**: The patch prevents `foldOpIntoPhi` from sinking a `freeze` into a `phi` if the incoming value is guaranteed not to be undef or poison. If we create a test case where a `freeze` is applied to a `phi` node, and one of the incoming values is guaranteed not to be undef or poison (e.g., a constant or an instruction that cannot produce poison), the old code would sink the `freeze` into the `phi`, creating `phi(..., freeze(x))`. Then, `pushFreezeToPreventPoisonFromPropagating` or another fold might push the `freeze` back out or simplify it away, leading to an infinite loop. We can trigger this by having a `freeze(phi(A, B))` where `A` is a constant (guaranteed not undef/poison) and `B` is an instruction that can produce poison.
- **Expected Issue**: InstCombine will enter an infinite loop, oscillating between `freeze(phi(..., x))` and `phi(..., freeze(x))`.

### Strategy 2: Redundant freeze creation in pushFreezeToPreventPoisonFromPropagating

- **Target**: Create a function with an instruction (e.g., `add`) that has a `freeze` instruction as one of its operands. Then, apply another `freeze` instruction to the result of the `add`. For example: `%f1 = freeze %x; %add = add %f1, %y; %f2 = freeze %add`.
- **Rationale**: The patch adds a check in `pushFreezeToPreventPoisonFromPropagating` to avoid pushing a `freeze` through an existing `freeze`. Without this check, if we have `freeze(freeze(x))` or a similar pattern where a `freeze` is pushed through an instruction that already has a `freeze` operand, it could repeatedly create new `freeze` instructions, leading to an infinite loop or excessive memory usage. We can trigger this by having a `freeze` applied to an instruction that already has a `freeze` as an operand, or by creating a chain of instructions where pushing the `freeze` would encounter another `freeze`.
- **Expected Issue**: InstCombine will enter an infinite loop, repeatedly creating redundant `freeze` instructions.

### Strategy 3: Freeze of PHI with multiple incoming values, some guaranteed not poison

- **Target**: Create a PHI node with one incoming value being a `noundef` argument or a constant, and another incoming value being an instruction that can generate poison (e.g., `add nsw`). Apply a `freeze` to the PHI node.
- **Rationale**: This is a variation of the first strategy, focusing on the specific condition `isGuaranteedNotToBeUndefOrPoison(InVal)`. If a PHI has multiple incoming values, and at least one is guaranteed not to be poison (e.g., an argument with `noundef` attribute, or a constant), sinking the freeze into the PHI will result in `freeze(constant)`, which simplifies to `constant`. The other incoming value might be a `freeze(poison_generating_inst)`. If the simplification causes the freeze to be removed, another pass might try to recreate it, causing oscillation.
- **Expected Issue**: InstCombine will enter an infinite loop due to oscillation between `freeze(phi)` and `phi(freeze)`.

### Strategy 4: Pushing freeze through an instruction with a frozen operand

- **Target**: Create a `select` or `phi` instruction where one of the operands is already a `freeze` instruction. Apply a `freeze` to the result of the `select` or `phi`.
- **Rationale**: The patch prevents `pushFreezeToPreventPoisonFromPropagating` from pushing through an existing `freeze`. If we have an instruction like `select %cond, (freeze %a), %b`, and we apply `freeze` to the `select`, the old code might try to push the `freeze` through the `select` to its operands. When it reaches `freeze %a`, it might try to push it further or create `freeze(freeze(%a))`, which simplifies back to `freeze(%a)`, potentially causing an infinite loop if the process repeats.
- **Expected Issue**: InstCombine will enter an infinite loop or create an excessive number of freeze instructions.

### Strategy 5: Freeze of PHI with all incoming values guaranteed not poison except one

Without the fix, it would sink, simplify the constant freezes, and potentially oscillate.
- **Expected Issue**: InstCombine will enter an infinite loop due to oscillation.
