## Test Strategies

### Strategy 1: Empty Block Between Guard and Preheader

- **Target**: Create a test case where `FC0.ExitBlock` branches to `FC1.GuardBlock`, but `FC0.GuardBranch` bypasses the loop and branches to a different block (not `FC1.GuardBlock`).
- **Rationale**: The fix checks `FC0.ExitBlock->getSingleSuccessor() == FC1.getEntryBlock()`. If there is an empty block between `FC0.ExitBlock` and `FC1.getEntryBlock()`, this condition will be false, and the loops will not be considered adjacent. However, if there is an empty block between the guard branch and the preheader of FC1, `FC1.getEntryBlock()` will be the guard block, and `FC0.ExitBlock->getSingleSuccessor()` might be the guard block. But wait, the fix is `FC0.ExitBlock->getSingleSuccessor() == FC1.getEntryBlock()`. If `FC1` is guarded, `FC1.getEntryBlock()` is `FC1.GuardBranch->getParent()`. So `FC0.ExitBlock->getSingleSuccessor()` must be `FC1.GuardBranch->getParent()`. What if `FC0.ExitBlock` has multiple successors? `getSingleSuccessor()` returns `nullptr`, so it won't match. What if `FC0.ExitBlock` is not the immediate predecessor of `FC1.getEntryBlock()`, but there are empty blocks in between? The fix requires them to be strictly adjacent (immediate successor). This is correct. But what if `FC0.ExitBlock` branches to `FC1.getEntryBlock()`, but `FC0.getNonLoopBlock()` is NOT `FC1.getEntryBlock()`? The commit message says: "This implicitly ensures that the non-loop successor of the FC0 guard branch is the entry block of FC1." Is this true? Let's check. `FC0.ExitBlock` is the exit block of `FC0`. For a guarded loop, the guard branch goes to `FC0.Preheader` and `FC0.getNonLoopBlock()`. The exit block of `FC0` goes to `FC0.ExitBlock->getSingleSuccessor()`. If `FC0.ExitBlock->getSingleSuccessor() == FC1.getEntryBlock()`, does it mean `FC0.getNonLoopBlock() == FC1.getEntryBlock()`? Not necessarily! `FC0.getNonLoopBlock()` could be something else, and `FC0.ExitBlock` could branch to `FC1.getEntryBlock()`. If `FC0.getNonLoopBlock()` is NOT `FC1.getEntryBlock()`, then `fuseGuardedLoops` will assert: `assert(FC0NonLoopBlock == FC1GuardBlock && "Loops are not adjacent");`. This is exactly the crash #166356! The fix assumes that if `FC0.ExitBlock->getSingleSuccessor() == FC1.getEntryBlock()`, then `FC0.getNonLoopBlock() == FC1.getEntryBlock()`. But this is NOT true if the guard branch bypasses the loop and goes to a different block, while the loop exit goes to `FC1.getEntryBlock()`.
- **Expected Issue**: The compiler will crash or produce incorrect code because the loops are not strictly adjacent, but the pass incorrectly identifies them as adjacent and attempts to fuse them.

### Strategy 2: Guard Branch Bypasses to Different Block

- **Target**: Modify a guarded loop test case so that the guard branch's non-loop successor is a different block (e.g., `ret void`), while the loop's exit block branches to the next loop's guard block.
- **Rationale**: The fix relies on `FC0.ExitBlock->getSingleSuccessor() == FC1.getEntryBlock()` to implicitly ensure `FC0.getNonLoopBlock() == FC1.getEntryBlock()`. However, `FC0.getNonLoopBlock()` is determined by the guard branch of `FC0`. If the guard branch of `FC0` evaluates to false, it jumps to `FC0.getNonLoopBlock()`. If the loop executes, it eventually exits to `FC0.ExitBlock`, which then jumps to `FC0.ExitBlock->getSingleSuccessor()`. These two destinations (`FC0.getNonLoopBlock()` and `FC0.ExitBlock->getSingleSuccessor()`) do not have to be the same block! If we construct a CFG where `FC0.ExitBlock` jumps to `FC1.GuardBlock`, but `FC0.GuardBranch` jumps to some other block (e.g., a return block or an error handling block), then `isStrictlyAdjacent` will return true (because `FC0.ExitBlock->getSingleSuccessor() == FC1.getEntryBlock()`), but `FC0.getNonLoopBlock()` will NOT be `FC1.GuardBlock`. When `fuseGuardedLoops` is called, it will hit the assertion `assert(FC0NonLoopBlock == FC1GuardBlock && "Loops are not adjacent");`.
- **Expected Issue**: The compiler will crash at `assert(FC0NonLoopBlock == FC1GuardBlock && "Loops are not adjacent");` in `fuseGuardedLoops`.

### Strategy 3: Guard Branch Bypasses to Different Block with Peeling

- **Target**: Modify a guarded loop test case with peeling enabled (`-loop-fusion-peel-max-count=3`) so that the guard branch's non-loop successor is a different block, while the loop's exit block branches to the next loop's guard block.
- **Rationale**: Similar to the previous strategy, but with loop peeling enabled. When peeling is enabled, `FC0.Peeled` is true, and `FC0.getNonLoopBlock()` returns `GuardBranch->getSuccessor(1)`. If the guard branch's non-loop successor is not `FC1.GuardBlock`, but `FC0.ExitBlock`'s successor is `FC1.GuardBlock`, the same assertion failure will occur. Peeling might complicate the CFG, making it more likely to expose edge cases in how `getNonLoopBlock()` and `ExitBlock` are updated.
- **Expected Issue**: The compiler will crash at `assert(FC0NonLoopBlock == FC1GuardBlock && "Loops are not adjacent");` in `fuseGuardedLoops`.

### Strategy 4: Guard Branch Bypasses to Exit Block Successor's Successor

- **Target**: Create a test case where `FC0.GuardBranch` branches to an empty block, which then branches to `FC1.GuardBlock`. `FC0.ExitBlock` branches directly to `FC1.GuardBlock`.
- **Rationale**: If `FC0.ExitBlock` branches to `FC1.GuardBlock`, but `FC0.GuardBranch` branches to a block that eventually reaches `FC1.GuardBlock` (e.g., an empty block in between), `FC0.getNonLoopBlock()` will be that empty block, not `FC1.GuardBlock`. The fix checks `FC0.ExitBlock->getSingleSuccessor() == FC1.getEntryBlock()`, which is true. But `FC0.getNonLoopBlock() == FC1.GuardBlock` is false. This will trigger the assertion.
- **Expected Issue**: The compiler will crash at `assert(FC0NonLoopBlock == FC1GuardBlock && "Loops are not adjacent");` in `fuseGuardedLoops`.

The fix checks `FC0.ExitBlock->getSingleSuccessor() == FC1.getEntryBlock()`, which is true. But `FC0.getNonLoopBlock()` is `FC0.ExitBlock`, not `FC1.GuardBlock`. This will trigger the assertion.
