## Test Strategies

### Strategy 1: Mismatched Operand Bitwidths

- **Target**: Operands of `llvm.clmul`
- **Rationale**: The `KnownBits::clmul` implementation asserts that the bitwidths of LHS and RHS match (`assert(BW == RHS.getBitWidth() && "Operand mismatch");`). While the intrinsic definition requires matching types, testing this edge case ensures robustness if malformed IR reaches this point.
- **Expected Issue**: The compiler might crash or produce incorrect known bits if the bitwidths of the operands do not match, although `llvm.clmul` requires matching bitwidths. If there's a way to create a mismatch (e.g., through a bug in another pass or a malformed IR that somehow bypasses the verifier), `APIntOps::clmul` will assert.

### Strategy 2: Poison/Undef in Operands

- **Target**: Operands of `llvm.clmul` with `undef` or `poison` values
- **Rationale**: The `KnownBits::clmul` logic uses `~LHS.Zero & ~LHS.One` to represent unknown bits. If an operand is partially poison or undef, the known bits might be overly optimistic or pessimistic. Testing with `undef` or `poison` elements in vector operands or scalar operands can reveal if the known bits are computed correctly.
- **Expected Issue**: The known bits calculation might incorrectly identify bits as known zero or one, leading to invalid optimizations later in the pipeline.

### Strategy 3: Vector Operands with Different Known Bits per Lane

- **Target**: Vector operands of `llvm.clmul` with different constants in each lane
- **Rationale**: `llvm.clmul` supports vector types. `computeKnownBits` for vectors typically computes the intersection of known bits across all demanded elements. If the demanded elements have different known bits, the resulting `KnownBits` might be too conservative or incorrect if the `clmul` logic doesn't handle the merged known bits properly.
- **Expected Issue**: The known bits calculation might be incorrect for vectors with different elements, leading to wrong optimizations.

### Strategy 4: Extreme Values (All Zeros, All Ones)

- **Target**: Operands of `llvm.clmul` set to 0 or -1
- **Rationale**: Carry-less multiplication with 0 should yield 0. With all ones, it's a complex XOR sum. Testing these extreme values ensures the `KnownBits::clmul` logic correctly handles the boundary conditions of the `ClMulOr` helper lambda.
- **Expected Issue**: The known bits calculation might be incorrect when one operand is all zeros or all ones, leading to missed optimizations or wrong code.

### Strategy 5: 1-bit Integer Boundary Conditions

- **Target**: `llvm.clmul` with `i1` operands
- **Rationale**: The `clmul` operation on `i1` is equivalent to logical AND. The `KnownBits::clmul` logic should correctly handle `i1` without out-of-bounds shifts or incorrect loop bounds in the `ClMulOr` lambda.
- **Expected Issue**: The known bits calculation might be incorrect for 1-bit integers, leading to wrong optimizations.
