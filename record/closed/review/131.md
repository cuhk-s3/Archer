## Test Strategies

### Strategy 1: Fuse loops with induction variables used after the loop

- **Target**: Create a test case with two fusible loops where the induction variable of the first loop is used after the fused loop.
- **Rationale**: The patch adds `SE.forgetLoop(FC0.L)` after `SE.forgetBlockAndLoopDispositions()` to forget cached SCEV values, including the induction variable, which may have changed after fusion. If we fuse two loops where the induction variable of the first loop is used after the loop (e.g., in a LCSSA phi node), the SCEV for that induction variable might be cached. After fusion, the loop bounds or the induction variable's evolution might change (e.g., if the second loop has a different trip count or if the loops are peeled/modified). If the SCEV is not forgotten, subsequent passes might use the stale SCEV, leading to incorrect optimizations or crashes.
- **Expected Issue**: The compiler might crash or produce incorrect code due to stale SCEV information for the induction variable or other loop-carried values.

### Strategy 2: Fuse loops with loop-carried values other than induction variables

- **Target**: Create a test case with two fusible loops where a reduction or other loop-carried value from the first loop is used after the fused loop.
- **Rationale**: Similar to the induction variable, other loop-carried values (e.g., reductions) might have their SCEVs cached. When the loops are fused, the structure of the loop changes, and the SCEVs for these values might become invalid. The patch aims to clear these cached values by calling `SE.forgetLoop(FC0.L)`. Testing with various loop-carried values ensures that the fix correctly invalidates their SCEVs.
- **Expected Issue**: The compiler might crash or produce incorrect code due to stale SCEV information for values defined in the loop.

### Strategy 3: Fuse guarded loops with induction variables

- **Target**: Create a test case with two fusible guarded loops where the induction variable of the first loop is used after the fused loop.
- **Rationale**: The patch modifies `fuseGuardedLoops` as well as `performFusion`. Guarded loops have a different control flow structure (with a guard branch bypassing the loop). Fusing guarded loops involves rewiring the guard branches and exit blocks. If the SCEVs for the induction variables or other values in the guarded loops are not properly forgotten, it could lead to issues. Testing guarded loops ensures the fix applies correctly to this specific control flow structure.
- **Expected Issue**: The compiler might crash or produce incorrect code due to stale SCEV information for the guarded loop.

- **Target**: Create a test case with two fusible loops where the first loop is peeled, and its induction variable is used after the fused loop.
- **Rationale**: Loop fusion can handle peeled loops (where the first iteration is peeled off). Peeling changes the loop structure and the initial values of induction variables. Fusing peeled loops might have different effects on SCEVs compared to non-peeled loops. Testing peeled loops ensures the fix correctly invalidates SCEVs in this scenario.
- **Expected Issue**: The compiler might crash or produce incorrect code due to stale SCEV information for the peeled loop.

### Strategy 5: Fuse nested loops with induction variables
