## Test Strategies

### Strategy 1: Signaling NaN Canonicalization

- **Target**: A test case where `exp` is called with a signaling NaN and the result is used in a context that only demands NaN (e.g., `fcmp uno`).
- **Rationale**: The patch changes the behavior when only NaN is demanded. Previously, it returned the original operand, dropping the signaling NaN quieting. Now, it returns a canonical quiet NaN if the demanded mask only requires NaN. We need to test if a signaling NaN input is correctly converted to a quiet NaN when only NaN is demanded.
- **Expected Issue**: The optimization might incorrectly return the original operand instead of a canonical NaN when the demanded mask only requires NaN and the input is a signaling NaN.

### Strategy 2: Signaling NaN with Other Demanded Classes

- **Target**: A test case where `exp` is called with a signaling NaN and the result is used in a context that demands NaN and other classes (e.g., `fcmp ueq`).
- **Rationale**: The patch adds a check `if ((DemandedMask & ~fcNan) == fcNone)` to return a canonical quiet NaN. If other classes are demanded, it falls through to returning the original operand. We need to ensure that if the input is a signaling NaN and other classes are demanded, the signaling NaN is not incorrectly propagated if it should be quieted.
- **Expected Issue**: The optimization might incorrectly return a non-canonical NaN or the original operand when the demanded mask requires NaN and other classes, but the input is known to be a signaling NaN.

### Strategy 3: Quiet NaN Propagation

- **Target**: A test case where `exp` is called with a quiet NaN with a specific payload and the result is used in a context that only demands NaN.
- **Rationale**: The patch returns a canonical quiet NaN if only NaN is demanded. If the input is already a quiet NaN, returning a canonical quiet NaN is correct but might lose payload. We need to test if the payload of a quiet NaN is preserved when other classes are demanded, or if it's correctly canonicalized when only NaN is demanded.
- **Expected Issue**: The optimization might incorrectly return a canonical NaN when the demanded mask requires NaN and other classes, but the input is known to be a quiet NaN.

### Strategy 4: Non-NaN Input with NaN Demanded

- **Target**: A test case where `exp` is called with a known non-NaN value and the result is used in a context that only demands NaN.
- **Rationale**: The patch returns a canonical quiet NaN if only NaN is demanded. If the input is known to be a non-NaN value (e.g., a positive number), returning a canonical quiet NaN is incorrect. The optimization should return poison or the original operand depending on the demanded mask.
- **Expected Issue**: The optimization might incorrectly return a canonical NaN when the demanded mask requires NaN and other classes, but the input is known to be a non-NaN value.

### Strategy 5: Exp Result Canonicalization

- **Target**: A test case where `exp` is called with a known value (e.g., positive infinity) and the result is used in a context that demands the specific class.
- **Rationale**: The patch adds `FPClassTest ValidResults = DemandedMask & Known.KnownFPClasses; return getFPClassConstant(VTy, ValidResults, /*IsCanonicalizing=*/true);` at the end of the `exp` handling. This means if the result is known to be a specific class (e.g., positive infinity), it will return a canonical constant for that class. We need to test if this canonicalization is correct and doesn't lose information or introduce incorrect values.
- **Expected Issue**: The optimization might incorrectly return a canonical NaN when the demanded mask requires NaN and other classes, but the input is known to be a non-NaN value.
