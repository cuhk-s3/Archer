## Test Strategies

### Strategy 1: Floating-point extended reduction without partial reduction

- **Target**: A loop with a floating-point reduction where the operand is an extended floating-point value (e.g., `fpext`), but the reduction does not qualify as a partial reduction (e.g., the scale factor doesn't match, or it's not a supported partial reduction pattern).
- **Rationale**: The patch adds `RedTy->isFloatingPointTy()` checks before calling `getExtendedReductionCost` in `VPlanRecipes.cpp` and `VPlanTransforms.cpp`. If we can create a scenario where a floating-point extended reduction is matched but it's NOT a partial reduction, the old code would have called `getExtendedReductionCost` and crashed or returned an invalid cost. The new code prevents this. We should test if the compiler handles floating-point extended reductions correctly (i.e., rejects them or handles them via partial reduction).
- **Expected Issue**: The compiler might crash or generate incorrect code because `getExtendedReductionCost` is called with a floating-point type, which it does not support.

### Strategy 2: Floating-point partial reduction without FastMathFlags

- **Target**: A loop with a floating-point partial reduction (e.g., `fadd` of `fpext` values) but without `fast` math flags on the `fadd`.
- **Rationale**: The patch passes `RedR->getFastMathFlags()` to `getPartialReductionCost` if `RedTy->isFloatingPointTy()`. If the loop has a floating-point partial reduction but lacks the necessary FastMathFlags (like `reassoc` and `contract`), the target's `getPartialReductionCost` might reject it or crash if it expects them. We should test a loop with `fadd` partial reduction but without `fast` or `reassoc contract` flags to ensure it's correctly rejected or costed as invalid.
- **Expected Issue**: The compiler might crash or generate incorrect code because `getPartialReductionCost` is called without FastMathFlags for a floating-point reduction, which might be required by the target (e.g., AArch64 requires `allowReassoc` and `allowContract`).

### Strategy 3: Floating-point partial reduction with FastMathFlags

- **Target**: A loop with a floating-point partial reduction (e.g., `fadd` of `fpext` values) and `fast` math flags on the `fadd`.
- **Rationale**: The patch enables support for floating-point partial reductions (like FDOT) by passing the correct FastMathFlags to `getPartialReductionCost`. We should test a loop with a valid floating-point partial reduction (e.g., `fadd` of `fpext` values) and the necessary FastMathFlags (`reassoc contract` or `fast`) to ensure it is correctly vectorized using the partial reduction intrinsic.
- **Expected Issue**: The compiler might incorrectly vectorize the loop using a partial reduction, or fail to vectorize it, due to incorrect cost calculation or missing FastMathFlags.

### Strategy 4: Floating-point extended reduction with fmul

- **Target**: A loop with a floating-point multiply-accumulate reduction (e.g., `sum += (float)a[i] * (float)b[i]`) with and without FastMathFlags.
- **Rationale**: The patch modifies `tryToMatchAndCreateExtendedReduction` to handle `fpext`. It also modifies `tryToMatchAndCreateMulAccumulateReduction` to handle `fadd` and `fmul`. We should test a loop with a floating-point multiply-accumulate reduction (e.g., `fadd` of `fmul` of `fpext` values) to ensure it's correctly handled and doesn't crash when querying costs.
- **Expected Issue**: The compiler might crash or generate incorrect code because `getExtendedReductionCost` is called with a floating-point type.

### Strategy 5: Floating-point extended reduction with fsub

- **Target**: A loop with a floating-point `fsub` reduction of extended values (e.g., `sum -= (float)a[i]`).
- **Rationale**: The patch modifies `tryToMatchAndCreateMulAccumulateReduction` to handle `fadd`. What about `fsub`? The code checks `Opcode != Instruction::Add && Opcode != Instruction::Sub && Opcode != Instruction::FAdd`. It doesn't explicitly check `FSub`. If a loop has an `fsub` reduction, it might fall through and cause issues. We should test a loop with an `fsub` reduction of extended values.
- **Expected Issue**: The compiler might crash or generate incorrect code because `getExtendedReductionCost` is called with a floating-point type.
