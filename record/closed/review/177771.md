## Test Strategies

### Strategy 1: Infer nnan/ninf on canonicalize with known non-NaN/Inf input

- **Target**: A test case with `llvm.canonicalize` where the input is known to be non-NaN or non-Inf (e.g., from an `uitofp` or `sitofp` instruction, or an instruction with `nnan`/`ninf` flags).
- **Rationale**: The patch infers fast math flags (like `nnan` and `ninf`) for `llvm.canonicalize` based on the known floating-point classes of its input. If the input is known to never be NaN or Inf, the `canonicalize` instruction will get `nnan` or `ninf` flags. We should test if this inference works correctly and if it drops UB-implying attributes when adding these flags.
- **Expected Issue**: The canonicalize instruction might incorrectly gain `nnan` or `ninf` flags, which could be propagated to other instructions or cause incorrect optimizations if the input can actually be NaN or Inf.

### Strategy 2: Infer nnan/ninf on canonicalize with known non-NaN/Inf input from FMF

- **Target**: A test case with `llvm.canonicalize` where the input is an instruction with `nnan` and/or `ninf` flags.
- **Rationale**: If the input to `llvm.canonicalize` has `nnan` or `ninf` flags, the `canonicalize` instruction should inherit these flags because the input is known not to be NaN or Inf.
- **Expected Issue**: The canonicalize instruction might not gain the `nnan` or `ninf` flags, missing an optimization opportunity.

### Strategy 3: Do not infer nnan/ninf on canonicalize with unknown input

- **Target**: A test case with `llvm.canonicalize` where the input is a generic floating-point value without any known properties.
- **Rationale**: If the input to `llvm.canonicalize` can be NaN or Inf, the `canonicalize` instruction should not get `nnan` or `ninf` flags.
- **Expected Issue**: The canonicalize instruction might incorrectly gain `nnan` or `ninf` flags, leading to incorrect optimizations.

### Strategy 4: Infer nnan/ninf on canonicalize with UB-implying attributes

- **Target**: A test case with `llvm.canonicalize` that has UB-implying attributes (like `noundef`) and where the input is known to be non-NaN or non-Inf.
- **Rationale**: The patch calls `dropUBImplyingAttrsAndMetadata()` when inferring new fast math flags. We should test if this correctly drops attributes like `noundef` when `nnan` or `ninf` are inferred.
- **Expected Issue**: The canonicalize instruction might incorrectly gain `nnan` or `ninf` flags, or UB-implying attributes might not be dropped correctly.

### Strategy 5: Infer nnan/ninf on canonicalize with denormal mode

- **Target**: A test case with `llvm.canonicalize` where the function has a specific denormal mode attribute, and the input is known to be non-NaN or non-Inf.
- **Rationale**: The behavior of `llvm.canonicalize` depends on the denormal mode. We should test if the inference of fast math flags works correctly under different denormal modes (e.g., `ieee`, `preserve-sign`, `positive-zero`).
- **Expected Issue**: The canonicalize instruction might incorrectly gain `nnan` or `ninf` flags, leading to incorrect optimizations.
