## Test Strategies

### Strategy 1: Basic Unstable Pointer PtrToAddr

- **Target**: A simple function with a `ptrtoaddr` instruction operating on an unstable pointer (e.g., `addrspace(2)` if configured as unstable).
- **Rationale**: The patch prevents SCEV from modeling `ptrtoaddr` for pointers with unstable representations. Testing a basic `ptrtoaddr` on an unstable pointer (e.g., in an address space defined as unstable in the DataLayout) ensures that SCEV correctly returns `SCEVUnknown` instead of a computed expression.
- **Expected Issue**: SCEV incorrectly models the address of the unstable pointer as a stable SCEV expression (e.g., SCEVUnknown with a specific type, or an AddRec), which could lead to invalid optimizations like hoisting or folding that assume the address is constant.

### Strategy 2: GEP of Unstable Pointer PtrToAddr

- **Target**: A `ptrtoaddr` instruction where the operand is a `getelementptr` derived from an unstable pointer.
- **Rationale**: SCEV often looks through GEPs. If an unstable pointer is used in a GEP, and then `ptrtoaddr` is applied, SCEV must still recognize the underlying pointer representation as unstable and return `SCEVUnknown`.
- **Expected Issue**: SCEV might try to sink the cast through the GEP and model the base pointer's address, leading to incorrect assumptions about the stability of the derived address.

### Strategy 3: PHI of Unstable Pointers PtrToAddr

- **Target**: A loop with a PHI node of unstable pointers, where `ptrtoaddr` is applied to the PHI node.
- **Rationale**: PHI nodes in loops can create AddRec expressions. If the PHI node contains unstable pointers, taking `ptrtoaddr` of the PHI should result in `SCEVUnknown` rather than an AddRec, because the addresses can change arbitrarily.
- **Expected Issue**: SCEV might incorrectly model the PHI node's address evolution, assuming the addresses of the incoming unstable pointers are stable across loop iterations.

### Strategy 4: Stable vs Unstable Pointer Comparison

- **Target**: Two similar functions or loops, one using a stable pointer (e.g., `addrspace(0)`) and one using an unstable pointer, both applying `ptrtoaddr`.
- **Rationale**: To ensure the fix is precise, we should test both a stable pointer and an unstable pointer in the same context. The stable pointer's `ptrtoaddr` should be modeled correctly (e.g., sinking the cast), while the unstable pointer's `ptrtoaddr` should be `SCEVUnknown`.
- **Expected Issue**: SCEV might return `SCEVUnknown` for the stable pointer, missing optimization opportunities, or compute an expression for the unstable pointer, leading to miscompilations.

### Strategy 5: Unstable Pointer AddRec PtrToAddr

- **Target**: A loop where an unstable pointer is incremented (forming a pointer AddRec), and `ptrtoaddr` is applied to the induction variable.
- **Rationale**: If an unstable pointer is incremented in a loop (e.g., via GEP), it forms an AddRec in the pointer domain. Applying `ptrtoaddr` to this AddRec should be blocked by the new check, returning `SCEVUnknown`.
- **Expected Issue**: SCEV might crash or incorrectly compute an AddRec for the unstable pointer's address, violating the assumption that unstable pointers cannot have predictable address evolutions.
