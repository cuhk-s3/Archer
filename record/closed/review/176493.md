## Test Strategies

### Strategy 1: Uninitialized Global Variable as Count

- **Target**: A test case where `Count` is a load from a tracked global variable. The global variable is initially `unknown` but later stored with a small constant range (e.g., `[0, 2)`).
- **Rationale**: When the load is initially `unknown`, `SCCPSolver` evaluates `get.vector.length` with an empty set for `Count`. `getUnsignedMax()` on an empty set returns `UINT_MAX`, causing the result to be `[0, MaxLanes + 1)`. Later, when the global variable resolves to `[0, 2)`, the result should be `[0, 2)`, but because lattice values only grow, it remains `[0, MaxLanes + 1)`, losing precision.
- **Expected Issue**: The range of `get.vector.length` will be overly conservative (e.g., `[0, 5)`) instead of the precise range (e.g., `[0, 2)`), missing optimization opportunities.

### Strategy 2: Both Count and VF are Unknown

- **Target**: A test case where both `Count` and `VF` are loads from tracked global variables, initially `unknown`.
- **Rationale**: If both are `unknown`, both `getUnsignedMax()` calls return `UINT_MAX`. `umin(UINT_MAX, UINT_MAX) + 1` overflows to `0`. `ConstantRange::getNonEmpty(0, 0)` returns a full set. The instruction is marked as overdefined. Later when they resolve to small ranges, the instruction remains overdefined.
- **Expected Issue**: The instruction is permanently marked as overdefined, completely losing all range information.

### Strategy 3: Unknown VF (Vectorization Factor)

- **Target**: A test case where `VF` is a load from a tracked global variable, initially `unknown`, later resolving to a small constant (e.g., `2`).
- **Rationale**: Similar to Strategy 1, if `VF` is `unknown`, `MaxLanes` is an empty set. `getUnsignedMax()` returns `UINT_MAX`. The result becomes `[0, Count + 1)`. Later when `VF` resolves to `2`, the result should be bounded by `2`, but it remains bounded by `Count`, losing precision.
- **Expected Issue**: The range of `get.vector.length` will be overly conservative, bounded only by `Count` instead of `min(Count, VF)`.

### Strategy 4: Undef Value as Count

- **Target**: A test case where `Count` is explicitly `undef`.
- **Rationale**: If `Count` is `undef`, `asConstantRange` returns a full set. `getUnsignedMax()` on a full set returns `UINT_MAX`. The result becomes `[0, MaxLanes + 1)`. While this is technically correct for `undef` (since `undef` can be anything), it might be better to return `unknown` or `undef` to allow further optimizations.
- **Expected Issue**: The result is not folded to a constant (e.g., `0`), missing an optimization opportunity.

### Strategy 5: PHI Node with Undef Incoming Value

- **Target**: A test case where `Count` is a PHI node with one incoming value being `undef` and another being a small constant (e.g., `1`).
- **Rationale**: `SCCPSolver` often struggles with `undef` in PHI nodes. If the PHI node evaluates to a range that includes `undef`, `get.vector.length` might compute a range based on the full set (due to `undef`), resulting in `[0, MaxLanes + 1)`. If we could choose `undef` to be `1`, the result could be precisely `[0, 2)`.
- **Expected Issue**: The range is overly conservative because `undef` forces a full set, missing the opportunity to assume a specific value for `undef`.
