### Strategy 1: Incorrect nnan inference for sqrt with negative source

- **Target**: A test with `sqrt` where the source is known to be negative (e.g., `-1.0`) and not NaN, and the result of `sqrt` is used in a context that does not demand NaN (e.g., `is.fpclass(sqrt(-1.0), fcPosNormal)`).
- **Rationale**: The `inferFastMathValueFlags` function infers `nnan` if all sources are known never NaN and the valid results don't include NaN. For `sqrt`, a negative source (which is not NaN) produces a NaN result. If the demanded mask doesn't demand NaN, `ValidResults` won't include NaN. Then `inferFastMathValueFlags` checks if `KnownSrc` is never NaN. If it is, it infers `nnan` for the `sqrt` instruction. However, `sqrt` of a negative number IS NaN. If we infer `nnan`, we are stating the result is not NaN, which is false and could lead to UB or incorrect folds later if the `sqrt` result is used in a way that relies on `nnan`. Wait, if the demanded mask doesn't demand NaN, does it matter? Yes, because `setFastMathFlags` modifies the instruction itself, which might have other uses? No, `SimplifyDemandedUseFPClass` is only called for one-use instructions. But wait, `setFastMathFlags` modifies the instruction. If the instruction is later used, or if the `nnan` flag triggers UB (poison) when the result is actually NaN, and that poison propagates to the demanded use, it could be a miscompilation.
- **Expected Issue**: The inferred fast math flags might be incorrect if the demanded mask does not demand NaN or Inf, but the instruction itself could produce them and they are used elsewhere (though SimplifyDemandedUseFPClass is only called for one-use instructions). Wait, the issue is that `inferFastMathValueFlags` takes `ArrayRef<KnownFPClass> KnownSrcs`. For `sqrt`, it passes `{KnownSrc}`. If `KnownSrc` is known never NaN, it infers `nnan`. But `sqrt` can produce NaN even if the source is not NaN (e.g., `sqrt(-1.0)`). So if `KnownSrc` is known never NaN, `inferFastMathValueFlags` will incorrectly infer `nnan` for `sqrt`!

### Strategy 2: Poison propagation from incorrectly inferred nnan on sqrt

- **Target**: `llvm.is.fpclass(llvm.sqrt(x), fcPosNormal)` where `x` can be negative.
- **Rationale**: If `sqrt` is given a negative number, it returns `NaN`. If we infer `nnan` on the `sqrt`, it will return `poison`. If the demanded mask doesn't demand `NaN`, it means the user of the `sqrt` only cares about certain classes. For example, `llvm.is.fpclass(sqrt(x), fcPosNormal)`. If `x = -1.0`, `sqrt(x)` is `NaN`. `is.fpclass(NaN, fcPosNormal)` is `false`. But if `sqrt` is marked `nnan`, it returns `poison`. `is.fpclass(poison, fcPosNormal)` is `poison`. This changes the result from `false` to `poison`, which is a miscompilation.
- **Expected Issue**: The `sqrt` instruction is incorrectly marked with `nnan`, which makes it return `poison` instead of `NaN`. This `poison` might then be used in the demanded context, changing the behavior from a well-defined `NaN` (which might evaluate to `false` in a check) to `poison` (which can trigger UB or evaluate to anything).

### Strategy 3: Incorrect ninf inference for sqrt with +Inf source

- **Target**: `llvm.is.fpclass(llvm.sqrt(x), fcPosNormal)` where `x` is a negative number.
- **Rationale**: Wait, `sqrt(+Inf)` is `+Inf`. If `KnownSrc` is known never Inf, `sqrt` is never Inf. So `ninf` inference is correct for `sqrt` if the source is never Inf. What about `nnan`? `sqrt` produces NaN if the source is negative. The patch adds: `if ((DemandedMask & fcNan) == fcNone) KnownSrc.knownNot((fcNegative & ~fcNegZero) | fcNan);` This means if the result is not demanded to be NaN, we assume the source is not negative and not NaN. Then `KnownSrc` will not contain negative values. Then `Known = KnownFPClass::sqrt(KnownSrc, Mode);` will compute the known classes of the result. Since `KnownSrc` has no negative values, `Known` will have no NaN. Then `ValidResults = DemandedMask & Known.KnownFPClasses` will have no NaN. Then `inferFastMathValueFlags` will see `ValidResults` has no NaN, and `KnownSrc` has no NaN. So it will infer `nnan`! But wait, `KnownSrc` was modified by `knownNot((fcNegative & ~fcNegZero) | fcNan)` based on the assumption that the result is not NaN. This is a backward propagation of demanded bits. If the result is not demanded to be NaN, we assume the source is not negative. But this assumption is ONLY valid for the demanded use! We cannot use this assumed `KnownSrc` to infer `nnan` on the `sqrt` instruction itself, because the `sqrt` instruction might actually receive a negative value at runtime! If it receives a negative value, it will produce NaN. If we marked it `nnan`, it will produce poison. And this poison will flow into the demanded use. Is this allowed? If the demanded use doesn't demand NaN, does it mean it doesn't care if the value is NaN? No! "Demanded" means the use only cares about the result IF it falls into the demanded classes. Wait, no. `SimplifyDemandedUseFPClass` means "we only need the result to be correct if it is in the demanded mask". If the actual value is NaN, and NaN is not demanded, the user will just return some value (e.g., `false` for `is.fpclass`). If we change the `sqrt` to `nnan`, it produces `poison`. `is.fpclass(poison)` is `poison`. This is a refinement, but `poison` can cause UB if branched upon. Wait, `is.fpclass` returning `poison` instead of `false` is a miscompilation!
- **Expected Issue**: The `sqrt` instruction is incorrectly marked with `ninf`, which makes it return `poison` instead of `+Inf`. This `poison` might then be used in the demanded context, changing the behavior from a well-defined `+Inf` to `poison`.
