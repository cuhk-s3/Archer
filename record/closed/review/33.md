## Test Strategies

### Strategy 1: Test trunc with demanded mask not demanding NaN but source is NaN

- **Target**: A `trunc` instruction where the demanded mask does not include `fcNan` (e.g., due to a `select` or `fcmp` that ignores NaN), but the source operand is a NaN.
- **Rationale**: The patch propagates NaN from source to result for rounding intrinsics. If the demanded mask does not demand NaN, `KnownSrc.knownNot(fcNan)` is called. If the source is actually a NaN, this creates a contradiction where `KnownSrc` says it's not NaN, but the actual value is NaN. This might lead to incorrect simplifications if the source is a constant NaN or if other optimizations rely on this.
- **Expected Issue**: The `trunc` instruction will be incorrectly simplified to `poison` or a wrong constant, leading to miscompilation.

### Strategy 2: Test trunc with demanded mask demanding only positive values but source is negative zero

- **Target**: A `trunc` instruction where the demanded mask demands `fcPosZero` but not `fcNegZero`, and the source operand is `-0.0`.
- **Rationale**: The patch adds `if (DemandedMask & fcPosZero) DemandedSrcMask |= fcPosSubnormal | fcPosNormal;`. If the demanded mask demands `fcPosZero` but not `fcNegZero`, and the source is `-0.0`, the `roundToIntegral` function might incorrectly assume the result cannot be `-0.0` if it doesn't properly handle the sign of zero. The patch says `if (KnownSrc.isKnownNever(fcNegFinite)) Known.knownNot(fcNegFinite);`. If the source is `-0.0`, it is `fcNegFinite`.
- **Expected Issue**: The `trunc` instruction will be incorrectly simplified to `poison` or a wrong constant, leading to miscompilation.

### Strategy 3: Test trunc with multi-unit FP type (ppc_fp128) and infinity

- **Target**: A `trunc` instruction and a `floor` instruction on `ppc_fp128` type with infinity as the source operand.
- **Rationale**: The patch has a special case for `ppc_fp128`: `if (IsTrunc || !IsMultiUnitFPType) { ... pass through infinities ... }`. This means for `trunc` on `ppc_fp128`, it *does* pass through infinities. But for other rounding intrinsics on `ppc_fp128`, it does *not* pass through infinities. We should test if `trunc` on `ppc_fp128` correctly handles infinities and if other rounding intrinsics correctly do not pass through infinities.
- **Expected Issue**: The `trunc` instruction will be incorrectly simplified to `poison` or a wrong constant, leading to miscompilation.

### Strategy 4: Test trunc with demanded mask demanding only positive values but source is negative subnormal

- **Target**: A `trunc` instruction where the demanded mask demands `fcPosZero` but not `fcNegZero`, and the source operand is a negative subnormal.
- **Rationale**: The patch says `if (DemandedMask & fcNegZero) DemandedSrcMask |= fcNegSubnormal | fcNegNormal;`. If the demanded mask demands `fcPosZero` but not `fcNegZero`, and the source is a negative subnormal, `trunc` of a negative subnormal is `-0.0`. If the demanded mask doesn't demand `-0.0`, the result might be incorrectly simplified.
- **Expected Issue**: The `trunc` instruction will be incorrectly simplified to `poison` or a wrong constant, leading to miscompilation.

### Strategy 5: Test trunc with demanded mask demanding only negative values but source is positive subnormal

- **Target**: A `trunc` instruction where the demanded mask demands `fcNegZero` but not `fcPosZero`, and the source operand is a positive subnormal.
- **Rationale**: Similar to the previous strategy, `trunc` of a positive subnormal is `+0.0`. If the demanded mask demands `fcNegZero` but not `fcPosZero`, and the source is a positive subnormal, the result might be incorrectly simplified.
- **Expected Issue**: The `trunc` instruction will be incorrectly simplified to `poison` or a wrong constant, leading to miscompilation.

I have found a clear miscompilation.
The bug is:
When `SimplifyDemandedUseFPClass` is called for `is.fpclass(trunc(x), Mask)`, it calls `SimplifyDemandedFPClass` on `trunc(x)` with `DemandedMask = Mask`.
If `Mask` is `fcPosZero` (32), `DemandedMask` is `fcPosZero`.
Then `trunc` computes `DemandedSrcMask`.
```cpp
      // Zero results imply valid subnormal sources.
      if (DemandedMask & fcNegZero)
        DemandedSrcMask |= fcNegSubnormal | fcNegNormal;

      if (DemandedMask & fcPosZero)
        DemandedSrcMask |= fcPosSubnormal | fcPosNormal;
```
Since `DemandedMask` is `fcPosZero`, `DemandedSrcMask` becomes `fcPosZero | fcPosSubnormal | fcPosNormal`.
Notice that `fcNegZero` is NOT in `DemandedSrcMask`.
Then it calls `SimplifyDemandedFPClass(x, DemandedSrcMask)`.
If `x` is `-0.0` (which is `fcNegZero`), `SimplifyDemandedFPClass` sees that `x`'s class (`fcNegZero`) is disjoint from `DemandedSrcMask`.
So it replaces `x` with `poison`!
Then `trunc(poison)` becomes `poison`.
Then `is.fpclass(poison, 32)` becomes `poison`.
Then `poison` can be evaluated to `true` (1).
But the original IR evaluates to `false` (0) because `trunc(-0.0)` is `-0.0`, and `is.fpclass(-0.0, 32)` is `false`.
This is a miscompilation!
I will report this bug.
