## Test Strategies

### Strategy 1: Division by Zero in URem

- **Target**: A test case with `urem x, n` where `n` can be zero, and the result is used in a comparison.
- **Rationale**: The patch adds the fact `result < n` for `urem x, n`. However, if `n` is zero, `urem` is undefined behavior. If the pass extracts this fact before the division by zero is executed or on a path where it's not guaranteed to execute, it might use the fact `result < 0` (which is impossible for unsigned integers) to incorrectly optimize other parts of the code.
- **Expected Issue**: The pass might incorrectly assume the remainder is less than the divisor even when the divisor is zero, leading to incorrect optimization or undefined behavior.

### Strategy 2: Division by Zero in UDiv

- **Target**: A test case with `udiv x, n` where `n` can be zero, and the result is used in a comparison.
- **Rationale**: The patch adds the fact `result <= x` for `udiv x, n`. If `n` is zero, `udiv` is undefined behavior. Similar to `urem`, extracting facts from an instruction that might cause UB if executed can lead to miscompilations if the fact is used on paths where the UB is not guaranteed to happen.
- **Expected Issue**: The pass might incorrectly assume the quotient is less than or equal to the dividend even when the divisor is zero, leading to incorrect optimization.

### Strategy 3: Poison Value from UDiv Exact

- **Target**: A test case with `udiv exact x, n` where the division is not exact, and the result is used in a comparison.
- **Rationale**: The patch checks `isGuaranteedNotToBePoison(BO)` before adding facts. However, `udiv exact` can produce a poison value if the division is not exact. If `isGuaranteedNotToBePoison` does not correctly handle `exact` flags or if the poison value is used in a way that propagates incorrect constraints, it could lead to miscompilations.
- **Expected Issue**: The pass might incorrectly use facts from a poison value, leading to incorrect optimization.

### Strategy 4: Poison Value from Operands

- **Target**: A test case with `urem` or `udiv` where one of the operands is a poison value, and the result is used in a comparison.
- **Rationale**: If the operands of `urem` or `udiv` are poison, the result is poison. The patch checks `isGuaranteedNotToBePoison(BO)`, but if this check is flawed or if the poison value is introduced later, the extracted facts might be invalid.
- **Expected Issue**: The pass might incorrectly use facts from a poison value, leading to incorrect optimization.

### Strategy 5: Vector Types in URem/UDiv

- **Target**: A test case with `urem` or `udiv` on vector types, and the result is used in a comparison.
- **Rationale**: The patch adds facts for `urem` and `udiv` using `ICMP_ULT` and `ICMP_ULE`. ConstraintElimination typically works on scalar integers. If `urem` or `udiv` is applied to vector types, adding these facts might cause crashes or incorrect behavior if the pass does not properly handle vector types.
- **Expected Issue**: The pass might incorrectly assume the remainder is less than the divisor for vector types, leading to incorrect optimization or crashes.
