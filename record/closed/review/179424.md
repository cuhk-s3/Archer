## Test Strategies

### Strategy 1: Cross-function Dominator Tree Check via Call Site Argument

- **Target**: Create a test case with a function call where the argument is a floating-point value. Add a dominating condition (e.g., a branch based on `fcmp`) or an `llvm.assume` in the caller function that affects the argument value. Ensure `AANoFPClass` is evaluated for the call site argument.
- **Rationale**: The fix changes `getAssociatedFunction()` to `getAnchorScope()` when setting up the `SimplifyQuery` context for `computeKnownFPClass`. `getAssociatedFunction()` for a call site argument returns the callee function, while `getAnchorScope()` returns the caller function. If `getAssociatedFunction()` is used, the `DominatorTree` and `AssumptionCache` retrieved will be for the callee function. However, the context instruction `CtxI` (which is the call instruction itself) belongs to the caller function. When `computeKnownFPClass` uses the `DominatorTree` to check dominance (e.g., `DT->dominates(Edge, CtxI->getParent())`), it will assert because the basic blocks belong to different functions. By creating a test case where `AANoFPClass` is evaluated for a call site argument and there are dominating conditions or assumptions in the caller, we can trigger this cross-function dominator tree check.
- **Expected Issue**: The compiler will crash with an assertion failure in `computeKnownFPClass` or `isValidAssumeForContext` because the dominator tree check is performed across different functions.

### Strategy 2: Cross-function Dominator Tree Check via Returned Value

- **Target**: Create a test case with a function call that returns a floating-point value. Add a dominating condition or assumption in the caller that affects the returned value, and ensure `AANoFPClass` is evaluated for the call site returned position.
- **Rationale**: Similar to the call site argument case, if `AANoFPClass` is evaluated for a returned value of a call site (`IRPosition::callsite_returned`), `getAssociatedFunction()` would return the callee function, while the context instruction is in the caller. If there are dominating conditions or assumptions in the caller that affect the returned value (e.g., if the returned value is used in a way that triggers `computeKnownFPClass` with the call instruction as context), it could lead to a cross-function dominator tree check. The fix ensures `getAnchorScope()` (the caller) is used.
- **Expected Issue**: The compiler will crash with an assertion failure in `computeKnownFPClass` or `isValidAssumeForContext` due to cross-function dominator tree queries.

### Strategy 3: Cross-function Dominator Tree Check via Callback Argument

- **Target**: Create a test case using a callback function (like `!callback` metadata or recognized callback APIs). Pass a floating-point value as the callback argument. Add a dominating condition or assumption in the caller affecting this value.
- **Rationale**: For callback calls (e.g., `pthread_create`), `getAssociatedFunction()` for a callback argument returns the callback callee function. The context instruction is the call site in the caller. If `AANoFPClass` is evaluated for the callback argument and there are dominating conditions in the caller, using `getAssociatedFunction()` would lead to a cross-function dominator tree check. The fix to use `getAnchorScope()` correctly uses the caller function.
- **Expected Issue**: The compiler will crash with an assertion failure in `computeKnownFPClass` or `isValidAssumeForContext` due to cross-function dominator tree queries.

- **Rationale**: If an inline assembly call has floating-point arguments, and `AANoFPClass` is evaluated for these arguments, `getAssociatedFunction()` might return null or something unexpected, but if it somehow resolves to a different scope than the caller, it could cause issues. While less likely to have a distinct associated function, ensuring `getAnchorScope()` is used guarantees the caller's dominator tree is used for the context instruction (the inline asm call).
- **Expected Issue**: The compiler will crash with an assertion failure in `computeKnownFPClass` or `isValidAssumeForContext` due to cross-function dominator tree queries.

### Strategy 5: Cross-function Dominator Tree Check via Invoke Instruction

- **Target**: Create a test case with an `invoke` instruction passing a floating-point argument. Add a dominating condition or assumption in the caller affecting this argument.
- **Rationale**: An `invoke` instruction is a `CallBase` and can have call site arguments. If `AANoFPClass` is evaluated for an argument of an `invoke`, the context instruction is the `invoke` itself. If `getAssociatedFunction()` returns the callee, the dominator tree check will fail because the `invoke` is in the caller. This is a variation of the call site argument strategy but uses `invoke` to ensure exception handling paths don't mask the issue.
- **Expected Issue**: The compiler will crash with an assertion failure in `computeKnownFPClass` or `isValidAssumeForContext` due to cross-function dominator tree queries.
