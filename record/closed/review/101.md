## Test Strategies

### Strategy 1: Missing TBAA on Store

- **Target**: A test case that spills a value to the coroutine frame and then reloads it, where the store could potentially alias with another memory access.
- **Rationale**: The patch adds TBAA metadata to the load instructions (`LI->setMetadata(LLVMContext::MD_tbaa, TBAATag);`) when reloading spilled values from the coroutine frame. However, it does not add the same TBAA metadata to the store instructions when spilling the values into the coroutine frame in `createStoreIntoFrame`. This means that while loads are protected from aliasing, stores are not, which could still lead to suboptimal alias analysis results.
- **Expected Issue**: The store instruction will not have the TBAA metadata, so alias analysis might still think the store to the frame slot aliases with other memory accesses, preventing optimizations like LICM.

### Strategy 2: Missing TBAA on ByVal Load

- **Target**: A test case with a byval argument that is spilled to the coroutine frame.
- **Rationale**: In `createStoreIntoFrame`, for byval arguments, a load is created (`Builder.CreateLoad(ByValTy, Def)`). This load does not receive the TBAA metadata. If this load aliases with other memory accesses, it could hinder optimizations.
- **Expected Issue**: The load instruction for the byval argument will not have the TBAA metadata, potentially leading to aliasing issues.

### Strategy 3: Missing TBAA on Alloca Load/Store

- **Target**: A test case with an alloca that is written to before CoroBegin and then spilled to the coroutine frame.
- **Rationale**: In `insertSpills`, for allocas that may write before CoroBegin, a load and a store are created (`Builder.CreateLoad(Alloca->getAllocatedType(), Alloca)` and `Builder.CreateStore(Value, G)`). Neither of these instructions receives the TBAA metadata.
- **Expected Issue**: The load and store instructions for allocas that may write before CoroBegin will not have the TBAA metadata, potentially leading to aliasing issues.

### Strategy 4: Missing TBAA on PromiseAlloca Load/Store

- **Target**: A test case with a PromiseAlloca that is accessed before CoroBegin and then spilled to the coroutine frame.
- **Rationale**: In `insertSpills`, for the PromiseAlloca, a load and a store are created (`Builder.CreateLoad(PA->getAllocatedType(), PA)` and `Builder.CreateStore(Value, G)`). Neither of these instructions receives the TBAA metadata.
- **Expected Issue**: The load and store instructions for the PromiseAlloca will not have the TBAA metadata, potentially leading to aliasing issues.

### Strategy 5: Language-Specific TBAA Root

- **Target**: A test case compiled from C or another language that uses a different TBAA root name, or a test case where the TBAA root is manually renamed.
- **Rationale**: The patch explicitly looks for "Simple C++ TBAA" (`MDString::getIfExists(C, "Simple C++ TBAA")`). If the module was compiled from a language other than C++ (e.g., C, which uses "Simple C/C++ TBAA" or similar, or Rust), the TBAA root might have a different name, and the optimization will be silently skipped.
- **Expected Issue**: The TBAA metadata is only added if "Simple C++ TBAA" exists. If a different TBAA root is used (e.g., in C or other languages), the metadata will not be added, and the optimization will not apply.

I have tested all the strategies and verified that the patch has several missed optimization issues. The patch adds TBAA metadata to the load instructions when reloading spilled values from the coroutine frame, but it fails to add the same TBAA metadata to the store instructions when spilling the values into the coroutine frame. It also misses adding TBAA metadata to loads and stores for byval arguments, allocas, and PromiseAlloca. These are all missed optimization issues, not miscompilations, so `difftest` did not find any execution differences. I will report these findings.
