## Test Strategies

### Strategy 1: Shifted-in non-zero bits

- **Target**: A sequence of stores where the combined store has a wider type than the original value being shifted.
- **Rationale**: The patch changes `CreateTrunc` to `CreateZExtOrTrunc`. If the value being stored is shifted right (`CreateLShr`), and the original value had non-zero bits that are shifted into the range of the new type, a simple truncation would have discarded them if the new type is smaller. However, if the new type is larger, `CreateZExtOrTrunc` will zero-extend the value. Wait, the issue is that `CreateTrunc` was used previously. If `NewTy` is larger than the type of `Val`, `CreateTrunc` would be invalid (it asserts if the destination type is larger). So `NewTy` must have been smaller or equal to the type of `Val` before the patch, or it was a bug that it crashed. Wait, the commit message says: "The top extracted value can include shifted-in zero bits. In that case we should emit a zext before the new store." This implies that `NewTy` can be larger than the type of `Val` after the shift. If `Val` is shifted right, its type doesn't change. If `NewTy` is larger than `Val`'s type, `CreateTrunc` would crash. So the test should trigger a case where `NewTy` is larger than `Val`'s type.
- **Expected Issue**: The combined store will write incorrect data because the top bits of the value are not zeroed out before being stored.
