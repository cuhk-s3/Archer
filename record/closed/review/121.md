## Test Strategies

### Strategy 1: Test fabs with value that can be -0.0 without nsz

- **Target**: A test case where `cannotBeOrderedLessThanZero` is true, `isKnownNeverNaN` is true, but `nsz` is missing on the `fabs` call.
- **Rationale**: The patch adds a condition `KnownClass.cannotBeOrderedLessThanZero() && KnownClass.isKnownNeverNaN() && Call->hasNoSignedZeros()`. If `Call->hasNoSignedZeros()` is false, it shouldn't fold. We should test that `fabs` is NOT folded if `nsz` is missing and the value can be `-0.0`.
- **Expected Issue**: The optimization might incorrectly fold `fabs(X)` to `X` when `X` is `-0.0`, because `cannotBeOrderedLessThanZero` is true for `-0.0`, but `fabs(-0.0)` is `+0.0`. If `nsz` is not present on the `fabs` call, this changes the sign of zero, which is observable.

### Strategy 2: Test fabs with nsz and value that can be -0.0

- **Target**: A test case where `X` is known to be `cannotBeOrderedLessThanZero` (e.g., `X = Y * Y` with `nnan`), and `fabs` has `nsz`.
- **Rationale**: To ensure the fold actually triggers when `nsz` is present, `cannotBeOrderedLessThanZero` is true, and `isKnownNeverNaN` is true.
- **Expected Issue**: The optimization might incorrectly fold `fabs(X)` to `X` when `X` is `-0.0` and `nsz` is present on the `fabs` call, but the user of the `fabs` result expects `+0.0` and does not have `nsz`. Wait, if `fabs` has `nsz`, it means the result of `fabs` can be `-0.0` if it's valid to return `-0.0` instead of `+0.0`. The fold replaces `fabs(X)` with `X`. If `X` is `-0.0`, the result is `-0.0`. Since `fabs` has `nsz`, returning `-0.0` is allowed. So this is correct.

### Strategy 3: Test fabs with nsz and value that can be -NaN

- **Target**: A test case where `X` can be `-NaN` (e.g., `X = Y * Y` without `nnan`), and `fabs` has `nsz`.
- **Rationale**: `cannotBeOrderedLessThanZero` is true for `NaN`. If `isKnownNeverNaN` is false, the fold should be prevented.
- **Expected Issue**: The optimization might incorrectly fold `fabs(X)` to `X` when `X` can be `NaN` with a sign bit of 1 (negative NaN). `cannotBeOrderedLessThanZero` returns true for `NaN`. If `isKnownNeverNaN` is false, the fold shouldn't happen because `fabs(-NaN)` is `+NaN`, which has a different sign bit. The patch checks `isKnownNeverNaN()`. We should test that it doesn't fold if `NaN` is possible.

### Strategy 4: Test fabs with nsz and value that can be negative

- **Target**: A test case where `X` can be negative, and `fabs` has `nsz`.
- **Rationale**: To ensure the fold does not trigger when the value can be strictly less than zero.
- **Expected Issue**: The optimization might incorrectly fold `fabs(X)` to `X` when `X` is `-0.0` and `nsz` is present on the `fabs` call, but `X` is not known to be `cannotBeOrderedLessThanZero`. Wait, the fold requires `cannotBeOrderedLessThanZero`. What if `X` is `+0.0` or `-0.0` but not known to be positive? Then `cannotBeOrderedLessThanZero` is false, so it won't fold.

### Strategy 5: Test fabs with nsz on vectors

- **Target**: A test case with vector types, `nsz` on `fabs`, and `cannotBeOrderedLessThanZero` true.
- **Rationale**: Vector types might have different behavior or partial undefs. We should ensure the fold works correctly for vectors.
- **Expected Issue**: The optimization might incorrectly fold `fabs(X)` to `X` when `X` is `-0.0` and `nsz` is present on the `fabs` call, but `X` is a vector and only some lanes have `nsz` or the analysis is incorrect for vectors.
