## Test Strategies

### Strategy 1: Invoke Unwind Edge Not Removed

- **Target**: A test case with `invoke` of a resume/destroy function on a noop coroutine, where the unwind destination block has a PHI node.
- **Rationale**: When `tryEraseCallInvoke` erases an `InvokeInst`, it replaces it with a branch to the normal destination. However, it fails to remove the predecessor edge from the unwind destination block. If the unwind destination block has PHI nodes, they will still expect an incoming value from the erased invoke's parent block, which is no longer a predecessor.
- **Expected Issue**: The invoke instruction is erased, but its unwind destination block still has the invoke's parent block as a predecessor in its PHI nodes. This leads to a malformed CFG and a crash in subsequent passes or verification.

### Strategy 2: CoroSubFnInst User is Not Call or Invoke

- **Target**: A test case where the result of `coro.subfn.addr` is used by an instruction other than a call or invoke (e.g., stored to memory or casted).
- **Rationale**: `visitIntrinsicInst` assumes that the unique undroppable user of `coro.subfn.addr` is either a `CallInst` or an `InvokeInst`. If it's something else (e.g., a `bitcast` or `store`), `tryEraseCallInvoke` returns false, and `SubFn->eraseFromParent()` is not called. While this might not crash, it leaves dead code.
- **Expected Issue**: The `coro.subfn.addr` instruction is not erased because `tryEraseCallInvoke` returns false, leaving a dead instruction in the IR.

### Strategy 3: CoroSubFnInst Has Multiple Users

- **Target**: A test case where the result of `coro.subfn.addr` has multiple users (e.g., two calls, or a call and a store).
- **Rationale**: `visitIntrinsicInst` calls `SubFn->getUniqueUndroppableUser()`. If `SubFn` has multiple users, this returns `nullptr`. Then `cast<Instruction>(User)` will crash with a null pointer dereference.
- **Expected Issue**: The `coro.subfn.addr` instruction is erased, but its user (the call/invoke) is not erased because it's not the unique undroppable user. This leaves a dangling pointer or causes a crash.

### Strategy 4: Use-After-Free in Worklist

- **Target**: A test case where a dead instruction deleted by `RecursivelyDeleteTriviallyDeadInstructions` is also in the `Worklist` but not as the direct user of the pointer.
- **Rationale**: `visitCallBase` calls `RecursivelyDeleteTriviallyDeadInstructions` with a callback `AboutToDeleteCallback` that calls `eraseFromWorklist`. However, `eraseFromWorklist` only removes the instruction if it is the *user* of the pointer. If an instruction in the worklist is deleted but it's not the user (e.g., it's an operand of the user), it might not be removed from the worklist, leading to a use-after-free. Also, `eraseFromWorklist` uses `erase_if` on `Worklist`, which is a `SmallVector`. This is O(N) and modifies the vector while iterating, but `erase_if` handles that. The issue is if the deleted instruction is the `U->get()` itself or something else. Wait, `eraseFromWorklist` checks `I == U.UseAndIsOffsetKnown.getPointer()->getUser()`. If `I` is deleted, and it was a user in the worklist, it's removed. But what if `I` is the `Use` itself? No, `I` is the `Instruction`.
- **Expected Issue**: The `RecursivelyDeleteTriviallyDeadInstructions` function deletes instructions that are still in the `Worklist` of `PtrUseVisitor`, leading to a use-after-free when the visitor tries to process them.

### Strategy 5: Pointer Escapes in Call Argument

- **Target**: A test case where the result of `coro.noop` is passed as an argument to a regular function call.
- **Rationale**: If the `coro.noop` pointer is passed as an argument to a function (not as the called operand), `visitCallBase` sets `ResumeOrDestroy` to false and does nothing. The `PtrUseVisitor` base class marks it as escaped. The `coro.noop` will then be lowered by `lowerCoroNoop`. This is correct behavior, but we should test it to ensure it doesn't crash or miscompile.
- **Expected Issue**: The `coro.noop` intrinsic is not fully elided, or a crash occurs because `PtrUseVisitor` visits a `CallBase` where the pointer is an argument, not the called operand, and `tryEraseCallInvoke` is not called, but the call is not deleted.

I have successfully tested all strategies on all test cases. I found two real bugs:
1. `Invoke Unwind Edge Not Removed`: When `coro-cleanup` erases an `invoke` instruction, it replaces it with a branch to the normal destination, but it fails to remove the predecessor edge from the unwind destination block. This leaves the CFG in an invalid state, causing `SimplifyCFG` to crash when it tries to process the PHI node in the unwind block.
2. `CoroSubFnInst Has Multiple Users`: The `coro-cleanup` pass assumes that `coro.subfn.addr` has a unique undroppable user. If it has multiple users, `getUniqueUndroppableUser()` returns `nullptr`, and `cast<Instruction>(nullptr)` crashes.

I will report the first bug as it is more subtle and directly related to the CFG manipulation logic in the patch.

Bug Triggering Analysis:
The provided test case triggers the bug because it contains an `invoke` instruction that calls the destroy function of a `coro.noop`. The unwind destination of this `invoke` is a block (`%unwind`) that has a PHI node. When the `coro-cleanup` pass processes this `invoke`, it replaces it with a branch to the normal destination (`%done`) and erases the `invoke`. However, it does not update the PHI node in the `%unwind` block to remove the incoming value from the `%coro` block. This results in a malformed CFG where the `%unwind` block still expects an incoming value from `%coro`, but `%coro` is no longer a predecessor. The subsequent `SimplifyCFG` pass crashes when it tries to process this malformed CFG.

Fix Weakness Analysis:
The weakness in the fix is in the `tryEraseCallInvoke` function. When it handles an `InvokeInst`, it correctly creates a branch to the normal destination and erases the invoke, but it completely forgets to update the unwind destination block. Specifically, it should call `II->getUnwindDest()->removePredecessor(II->getParent())` before erasing the invoke. This is a common pitfall when manually manipulating the CFG in LLVM passes.
