## Test Strategies

### Strategy 1: Context-Leakage in Multi-Use Instruction Simplification

- **Target**: A test case where `fpext` is conditionally executed and its operand is known never NaN only under that condition.
- **Rationale**: The fix uses `simplifyDemandedFPClassResult` to infer `nnan` and `ninf` flags on the `fpext` instruction based on `KnownSrc`. However, `KnownSrc` is computed using `SimplifyDemandedFPClass` which can use context-sensitive information (like dominating conditions) that are only valid at the specific use site `I`. If `I` (the `fpext` instruction) has multiple uses, modifying it in-place with `setFastMathFlags` leaks these context-specific assumptions to all other uses of the instruction, corrupting the data flow for execution paths that do not share those constraints. Wait, `SimplifyDemandedUseFPClass` asserts `I->hasOneUse()`. So `I` has exactly one use. But what if `I` is moved later? No, if it has one use, it's fine. Wait, what if `I` is a `fpext` that is conditionally executed, but the `KnownSrc` is derived from a condition that dominates `I`? That's fine. What if `KnownSrc` is derived from the demanded mask? `simplifyDemandedFPClassResult` uses `ValidResults = DemandedMask & Known.KnownFPClasses`. If `DemandedMask` doesn't demand NaN, `ValidResults` won't have NaN. Then `inferFastMathValueFlags` checks `if (!FMF.noNaNs() && (ValidResults & fcNan) == fcNone)`. If `ValidResults` has no NaN, it checks `all_of(KnownSrcs, isKnownNeverNaN)`. So it only sets `nnan` if the *sources* are known never NaN. It doesn't set `nnan` just because the result doesn't demand NaN. This is safe. Wait, what if `KnownSrc` is known never NaN because of a context-sensitive condition, but the `fpext` is hoisted later? Fast math flags are poison-generating, so hoisting an instruction with `nnan` can trigger UB if the condition no longer holds.
- **Expected Issue**: The `fpext` instruction will be incorrectly marked with `nnan` or `ninf` flags, which might be propagated to other uses or cause incorrect optimizations if the `fpext` is moved or if the context-specific condition does not apply globally.

### Strategy 2: Poison-Generating Flags on Hoisted Instructions

- **Target**: A test case where `fpext` is guarded by a condition that ensures its operand is not NaN/Inf, and the `fpext` could potentially be hoisted.
- **Rationale**: Fast-math flags like `nnan` and `ninf` are poison-generating. If `simplifyDemandedFPClassResult` infers these flags based on context-sensitive information (e.g., a dominating `icmp` or `assume`), and the `fpext` instruction is later hoisted or its guarding condition is removed, the instruction may execute with inputs that are NaN or Inf, generating poison. If this poison is not properly handled, it can lead to miscompilations.
- **Expected Issue**: The `fpext` instruction will be incorrectly marked with `nnan` or `ninf` flags, which might cause incorrect optimizations if the `fpext` is hoisted or if the flags are propagated incorrectly.

### Strategy 3: Incorrect Fast-Math Flag Inference based on DemandedMask

- **Target**: A test case where `DemandedMask` does not demand NaN/Inf, and `KnownSrc` is computed.
- **Rationale**: `simplifyDemandedFPClassResult` infers `nnan` and `ninf` flags based on `ValidResults` and `KnownSrcs`. If `DemandedMask` does not demand NaN/Inf, `ValidResults` will not contain NaN/Inf. If `KnownSrcs` are also known never NaN/Inf, the flags are set. However, if `KnownSrcs` are known never NaN/Inf only because of the `DemandedMask` passed down to `SimplifyDemandedFPClass`, this could be circular reasoning. Wait, `SimplifyDemandedFPClass` computes `KnownSrc` based on `SrcDemandedMask`. If `SrcDemandedMask` doesn't demand NaN, `KnownSrc` might not contain NaN? No, `KnownFPClass` represents the *actual* possible values, not just the demanded ones. `SimplifyDemandedFPClass` returns true if it simplified something, otherwise it computes `KnownSrc`. So `KnownSrc` is the actual known classes.
- **Expected Issue**: The `fpext` instruction will be incorrectly marked with `nnan` or `ninf` flags, leading to incorrect poison generation.

### Strategy 4: Subnormal to Normal Conversion in fpext

- **Target**: A test case involving subnormal numbers and `fpext`.
- **Rationale**: The patch adds `simplifyDemandedFPClassResult` to `fpext`. `fpext` can convert subnormal numbers to normal numbers. If the source is known to be subnormal, the result will be normal. The patch correctly handles this by adding `fcNegSubnormal` and `fcPosSubnormal` to `SrcDemandedMask` if `fcNegNormal` or `fcPosNormal` are demanded. However, does `fpext` preserve NaN payloads and signs? Yes. Does it preserve Inf? Yes. Are there any edge cases where `fpext` changes the FP class in a way that invalidates `nnan` or `ninf`? No, `fpext` of NaN is NaN, `fpext` of Inf is Inf. So if the source is never NaN, the result is never NaN. This seems correct.
- **Expected Issue**: The `fpext` instruction will be incorrectly marked with `nnan` or `ninf` flags, leading to incorrect poison generation.
