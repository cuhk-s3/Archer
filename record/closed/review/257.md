## Test Strategies

### Strategy 1: SSA Dominance Violation due to Incorrect Insert Point

- **Target**: Create a test case where an `fdiv` instruction is an indirect operand of the root instruction being simplified by `SimplifyDemandedFPClass` (e.g., `fdiv` -> `fadd` -> `fcmp` or `fdiv` -> `fpext` -> `fcmp`). The `fdiv` should have only one use so that `SimplifyDemandedUseFPClass` is called. The conditions for the `fdiv` -> `copysign` fold must be met (e.g., `KnownLHS.isKnownAlways(fcPosZero)` and `KnownRHS.isKnownNeverNaN()`).
- **Rationale**: When `SimplifyDemandedUseFPClass` simplifies an `fdiv` instruction to a `copysign` instruction, it creates the new `copysign` instruction using the `IRBuilder`. If the `Builder`'s insert point is not explicitly set to the `fdiv` instruction, the new `copysign` will be inserted at the `Builder`'s current insert point, which is typically the root instruction of the `SimplifyDemanded` recursive traversal. If the `fdiv` instruction is not the root instruction (e.g., it is an operand of an operand of the root instruction), the new `copysign` will be inserted after the users of the `fdiv` instruction (other than the root). When the `fdiv` is replaced by the `copysign`, those intermediate users will now use a value (`copysign`) that is defined after them, violating SSA dominance rules. The patch fixes this by explicitly setting the insert point to the `fdiv` instruction before creating the `copysign`.
- **Expected Issue**: The `copysign` instruction will be inserted before the `fdiv` instruction, but it uses the result of the `fdiv` instruction (or its operands, but in this case it uses `X` and `Y` which are operands of `fdiv`). Wait, the `copysign` uses `X` and `Y`, which are operands of `fdiv`. So inserting before `fdiv` is fine for `X` and `Y`. However, if `fdiv` is a multi-use instruction, `SimplifyDemandedUseFPClass` might be called from a user of `fdiv`. If we insert `copysign` at `I` (which is `fdiv`), it's fine. But wait, the patch changes `Builder.SetInsertPoint(I)` to `Builder.SetInsertPoint(I)`. Wait, the patch *adds* `Builder.SetInsertPoint(I)`. Before the patch, where was the insert point? It was at the user of `I`! Because `SimplifyDemandedUseFPClass` is called with `I` being the `fdiv` instruction, but `Builder`'s insert point is set by the caller of `SimplifyDemandedUseFPClass` to the instruction being simplified (the user of `fdiv`). If `copysign` is inserted at the user, but it uses `X` and `Y` (operands of `fdiv`), that's also fine, because `X` and `Y` dominate `fdiv` and thus dominate the user. BUT what if `fdiv` is NOT dominating the user? No, `fdiv` must dominate its user. Wait, what if the user is a PHI node? If the user is a PHI node, the insert point might be at the beginning of the incoming block. If we insert at the incoming block, `X` and `Y` still dominate it.
Wait, why did the patch add `Builder.SetInsertPoint(I)`?
Ah! If `fdiv` is simplified to `copysign(X, Y)`, the new `copysign` instruction is created. If the insert point is at the user of `fdiv`, and `fdiv` has multiple uses, we are replacing the use of `fdiv` with `copysign`. Wait, `SimplifyDemandedUseFPClass` is called from `SimplifyDemandedFPClass`. If `VInst->hasOneUse()` is true, it calls `SimplifyDemandedUseFPClass(VInst, ...)`. Here `VInst` is `fdiv`. The caller `SimplifyDemandedFPClass` is simplifying an operand of some instruction `I`. The `Builder`'s insert point is typically set to `I` (the user).
If `VInst` (the `fdiv`) has ONE use, we replace the use with `copysign`. The `copysign` is inserted at `I` (the user). Is that a problem?
What if `X` or `Y` do not dominate `I`? They must, because they dominate `fdiv`, which dominates `I`.
Wait, what if `fdiv` is an instruction, and we are replacing it. If we insert `copysign` at `I` (the user), and `fdiv` is later removed, that's fine.
But wait! What if `fdiv` is NOT removed? It has one use, so it will be removed.
Wait, what if the `copysign` is inserted at `I`, but `I` is a `phi` node? `InstCombinerImpl::SimplifyDemandedUseFPClass` is called from `SimplifyDemandedFPClass`.
Let's look at the commit message or bug report if possible. "Fix wrong insert point for fdiv->copysign simplify".
If `Builder.SetInsertPoint(I)` is added, it means `I` is the `fdiv` instruction. So the `copysign` is now inserted at `fdiv` instead of the user.
Why is inserting at the user wrong?
Because `SimplifyDemandedUseFPClass` is simplifying the *use* of `fdiv`. Wait, `I` in `SimplifyDemandedUseFPClass(Instruction *I, ...)` is the instruction being simplified!
Let's check `SimplifyDemandedUseFPClass` signature:
`Value *InstCombinerImpl::SimplifyDemandedUseFPClass(Instruction *I, FPClassTest DemandedMask, KnownFPClass &Known, Instruction *CxtI, unsigned Depth)`
Here `I` is the `fdiv` instruction!
If `Builder`'s insert point was NOT set, where was it? It was left wherever it was before `SimplifyDemandedUseFPClass` was called.
Usually, `InstCombine` sets the `Builder` insert point to the instruction being visited. So if we are visiting `User`, the insert point is at `User`.
If we create `copysign` at `User`, it uses `X` and `Y`. This is perfectly valid SSA.
BUT wait! `SimplifyDemandedUseFPClass` can call `SimplifyDemandedFPClass` recursively!
Look at `SimplifyDemandedUseFPClass` for `FDiv`:
```cpp
    if (SimplifyDemandedFPClass(I, 0, LHSDemandedMask, KnownLHS, Depth + 1) ||
        SimplifyDemandedFPClass(I, 1, RHSDemandedMask, KnownRHS, Depth + 1))
      return I;
```
If it recurses, it might modify things. But that doesn't explain the insert point.
Wait, what if `I` (the `fdiv`) is NOT the instruction currently being visited by InstCombine?
Yes, `SimplifyDemandedFPClass` can be called on operands of the visited instruction, and their operands, etc. (up to `Depth` limit).
If we create a new instruction `copysign` to replace `fdiv`, and we don't set the insert point, it gets inserted at the `Builder`'s current insert point, which is the root instruction of the `SimplifyDemanded` tree (the one originally visited by InstCombine).
So `copysign` is inserted at the root instruction.
Is that a problem?
Suppose the root instruction is `Root`, and we have `fdiv` dominating `Root`.
If we insert `copysign` at `Root`, it uses `X` and `Y`. This is fine for SSA.
BUT what if `fdiv` is used by `Root`, AND `fdiv` is also used by something else?
Wait, `SimplifyDemandedUseFPClass` is ONLY called if `VInst->hasOneUse()` is true!
```cpp
  if (VInst->hasOneUse()) {
    // If the instruction has one use, we can directly simplify it.
    NewVal = SimplifyDemandedUseFPClass(VInst, DemandedMask, Known, I, Depth);
```
So `fdiv` has exactly ONE use.
If it has one use, it will be replaced by `copysign` and then `fdiv` will be dead.
Wait, if `fdiv` has one use, its use is some instruction `U1`. `U1` might have one use `U2`, ..., up to `Root`.
If we insert `copysign` at `Root`, and replace `fdiv` with `copysign`, then `U1` now uses `copysign`.
But `U1` is BEFORE `Root`!
Ah!!!
If `U1` is before `Root`, and `U1` uses `copysign`, but `copysign` is inserted at `Root`, then `U1` uses a value defined AFTER it! This breaks SSA dominance!
Let's verify this.
Suppose we have:
```llvm
%fdiv = fdiv %x, %y
%u1 = fadd %fdiv, %z
%root = fmul %u1, %w
```
InstCombine visits `%root`. It calls `SimplifyDemandedFPClass` on `%u1`.
`SimplifyDemandedFPClass` on `%u1` calls `SimplifyDemandedUseFPClass` for `fadd`.
`fadd` calls `SimplifyDemandedFPClass` on `%fdiv`.
`SimplifyDemandedFPClass` on `%fdiv` calls `SimplifyDemandedUseFPClass` for `fdiv`.
In `fdiv`, it decides to return `copysign(X, Y)`.
It creates `copysign` using `Builder.CreateCopySign`.
The `Builder`'s insert point is still at `%root`!
So `copysign` is inserted BEFORE `%root`.
But `SimplifyDemandedUseFPClass` returns `copysign`, which replaces `%fdiv` in `%u1`.
So `%u1` now uses `copysign`.
But `%u1` is BEFORE `%root` (and before `copysign`).
This violates SSA dominance! `%u1` uses a value defined after it.

The test case uses a chain of instructions where `SimplifyDemandedFPClass` is called recursively.

Since `%u1` is before `%mul`, `%u1` now uses `copysign` which is defined after it, causing an SSA dominance violation.
The patch fixes this by explicitly setting the insert point to `%div` before creating the `copysign` instruction.
