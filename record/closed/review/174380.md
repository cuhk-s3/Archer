## Test Strategies

### Strategy 1: ShrinkDemandedConstant with LHS known zeros

- **Target**: An `add` instruction where LHS has known trailing zeros, the low bits are not demanded, and RHS is a constant.
- **Rationale**: The patch adds a call to `ShrinkDemandedConstant(I, 1, DemandedFromRHS)` when `NtzLHS` (number of trailing zeros in LHS) is greater than 0. If `ShrinkDemandedConstant` modifies the constant operand (RHS) by clearing bits that are not demanded, it might change the value of the constant in a way that affects the result of the addition, especially if the addition can overflow or if the cleared bits were actually needed for a carry. However, the logic clears low bits of `DemandedFromRHS` up to `NtzLHS`. Since LHS has known zeros in those low bits, the addition in those low bits is just `0 + RHS_low`. If those low bits are not demanded, we don't care about the result of `0 + RHS_low`. But wait, what if `RHS_low` generates a carry into the demanded bits? If we clear `RHS_low`, we might lose a carry that would have been generated. The patch clears `DemandedFromRHS` low bits up to `NtzLHS`. `NtzLHS` is `(~DemandedMask & LHSKnown.Zero).countr_one()`. This means the low `NtzLHS` bits are NOT demanded AND are known zero in LHS. Since they are known zero in LHS, the addition in these bits is `0 + RHS_bit`. This can NEVER generate a carry into bit `NtzLHS`! Because `0 + 1 = 1` (no carry), `0 + 0 = 0` (no carry). So the carry out of the first `NtzLHS` bits is always 0. Therefore, the value of RHS in the first `NtzLHS` bits does not affect any higher bits. And since these bits are not demanded, they don't affect the demanded result. So shrinking the constant by clearing these bits is mathematically sound. Wait, what if `ShrinkDemandedConstant` clears bits in RHS that ARE demanded? No, it only clears bits that are NOT in `DemandedFromRHS`. `DemandedFromRHS` has its low `NtzLHS` bits cleared. So `ShrinkDemandedConstant` will clear the low `NtzLHS` bits of the constant. This is correct.
- **Expected Issue**: The constant is shrunk incorrectly, leading to a miscompilation where the result of the add is wrong.

### Strategy 2: Wrap flags preservation

- **Target**: An `add` instruction with `nsw` or `nuw` flags, where LHS has known trailing zeros, the low bits are not demanded, and RHS is a constant.
- **Rationale**: The patch returns `disableWrapFlagsBasedOnUnusedHighBits(I, NLZ)` if `ShrinkDemandedConstant` succeeds. `disableWrapFlagsBasedOnUnusedHighBits` drops `nsw`/`nuw` if `NLZ > 0` (i.e., high bits are not demanded). But what if `NLZ == 0`? Then `nsw`/`nuw` are preserved. If we change the constant by clearing its low bits, does it affect whether the addition wraps? Yes! Suppose we have `add nsw i8 %x, 3`. `%x` has 2 known trailing zeros. We demand all bits except the lowest 2. `NLZ == 0`. We clear the lowest 2 bits of the constant, changing it from 3 to 0. The instruction becomes `add nsw i8 %x, 0`. This is fine because `x + 0` won't wrap if `x + 3` didn't wrap? Wait. If `x + 3` didn't wrap, does `x + 0` wrap? No, `x + 0 = x`, which never wraps. What if the constant was negative? `add nsw i8 %x, -1` (255). `%x` has 2 trailing zeros. We clear the lowest 2 bits of -1 (11111111 -> 11111100 = -4). The instruction becomes `add nsw i8 %x, -4`. If `x + (-1)` didn't wrap, does `x + (-4)` wrap? Suppose `x = -128`. `-128 + (-1) = -129` (wraps). So `x` couldn't be -128. Suppose `x = -124`. `-124 + (-1) = -125` (no wrap). `-124 + (-4) = -128` (no wrap). Suppose `x = 124`. `124 + (-1) = 123` (no wrap). `124 + (-4) = 120` (no wrap). Wait, what if `x = -128` is not possible, but `x = -126`? `%x` has 2 trailing zeros, so `%x` is a multiple of 4. So `%x` can be -128, -124, ..., 124. If `%x + C` doesn't wrap, does `%x + (C & ~3)` wrap? Let `C' = C & ~3`. `C' <= C` (as unsigned). As signed, `C'` can be smaller than `C`. For example, `C = 3`, `C' = 0`. `C = -1`, `C' = -4`. Since `%x` is a multiple of 4, `%x + C` and `%x + C'` have the same bits from bit 2 upwards. Can `%x + C'` wrap while `%x + C` doesn't? Let `%x = -128`. `%x + 3 = -125` (no wrap). `%x + 0 = -128` (no wrap). Let `%x = 124`. `%x + 3 = 127` (no wrap). `%x + 0 = 124` (no wrap). Let `%x = 124`. `%x + (-1) = 123` (no wrap). `%x + (-4) = 120` (no wrap). Let `%x = -124`. `%x + (-1) = -125` (no wrap). `%x + (-4) = -128` (no wrap). It seems it never wraps if the original didn't wrap. Wait, what if `C = 127`? `C' = 124`. `%x = 0`. `0 + 127 = 127` (no wrap). `0 + 124 = 124` (no wrap). What if `C = -125`? `C' = -128`. `%x = 4`. `4 + (-125) = -121` (no wrap). `4 + (-128) = -124` (no wrap). Is it possible that `%x + C'` wraps? Let's check unsigned wrap. `nuw`. `%x + C` doesn't wrap. `C' <= C` (unsigned). So `%x + C' <= %x + C`. Since `%x + C` doesn't wrap, `%x + C'` cannot wrap unsigned! What about signed wrap? `nsw`. `%x + C` doesn't wrap. `C' = C - (C & 3)`. So `C - 3 <= C' <= C`. Thus `%x + C - 3 <= %x + C' <= %x + C`. If `%x + C` doesn't wrap, it is in `[-128, 127]`. Since `%x` is a multiple of 4, `%x + C` mod 4 is `C` mod 4. So `%x + C` cannot be at the very bottom of the range if `C mod 4 != 0`. Specifically, the minimum value of `%x + C` is `-128 + (C mod 4)`. So `%x + C - (C mod 4) >= -128`. Thus `%x + C'` is always >= -128. And since `C' <= C`, `%x + C' <= 127`. So it never wraps signed either! Wait, is this true? Let's test: `x` is multiple of 4. `x \in [-128, 124]`. `C \in [-128, 127]`. `C' = C & ~3`. `x + C \in [-128, 127]`. `x + C' = x + C - (C & 3)`. Since `x \equiv 0 \pmod 4`, `x + C \equiv C \pmod 4`. So `x + C - (C & 3)` is exactly rounding down `x + C` to the nearest multiple of 4. Since `x + C \ge -128` and `-128` is a multiple of 4, rounding down `x + C` to the nearest multiple of 4 cannot go below `-128`. So `x + C' \ge -128`. And since `x + C' \le x + C \le 127`, it cannot overflow either. So `nsw` and `nuw` are always preserved! Wait, what if `NtzLHS` is not just trailing zeros, but some zeros in the middle? The code says: `unsigned NtzLHS = (~DemandedMask & LHSKnown.Zero).countr_one();` This counts the number of contiguous zeros starting from bit 0 in `(~DemandedMask & LHSKnown.Zero)`. So they are strictly trailing bits. So my proof holds.
- **Expected Issue**: The `nsw`/`nuw` flags are not dropped when they should be, leading to poison being generated incorrectly.

### Strategy 3: ShrinkDemandedConstant return value

- **Target**: An `add` instruction where `ShrinkDemandedConstant` modifies the constant.
- **Rationale**: If `ShrinkDemandedConstant` returns `true`, it means the constant was modified. The code then returns `disableWrapFlagsBasedOnUnusedHighBits(I, NLZ)`. This is correct.
- **Expected Issue**: The optimization might be applied when `DemandedFromRHS` is modified, but `ShrinkDemandedConstant` might not actually shrink the constant if the constant already has zeros in the cleared bits. In this case, `ShrinkDemandedConstant` returns `false`, and the optimization doesn't trigger `disableWrapFlagsBasedOnUnusedHighBits`. This is correct behavior. But what if `ShrinkDemandedConstant` returns `true`? It modifies the constant in place.

### Strategy 4: Sub instruction symmetry

- **Target**: A `sub` instruction with LHS known trailing zeros.
- **Rationale**: Checking if `sub` should have the same optimization. As reasoned, it shouldn't, because `0 - RHS_low` generates a borrow if `RHS_low != 0`, whereas `0 + RHS_low` never generates a carry. So the patch is correct to only apply this to `add`.
- **Expected Issue**: The patch only handles `add`. What about `sub`? The patch does not add this logic to `sub`. Is it missing for `sub`? For `sub`, the existing code has: `unsigned NTZ = (~DemandedMask & RHSKnown.Zero).countr_one(); DemandedFromLHS.clearLowBits(NTZ); if (ShrinkDemandedConstant(I, 0, DemandedFromLHS) ...)` This is the exact same logic, but for `sub`, it uses `RHSKnown.Zero` to simplify LHS. The patch adds the symmetric case for `add` (using `LHSKnown.Zero` to simplify RHS). For `sub`, can we use `LHSKnown.Zero` to simplify RHS? If LHS has known trailing zeros, and we don't demand them, can we clear the trailing bits of RHS? `LHS - RHS`. In the lowest `N` bits, LHS is 0. So `0 - RHS_low`. This is `-RHS_low`. Does this generate a borrow into the higher bits? Yes! If `RHS_low != 0`, `0 - RHS_low` will generate a borrow. If we clear `RHS_low` to 0, `0 - 0 = 0`, which does NOT generate a borrow. So changing `RHS_low` to 0 will change the borrow into the demanded bits, which changes the result! So we CANNOT do this for `sub`. The patch correctly only adds this to `add`.

### Strategy 5: Contiguous trailing zeros

- **Target**: An `add` instruction where LHS has known zeros, but bit 0 is demanded.
- **Rationale**: The use of `countr_one` ensures that the bits are contiguous starting from the LSB. If there was a gap, a carry could be generated. The logic is sound.
- **Expected Issue**: Wait, what if `NtzLHS` is calculated incorrectly? `unsigned NtzLHS = (~DemandedMask & LHSKnown.Zero).countr_one();` This counts the number of contiguous 1s starting from the LSB in `~DemandedMask & LHSKnown.Zero`. This means it finds the longest prefix of bits (from LSB) that are BOTH not demanded AND known zero in LHS. This is exactly the condition we need: the bits must be known zero in LHS, and they must not be demanded. Since they are contiguous from LSB, there are no bits below them that could generate a carry. Wait! What if bit 0 is demanded, but bit 1 is not demanded and known zero? Then `countr_one` will return 0, because bit 0 is demanded (so `~DemandedMask` has 0 at bit 0). So it only clears bits if the contiguous block STARTS from bit 0. This is required because if bit 0 was not known zero, it could generate a carry into bit 1. Since we only clear bits up to `NtzLHS`, and `NtzLHS` starts from bit 0, all bits below `NtzLHS` are known zero in LHS. So no carry can be generated from ANY of the first `NtzLHS` bits. This is perfectly safe.
