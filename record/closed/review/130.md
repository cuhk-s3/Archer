## Test Strategies

### Strategy 1: Poison Propagation in Inverted Condition

- **Target**: Vector selects with poison elements in the condition.
- **Rationale**: The fix introduces `Builder.CreateNot(TrueSI->getCondition())` and `Builder.CreateNot(FalseSI->getCondition())`. If the condition of the inner select is poison, `CreateNot` will produce poison. The subsequent `CreateLogicalAnd` or `CreateLogicalOr` will then propagate this poison if the outer condition is true (for AND) or false (for OR). In the original code, `select(C0, select(poison, b, a), b)` with `C0 = false` evaluates to `b`. In the optimized code, `select(C0 && !poison, a, b)` with `C0 = false` evaluates to `b` (since `false && poison` is `false`). Wait, `false && poison` is `false`. What if `C0 = true`? Then `select(true, select(poison, b, a), b)` evaluates to `select(poison, b, a)` which is poison. So poison behavior seems correct for scalar. What about vectors? If `C0` is a vector `<true, false>` and `C1` is `<poison, poison>`, `select(<true, false>, select(<poison, poison>, <b, b>, <a, a>), <b, b>)` -> `<poison, b>`. Optimized: `select(<true, false> && !<poison, poison>, <a, a>, <b, b>)` -> `select(<poison, false>, <a, a>, <b, b>)` -> `<poison, b>`. This seems correct. Let's reconsider. What if `C1` is poison, and we use `CreateLogicalAnd`? `CreateLogicalAnd` is `select C0, C1, false`. So `select C0, !C1, false`. If `C0` is false, it's false. If `C0` is true, it's `!C1`. This matches the original behavior.
- **Expected Issue**: The optimized code will evaluate to poison instead of the expected value, because the logical AND/OR will propagate poison from the inverted condition, whereas the original nested select would have shielded the poison.

### Strategy 2: Fast-Math Flags Dropped

- **Target**: Select instructions with Fast-Math Flags.
- **Rationale**: The original code had `select` instructions which might have had fast-math flags (e.g., `nnan`, `nsz`). The optimization replaces the nested selects with a new logical AND/OR and a single select. The new select is created using `replaceOperand`, so it reuses the outer select instruction. However, the inner select's fast-math flags are completely ignored and dropped. If the inner select had FMF that are needed, they are lost. Wait, `replaceOperand` modifies the outer select in place. The outer select keeps its FMF. The inner select's FMF are lost. Is this a problem? If the inner select had `nnan` and the outer didn't, the result loses `nnan`. This is generally safe (dropping FMF is safe). What if the outer select had `nnan` but the inner didn't? The outer select keeps `nnan`, which applies to the result. Since the result is now a combination of both, is it safe to keep the outer select's FMF? Yes, because the outer select's FMF applied to its result, which could have been the inner select's result. So if the outer select had `nnan`, it assumed the final result is not NaN. This is still true.
- **Expected Issue**: The optimized code might drop fast-math flags or other instruction-level metadata that were present on the original select instructions.

### Strategy 3: Multi-Use Inner Select Condition

- **Target**: Inner select condition is undef.
- **Rationale**: The optimization checks `TrueSI->hasOneUse()` and `FalseSI->hasOneUse()`. This ensures the inner select itself is only used once. However, it does not check if the *condition* of the inner select (`TrueSI->getCondition()`) has multiple uses or is safe to evaluate. Wait, the condition is already evaluated before the select, so it's an existing value. Creating a `Not` of the condition (`Builder.CreateNot`) creates a new instruction. If the condition is a complex expression, we are just adding a `Not` instruction. This is safe. What if the inner select's condition is poison? We discussed this. What if the inner select's condition is `undef`? `!undef` is `undef`. `C0 && undef` is `undef` if `C0` is true, `false` if `C0` is false. Original: `select(C0, select(undef, b, a), b)`. If `C0` is true, result is `undef ? b : a`, which can be either `a` or `b`. Optimized: `select(C0 && !undef, a, b)`. If `C0` is true, condition is `undef`, result is `undef ? a : b`, which can be either `a` or `b`. This is also safe.
- **Expected Issue**: The optimized code might execute an instruction that was previously guarded by the outer select, leading to poison or UB if the instruction is not safe to execute unconditionally.

### Strategy 4: Vector Undef in Condition

- **Target**: Vector selects with undef elements in the condition.
- **Rationale**: For vectors, `CreateLogicalAnd(CondVal, InvertedCond)` creates `select CondVal, InvertedCond, zeroinitializer`. If `CondVal` has an `undef` element, the `select` will choose either `InvertedCond` or `0`. In the original code: `select(CondVal, select(C1, b, a), b)`. If `CondVal` is `undef`, it can choose either `select(C1, b, a)` or `b`. If it chooses `b`, the result is `b`. If it chooses `select(C1, b, a)`, the result is `a` or `b` depending on `C1`. So the result can be `a` or `b`. In the optimized code: `select(CondVal && !C1, a, b)`. `CondVal && !C1` is `select(CondVal, !C1, false)`. If `CondVal` is `undef`, this evaluates to either `!C1` or `false`. Then the outer select is `select(..., a, b)`. If `CondVal` was `undef`, the outer select condition is `undef ? !C1 : false`. If it's `false`, result is `b`. If it's `!C1`, result is `a` or `b`. This matches the original behavior.
- **Expected Issue**: The optimized code might evaluate to poison for vector types when the condition contains undef elements, due to the way logical AND/OR are implemented for vectors.

### Strategy 5: Scalar Condition with Vector Values

- **Target**: Outer select has vector condition, inner select has scalar condition.
- **Rationale**: LLVM IR allows `select i1 <cond>, <n x ty> <val1>, <n x ty> <val2>`. The optimization uses `Builder.CreateLogicalAnd(CondVal, TrueSI->getCondition())`. If `CondVal` is `i1` and `TrueSI->getCondition()` is `<n x i1>`, `CreateLogicalAnd` will assert or fail because the types don't match. Wait, `TrueSI` is the inner select. If the outer select has an `i1` condition and vector values, the inner select must also have vector values. But can the inner select have a vector condition `<n x i1>` while the outer select has a scalar condition `i1`? Yes, LLVM allows this! If `CondVal` is `i1` and `TrueSI->getCondition()` is `<n x i1>`, `Builder.CreateLogicalAnd(CondVal, TrueSI->getCondition())` will crash because `CreateLogicalAnd` expects both operands to have the same type. Let's check `CreateLogicalAnd` implementation: `CreateSelect(Cond1, Cond2, ConstantInt::getNullValue(Cond2->getType()))`. If `Cond1` is `i1` and `Cond2` is `<n x i1>`, `CreateSelect` allows scalar condition with vector values! Wait, `CreateSelect(i1, <n x i1>, <n x i1>)` is valid. It will return `<n x i1>`. Then the outer select will have its condition replaced with `<n x i1>`. But the outer select originally had an `i1` condition! `replaceOperand(SI, 0, And)` will change the condition of the outer select from `i1` to `<n x i1>`. This is valid in LLVM IR, but it changes the semantics. Original: `select i1 C0, (select <n x i1> C1, <n x ty> b, <n x ty> a), <n x ty> b`. If `C0` is true, it does element-wise selection based on `C1`. If `C0` is false, it returns `b` for all elements. Optimized: `And = select i1 C0, <n x i1> C1, <n x i1> zeroinitializer`. This is `<n x i1>`. Then `select <n x i1> And, <n x ty> a, <n x ty> b`. This does element-wise selection. If `C0` is true, `And` is `C1`, so it selects `a` or `b` based on `C1`. If `C0` is false, `And` is all-zeros, so it selects `b` for all elements. The semantics are exactly the same! But wait, what if `CondVal` is `<n x i1>` and `TrueSI->getCondition()` is `i1`? Then `CreateLogicalAnd` does `CreateSelect(<n x i1>, i1, i1)`. This is INVALID! `select` cannot have a vector condition and scalar values. `CreateSelect` will assert or create invalid IR. Let's verify this.
- **Expected Issue**: The optimization might incorrectly fold when the true/false values are vectors and the condition is a scalar, which is allowed in LLVM IR.
