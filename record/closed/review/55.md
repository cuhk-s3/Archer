## Test Strategies

### Strategy 1: Select with undef arm and condition

- **Target**: A select instruction where one arm is `undef` and the condition implies a specific FP class for that arm.
- **Rationale**: The fix checks `isGuaranteedNotToBeUndef(Arm, ...)` to decide whether to use the known FP class from the condition. If the arm is undef, `isGuaranteedNotToBeUndef` returns false, and the known FP class from the condition is not applied. We should test if this correctly prevents incorrect optimizations when the arm is undef.
- **Expected Issue**: The optimization might incorrectly assume the select result is not undef/poison, leading to incorrect simplifications or folding.

### Strategy 2: Select with poison arm and condition

- **Target**: A select instruction where one arm is `poison` and the condition implies a specific FP class for that arm.
- **Rationale**: Similar to the undef case, if the arm is poison, `isGuaranteedNotToBeUndef` (which actually checks for undef or poison in many contexts, though the name says undef) should return false. We should test if this correctly prevents incorrect optimizations when the arm is poison.
- **Expected Issue**: The optimization might incorrectly assume the select result is not poison, leading to incorrect simplifications or folding.

### Strategy 3: Select with conditionally undef arm

- **Target**: A select instruction where one arm is an instruction that can produce poison/undef (e.g., `add nsw` with potential overflow) and the condition implies a specific FP class for that arm.
- **Rationale**: If the arm is only conditionally undef (e.g., a load that might be out of bounds, or an operation that might produce poison), `isGuaranteedNotToBeUndef` should return false. We should test if this correctly prevents incorrect optimizations.
- **Expected Issue**: The optimization might incorrectly assume the select result is not undef/poison, leading to incorrect simplifications or folding.

### Strategy 4: Select with vector undef elements

- **Target**: A select instruction with vector types where one arm has `undef` elements and the condition implies a specific FP class for that arm.
- **Rationale**: For vector types, `undef` can be per-element. `isGuaranteedNotToBeUndef` should handle vector undefs correctly. We should test if this correctly prevents incorrect optimizations when the arm is a vector with undef elements.
- **Expected Issue**: The optimization might incorrectly assume the select result is not undef/poison, leading to incorrect simplifications or folding.

### Strategy 5: Select with vector poison elements

- **Target**: A select instruction with vector types where one arm has `poison` elements and the condition implies a specific FP class for that arm.
- **Rationale**: Similar to vector undefs, vector poison elements should be handled correctly. We should test if this correctly prevents incorrect optimizations when the arm is a vector with poison elements.
- **Expected Issue**: The optimization might incorrectly assume the select result is not undef/poison, leading to incorrect simplifications or folding.
