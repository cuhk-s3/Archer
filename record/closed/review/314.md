## Test Strategies

### Strategy 1: Propagate nofpclass from callsite argument that can be undef or poison

- **Target**: Create a test case with a function `F` taking a float argument `arg`. Inside `F`, call another function `G` passing `arg` as an argument. Add the `nofpclass` attribute to the callsite argument, but do *not* add `noundef`. Ensure the call is guaranteed to execute.
- **Rationale**: The fix adds a check `CB->paramHasAttr(ArgNo, Attribute::NoUndef)` before propagating the `nofpclass` attribute. This is because `nofpclass` on a callsite argument only guarantees that the value is not of the specified floating-point classes *if* it is not `undef` or `poison`. If the callsite argument can be `undef` or `poison`, it might take on any value, including the disallowed floating-point classes. Therefore, it's unsafe to propagate `nofpclass` to the caller's argument unless we also know the callsite argument is `noundef`. The test strategy should create a scenario where a callsite argument has `nofpclass` but *not* `noundef`, and verify that the attribute is *not* propagated.
- **Expected Issue**: The `nofpclass` attribute will be incorrectly propagated to the caller's argument, even though the callsite argument might be `undef` or `poison`. This could lead to miscompilations if the caller's argument is later used in a context where `undef` or `poison` is not allowed, or if the `nofpclass` attribute is used to optimize away checks that are actually necessary.

- **Rationale**: Similar to the previous strategy, but here the caller's argument *does* have `noundef`. Wait, if the callsite argument *is* the caller's argument, and the caller's argument is `noundef`, then the callsite argument is also `noundef`. The pass checks `CB->paramHasAttr(ArgNo, Attribute::NoUndef)`. If the caller's argument has `noundef`, does `CB->paramHasAttr` return true? `paramHasAttr` checks the callsite attributes and the called function's attributes. It does *not* check the attributes of the `Value` being passed. So if the caller's argument has `noundef`, but the callsite/callee does not, `CB->paramHasAttr` will return false, and `nofpclass` will *not* be propagated. This is actually a missed optimization, not a miscompilation. But let's focus on the miscompilation: what if the callsite has `nofpclass` but not `noundef`? The fix prevents propagation. We should test that it is indeed prevented.
