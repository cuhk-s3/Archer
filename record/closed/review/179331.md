## Test Strategies

### Strategy 1: Preserve !prof metadata on select

- **Target**: A `udiv` instruction where the denominator is a `select` between two power-of-2 constants, and the `select` has `!prof` metadata.
- **Rationale**: The `takeLog2` function folds `log2(Cond ? X : Y)` into `Cond ? log2(X) : log2(Y)`. When `X` and `Y` are power-of-2 constants (e.g., in a `udiv` where the denominator is a `select`), the new `select` should inherit the `!prof` metadata (branch weights) from the original `select`.
- **Expected Issue**: Without the fix, the `!prof` metadata on the original `select` instruction is lost when it is folded into a new `select` instruction representing the log2 of the operands.

### Strategy 2: Preserve !unpredictable metadata on select

- **Target**: A `udiv` instruction where the denominator is a `select` between two power-of-2 constants, and the `select` has `!unpredictable` metadata.
- **Rationale**: Similar to `!prof` metadata, `!unpredictable` metadata on a `select` instruction should be preserved when the `select` is transformed by `takeLog2`.
- **Expected Issue**: Without the fix, the `!unpredictable` metadata on the original `select` instruction is lost when it is folded into a new `select` instruction.

### Strategy 3: Preserve metadata with variable power-of-2 operands

- **Target**: A `udiv` instruction where the denominator is a `select` between a power-of-2 constant and a variable known to be a power of 2 (e.g., `1 << Z`), and the `select` has `!prof` metadata.
- **Rationale**: The `takeLog2` function can also fold variables that are known to be powers of 2 (e.g., `1 << Z`). The metadata preservation should work correctly even when the operands of the `select` are not constants but variables that can be folded by `takeLog2`.
- **Expected Issue**: Without the fix, the `!prof` metadata is lost when the `select` is folded.

### Strategy 4: Preserve metadata in vector types

- **Target**: A vector `udiv` instruction where the denominator is a vector `select` between two power-of-2 vector constants, and the `select` has `!prof` metadata.
- **Rationale**: The `takeLog2` transformation applies to vector types as well. A vector `select` with `!prof` metadata should have its metadata preserved when transformed into a new vector `select`.
- **Expected Issue**: Without the fix, the `!prof` metadata on the original vector `select` instruction is lost.

### Strategy 5: Preserve multiple metadata types

- **Target**: A `udiv` instruction where the denominator is a `select` between two power-of-2 constants, and the `select` has both `!prof` and `!unpredictable` metadata.
- **Rationale**: A `select` instruction can have multiple types of metadata attached to it (e.g., both `!prof` and `!unpredictable`). The new `select` created by `takeLog2` should inherit all relevant metadata from the original `select`.
- **Expected Issue**: Without the fix, all metadata on the original `select` instruction is lost.

I have tested all the strategies on all the test cases. The fix correctly preserves the metadata on the `select` instruction when it is folded by `takeLog2`. I did not find any bugs in the fix.

Summary of the fix:
The fix modifies `InstCombinerImpl::takeLog2` to preserve metadata (like `!prof` and `!unpredictable`) when folding a `select` instruction. Before the fix, the new `select` instruction was created without any metadata from the original `select` instruction. After the fix, the new `select` instruction inherits the metadata from the original `select` instruction, unless `ProfcheckDisableMetadataFixes` is true. This ensures that profile-guided optimizations (PGO) and other metadata-dependent passes can still benefit from the metadata after the `takeLog2` transformation.
