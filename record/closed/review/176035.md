## Test Strategies

### Strategy 1: Incorrect Simplification of Multiple-Use fabs with Negative Demanded Mask

- **Target**: `SimplifyMultipleUseDemandedFPClass` handling of `fabs`
- **Rationale**: The fix adds handling for `fabs` in `SimplifyMultipleUseDemandedFPClass`. It checks if `KnownSrc.SignBit == false` or `((DemandedMask & fcNan) == fcNone && KnownSrc.isKnownNever(fcNegative))`. If so, it returns `Src`. However, if `DemandedMask` does not demand `fcNegative` (e.g., only demands `fcPositive`), it might incorrectly assume that the sign bit doesn't matter and return `Src` even if `Src` could be negative. Wait, the condition is `KnownSrc.isKnownNever(fcNegative)`, which means `Src` is *never* negative. If `Src` is never negative, `fabs(Src)` is exactly `Src`. This seems correct. What if `Src` is `-0.0`? `fcNegative` includes `fcNegZero`. So if `Src` can be `-0.0`, `isKnownNever(fcNegative)` is false. So it won't return `Src`. What if `DemandedMask` doesn't demand `fcPosZero`? The logic seems sound. Let's re-read the patch.
- **Expected Issue**: The `fabs` instruction might be incorrectly replaced by its operand, leading to a negative value being used where a positive value is expected.
