## Test Strategies

### Strategy 1: powi(SNaN, 0) with strict FP semantics

- **Target**: A test case with `llvm.powi(SNaN, 0)` where the SNaN is explicitly passed.
- **Rationale**: The patch explicitly handles `powi(SNaN, 0)` by checking `KnownSrc.isKnownNever(fcSNan)`. If the source can be an SNaN, it does not exclude `fcNan` from the result. However, if the source is a QNaN, it assumes the result is 1.0 (not NaN). We should test if `powi(SNaN, 0)` correctly retains the possibility of returning NaN.
- **Expected Issue**: The compiler might incorrectly assume the result is not NaN, leading to invalid optimizations (e.g., removing `isnan` checks or folding to a non-NaN value).

### Strategy 2: powi(QNaN, 0) with strict FP semantics

- **Target**: A test case with `llvm.powi(QNaN, 0)` where the QNaN is explicitly passed.
- **Rationale**: The patch assumes `powi(QNaN, 0)` returns 1.0 and thus excludes `fcNan` from the result. We should verify this behavior is correctly modeled and doesn't conflict with strict FP semantics if applicable.
- **Expected Issue**: The compiler might incorrectly assume the result is not NaN, leading to invalid optimizations.

### Strategy 3: powi(NaN, 0) with unknown NaN type

- **Target**: A test case with `llvm.powi(x, 0)` where `x` is known to be NaN but not specifically SNaN or QNaN.
- **Rationale**: If the source is known to be NaN but it's unknown whether it's SNaN or QNaN, the patch should conservatively assume it could be SNaN and thus could return NaN. We should test this conservative fallback.
- **Expected Issue**: The compiler might incorrectly assume the result is not NaN, leading to invalid optimizations.

### Strategy 4: powi(x, 0) with unknown source

- **Target**: A test case with `llvm.powi(x, 0)` where `x` is completely unknown.
- **Rationale**: If the source is completely unknown, it could be an SNaN. The patch should conservatively assume the result could be NaN. We should test this conservative fallback.
- **Expected Issue**: The compiler might incorrectly assume the result is not NaN, leading to invalid optimizations.

### Strategy 5: powi(x, 0) with source known not to be SNaN

- **Target**: A test case with `llvm.powi(x, 0)` where `x` is known not to be SNaN (e.g., via `nnan` flag or other analysis).
- **Rationale**: If the source is known not to be SNaN, the patch assumes the result is 1.0 and excludes `fcNan`. We should verify this optimization triggers correctly.
- **Expected Issue**: The compiler might incorrectly assume the result is not NaN, leading to invalid optimizations.
