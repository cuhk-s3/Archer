## Test Strategies

### Strategy 1: Test with multiple users where one introduces a broadcast and another doesn't

- **Target**: A loop with a single-scalar recipe that has multiple users, some of which are extracts and some of which are vector users.
- **Rationale**: The fix changes the condition from `!all_of(users, ...)` to `any_of(users, IntroducesBCastOf)`. If there are multiple users, and one user introduces a broadcast while another does not, the new logic might behave differently. We need to ensure that if ANY user introduces a broadcast, the recipe is NOT narrowed, or if NO user introduces a broadcast, it IS narrowed. The previous logic checked if NOT ALL users introduce a broadcast (meaning AT LEAST ONE does NOT introduce a broadcast). The new logic checks if ANY user introduces a broadcast. Wait, the old logic was `!all_of(users, [](U) { if (Extract) return true; return U->usesScalars(); })`. This means "not all users are extracts or use scalars". So if there is at least one user that is NOT an extract and DOES NOT use scalars, it returns true (skip narrowing). The new logic is `any_of(users, IntroducesBCastOf)`. `IntroducesBCastOf` returns `false` for extracts, and `!U->usesScalars()` otherwise. So it returns true if a user is NOT an extract and DOES NOT use scalars. Thus, `any_of(..., IntroducesBCastOf)` is exactly equivalent to `!all_of(..., !IntroducesBCastOf)`. The logic for users seems identical in outcome but refactored. The real change is in the operands check.
- **Expected Issue**: The vectorizer might incorrectly narrow a recipe to a single scalar, leading to missing broadcasts or incorrect scalar values being used in subsequent vector operations.

### Strategy 2: Test with an operand used by multiple recipes, where one user introduces a broadcast

- **Target**: A loop where a single-scalar recipe's operand is used by another recipe that only uses scalars (e.g., another single-scalar recipe or an extract).
- **Rationale**: The old logic checked `none_of(operands, [](Op) { if (Op->getSingleUser() != RepOrWidenR) return false; ... })`. This meant that if an operand had MULTIPLE users, it would return `false` for that operand, meaning `none_of` might fail if other conditions were met, but actually `return false` in `none_of` means the predicate is false, so it continues checking other operands. Wait. `none_of` returns true if the predicate is false for ALL operands. The old predicate was: `if (Op->getSingleUser() != RepOrWidenR) return false; return LiveInNeedsBroadcast || ...`. So if an operand had multiple users, the predicate returned `false`. If ALL operands had multiple users, `none_of` would return `true`, and the recipe WOULD be skipped (if the user condition was also met). Wait, `if (user_cond && none_of(...)) continue;`. So if `none_of` is true, we SKIP narrowing. If an operand has multiple users, the old predicate returned `false`. If all operands returned `false`, `none_of` is `true`, so we SKIP narrowing.
The new logic: `none_of(operands, [&](Op) { if (any_of(Op->users() != RepOrWidenR, IntroducesBCastOf)) return false; return LiveInNeedsBroadcast || ... })`.
If an operand has multiple users, and ANY of those OTHER users introduces a broadcast, the predicate returns `false`. If ALL operands return `false`, `none_of` is `true`, so we SKIP narrowing.
Wait, if `any_of` is true, it returns `false`. What if `any_of` is false (i.e., NO other user introduces a broadcast)? Then it checks `LiveInNeedsBroadcast || ...`. If that is true, the predicate returns `true`, so `none_of` becomes `false`, and we DO NOT skip narrowing (we proceed to narrow).
So the new logic ALLOWS narrowing even if an operand has multiple users, AS LONG AS none of those other users introduce a broadcast!
This is the core of the fix: "narrowToSingleScalarRecipes' operands check is a bit too restrictive by permitting a single user. Factor out and reuse the existing introduces-broadcast logic to improve results."
We need to test a scenario where an operand has MULTIPLE users, NONE of which introduce a broadcast (e.g., they all use scalars or are extracts). In the old logic, this would be skipped (not narrowed). In the new logic, it WILL be narrowed.
- **Expected Issue**: The vectorizer might incorrectly narrow a recipe, causing a broadcast to be removed when it is actually needed by another user of the operand, leading to a miscompilation or a crash during code generation.

### Strategy 3: Test with an operand used by multiple recipes, where one OTHER user DOES introduce a broadcast

If all operands return `false`, `none_of` is `true`, and we SKIP narrowing. This ensures we don't narrow and then have to broadcast again. We should verify that this case is still skipped correctly.
- **Expected Issue**: The vectorizer might incorrectly narrow a recipe, leading to missing broadcasts for operands that actually need them, causing a crash or incorrect execution.

### Strategy 4: Test with an operand that is a live-in requiring a broadcast, but has multiple users

- **Target**: A loop with a live-in value used by multiple single-scalar recipes.
- **Rationale**: If an operand is a live-in that requires a broadcast, and it has multiple users, none of which introduce a broadcast. The new logic will check `any_of(other_users, IntroducesBCastOf)`. This will be `false`. Then it evaluates `LiveInNeedsBroadcast`, which is `true`. So the predicate returns `true`. `none_of` returns `false`. So we DO NOT skip narrowing (we proceed to narrow). Wait, if we proceed to narrow, we replace the recipe with a single-scalar recipe. The live-in will be used as a scalar. This is correct because no other user needs it as a vector! In the old logic, because it had multiple users, the predicate returned `false`, `none_of` might be `true`, and we would SKIP narrowing, leaving it as a vector recipe, which would require broadcasting the live-in. The new logic avoids this broadcast.
- **Expected Issue**: The vectorizer might incorrectly narrow a recipe, leading to missing broadcasts for operands that actually need them, causing a crash or incorrect execution.
