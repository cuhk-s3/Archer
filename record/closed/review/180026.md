### Strategy 2: Force optnone on a function with optsize

- **Target**: A function with `optsize` attribute. Mutate to run `-passes=forceattrs -force-attribute=optnone`.
- **Rationale**: The patch only checks for conflicts between `alwaysinline` and `noinline`. However, there are other mutually exclusive attributes in LLVM, such as `optnone` and `optsize`/`minsize`. If a function already has `optsize` and the user forces `optnone` (or vice versa), the pass will add the attribute without removing the conflicting one, leading to a Verifier failure.
- **Expected Issue**: The Verifier will fail because `optnone` and `optsize` are incompatible.

### Strategy 3: Force optsize on a function with optnone

- **Target**: A function with `optnone` and `noinline` attributes. Mutate to run `-passes=forceattrs -force-attribute=optsize`.
- **Rationale**: Similar to the above, forcing `optsize` on a function that already has `optnone` will result in both attributes being present, which is invalid and will fail the Verifier. The patch fails to generalize the conflict checking to other known incompatible attributes.
- **Expected Issue**: The Verifier will fail because `optnone` and `minsize` are incompatible.

### Strategy 4: Force alwaysinline and force-remove noinline on a function with noinline

- **Target**: A function with `noinline` attribute. Mutate to run `-passes=forceattrs -force-attribute=alwaysinline -force-remove-attribute=noinline`.
- **Rationale**: The `forceAttributes` function processes additions (`ForceAttributes`) before removals (`ForceRemoveAttributes`). If a function currently has `noinline`, and the user specifies both `-force-attribute=alwaysinline` and `-force-remove-attribute=noinline`, the addition of `alwaysinline` will be skipped because `hasConflictingFnAttr` sees the existing `noinline`. Then `noinline` is removed. The end result is the function has neither, whereas the user explicitly requested to replace `noinline` with `alwaysinline`.
- **Expected Issue**: The function will not have `alwaysinline` added, even though `noinline` is explicitly removed in the same pass, contrary to the user's likely intent.

### Strategy 5: Force writable on a function with incompatible memory attribute

- **Target**: A function with `memory(read)` or `readnone` attribute. Mutate to run `-passes=forceattrs -force-attribute=writable`.
- **Rationale**: The Verifier checks that `writable` is incompatible with `memory` attributes that don't allow `argmem: write` (or `readnone`/`readonly`). The `ForceFunctionAttrs` pass does not check for this conflict when forcing the `writable` attribute, which can lead to invalid IR.
- **Expected Issue**: The Verifier will fail because `memory` (without argmem: write) and `writable` are incompatible.

### Reproducer

}

**Command:**

opt -S -passes=forceattrs -force-attribute=optnone

### Analysis

Bug Triggering Analysis: The provided test case defines a function `@hoo` with the `optsize` attribute. When `opt` is run with `-passes=forceattrs -force-attribute=optnone`, the `ForceFunctionAttrs` pass adds the `optnone` attribute to the function. However, the Verifier requires that `optnone` must be accompanied by `noinline`, and it is incompatible with `optsize`. Since the pass does not check for these incompatibilities (it only checks for `alwaysinline` vs `noinline`), it blindly adds `optnone`, resulting in a function with both `optsize` and `optnone` (and missing `noinline`). This causes the Verifier to fail and `opt` to crash.

Fix Weakness Analysis: The fix introduced `hasConflictingFnAttr` to prevent adding `alwaysinline` to a function with `noinline` and vice versa. However, it failed to generalize this check to other mutually exclusive attributes defined in LLVM, such as `optnone` vs `optsize`/`minsize`/`optdebug`, or `writable` vs `readnone`. By only addressing the specific case of `alwaysinline` and `noinline`, the fix left the pass vulnerable to Verifier crashes when other conflicting attributes are forced.
