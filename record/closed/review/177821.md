### Strategy 1: NaN Flag Propagation with fcmp nnan and select without nnan

- **Target**: A test case with `fcmp nnan` and a `select` without `nnan`.
- **Rationale**: The patch unconditionally sets `nnan` on the generated `minnum`/`maxnum` intrinsic if the `fcmp` has `nnan`. This is safe because if the `fcmp` assumes no NaNs, the presence of a NaN would result in poison, making the entire operation poison. Therefore, the intrinsic can safely assume no NaNs.
- **Expected Issue**: The generated minnum/maxnum intrinsic will have the `nnan` flag set, which allows it to be lowered back to an fcmp+select. However, if the original fcmp did not have the `nnan` flag, setting `nnan` on the intrinsic is incorrect and could lead to miscompilations if NaNs are actually present. The patch explicitly checks `FCmp->hasNoNaNs()`, so this is handled. But what if the select has `nnan` but the fcmp does not? The patch requires `FCmp->hasNoNaNs()`. Wait, the patch says `if (FCmp && FCmp->hasNoNaNs() && ...)`. So it requires the fcmp to have `nnan`. What if the fcmp has `nnan` but the select does not? The patch sets `BinIntrInst->setHasNoNaNs(true)`. This is safe because the fcmp has `nnan`, meaning the comparison assumes no NaNs. If a NaN is passed, the fcmp behavior is poison, so the whole select is poison. Thus, setting `nnan` on the intrinsic is safe.

### Strategy 2: Signed Zero Flag Propagation with single use ignoring sign of zero

- **Target**: A test case where the select does not have `nsz`, but its single use ignores the sign of zero (e.g., `fcmp`), and there is another identical select that does not have `nsz` and its use *does* care about the sign of zero.
- **Rationale**: The patch sets `nsz` on the intrinsic if the select's single use ignores the sign of zero. This is safe for the current use, but we should verify that the `nsz` flag doesn't leak to other uses if the instruction is CSE'd or if the use changes.
- **Expected Issue**: The generated minnum/maxnum intrinsic will have the `nsz` flag set. The patch sets `BinIntrInst->setHasNoSignedZeros(true)` unconditionally if the transformation triggers. The transformation triggers if `SIFPOp->hasNoSignedZeros() || (SIFPOp->hasOneUse() && canIgnoreSignBitOfZero(*SIFPOp->use_begin()))`. If the select does not have `nsz`, but its single use ignores the sign bit of zero (e.g., it's an `fadd` with 0.0, or an `fcmp`), the intrinsic gets `nsz`. If this intrinsic is later used in a context where the sign of zero *does* matter (e.g., the single use is replaced or modified, or the intrinsic is CSE'd with another use), the `nsz` flag might cause incorrect sign of zero. However, since the original select had a single use that ignored the sign of zero, replacing the select with an intrinsic with `nsz` is safe for that specific use. But what if the intrinsic is CSE'd? LLVM's CSE does not merge instructions if it would add poison-generating flags unless they are valid for both. Wait, `nsz` is a fast-math flag, not a poison-generating flag. FMFs are usually intersected during CSE. So CSE is safe. What if the single use is an instruction that ignores the sign of zero, but later optimizations change it to something that cares? The `nsz` flag on the intrinsic would allow the backend to return either +0.0 or -0.0. Since the original select didn't have `nsz`, it would have returned a specific zero. But the only use didn't care. So it's fine.

Thus, the intrinsic can safely assume no infinities. This is safe.
