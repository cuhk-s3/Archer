### Strategy 1: Hoist non-select instruction with empty ProfileMetadataToPreserve

- **Target**: A loop with a hoistable `CallInst` (e.g., indirect call or call with profile data) where `ProfcheckDisableMetadataFixes` is true or the instruction is not in the header.
- **Rationale**: The patch adds a check `if (ProfileMetadataToPreserve.empty() && isa<SelectInst>(I))` to set explicitly unknown branch weights. If `I` is not a `SelectInst`, it won't set the explicitly unknown branch weights. However, the previous code `I->dropUnknownNonDebugMetadata(ProfileMetadataToPreserve);` drops all unknown non-debug metadata except those in `ProfileMetadataToPreserve`. If `ProfileMetadataToPreserve` is empty, all unknown non-debug metadata (including profile metadata) is dropped. If `I` is a `SelectInst`, it sets explicitly unknown branch weights. What if `I` is a `BranchInst` or `SwitchInst`? `makeLoopInvariant` is called on `Instruction *I`. Can `I` be a `BranchInst` or `SwitchInst`? `makeLoopInvariant` is used to hoist instructions out of the loop. Usually, terminators are not hoisted. But if they are, or if there are other instructions that need profile metadata, they might lose it. Wait, the patch specifically targets `SelectInst`. Let's check if there are other instructions that might need explicitly unknown branch weights. `setExplicitlyUnknownBranchWeightsIfProfiled` is called for `SelectInst`. What about `CallInst`? `CallInst` can have branch weights (for indirect calls). If a `CallInst` is hoisted, its profile metadata is dropped. Should it also get explicitly unknown branch weights?
- **Expected Issue**: The hoisted instruction will have explicitly unknown branch weights, which might be incorrect if the instruction is not a select instruction or if it's not appropriate to set explicitly unknown branch weights for it.

### Strategy 2: Hoist instruction from loop header with ProfcheckDisableMetadataFixes=false

- **Target**: A loop with a hoistable `CallInst` with value profile data in the loop header.
- **Rationale**: The patch preserves profile metadata if `OriginalParent == getHeader()` and `!ProfcheckDisableMetadataFixes`. The rationale is that the header is executed if the loop is entered, so the profile metadata is valid outside the loop (in the preheader). However, if the loop is never entered (e.g., the loop is guarded by a condition that is false), the hoisted instruction will be executed in the preheader, but its profile metadata was collected inside the loop. Is the profile metadata still valid? Yes, because if the loop is never entered, the profile counts would be zero, which is still valid. But what if the instruction is hoisted above a condition that determines whether the loop is entered? The preheader is executed before the loop condition is checked (if the loop is a do-while loop, the header is executed at least once; if it's a while loop, the header is the condition block). Wait, in LLVM, the loop header is the entry point of the loop. If the loop is a while loop, the header contains the loop condition. Hoisting an instruction from the header to the preheader means it's executed before the loop condition. If the instruction's profile metadata depends on the loop condition, preserving it might be incorrect. For example, a `select` instruction in the header. Its profile metadata reflects the branch weights inside the loop. If hoisted to the preheader, it's executed only once per loop invocation, not per loop iteration. The branch weights (which are relative) might still be valid, but the absolute counts (if any) would be wrong. However, `MD_prof` for `select` usually contains branch weights, which are relative. What if it's a `CallInst` with value profile data? The value profile data contains absolute counts. If hoisted, the counts would be wrong.
- **Expected Issue**: The hoisted instruction retains its profile metadata, which might be incorrect if the profile metadata is control-dependent on a condition inside the loop that is not guaranteed to be true outside the loop.

### Strategy 3: Hoist SelectInst from non-header block

- **Target**: A loop with a hoistable `SelectInst` in a non-header block (e.g., a block that is conditionally executed inside the loop).
- **Rationale**: If a `SelectInst` is hoisted from a non-header block, `ProfileMetadataToPreserve` will be empty. The patch drops the profile metadata and sets explicitly unknown branch weights. This is conservative and correct, but we should test that this path is actually taken and works as expected.
- **Expected Issue**: The hoisted instruction loses its profile metadata, and since it's a `SelectInst`, it gets explicitly unknown branch weights. This might degrade optimization quality compared to preserving the profile metadata.

### Strategy 4: Hoist non-SelectInst from non-header block

- **Target**: A loop with a hoistable non-`SelectInst` (e.g., `CallInst`) in a non-header block.
- **Rationale**: If a non-`SelectInst` (e.g., a `CallInst` or a regular arithmetic instruction with some metadata) is hoisted from a non-header block, its profile metadata is dropped and no explicitly unknown branch weights are set. We should test this to ensure no crashes or unexpected behavior occurs.
- **Expected Issue**: The hoisted instruction loses its profile metadata, and since it's not a `SelectInst`, it doesn't get explicitly unknown branch weights. This might be correct, but we should verify the behavior.

### Strategy 5: Hoist instruction from header of a nested loop

- **Target**: A nested loop where an instruction in the inner loop's header is hoisted to the inner loop's preheader, and then to the outer loop's preheader.
- **Rationale**: If an instruction is hoisted from the header of an inner loop to its preheader, and then potentially hoisted again to the outer loop's preheader. The patch checks `OriginalParent == getHeader()`. If it's hoisted one level, it preserves metadata. If it's hoisted again, `OriginalParent` is now the inner loop's preheader, which is not the outer loop's header. So it will drop the metadata. This seems correct, but we should test nested loop hoisting to ensure the metadata is handled correctly at each level.
- **Expected Issue**: The hoisted instruction retains its profile metadata, but the metadata is invalid in the preheader context.
