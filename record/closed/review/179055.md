## Test Strategies

### Strategy 1: Vector with Poison Lanes in All-Ones

- **Target**: A vector `icmp sgt` or `icmp sle` where the RHS is a constant vector containing `-1` and `poison` elements.
- **Rationale**: The `m_AllOnes()` matcher in LLVM allows poison elements in vectors. If `RHS` is a vector like `<i32 -1, i32 poison>`, `matchEquivZeroRHS` will treat it as an all-ones vector and transform `icmp sgt X, <-1, poison>` to `icmp sge X, <0, 0>`. However, the original comparison with poison would yield poison for that lane, while the transformed comparison yields a concrete boolean value (true or false) depending on `X`. This is an invalid refinement because it replaces poison with a potentially non-poison value.
- **Expected Issue**: The optimization might incorrectly fold the comparison to true/false or another constant, ignoring the poison lanes, which could lead to incorrect behavior if the poison lanes are later observed.

- **Target**: A vector `icmp sge`, `icmp uge`, `icmp slt`, or `icmp ult` where the RHS is a constant vector containing `1` and `poison` elements.
- **Rationale**: Similar to the all-ones case, the `m_One()` matcher allows poison elements in vectors. If `RHS` is a vector like `<i32 1, i32 poison>`, `matchEquivZeroRHS` will treat it as a ones vector and transform `icmp sge X, <1, poison>` to `icmp sgt X, <0, 0>`. The original comparison with poison would yield poison for that lane, while the transformed comparison yields a concrete boolean value. This is an invalid refinement.
- **Expected Issue**: The optimization might incorrectly fold the comparison to true/false or another constant, ignoring the poison lanes, which could lead to incorrect behavior if the poison lanes are later observed.

### Strategy 3: Vector with Undef Lanes in All-Ones

- **Target**: A vector `icmp sgt` or `icmp sle` where the RHS is a constant vector containing `-1` and `undef` elements.
- **Rationale**: The `m_AllOnes()` matcher in LLVM allows undef elements in vectors. If `RHS` is a vector like `<i32 -1, i32 undef>`, `matchEquivZeroRHS` will treat it as an all-ones vector and transform `icmp sgt X, <-1, undef>` to `icmp sge X, <0, 0>`. The original comparison with undef could yield either true or false for that lane, but the transformed comparison forces it to be compared against 0. While replacing undef with a specific value is generally allowed, replacing it with a value that changes the semantics of the comparison (from comparing with undef to comparing with 0) might be problematic if it interacts with other optimizations that rely on the undef behavior. However, the more critical issue is if it replaces poison. Let's focus on poison. Wait, `m_AllOnes()` matches undef as well. If it's undef, `icmp sgt X, undef` can be evaluated to `false` or `true`. `icmp sge X, 0` evaluates to `true` or `false`. This might be a valid refinement. But what if `m_AllOnes()` matches a vector with `undef` and we replace it with `0`? The original RHS had `undef`, the new RHS has `0`. This is a valid refinement of `undef` to `0`. But wait, `simplifyICmpWithZero` doesn't create a new instruction, it just returns a folded value. If it returns `getTrue(ITy)` or `getFalse(ITy)`, it replaces the whole vector comparison with an all-true or all-false vector. If the original vector had poison, replacing poison with true/false is invalid.
- **Expected Issue**: The optimization might incorrectly fold the comparison to true/false or another constant, ignoring the undef lanes, which could lead to incorrect behavior if the undef lanes are later observed.
