## Test Strategies

### Strategy 1: Scalable Vectorization with Tail Folding and Runtime Trip Count

- **Target**: A loop with a runtime trip count, compiled with scalable vectorization (e.g., SVE or RISC-V V) and tail folding enabled.
- **Rationale**: When tail folding is enabled for scalable vectors and the trip count is not a compile-time constant, `addMinimumIterationCheck` needs to emit an overflow check. Before the fix, it emitted the `Sub` instruction before the `ExpandSCEV` recipe, violating the VPlan invariant that all `VPExpandSCEVRecipe`s must appear before any other recipes in the entry block.
- **Expected Issue**: The compiler will crash due to an assertion failure in `VPlanTransforms::expandSCEVs` because a `VPExpandSCEVRecipe` is found after a `VPInstruction`.

### Strategy 2: Fixed-Width Vectorization with Tail Folding and Overflow Check

- **Target**: A loop with a runtime trip count, compiled with fixed-width vectorization, tail folding enabled, and an induction variable that requires an overflow check.
- **Rationale**: Even with fixed-width vectors, if tail folding is enabled and the compiler cannot prove that the induction variable will not overflow (e.g., due to a large step or specific types), it will generate an overflow check. This will trigger the same buggy code path if the trip count is a runtime value.
- **Expected Issue**: The compiler will crash due to an assertion failure in `VPlanTransforms::expandSCEVs`.
