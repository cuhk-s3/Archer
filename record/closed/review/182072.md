## Test Strategies

### Strategy 1: Integer Sub-Reduction with Vectorized Epilogue

- **Target**: A loop with an integer sub-reduction, compiled with flags to force vectorization and epilogue vectorization (e.g., `-force-vector-width=4 -epilogue-vectorization-force-VF=2`).
- **Rationale**: This is the primary scenario targeted by the fix. A loop with an integer sub-reduction (e.g., `sum = sum - array[i]`) needs to be vectorized such that both a main vector loop and a vectorized epilogue loop are generated. The initial value of the reduction should be non-zero to clearly observe the difference between the original start value and the resume value.
- **Expected Issue**: The epilogue loop would incorrectly use the resume value as the PHI start value and the original start value for the final subtraction, leading to an incorrect final reduction result.

### Strategy 2: Floating-Point Sub-Reduction with Vectorized Epilogue

- **Target**: A loop with a floating-point sub-reduction (e.g., `sum = sum - array[i]`) with `fast` math flags, forcing main and epilogue vectorization.
- **Rationale**: To ensure the fix correctly handles floating-point sub-reductions (with `fast` math flags allowing reassociation) if they are represented using `RecurKind::Sub` and partial reductions.
- **Expected Issue**: If floating-point sub-reductions are handled similarly and fall under `RecurKind::Sub`, the same incorrect initialization and subtraction would occur, leading to wrong FP results.

### Strategy 3: Multiple Sub-Reductions in the Same Loop

- **Target**: A loop containing two or more independent sub-reductions (e.g., `sum1 = sum1 - a[i]; sum2 = sum2 - b[i];`), vectorized with an epilogue.
- **Rationale**: Testing a loop with multiple independent sub-reductions ensures that the fix correctly identifies and updates the final subtraction for each reduction independently without mixing up their start values or resume values.
- **Expected Issue**: The compiler might crash if the single-user assumption is violated, or generate incorrect code if the multiple sub-reductions interfere.

### Strategy 4: Sub-Reduction with Variable Initial Value

- **Target**: A loop with a sub-reduction where the initial value is passed as a function argument, vectorized with an epilogue.
- **Rationale**: The start value of the reduction might be a runtime variable rather than a compile-time constant. The fix must correctly replace the operand of the subtraction with the dynamic resume value.
- **Expected Issue**: The test ensures that the fix correctly handles cases where the start value is a variable (e.g., passed as a function argument) rather than a constant, verifying that the IR replacement works for dynamic values.

### Strategy 5: Sub-Reduction with Scalar Epilogue (Negative Test)

- **Target**: A loop with a sub-reduction, vectorized such that the main loop is vectorized but the epilogue remains scalar (e.g., by disabling epilogue vectorization).
- **Rationale**: The fix specifically checks `PhiR->getVFScaleFactor() > 1` to apply the logic only to vectorized epilogues. A test case that generates a scalar epilogue ensures that this condition correctly prevents the logic from applying where it shouldn't, and that scalar epilogues are already handled correctly.
- **Expected Issue**: If the fix incorrectly applies to scalar epilogues or fails to trigger, the scalar epilogue might compute the wrong result or the compiler might crash.
