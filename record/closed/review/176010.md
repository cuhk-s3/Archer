## Test Strategies

### Strategy 1: Missing Resize for Smaller Source Alloca

- **Target**: A test case where `SrcAlloca` is smaller than `DestAlloca`, and `SrcAlloca` has `lifetime.start`/`lifetime.end` intrinsics with the smaller size.
- **Rationale**: The patch adds logic to resize the source alloca if the destination alloca is larger (`DestSize->getFixedValue() > SrcSize->getFixedValue()`). However, if the source alloca is smaller, it only resizes it if `DestSize` is larger. Wait, the patch *does* resize it: `if (DestSize->getFixedValue() > SrcSize->getFixedValue()) { SrcAlloca->setAllocatedType(DestAlloca->getAllocatedType()); SrcAlloca->setOperand(0, DestAlloca->getArraySize()); }`. But what if the source alloca is *larger* than the destination alloca? The patch allows `*SrcSize != *DestSize` as long as both are fixed. If `SrcSize > DestSize`, it doesn't resize `SrcAlloca`. This is fine because `SrcAlloca` is already large enough. But what if `DestAlloca` has a larger alignment? The patch does `SrcAlloca->setAlignment(std::max(SrcAlloca->getAlign(), DestAlloca->getAlign()));`. What if `DestAlloca` is used in ways that assume its exact type, and replacing it with `SrcAlloca` (which has a different type) causes type mismatches in instructions that require exact types (like `getelementptr` with `inbounds` or `byval` arguments)? Actually, `replaceAllUsesWith` might create invalid IR if the types of the allocas don't match, because `AllocaInst` returns a pointer, and in opaque pointers, the pointer type is just `ptr`. So type mismatch is less of an issue for the pointer itself. But `getAllocatedType()` is used by `byval` and `sret`. If `DestAlloca` was passed as `byval(Ty)`, and we replace it with `SrcAlloca` which has a different allocated type, does it matter? No, `byval` has its own type. Wait, what if `SrcSize < DestSize`? The patch resizes `SrcAlloca` to `DestAlloca`'s type and array size. But what if `SrcAlloca` is used by a `getelementptr` that assumes its original type? With opaque pointers, GEPs have their own source element type, so changing the alloca's allocated type doesn't invalidate the GEPs. However, what if `SrcAlloca` is used by a `lifetime.start` or `lifetime.end` with the original smaller size? The patch drops unknown metadata, but doesn't update the sizes in `lifetime` intrinsics for `SrcAlloca`. If `SrcAlloca` is resized to be larger, its original `lifetime.start` might only cover the smaller size, leaving the rest of the new larger alloca uninitialized or dead according to stack coloring!
- **Expected Issue**: The optimization will incorrectly merge the allocas, causing out-of-bounds accesses to the smaller source alloca to overwrite unrelated memory or cause a crash, because the source alloca was not resized to accommodate the larger destination alloca's size.

### Strategy 2: Unupdated Lifetime Markers for Resized Source Alloca

- **Target**: A test case with `SrcSize < DestSize`, where `SrcAlloca` has `llvm.lifetime.start` and `llvm.lifetime.end` intrinsics.
- **Rationale**: When `SrcAlloca` is resized to match a larger `DestAlloca`, its size increases. The patch removes `DestAlloca`'s lifetime markers (if any were collected in `LifetimeMarkers`), but it does *not* update the size arguments of `SrcAlloca`'s existing `lifetime.start` and `lifetime.end` intrinsics. If `SrcAlloca` originally had a `lifetime.start` of size 8, and is resized to size 16, the `lifetime.start` still says 8. Users of the old `DestAlloca` might access bytes 8-15, which are now outside the active lifetime of `SrcAlloca`.
- **Expected Issue**: The optimization will proceed, but the `lifetime.start` and `lifetime.end` intrinsics for `SrcAlloca` will still specify the old, smaller size. Stack coloring or other passes might then assume the extra bytes (which now belong to the merged alloca and are used by `DestAlloca`'s users) are dead, leading to miscompilations (e.g., overlapping stack slots).

### Strategy 3: Inalloca Type Mismatch

- **Target**: A test case where `SrcAlloca` is used as an `inalloca` argument and is smaller than `DestAlloca`.
- **Rationale**: The patch changes the allocated type of `SrcAlloca` if `DestSize > SrcSize`. If `SrcAlloca` is used in an `inalloca` call, changing its allocated type might invalidate the `inalloca` contract, as `inalloca` requires the alloca to match the argument type exactly.
- **Expected Issue**: The optimization will replace `DestAlloca` with `SrcAlloca`. If `SrcAlloca` is larger, `DestAlloca`'s users might now access a larger allocation. This is generally safe. But what if `SrcAlloca` is smaller, and we resize it to `DestAlloca`'s type? If `SrcAlloca` is part of a `byval` argument, changing its allocated type might change the semantics if the `byval` type relies on the alloca's type (though in modern LLVM, `byval` has an explicit type). What about `inalloca`? `inalloca` relies on the allocated type. If we change `SrcAlloca`'s allocated type, it might break `inalloca` assumptions.

### Strategy 4: Lifetime Marker Scope Mismatch

- **Target**: A test case where `SrcAlloca` has a `lifetime.end` before a use of `DestAlloca`.
- **Rationale**: The patch deletes `DestAlloca`'s lifetime markers (which were collected in `LifetimeMarkers`). It leaves `SrcAlloca`'s lifetime markers intact. If `DestAlloca` is used outside the region where `SrcAlloca` is marked alive (e.g., `SrcAlloca` has a `lifetime.end` before a use of `DestAlloca`), then replacing `DestAlloca` with `SrcAlloca` will result in `DestAlloca`'s users accessing dead memory. The patch does not check if `SrcAlloca`'s lifetime encompasses `DestAlloca`'s uses, nor does it remove `SrcAlloca`'s lifetime markers.
- **Expected Issue**: The optimization will replace `DestAlloca` with `SrcAlloca`. If `SrcAlloca` is larger than `DestAlloca`, `DestAlloca`'s lifetime markers are deleted. But `SrcAlloca`'s lifetime markers remain. If `DestAlloca`'s live range extends beyond `SrcAlloca`'s live range, removing `DestAlloca`'s lifetime markers and relying on `SrcAlloca`'s lifetime markers might cause the memory to be considered dead while `DestAlloca`'s users are still accessing it. Wait, the patch checks that `SrcAlloca` dominates `DestAlloca`'s users? No, it moves `SrcAlloca` to the entry block if it doesn't dominate. But it doesn't extend `SrcAlloca`'s lifetime markers to cover `DestAlloca`'s uses!

### Strategy 5: GEP Source Element Type Mismatch (Minor)

- **Target**: A test case where `SrcAlloca` is used by a GEP and is resized to a different type.
- **Rationale**: Changing the allocated type of `SrcAlloca` might cause a mismatch with the source element type of existing GEPs using `SrcAlloca`. While opaque pointers mitigate this, some analyses might still expect them to match.
- **Expected Issue**: The optimization will replace `DestAlloca` with `SrcAlloca`. If `SrcAlloca` is smaller than `DestAlloca`, it resizes `SrcAlloca` by setting its allocated type to `DestAlloca`'s allocated type. However, if `SrcAlloca` is an array allocation (e.g., `[4 x i32]`) and `DestAlloca` is a scalar allocation (e.g., `i64`), setting `SrcAlloca`'s type to `i64` changes its type. If there are `getelementptr` instructions that use `SrcAlloca`'s original allocated type as their source element type, they will not be invalidated (since GEPs store their own source element type), but if any pass relies on the alloca's allocated type matching the GEP's source element type, it might fail. More importantly, what if `SrcAlloca` has a custom alignment that is smaller than `DestAlloca`'s, the patch does `SrcAlloca->setAlignment(max)`. What if `SrcAlloca` has a different address space? The patch checks `SrcAlloca->getAddressSpace() != DestAlloca->getAddressSpace()`. What if `SrcSize` and `DestSize` are scalable? The patch checks `!SrcSize->isFixed() || !DestSize->isFixed()`.
