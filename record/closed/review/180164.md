### Strategy 1: Both fpexts have multiple uses, no subsequent fptrunc

- **Target**: min/max with fpext operands where one has multiple uses and the other has one use.
- **Rationale**: The patch relaxes the one-use check to allow the optimization when at least one fpext has one use. However, if both fpexts have multiple uses, the optimization is not applied. Wait, the patch says `(Arg0->hasOneUse() || Arg1->hasOneUse())`. So it requires at least one to have one use. If both have multiple uses, it won't apply. The patch description says "If only of the operands is one-use, the total number of fpexts stays the same, but the min/max is performed on a narrowed type. Additionally, the fpext may fold with a following fptrunc." Wait, if one operand has multiple uses, the fpext for that operand is kept. The new fpext is created. So the number of fpexts increases by 1 (the new one), but the one-use fpext is removed. So the total number of fpexts stays the same. If both have multiple uses, the number of fpexts would increase by 1. The patch correctly prevents this by requiring at least one to have one use. What if the new fpext is not folded? The number of fpexts stays the same, but we have a narrower min/max. Is there any case where a narrower min/max is worse? Usually narrower is better or equal. What about fast-math flags? The new min/max inherits the flags from the original min/max.
- **Expected Issue**: The optimization might be applied when both fpexts have multiple uses, but the resulting fpext might not be folded with a subsequent fptrunc, leading to an extra fpext instruction and potentially worse performance.

### Strategy 2: NaN handling with fpext and min/max

- **Target**: min/max with fpext operands where one is a signaling NaN.
- **Rationale**: Check if fpext followed by min/max has the same NaN semantics as min/max followed by fpext, especially for signaling NaNs.
- **Expected Issue**: The new min/max might have different semantics if the narrower type handles NaNs or subnormals differently, or if the fast-math flags are not preserved correctly. Wait, fpext is exact, so the values are exactly representable in the wider type. min/max on the wider type will return one of the exact values. fpext of the min/max on the narrower type will return the fpext of the narrower min/max. Since min/max just selects one of the operands (or handles NaN), the result should be identical. What about +0.0 and -0.0? minnum(-0.0, +0.0) can return either. If we do it in narrower type, it also returns either. fpext(-0.0) is -0.0, fpext(+0.0) is +0.0. So it's the same. What about sNaN? fpext of sNaN might quiet it? LLVM's fpext does not quiet sNaNs. minnum on sNaN might return NaN or quiet it. If we do minnum on narrower type, it might quiet it, then fpext quiets it? Wait, fpext of sNaN is sNaN in LLVM. minnum on sNaN might return the other operand if it's a number. If both are sNaN, it returns qNaN. fpext of qNaN is qNaN. So semantics should be preserved.

This copies the fast-math flags from `II` (the original min/max). The new `fpext` is created with `new FPExtInst(NewCall, II->getType())`. But fpext FMF is rarely used. Wait, the original min/max had FMF, which are copied to the new min/max. What if the original fpexts had `nnan` and the new fpext doesn't? ### Strategy 4: Scalable vectors

- **Target**: min/max with fpext operands of scalable vector types.
- **Rationale**: Check if the optimization works correctly for scalable vectors.
- **Expected Issue**: The optimization might be applied to vectors of fpexts. If `X` and `Y` are vectors, `X->getType() == Y->getType()` checks if they have the same vector type. `Builder.CreateBinaryIntrinsic` works for vectors. `FPExtInst` works for vectors. What if the vector has a mix of one-use and multi-use elements? `hasOneUse()` checks the whole vector instruction. So it's fine. What if the types are scalable vectors? `FPExtInst` and `CreateBinaryIntrinsic` support scalable vectors.

### Strategy 5: Target legality for narrower min/max

- **Target**: min/max with fpext operands where the narrower type is not natively supported (e.g., f16 on x86 without AVX512FP16).
- **Rationale**: Check if the optimization is applied even when the narrower min/max is not natively supported by the target, potentially leading to worse code generation.
- **Expected Issue**: The optimization might increase the critical path length if the multi-use fpext is computed early, but the one-use fpext is computed late. Wait, the new min/max depends on X and Y. X and Y are available before the fpexts. So the new min/max can be computed earlier than the original min/max. The new fpext depends on the new min/max. So the critical path for the result of min/max is: X, Y -> min/max -> fpext. Originally it was: X -> fpext, Y -> fpext -> min/max. The new path has the same length. For the multi-use fpext, its critical path is unchanged. So performance should be fine. What if the target doesn't support the narrower min/max natively, but supports the wider one? Then the narrower min/max will be expanded, potentially causing a performance regression. InstCombine generally assumes narrower operations are better or equal, but for some targets (e.g., f16 on older x86), f32 is natively supported while f16 is not. However, InstCombine doesn't check target legality for this fold. This is a general InstCombine issue, not specific to this patch.
