## Test Strategies

### Strategy 1: Vectorize sadd.with.overflow with scalarization

- **Target**: A loop containing `llvm.sadd.with.overflow` where the cost model forces scalarization (e.g., by using a specific target or VF=1).
- **Rationale**: The patch adds `sadd.with.overflow` to `isTriviallyVectorizable`, but removes it from `isTriviallyScalarizable`. If the cost model decides to scalarize the intrinsic (e.g., due to VF=1 or target cost), the scalarizer might not handle the struct return type correctly, especially since it's no longer considered trivially scalarizable.
- **Expected Issue**: The vectorizer will crash or generate invalid IR because it attempts to extract a vector element from a struct type, or it fails to scalarize the struct return type properly.

### Strategy 2: Vectorize uadd.with.overflow with struct return type

- **Target**: A loop containing `llvm.uadd.with.overflow` that gets vectorized.
- **Rationale**: The patch modifies `VPWidenIntrinsicRecipe::execute` to handle struct return types by iterating over `getContainedTypes(RetTy)`. However, `toVectorizedTy` might not correctly vectorize all elements of the struct if they have different scalar types (e.g., `i32` and `i1`), or the intrinsic declaration might expect a specific combination of vector types that `toVectorizedTy` doesn't produce.
- **Expected Issue**: The vectorizer will generate a vector intrinsic call with incorrect return type (e.g., `{ <4 x i32>, i1 }` instead of `{ <4 x i32>, <4 x i1> }`), leading to a verifier error or miscompilation.

### Strategy 3: Cost model for overflow intrinsics

- **Target**: A loop containing `llvm.smul.with.overflow` where the cost model is queried (e.g., during VPlan construction).
- **Rationale**: The patch adds overflow intrinsics to `isTriviallyVectorizable`, which means they will be costed using `getIntrinsicInstrCost`. The cost model might not correctly handle the struct return type of these intrinsics, especially since `toVectorizedTy` is used to determine the return type for the cost attributes.
- **Expected Issue**: The cost model will compute an incorrect cost for the intrinsic, potentially leading to suboptimal vectorization decisions or crashes if the cost function expects a different type structure.

### Strategy 4: ExtractValue from vectorized overflow intrinsic

- **Target**: A loop containing `llvm.usub.with.overflow` followed by `extractvalue` instructions to use both the result and the overflow flag.
- **Rationale**: Overflow intrinsics return a struct `{ result, overflow_flag }`. When vectorized, this becomes a struct of vectors `{ <v x result>, <v x overflow_flag> }`. If the original scalar code uses `extractvalue` to get the result or the flag, the vectorizer needs to correctly translate this to extracting from the vectorized struct. The patch modifies `VPWidenIntrinsicRecipe` to handle the struct return, but it's unclear if `extractvalue` recipes correctly handle extracting from a struct of vectors.
- **Expected Issue**: The vectorizer will generate invalid IR because it attempts to extract an element from a struct type using an incorrect index or type, or it fails to handle the multiple return values correctly.

### Strategy 5: Target support for vectorized overflow intrinsics

- **Target**: A loop containing `llvm.ssub.with.overflow` compiled for a target with limited vector support (e.g., an older ARM or x86 architecture).
- **Rationale**: The patch enables vectorization of overflow intrinsics globally. However, not all targets might support the vectorized versions of these intrinsics (e.g., `<4 x i32> @llvm.sadd.with.overflow.v4i32`). If a target doesn't support it, the backend might fail to lower it, or the vectorizer might generate invalid IR if it assumes target support.
- **Expected Issue**: The vectorizer will crash or generate invalid IR because it attempts to use a vector intrinsic that the target does not support, or it fails to lower the intrinsic correctly.
