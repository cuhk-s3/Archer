## Test Strategies

### Strategy 1: Fabs Removal with NSZ on Fabs but not on User

- **Target**: A test case where `fabs` has the `nsz` flag, its operand can be `-0.0`, and its user does not have the `nsz` flag and is sensitive to the sign of zero.
- **Rationale**: The fix uses `FMF.noSignedZeros()` (which comes from the `fabs` instruction) to decide whether to ignore the sign bit difference for `-0.0`. If `nsz` is present on the `fabs` instruction, it might return the operand directly. However, if the user of the `fabs` instruction does not have `nsz`, it might rely on the result being `+0.0` instead of `-0.0`. Returning the operand directly (which could be `-0.0`) violates the semantics for the user.
- **Expected Issue**: The `fabs` instruction is incorrectly removed, and its operand is used directly. If the operand is `-0.0`, the user will receive `-0.0` instead of `+0.0`. Since the user does not have the `nsz` flag, it might distinguish between `-0.0` and `+0.0` (e.g., via `copysign` or division), leading to incorrect results.

### Strategy 2: Fabs Removal with NSZ on Fabs but not on User (Multiple Uses)

- **Target**: A test case where `fabs` has multiple uses, has the `nsz` flag, its operand can be `-0.0`, and at least one user does not have the `nsz` flag and is sensitive to the sign of zero.
- **Rationale**: Similar to the single-use case, the multiple-use handling in `SimplifyMultipleUseDemandedFPClass` also uses `FMF.noSignedZeros()` from the `fabs` instruction to ignore the sign bit difference for `-0.0`. If the `fabs` has `nsz` but its users do not, returning the operand directly can lead to incorrect results for users sensitive to the sign of zero.
- **Expected Issue**: The `fabs` instruction is incorrectly removed, and its operand is used directly. If the operand is `-0.0`, the user will receive `-0.0` instead of `+0.0`. Since the user does not have the `nsz` flag, it might distinguish between `-0.0` and `+0.0` (e.g., via `copysign` or division), leading to incorrect results.

### Strategy 3: FNeg(Fabs) Removal with NSZ on Fabs but not on FNeg or User

- **Target**: A test case where `fneg(fabs(x))` is used, `fabs` has the `nsz` flag, `fneg` does not have the `nsz` flag, the operand can be `+0.0`, and the user of `fneg` does not have the `nsz` flag and is sensitive to the sign of zero.
- **Rationale**: In `SimplifyMultipleUseDemandedFPClass` for `FNeg`, it checks `FMF.noSignedZeros() || FabsFMF.noSignedZeros()` to ignore the sign bit difference for `-0.0`. If `FabsFMF` has `nsz` but `FMF` (from `fneg`) does not, it might return the operand directly. If the user of `fneg` does not have `nsz`, it might rely on the result being `-0.0` instead of `+0.0`. Returning the operand directly (which could be `+0.0`) violates the semantics for the user.
- **Expected Issue**: The `fabs` instruction is incorrectly removed, and its operand is used directly. If the operand is `-0.0`, the user will receive `-0.0` instead of `+0.0`. Since the user does not have the `nsz` flag, it might distinguish between `-0.0` and `+0.0` (e.g., via `copysign` or division), leading to incorrect results.
