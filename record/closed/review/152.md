## Test Strategies

### Strategy 1: External use by an unvectorizable store

- **Target**: A vectorizable tree where one of the scalar values is also used by a store instruction that is intentionally made unvectorizable (e.g., volatile store).
- **Rationale**: The `IsExternallyUsed` lambda explicitly ignores uses that are `StoreInst`s, assuming they will likely be vectorized later. However, if the store cannot be vectorized (e.g., it is volatile, has an incompatible alignment, or the pointers are not consecutive), the scalar will actually need to be extracted from the vector. Ignoring this use underestimates the extraction cost.
- **Expected Issue**: The tree is vectorized despite high extraction costs, leading to performance regression because the extraction cost was underestimated.

### Strategy 2: External use by an unvectorizable insertelement

- **Target**: A vectorizable tree where a scalar is used by an `insertelement` instruction that cannot be vectorized by SLP.
- **Rationale**: Similar to stores, `IsExternallyUsed` ignores uses that are `InsertElementInst`s. If the `insertelement` is part of a sequence that SLP cannot vectorize (e.g., inserting into a vector of a different size or type that isn't handled), the extraction cost is missed.
- **Expected Issue**: The extraction cost is underestimated, leading to unprofitable vectorization.

### Strategy 3: Minimum bitwidth mismatch for external use

- **Target**: A vectorizable tree with operations that can be demoted to a smaller bitwidth, where a scalar is externally used by an instruction requiring the original larger bitwidth (e.g., a `ret` or a call).
- **Rationale**: The patch uses the minimum bitwidth (`MinBWs`) to calculate the extraction cost. If a tree is demoted to a smaller bitwidth (e.g., `i8`), the extraction cost is calculated for `i8`. However, if the external use requires the original larger type (e.g., `i32`), the actual cost might involve extraction plus sign/zero extension, which might have a different cost than just extracting the smaller type.
- **Expected Issue**: Incorrect extraction cost calculation, potentially leading to wrong trimming decisions (either over-vectorizing or under-vectorizing).

### Strategy 4: High number of internal uses exceeding NumVectScalars

- **Target**: A vectorizable tree where a single scalar value is used many times across different lanes and instructions within the same tree, exceeding the total number of tree entries, but has no uses outside the tree.
- **Rationale**: `IsExternallyUsed` uses a heuristic: `if (V->hasNUsesOrMore(ScalarToTreeEntries.size() + 1)) return true;`. If a scalar value is heavily reused *within* the vectorizable tree (e.g., a splat-like behavior or a common operand across many lanes), it might exceed this threshold even if it has no external uses. This would incorrectly add an extraction cost.
- **Expected Issue**: Overestimation of extraction cost, leading to premature trimming of a profitable tree.

### Strategy 5: External use in a different basic block

- **Target**: A vectorizable tree in one basic block, where a scalar is used in a different, perhaps conditionally executed, basic block.
- **Rationale**: External uses in different basic blocks might have different cost implications, especially if they are in loops or cold blocks. The patch calculates a flat extraction cost. Testing this ensures the cost model handles cross-block external uses robustly.
- **Expected Issue**: The extraction cost might be miscalculated or the tree might be incorrectly trimmed if the cross-block extraction is not modeled properly.
