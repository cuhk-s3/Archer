## Test Strategies

### Strategy 1: Missing FastMathFlags on Widened Select

- **Target**: A loop with a floating-point phi node that has fast-math flags (e.g., `nnan`, `nsz`) and is vectorized into a blend (select).
- **Rationale**: The patch aims to propagate FastMathFlags from phi nodes to the widened select instructions (blends). If a phi node has fast-math flags (e.g., `nnan`, `nsz`), these flags should be preserved on the resulting select instruction. A test case with a floating-point phi node having fast-math flags should be vectorized, and the resulting select instruction should be checked for the presence of these flags.
- **Expected Issue**: The generated select instruction will not have the fast-math flags that were present on the original phi node, leading to missed optimization opportunities or incorrect semantics if the flags were relied upon.

### Strategy 2: FastMathFlags on Integer Select

- **Target**: A loop with an integer phi node that is vectorized into a blend (select).
- **Rationale**: FastMathFlags are only valid for floating-point operations. The patch adds `VPIRFlags` to `VPBlendRecipe` and `VPPhi`. If the logic incorrectly applies fast-math flags to an integer select instruction (e.g., by copying flags from an unrelated instruction or failing to check the type), it would result in invalid IR. A test case with an integer phi node should be vectorized to ensure no fast-math flags are applied to the resulting select.
- **Expected Issue**: The generated select instruction will have fast-math flags, which is invalid for integer types and may cause a verifier error or miscompilation.

### Strategy 3: FastMathFlags Propagation through VPlan CFG Builder

- **Target**: A loop with a floating-point phi node having fast-math flags, ensuring the flags are present on the `VPPhi` and `VPBlendRecipe` in the VPlan dump.
- **Rationale**: The patch modifies `PlainCFGBuilder::createVPInstructionsForVPBB` to pass the underlying `PHINode` to `createScalarPhi` to copy its flags. If this propagation fails or is incomplete, the flags won't reach the `VPPhi` and subsequently the `VPBlendRecipe`. A test case should verify that the flags are correctly propagated through the VPlan construction phase.
- **Expected Issue**: The generated select instruction will not have the fast-math flags, or the compiler might crash during VPlan construction or execution due to missing flag propagation.

### Strategy 4: FastMathFlags Propagation through VPlan Predicator

- **Target**: A loop with a floating-point phi node having fast-math flags that requires predication (e.g., inside an if-statement within the loop), ensuring the flags are present on the `VPBlendRecipe` in the VPlan dump.
- **Rationale**: The patch modifies `VPPredicator::convertPhisToBlends` to pass the `VPPhi` to the `VPBlendRecipe` constructor to copy its flags. If this propagation fails, the flags won't reach the `VPBlendRecipe`. A test case should verify that the flags are correctly propagated through the VPlan predication phase.
- **Expected Issue**: The generated select instruction will not have the fast-math flags, or the compiler might crash during VPlan predication due to missing flag propagation.

### Strategy 5: FastMathFlags Propagation during VPlan Execution

- **Target**: A loop with a floating-point phi node having fast-math flags, ensuring the final LLVM IR contains a select instruction with the correct fast-math flags.
- **Rationale**: The patch modifies `VPInstruction::generate` for `Instruction::Select` to use `Builder.CreateSelectFMF` and pass the fast-math flags. If this logic is flawed, the flags won't be applied to the final IR instruction. A test case should verify that the final LLVM IR contains the select instruction with the correct fast-math flags.
- **Expected Issue**: The generated select instruction will not have the fast-math flags, or the compiler might crash during VPlan execution due to missing flag propagation.
