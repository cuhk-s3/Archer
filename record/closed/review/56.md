## Test Strategies

### Strategy 1: Test maximum with NaN and non-NaN operands

- **Target**: InstCombine SimplifyDemandedUseFPClass for maximum/minimum/maximumnum/minimumnum
- **Rationale**: The patch changes the logic for `maximum` and `minimum` to use `cannotOrderStrictlyLess` and `cannotOrderStrictlyGreater`. It adds checks for `KnownRHS.isKnownNever(fcNan)` and `KnownLHS.isKnownNever(fcNan)`. We should test cases where one operand is known to be NaN and the other is not, or where both can be NaN, to ensure the correct operand is returned or no incorrect optimization is performed.
- **Expected Issue**: The optimization might incorrectly return the LHS operand when it should return the RHS operand, or vice versa, because it assumes that if LHS cannot be strictly less than RHS, then LHS must be greater than or equal to RHS. However, if LHS and RHS can be unordered (e.g., one is NaN), this assumption is false. The patch attempts to fix this by adding `KnownRHS.isKnownNever(fcNan)` and `KnownLHS.isKnownNever(fcNan)`, but it might still be flawed if both can be NaN or if the logic doesn't cover all cases.

### Strategy 2: Test minimum/maximum with -0.0 and +0.0

- **Target**: InstCombine SimplifyDemandedUseFPClass for maximum/minimum/maximumnum/minimumnum with zero operands
- **Rationale**: The patch explicitly mentions that it "can now consider -0 as less than or equal to +0 as appropriate, account for nsz." We should test cases involving `-0.0` and `+0.0` with and without the `nsz` (no signed zeros) fast-math flag to ensure the correct zero is returned.
- **Expected Issue**: The optimization might incorrectly fold `maximum(-0.0, +0.0)` to `-0.0` or `minimum(+0.0, -0.0)` to `+0.0` if it doesn't correctly handle the sign of zero. The patch introduces `OrderedZeroSign = !FMF.noSignedZeros()`, which affects how `cannotOrderStrictlyLess` and `cannotOrderStrictlyGreater` treat zeros.

### Strategy 3: Test with positive normal and negative subnormal

- **Target**: InstCombine SimplifyDemandedUseFPClass for maximum/minimum/maximumnum/minimumnum with normal and subnormal operands
- **Rationale**: The patch description states: "It additionally can handle cases like one half is known positive normal and the other subnormal." We should test cases where one operand is a positive normal and the other is a negative subnormal (or vice versa) to ensure the new logic correctly identifies the maximum/minimum.
- **Expected Issue**: The optimization might incorrectly fold the intrinsic if it assumes that a positive subnormal is always greater than a negative normal, but fails to account for the fact that subnormals might be flushed to zero depending on the denormal mode.

### Strategy 4: Test maximumnum/minimumnum with NaN

- **Target**: InstCombine SimplifyDemandedUseFPClass for maximumnum/minimumnum with NaN operands
- **Rationale**: The patch updates the logic for `maximumnum` and `minimumnum` as well. These intrinsics have different NaN handling semantics compared to `maximum` and `minimum`. We should test cases where one operand can be NaN to ensure the optimization respects the specific NaN semantics of `maximumnum` and `minimumnum`.
- **Expected Issue**: The optimization might incorrectly return the wrong operand if it doesn't correctly handle the difference between `maximum` (propagates NaN) and `maximumnum` (returns the non-NaN number).

### Strategy 5: Test cannotOrderStrictlyLess/Greater edge cases

- **Target**: InstCombine SimplifyDemandedUseFPClass for maximum/minimum/maximumnum/minimumnum with overlapping FP classes
- **Rationale**: The core of the patch replaces sign bit checks with `cannotOrderStrictlyLess` and `cannotOrderStrictlyGreater`. We should test edge cases of these functions, such as when both operands have overlapping possible classes (e.g., both can be positive zero or negative zero), to ensure the optimization doesn't make incorrect assumptions.
- **Expected Issue**: The optimization might incorrectly fold the intrinsic if it assumes that `cannotOrderStrictlyLess` implies `greater than or equal`, but fails to account for the fact that `cannotOrderStrictlyLess` might return true when the operands are unordered (e.g., due to NaN). The patch adds `isKnownNever(fcNan)` checks, but we should verify if there are any edge cases where this is insufficient.
