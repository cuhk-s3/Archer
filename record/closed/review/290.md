## Test Strategies

### Strategy 1: Dead Internal Function with Non-FP Arguments

- **Target**: A module with an internal function that has no users and takes non-FP arguments (e.g., `i32`).
- **Rationale**: The `addNoFPClassAttrsTopDown` function initializes `RetNoFPClass` and `ArgsNoFPClass` to `fcAllFlags`. If the function has no users (e.g., it is a dead internal function), the loop over users is skipped, and these masks remain `fcAllFlags`. The pass will then add `nofpclass(all)` to the return type and all arguments. If the types are not floating-point types, this will cause a verifier failure because `nofpclass` is only valid for FP types.
- **Expected Issue**: Verifier failure: "Attribute 'nofpclass' applied to incompatible type!"

### Strategy 2: Dead Internal Function with Non-FP Return Type

- **Target**: A module with an internal function that has no users and returns a non-FP type (e.g., `void` or `i32`).
- **Rationale**: Similar to the previous strategy, if a dead internal function returns a non-FP type (e.g., `void` or `i32`), `RetNoFPClass` remains `fcAllFlags` and `nofpclass(all)` is added to the return type, causing a verifier failure.
- **Expected Issue**: Verifier failure: "Attribute 'nofpclass' applied to incompatible type!"

### Strategy 3: Function with Users but Non-CallBase Users

- **Target**: A module with an internal function that has no users and returns a non-FP type.
- **Rationale**: If a function has users, but none of them are `CallBase` (e.g., its address is taken and stored, or used in a `ptrtoint`), the loop over users will return `false` early. Wait, if it returns `false` early, it doesn't add the attributes. So this is safe. But what if it has NO users? Then it doesn't return `false` early, and proceeds to add the attributes. So the target should be a function with NO users.
- **Expected Issue**: Verifier failure: "Attribute 'nofpclass' applied to incompatible type!"

### Strategy 4: Function with No Users and Mixed FP/Non-FP Arguments

- **Target**: A module with an internal function that has no users and takes both FP and non-FP arguments.
- **Rationale**: If a function has no users and takes a mix of FP and non-FP arguments, `nofpclass(all)` will be added to all arguments, including the non-FP ones, causing a verifier failure.
- **Expected Issue**: Verifier failure: "Attribute 'nofpclass' applied to incompatible type!"

### Strategy 5: Dead Internal Function with FP Arguments/Return Type

- **Target**: A module with an internal function that has no users and takes FP arguments and returns an FP type.
- **Rationale**: Even if the function has FP arguments and return type, adding `nofpclass(all)` to them when the function has no users is semantically incorrect (it means the function always returns poison and takes poison arguments). While this might not crash the verifier, it is an unintended consequence of the initialization logic.
- **Expected Issue**: Incorrect `nofpclass(all)` attribute added to FP arguments/return type

### Analysis

Bug Triggering Analysis:
The `addNoFPClassAttrsTopDown` function checks if `OldAttr != RetNoFPClass`. If they are different, it adds `RetNoFPClass` to the function and sets `Changed = true`.
However, `RetNoFPClass` is the intersection of the `nofpclass` attributes from all callsites.
If the function already has a `nofpclass` attribute that is a SUPERSET of `RetNoFPClass` (e.g., the function has `nofpclass(nan inf)` but the callsite only has `nofpclass(nan)`), then `OldAttr` (`nan inf`) is different from `RetNoFPClass` (`nan`).
The pass will call `F.addRetAttr(Attribute::getWithNoFPClass(Ctx, RetNoFPClass))`.
Because `addAttribute` merges the attributes, the new attribute will be `nan inf | nan = nan inf`.
So the IR is NOT actually changed!
But the pass sets `Changed = true`!
This means the pass reports that it modified the IR, even though it didn't.
If the pass manager runs this pass in a loop (e.g., CGSCC pass manager), it might loop infinitely or cause unnecessary invalidation of analyses!

Fix Weakness Analysis:
The weakness is that the pass doesn't check if `RetNoFPClass` is a SUBSET of `OldAttr`.
It only checks `OldAttr != RetNoFPClass`.
It should check `(OldAttr | RetNoFPClass) != OldAttr` or `(RetNoFPClass & ~OldAttr) != 0`.
The same issue applies to `ArgNoFPClass`.
If `OldAttr == ArgNoFPClass` is false, it adds the attribute and sets `Changed = true`.
But if `ArgNoFPClass` is a subset of `OldAttr`, adding it does nothing, but `Changed` is set to `true`.
This is a classic "infinite loop in pass manager" bug.
