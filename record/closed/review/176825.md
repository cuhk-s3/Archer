## Test Strategies

### Strategy 1: Alias Store Escape

- **Target**: A test case where the tracked pointer is stored into an alloca, but that alloca is accessed via an alias (e.g., a pointer passed as an argument or returned from a function).
- **Rationale**: The tracker assumes that storing the tracked pointer into an alloca is safe and propagates tracking to that alloca. However, if the destination alloca is aliased by another pointer, and the tracked pointer is loaded through that alias, the tracker will miss the load and incorrectly assume the pointer hasn't escaped.
- **Expected Issue**: The pass will incorrectly eliminate the tail call, leading to a miscompilation where the loaded value is incorrect because the store to the alias was not considered an escape.

### Strategy 2: Load from Non-Alloca Location

- **Target**: A test case where the tracked pointer is stored into an alloca, loaded from the alloca as an integer (e.g., via a bitcast or ptrtoint), and then stored into a global.
- **Rationale**: The tracker only tracks loads if the load pointer is in `LocationsWithAllocaAddresses`. If the tracked pointer is stored into an alloca, and then that alloca's address is stored into a global or passed to a function, the tracker will mark the alloca as escaped. However, if the tracked pointer is loaded from the alloca *before* the alloca escapes, the tracker will miss the load if it doesn't properly handle the order of instructions or if it assumes that an escaped alloca means all its contents are escaped. Wait, the tracker marks the *store* as an escape point if the destination is not an alloca. But what if the tracked pointer is stored into an alloca, and then the tracked pointer is loaded from that alloca, and *then* the loaded pointer is stored into a global? The tracker will see the load, but since it only continues tracking if the loaded value is a pointer type, it will track the loaded value. But what if the loaded value is cast to an integer and then stored? The tracker will stop tracking at the load because `I->getType()->isPointerTy()` is false.
- **Expected Issue**: The pass will incorrectly eliminate the tail call, leading to a miscompilation where the loaded value is incorrect because the load was not considered an escape.

### Strategy 3: Load Type Mismatch

- **Target**: A test case where the tracked pointer is stored into an alloca, loaded as an integer type, and then stored into a global or used in a way that escapes.
- **Rationale**: The tracker only continues tracking a loaded value if `I->getType()->isPointerTy()` is true. If the tracked pointer is stored into an alloca, and then loaded as an integer type (e.g., `i64`), the tracker will stop tracking it. If this integer is then cast back to a pointer and used to access the alloca, or stored into a global, the tracker will miss the escape.
- **Expected Issue**: The pass will incorrectly eliminate the tail call, leading to a miscompilation where the loaded value is incorrect because the load was not considered an escape.

### Strategy 4: PHI Node Tracking

- **Target**: A test case with a loop where the tracked pointer is stored into an alloca at the end of the loop, and loaded from the alloca at the beginning of the loop.
- **Rationale**: The tracker handles `PHI` nodes by adding them to the worklist. However, if the tracked pointer is stored into an alloca, and the alloca's address is used in a `PHI` node, the tracker will track the `PHI` node. But what if the tracked pointer itself is used in a `PHI` node, and the `PHI` node is stored into a global? The tracker handles `PHI` nodes by adding them to the worklist, so it will track the `PHI` node and see the store to the global. What if the tracked pointer is stored into an alloca, and the alloca is loaded, and the loaded value is used in a `PHI` node? The tracker will track the loaded value (if it's a pointer), and then track the `PHI` node. What if the tracked pointer is stored into an alloca, and the alloca is used in a `Select` instruction? The tracker handles `Select` by adding it to the worklist. What if the tracked pointer is stored into an alloca, and the alloca is used in a `GetElementPtr`? The tracker handles `GetElementPtr` by adding it to the worklist. What if the tracked pointer is stored into an alloca, and the alloca is used in a `BitCast`? The tracker handles `BitCast` by adding it to the worklist. What if the tracked pointer is stored into an alloca, and the alloca is used in an `AddrSpaceCast`? The tracker handles `AddrSpaceCast` by adding it to the worklist. What if the tracked pointer is stored into an alloca, and the alloca is used in an `ICmp`? The tracker handles `ICmp` by marking it as an escape point (default case). This seems correct. Let's look at the `Store` case again. `if (Value *Dest = SI->getPointerOperand()->stripPointerCasts(); isa<AllocaInst>(Dest))` - it strips pointer casts from the destination. What if the destination is a `GetElementPtr` of an alloca? `isa<AllocaInst>(Dest)` will be false, so it will mark it as an escape point. This is conservative but correct. What if the destination is a `PHI` node that resolves to an alloca? `isa<AllocaInst>(Dest)` will be false, so it will mark it as an escape point. This is also conservative but correct. What if the destination is an alloca, but the store is not dominated by the alloca? That's impossible. What if the store is in a loop? The tracker uses `DT.dominates(SI, UserInst)` to filter uses. If the store is in a loop, and the load is in the same loop but before the store, `DT.dominates(SI, UserInst)` will be false. But the load could read the value stored in the previous iteration! This is a classic flow-sensitivity bug in loops.
- **Expected Issue**: The pass will incorrectly eliminate the tail call, leading to a miscompilation where the loaded value is incorrect because the PHI node was not considered an escape.

### Strategy 5: Loop Flow Sensitivity Bug

- **Target**: A test case with a loop where an alloca address is stored into a local alloca at the end of the loop body, and loaded from that local alloca at the beginning of the loop body. The loaded address is then escaped or used in a way that should prevent TRE.
- **Rationale**: The tracker uses `DT.dominates(SI, UserInst)` to filter uses of the destination alloca. It assumes that if a use is not dominated by the store, it cannot read the stored value. However, in a loop, a load that is chronologically before the store in the loop body can read the value stored in the previous iteration. Since the load is not dominated by the store, the tracker will ignore it, missing a potential escape or use of the tracked pointer.
- **Expected Issue**: The pass will incorrectly eliminate the tail call, leading to a miscompilation where the loaded value is incorrect because the load in the loop was not considered an escape.
