### Strategy 1: FP cmpxchg with +0.0 and -0.0

- **Target**: A select instruction where the condition is the success flag of a cmpxchg, the true value is the loaded value of the cmpxchg, and the false value is a float. The compare operand of the cmpxchg is a bitcast of the false value. The false value is -0.0, and the compare operand is +0.0.
- **Rationale**: The fix assumes that if the compare operand is a bitcast of the select value, they are equivalent. However, for floating-point values, +0.0 and -0.0 have different integer representations but compare equal in FP. If the compare operand is +0.0 and the select value is -0.0, the cmpxchg will fail, and the select will return the select value (-0.0). But the folded instruction will return the compare operand (+0.0), which is incorrect.
- **Expected Issue**: The select instruction will be incorrectly folded to the false value, which is the compare operand. However, the compare operand is a bitcast of the false value, and the false value is a float. If the float is -0.0, its integer representation is 0x80000000. If the compare operand is +0.0, its integer representation is 0x00000000. The cmpxchg will fail, and the select will return the false value, which is -0.0. But the folded instruction will return the compare operand, which is +0.0. This is incorrect.

define double @cmpxchg_fp(ptr %ptr, double %compare, i64 %new_value) {
  %bc = bitcast double %compare to i64
  %t0 = cmpxchg ptr %ptr, i64 %bc, i64 %new_value acq_rel monotonic, align 8
  %t1 = extractvalue { i64, i1 } %t0, 1

I have tested all the strategies on all the tests.
I found that the transformation is actually correct.
The patch extends `foldSelectCmpXchg` to handle floating-point and vector compare-exchange operations. It allows peeking through a `bitcast` when extracting the value loaded by a `cmpxchg`. It also introduces `isCompareSameAsValue` to check if the compare value of a `cmpxchg` matches another value, handling `bitcast` and constant FP/Int conversions.
My initial analysis in Phase 1 was that if the compare operand is `+0.0` and the select value is `-0.0`, the `cmpxchg` will fail, and the `select` will return the select value (`-0.0`). But the folded instruction will return the compare operand (`+0.0`), which is incorrect.
However, `isCompareSameAsValue` only returns true if the compare operand and the select value are bitwise equal.
If they are bitwise equal, then if `success` is true, the loaded value is bitwise equal to the compare operand, which is bitwise equal to the select value. So the loaded value is bitwise equal to the select value. Returning the select value instead of the loaded value is correct.
If `success` is false, the original `select` returns the false value. The transformed IR returns the false value. So they are exactly the same.
Therefore, there is no bug in the patch. The transformation is correct.
