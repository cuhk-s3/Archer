## Test Strategies

### Strategy 1: Test maxnum with NaN and positive number

- **Target**: A `maxnum` instruction where LHS is known positive and RHS is known negative or NaN.
- **Rationale**: The fix adds support for `maxnum` and `minnum` in `simplifyDemandedFPClassMinMax`. The logic relies on `cannotOrderStrictlyLess` and `cannotOrderStrictlyGreater`. If one operand is known to be greater/less than the other, it returns that operand, UNLESS the other operand is a NaN. The fix checks `KnownRHS.isKnownNever(fcNan)` or `KnownLHS.isKnownNever(fcNan)`. If we provide an input where one operand is a NaN and the other is a number, `maxnum` should return the number. However, if the optimization incorrectly assumes it can just return the LHS/RHS without checking for NaN, it might return the NaN instead. Wait, `maxnum` returns the number if one is NaN. The code says: `if (cannotOrderStrictlyLess(...) && KnownLHS.isKnownNever(fcNan)) return CI->getArgOperand(0);`. If `LHS` is known never NaN, and `LHS >= RHS`, it returns `LHS`. If `RHS` could be NaN, `maxnum(LHS, NaN)` is `LHS`. So returning `LHS` is correct! What if `LHS` could be NaN? Then it doesn't return `LHS`. What if `RHS` is known never NaN, and `LHS >= RHS`? It doesn't return `LHS` because `LHS` could be NaN, and `maxnum(NaN, RHS)` is `RHS`. So the logic is correct. Let's test the case where `LHS` is known never NaN, `RHS` could be NaN, and `LHS >= RHS`. The optimization should return `LHS`.
- **Expected Issue**: The optimization will incorrectly simplify the instruction, potentially returning the wrong operand or dropping NaN propagation.
