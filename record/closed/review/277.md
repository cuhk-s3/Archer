## Test Strategies

### Strategy 1: Test Attribute Propagation for Promoted Internal Functions

- **Target**: Create a ThinLTO test with an internal function that gets promoted (e.g., by being exported). Check if attributes like `norecurse` or `nounwind` are correctly propagated to it and its callers.
- **Rationale**: The patch modifies `calculatePrevailingSummary` in `FunctionAttrs.cpp` to consider promoted symbols as prevailing (`assert(IsPrevailing(VI.getGUID(), GVS.get()) || GVS->wasPromoted());`). If a function is promoted from internal to external linkage, it should still be considered for attribute propagation. We need to ensure that when an internal function is promoted, its attributes are correctly propagated to its callers, and it is correctly identified as prevailing.
- **Expected Issue**: The function attributes (like `norecurse`, `nounwind`) might not be correctly propagated to the promoted function, or the function might be incorrectly analyzed as not prevailing, leading to missed optimization opportunities or incorrect attribute propagation.

### Strategy 2: Test Linkage Resolution for Promoted Symbols

- **Target**: Create a ThinLTO test where an internal symbol is promoted (e.g., due to being exported). Verify that its linkage in the summary and the final IR remains `ExternalLinkage` and is not changed to `AvailableExternallyLinkage`.
- **Rationale**: The patch modifies `thinLTOResolvePrevailingGUID` in `LTO.cpp` to avoid setting the linkage of promoted symbols to `AvailableExternallyLinkage`. Promoted symbols should retain their `ExternalLinkage`. We need to test a scenario where a symbol is promoted and ensure its linkage is not downgraded to `AvailableExternallyLinkage` during the prevailing resolution phase.
- **Expected Issue**: The promoted symbol might be incorrectly assigned `AvailableExternallyLinkage` instead of keeping its `ExternalLinkage`, leading to a link error if the definition is elided.

### Strategy 3: Test Assertion on Prevailing Promoted Symbols

- **Target**: Create a ThinLTO test with multiple modules where an internal symbol is promoted, and try to force a situation where it might be checked for prevailing-ness, perhaps by having a weak symbol with the same name (though name mangling should prevent this, it's worth testing the boundary).
- **Rationale**: The patch adds an assertion in `thinLTOResolvePrevailingGUID` that a promoted symbol should not have a prevailing variant. We should try to construct an edge case where a promoted symbol might be evaluated by `isPrevailing` to see if the assertion holds or if it can be violated. This might involve complex import/export scenarios or name collisions.
- **Expected Issue**: The compiler might crash due to the assertion `assert(!S->wasPromoted() && "promoted symbols used to be internal linkage and shouldn't have a prevailing variant");` if a promoted symbol is somehow considered prevailing in the traditional sense (e.g., if it was incorrectly marked as prevailing or if there's a logic error in how prevailing-ness is determined for promoted symbols).

### Strategy 4: Test `setExternalLinkageForTest` Behavior

- **Target**: Run existing tests that use `llvm-link` with summary index or `opt -function-import` to ensure they don't crash on the new assertions in `setLinkage` and that they correctly handle the simulated external linkage.
- **Rationale**: The patch introduces `setExternalLinkageForTest` to replace `setLinkage(GlobalValue::ExternalLinkage)` in test-specific code paths (`FunctionImport.cpp` and `llvm-link.cpp`). This is because `setLinkage` now asserts that the linkage is not external (unless `promote` is used). We need to ensure that tests relying on this behavior still work correctly and that the `Promoted` flag is NOT set in these test scenarios, as they are just simulating external linkage for testing purposes.
- **Expected Issue**: The test might fail or behave incorrectly if the `Promoted` flag is not set or if the linkage is not correctly forced to `ExternalLinkage` during the test setup.

### Strategy 5: Test Linkage Modification After Promotion

- **Target**: Create a ThinLTO test that triggers promotion and then goes through various ThinLTO passes (like internalization, weak resolution, etc.) to ensure none of them attempt to call `setLinkage` on the promoted symbol.
- **Rationale**: The patch adds an assertion in `setLinkage` that the symbol was not promoted (`assert(!wasPromoted());`). This means once a symbol is promoted, its linkage should not be changed via `setLinkage`. We need to ensure there are no paths in the ThinLTO pipeline that attempt to change the linkage of a promoted symbol after it has been promoted.
- **Expected Issue**: The compiler might crash due to the assertion `assert(!wasPromoted());` in `setLinkage` if a promoted symbol's linkage is attempted to be changed later in the pipeline.

### Analysis

Bug Triggering Analysis:
The provided test case triggers a crash in the `function-import` pass when run via `opt`. The crash occurs because the `function-import` pass requires a summary file (`-summary-file`), which is not provided in the command line arguments. This is a known requirement for the `function-import` pass, and the crash is a direct result of missing this required argument.

Fix Weakness Analysis:
The fix introduces a new `Promoted` flag in `GlobalValueSummary` and modifies how promoted symbols are handled during ThinLTO. However, the crash observed in the test case is not directly related to the logic introduced by the fix. Instead, it highlights a weakness in the test setup or the way the `function-import` pass is invoked without the necessary summary file. The fix itself seems to correctly handle the promotion of internal symbols and their linkage resolution, but the test case fails due to a missing prerequisite for the pass.

To properly test the fix, the test case should be run with a valid summary file, or the test setup should be adjusted to generate and provide the summary file. The current test case demonstrates that the `function-import` pass cannot be run in isolation without a summary file, which is a limitation of the pass itself rather than a weakness in the fix.
