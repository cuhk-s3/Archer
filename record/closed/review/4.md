## Test Strategies

### Strategy 1: Non-contiguous or unaligned loads

- **Target**: A sequence of scalar loads that are not contiguous or have different alignments, followed by zext and shl/or operations.
- **Rationale**: The patch adds support for scalar loads + bswap pattern. It checks if the source of the ZExt is a load. However, it might not properly check if the loads are contiguous or if they have the correct alignment for vectorization, especially when combined with bswap.
- **Expected Issue**: The vectorizer might incorrectly assume that the load is contiguous or has a specific alignment, leading to incorrect vectorization or a crash during code generation.

### Strategy 2: Different types or sizes

- **Target**: A sequence of scalar loads with types other than i8 (e.g., i16, i32) followed by zext and shl/or operations.
- **Rationale**: The patch checks if the type size in bits is equal to `ByteSize` (8 bits). It might not correctly handle cases where the types are different or the sizes do not match the expected byte size, especially when dealing with bswap which requires specific type sizes.
- **Expected Issue**: The vectorizer might incorrectly combine the loads and bswap, leading to incorrect results or a crash if the types do not match the expected sizes.

### Strategy 3: Multiple uses of the load

- **Target**: A sequence of scalar loads where some loads have multiple uses, followed by zext and shl/or operations.
- **Rationale**: The patch checks if the load has one use (`all_of(SrcTE->Scalars, [](Value *V) { return V->hasOneUse(); })`). However, there might be edge cases where the load has multiple uses, but the vectorizer still tries to vectorize it, or the check is bypassed.
- **Expected Issue**: The vectorizer might incorrectly vectorize the loads, ignoring the multiple uses, which could lead to incorrect results or a crash if the other uses expect the original scalar values.

### Strategy 4: BSwap intrinsic not supported

- **Target**: A sequence of scalar loads followed by zext and shl/or operations on a target or with a type where the bswap intrinsic is not supported.
- **Rationale**: The patch adds support for the bswap intrinsic. However, the bswap intrinsic might not be supported for all types or targets. The vectorizer might incorrectly assume it is supported and generate invalid code.
- **Expected Issue**: The vectorizer might incorrectly combine the loads and bswap, leading to incorrect results or a crash if the bswap intrinsic is not supported for the specific type or target.

### Strategy 5: Cost calculation issues

- **Target**: A sequence of scalar loads followed by zext and shl/or operations with varying costs (e.g., different alignments, different address spaces).
- **Rationale**: The patch modifies the cost calculation for the new `ReducedBitcastLoads` and `ReducedBitcastBSwapLoads` combined opcodes. There might be edge cases where the cost is calculated incorrectly, especially when dealing with poison values or different cost kinds.
- **Expected Issue**: The vectorizer might incorrectly calculate the cost, leading to suboptimal vectorization or incorrect decisions.
