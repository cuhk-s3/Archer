## Test Strategies

### Strategy 1: Non-Call/Invoke User of coro.subfn.addr

- **Target**: Create a test where `coro.subfn.addr` is used by something other than a direct call/invoke, or has multiple uses.
- **Rationale**: The `elideCoroNoop` function iterates over users of `coro.noop`, finds `coro.subfn.addr`, and then gets its unique undroppable user. It assumes this user is either a `CallInst` or `InvokeInst`. If it's something else (e.g., a bitcast, or stored to memory), `dyn_cast<CallInst>` and `dyn_cast<InvokeInst>` will both return null, and the loop will just `continue`, leaving the `coro.subfn.addr` and its user intact, which might lead to a crash later or incorrect code generation since the `coro.noop` is elided but its subfn is not. Wait, if it continues, it doesn't erase `Fn` (the `coro.subfn.addr`). Then `lowerCoroNoop` might be called if `!II->user_empty()`. But wait, `elideCoroNoop` doesn't remove `Fn` from `II`'s users if it's not a call/invoke. So `II` will not be user_empty, and `lowerCoroNoop` will replace `II` with the noop coroutine frame. This is actually safe! Wait, what if the user of `coro.subfn.addr` is a `CallInst` but it has uses? `Call->eraseFromParent()` will assert if `Call` has uses. A resume/destroy call returns `void`, so it shouldn't have uses. But what if it's an invoke? `I->eraseFromParent()` will also assert if it has uses. What if the `coro.subfn.addr` has multiple users? `getUniqueUndroppableUser()` will return `nullptr`. Then `dyn_cast<CallInst>(nullptr)` is null, so it continues. This is also safe. Wait, what if the `CallInst` is an indirect call that doesn't return void? Resume/destroy calls return void.
- **Expected Issue**: The elision logic will crash or misbehave because it assumes the user of `coro.subfn.addr` is a `CallInst` or `InvokeInst` and directly calls `eraseFromParent()` on it, but it could be a `BitCastInst` or other instruction if the user is not a direct call.

### Strategy 2: InvokeInst Unwind Edge Not Removed

- **Target**: Create a test where `coro.noop` is used in an `invoke` instruction, and the unwind destination has a `landingpad` and a `phi` node.
- **Rationale**: When `elideCoroNoop` processes an `InvokeInst`, it creates a branch to the normal destination (`I->getNormalDest()`) and erases the `InvokeInst`. However, it completely ignores the unwind destination (`I->getUnwindDest()`). The unwind block will still think the block containing the erased `InvokeInst` is a predecessor, but there is no longer a branch to it. `removePredecessor` must be called on the unwind destination to update PHI nodes and maintain CFG consistency.
- **Expected Issue**: The `InvokeInst` is erased, but its `unwind` destination block might still have a PHI node that expects an incoming value from the `InvokeInst`'s block, or the `unwind` block might become unreachable but still have predecessors. The `elideCoroNoop` function replaces the `InvokeInst` with a branch to the normal destination, but it does not remove the predecessor edge from the unwind destination block. This will cause a malformed CFG and a verifier error: "PHI node has entry for basic block that is not a predecessor!" or similar.

### Strategy 3: coro.subfn.addr Passed as Argument to Call/Invoke

- **Target**: Create a test where the result of `coro.subfn.addr` is passed as an argument to a function call, rather than being the called function.
- **Rationale**: `elideCoroNoop` checks if the unique user of `coro.subfn.addr` is a `CallInst` or `InvokeInst`. It does NOT check if the `coro.subfn.addr` is the actual callee of the `CallInst`/`InvokeInst`. If the subfn address is passed as an argument to a completely different function call, `elideCoroNoop` will blindly erase that function call, which is a severe miscompilation.
- **Expected Issue**: The `coro.noop` intrinsic is elided, but the `coro.subfn.addr` might be used in a `CallInst` that has operand bundles (e.g., `funclet`). Erasing the call is fine, but what if the call is an `InvokeInst`? The replacement `BranchInst` will not have the operand bundles, which is fine since it's just a branch. But wait, what if the `CallInst` or `InvokeInst` is not actually calling the resume/destroy function, but passing the function pointer as an argument to another function? `Fn->getUniqueUndroppableUser()` returns the `CallInst`. If the `CallInst` is calling some other function and passing `Fn` as an argument, `elideCoroNoop` will erase the `CallInst`! This changes the semantics of the program incorrectly.

### Strategy 4: CallInst with Uses Erased

- **Target**: Create a test where the `CallInst` using `coro.subfn.addr` has a use (e.g., assigned to a value and returned).
- **Rationale**: `elideCoroNoop` calls `Call->eraseFromParent()` without checking if `Call` has uses. If the IR contains a call to the resume/destroy function pointer that somehow has uses (e.g., if the function type was bitcasted to return a value), erasing it will cause an assertion failure in LLVM.
- **Expected Issue**: The `coro.noop` intrinsic is elided, but the `coro.subfn.addr` might be used in a `CallInst` where the `coro.subfn.addr` is the callee, but the call has a non-void return type (which is invalid for resume/destroy, but possible in IR). If the call has uses, `Call->eraseFromParent()` will assert.

### Strategy 5: Partial Elision of coro.noop

- **Target**: Create a test where `coro.noop` is used by a `coro.subfn.addr` that is directly called, and also used by another instruction (e.g., `ptrtoint` or stored to memory).
- **Rationale**: If `coro.noop` has multiple uses, and only some of them are elided (e.g., one is a direct call, another is stored to memory), `elideCoroNoop` will erase the direct call and leave the store. Then `lowerCoroNoop` will replace `coro.noop` with the global noop frame. We should test that this partial elision works correctly and doesn't crash.
- **Expected Issue**: The `coro.noop` intrinsic is elided, but the `coro.subfn.addr` might be used in a `CallInst` that is a tail call. Erasing it is fine. What if the `coro.noop` is used by multiple `coro.subfn.addr` instructions, and some are elided while others are not? The ones that are not elided will keep `coro.noop` alive, so `lowerCoroNoop` will be called. `lowerCoroNoop` replaces `coro.noop` with a pointer to the noop frame. This is correct. What if `NoopCoro` is created, but the module has no datalayout? `M.getDataLayout().getProgramAddressSpace()` might fail or return a default. This is probably fine. What about `buildDebugInfoForNoopResumeDestroyFunc`? It creates a DISubprogram. If the module has debug info, it adds it. But it doesn't add the subprogram to the CU's subprogram list, which might be required in older LLVM versions, but in modern LLVM `setSubprogram` is enough. Wait, `buildDebugInfoForNoopResumeDestroyFunc` creates a `DISubprogram` with `LineNo=0`. Is it possible that `M.debug_compile_units_begin()` is invalid if `debug_compile_units()` is not empty but the iterator is somehow invalidated? No.

Bug Triggering Analysis:
The provided test case triggers the bug because it uses `invoke` to call the `coro.subfn.addr` of a `coro.noop`. The `CoroCleanup` pass attempts to elide this call by replacing the `invoke` with a `br` to the normal destination and erasing the `invoke`. However, it fails to remove the predecessor edge from the unwind destination block. The unwind block contains a `phi` node that expects an incoming value from the `entry` block (where the `invoke` was). Since the `invoke` is erased and no branch goes to the unwind block anymore, the CFG becomes invalid, causing a verifier error: "PHI nodes not grouped at top of basic block!" (which is a symptom of the broken CFG where the block is no longer a predecessor).

Fix Weakness Analysis:
The fix weakness is in `Lowerer::elideCoroNoop` where it handles `InvokeInst`:
```cpp
    if (auto *I = dyn_cast<InvokeInst>(User)) {
      Builder.SetInsertPoint(I);
      Builder.CreateBr(I->getNormalDest());
      I->eraseFromParent();
      Fn->eraseFromParent();
    }
```
It completely ignores the `unwind` destination of the `InvokeInst`. When an `InvokeInst` is removed or replaced with a branch that doesn't go to the unwind destination, the unwind destination must have its predecessor removed using `I->getUnwindDest()->removePredecessor(I->getParent())`. Failing to do so leaves the CFG in an inconsistent state, leading to crashes in subsequent passes or the verifier.
