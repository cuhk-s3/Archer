## Test Strategies

### Strategy 1: Poison Propagation with nsw/nuw Flags

- **Target**: Test cases involving `icmp` over `min/max` with `nsw`/`nuw` flags or poison values.
- **Rationale**: The patch decomposes `icmp pred min/max(X, Y), Z` into `(X pred Z) and/or (Y pred Z)`. If the original `min/max` intrinsic or the `icmp` instruction had poison-generating flags (like `nsw`, `nuw`, `exact`), these flags might be incorrectly propagated or dropped, leading to incorrect poison semantics. We should test cases where `X` or `Y` are poison or where the comparison involves poison-generating flags.
- **Expected Issue**: The optimization might incorrectly fold the comparison, leading to a miscompilation where the result of the comparison is wrong for poison values.

### Strategy 2: Vector Types with Poison Elements

- **Target**: Test cases involving `icmp` over `min/max` with vector types containing poison elements.
- **Rationale**: The patch applies to vector types as well. If the vector contains poison elements, the decomposition into `and`/`or` might not handle the poison elements correctly, especially if the original `min/max` intrinsic had specific poison semantics for vector elements. We should test cases with vector types containing poison elements.
- **Expected Issue**: The optimization might incorrectly fold the comparison, leading to a miscompilation where the result of the comparison is wrong for vector types with poison elements.

### Strategy 3: Non-Splat Vector Constants

- **Target**: Test cases involving `icmp` over `min/max` where the constant `Z` is a non-splat vector constant.
- **Rationale**: The patch checks `isa<Constant>(Z)`. For vector types, `Z` could be a non-splat vector constant. The decomposition into `and`/`or` might not be valid or might have different semantics for non-splat vector constants. We should test cases where `Z` is a non-splat vector constant.
- **Expected Issue**: The optimization might incorrectly fold the comparison, leading to a miscompilation where the result of the comparison is wrong for non-splat vector constants.

### Strategy 4: Floating-Point Types with NaN Values

- **Target**: Test cases involving `fcmp` over `minnum/maxnum` with NaN values (if applicable, though the patch specifically mentions `icmp`).
- **Rationale**: Although the patch is in `InstCombineCompares.cpp` and seems to target integer comparisons (`ICmpInst`), we should ensure it doesn't accidentally apply to floating-point comparisons (`FCmpInst`) or that similar logic for floating-point comparisons handles NaN values correctly. If it does apply to floating-point, the decomposition might not preserve NaN semantics.
- **Expected Issue**: The optimization might incorrectly fold the comparison, leading to a miscompilation where the result of the comparison is wrong for floating-point types with NaN values.

### Strategy 5: Multi-Use Min/Max Intrinsics

- **Target**: Test cases involving `icmp` over `min/max` where the `min/max` intrinsic has multiple uses.
- **Rationale**: The patch checks `MinMax->hasOneUse()`. We should verify that the optimization is NOT applied when the `min/max` intrinsic has multiple uses, as this could lead to code bloat or incorrect semantics if the other uses rely on the original `min/max` value.
- **Expected Issue**: The optimization might incorrectly fold the comparison, leading to a miscompilation where the result of the comparison is wrong for multi-use `min/max` intrinsics.
