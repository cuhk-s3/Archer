## Test Strategies

### Strategy 1: EH Pad Predecessor Split

- **Target**: A test case with a conditional branch in an EH pad (e.g., `catchpad` or `cleanuppad`) where the condition is known in a predecessor.
- **Rationale**: The fix adds a null check for `EdgeBB` returned by `SplitBlockPredecessors`. `SplitBlockPredecessors` returns `nullptr` if `BB->canSplitPredecessors()` is false. `canSplitPredecessors()` returns false for EH pads (like `catchswitch`, `catchpad`, `cleanuppad`). If `foldCondBranchOnValueKnownInPredecessorImpl` is called on a block that is an EH pad, `SplitBlockPredecessors` will return `nullptr`, and without the fix, it would dereference it.
- **Expected Issue**: The compiler will crash with a null pointer dereference when trying to access `EdgeBB` because `SplitBlockPredecessors` returns `nullptr` when the block cannot be split (e.g., it's an EH pad).

Bug Triggering Analysis:
The test case contains a `cleanuppad` block (`cleanup`) with a conditional branch `br i1 %c, label %cleanup_2, label %cpret`. The condition `%c` is a phi node that depends on the predecessor (`true` from `entry`, `false` from `bb1`).
When `SimplifyCFG` processes this block, it calls `foldCondBranchOnValueKnownInPredecessorImpl`.
It checks if the block is simple enough, which it is.
It then computes `NonLocalUseBlocks`. The `cleanuppad` token `%cp` is used in `cpret` by `cleanupret from %cp`. So `cpret` is added to `NonLocalUseBlocks`.
It then computes `ReachesNonLocalUseBlocks` by searching backward from `cpret`. This adds `cpret`, `cleanup`, `entry`, and `bb1`.
Crucially, `cleanup_2` is NOT added to `ReachesNonLocalUseBlocks` because it is not a predecessor of any block in the set (it loops to itself).
For the `true` value (from `entry`), the destination is `cleanup_2`.
Since `cleanup_2` is not in `ReachesNonLocalUseBlocks`, the optimization proceeds and calls `SplitBlockPredecessors(cleanup, {entry}, ".critedge", DTU)`.
Because `cleanup` is an EH pad, `SplitBlockPredecessors` returns `nullptr`.
Without the fix, the code then dereferences this null pointer (`EdgeBB->setName(...)`), causing a compiler crash.

Fix Weakness Analysis:
The fix adds a null check `if (!EdgeBB) continue;` after calling `SplitBlockPredecessors`. This correctly handles the case where the block cannot be split (e.g., it's an EH pad) and prevents the null pointer dereference. The test case demonstrates the exact scenario that the fix addresses.
