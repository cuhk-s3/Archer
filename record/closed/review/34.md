## Test Strategies

### Strategy 1: Obscured Increasing Loop Bound

- **Target**: An increasing loop with a range check against `limit`, guarded by `start < limit` and `limit < C`.
- **Rationale**: When a loop has a guard like `start < limit` and another guard that refines `limit` (e.g., `limit < 100`), `applyLoopGuards` might rewrite `limit` to `smin(limit, 99)`. The SCEV analysis might fail to prove `start < smin(limit, 99)` even though `start < limit` is explicitly guarded. The fix should allow IRCE to prove this using the original SCEVs.
- **Expected Issue**: IRCE fails to eliminate the range check because `applyLoopGuards` obscures the loop bound, making `isLoopEntryGuardedByCond` fail on the rewritten SCEVs.

### Strategy 2: Obscured Decreasing Loop Bound

- **Target**: A decreasing loop with a range check against `limit`, guarded by `start > limit` and `limit > C`.
- **Rationale**: Similar to the increasing loop, a decreasing loop with a guard `start > limit` and a refinement on `limit` (e.g., `limit > 0`) might have its `limit` rewritten to `smax(limit, 1)`. Proving `start > smax(limit, 1)` might fail, whereas `start > limit` is trivially true from the guard.
- **Expected Issue**: IRCE fails to eliminate the range check for a decreasing loop due to obscured bounds.

### Strategy 3: Guarded SCEV Required for Proof

- **Target**: A loop where the safety of the bound relies on an `llvm.assume` or a condition that requires `applyLoopGuards` to simplify the SCEV before `isLoopEntryGuardedByCond` can prove it.
- **Rationale**: To ensure the fallback to `applyLoopGuards` still works, we need a test where the original SCEVs cannot prove the predicate, but the rewritten SCEVs (via `applyLoopGuards`) can. For example, using an `assume` or a complex dominating condition that `isLoopEntryGuardedByCond` doesn't natively handle but `applyLoopGuards` does.
- **Expected Issue**: IRCE fails to eliminate the range check because the original SCEVs are not sufficient, and the fix incorrectly skips or fails the guarded SCEV check.

### Strategy 4: MinusOne/Limit Computation with Original SCEV

- **Target**: A loop where `BoundSCEV - 1` wraps (e.g., `BoundSCEV` can be `INT_MIN`), but a loop guard prevents the loop from executing if it wraps.
- **Rationale**: The patch changes `MinusOne` and `Limit` to be computed from `BoundSCEV` instead of `BoundLG`. If `BoundSCEV` can wrap when subtracting 1, but `BoundLG` could not (due to guards), computing `BoundSCEV - 1` might create an unexpected SCEV expression. We should test if this causes any issues when the guarded version is later evaluated.
- **Expected Issue**: IRCE might fail or miscompile if the computation of `MinusOne` or `Limit` using the original `BoundSCEV` instead of `BoundLG` causes an overflow or underflow that wasn't present with `BoundLG`.

### Fix Summary
The patch addresses a missed optimization in the Inductive Range Check Elimination (IRCE) pass. Previously, IRCE unconditionally applied loop guards to the `Start` and `Bound` SCEVs before checking if the loop entry is guarded by the condition. However, `applyLoopGuards` can rewrite simple SCEVs into more complex expressions (e.g., using `smin` or `smax`), which can obscure the relationship between the induction variable and the bound. This caused `isLoopEntryGuardedByCond` to fail even when the predicate was trivially implied by a loop guard in the preheader.

The fix introduces a helper function `isLoopEntryGuardedByCond` that first attempts to prove the predicate using the *original* SCEVs. If that fails, it falls back to trying with the *guarded* SCEVs. This ensures that IRCE can still eliminate range checks when the original SCEVs are sufficient, while retaining the ability to use guarded SCEVs when necessary.

### Test Case Analysis
The provided test case demonstrates the "Obscured Increasing Loop Bound" scenario. It includes a guard `limit < 100` in the preheader, which causes `applyLoopGuards` to rewrite `limit` to `smin(limit, 99)`. The loop guard `start < limit` is sufficient to prove the bound, but without the fix, IRCE would fail to prove `start < smin(limit, 99)`. With the fix, IRCE successfully proves the bound using the original SCEVs and eliminates the range check.
