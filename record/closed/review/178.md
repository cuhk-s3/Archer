## Test Strategies

### Strategy 1: Out-of-bounds shuffle mask index

- **Target**: A test case with a load of a vector of elements larger than 1 byte (e.g., `<4 x i32>`), where the shuffle mask only uses the upper elements (e.g., `<2, 3, 2, 3>`).
- **Rationale**: The patch reverts a change that allowed trimming the low end of loads. The reverted code calculated `NewNumElements = Indices->second + 1u` and `MaxIndex = NewNumElements * 2u`. It then checked if any index in the old mask was `>= MaxIndex`. If the old mask contained an index that was valid for the old load but out of bounds for the new load (e.g., an index referencing the second operand, which is poison), the check `Index >= static_cast<int>(MaxIndex)` would fail, and the optimization would be aborted. However, if the index was within `MaxIndex` but still referenced the second operand, it might be valid. Wait, the issue is that the old mask is used directly with the new load. The new load has `NewNumElements`. The shufflevector has two operands, both of type `<NewNumElements x Ty>`. The valid indices are `0` to `2 * NewNumElements - 1`. If the old mask contained an index `>= 2 * NewNumElements`, it would be invalid for the new shufflevector. The reverted code checks `Index >= static_cast<int>(MaxIndex)` where `MaxIndex = NewNumElements * 2u`. This correctly prevents invalid indices. Wait, what if the old mask had an index that was valid for the old load (e.g., referencing the second operand of the old shuffle) but is now out of bounds for the new load? The reverted code checks this and returns false. So what is the bug in the reverted code?
Let's re-read the patch. The patch *reverts* a previous commit. The previous commit added the ability to trim the low end of loads.
The diff shows:
```cpp
-    unsigned const NewNumElements = (Indices->second + 1) - Indices->first;
+    unsigned const NewNumElements = Indices->second + 1u;
```
And:
```cpp
-      unsigned const LowOffset = Indices->first;
-      unsigned const HighOffset = OldNumElements - (Indices->second + 1);
+      unsigned const MaxIndex = NewNumElements * 2u;
```
And:
```cpp
-        NewUses.push_back({Shuffle, {}});
-        std::vector<int> &NewMask = NewUses.back().second;
+        NewUses.push_back({Shuffle, OldMask});
+
+        // Validate mask indices.
         for (int Index : OldMask) {
-          int NewIndex = Index >= static_cast<int>(OldNumElements)
-                             ? Index - LowOffset - HighOffset
-                             : Index - LowOffset;
-          if (NewIndex >= static_cast<int>(NewNumElements * 2u))
+          if (Index >= static_cast<int>(MaxIndex))
             return false;
-          NewMask.push_back(NewIndex);
         }
```
And:
```cpp
-      Type *IndexTy = DL->getIndexType(PtrOp->getType());
-      Value *NewPtr = LowOffset > 0u
-                          ? Builder.CreateInBoundsPtrAdd(
-                                PtrOp, ConstantInt::get(IndexTy, LowOffset))
-                          : PtrOp;
-
       auto *NewLoad = cast<LoadInst>(
-          Builder.CreateAlignedLoad(NewLoadTy, NewPtr, OldLoad->getAlign()));
+          Builder.CreateAlignedLoad(NewLoadTy, PtrOp, OldLoad->getAlign()));
```
The patch *removes* the ability to trim the low end of the load. Why was it reverted?
Ah! The commit message says: "Revert "[VectorCombine] Trim low end of loads used in shufflevector rebroadcasts." Reverts llvm/llvm-project#149093".
Why would trimming the low end be wrong?
If we trim the low end, we adjust the pointer: `Builder.CreateInBoundsPtrAdd(PtrOp, ConstantInt::get(IndexTy, LowOffset))`.
Wait, `LowOffset` is the number of *elements*, not *bytes*!
`CreateInBoundsPtrAdd` adds a byte offset, not an element offset!
Let's check `IRBuilder::CreateInBoundsPtrAdd`.
```cpp
  Value *CreateInBoundsPtrAdd(Value *Ptr, Value *Offset,
                              const Twine &Name = "") {
    return CreateGEP(getInt8Ty(), Ptr, Offset, Name,
                     GEPNoWrapFlags::inBounds());
  }
```
Yes! `CreateInBoundsPtrAdd` treats the offset as a byte offset because it uses `getInt8Ty()`.
But `LowOffset` is the number of elements (`Indices->first`).
So if the element type is not 1 byte (e.g., `i32`, which is 4 bytes), `CreateInBoundsPtrAdd` will add `LowOffset` bytes instead of `LowOffset * 4` bytes!
This will result in an unaligned/incorrect pointer!
This is a classic bug: confusing element offset with byte offset when doing pointer arithmetic with opaque pointers.
- **Expected Issue**: The verifier will fail because the shufflevector mask contains an index that is out of bounds for the new load type.
