## Test Strategies

### Strategy 1: Call reading memory clobbering a load

- **Target**: A test case with a memory definition (e.g., a store), followed by a call that only reads memory (e.g., `readonly` attribute), followed by a use of the memory (e.g., a load).
- **Rationale**: Before the patch, `isModOrRefSet(I)` was used, meaning if a call only reads memory (`Ref`), it would be considered a clobber for a use. After the patch, `isModSet(I)` is used, so a call that only reads memory is no longer a clobber. We can test this by having a call that only reads memory between a memory definition and its use.
- **Expected Issue**: The call instruction should not be considered a clobber for the use, but MemorySSA treats it as a clobber because it reads memory, leading to missed optimization opportunities (e.g., missed load hoisting or GVN).

### Strategy 2: Call writing memory clobbering a load

- **Target**: A test case with a memory definition, followed by a call that writes memory (e.g., no `readonly` attribute, or explicitly `writeonly`), followed by a use of the memory.
- **Rationale**: The patch changes the check to `isModSet(I)`. If a call writes memory (`Mod`), it should still be considered a clobber. We need to ensure that calls that write memory are still correctly identified as clobbers.
- **Expected Issue**: The call instruction should be considered a clobber for the use, and MemorySSA should treat it as a clobber because it writes memory. If the patch incorrectly relaxed this, it might not be treated as a clobber, leading to incorrect optimizations (e.g., hoisting a load above a clobbering call).

### Strategy 3: Call reading memory clobbering a call

- **Target**: A test case with a memory definition, followed by a call that only reads memory, followed by another call that uses the memory.
- **Rationale**: The patch applies to `UseInst` being a `CallBase`. If the use is a call, and the potential clobber is also a call, the same logic applies. We can test if a call that only reads memory is considered a clobber for another call.
- **Expected Issue**: The call instruction should not be considered a clobber for the use, but MemorySSA treats it as a clobber because it reads memory, leading to missed optimization opportunities.

### Strategy 4: Call writing memory clobbering a call

- **Target**: A test case with a memory definition, followed by a call that writes memory, followed by another call that uses the memory.
- **Rationale**: Similar to the previous strategy, but testing that a call that writes memory is still correctly identified as a clobber for another call.
- **Expected Issue**: The call instruction should be considered a clobber for the use, and MemorySSA should treat it as a clobber because it writes memory.

### Strategy 5: Call reading memory clobbering a store

- **Target**: A test case where `UseInst` is a call, and `DefInst` is an instruction that only reads memory (e.g., a load or a readonly call).
- **Rationale**: The patch applies to `UseInst` being a `CallBase`. Wait, the patch is in `instructionClobbersQuery(const MemoryDef *MD, const MemoryLocation &UseLoc, const Instruction *UseInst, AliasAnalysisType &AA)`. The `UseInst` is the instruction we are querying about (the one that might be clobbered). The `DefInst` is the potential clobber. The patch checks `if (auto *CB = dyn_cast_or_null<CallBase>(UseInst)) { ModRefInfo I = AA.getModRefInfo(DefInst, CB); return isModSet(I); }`. This means if the *use* is a call, we check if the *def* modifies the call. Before the patch, it checked if the *def* modifies or references the call. Wait, `AA.getModRefInfo(DefInst, CB)` returns how `DefInst` affects `CB`. If `DefInst` only reads memory, `isModSet(I)` will be false. So if `DefInst` is a read, it won't clobber the call `CB`.
- **Expected Issue**: The call instruction should not be considered a clobber for the use, but MemorySSA treats it as a clobber because it reads memory, leading to missed optimization opportunities.
