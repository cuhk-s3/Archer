## Test Strategies

### Strategy 1: Test maximum with NaN and normal value

- **Target**: `maximum` intrinsic with `NaN` and `fcPosNormal` operands, used by an instruction that does not demand `fcNan`.
- **Rationale**: The fix modifies `KnownLHS` and `KnownRHS` to not contain `fcNan` if `DemandedMask` does not demand `fcNan`. This causes `cannotOrderStrictlyLess` to return true when `a` is `NaN` and `b` is `fcPosNormal`, returning `a` (which is `NaN`). If the user is `fadd nnan`, it will receive `NaN` and produce poison, which is a valid refinement. However, if the user is something else that doesn't demand `fcNan` but doesn't produce poison, it might be a miscompile.
- **Expected Issue**: The optimization might incorrectly replace `maximum(a, b)` with `a` when `a` is `NaN` and `b` is `fcPosNormal`, leading to a miscompile if the user of `maximum` does not demand `fcNan` but still expects a normal value (e.g., `fadd nnan`).

### Strategy 2: Test maximumnum with NaN and normal value

- **Target**: `maximumnum` intrinsic with `NaN` and `fcPosNormal` operands.
- **Rationale**: For `maximumnum`, `PropagateNaN` is false, so `KnownLHS` and `KnownRHS` are not modified. If `a` is `NaN` and `b` is `fcPosNormal`, `cannotOrderStrictlyLess` returns true. However, the fix checks `KnownLHS.isKnownNever(fcNan)` before returning `a`, which prevents the miscompile. This test ensures the check is effective.
- **Expected Issue**: The optimization might incorrectly replace `maximumnum(a, b)` with `a` when `a` is `NaN` and `b` is `fcPosNormal`, leading to a miscompile.

### Strategy 3: Test maximum with equal zero values

- **Target**: `maximum` intrinsic with `fcPosZero` and `fcPosZero` operands.
- **Rationale**: `cannotOrderStrictlyLess` returns true for equal values if they are `fcPosZero`, `fcNegZero`, `fcPosInf`, or `fcNegInf`. This test ensures the logic handles equal values correctly.
- **Expected Issue**: The optimization might incorrectly replace `maximum(a, b)` with `a` when `a` is `fcPosZero` and `b` is `fcPosZero`, which is correct but might expose issues in `cannotOrderStrictlyLess` logic for equal values.

### Strategy 4: Test KnownFPClass propagation for maximum

- **Target**: `maximum` intrinsic used by an instruction that does not demand `fcNan`, where `SimplifyDemandedFPClass` returns false.
- **Rationale**: The fix modifies `KnownLHS` and `KnownRHS` to not contain `fcNan` before computing `Known` for `maximum`. If `SimplifyDemandedFPClass` returns false, the caller receives a `Known` without `fcNan`. This test ensures the caller handles this correctly.
- **Expected Issue**: The optimization might incorrectly compute `Known` for `maximum` without `fcNan` if `DemandedMask` does not demand `fcNan`, leading to incorrect `Known` propagation to the caller.

### Strategy 5: Test hasNoNaNs flag setting for maximum

- **Target**: `maximum` intrinsic with one use that does not demand `fcNan`.
- **Rationale**: The fix sets `hasNoNaNs(true)` on `maximum` if `DemandedMask` does not demand `fcNan`. This is valid if `maximum` has only one use, but this test ensures it doesn't cause issues.
- **Expected Issue**: The optimization might incorrectly set `hasNoNaNs(true)` on `maximum` if `DemandedMask` does not demand `fcNan`, leading to a miscompile if `maximum` actually produces `NaN`.
