## Test Strategies

### Strategy 1: Bitcast between bfloat and half

- **Target**: Bitcast from fp128 to ppc_fp128 or vice versa
- **Rationale**: The patch adds a specific case to fold bitcasts between bfloat and half. It extracts the APInt from the source ConstantFP and creates a new ConstantFP with the destination type's semantics using the same APInt. However, bfloat and half have different exponent and mantissa sizes. A bitcast between them should preserve the bit pattern, but creating a new ConstantFP with the same APInt and different semantics will change the numerical value. The patch does exactly this: `APInt Val = FP->getValueAPF().bitcastToAPInt(); APFloat ResultFP(DestTy->getFltSemantics(), Val); return ConstantFP::get(DestTy->getContext(), ResultFP);`. This is correct for bitcast, as bitcast preserves the bit pattern. Wait, is it? Let's re-evaluate. `APFloat ResultFP(DestTy->getFltSemantics(), Val);` creates an APFloat from an APInt. The constructor `APFloat(const fltSemantics &, const APInt &)` interprets the APInt as the bit representation of the floating-point number according to the given semantics. So it does preserve the bit pattern. Is there any issue? What if the bitcast is between vectors of bfloat and half? The patch explicitly checks `!isa<VectorType>(SrcTy)`. What if the bitcast is between bfloat and half, but they are not the only floating-point types of the same size? No, they are the only ones of size 16. What if the bitcast is between bfloat/half and an integer type? The patch only triggers if `DestTy->isFloatingPointTy()`. What if the bitcast is between bfloat and half, and the value is a NaN? Does `APFloat` preserve the exact NaN payload when constructed from APInt? Yes, `APFloat(Semantics, APInt)` just copies the bits. Wait, what if the bitcast is between `float` and `i32`? The patch is inside `if (ConstantFP *FP = dyn_cast<ConstantFP>(V))`. It checks `DestTy->isFloatingPointTy()`. So it handles `ConstantFP` to `ConstantFP`. The only way to have a bitcast between two different FP types of the same size is if they are `bfloat` and `half`. Wait, are there other FP types of the same size? `double` and `x86_mmx`? No, `x86_mmx` is not an FP type. What about `ppc_fp128` and `fp128`? They are both 128-bit FP types! If `SrcTy` is `fp128` and `DestTy` is `ppc_fp128`, or vice versa, this code will trigger! Let's check if `ppc_fp128` and `fp128` can be bitcasted to each other. Yes, they have the same size (128 bits). If we bitcast `fp128` to `ppc_fp128`, the patch will do: `APInt Val = FP->getValueAPF().bitcastToAPInt(); APFloat ResultFP(DestTy->getFltSemantics(), Val);`. But wait, `ppc_fp128` has a different memory layout and its `bitcastToAPInt()` might not be a simple bitwise representation, or maybe `APFloat(semPPCDoubleDouble, APInt)` is not a simple bitwise copy! Let's check `DoubleAPFloat::DoubleAPFloat(const fltSemantics &S, const APInt &I)`. It creates two `IEEEdouble` APFloats from the two 64-bit halves of the APInt. But the memory layout of `ppc_fp128` depends on endianness! As the comment in the same function says: "PPC_FP128 is really the sum of two consecutive doubles, where the first double is always stored first in memory, regardless of the target endianness. The memory layout of i128, however, depends on the target endianness, and so we can't fold this without target endianness information." If we bitcast `fp128` to `ppc_fp128`, we are bypassing this endianness check!
- **Expected Issue**: The bitcast from bfloat to half or half to bfloat will be incorrectly folded into a ConstantFP with the same underlying APInt value, but interpreted with the destination type's semantics. This changes the numerical value of the constant.

### Strategy 2: Bitcast between fp128 and ppc_fp128

- **Target**: Bitcast from fp128 to ppc_fp128 or vice versa
- **Rationale**: The patch adds a fold for `ConstantFP` to `ConstantFP` bitcasts: `if (!isa<VectorType>(SrcTy) && DestTy->isFloatingPointTy()) { APInt Val = FP->getValueAPF().bitcastToAPInt(); APFloat ResultFP(DestTy->getFltSemantics(), Val); return ConstantFP::get(DestTy->getContext(), ResultFP); }`. This was intended for `half` <-> `bfloat`, but it applies to ANY two floating-point types of the same size. The only other pair of FP types with the same size is `fp128` and `ppc_fp128`. However, `ppc_fp128` is a double-double format, and its bitwise representation depends on the target endianness. The comment just below the patch explicitly states: "PPC_FP128 is really the sum of two consecutive doubles... The memory layout of i128, however, depends on the target endianness, and so we can't fold this without target endianness information." By unconditionally folding `fp128` <-> `ppc_fp128` using `bitcastToAPInt()` and the `APFloat` constructor, the compiler assumes a specific endianness (likely the host's or a fixed one in APFloat), which will be incorrect for targets with the opposite endianness.
- **Expected Issue**: The bitcast will be folded incorrectly because the endianness is not considered when converting between fp128 and ppc_fp128.

### Strategy 3: Bitcast from ppc_fp128 to fp128

- **Target**: Bitcast from ppc_fp128 to fp128
- **Rationale**: Similar to the above, bitcasting from `ppc_fp128` to `fp128` will use `FP->getValueAPF().bitcastToAPInt()`. For `ppc_fp128`, `bitcastToAPInt()` returns a 128-bit APInt. Then it constructs an `fp128` APFloat from this APInt. This conversion implicitly assumes a certain endianness mapping between the two 64-bit doubles in `ppc_fp128` and the 128-bit integer, which might not match the target's endianness. The existing code explicitly prevented folding `ppc_fp128` to `i128` because of endianness: `if (SrcTy->isPPC_FP128Ty()) return nullptr;`. But the new patch is inserted *before* this check, and it handles `DestTy->isFloatingPointTy()`. So a bitcast from `ppc_fp128` to `fp128` will be caught by the new patch and folded, bypassing the `isPPC_FP128Ty()` check and ignoring endianness.
- **Expected Issue**: The bitcast will be folded into a ConstantFP, but the value might be incorrect or it might crash if APFloat doesn't support bitcasting between these types directly without endianness.

### Strategy 4: Endianness-dependent bitcast between fp128 and ppc_fp128

- **Target**: Bitcast between fp128 and ppc_fp128 on different endian targets
- **Rationale**: The patch aims to fold `half` <-> `bfloat` bitcasts but uses a generic check `DestTy->isFloatingPointTy()`. This inadvertently catches `fp128` <-> `ppc_fp128` bitcasts. Since `ppc_fp128`'s memory layout is endian-dependent (the most significant double is always first in memory, regardless of endianness, whereas `fp128` follows normal endianness rules), a simple APInt-based bitcast is semantically incorrect without knowing the target data layout. The patch performs this fold without any DataLayout, leading to miscompilations on either big-endian or little-endian targets (depending on what APFloat assumes).
- **Expected Issue**: The compiler will incorrectly fold the bitcast, leading to wrong numerical values on certain endian targets.
