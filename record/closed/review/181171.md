## Test Strategies

### Strategy 1: Samesign SLT mixed with unsigned

- **Target**: Loop invariant exit condition with `icmp samesign slt`.
- **Rationale**: The `samesign` flag on `slt` might allow it to be treated as `ult` in some contexts (like backedge guard checks) but `slt` in others (like overflow checks). Dropping `samesign` forces consistent use of `slt`.
- **Expected Issue**: Miscompilation or incorrect loop invariant hoisting due to mixed signed/unsigned reasoning.

### Strategy 2: Samesign ULT mixed with signed

- **Target**: Loop invariant exit condition with `icmp samesign ult`.
- **Rationale**: Similar to SLT, `samesign ult` could be treated as `slt` in some checks and `ult` in others, leading to a mismatch in the mathematical model of the loop exit.
- **Expected Issue**: Incorrect loop exit condition evaluation or infinite loop.

### Strategy 3: Decreasing IV with samesign predicate

- **Target**: Loop with a decreasing induction variable (step -1) and a `samesign` exit condition.
- **Rationale**: The patch changes `getSwappedCmpPredicate` to `getSwappedPredicate` for the `MinusOne` step case. Testing a decreasing IV ensures this path is exercised and `samesign` is correctly dropped without causing type/assertion issues.
- **Expected Issue**: Crash or incorrect overflow check for decreasing IVs.

### Strategy 4: Samesign predicate with negative start value

- **Target**: Loop where the IV starts negative, but the exit condition uses `samesign ult` or `samesign slt`.
- **Rationale**: `samesign` implies the operands have the same sign. If the IV starts negative, and we use `samesign ult`, it might be valid for the first iteration but invalid later. Dropping `samesign` ensures we don't incorrectly assume `samesign` holds for all iterations.
- **Expected Issue**: Incorrect loop invariant exit condition evaluation.
