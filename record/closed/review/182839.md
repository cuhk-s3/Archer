## Test Strategies

### Strategy 1: Function with no uses and no FP arguments/return values

- **Target**: Create a function with internal linkage, no uses, and no floating-point arguments or return values (e.g., returns `void` or `i32`).
- **Rationale**: The patch prevents `addNoFPClassAttrsTopDown` from being called on functions with no uses. If a function has no uses, `F.users()` is empty. In `addNoFPClassAttrsTopDown`, if `F.users()` is empty, the loop `for (User *U : F.users())` is skipped. Then it checks `RetNoFPClass != fcNone` (which is true, as it's initialized to `fcAllFlags`). It then adds `RetNoFPClass` to the return attribute. If the function doesn't return an FP type, adding `nofpclass` is illegal and causes a verifier error. The patch avoids this by not processing functions with no uses.
- **Expected Issue**: The pass might crash or incorrectly infer attributes for a function with no uses, or it might add `nofpclass` to a function that doesn't have FP arguments or return values, leading to invalid IR.

### Strategy 2: Function with no uses but with FP arguments/return values

- **Target**: Create a function with internal linkage, no uses, and floating-point arguments or return values.
- **Rationale**: Even if the function has FP arguments or return values, if it has no uses, the top-down inference has no callers to infer from. The initialization `RetNoFPClass = fcAllFlags` and `ArgsNoFPClass = fcAllFlags` means that if there are no users, it will infer `fcAllFlags` (i.e., `nofpclass(all)`) for all arguments and the return value. This might be overly aggressive or incorrect if the function is somehow called indirectly or if the call graph is stale.
- **Expected Issue**: The pass might incorrectly infer `nofpclass` for a function with no uses but with FP arguments/return values, which is technically not illegal but might be unintended or based on stale call graph edges.

### Strategy 3: Function with non-call uses

- **Target**: Create a function with internal linkage, whose address is taken (e.g., stored in a global variable or passed as an argument), but is never called directly.
- **Rationale**: If a function has uses, but none of them are `CallBase` where the function is the called function, the loop in `addNoFPClassAttrsTopDown` will return `false` early (`if (!CB || CB->getCalledFunction() != &F) return false;`). This means it won't infer any attributes. However, the patch only checks `!F.use_empty()`. If a function has only non-call uses, it will still be added to the `Worklist`, but `addNoFPClassAttrsTopDown` will safely return `false`. This is a good edge case to ensure the logic holds.
- **Expected Issue**: The pass might incorrectly infer `nofpclass` for a function whose only uses are not calls (e.g., address taken but not called directly).

### Strategy 4: Stale call graph edge

- **Target**: Create a test case where a function has no uses, to simulate the state after a call is removed.
- **Rationale**: The commit message mentions "This would only happen if the call graph analysis had stale edges." If a function had a caller, but the caller was optimized away or the call was removed, the function might end up with no uses. The patch handles this by checking `!F.use_empty()`. We can simulate this by having a pass pipeline that removes a call but doesn't update the call graph before `FunctionAttrs` runs, or simply by providing a test case with a function with no uses.
- **Expected Issue**: The pass might incorrectly infer `nofpclass` for a function that is called, but the call graph is stale and the call was removed.
