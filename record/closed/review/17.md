## Test Strategies

### Strategy 1: Test exp with known NaN source and demanded NaN

- **Target**: A test case calling `@llvm.exp` with a known NaN input (e.g., a constant NaN or a value that is known to be NaN through other means) where the result is used in a way that demands NaN.
- **Rationale**: The patch moves the `KnownSrc.isKnownAlways(fcNan)` check above the `KnownSrc.isKnownAlways(fcZero | fcNan)` check. If the source is known to be NaN, it now directly returns the operand, dropping the canonicalize/quieting of signaling NaN. We should test if this behaves correctly when the source is known to be NaN and the demanded mask includes NaN.
- **Expected Issue**: The fadd instruction might be incorrectly simplified or retained when it shouldn't be, or the NaN payload might be lost if the fadd is simplified away.

### Strategy 2: Test exp with known NaN source and demanded non-NaN

- **Target**: A test case calling `@llvm.exp` with a known NaN input, but the result is used in a way that does not demand NaN (e.g., an `fcmp ord`).
- **Rationale**: If the source is known to be NaN, but the demanded mask does not include NaN, the earlier check `if ((DemandedMask & fcNan) == fcNone) KnownSrc.knownNot(fcNan);` might have modified `KnownSrc`. We should test this interaction.
- **Expected Issue**: The fadd instruction might be incorrectly simplified or retained when it shouldn't be, or the NaN payload might be lost if the fadd is simplified away.

### Strategy 3: Test exp with known Zero or NaN source

- **Target**: A test case calling `@llvm.exp` with an input that is known to be either zero or NaN (e.g., the result of a `select` between 0.0 and NaN).
- **Rationale**: The patch avoids a spurious diff in an upcoming patch for fadd support by reordering the checks. We should test the `KnownSrc.isKnownAlways(fcZero | fcNan)` case to ensure it still works correctly when the source can be either zero or NaN, and that it correctly creates the `fadd` instruction.
- **Expected Issue**: The fadd instruction might be incorrectly simplified or retained when it shouldn't be, or the NaN payload might be lost if the fadd is simplified away.

### Strategy 4: Test exp with known Zero or NaN source and fast-math flags

- **Target**: A test case calling `@llvm.exp` with fast-math flags (e.g., `nnan`, `ninf`) and an input that is known to be either zero or NaN.
- **Rationale**: The `fadd` created for the `fcZero | fcNan` case uses the fast-math flags from the `exp` call. We should test if these flags are correctly propagated and if they interact correctly with the simplification.
- **Expected Issue**: The fadd instruction might be incorrectly simplified or retained when it shouldn't be, or the NaN payload might be lost if the fadd is simplified away.

### Strategy 5: Test exp with known Inf or NaN source

- **Target**: A test case calling `@llvm.exp` with an input that is known to be either infinity or NaN.
- **Rationale**: The `KnownSrc.isKnownAlways(fcInf | fcNan)` case is also present. We should test this case to ensure it works correctly and doesn't interact negatively with the reordered checks.
- **Expected Issue**: The fadd instruction might be incorrectly simplified or retained when it shouldn't be, or the NaN payload might be lost if the fadd is simplified away.
