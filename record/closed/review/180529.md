## Test Strategies

### Strategy 1: Test minimumnum/maximumnum with -X and X

- **Target**: Create tests with `minimumnum(X, -X)` and `maximumnum(X, -X)` where `X` can be NaN, sNaN, or +/-0.0.
- **Rationale**: The patch adds `minimumnum` and `maximumnum` to the `IsMinMaxOrXNegX` fold, which transforms `min(X, -X)` to `-(fabs X)` and `max(X, -X)` to `fabs X`. However, `minimumnum` and `maximumnum` have specific NaN handling semantics (they return the number if one operand is NaN and the other is a number). If `X` is NaN, `-X` is also NaN. The result of `minimumnum(NaN, -NaN)` should be NaN. `fabs(NaN)` is also NaN, but the sign bit might be cleared. Also, if `X` is a signaling NaN, `minimumnum` might have different behavior than `minnum`. We need to ensure the fold is correct for NaNs and signed zeros.
- **Expected Issue**: The optimization might incorrectly fold `minimumnum(X, -X)` to `-(fabs X)` without preserving the NaN payload or sign bit correctly, or it might incorrectly propagate `poison` if `X` is a signaling NaN.

### Strategy 2: Test constant folding of nested minimumnum/maximumnum

- **Target**: Create tests with `minimumnum(minimumnum(X, C2), C1)` where `C1` and `C2` are various constants (normal, NaN, sNaN, +/-0.0).
- **Rationale**: The patch adds `minimumnum` and `maximumnum` to the `m(m(X, C2), C1) -> m(X, C)` fold. We need to verify that `minimumnum(C1, C2)` is computed correctly when `C1` and `C2` are constants, including NaNs, sNaNs, and +/-0.0. Also, the fast-math flags intersection might be incorrect.
- **Expected Issue**: The optimization might incorrectly fold `minimumnum(minimumnum(X, C2), C1)` to `minimumnum(X, C)` where `C = minimumnum(C1, C2)`. If `C1` and `C2` are NaNs, or if `X` is NaN, the behavior might differ from the original sequence, especially regarding signaling NaNs or NaN payloads.

### Strategy 3: Test negation fold for minimumnum/maximumnum

- **Target**: Create tests with `minimumnum(-X, -Y)` and `maximumnum(-X, -Y)` where `X` and `Y` can be NaNs, sNaNs, or +/-0.0.
- **Rationale**: The patch adds `minimumnum` and `maximumnum` to the `min(-X, -Y) --> -(max(X, Y))` fold. We need to ensure that `minimumnum(-X, -Y)` is strictly equivalent to `-(maximumnum(X, Y))` for all inputs, including NaNs, sNaNs, and +/-0.0.
- **Expected Issue**: The optimization might incorrectly invert `minimumnum(-X, -Y)` to `-(maximumnum(X, Y))`. If `X` or `Y` is a signaling NaN, the negation might change the sNaN to qNaN or vice versa, or the `maximumnum` might handle the sNaN differently than `minimumnum`. Also, the sign of zero might be mishandled.

### Strategy 4: Test fpext fold for minimumnum/maximumnum

- **Target**: Create tests with `minimumnum(fpext X, fpext Y)` where `X` and `Y` can be sNaNs.
- **Rationale**: The patch adds `minimumnum` and `maximumnum` to the `m((fpext X), (fpext Y)) -> fpext (m(X, Y))` fold. We need to verify that this fold is valid for `minimumnum` and `maximumnum`, especially regarding sNaNs, as `fpext` of an sNaN might quiet it, whereas `minimumnum` of an sNaN might have different behavior.
- **Expected Issue**: The optimization might incorrectly fold `minimumnum(fpext X, fpext Y)` to `fpext (minimumnum(X, Y))`. If `X` or `Y` is a signaling NaN, the `fpext` might quiet the sNaN, changing the behavior of the subsequent `minimumnum`.

### Strategy 5: Test fpext with constant fold for minimumnum/maximumnum

- **Target**: Create tests with `minimumnum(fpext X, C)` where `C` is a constant that can be truncated, including NaNs and sNaNs.
- **Rationale**: The patch adds `minimumnum` and `maximumnum` to the `m(fpext X, C) -> fpext m(X, TruncC)` fold. We need to verify that this fold is valid for `minimumnum` and `maximumnum`, especially when `C` is a NaN or sNaN.
- **Expected Issue**: The optimization might incorrectly fold `minimumnum(fpext X, C)` to `fpext (minimumnum(X, TruncC))`. If `C` is a signaling NaN that can be losslessly truncated, the behavior might change.

### Analysis

Bug Triggering Analysis:
The provided test case triggers the bug by passing a signaling NaN (sNaN) to `minimumnum(X, -X)`.
In the original code, `minimumnum(sNaN, -sNaN)` is evaluated. According to the semantics of `minimumnum`, it quiets signaling NaNs. So the result is a quiet NaN (qNaN).
In the optimized code, the expression is folded to `-(fabs X)`. `fabs` and `fneg` do not quiet signaling NaNs; they only manipulate the sign bit. So the result is a signaling NaN with its sign bit flipped.
This causes a semantic difference: the original code returns a quiet NaN, while the optimized code returns a signaling NaN. This can lead to unexpected traps or undefined behavior in subsequent operations that expect a quiet NaN.

Fix Weakness Analysis:
The fix added `minimumnum` and `maximumnum` to the `IsMinMaxOrXNegX` fold, which transforms `min(X, -X)` to `-(fabs X)` and `max(X, -X)` to `fabs X`.
However, the fix failed to account for the fact that `minimumnum` and `maximumnum` quiet signaling NaNs, whereas `fabs` and `fneg` do not.
By unconditionally applying this fold to `minimumnum` and `maximumnum`, the optimization pass incorrectly preserves signaling NaNs, violating the semantics of the intrinsics.
The test case reveals this weakness by explicitly checking the result of `minimumnum(X, -X)` when `X` is a signaling NaN, demonstrating the semantic mismatch.
