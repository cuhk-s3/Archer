### Strategy 1: fsub x, x with nnan ninf

- **Target**: fsub x, x
- **Rationale**: The patch restricts `Self` handling to `FAdd` in `computeKnownFPClass` and `InstCombineSimplifyDemanded`. However, `fsub x, x` is also a common pattern. If `fsub x, x` is not handled as `Self`, it falls back to `fsub(KnownLHS, KnownRHS)`. If `KnownLHS` and `KnownRHS` are the same, `fsub` will do `fadd(KnownLHS, fneg(KnownLHS))`. This might lose precision compared to knowing they are exactly the same value. But wait, `fsub x, x` is exactly `0.0` (if no NaN/Inf). If we don't handle it as `Self`, we might not realize it's exactly `0.0`.
- **Expected Issue**: The compiler might incorrectly assume the result of `fsub x, x` cannot be `-0.0` when the rounding mode is round-toward-negative-infinity, or it might incorrectly assume `fsub x, x` is always `+0.0` and fold it to `+0.0` even when `x` could be `-0.0` or `NaN` or `Inf`. Wait, `fsub x, x` is not folded to `+0.0` here, but `KnownFPClass` might say it's `+0.0`. Actually, `fsub x, x` is not handled as `Self` in `computeKnownFPClass` anymore because `Self` is restricted to `Opc == Instruction::FAdd`. But wait, `InstCombineSimplifyDemanded` handles `fsub x, x` as `Self`? No, `I->getOpcode() == Instruction::FAdd` is checked.

### Strategy 2: fsub with round-down rounding mode

- **Target**: fsub with constrained intrinsics or dynamic rounding mode
- **Rationale**: LLVM's `KnownFPClass` assumes default rounding mode for `fadd` and `fsub`. If the rounding mode is changed dynamically or via constrained intrinsics, the assumptions about `+0.0` and `-0.0` might be violated. However, `KnownFPClass` is typically only used for default floating-point environment.
- **Expected Issue**: The compiler might incorrectly fold `fsub x, -0.0` or similar due to incorrect `fcNegZero` tracking. Wait, `fsub x, -0.0` is `x + 0.0`. If `x` is `-0.0`, `-0.0 + 0.0 = +0.0`. So it cannot be `-0.0`. But what if the rounding mode is round-down? Then `-0.0 + 0.0 = -0.0`. The `fadd` logic assumes default rounding mode (round-to-nearest). If the function has a different rounding mode, `fadd` might be wrong. But LLVM generally assumes default rounding mode unless constrained intrinsics are used.

### Strategy 3: fsub Inf, X with nnan

- **Target**: fsub Inf, X with nnan
- **Rationale**: The patch adds a fold for `fsub` where LHS is known infinity and `nnan` is present. It folds to LHS. If RHS is also infinity, the result is NaN, which is poison under `nnan`, so folding to LHS (Inf) is valid. If RHS is finite, `Inf - finite = Inf`, so folding to LHS is valid. What if LHS is `-Inf` and RHS is `finite`? `-Inf - finite = -Inf`. Fold to LHS is valid. What if LHS is `-Inf` and RHS is `-Inf`? `-Inf - (-Inf) = NaN` -> poison. Fold to LHS is valid. What if LHS is `-Inf` and RHS is `+Inf`? `-Inf - (+Inf) = -Inf`. Fold to LHS is valid. So this fold seems completely correct.
- **Expected Issue**: The compiler might incorrectly fold `fsub {+/-}Inf, X` to `{+/-}Inf` when `X` is also `{+/-}Inf`, resulting in `NaN` instead of `{+/-}Inf`. The patch adds `With nnan: {+/-}Inf - X --> {+/-}Inf` by checking `KnownLHS.isKnownAlways(fcInf | fcNan)` and `KnownRHS.isKnownNever(fcNan)`. But if `X` is `Inf` with the same sign, `Inf - Inf = NaN`. Since `nnan` is present, `NaN` is poison. So folding to `Inf` is technically valid because the original program would produce `NaN` which is UB/poison under `nnan`. Wait, if `X` is `-Inf`, `Inf - (-Inf) = Inf`. If `X` is `Inf`, `Inf - Inf = NaN` -> poison -> can be `Inf`. So this fold is correct under `nnan`.

### Strategy 4: Missed optimization: fsub X, Inf with nnan

- **Target**: fsub X, Inf with nnan
- **Rationale**: The patch explicitly restricts the `X + {+/-}Inf --> {+/-}Inf` fold to `FAdd`. For `FSub`, `X - Inf` should fold to `-Inf`, and `X - (-Inf)` should fold to `+Inf`. By not handling `FSub` here, the compiler misses an optimization opportunity.
- **Expected Issue**: The compiler might incorrectly fold `fsub X, {+/-}Inf` to `{-/+}Inf`. Wait, the patch does NOT add a fold for `X - {+/-}Inf`. It only adds `With nnan: {+/-}Inf - X --> {+/-}Inf`. For `FAdd`, it has `X + {+/-}Inf --> {+/-}Inf`. For `FSub`, `X - {+/-}Inf` should be `- (+/-Inf) = -/+Inf`. But the patch does not add this fold! It only adds `I->getOpcode() == Instruction::FAdd` for the `X + {+/-}Inf` case. This is a missed optimization, not a miscompilation.

### Strategy 5: fsub with denormal flushing

- **Target**: fsub with denormal inputs and DAZ/FTZ modes
- **Rationale**: Denormal flushing behavior might be incorrectly modeled if `fsub` is treated as `fadd(x, fneg(y))`. However, since `fneg` only flips the sign bit and doesn't change the magnitude, a subnormal remains a subnormal. When passed to `fadd`, the denormal mode will correctly flush it if necessary. So this seems correct.
- **Expected Issue**: The compiler might incorrectly compute `KnownFPClass` for `fsub` when `DenormalMode` flushes inputs or outputs to zero. For example, if `fsub` operands are subnormal, they might be flushed to zero. `fsub(subnormal, subnormal)` could become `fsub(0, 0) = 0`. `KnownFPClass::fsub` uses `fadd(KnownLHS, fneg(KnownRHS), Mode)`. `fneg` does not flush denormals. `fadd` handles denormals. Is this correct? Yes, `fneg` is a bitwise operation and doesn't flush. The flushing happens during `fadd`.
