## Test Strategies

### Strategy 1: Invalidate Chain After Clamping VFRange

- **Target**: A loop with a partial reduction pattern where the extend instructions have outside users, or the reduction bin op has invalid users, causing the chain to be invalidated after `getScaledReductions` has already clamped the VFRange.
- **Rationale**: The fix moves the `isValidPartialReduction` check (which calls `getDecisionAndClampRange`) into `getScaledReductions`. This means `Range` is clamped during the discovery phase. However, later in `createPartialReductions`, there are validation checks (e.g., `ExtendUsersValid`, checking if reduction bin ops are only used by other partial reductions). If these later checks fail, the chain is cleared (`Chains.clear()`). But the `Range` has already been clamped based on the assumption that this partial reduction would be used! This could lead to a VFRange that is incorrectly restricted, potentially causing crashes or suboptimal vectorization if the clamped range is empty or excludes valid VFs.
- **Expected Issue**: The compiler will crash or generate incorrect code because the partial reduction chain is invalidated but the VFRange was already clamped based on the invalid chain.

### Strategy 2: Multiple Chains with Conflicting VFRanges

- **Target**: A loop with multiple partial reductions (e.g., two different reduction variables) that have different cost profiles or valid VF ranges, causing conflicting clamps on the `VFRange`.
- **Rationale**: `getScaledReductions` is called for each reduction PHI. Each call can modify the `Range` via `isValidPartialReduction` -> `getDecisionAndClampRange`. If there are multiple reduction PHIs, or multiple chains, they all mutate the same `Range` object. If one chain requires a specific VF range and another chain requires a disjoint VF range, `getDecisionAndClampRange` might clamp the range to be empty, or the order of evaluation might lead to incorrect range clamping.
- **Expected Issue**: The compiler will crash or generate incorrect code because the VFRange is clamped multiple times for different chains, potentially resulting in an empty range or conflicting decisions.

### Strategy 3: Nested Scaled Reductions with Invalid Outer Chain

- **Target**: A loop with a chained partial reduction where the inner reduction is valid but the outer reduction is invalid (e.g., due to mismatched types or invalid cost).
- **Rationale**: `getScaledReductions` is recursive. It first calls itself on the operand (`Op`). If the inner call succeeds, it adds to `Chains` and clamps `Range`. Then it checks the outer reduction. If the outer reduction fails the `isValidPartialReduction` check, it returns `false`. However, the inner call already added a chain to `Chains` and clamped the `Range`. Wait, `Chains` is passed by reference. If the outer fails, does it remove the inner chain? No, `Chains.push_back(Chain)` happens at the end of `getScaledReductions`. But the inner call already pushed its chain! If the outer fails, the inner chain remains in `Chains`, and the `Range` remains clamped. Is this intended? If the outer fails, maybe the inner is still a valid partial reduction on its own. But the `RedPhiR` was updated. Let's check: `if (getScaledReductions(RedPhiR, Op, Chains, CostCtx, Range)) { RedPhiR = Chains.rbegin()->ReductionBinOp; ... }`. If the outer fails later, it just returns `false`, but `Chains` still has the inner chain. This seems correct, but what if the outer failing means the whole thing is invalid? Actually, the issue is if `isValidPartialReduction` clamps the range, but then we return `false` for some other reason, the range is still clamped.
- **Expected Issue**: The compiler will crash or generate incorrect code because the inner chain's validity check clamps the range, but the outer chain might be invalid.

### Strategy 4: Incorrect PhiType in Nested Chains

- **Target**: A loop with a chained partial reduction (e.g., `reduce.add(ext(A) * ext(B)) + ext(C) * ext(D)`) where the intermediate accumulator type might cause issues in cost calculation.
- **Rationale**: In `getScaledReductions`, `PhiType` is inferred from `RedPhiR`: `Type *PhiType = CostCtx.Types.inferScalarType(RedPhiR);`. For a nested chain, `RedPhiR` is updated to `Chains.rbegin()->ReductionBinOp`. The `PhiType` inferred from this intermediate `ReductionBinOp` might be different from the original PHI type, or it might be correct. However, `isValidPartialReduction` uses this `PhiType` to check the cost. If the cost model expects the original PHI type, this might be wrong. Wait, `RedPhiR` is the accumulator. Its type should be the wide type. Let's ensure the cost model handles intermediate accumulator types correctly.
- **Expected Issue**: The compiler will crash or generate incorrect code because `isValidPartialReduction` uses `PhiType` which might not match the actual type of the intermediate reduction in a chain.

### Strategy 5: Empty VFRange after getDecisionAndClampRange

- **Target**: A loop with a partial reduction where the cost is only valid for a subset of VFs, and another constraint (or invalidation) conflicts with this subset.
- **Rationale**: `getDecisionAndClampRange` modifies `Range.End` if the predicate changes. If the predicate is false at `Range.Start`, it returns `false`. If `isValidPartialReduction` returns `false`, `getScaledReductions` returns `false`. But what if it returns `true` but clamps the range, and then a subsequent check fails? As noted in strategy 1, the range is already clamped. If the range becomes empty, `LoopVectorizationPlanner` might assert or fail.
- **Expected Issue**: The compiler will crash or generate incorrect code because `getDecisionAndClampRange` might return false, but the range is still modified.
