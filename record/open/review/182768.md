## Test Strategies

### Strategy 1: Invoke Instruction Insertion Point

- **Target**: Create a test case where `V0` is the result of an `invoke` instruction (which is an intrinsic that can be shuffled, though `invoke` of intrinsics is rare, or `V0` is an `invoke` of a normal function and we are shuffling it, wait, `foldShuffleOfIntrinsics` only works on intrinsics. Can we `invoke` an intrinsic? Yes, some intrinsics can be invoked. Or maybe `V0` is a `phi` node? If `V0` is a `phi` node, `++It` might insert before other `phi` nodes, which is also invalid IR. Let's target `phi` nodes as well.)
- **Rationale**: The `findEarliestInsertionPoint` function assumes that if `V0` is an instruction and `V1` is not, it can safely insert after `V0` by doing `++It`. However, if `V0` is an `invoke` instruction, it is a terminator, and inserting after it in the same basic block is invalid. The insertion point should be at the beginning of the normal destination block of the invoke, or the function should bail out.
- **Expected Issue**: The compiler will crash or generate invalid IR because it attempts to insert an instruction after a terminator instruction (invoke) in the same basic block.

### Strategy 2: PHI Node Insertion Point

- **Target**: Create a test case where `V0` is a `phi` node, `V1` is a constant, and there is at least one more `phi` node after `V0` in the same basic block. The optimization should attempt to create a shuffle of `V0` and `V1`. Wait, `foldShuffleOfIntrinsics` requires `V0` and `V1` to be intrinsics. But `getOrCreateShuffle` is called with the arguments of the intrinsics. So `V0` and `V1` are the *arguments* of the intrinsics. Thus, `V0` can easily be a `phi` node or an `invoke` instruction!
- **Rationale**: If `V0` is a `phi` node and `V1` is not an instruction (e.g., a constant), `findEarliestInsertionPoint` will return the iterator immediately following `V0`. If there are other `phi` nodes after `V0` in the same basic block, the new `shufflevector` instruction will be inserted among the `phi` nodes, which violates the LLVM IR rule that all `phi` nodes must be grouped at the beginning of a basic block.
- **Expected Issue**: The compiler will generate invalid IR by inserting a non-PHI instruction among PHI nodes.

### Strategy 3: Invoke Argument Insertion Point

- **Target**: Create a test case where an intrinsic's argument `V0` is the result of an `invoke` instruction, and the other intrinsic's argument `V1` is a constant. The pass will try to shuffle `V0` and `V1` and insert after `V0`.
- **Rationale**: Since `getOrCreateShuffle` is called on the arguments of the intrinsics being folded, `V0` can be the result of an `invoke` instruction. If `V1` is a constant, `findEarliestInsertionPoint` will try to insert the shuffle immediately after the `invoke` instruction, which is a terminator. This will result in invalid IR or a crash.
- **Expected Issue**: The compiler will crash or generate invalid IR because it attempts to insert an instruction after a terminator instruction (invoke) in the same basic block.

### Strategy 4: Different Blocks Dominance Issue

- **Target**: Create a test case with two independent `shufflevector` instructions that fold identical intrinsics. The intrinsics use the same arguments `V0` and `V1` (which are in different blocks, so the shuffle is inserted at the first `shufflevector`). The second `shufflevector` is in a different basic block not dominated by the first one. The pass will reuse the cached shuffle, causing a dominance violation.
- **Rationale**: When `V0` and `V1` are instructions in different basic blocks, `findEarliestInsertionPoint` falls back to `Builder.GetInsertPoint()`. The builder's insertion point is currently set to the `shufflevector` instruction `I` that is being folded (in `FoldInst`: `Builder.SetInsertPoint(&I);`). This is generally safe because `I` must be dominated by both `V0` and `V1` (since `I` uses the intrinsics which use `V0` and `V1`). However, if the new shuffle is cached and later reused for another fold where the insertion point does not dominate the new use, it could lead to invalid IR. Wait, the cache is global per function. If we cache a shuffle inserted at `I`, and later another instruction `I2` (not dominated by `I`) tries to get the same shuffle, it will reuse the cached shuffle, which does not dominate `I2`.
- **Expected Issue**: The compiler will generate invalid IR because the shuffle is inserted at a point where it does not dominate its uses, or it crashes due to unreachable insertion points.

### Strategy 5: CatchPad/CleanupPad Insertion Point

- **Target**: Create a test case where `V0` is a `phi` node in a basic block that starts with a `catchpad` or `cleanuppad`. `findEarliestInsertionPoint` will insert the shuffle after the `phi` node, which might be before the `catchpad`, violating IR rules.
- **Rationale**: If `V0` is an instruction like `catchpad` or `cleanuppad` (or if `V0` is an argument of an intrinsic that is placed immediately after a pad), inserting after `V0` might be invalid if the pad instruction requires to be the first non-PHI instruction in the block. Actually, `catchpad` and `cleanuppad` must be the first non-PHI instruction in their basic blocks. If `V0` is a `phi` node in a catchpad block, inserting after `V0` might place the shuffle before the `catchpad`, which is invalid.
- **Expected Issue**: The compiler will generate invalid IR by inserting a shuffle instruction before a CatchPad or CleanupPad instruction.
