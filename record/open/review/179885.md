### Strategy 1: Switch Case Constant Replacement

- **Target**: A test case with a `switch` instruction where the case values differ between two otherwise identical regions.
- **Rationale**: The patch replaces constants with arguments by iterating over instruction operands. However, some instructions require their operands to be constants. For example, the case values of a `switch` instruction must be `ConstantInt`s. If the outliner elevates a `switch` case value to an argument and replaces it with an `Argument` (which is not a constant), it will produce invalid IR and crash the compiler.
- **Expected Issue**: The outliner might incorrectly replace a constant inside a ConstantExpr if the ConstantExpr is an operand of an instruction, but the constant we want to replace is nested inside the ConstantExpr. Wait, the patch specifically avoids replacing constants inside ConstantExprs by only iterating over direct operands of instructions. However, if the ConstantExpr itself is the constant we want to replace, it will be replaced. If the constant we want to replace is an operand of the ConstantExpr, it will NOT be replaced. The patch says: "The new code does not use the constant's use list (which the buggy change avoided), and instead assumes that GVN is only numbering operands of instructions, rather than iterating into ConstantExprs to number the entire expression."
Wait, if the constant is used in a ConstantExpr, and the ConstantExpr is used in an instruction, the ConstantExpr is an operand of the instruction. The patch checks `if (auto *CCandidate = dyn_cast<Constant>(U)) { if (CCandidate == CST) U = Arg; }`. If `CST` is the ConstantExpr itself, it will be replaced. If `CST` is a constant used *inside* the ConstantExpr, it will not be replaced.
Is it possible that `CST` is a constant used inside a ConstantExpr, and we *should* replace it? The patch assumes GVN only numbers operands of instructions. If GVN numbers the ConstantExpr as a whole, then `CST` would be the ConstantExpr. If GVN numbers the operands of the ConstantExpr, then `CST` would be the operand, and it wouldn't be replaced.
Let's check if GVN numbers operands of ConstantExprs. IRSimilarityIdentifier uses `IRInstructionData` which only looks at operands of instructions. So `CST` is always a direct operand of an instruction.
Wait, what if the same constant is used as a direct operand in one place, and inside a ConstantExpr in another place? The patch will replace the direct operand, but not the one inside the ConstantExpr. Is this correct? Yes, because the ConstantExpr wasn't numbered/outlined as having that constant as an operand.
What if the instruction has a ConstantExpr operand, and we want to replace the ConstantExpr itself? The patch will replace it because `CCandidate == CST` will be true (ConstantExpr is a Constant).
Wait, what if the ConstantExpr is identical in both regions, but it contains a constant that is different? IRSimilarityIdentifier treats ConstantExprs as a single value. If they are different, they get different GVNs. Then the ConstantExpr itself is elevated to an argument. The patch will replace the ConstantExpr with the argument.
What if the ConstantExpr is the same, but we want to replace a constant inside it? IRSimilarityIdentifier doesn't look inside ConstantExprs, so it would just elevate the whole ConstantExpr.
Is there any case where `U = Arg` is invalid? `U` is a `Use`. `Arg` is an `Argument`. If `U` is an operand of an instruction, replacing it with an `Argument` is valid, UNLESS the instruction requires a Constant!
For example, `switch` instruction requires a `ConstantInt` for its cases. If we replace a `ConstantInt` case value with an `Argument`, it will result in invalid IR!
Let's check if `switch` cases can be outlined and elevated to arguments.
IRSimilarityIdentifier might outline `switch` instructions. If the case values are different between two regions, they might be elevated to arguments. But `switch` cases MUST be constants. If we replace a `switch` case with an `Argument`, the IR verifier will crash.
Another example: `getelementptr` with `inrange` or struct indices. Struct indices MUST be constants. If we elevate a struct index to an argument, and replace it with an `Argument`, the IR verifier will crash.
Another example: `extractvalue` / `insertvalue` indices. They are not operands (they are part of the instruction itself), so they are not `Use`s.
Another example: `shufflevector` mask. It's a constant, but in newer LLVM it's an array of ints, not an operand.
Another example: `call` with `immarg` attribute. The argument MUST be a constant. If we replace it with an `Argument`, the verifier will crash.
Let's propose a test strategy for `switch` cases.

The `trans` tool crashed with `immarg operand has non-immediate parameter`. This confirms my hypothesis that the outliner incorrectly replaces constants that MUST be constants (like `immarg` parameters) with arguments.
I will report this bug.

Bug Triggering Analysis:
The provided test case contains two regions that are structurally identical except for the `isvolatile` parameter of the `llvm.memcpy` intrinsic. In one region, it is `false`, and in the other, it is `true`. The `isvolatile` parameter is marked with the `immarg` attribute, which means it must be a constant immediate value.
The `IROutliner` pass identifies these two regions as similar and decides to outline them into a single function. Because the `isvolatile` parameter differs between the two regions, the outliner elevates this constant to an argument of the newly created outlined function.
When the outliner replaces the constants in the outlined function with the new argument, it replaces the `immarg` parameter of the `llvm.memcpy` call with an `Argument` value. This violates the `immarg` constraint, causing the LLVM IR verifier to crash with the error `immarg operand has non-immediate parameter`.

Fix Weakness Analysis:
The fix in the patch changed how constants are replaced in the outlined function. Instead of using `ValueMapper` (which incorrectly replaced constants inside `ConstantExpr`s), the new code manually iterates over the direct operands of instructions and replaces them if they match the constant being elevated.
However, the fix assumes that *any* direct operand of an instruction can be safely replaced with an `Argument`. It fails to account for instructions or intrinsics that require certain operands to be constants (e.g., `immarg` parameters, `switch` cases, `getelementptr` struct indices). By blindly replacing these required-constant operands with non-constant `Argument`s, the outliner generates invalid IR, leading to compiler crashes during verification. The fix only addressed the `ConstantExpr` issue but introduced or left exposed this broader issue of replacing constants that must remain constants.
