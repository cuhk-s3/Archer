## Test Strategies

### Strategy 1: Reverse-direction non-integer ratio (compression path)

- **Target**: Add/mutate InstSimplify bitcast constant tests with `<5 x i24>` -> `<2 x i60>` and similarly odd-ratio compression shapes.
- **Rationale**: The patch rewrites only the NumDstElt > NumSrcElt path. The NumDstElt < NumSrcElt path still computes `Ratio = NumSrcElt / NumDstElt` with integer division and packs exactly `Ratio` source elements per destination lane. For legal casts like `<5 x i24>` -> `<2 x i60>` (120 bits each side), true ratio is 2.5, so floor division risks dropping/offsetting bits.
- **Expected Issue**: Incorrect folded constants (or malformed construction leading to downstream crash) for legal equal-bitwidth casts where NumDstElt < NumSrcElt but element-count ratio is non-integer.

### Strategy 2: Big-endian non-integer widening validation

- **Target**: Duplicate non-integer widening cases in a BE datalayout test file (e.g., powerpc64) for `<3 x i32> -> <4 x i24>` and `<3 x i64> -> <8 x i24>` and check exact constants.
- **Rationale**: New logic introduces `Rest/RestSize` streaming with distinct BE code paths (`Rest << ...`, `NextVecElem.lshrInPlace(...)`, `ShiftAmt = SrcBitSize - DstBitSize`). BE mistakes are common and existing new tests appear LE-oriented.
- **Expected Issue**: Wrong lane order/values under big-endian for non-integer-ratio widening; no crash, but semantic mismatch.

### Strategy 3: Remainder carry-chain stress

- **Target**: Introduce long-stream equal-bit tests with odd widths (e.g., `<9 x i40>` -> `<15 x i24>`) using recognizable bit patterns to detect shifted boundaries.
- **Rationale**: Correctness hinges on iterative invariant maintenance: drain while `RestSize >= DstBitSize`, otherwise merge next source and update `RestSize = SrcBitSize + RestSize - Ratio*DstBitSize`. Larger vectors with odd widths amplify small arithmetic mistakes.
- **Expected Issue**: Bit misalignment or element count boundary errors after multiple remainder cycles (off-by-one in `RestSize` evolution).

### Strategy 4: Undef across partial chunk boundaries

- **Target**: Mutate existing undef constexpr tests around non-integer ratios (e.g., `<3 x i32>` -> `<4 x i24>`) by placing `undef` in each source position and checking resulting per-lane form.
- **Rationale**: For non-divisible widths, undef source lanes are replaced with zeros (`NextVecElem=0`) while divisible widths append undef lanes. This asymmetry may over-concretize results compared to alternative legal folds or expected poison/undef behavior patterns.
- **Expected Issue**: Potential semantic/canonical inconsistency in undef handling for non-divisible widths (values become concretized where undef-taint may be expected).

### Strategy 5: Metamorphic equivalence with integer pivot

- **Target**: Create paired tests in InstSimplify where both formulations are constants and must fold identically for non-integer ratios.
- **Rationale**: Bitcast is defined as raw bit reinterpretation; `bitcast <m x iA> -> <n x iB>` should match `bitcast <m x iA> -> i(m*A) -> <n x iB>`. If the streaming implementation mishandles remainders/endianness, the two forms can differ.
- **Expected Issue**: Inconsistency between equivalent bitcast formulations (direct vector->vector vs vector->integer->vector), exposing incorrect bitstream assembly in the new path.

### Analysis

Bug Triggering Analysis:
- This IR encodes a metamorphic equivalence: direct bitcast `<5 x i24> -> <2 x i60>` must equal pivoted bitcast `<5 x i24> -> i120 -> <2 x i60>`.
- By LangRef, bitcast is pure bit reinterpretation; both forms are semantically identical.
- InstSimplify (via ConstantFold/FoldBitCast) folds `@t` to `ret i1 false`, proving it believes the two results differ.
- Alive2 verification reports the transform is incorrect with a concrete mismatch example, demonstrating a real miscompile.
- Root cause aligns with patch gap: in `FoldBitCast`, the `NumDstElt < NumSrcElt` path still uses `Ratio = NumSrcElt / NumDstElt` integer truncation and assumes integral grouping, which is invalid for non-integer element-count ratios (here 5/2).

Fix Weakness Analysis:
- The patch fixed non-integer-ratio handling only for `NumDstElt > NumSrcElt` (widening to smaller elements) by adding bit-stream logic.
- It did not symmetrically fix the reverse compression path (`NumDstElt < NumSrcElt`), leaving old floor-ratio packing logic.
- This test directly targets that asymmetry and exposes that legal equal-total-bit casts with non-integer element-count ratio are still folded incorrectly.
- Therefore the fix is incomplete and the provided test is a strong regression candidate for the remaining bug surface.
