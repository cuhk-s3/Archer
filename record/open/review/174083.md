## Test Strategies

### Strategy 1: Context-sensitive equality analysis

- **Target**: Add CxtI to SimplifyQuery and test with dominating conditions
- **Rationale**: The patch creates SimplifyQuery without CxtI, so isKnownNonEqual may miss context-specific inequalities from dominating branches. This could lead to incorrect folding when a != b is guaranteed by control flow.
- **Expected Issue**: Incorrect transformation when a branch before the call ensures a != b but isKnownNonEqual returns false due to missing context.

### Strategy 2: Invoke instruction handling

- **Target**: Extend to InvokeInst and test invoke-specific patterns
- **Rationale**: The patch only handles CallInst but not InvokeInst, missing optimization opportunities and creating inconsistency between call and invoke handling.
- **Expected Issue**: Missing optimization for invoke instructions or inconsistent behavior between call and invoke.

### Strategy 3: Overflow flag semantics

- **Target**: Test sub with nuw/nsw flags and poison propagation
- **Rationale**: Subtraction with overflow flags has different poison semantics. Folding sub nuw/nsw a, b to 0 when a == b might be correct but could miss flag-specific optimizations or affect poison propagation.
- **Expected Issue**: Incorrect poison semantics or missed optimization opportunities for flagged subtractions.

### Strategy 4: Multiple use analysis

- **Target**: Test subtraction used in multiple places (call arg + other uses)
- **Rationale**: The patch modifies only the call argument, leaving the subtraction instruction intact. If the subtraction has other uses, this could create inconsistency or miss broader optimization opportunities.
- **Expected Issue**: Inconsistent IR where subtraction is replaced in call but remains for other uses, or missed full elimination of dead subtraction.

### Strategy 5: Vector type handling

- **Target**: Test vector subtractions as call arguments
- **Rationale**: Vector types may have different behavior in isKnownNonEqual and constant folding. The patch uses ConstantInt::get which works for vectors, but analysis might differ.
- **Expected Issue**: Incorrect folding for vector types or mismatched constant creation.

### Strategy 6: Recursive depth limits

- **Target**: Test complex nested expressions hitting MaxAnalysisRecursionDepth
- **Rationale**: isKnownNonEqual has recursion depth limits. Complex expressions might hit limits and return false, leading to incorrect folding when deeper analysis would reveal inequality.
- **Expected Issue**: Incorrect folding due to analysis depth limits causing false negative from isKnownNonEqual.

### Strategy 7: Call argument side effects

- **Target**: Test calls where argument value affects side effects (e.g., memory operations)
- **Rationale**: Some calls may have side effects dependent on argument values (e.g., memset length, memcpy size). Changing argument from non-zero to 0 could significantly alter behavior.
- **Expected Issue**: Changed program semantics for calls with argument-dependent side effects.

### Strategy 8: Cache invalidation

- **Target**: Test sequences where setArgOperand modifies IR during GVN iteration
- **Rationale**: setArgOperand modifies the IR; GVN's internal caches (value numbering, leader table) might become stale if not properly invalidated.
- **Expected Issue**: Stale cache entries leading to incorrect optimizations or crashes.

### Strategy 9: Integer type mismatches

- **Target**: Test type conversions between subtraction and call argument types
- **Rationale**: The subtraction type might not exactly match the call argument type (e.g., after extensions/truncations). ConstantInt::get with Sub->getType() might create wrong constant type.
- **Expected Issue**: Type mismatch or incorrect constant creation for type-converted arguments.

### Strategy 10: Poison/undef propagation

- **Target**: Test with poison/undef operands in subtraction
- **Rationale**: If a or b is poison/undef, a - b might be poison, but 0 is not. This could change poison propagation semantics in the program.
- **Expected Issue**: Changed poison propagation behavior leading to different program semantics.

### Analysis

## Bug Triggering Analysis

The test case `test_context_sensitive_bug4` exposes a critical bug in the GVN patch. The bug occurs because:

1. **Missing Context in SimplifyQuery**: The patch creates a `SimplifyQuery` without setting `CxtI` (context instruction) to the call instruction being processed. This means `isKnownNonEqual` cannot use context-specific information from dominating conditions.

2. **Incorrect Folding**: In the test, we have a branch `icmp ne i64 %x, %y` that ensures `%x != %y` when we take the `ne_block` path. However, because `isKnownNonEqual` lacks context, it returns `false` (cannot prove inequality), causing the patch to incorrectly fold `%sub = sub i64 %x, %y` to `0`.

3. **Semantic Violation**: When `%x = 0` and `%y = 1`, the subtraction should be `-1` (or `18446744073709551615` in unsigned), but the patch transforms it to `0`. This changes program semantics.

The alive2 verification confirms this bug: "Source is more defined than target" with example values `%x = 0, %y = 1`.

## Fix Weakness Analysis

The weakness in the fix is that it assumes `!isKnownNonEqual(a, b)` implies it's safe to fold `a - b` to `0` in the current context. However:

1. **Context Sensitivity**: The patch doesn't pass `CxtI` to `SimplifyQuery`, so `isKnownNonEqual` can't use dominating condition information. In the test, there's a dominating `icmp ne` that guarantees inequality, but this isn't considered.

2. **Overly Aggressive Transformation**: The transformation replaces subtraction with `0` whenever equality cannot be disproven, but this is incorrect when the subtraction is reachable via paths where the operands are definitely not equal.

3. **Missing Precondition Check**: The patch should verify that the call is actually on an "eq-dominated path" (i.e., only reachable when `a == b`), not just that we can't prove `a != b`.

The fix needs to either:
- Pass proper context (`CxtI`) to `isKnownNonEqual`
- Add stronger preconditions about the execution context
- Or reconsider the transformation's safety conditions
