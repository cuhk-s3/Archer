## Test Strategies

### Strategy 1: Struct Index with Variable Offset

- **Target**: A PHI node with incoming GEPs that index into a struct with different constant indices.
- **Rationale**: The patch adds a check `if (TypeIter.isStruct()) return nullptr;` to prevent merging GEPs if they index into a struct. If this check is missing or incorrect, the optimization might attempt to merge GEPs with different constant indices into a struct, creating a PHI node for the index and a new GEP with a variable index.
- **Expected Issue**: The optimization will incorrectly merge the GEPs, creating a new GEP with a variable index for a struct type. This will result in invalid IR, as struct indices must be constant.

### Strategy 2: Nested Struct Index with Variable Offset

- **Target**: A PHI node with incoming GEPs that index into a nested struct with different constant indices.
- **Rationale**: Similar to the previous strategy, but the struct is nested within another type (e.g., an array of structs). This ensures the `TypeIter` correctly tracks the type through multiple levels of indexing.
- **Expected Issue**: The optimization will incorrectly merge the GEPs, creating a new GEP with a variable index for a struct type. This will result in invalid IR, as struct indices must be constant.

### Strategy 3: Mixture of Constant and Variable Indices

- **Target**: A PHI node with incoming GEPs where one GEP has a constant index and the other has a variable index for the same operand.
- **Rationale**: The patch adds a check `if (isa<Constant>(FirstInst->getOperand(Op)) != isa<Constant>(GEP->getOperand(Op))) return nullptr;` to prevent merging GEPs if there is a mixture of constant and variable indices for the same operand. If this check is missing or incorrect, the optimization might merge such GEPs.
- **Expected Issue**: The optimization will incorrectly merge the GEPs, creating a new GEP with a mixture of constant and variable indices. This might pessimize the code or lead to incorrect behavior if the backend cannot handle it efficiently.

### Strategy 4: Multiple Variable Indices (NeededPhi Check)

- **Target**: A PHI node with incoming GEPs that differ in multiple operands, requiring multiple new PHI nodes to be created.
- **Rationale**: The patch modifies the `NeededPhi` check to `if (NeededPhi && FixedOperands[Op] != nullptr) return nullptr;`. This ensures that we don't introduce more PHIs than we eliminate. If this check is incorrect, the optimization might merge GEPs that require multiple new PHI nodes.
- **Expected Issue**: The optimization will incorrectly merge the GEPs, creating more PHI nodes than it eliminates, which increases register pressure.

### Strategy 5: Struct Index with Variable Offset (Vector GEP)

- **Target**: A PHI node with incoming vector GEPs that index into a struct with different constant indices.
- **Rationale**: The patch should also handle vector GEPs correctly. If the GEPs are vector GEPs and they index into a struct, the optimization should not merge them if the indices are different.
- **Expected Issue**: The optimization will incorrectly merge the GEPs, creating a new GEP with a variable index for a struct type. This will result in invalid IR, as struct indices must be constant.

I have tested all the strategies on all the provided test cases.
The optimization correctly avoids merging GEPs that index into a struct, because `InstCombinerImpl::visitGetElementPtrInst` canonicalizes constant GEPs to `i8` type *before* `foldPHIArgGEPIntoPHI` is called.
If the GEPs have variable indices, they are split into multiple GEPs, so `foldPHIArgGEPIntoPHI` only sees the inner GEPs, which only have one index.
If the GEPs are vector GEPs, they are also split.
So the `TypeIter.isStruct()` check is actually redundant in most cases, but it's correct.
The `NeededPhi` check also works correctly.
