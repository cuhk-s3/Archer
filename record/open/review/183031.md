## Test Strategies

### Strategy 1: SCEV Brute-Force Exit Count with Denormal Addition

- **Target**: A loop with a floating-point phi node and an `fadd` instruction where the value becomes denormal during the loop iterations, and the loop exit condition depends on this value.
- **Rationale**: SCEV uses `EvaluateExpression` to compute brute-force exit counts by simulating loop iterations. If the loop involves a floating-point addition that produces or consumes a denormal value, SCEV might constant-fold it assuming IEEE semantics. However, if the target or environment flushes denormals to zero, the actual runtime loop exit count will differ from SCEV's prediction. The patch prevents this by returning `nullptr` when `AllowNonDeterministic` is false.
- **Expected Issue**: SCEV computes an incorrect brute-force exit count by assuming IEEE semantics for the denormal addition, leading to incorrect loop optimizations (e.g., loop unrolling or vectorization) that do not match the runtime behavior where the denormal is flushed to zero.

### Strategy 2: SCEV Brute-Force Exit Count with Denormal FCMP

- **Target**: A loop where the exit condition is an `fcmp` instruction comparing a floating-point phi node against a denormal constant.
- **Rationale**: If a loop's exit condition is an `fcmp` comparing a loop-variant floating-point value against a denormal constant, SCEV might simulate the loop and constant-fold the `fcmp` assuming IEEE semantics. If the runtime environment flushes denormals to zero, the comparison might yield a different result, causing the loop to exit earlier or later than SCEV predicted.
- **Expected Issue**: SCEV computes an incorrect exit count because it constant-folds the `fcmp` assuming the denormal is preserved, whereas at runtime it is flushed to zero, changing the comparison result and the number of iterations.

### Strategy 3: SCEV Brute-Force Exit Count with Denormal Multiplication

- **Target**: A loop with a floating-point phi node and an `fmul` instruction that produces a denormal value, with the loop exit condition depending on this value.
- **Rationale**: Similar to addition, a loop that multiplies a floating-point value by a constant such that it underflows to a denormal value can be simulated by SCEV. If SCEV constant-folds the `fmul` assuming IEEE semantics, it will predict an exit count that diverges from the runtime behavior if denormals are flushed to zero.
- **Expected Issue**: SCEV incorrectly constant-folds the `fmul` operation assuming IEEE semantics, leading to a wrong exit count prediction compared to the runtime behavior where the denormal is flushed to zero.

### Strategy 4: SCEV Brute-Force Exit Count with Denormal Initial Value

- **Target**: A loop where the initial value of a floating-point phi node is a denormal constant, and the loop exit condition depends on this phi node.
- **Rationale**: If a loop's floating-point phi node starts with a denormal constant, SCEV might simulate the loop starting with that denormal value. If the runtime environment flushes denormals to zero, the initial value will effectively be zero, leading to a different execution path and exit count.
- **Expected Issue**: SCEV computes an incorrect exit count because it constant-folds the `fcmp` assuming the denormal initial value is preserved, while at runtime it is flushed to zero before the first comparison.

### Strategy 5: SCEV Brute-Force Exit Count with Denormal Vector Element

- **Target**: A loop that performs operations on a vector containing denormal floating-point constants, with the loop exit condition depending on the result of these vector operations.
- **Rationale**: The patch also affects vector constants. If a loop uses a vector of floating-point values where one or more elements are denormal, SCEV might simulate the loop and constant-fold operations on this vector. If the runtime flushes denormals to zero, the vector operations will yield different results, leading to an incorrect exit count.
- **Expected Issue**: SCEV computes an incorrect exit count because it constant-folds the cast and subsequent operations assuming the denormal is preserved, whereas at runtime it is flushed to zero.

### Analysis

CRITICAL INSTRUCTION 1: Always prioritize using the most specific tool for the task at hand.
CRITICAL INSTRUCTION 2: Related tools: `report`. I will use `report` to submit the bug.

I have found a bug in the patch!
The patch aims to prevent SCEV from computing brute-force exit counts when the exit count depends on denormal flushing, because denormal flushing is non-deterministic (except for IEEE mode).
To achieve this, the patch adds an `AllowNonDeterministic` flag to `FlushFPConstant` and `ConstantFoldFPInstOperands`.
When SCEV evaluates expressions, it passes `AllowNonDeterministic=false`.
However, for `fcmp` instructions, SCEV calls `ConstantFoldCompareInstOperands`.
The patch modifies `ConstantFoldCompareInstOperands` to flush denormals, but it HARDCODES `AllowNonDeterministic=true`!
```cpp
  if (CmpInst::isFPPredicate(Predicate)) {
    // Flush any denormal constant float input according to denormal handling
    // mode.
    // TODO: Pass through AllowNonDeterministic flag.
    Ops0 = FlushFPConstant(Ops0, I, /*IsOutput=*/false,
                           /*AllowNonDeterministic=*/true);
    if (!Ops0)
      return nullptr;
    Ops1 = FlushFPConstant(Ops1, I, /*IsOutput=*/false,
                           /*AllowNonDeterministic=*/true);
    if (!Ops1)
      return nullptr;
  }
```
Because of this hardcoded `true`, SCEV WILL constant fold `fcmp` instructions involving denormals, even when `AllowNonDeterministic=false`!
This completely defeats the purpose of the patch for `fcmp` instructions, allowing SCEV to compute incorrect brute-force exit counts based on non-deterministic denormal flushing.

Bug Triggering Analysis:
The provided test case `test_fp_fcmp_denormal_preservesign` has a loop that exits based on an `fcmp` between a denormal constant (`0x36A0000000000000`) and `0.0`.
Because the function has the `denormal_fpenv(preservesign)` attribute, the denormal mode is `preservesign`.
When SCEV evaluates the loop, it calls `ConstantFoldCompareInstOperands` for the `fcmp`.
Since `AllowNonDeterministic` is hardcoded to `true`, `FlushFPConstant` flushes the denormal to `0.0`.
The `fcmp one float 0.0, 0.0` evaluates to `false`.
SCEV then computes the loop's max backedge-taken count as `0`.
This allows optimization passes like `instcombine` and `loop-deletion` to completely remove the loop and return `0`, even though the actual runtime behavior might not flush the denormal to zero (since it's non-deterministic), which would cause the loop to run 10 times.

Fix Weakness Analysis:
The weakness in the fix is the failure to propagate the `AllowNonDeterministic` flag through `ConstantFoldCompareInstOperands`. The author even left a `TODO: Pass through AllowNonDeterministic flag.` comment, but forgot to implement it before committing the patch. This leaves a significant gap where `fcmp` instructions are still constant-folded non-deterministically by SCEV, leading to the exact same incorrect loop optimizations the patch was supposed to prevent.
