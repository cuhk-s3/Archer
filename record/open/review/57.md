## Test Strategies

### Strategy 1: Test with `smin` where `Op0` is negative

- **Target**: `Op1 = smin(Op0, V)`
- **Rationale**: The function `isKnownNonNegativeFromMinOrGuardedSub` returns true if `Op1 = smin(Op0, V)`. The caller `computeKnownBitsAddSub` then assumes `Op0 - Op1 >= 0`. However, if `Op0 = -5` and `V = 0`, `Op1 = smin(-5, 0) = -5`. Then `Op0 - Op1 = -5 - (-5) = 0 >= 0`. Wait, if `Op0 = -5` and `V = -10`, `Op1 = smin(-5, -10) = -10`. Then `Op0 - Op1 = -5 - (-10) = 5 >= 0`. In fact, `smin(Op0, V) <= Op0` is always true, so `Op0 - smin(Op0, V) >= 0` is always true. Wait, what if `Op0 - Op1` overflows? The caller checks `NSW`, so overflow is undefined behavior. So `smin` is actually correct. Let's re-evaluate the guarded sub.
- **Expected Issue**: The compiler will incorrectly assume the result of `Op0 - Op1` is non-negative, leading to incorrect optimizations (e.g., replacing a subsequent `icmp slt` with `false`).

### Strategy 2: Test with guarded sub where `Op0` is negative and `A` is positive

- **Rationale**: The function `isKnownNonNegativeFromMinOrGuardedSub` returns true if `Op1 = Op0 < A ? 0 : Op0 - A`. The caller assumes `Op0 - Op1 >= 0`. If `Op0 = -5` and `A = 0`, `Op0 < A` is true, so `Op1 = 0`. Then `Op0 - Op1 = -5 - 0 = -5`, which is NEGATIVE! The assumption `Op0 - Op1 >= 0` is violated. The patch incorrectly assumes `Op0 - Op1 >= 0` for this pattern.
- **Expected Issue**: The compiler will incorrectly assume the result of `Op0 - Op1` is non-negative, leading to incorrect optimizations.

### Strategy 4: Test with guarded sub where `Op0 - A` overflows

- **Target**: `Op1 = select(Op0 < A, 0, Op0 - A)` with `Op0` negative
- **Rationale**: The pattern matches `m_NSWSub(Op0, A)`. If `Op0 - A` overflows, it's poison. But what if the `select` condition shields the poison? If `Op0 < A` is true, `Op1 = 0`, and `Op0 - A` is not evaluated (or its poison is shielded by select). Then `Op0 - Op1 = Op0 - 0 = Op0`. If `Op0` is negative, `Op0 - Op1` is negative. The `NSW` flag on the outer sub `Op0 - Op1` might not be violated if `Op0` is just a small negative number.
- **Expected Issue**: The compiler will incorrectly assume the result of `Op0 - Op1` is non-negative, leading to incorrect optimizations.

- **Target**: `Op0 - select(Op0 < A, 0, Op0 - A)`
- **Rationale**: The Alive2 proofs provided in the commit message likely assumed `Op0 >= 0` or `A >= 0` implicitly, or the proof was for `Op1 - Op0` instead of `Op0 - Op1`. If `Op0 >= A`, result is `Op0 - (Op0 - A) = A`. So the result is `Op0 < A ? Op0 : A`. This is exactly `smin(Op0, A)`. Is `smin(Op0, A)` always non-negative? NO! If `Op0 = -5` and `A = 0`, `smin(-5, 0) = -5`, which is negative! So `Op0 - Op1` is NOT known non-negative. ### Analysis

CRITICAL INSTRUCTION 1: Prioritize specific tools.
CRITICAL INSTRUCTION 2: List related tools.
I have found a bug and confirmed it. I will report it.
Bug Triggering Analysis:
The test case triggers the bug because it matches the pattern `Op1 = select(Op0 < A, 0, Op0 - A)` in `isKnownNonNegativeFromMinOrGuardedSub`. The function incorrectly assumes that `Op0 - Op1` is non-negative. However, if `Op0` is negative and `A` is positive, `Op0 < A` is true, so `Op1 = 0`. Then `Op0 - Op1 = Op0 - 0 = Op0`, which is negative. The compiler incorrectly optimizes `%cmp2 = icmp sge i8 %sub2, 0` to `true`.
Fix Weakness Analysis:
The fix added `isKnownNonNegativeFromMinOrGuardedSub` to deduce that `Op0 - Op1 >= 0`. It assumes that `Op0 - (Op0 < A ? 0 : Op0 - A)` is always non-negative. This is equivalent to assuming `smin(Op0, A) >= 0`, which is only true if both `Op0 >= 0` and `A >= 0`. The fix failed to consider the case where `Op0` is negative. The test case exposes this weakness by providing a negative `Op0` and a positive `A`, and adding a `store` instruction to prevent InstCombine from simplifying the `sub` to `smin` before `computeKnownBits` is called.
