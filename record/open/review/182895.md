### Strategy 1: All Zeros with is_zero_poison=false

- **Target**: Mutate the second argument of `llvm.experimental.cttz.elts` to `undef`.
- **Rationale**: The patch checks `if (Op2->isOne()) return PoisonValue::get(Ty);` and then returns `ConstantInt::get(Ty, FVTy->getNumElements());`. However, if the input vector is all zeros, the loop `for (unsigned I = 0; I < FVTy->getNumElements(); ++I)` will finish without returning. Then it checks `Op2->isOne()`. If `Op2` is false (0), it correctly returns `FVTy->getNumElements()`. Wait, let me re-read the patch.
```cpp
    for (unsigned I = 0; I < FVTy->getNumElements(); ++I) {
      Constant *Elt = Operands[0]->getAggregateElement(I);
      if (!Elt)
        return nullptr;
      if (isa<UndefValue>(Elt) || Elt->isNullValue())
        continue;
      return ConstantInt::get(Ty, I);
    }
    if (Op2->isOne())
      return PoisonValue::get(Ty);
    return ConstantInt::get(Ty, FVTy->getNumElements());
```
If the vector is all zeros, `Elt->isNullValue()` is true, so it continues. After the loop, if `Op2->isOne()` is false, it returns `FVTy->getNumElements()`. This seems correct.
What if the vector contains `undef` and `zero`? It will also return `FVTy->getNumElements()`. But `undef` could be non-zero. If `undef` is treated as zero, it returns `FVTy->getNumElements()`. If `undef` is treated as non-zero, it should return the index of the first `undef`. The current implementation skips `undef` and `zero`. So it treats `undef` as zero. This is correct because we want to find the first non-zero element. If we treat `undef` as zero, we might return a larger index, which is valid since `undef` can be anything. Wait, if we treat `undef` as non-zero, we could return its index. By skipping it, we are choosing `undef = 0`. This is a valid choice.
Wait, what if the vector is `<undef, 1>`? The loop skips `undef` (index 0), and returns index 1. This means `undef` is treated as 0. This is valid.
What if the vector is `<1, undef>`? The loop returns index 0. This is valid.
What if the vector is `<undef, undef>`? The loop finishes. If `is_zero_poison` is true, it returns `poison`. If false, it returns 2. This means both `undef`s are treated as 0. This is valid.

Wait, what if the vector is scalable? `dyn_cast<FixedVectorType>` will return `nullptr`, so it returns `nullptr`. This is correct.

What if `Op2` is not a `ConstantInt`? The patch does `auto *Op2 = cast<ConstantInt>(Operands[1]);`. If `Operands[1]` is not a `ConstantInt`, this will crash!

### Strategy 2: Second Argument is UndefValue

- **Target**: Mutate the second argument of `llvm.experimental.cttz.elts` to `undef` or `poison` in a test case.
- **Rationale**: The patch uses `cast<ConstantInt>(Operands[1])` without checking if `Operands[1]` is actually a `ConstantInt`. In LLVM IR, the second argument of `llvm.experimental.cttz.elts` is required to be a constant flag, but it can be an `UndefValue` or a `PoisonValue` (which are `Constant`s but not `ConstantInt`s). If a user provides `undef` or `poison` as the second argument, `cast<ConstantInt>` will trigger an assertion failure and crash the compiler.
- **Expected Issue**: The compiler will crash due to `cast<ConstantInt>` failing when the second argument is a `ConstantExpr` or `UndefValue`.

### Strategy 3: Second Argument is ConstantExpr

- **Target**: Mutate the second argument of `llvm.experimental.cttz.elts` to a `ConstantExpr` (e.g., `icmp eq (i32 ptrtoint (ptr @global to i32), i32 0)`).
- **Rationale**: Similar to the `UndefValue` case, the second argument could potentially be a `ConstantExpr` that evaluates to `i1`, such as `ptrtoint` or `icmp`. While the LangRef says "constant flag", `ConstantExpr` is a subclass of `Constant`. If `Operands[1]` is a `ConstantExpr`, `cast<ConstantInt>` will crash. (Though `i1` constant expressions are rare, they are possible).
- **Expected Issue**: The compiler will crash due to `cast<ConstantInt>` failing when the second argument is a `ConstantExpr`.
