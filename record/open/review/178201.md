## Test Strategies

### Strategy 1: Unsafe nsw Flag Propagation in Distribution

- **Target**: A test case containing `(A + B) * C` where both the `add` and `mul` have `nsw` flags, and the inputs are chosen such that `A * C` overflows but `(A + B) * C` does not.
- **Rationale**: The patch copies the `nsw` flag from the original `mul` to the newly created distributed `mul` instructions, and from the original `add` to the new `add`. However, algebraic distribution does not preserve no-signed-wrap semantics. For example, in i8, if A=100, B=-99, C=2, `(A+B)*C` is `1*2=2` (no overflow). But `A*C` is `200`, which overflows i8. Copying the `nsw` flag makes the intermediate result `poison`.
- **Expected Issue**: The distributed multiplications (`A * C` and `B * C`) or the new addition will incorrectly inherit the `nsw` flag. For inputs where the intermediate products overflow but the final result does not, this will produce `poison`, leading to a miscompilation.

### Strategy 2: Unsafe nuw Flag Propagation in Distribution

- **Target**: A test case containing `(A + B) * C` where both the `add` and `mul` have `nuw` flags, with a large unsigned constant `C`.
- **Rationale**: Similar to `nsw`, copying the `nuw` (no-unsigned-wrap) flag to the distributed operations is unsafe. The intermediate products `A * C` and `B * C` can wrap unsignedly even if the final result does not. For example, in i8, A=200, B=100, C=2. `A+B` = 300 = 44 (wrap). Wait, if `add` has `nuw`, `A+B` cannot wrap. But `A*C` can wrap even if `(A+B)*C` doesn't? Actually, if `A+B` doesn't wrap and `(A+B)*C` doesn't wrap, `A*C` can still wrap if we consider negative numbers, but `nuw` is unsigned. For unsigned, if `A+B` doesn't wrap, `A < A+B`. So `A*C <= (A+B)*C`. Thus if `(A+B)*C` doesn't wrap, `A*C` cannot wrap. However, the new `add` might wrap if `A*C + B*C` is computed, but mathematically it equals `(A+B)*C`. Wait, what if `C` is negative (large unsigned)? Then `A*C` can wrap. So `nuw` is still unsafe to blindly copy.
- **Expected Issue**: The distributed multiplications or the new addition will incorrectly inherit the `nuw` flag, producing `poison` for inputs that previously computed a valid result without wrapping.

### Strategy 3: Vector Constants with Poison Lanes

- **Target**: A test case using vector types for `(A + B) * C` where `C` or one of the addends is a vector constant containing `poison` lanes.
- **Rationale**: The profitability check uses `isa<Constant>` which returns true for vector constants, including those with `poison` lanes. Furthermore, the factorization heuristic explicitly checks `!isa<UndefValue>(MaxOccVal)`. Since `PoisonValue` is a subclass of `UndefValue`, it is excluded, but testing with explicit `poison` lanes ensures that the heuristic and distribution logic correctly handle modern LLVM poison semantics without propagating poison to safe lanes.
- **Expected Issue**: The optimization might incorrectly treat partially poisoned vectors as fully defined, or the factorization heuristic might behave unexpectedly when encountering `poison` elements instead of `undef`.

### Strategy 4: Factorization Heuristic with Explicit Poison Values

- **Target**: A reassociation test case where one of the repeated factors is an explicit `poison` value, competing with a regular constant and a non-constant instruction.
- **Rationale**: The patch updates the factorization heuristic to prefer non-constant factors over constant factors when occurrence counts are equal, explicitly excluding `UndefValue`. It is important to verify that explicit `PoisonValue` (which is a subclass of `UndefValue`) is also correctly handled and does not cause unintended prioritization during the `MaxOcc` selection.
- **Expected Issue**: The factorization heuristic might incorrectly prioritize a `poison` value over a valid constant, or vice versa, potentially altering the poison semantics of the program or missing optimization opportunities.

### Strategy 5: Interaction with Negative Constants and Factorization

- **Target**: A test case containing `(A + B) * C - (A * C)` where `C` is a negative constant.
- **Rationale**: The distribution creates `A * C + B * C`. If `C` is a negative constant, the subsequent factorization logic (which has specific handling for negative constants, e.g., `LowerNegateToMultiply`) might try to extract `-C` or interact with the newly distributed negative constants. This could expose edge cases in how the pass canonicalizes negative factors.
- **Expected Issue**: The interaction between the new distribution logic and the existing negative factor extraction might lead to infinite loops, missed optimizations, or incorrect signs in the final reassociated expression.

### Bug Triggering Analysis
The provided test case triggers the bug because it contains an expression `(x + y) * 20` where both the `add` and `mul` instructions have the `nsw` (no signed wrap) flag. The `ReassociatePass` distributes this multiplication into `x * 20 + y * 20`. During this distribution, the pass explicitly copies the `nsw` flag from the original `mul` to the newly created `mul` instructions (`x * 20` and `y * 20`), and from the original `add` to the new `add` instruction.

However, algebraic distribution does not preserve no-signed-wrap semantics. For example, if `x = 4` and `y = -1640`, `x + y = -1636`. `(-1636) * 20 = -32720`, which fits in a signed 16-bit integer (range [-32768, 32767]), so the original `mul` does not overflow. But in the distributed form, `y * 20 = -1640 * 20 = -32800`, which overflows the 16-bit signed integer range. Because the new `mul` instruction inherited the `nsw` flag, this overflow results in a `poison` value. The `poison` value then propagates to the final result, causing a miscompilation where a valid computation is turned into `poison`.

### Fix Weakness Analysis
The weakness in the fix is the assumption that flags can be safely copied during algebraic distribution. The patch includes the following code:
```cpp
  if (auto *MulInst = dyn_cast<BinaryOperator>(AC))
    MulInst->copyIRFlags(I); // Copy nsw/nuw from original mul
  if (auto *MulInst = dyn_cast<BinaryOperator>(BC))
    MulInst->copyIRFlags(I);

  Value *NewAdd = Builder.CreateAdd(AC, BC);
  if (auto *AddInst = dyn_cast<BinaryOperator>(NewAdd))
    AddInst->copyIRFlags(Add); // Copy nsw/nuw from original add
```
This logic blindly copies the `nsw` and `nuw` flags without verifying if the intermediate distributed operations are guaranteed not to wrap. As demonstrated by the test case, the intermediate products (`A * C` or `B * C`) can overflow even if the original factored expression `(A + B) * C` does not. By copying the `nsw` flag, the pass introduces undefined behavior (poison) for inputs that were previously valid. The fix should either drop the flags or perform a rigorous analysis to prove that the distributed operations cannot wrap before copying the flags.

Additionally, the patch introduces a secondary bug by using `dbgs() << "DISTRIBUTING: ..."` without wrapping it in `LLVM_DEBUG(...)`. This causes the pass to unconditionally print debug information to standard error during normal compilation, which breaks tools and tests that expect clean output.
