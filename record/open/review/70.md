### Analysis

### Bug Triggering Analysis
The provided test case triggers a bug in the `optimizeFindIVReductions` function when the selected value is a binary expression where the first operand is loop-invariant and the second operand is the induction variable (e.g., `sdiv i64 100, %iv`).

In the original scalar loop, the reduction variable `%rdx` tracks the last evaluated value of `%expr` (`100 / %iv`) when `%cmp` is true.
When vectorized, the compiler attempts to optimize this by tracking the underlying IV instead of the expression result, and then applying the expression to the reduced IV in the middle block.

However, the compiler incorrectly constructs the new backedge select instruction in the vector loop.
In `optimizeFindIVReductions`:
```cpp
    VPValue *NewBackedgeVal = CondSelect;
    // If IV is an expression, create a new select that tracks the underlying
    // IV directly and update the backedge value.
    if (ExpressionIV) {
      auto *SelectR = BackedgeVal->getDefiningRecipe();
      VPBuilder LoopBuilder(SelectR);
      NewBackedgeVal = LoopBuilder.createSelect(Cond, ExpressionIV, PhiR,
                                                SelectR->getDebugLoc());
    }
```
Here, `PhiR` is the original reduction PHI recipe, which tracks the *expression* values, not the *IV* values.
The new select instruction `NewBackedgeVal` selects between `ExpressionIV` (the IV values) and `PhiR` (the expression values).
This mixes IV values and expression values in the same vector!

Later, in the middle block, the compiler performs a reduction (e.g., `smax`) on this mixed vector, assuming it contains only IV values.
```cpp
    VPValue *ReducedIV = MiddleBuilder.createNaryOp(
        VPInstruction::ComputeReductionResult, NewBackedgeVal, Flags, ExitDL);
```
Because the vector contains expression values (which could be larger than the IV values, e.g., `100` vs `1`), the `smax` reduction yields the expression value instead of the correct IV value.
Finally, the compiler applies the expression again to this incorrect reduced value:
```cpp
      if (ExpressionIV)
        ResultVal =
            sinkBinOpToMiddleBlock(IV, ReducedIV, ExpressionIV, MiddleBuilder);
```
This results in evaluating the expression twice (e.g., `100 / 100 = 1`), leading to an incorrect final result.

### Fix Weakness Analysis
The weakness in the fix is that it reuses the original reduction PHI (`PhiR`) as the "false" operand for the new select instruction that is supposed to track the underlying IV.
The original reduction PHI tracks the result of the expression, not the IV itself.
When the condition is false, the new select instruction takes the value from `PhiR`, which is the expression result from previous iterations (or the initial value).
This pollutes the vector with expression results, breaking the assumption that the vector contains only IV values (or the sentinel value) for the subsequent min/max reduction.

To fix this, the compiler should create a new reduction PHI that tracks the IV values (initialized with the sentinel value) and use that new PHI as the "false" operand for the new select instruction. The original reduction PHI should be replaced entirely.
