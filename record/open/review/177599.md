## Test Strategies

### Strategy 1: Interleave Count > 1 with Partial Alias Masking

- **Target**: A loop with a partial alias dependency, compiled with `-force-partial-aliasing-vectorization` and `-force-vector-interleave=2` or `#pragma clang loop interleave_count(2)`.
- **Rationale**: The patch explicitly states that partial alias masking is only valid for IC = 1. The code in `LoopVectorizePass::processLoop` sets `IC = 1` if `CM.maskPartialAliasing()` is true. However, if a user forces an interleave count using `#pragma clang loop interleave_count(2)` or `-force-vector-interleave=2`, the compiler might still try to interleave, or the check might be bypassed. We need to test if forcing IC > 1 is correctly handled (either by ignoring the pragma/flag or by disabling partial alias masking).
- **Expected Issue**: The compiler will crash or generate incorrect code because it attempts to use partial alias masking with an interleave count > 1, which is not supported.

### Strategy 2: Reverse Iteration with Partial Alias Masking

- **Target**: A loop with a negative stride (e.g., `for (int i = n; i > 0; i--)`) and a partial alias dependency, compiled with `-force-partial-aliasing-vectorization`.
- **Rationale**: The patch states that reverse iteration is not supported. The code in `LoopVectorizationCostModel::memoryInstructionCanBeWidened` checks if `Stride < 0` and calls `disablePartialAliasMaskingIfEnabled()`. We need to test a loop with a negative stride and a partial alias dependency to ensure that partial alias masking is correctly disabled.
- **Expected Issue**: The compiler will crash or generate incorrect code because it attempts to use partial alias masking with reverse iteration, which is not supported.

### Strategy 3: Epilogue Vectorization with Partial Alias Masking

- **Target**: A loop with a partial alias dependency, compiled with `-force-partial-aliasing-vectorization` and `-epilogue-vectorization-force-VF=4` or similar flags to force epilogue vectorization.
- **Rationale**: The patch adds a check in `LoopVectorizationPlanner::selectEpilogueVectorizationFactor` to return `FixedScalableVFPair::getNone()` if `CM.maskPartialAliasing()` is true. We need to test if epilogue vectorization is correctly disabled when partial alias masking is enabled.
- **Expected Issue**: The compiler will crash or generate incorrect code because it attempts to use partial alias masking with epilogue vectorization, which is not supported.

### Strategy 4: Fixed Order Recurrences with Partial Alias Masking

- **Target**: A loop with a fixed order recurrence (e.g., `a[i] = b[i-1]`) and a partial alias dependency, compiled with `-force-partial-aliasing-vectorization`.
- **Rationale**: The patch adds a check in `LoopVectorizationCostModel::checkIfPartialAliasMaskingIsEnabled` to disable partial alias masking if `!Legal->getFixedOrderRecurrences().empty()`. We need to test a loop with a fixed order recurrence and a partial alias dependency to ensure that partial alias masking is correctly disabled.
- **Expected Issue**: The compiler will crash or generate incorrect code because it attempts to use partial alias masking with fixed order recurrences, which is not supported.

### Strategy 5: Scalable Vectors with Partial Alias Masking

- **Target**: A loop with a partial alias dependency, compiled for AArch64 with SVE enabled (`-mattr=+sve`) and `-force-partial-aliasing-vectorization`.
- **Rationale**: The patch adds support for partial alias masking, but it's unclear if it fully supports scalable vectors. The `llvm.loop.dependence.war.mask` intrinsic supports scalable vectors (`<vscale x 16 x i1>`), but the interaction with `NumActiveLanes` and the clamped VF might have issues. We need to test a loop with a partial alias dependency compiled for a target with scalable vectors (e.g., AArch64 SVE) with `-force-partial-aliasing-vectorization`.
- **Expected Issue**: The compiler will crash or generate incorrect code because it attempts to use partial alias masking with scalable vectors, which might not be fully supported or tested.
