## Test Strategies

### Strategy 1: Poison Shielding Select with Poisonous Condition

- **Target**: A select instruction where the condition is derived from an instruction that can produce poison (e.g., `add nsw` or `icmp samesign`), and the arms are identical after operand replacement but do not themselves produce poison.
- **Rationale**: The fix attempts to prevent refinement when simplifying a select instruction by checking if the condition can be poison and the arms are not. However, the check `areTheseAllTheValuesThatMayCausePoisonInValue` might not correctly identify all sources of poison, especially when the condition is derived from complex instructions or when the arms themselves have poison-generating flags that are not properly accounted for. If the condition is poison, the select evaluates to poison. If the simplified value is not poison, this is an invalid refinement.
- **Expected Issue**: The select instruction is incorrectly simplified to a value that can be poison when the original select would have shielded the poison.

### Strategy 2: Select with Poison-Generating Flags on Arms

- **Target**: A select instruction where the arms are instructions with poison-generating flags (e.g., `add nsw`), and the condition is an equality comparison that allows operand replacement.
- **Rationale**: The fix checks if the select itself has poison-generating flags (`hasPoisonGeneratingAnnotations()`), but it might not correctly handle cases where the arms of the select have poison-generating flags. If the arms are instructions with poison-generating flags (e.g., `add nsw`), and the select is simplified to one of these arms, the poison-generating flags on the arm might be dropped or incorrectly propagated, leading to an invalid refinement.
- **Expected Issue**: The select instruction is incorrectly simplified, dropping poison-generating flags that should have been preserved or causing a refinement.

### Strategy 3: Select with Vector Condition and Poison Elements

- **Target**: A select instruction with a vector condition where some elements can be poison, and the arms are identical after operand replacement but do not have poison in the corresponding elements.
- **Rationale**: The fix uses `impliesPoison` and `areTheseAllTheValuesThatMayCausePoisonInValue` to check for poison propagation. However, these functions might not correctly handle vector types with partial poison elements. If the condition is a vector with some poison elements, the select evaluates to poison only for those elements. If the simplified value does not have poison in those elements, this is an invalid refinement.
- **Expected Issue**: The select instruction is incorrectly simplified to a value that can be poison when the original select would have shielded the poison.

### Strategy 4: Select Condition Derived from Freeze

- **Target**: A select instruction where the condition is an equality comparison involving a variable derived from a `freeze` instruction, and the arms are identical after operand replacement.
- **Rationale**: The fix checks if the condition can be poison. If the condition is derived from a `freeze` instruction, it cannot be poison. However, the `freeze` instruction guarantees a consistent value, not a specific value. If the simplification assumes a specific value for the frozen variable based on the equality comparison, it might violate the consistency guarantee of `freeze`, leading to an invalid refinement.
- **Expected Issue**: The select instruction is incorrectly simplified, leading to an invalid refinement when the condition is derived from a freeze instruction.

### Strategy 5: Select Condition with Pointer Equality and Different Provenances

- **Target**: A select instruction where the condition is an equality comparison between pointers with different provenances, and the arms involve these pointers.
- **Rationale**: The fix allows operand replacement based on equality comparisons. If the condition is an equality comparison between pointers, the pointers might have different provenances even if their addresses are equal. Replacing one pointer with the other in the arms of the select might strip or alter provenance information, violating the memory model and leading to an invalid refinement.
- **Expected Issue**: The select instruction is incorrectly simplified, leading to an invalid refinement when the condition involves pointers with different provenances.
