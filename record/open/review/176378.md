## Test Strategies

### Strategy 1: Operand not available at loop entry

- **Target**: Create a test case with two adjacent loops eligible for fusion. In the first loop, have a memory access whose pointer is an AddRec. Make one of the operands of this AddRec (e.g., the start or step) depend on a value that is not available at the entry of the second loop. For example, a value defined inside the first loop but outside the AddRec's cycle, or a value that is loop-variant in a way that makes it unavailable at the second loop's entry.
- **Rationale**: The fix adds a check to ensure that all operands of a `SCEVAddRecExpr` are available at the entry of the target loop (`NewL`) before rewriting the expression. If an operand is not available, it bails out. To test this, we need a scenario where an operand of an AddRec in the first loop (`OldL`) is defined inside the first loop but outside the AddRec itself, or is otherwise not available at the entry of the second loop (`NewL`). When LoopFuse attempts to rewrite the access function of the first loop to be in terms of the second loop, it will try to create a new AddRec in `NewL` with this operand. Without the fix, `SE.getAddRecExpr` will assert because the operand is not available at `NewL`'s entry.
- **Expected Issue**: The compiler will crash with an assertion failure in `ScalarEvolution::getAddRecExpr` because the operand is not available at the loop entry of the new loop.

### Strategy 2: Step operand not available at loop entry

- **Target**: Create a test case with two adjacent loops. In the first loop, have a memory access with an AddRec where the step is a value defined inside the first loop (e.g., a loop-invariant value within the first loop but not available before it, or a value loaded within the first loop). Attempt to fuse the loops.
- **Rationale**: Similar to the first strategy, but specifically targeting the step operand of the `SCEVAddRecExpr`. The step operand might be a complex SCEV expression that involves values not available at the entry of the second loop. This ensures the check correctly iterates over all operands, including the step.
- **Expected Issue**: The compiler will crash with an assertion failure in `ScalarEvolution::getAddRecExpr` because the step operand is not available at the loop entry of the new loop.

### Strategy 3: Nested AddRec operand not available at loop entry

- **Target**: Create a test case with nested loops or complex SCEV expressions where an inner AddRec or a complex expression is used as an operand for the AddRec being rewritten. Ensure that some component of this complex operand is not available at the entry of the second loop.
- **Rationale**: The `SCEVAddRecExpr` might have operands that are themselves complex SCEV expressions, such as nested AddRecs or other expressions containing values not available at the entry of the second loop. The fix checks `SE.isAvailableAtLoopEntry(Op, &NewL)` for each direct operand. If an operand is a complex expression, `isAvailableAtLoopEntry` should correctly determine if all its components are available. Testing with nested expressions ensures the availability check is robust.
- **Expected Issue**: The compiler will crash with an assertion failure in `ScalarEvolution::getAddRecExpr` because a nested AddRec operand is not available at the loop entry of the new loop.

### Strategy 4: Start operand not available at loop entry

- **Target**: Create a test case where the start value of an AddRec in the first loop is a value that is not available at the entry of the second loop. For example, a value loaded or computed inside the first loop's body or a block that doesn't dominate the second loop's header.
- **Rationale**: Specifically target the start operand of the `SCEVAddRecExpr`. If the start value is defined inside the first loop (e.g., a value computed in the preheader of the first loop but the preheader is not dominating the second loop, or a value computed inside the first loop), it won't be available at the entry of the second loop.
- **Expected Issue**: The compiler will crash with an assertion failure in `ScalarEvolution::getAddRecExpr` because the start operand is not available at the loop entry of the new loop.

### Strategy 5: Operand defined between loops (if applicable)

- **Target**: Construct a control flow where a value used in the first loop's AddRec is defined in a location that does not strictly dominate the second loop's header, if such a configuration can still reach the fusion checks.
- **Rationale**: While LoopFuse typically works on adjacent loops, the definition of "available at loop entry" relies on dominance. If an operand is defined in a way that it doesn't dominate the entry of the second loop, it shouldn't be used to construct a SCEV in the second loop. This tests the dominance aspect of `isAvailableAtLoopEntry`.
- **Expected Issue**: The compiler will crash with an assertion failure in `ScalarEvolution::getAddRecExpr` because an operand is defined in a block between the two loops that is not executed before the second loop's entry (though this is unlikely for adjacent loops, it's worth considering dominance).

### Bug #3

**Analysis:**
CRITICAL INSTRUCTION 1: I must use the most specific tool for the task.
CRITICAL INSTRUCTION 2: I am using `trans` to test the mutated IR with the `loop-fusion` pass. The previous mutation didn't crash because `%start.phi` was not loop-invariant (it was `[ %start, %bb2.cond ], [ 0, %bb2 ]`). I will change it to `[ %start, %bb2.cond ], [ %start, %bb2 ]` so that it IS loop-invariant. Wait, if it's `[ %start, %bb2 ]`, then `%start` must be defined before `bb2`! But `%start` is defined in `bb2.cond`. So it can't be used in the `bb2` incoming edge of the PHI node.
Ah! If `%start` is defined in `bb2.cond`, it can't be used in `bb2.merge` from `bb2`.
So I can't make `%start.phi` loop-invariant this way.
How can I make an instruction loop-invariant but not dominate the exit block?
What if the instruction is NOT used in a PHI node, but used directly in the AddRec?
If it's used directly in the AddRec, it must dominate the AddRec!
If the AddRec is in `bb2.merge`, the instruction must dominate `bb2.merge`.
If it dominates `bb2.merge`, it dominates the exit block `bb3`.
So it dominates `NewL`'s header `bb5`.
So it IS available at `NewL`'s entry!
Wait, is there ANY way an instruction can be used in an AddRec but NOT dominate the exit block?
What if the AddRec is NOT in `bb2.merge`, but in `bb2.cond`?
If the AddRec is in `bb2.cond`, it's conditionally executed.
Does `LoopFuse` check dependencies for conditionally executed instructions?
Yes, if they access memory.
Let's put the memory access and the AddRec in `bb2.cond`!
