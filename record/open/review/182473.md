## Test Strategies

### Strategy 1: Missing FastMathFlags on Min/Max Reductions

- **Target**: A loop with a floating-point min/max reduction (using fcmp and select) that lacks the 'nnan' and 'nsz' fast math flags.
- **Rationale**: The patch adds support for parallelizing min/max reductions during loop unrolling. It uses `createMinMaxOp` to create the reduction operations. However, `createMinMaxOp` does not take `FastMathFlags` as an argument, nor does it extract them from the `IRBuilder` (unlike some other reduction creation functions). The `IRBuilder` has its fast math flags set before the loop (`Builder.setFastMathFlags(Reductions.begin()->second.getFastMathFlags());`), but `createMinMaxOp` creates an intrinsic or a cmp/select pair. For intrinsics, `IRBuilder::CreateIntrinsic` does not automatically apply the builder's fast math flags to the created call instruction unless explicitly done or if the builder's method handles it (which `CreateIntrinsic` does not do for FMF automatically in all cases, or rather, `createMinMaxOp` might not be applying them correctly if it creates cmp/select). Wait, `IRBuilder::CreateIntrinsic` *does* apply FMF if the intrinsic supports it. Let's check `createMinMaxOp`. It creates `rdx.minmax` intrinsic. If it's an FP min/max intrinsic, it should get the FMF from the builder. But wait, what if it creates a cmp/select pair? `Builder.CreateCmp` and `Builder.CreateSelect` do not take FMF, but `SelectInst` doesn't have FMF, and `FCmpInst` does. `IRBuilder::CreateFCmp` applies FMF. So FMF might be applied. Let's look closer at `createMinMaxOp`. It creates an intrinsic for `FMinNum` etc. Intrinsics like `llvm.minnum` support FMF. The builder applies FMF to calls. So FMF might be fine. Let's re-evaluate. What about the `NeedToFixLCSSA` loop? `for (Instruction *RdxPart : PartialReductions) RdxPart->dropPoisonGeneratingFlags();` This drops poison generating flags (like nsw/nuw) from the partial reductions. But what about the newly created `RdxResult` instructions? They are created with `createMinMaxOp`. If `createMinMaxOp` creates a `cmp` and `select`, they don't have `nsw/nuw`. If it creates an intrinsic, it doesn't have `nsw/nuw`. So that's fine. Wait, what if the reduction is an integer min/max? `createMinMaxOp` creates an intrinsic `llvm.smax` etc. Intrinsics don't have `nsw/nuw`. What if the reduction is a floating point min/max? The patch says `// Skip unsupported reductions. Min/max (integer and FP) are supported.` But wait, `canParallelizeReductionWhenUnrolling` checks `if (RdxDesc.hasExactFPMath()) return std::nullopt;`. So it doesn't parallelize if exact FP math is required. Wait, `RecurrenceDescriptor::isMinMaxRecurrenceKind(RK)` is used. Is there any issue with the identity value? `NewPHI->setIncomingValueForBlock(L->getLoopPreheader(), getRecurrenceIdentity(RdxDesc.getRecurrenceKind(), OrigPHI->getType(), RdxDesc.getFastMathFlags()));` For FP min/max, `getRecurrenceIdentity` requires `nnan` and `nsz` to be set for FP min/max reduction! `assert((!(K == RecurKind::FMin || K == RecurKind::FMax) || (FMF.noNaNs() && FMF.noSignedZeros())) && "nnan, nsz is expected to be set for FP min/max reduction.");` If the original loop has an FP min/max reduction but does NOT have `nnan` and `nsz` fast math flags, `getRecurrenceIdentity` will assert and crash! Let's check `canParallelizeReductionWhenUnrolling`. It does not check if `nnan` and `nsz` are set for FP min/max reductions. It just allows them. Then during unrolling, it calls `getRecurrenceIdentity`, which asserts if `nnan` and `nsz` are missing for `FMin`/`FMax`. Wait, `FMin`/`FMax` are `RecurKind::FMin` and `RecurKind::FMax`. Are these used for `llvm.minnum`? No, `FMinNum` is used. Let's check `RecurKind`. `FMin`/`FMax` are for `cmp` + `select` patterns that don't match `minnum`/`minimum` semantics exactly, or maybe they do. If `RK` is `FMin` or `FMax`, `getRecurrenceIdentity` asserts if `FMF.noNaNs()` and `FMF.noSignedZeros()` are not both true. This means if we have a loop with a custom FP min/max reduction (e.g., `cmp` + `select`) without `nnan` and `nsz`, unrolling it with parallel reductions will cause a compiler crash!
- **Expected Issue**: The unrolled loop will compute an incorrect reduction result because the fast math flags from the original reduction are not propagated to the newly created min/max operations.

### Reproducer

**LLVM IR:**
```llvm
define float @test_fmin_reduction(ptr %a) {
entry:
  br label %loop

loop:                                             ; preds = %loop, %entry
  %iv = phi i64 [ 0, %entry ], [ %iv.next, %loop ]
  %min = phi float [ 0x7FF0000000000000, %entry ], [ %rdx.next, %loop ]
  %gep.a = getelementptr inbounds float, ptr %a, i64 %iv
  %0 = load float, ptr %gep.a, align 4
  %cmp = fcmp reassoc nnan nsz olt float %min, %0
  %rdx.next = select reassoc i1 %cmp, float %min, float %0
  %iv.next = add nuw nsw i64 %iv, 1
  %ec = icmp eq i64 %iv.next, 1024
  br i1 %ec, label %exit, label %loop, !llvm.loop !0

exit:                                             ; preds = %loop
  %res = phi float [ %rdx.next, %loop ]
  ret float %res
}

!0 = distinct !{!0, !1}
!1 = !{!"llvm.loop.unroll.count", i32 2}
```

### Analysis

Bug Triggering Analysis:
The test case contains a floating-point min reduction loop. The `fcmp` instruction has `reassoc nnan nsz` fast math flags, but the `select` instruction only has `reassoc`.
When `LoopUnroll` attempts to parallelize this reduction, it calls `canParallelizeReductionWhenUnrolling`, which uses `RecurrenceDescriptor::isReductionPHI` to identify the reduction. `isReductionPHI` successfully identifies this as a min reduction because the `fcmp` has the required `nnan` and `nsz` flags (which are needed to match `m_OrdFMax` or similar patterns).
However, when `LoopUnroll` creates the new parallel reduction phis, it needs to initialize them with the identity value for the reduction. It calls `getRecurrenceIdentity(RdxDesc.getRecurrenceKind(), OrigPHI->getType(), RdxDesc.getFastMathFlags())`.
The `FastMathFlags` for the reduction descriptor (`RdxDesc`) are collected from the reduction instructions. Since the `select` instruction lacks `nnan` and `nsz`, the combined `FastMathFlags` for the reduction descriptor will NOT have `nnan` and `nsz` set.
Inside `getRecurrenceIdentity`, there is an assertion:
`assert((!(K == RecurKind::FMin || K == RecurKind::FMax) || (FMF.noNaNs() && FMF.noSignedZeros())) && "nnan, nsz is expected to be set for FP min/max reduction.");`
Because `FMF.noNaNs()` and `FMF.noSignedZeros()` are false (due to the `select` lacking these flags), this assertion will fail, causing the compiler to crash.

Fix Weakness Analysis:
The patch added support for parallelizing min/max reductions by allowing `RecurrenceDescriptor::isMinMaxRecurrenceKind(RK)` in `canParallelizeReductionWhenUnrolling`. However, it failed to account for the strict precondition in `getRecurrenceIdentity` that requires `nnan` and `nsz` flags to be present in the `FastMathFlags` of the reduction descriptor for `FMin` and `FMax` recurrence kinds.
While `isReductionPHI` might require `nnan` and `nsz` on the `fcmp` to recognize the pattern, it does not enforce that these flags are present on ALL instructions in the reduction chain (e.g., the `select`). As a result, the combined `FastMathFlags` for the reduction descriptor might lack `nnan` and `nsz`, leading to a crash when `getRecurrenceIdentity` is called during unrolling. The fix should have either ensured that `nnan` and `nsz` are present in the reduction descriptor's `FastMathFlags` before allowing parallelization, or handled the identity value generation without asserting.
