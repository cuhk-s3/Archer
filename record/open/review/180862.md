## Test Strategies

### Strategy 1: Poison-generating instruction used for constant range fact

- **Target**: A test case with a comparison, where one of the operands is an instruction with poison-generating flags (like `add nuw` or `shl nuw`) located *after* another comparison in the same block, or in a block that doesn't strictly dominate the check.
- **Rationale**: The patch extracts constant range facts from operands of comparisons. If an operand is an instruction with poison-generating flags (e.g., `add nuw`), `computeConstantRange` might use these flags to compute a tighter range. However, if the comparison itself is not guaranteed to execute (e.g., it's in a different basic block or after a potential exit), or if the fact is applied to a check that dominates the poison-generating instruction, it could lead to incorrect simplifications based on unobserved poison. The patch adds facts at the `DTN` of the instruction's parent block, which means the fact is assumed to hold for all instructions dominated by that block. But if the instruction is not guaranteed to execute within that block (e.g., it's after a call that might not return), or if the fact is used to simplify a check that comes *before* the instruction in the same block, it's invalid. Wait, the patch adds the fact at `DT.getNode(OpI->getParent())`. This means the fact is available for any check dominated by `OpI->getParent()`. If `OpI` is in the same block as a check, but *after* the check, the fact will still be available because `stable_sort` sorts by `NumIn` (which is the same for the whole block) and then by relative order. Wait, `stable_sort` sorts by relative order in the basic block! So facts from `OpI` will come *after* checks that precede `OpI` in the block? Let's check `stable_sort` logic. `stable_sort` puts facts before checks if they have the same `NumIn`? No, it uses `InstA->comesBefore(InstB)`. But `addConstantRangeFact` creates a `ConditionFact` with `Inst` set to `nullptr`? No, `getConditionFact` doesn't set `Inst`. Wait, `getConditionFact` sets `Ty = EntryTy::ConditionFact`. In `stable_sort`, if both are `ConditionFact`, it sorts by `HasNoConstOp`. If one is `ConditionFact` and the other is not, `ConditionFact` comes first! So `ConditionFact`s are processed *before* `InstCheck`s and `UseCheck`s in the same block, regardless of their actual position in the block! This means a fact derived from an instruction at the end of the block will be used to simplify a check at the beginning of the block. If the instruction at the end of the block produces poison (e.g., `add nuw`), and the check at the beginning of the block is simplified based on this, it's a bug if the block can exit before reaching the instruction, or if the instruction's poison is not guaranteed to be observed.
- **Expected Issue**: The pass might incorrectly simplify a comparison that executes before the poison-generating instruction, leading to a miscompilation where the comparison result is based on a poison value that hasn't been observed yet.

### Strategy 2: Fact from instruction not guaranteed to execute

- **Target**: A test case with a check, followed by a throwing/non-returning call, followed by a comparison that uses an instruction with poison-generating flags.
- **Rationale**: The patch extracts constant range facts from operands of comparisons. It iterates over `Cmp->operands()` and their operands, and adds facts at `DT.getNode(OpI->getParent())`. If `OpI` is in a basic block, the fact is added at that block's `DTN`. Because `ConditionFact`s are sorted before checks in the same block, the fact will be applied to all checks in the block. However, `OpI` might not be guaranteed to execute. For example, if there is a function call that might throw or not return before `OpI` in the same block, `OpI` might never execute. If `OpI` has poison-generating flags, `computeConstantRange` might assume those flags hold. But if `OpI` never executes, those flags are irrelevant, and using them to simplify a check before the throwing call is invalid.
- **Expected Issue**: The pass might incorrectly simplify a comparison based on a fact derived from an instruction that is never executed, leading to a miscompilation.

### Strategy 3: Fact from conditionally executed instruction in the same block

- **Target**: A test case where an instruction with `nuw`/`nsw` is used in a comparison, and the fact derived from it is used to simplify a check that executes *before* the instruction, or on a path where the instruction's poison is not observed.
- **Rationale**: Wait, `OpI` is an operand of `Cmp`. If `Cmp` is in block `B`, `OpI` must dominate `Cmp`. So `OpI` is either in `B` or in a dominator of `B`. The fact is added at `DT.getNode(OpI->getParent())`. This means the fact is assumed to hold for all blocks dominated by `OpI->getParent()`. This is generally correct *if* the fact is an inherent property of `OpI`'s result. `computeConstantRange` computes the range of `OpI`'s result. If `OpI` is `add nuw x, y`, the result is in a certain range *if* it's not poison. If it is poison, the result is poison, and any use of it is poison. If `Cmp` uses `OpI`, and `Cmp` is reached, then `OpI` was executed. But wait! What if `OpI` is used by `Cmp`, but we use the fact to simplify *another* check that uses `x`? The fact `x + y >= UMin` is derived from `add nuw x, y`. This fact is added at `OpI->getParent()`. It will be used to simplify any check dominated by `OpI->getParent()`. But what if `OpI` is poison? If `OpI` is poison, `x + y >= UMin` might be false. Is it valid to assume `x + y >= UMin` for other checks just because `OpI` exists? No! If `OpI` is poison, it doesn't trigger UB unless it's used in a UB-triggering way. `Cmp` using poison just produces poison, which might branch on poison (UB), but what if `Cmp` is part of a `select` or `phi` and never branched on? Or what if the other check is executed *before* `OpI`? If the other check is executed before `OpI`, and we simplify it based on `OpI`'s `nuw` flag, we are assuming `OpI` is not poison. But `OpI` hasn't even executed yet! This is a classic poison propagation bug.
- **Expected Issue**: The pass might incorrectly simplify a comparison based on a fact derived from an instruction in a different function or an unreachable block, though less likely. More importantly, it might extract facts from instructions that are conditionally executed but add them at the block level.

### Strategy 4: Poison-generating instruction used in a non-branching context

- **Target**: A test case where `OpI` with `nuw`/`nsw` is used in a `icmp`, which is used in a `select` (so no UB if poison). Another `icmp` in the same block or a dominated block is simplified based on the fact from `OpI`.
- **Rationale**: If `OpI` is `add nuw x, 1`, and it's used in `Cmp`, and `Cmp` is used in a `select`, then `OpI` being poison just makes the `select` result poison. It doesn't trigger UB. But the patch adds the fact `OpI >= UMin` at `OpI->getParent()`. This fact is then used to simplify other checks in `OpI->getParent()` or dominated blocks. If another check `x < 10` is simplified based on this fact, it might be miscompiled because `OpI` being poison doesn't guarantee `x` satisfies the `nuw` condition. The fact is only valid if `OpI` is guaranteed to not be poison, or if the simplified check is only used when `OpI` is not poison. But the patch adds the fact unconditionally for all dominated checks!
- **Expected Issue**: The pass might incorrectly simplify a comparison based on a fact derived from an instruction that is not guaranteed to execute, because the fact is added at the instruction's parent block, which might not dominate the check, or the check might be before the instruction.

### Strategy 5: Fact from instruction in a dominating block used in a dominated block

- **Target**: A test case with `OpI` in block A, used in a `select` in block A. Block B is dominated by A and contains a check that can be simplified if `OpI`'s `nuw`/`nsw` flags are true.
- **Rationale**: Similar to the above, if `OpI` is in block A, and it's used in a `icmp` in block A, and the `icmp` is used in a `select`. Block B is dominated by A. A check in block B is simplified using the fact from `OpI`. If `OpI` is poison, the program might still be well-defined (e.g., the `select` result is unused or doesn't cause UB). But the check in block B was simplified assuming `OpI` is not poison. This leads to a miscompilation.
- **Expected Issue**: The pass might incorrectly simplify a comparison based on a fact derived from an instruction that is in a different block, where the instruction's poison is not observed.
