## Test Strategies

### Strategy 1: Poison-only suffix is not recognized

- **Target**: Mutate an InstCombine store test to: `store <4 x i32> <i32 7, i32 poison, i32 poison, i32 poison>, ptr %p` and FileCheck for narrowing to `store i32 7, ptr %p` (or `<1 x i32>` canonical form).
- **Rationale**: `getDefinedPrefix()` checks only `isa<UndefValue>(Elt)` and never treats `PoisonValue` as suffix, despite the patch contract saying undef/poison.
- **Expected Issue**: The new fold does not trigger at all for poison suffix vectors, so the intended narrowing is missed.

### Strategy 2: Mixed undef+poison tail mismatch

- **Target**: Add case like `store <4 x i32> <i32 1, i32 undef, i32 poison, i32 poison>, ptr %p` and check whether InstCombine narrows (it likely won’t).
- **Rationale**: Once `SawUndef` is set, any later non-undef element causes bailout. A poison after undef is treated as a defined element, so the transform rejects a semantically intended case.
- **Expected Issue**: No narrowing occurs for valid prefix + mixed undef/poison tail, contrary to the documented transform intent.

### Strategy 3: End-to-end memcpyopt regression with poison tail

- **Target**: Create/modify a PhaseOrdering test (`instcombine,memcpyopt`) from the motivating pattern but with `<..., poison, ...>` tail and check that no oversized memset is formed.
- **Rationale**: The motivating fix relies on narrowing before memcpyopt combines stores. If SROA/canonicalization emits poison lanes, the new transform misses them and memcpyopt may still form a larger memset that initializes bytes that should remain uninitialized/poison.
- **Expected Issue**: Pipeline still regresses to over-initialization (e.g., memset of full prefix range) for poison-tailed vectors because InstCombine fails to narrow; this recreates the motivating class of bug when poison is used instead of undef.

### Strategy 4: Insertelement-built prefix not handled

- **Target**: Mutate an existing InstCombine vector-store test to use insertelement-built vectors with undefined tail and verify no narrowing happens (gap).
- **Rationale**: The new logic requires `Val` to be `Constant`. Patterns like `%v = insertelement <4 x float> poison, float %x, i32 0; store <4 x float> %v` are explicitly mentioned as TODO and remain uncovered.
- **Expected Issue**: Missed fold for non-constant vector values with equivalent semantics, leaving the same regression path open for non-literal construction patterns.

### Strategy 5: Width-changing metadata carryover

- **Target**: Add a narrowing candidate store decorated with metadata (especially `!tbaa_struct`, `!alias.scope`, `!noalias`, `!nontemporal`) and run `opt -passes=instcombine -verify`. Check for malformed IR/metadata propagation anomalies.
- **Rationale**: `combineStoreToNewValue()` was written for type-only changes (same width) and aggressively preserves store metadata. The new fold changes byte width, but still reuses this helper unchanged.
- **Expected Issue**: Potential metadata inconsistency after width-changing store rewrite (stale metadata copied from wider store). In worst case this can trip verifier or mislead later analyses.
