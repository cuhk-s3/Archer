## Test Strategies

### Strategy 1: Vector and poison-lane exclusion stress

- **Target**: Mutate scalar tests to `<2 x i8>` / `<4 x i32>` forms, including splat constants, non-splat constants, and poison/undef elements in vector constants.
- **Rationale**: Although the implementation gates on `I.getType()->isIntegerTy()`, the underlying `m_APInt`/`m_Not` matchers have vector-splat behavior. A future or mistaken relaxation to `isIntOrIntVectorTy()` would be risky, especially because the patch summary says vector splats were intentionally excluded after Alive2 timeouts.
- **Expected Issue**: If the fold is accidentally applied to vector constants with poison/undef lanes or non-splat masks, it may incorrectly turn lane-local masked merges into a single vector `and`, changing poison/undef behavior. If correctly restricted, these should remain unfused or only scalar/safe splat cases should fold.

### Strategy 2: Non-power-of-two and degenerate-width masks

- **Target**: Mutate mask-1 and 0x55 tests to `i1`, `i3`, `i5`, `i33`, `i77`, with masks such as `0`, `-1`, `1`, alternating bit patterns, and sign-looking constants.
- **Rationale**: The transform relies on exact APInt complement equality `*NotMask == ~*Mask`. Non-byte and one-bit widths are common sources of mistakes in mask folds, especially for all-ones/all-zero complements and textual negative constants.
- **Expected Issue**: The fold should still be correct and not assert or mishandle APInt complements for unusual bit widths. Bugs would appear as missed folds, wrong constants, or malformed IR for types like `i1`, `i3`, `i5`, `i33`, `i77`.

### Strategy 3: `or disjoint` poison-boundary near misses

- **Target**: Mutate positive `or disjoint` cases so `OldBits` uses `~M` with one bit flipped, `X & M` uses a different mask, or `Y & M` uses a superset/subset mask, while keeping the IR superficially similar.
- **Rationale**: `or disjoint` is poison if both operands share a set bit. The intended pattern makes `(X & ~M)` and `(X & M) & ...` disjoint by construction, but one-bit mask differences can break that guarantee. Negative tests should ensure exact complement and exact reused mask checks are necessary.
- **Expected Issue**: If `or disjoint` poison semantics are not truly redundant for a mutated pattern, replacing it by plain `and` could remove or introduce poison. Correct behavior: fold only when masks guarantee disjointness for all defined scalar inputs; near-miss masks must not fold.

### Strategy 4: Multi-use sharing and reuse of `NotYAndM`

- **Target**: Mutate tests so `%x.mask`, `%y.mask`, `%not.ym`, `%new.masked`, and `%old.bits` each have additional uses (calls, returns through an aggregate-like chain), and check only the root `or` is replaced by `and %x, %not.ym`.
- **Rationale**: The patch explicitly says it reuses the existing complement to avoid duplicating operands. Multi-use values are a common InstCombine edge case: the root `or` can be replaced, but shared intermediate values must remain for their other uses and not be cloned into a new expression tree.
- **Expected Issue**: A bad implementation might duplicate or rebuild `~(Y & M)` and increase instruction count or change undef/poison sharing; a too-narrow implementation might miss safe folds when only the root has one use. Correct behavior is to reuse the existing `not` value and avoid duplicating the `Y & M` computation.

### Strategy 5: Deep commutation and specific-X identity stress

- **Target**: Mutate positive cases by commuting outer `or`, old-bits `and`, new-bits outer `and`, and inner `X & M`. Add negative cases where the second `X` is `freeze %x`, `or %x, 0`, a separate argument `%x2`, or a recomputed equivalent value.
- **Rationale**: The matcher uses `m_c_And` at both the old-bits and new-bits levels and then `m_Specific(X)` to require pointer identity. This should handle operand order but not semantic equivalence. Tests should stress the exact identity requirement and catch accidental use of a too-permissive matcher.
- **Expected Issue**: If the matcher captures the wrong `X` when operands are commuted at multiple levels, it could either miss valid folds or fold invalid expressions where the two `X` occurrences are not identical. Correct behavior: all commuted variants with identical `X` fold; variants with distinct but equivalent-looking values do not.
