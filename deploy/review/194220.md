## Test Strategies

### Strategy 1: Bit-C boundary violation for K

- **Target**: Test `lshr (add (or (shl X, C), K), C)` with `K = 1<<C` and with K containing higher bits; ensure no fold.
- **Rationale**: Transform is valid only if K >> C == 0. Any set bit at position C breaks the identity.
- **Expected Issue**: InstCombine may fold when K has bit C set (or above), causing wrong value.

### Strategy 2: Negative K on non-power-of-two widths

- **Target**: Use i9/i33 forms with small C and K=-1 or -3; check fold does not happen.
- **Rationale**: For i9/i17/i33, constants like -1/-3 have high bits set; fold must not fire.
- **Expected Issue**: Negative APInt constants on odd widths might be misclassified as low-bit-only, leading to miscompile.

### Strategy 3: Positive fold on odd widths with carry-heavy Y+K

- **Target**: Create i33/i65 positive cases with valid low-bit K (e.g. (1<<C)-1) and Y values causing wrap/carry; verify folded pattern.
- **Rationale**: The rewrite uses `(Y+K)>>C` and `getLowBitsSet(BitWidth, BitWidth-C)`. Odd widths stress APInt paths.
- **Expected Issue**: Wrong mask or carry interaction in folded form on unusual widths can change semantics.

### Strategy 4: Shift amount boundaries

- **Target**: Add C=0 and C=BitWidth variants; ensure C=0 behaves correctly and C=BitWidth is not folded/crashing.
- **Rationale**: New check explicitly requires `ShAmtC < BitWidth` only for OR+K path; boundary behavior should be guarded.
- **Expected Issue**: Boundary C handling (0 or BW) may fold incorrectly or crash.

### Strategy 5: Vector and multi-use robustness

- **Target**: Add negative tests with extra uses of shl/or/add and vectorized analogs; ensure transform does not apply unexpectedly.
- **Rationale**: New pattern uses `m_APInt` and nested `m_OneUse`; incorrect relaxation can leak poison/context assumptions.
- **Expected Issue**: Matcher/use constraints may regress for vector or multi-use DAGs, causing illegal transforms.

Bug Triggering Analysis:
I exercised all managed tests and covered every Phase-1 strategy per test index via differential testing (`difftest`) against `-passes=instcombine`, including:
1) K boundary violations (K with bit C set / high bits set),
2) negative K on odd widths (i33),
3) positive valid folds on odd widths with carry-heavy `Y+K` (i33/i65),
4) shift boundaries (C=0 and C=BW poison cases),
5) vector and multi-use robustness (extra uses of `shl`/`or`, vector lanes).

Observed behavior was consistent between original and optimized IR in all executions tested. For boundary poison cases, both sides produced consistent UB behavior. Fix Weakness Analysis:
The patch’s key guard `ShAmtC < BitWidth && LowBitsC->lshr(ShAmtC).isZero()` held up under stress, including odd bitwidth APInt constants and negative constants. One-use matcher constraints also prevented problematic rewrites in multi-use shapes. Vector variants either folded consistently or simplified through other safe canonicalizations, with no differential miscompile observed.

Conclusion: the fix appears robust for the explored edge cases; no regression or new miscompile was identified.
