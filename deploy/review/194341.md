## Test Strategies

### Strategy 1: Multi-use outer binop blocks equal-op distributive fold

- **Target**: Mutate an existing `or/and/xor` shift-combine test into `%sx = shl i32 %x, 3; %base = or i32 %sx, %m; %sy = shl i32 %y, 3; %r0 = or i32 %base, %sy; %r1 = xor/add/or i32 %base, %z; ret/use both`, so `%base` has more than one use while both shifts remain one-use.
- **Rationale**: The patch specifically guards the enclosing `m_c_BinOp`, and the earliest fold path (`BinOpc == I.getOpcode()` with bitwise ops or `add`+`shl`) does not require constant shift/mask. A shared intermediate such as `%base = or (shl %x, C), %mask` feeding two downstream `or` consumers was the reported growth pattern. This is the most direct regression target for the fix.
- **Expected Issue**: If the one-use guard is incomplete, InstCombine still rewrites the target consumer of the shared intermediate, leaving the original intermediate for the other use and adding a fresh unshifted binop plus shift/top binop. The CHECK should show no duplicated reassociated shift fold for that consumer when the intermediate has multiple users.

### Strategy 2: Multi-use outer binop blocks constant-mask transfer fold

- **Target**: Create/mutate a test like `%sx = lshr i32 %x, 4; %base = xor i32 %sx, 240; %sy = lshr i32 %y, 4; %r0 = and i32 %base, %sy; %r1 = add i32 %base, %z; ret/use both`. Also vary top/binop pair among `and` with `or/xor/add` and `shl` cases with a constant shift amount.
- **Rationale**: The second fold path shifts a constant mask with `GetInvShift` and applies `CanDistributeBinops`. It has different profitability and legality checks from the equal-op path. The added `m_OneUse` should dominate this path too, especially for cases where top binop is `and` and thus `CanDistributeBinops` accepts any constant mask.
- **Expected Issue**: InstCombine may still fire on a shared outer binop in the constant-mask path, causing new mask-shifted/reassociated instructions to be created while the old outer binop remains live for the extra use.

### Strategy 3: Multi-use outer binop blocks AShr-not fold

- **Target**: Mutate an arithmetic-shift test into `%sx = ashr i32 %x, 5; %notx = xor i32 %sx, -1; %sy = ashr i32 %y, 5; %r0 = and/or/xor i32 %notx, %sy; %r1 = add/xor i32 %notx, %z; ret/use both`.
- **Rationale**: The AShr path has separate semantics: it recognizes `BinOp2 == xor` with all-ones mask as a `not`, and only top-level bitwise ops are allowed. Because this path does not go through constant mask shifting or `CanDistributeBinops`, it needs an explicit regression test that the new outer one-use wrapper applies to it as well.
- **Expected Issue**: The arithmetic-shift/not special case may still transform one user of a shared `xor (ashr X,C), -1` into `ashr (binop Y, not X), C`, leaving the original `xor` live and increasing IR.

### Strategy 4: Operand-commutation coverage for multi-use outer binop

- **Target**: Generate four variants from the same base: top-level as `binop %base, %sy` and `binop %sy, %base`; intermediate as `or %sx, %mask` and `or %mask, %sx`; add an extra use of `%base` in every variant. Use FileCheck to ensure `%base` remains and no folded `shift (binop %x,%y)` replacement appears.
- **Rationale**: `MatchBinOp(0)` and `MatchBinOp(1)` try both top-level operand positions, and `m_c_BinOp` may match the shifted value on either side of the intermediate binop. The new `m_OneUse(m_c_BinOp(...))` must guard the actual intermediate regardless of operand order. Missing a commuted form would preserve the original bug for syntactically different but canonicalizable IR.
- **Expected Issue**: A matcher-order bug could allow the fold when the shared outer binop is on the opposite operand of the top-level binop, or when the shifted operand and mask are commuted inside the intermediate binop.

### Strategy 5: Vector and non-splat mask multi-use guard

- **Target**: Mutate scalar tests to `<2 x i32>`/`<4 x i8>` with a vector constant shift amount and non-splat mask, e.g. `%sx = lshr <2 x i32> %x, <i32 1, i32 3>; %base = or <2 x i32> %sx, <i32 2, i32 24>; %sy = lshr <2 x i32> %y, <i32 1, i32 3>; %r0 = xor/and <2 x i32> %base, %sy; %r1 = and <2 x i32> %base, %z; ret/use both`.
- **Rationale**: `CanDistributeBinops` explicitly mentions non-splat vectors, and shifts are lane-wise poison when an element shift amount is out of range. The patch is profitability-oriented, but vector constants exercise different `ConstantFoldBinaryOpOperands` behavior and m_ImmConstant matching. The outer one-use requirement should be uniformly enforced for scalar and vector forms.
- **Expected Issue**: The fold may still cause IR growth for vector/non-splat constants or poison/undef element masks if the outer binop is multi-use, especially because the mask-distribution predicate has vector-specific comments and constant-folding behavior.

### Analysis

Bug Triggering Analysis: The generated tests target all five Phase 1 strategies for the patched matcher in foldBinOpShiftWithShift. Each function keeps the two shift instructions otherwise eligible for the fold but gives the enclosing intermediate binop (%base) an additional use. This is exactly the structural condition that was missing before the patch. The equal-op, constant-mask, AShr-not, commuted, and vector cases exercise the distinct matcher/control-flow paths that could otherwise bypass the new m_OneUse(m_c_BinOp(...)) requirement. Alive2 verification reported no semantic miscompile for the optimized results, and differential execution of scalar tests under llubi (and vector tests under lli because llubi crashes on vector return postprocessing) showed matching outputs.

Fix Weakness Analysis: No correctness bug was found in the patched implementation. The tests are valuable regression coverage because an incomplete fix would still produce folded forms such as a new `shift (binop X, Y)` while leaving `%base` live for its extra use, increasing IR. The observed optimized IR preserves the shared intermediate in the targeted fold positions, with only unrelated reassociation/canonicalization of surrounding xor/add expressions. All tests managed by tests_manager were processed and marked tested, and every proposed Phase 1 strategy was covered.
