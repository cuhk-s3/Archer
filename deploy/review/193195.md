## Test Strategies

### Strategy 1: Cross-family nnan source asymmetry checks

- **Target**: Mutate existing 2-level min/max tests in `minimum.ll`, `maximum.ll`, `minnum.ll`, `maxnum.ll`, `minimumnum.ll`, `maximumnum.ll` to 3-node trees with mixed FMF on only one inner node and/or only outer node; check exact FMF on the factorized call.
- **Rationale**: The patch has intrinsic-family-specific nnan sourcing rules: `minimum/maximum` take nnan from II, `minimumnum/maximumnum` from folded-op, and `minnum/maxnum` from intersection. This is easy to get wrong in one family or one operand-orientation branch.
- **Expected Issue**: Incorrect `nnan` propagation on the new outer call (either missing when safe or present when unsafe), especially for vector forms.

### Strategy 2: Non-nnan FMF intersection stress

- **Target**: Create four structural variants that select each `factorizeMinMaxTree` branch (LHS one-use A/B match, RHS one-use C/D match), decorate only one inner with extra FMF bits, and verify post-transform FMF is exactly expected.
- **Rationale**: Implementation computes `RetFMF = IIFMF & FoldedOpFMF`, where `FoldedOpFMF` is the eliminated sibling. If that choice is wrong on some reuse branch, result may carry flags unsupported by a surviving operand path.
- **Expected Issue**: Overly aggressive propagation of non-nnan FMF (`reassoc`, `arcp`, `afn`, `contract`, `ninf`, `nsz`) to transformed call, enabling illegal assumptions.

### Strategy 3: Vector lane FMF parity tests

- **Target**: Mutate existing vector tests in `minnum.ll`/`maxnum.ll`/`minimum.ll`/`maximum.ll` into factorization shape `m(m(a,b),m(c,b))`, with lane-wise constants including `-0.0/+0.0` and NaN constants; check transform happens and FMF matches scalar expectations.
- **Rationale**: Patch logic is scalar-generic but FMF semantics for min/max intrinsics are vector-sensitive (esp. nsz/nnan behavior with signed zero and NaN per lane). Existing copied tests may under-cover vector trees with 3-node factorization.
- **Expected Issue**: Lost/incorrect FMF in vector lanes due to missing vector coverage for factorization+FMF logic.

### Strategy 4: One-use steering robustness

- **Target**: Start from same algebraic tree and add/remove dummy uses to flip which inner node is reused. Verify transformed call’s FMF remains semantically correct for each intrinsic family (especially nnan special-cases).
- **Rationale**: `FoldedOpFMF` is selected via `(MinMaxOp == LHS ? RHS : LHS)`. If `MinMaxOp` chosen differently by use-count steering, FMF result may differ unintentionally between equivalent DAGs.
- **Expected Issue**: Wrong folded-op FMF source when operand reuse choice flips due to `hasOneUse`, causing nondeterministic FMF output across equivalent shapes.

### Strategy 5: NaN/sNaN semantic differential patterns

- **Target**: Mutate existing tree tests to inject quiet/signaling NaN-producing operands (via bitcast constants and operations) plus selective nnan on only outer/inner calls; verify no invalid strengthening/weakening of resulting call FMF and no unexpected canonicalization to different intrinsic forms.
- **Rationale**: The transform is semantic-only equal if the selected FMF and intrinsic family preserve NaN behavior. `minnum/maxnum` and `minimumnum/maximumnum` differ in NaN classes (qNaN vs all NaN handling distinctions), so flag/source mismatch can silently miscompile.
- **Expected Issue**: Behavioral mismatch in NaN/sNaN corner semantics after factorization, potentially exposed under `-verify-each` or Alive-style reductions.

### Analysis

Bug Triggering Analysis:
- I targeted the new FP extension of `factorizeMinMaxTree` and especially FMF recomputation paths.
- For scalar cases:
  - `maximum` with `nnan` only on the folded inner did not incorrectly propagate `nnan` to the new root.
  - `maximumnum` with `nnan` only on the folded inner did propagate `nnan` to the new root as intended.
  - `maxnum` non-`nnan` FMF intersection behavior matched expectations in transformed IR (`arcp` preserved while unsupported bits were dropped).
- For vector case (`maximum_vec_common`), Alive2 produced a counterexample that depends on poison/sNaN/signed-zero lane behavior under `nnan/nsz` interactions. Differential execution with concrete non-NaN inputs did not show divergence, and llubi instability required lli fallback. This points to a subtle semantic boundary rather than a confirmed executable miscompile in the tested concrete domain.

Fix Weakness Analysis:
- The patch’s FMF logic is nuanced and family-specific. The tests confirm core intended behavior for scalar `nnan` asymmetry and FMF intersection under factorization.
- The remaining weak area is vector + poison/sNaN interaction in proofs: this is where formal verification can still fail due to difficult semantics, even when concrete tests do not diverge.
- Overall, no concrete miscompile was confirmed from the generated targeted cases; the fix appears directionally correct for the exercised scalar paths and expected FMF transfer rules.
