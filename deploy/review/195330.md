## Test Strategies

### Strategy 1: Partial deinterleave shuffle length mismatch

- **Target**: Mutate the canonical <8 x i32> deinterleave masks to partial masks such as <0,2> and <1,3>, with zexts to <2 x i64>.
- **Rationale**: The matcher only checks that the mask is an arithmetic deinterleave pattern, but it does not check that the shuffle result length is exactly half the input vector length. The created bitcast always has half as many elements as the input, while the zext destination has as many elements as the shuffle result.
- **Expected Issue**: Verifier assertion or invalid IR due to replaceInstUsesWith attempting to replace a <2 x i64> zext with a <4 x i64> and/lshr value (or analogous mismatched vector lengths).

### Strategy 2: RHS-selected mask accepted as deinterleave

- **Target**: Create shufflevector <2 x i32> %v, poison, <2 x i32> <i32 1, i32 3>, then zext the <2 x i32> result to <2 x i64>.
- **Rationale**: m_Shuffle(m_Value(V), m_Undef(), ...) permits an undef/poison RHS, and isDeInterleaveMaskOfFactor ignores whether mask indices are within the first input vector. A mask like <1,3> on a <2 x i32> LHS is accepted even though lane 3 selects the RHS, so the transform builds <1 x i64> but the zext has type <2 x i64>.
- **Expected Issue**: Verifier assertion or invalid IR from replacing a zext of a shuffle that selects from the undef/poison RHS with a value whose vector length is based only on the LHS input length.

### Strategy 3: Cross-lane poison contamination for field 0

- **Target**: Use a constant vector or constructed vector with poison in odd lanes and defined values in even lanes; deinterleave field 0 and zext it, then observe through freeze/extract/return or FileCheck the folded constant.
- **Rationale**: The original shuffle/zext reads each selected i32 lane independently. The new bitcast groups an even and odd i32 into one i64 lane; if either half is poison, the whole i64 lane is poison, and bitwise and does not recover the defined low half. This can make a formerly defined field0 result poison when the adjacent field1 input element is poison.
- **Expected Issue**: Miscompile by introducing poison into a lane that was originally defined; e.g. zext(shuffle lane 0) of <i32 42, i32 poison> is 42, but bitcast to i64 becomes poison and the subsequent and is poison.

### Strategy 4: Cross-lane poison contamination for field 1

- **Target**: Use poison in even lanes and defined values in odd lanes; deinterleave field 1 using shuffle or llvm.vector.deinterleave2, zext, and observe the result.
- **Rationale**: For field1, the original result depends only on odd lanes. The replacement lshr operates on the bitcasted i64 lane, which is poison if the even half is poison. Thus lshr produces poison even though the original zext of the odd lane was defined.
- **Expected Issue**: Miscompile by introducing poison into field1 when the even paired lane is poison but the odd lane selected by field1 is defined.

### Strategy 5: Intrinsic deinterleave2 poison isolation

- **Target**: Mutate a vector_deinterleave2 test to pass a vector constant such as <i32 7, i32 poison, i32 9, i32 poison> and use only extractvalue 0 zexts, and symmetrically poison even lanes for extractvalue 1.
- **Rationale**: The same lane-pairing poison issue applies to the intrinsic path. llvm.vector.deinterleave2 semantically separates even and odd elements, but the replacement combines each pair into an i64 lane before extracting with and/lshr, so poison in the unobserved field can poison the observed field.
- **Expected Issue**: Miscompile or excessive poison propagation for llvm.vector.deinterleave2 with poison in one field contaminating the other field after bitcast grouping.

### Analysis

Bug Triggering Analysis:
The new foldExtractionOfVectorDeinterleave helper recognizes shufflevector masks using ShuffleVectorInst::isDeInterleaveMaskOfFactor(mask, 2, Index). That predicate accepts a partial deinterleave mask such as <0,2> because every non-poison lane follows the arithmetic sequence Index + I*2. However, the helper does not check that the shuffle result element count is exactly half of the input vector element count.

For the reproducer, the source vector is <8 x i32>, so the helper creates BitcastedTy by extending element width and halving the input element count: <8 x i32> -> <4 x i64>. It then creates NewField0 as an `and <4 x i64> ...`. But the actual zext being replaced has type <2 x i64>, because the shuffle result mask has only two lanes. When replaceInstUsesWith attempts to RAUW the <2 x i64> zext with the <4 x i64> and, opt aborts with:

  Assertion `New->getType() == getType() && "replaceAllUses of value with new value of different type!"' failed.

This was verified with trans/difftest, both of which observed an opt crash during InstCombine.

A second related crash was also verified with:

```llvm
define <2 x i64> @rhs_selected_mask(<2 x i32> %v) {
  %f1 = shufflevector <2 x i32> %v, <2 x i32> poison, <2 x i32> <i32 1, i32 3>
  %z1 = zext <2 x i32> %f1 to <2 x i64>
  ret <2 x i64> %z1
}
```

Here the mask <1,3> is accepted as deinterleave index 1, but index 3 selects from the second shuffle operand. The replacement type is again derived only from the first operand, producing <1 x i64> while the original zext has <2 x i64>.

Fix Weakness Analysis:
The patch assumes that any factor-2 deinterleave mask implies a full deinterleave of the first input vector, but the IR predicate used does not guarantee either (1) mask length == input length / 2 or (2) all mask indices select from operand 0. The fix should add explicit type/length validation before building the bitcast replacement, e.g. require Field->getType() == VectorType::getHalfElementsVectorType(InputVecTy) (or equivalent destination type equality for NewField0/NewField1 and each zext), and require every non-negative mask element to be less than the first input vector element count.
