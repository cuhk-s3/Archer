## Test Strategies

### Strategy 1: Byval Argument Memcpy Size Mismatch

- **Target**: A coroutine that takes a byval argument with a small alignment, and the coroutine frame assigns it a larger alignment (e.g., due to other fields or max frame alignment).
- **Rationale**: The patch changes `createStoreIntoFrame` to use `Builder.CreateMemCpy` for byval arguments. It computes the size using `DL.getTypeStoreSize(ByValTy)`. However, `getTypeStoreSize` returns the size in bytes including padding, which might be larger than the actual size of the byval argument if it's not a multiple of bytes or has padding. More importantly, `ByValTy` is the type of the byval argument, but the original code used `Builder.CreateLoad(ByValTy, Def)` and `Builder.CreateAlignedStore`. The new code uses `Builder.CreateMemCpy(G, SpillAlignment, Def, SpillAlignment, Size)`. The source alignment is set to `SpillAlignment` (which is the alignment of the field in the coroutine frame), but the actual alignment of the byval argument `Def` might be smaller than `SpillAlignment`. This can lead to unaligned reads from the source `Def`.
- **Expected Issue**: The memcpy will copy more bytes than the actual size of the byval argument, potentially reading out of bounds of the source argument or overwriting other fields in the coroutine frame.

### Strategy 2: Alloca Memcpy Size Mismatch

- **Target**: A coroutine with an alloca that is accessed before `coro.begin` and has a dynamic alignment requirement.
- **Rationale**: The patch introduces `handleAccessBeforeCoroBegin` which uses `Builder.CreateMemCpy` to copy the contents of an alloca to the coroutine frame. It computes the size using `Builder.CreateAllocationSize(Builder.getInt64Ty(), Alloca)`. However, `CreateAllocationSize` returns the size of the allocated type multiplied by the array size. If the alloca has a dynamic alignment buffer in the coroutine frame (i.e., `FrameData.getDynamicAlign(Alloca) != 0`), the pointer `G` returned by `createGEPToFramePointer` is dynamically aligned. The size of the memcpy is just the allocation size, which is correct, but the destination pointer `G` is dynamically aligned. Wait, the issue is that `handleAccessBeforeCoroBegin` uses `FrameData.getAlign(Alloca)` as the destination alignment for the memcpy. But if the alloca has dynamic alignment, `G` is dynamically aligned to `Alloca->getAlign()`, not `FrameData.getAlign(Alloca)`. `FrameData.getAlign(Alloca)` is the alignment of the field in the frame, which might be smaller than the dynamic alignment (e.g., capped by `MaxFrameAlignment`). So the memcpy might be generated with an alignment assumption that is incorrect, or it might use the wrong alignment. Actually, `FrameData.getAlign(Alloca)` is the alignment of the field in the frame, which is capped at `MaxFrameAlignment`. But `G` is dynamically aligned to `Alloca->getAlign()`. So the actual alignment of `G` is `Alloca->getAlign()`. Using `FrameData.getAlign(Alloca)` as the destination alignment is safe but suboptimal. However, what if `Alloca->getAlign()` is smaller than `FrameData.getAlign(Alloca)`? That shouldn't happen. Let's look at the source alignment: `Alloca->getAlign()`. That is correct.
- **Expected Issue**: The memcpy will copy more bytes than the actual size of the alloca, potentially reading out of bounds of the alloca or overwriting other fields in the coroutine frame.

### Strategy 3: Array Alloca Memcpy Size

- **Target**: A coroutine with a static array alloca that is accessed before `coro.begin`.
- **Rationale**: The patch replaces a fatal error for array allocas with a memcpy. We should test if array allocas are correctly copied to the coroutine frame when accessed before coro.begin.
- **Expected Issue**: The memcpy will copy the entire array size, but the original code only copied the first element or failed. Wait, the original code had `report_fatal_error("Coroutines cannot handle copying of array allocas yet")`. The new code removes this and uses `CreateMemCpy` with `CreateAllocationSize`. This is actually a feature enhancement, but it might be buggy if the array size is not a constant. The patch removes the check for `Alloca->isArrayAllocation()` and `ConstantInt`. If the alloca is a VLA (Variable Length Array), `CreateAllocationSize` will return a dynamic size. But `addFieldForAlloca` checks if the size is fixed: `if (!Size || !Size->isFixed()) report_fatal_error(...)`. So VLAs are still rejected. What if the alloca is an array of structs with padding? `CreateAllocationSize` returns `getTypeAllocSize` * array size. This is correct.

### Strategy 4: Address Space Cast for Alloca

- **Target**: A coroutine with an alloca in a non-default address space.
- **Rationale**: The patch uses `CreateAddrSpaceCast` when the type of the frame pointer (which is `ptr` in the default address space) differs from the type of the alloca. If the alloca is in a different address space, the frame pointer needs to be cast to that address space. We should test if this cast works correctly.
- **Expected Issue**: The `createGEPToFramePointer` function casts the pointer to `Orig->getType()`. If `Orig` is an `AllocaInst`, its type is a pointer to the allocated type (in opaque pointers, it's just `ptr`). The cast `Builder.CreateAddrSpaceCast(Ptr, Orig->getType(), ...)` is used. But wait, `Ptr` is already a `ptr` (from `CreateInBoundsPtrAdd`). If `Orig` is also a `ptr`, `CreateAddrSpaceCast` might be a no-op or invalid if the address spaces are the same. In LLVM 15+, opaque pointers are the default, so both are `ptr`. If they have different address spaces, `AddrSpaceCast` is correct. But what if `Orig` is not an `AllocaInst`? The cast is only done `if (auto *AI = dyn_cast<AllocaInst>(Orig))`. For spills, it returns `Ptr` directly, which is a `ptr` in the frame's address space. Then `insertSpills` does `Builder.CreateAlignedLoad(E.first->getType(), GEP, ...)`. Wait, `E.first->getType()` is the type of the spilled value. `GEP` is the pointer to the frame slot. `CreateAlignedLoad` takes the type of the loaded value. This is correct. What about `createStoreIntoFrame`? It does `Builder.CreateAlignedStore(Def, G, SpillAlignment)`. `G` is the pointer to the frame slot. This is also correct.

### Strategy 5: Byval Argument Memcpy Source Alignment

- **Target**: A coroutine that takes a byval argument with a small alignment (e.g., `align 1`), but the coroutine frame assigns it a larger alignment (e.g., due to `MaxFrameAlignment` or other fields).
- **Rationale**: In `createStoreIntoFrame`, when copying a byval argument to the coroutine frame, the patch uses `Builder.CreateMemCpy(G, SpillAlignment, Def, SpillAlignment, Size)`. The fourth argument is the source alignment. It passes `SpillAlignment`, which is the alignment of the destination slot in the coroutine frame. However, the source pointer `Def` (the byval argument) might have a smaller alignment than the frame slot. Passing a larger alignment for the source pointer in `memcpy` can lead to undefined behavior (e.g., unaligned memory access).
- **Expected Issue**: The `createGEPToFramePointer` function uses `CreateInBoundsPtrAdd` to compute the pointer to the frame slot. If the offset is large, this is fine. But wait, `createStoreIntoFrame` also uses `CreateInBoundsPtrAdd`. If `Offset != 0`, it adds the offset. But what if `Offset == 0`? It uses `Shape.FramePtr` directly. `Shape.FramePtr` is the result of `coro.begin`, which is a `ptr`. This is correct. However, in `createGEPToFramePointer`, it always does `Builder.CreateInBoundsPtrAdd(Shape.FramePtr, OffsetVal)`. If `Offset == 0`, it adds 0. This is fine. But wait, `createStoreIntoFrame` does: `if (Offset != 0) G = Builder.CreateInBoundsPtrAdd(G, OffsetVal);`. Why does `createGEPToFramePointer` not check `Offset != 0`? It's a minor optimization difference. More importantly, `createStoreIntoFrame` for byval arguments uses `Builder.CreateMemCpy(G, SpillAlignment, Def, SpillAlignment, Size)`. The source alignment is `SpillAlignment`. But `Def` is the byval argument pointer. Its alignment is the alignment of the byval argument, which might be different from `SpillAlignment`. `SpillAlignment` is `FrameData.getAlign(Def)`. If the byval argument has a smaller alignment than the frame slot, using `SpillAlignment` for the source pointer `Def` in `memcpy` will result in an unaligned read being marked as aligned, causing undefined behavior.
