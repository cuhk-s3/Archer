## Test Strategies

### Strategy 1: FAdd Self-Add with PreserveSign Denormal Mode

- **Target**: A function with `denormal-fp-math` set to `preserve-sign,preserve-sign` containing an `fadd x, x` where `x` is known not to be `+0.0` but can be a positive subnormal.
- **Rationale**: The patch adds a check `if (SelfAdd && KnownRHS.isKnownNeverLogicalPosZero(Mode) && (Mode.Output == DenormalMode::IEEE || Mode.Output == DenormalMode::PreserveSign)) Known.knownNot(fcPosZero);`. However, if `Mode.Output == DenormalMode::PreserveSign`, a negative subnormal input `x` will be flushed to `-0.0` on output, but wait, `x + x` where `x` is a negative subnormal. If `x` is a negative subnormal, `x + x` is also a negative subnormal (or negative normal). If it's a negative subnormal, and `Mode.Output == DenormalMode::PreserveSign`, it flushes to `-0.0`. So it doesn't produce `+0.0`. What if `x` is a positive subnormal? If `x` is a positive subnormal, `x + x` is a positive subnormal (or positive normal). If it's a positive subnormal, and `Mode.Output == DenormalMode::PreserveSign`, it flushes to `+0.0`. But wait, `KnownRHS.isKnownNeverLogicalPosZero(Mode)` means `x` is never logical positive zero. If `Mode.Input == DenormalMode::PreserveSign`, a positive subnormal is NOT a logical positive zero (it's treated as positive subnormal, not flushed to zero on input). So `x` can be a positive subnormal. Then `x + x` is a positive subnormal. Then on output, it is flushed to `+0.0` because `Mode.Output == DenormalMode::PreserveSign` flushes positive subnormals to `+0.0`. Thus, `x + x` CAN be `+0.0`. But the patch says `Known.knownNot(fcPosZero)`, which is incorrect!
- **Expected Issue**: The compiler will incorrectly deduce that the result of `fadd x, x` cannot be `+0.0`, which might lead to incorrect optimizations (e.g., replacing a check for `+0.0` with `false`).
