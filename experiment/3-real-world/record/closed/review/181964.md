## Test Strategies

### Strategy 1: Vector Types in Select Pattern

- **Target**: A test case with a vector select pattern like `select <2 x i1> %cmp, <2 x i32> zeroinitializer, <2 x i32> %sub`.
- **Rationale**: The `isSelectKnownNonNegative` function uses `dyn_cast<ConstantInt>(ZeroVal)` to check if the true/false value is zero. This cast will fail for vector types (e.g., `<2 x i32>`), meaning the optimization will be missed for vectors. However, if it were changed to `m_Zero()`, it might incorrectly handle vectors with poison elements if not careful. The current implementation simply misses the optimization for vectors.
- **Expected Issue**: The compiler might incorrectly assume the result is non-negative, leading to incorrect optimizations (e.g., removing a subsequent `smax(..., 0)` or `icmp sge ..., 0`).

### Strategy 2: Poison Elements in Vector Constants

- **Target**: A test case with a vector select pattern and a partially poisonous zero vector.
- **Rationale**: If the `isSelectKnownNonNegative` function is ever updated to support vectors (e.g., using `m_Zero()`), it might incorrectly handle vectors with poison elements (e.g., `<0, poison>`). The current implementation uses `ConstantInt`, which only matches scalar integers, so it's safe from this specific issue, but it's a common pitfall in ValueTracking.
- **Expected Issue**: The compiler might incorrectly assume the result is non-negative, leading to incorrect optimizations.

### Strategy 3: Mismatched Operands in Select Pattern

- **Target**: A test case with `b = 1` and `a = INT_MIN` where `b - a` overflows.
- **Rationale**: The `isSelectKnownNonNegative` function checks if `Sub->getOperand(0) == CmpLHS` and `Sub->getOperand(1) == CmpRHS`. It does not check if the subtraction has `nsw` or `nuw` flags. If the subtraction overflows, the result might not be non-negative. For example, if `b = INT_MIN` and `a = 1`, `b < a` is true, so the result is `0`. If `b = 1` and `a = INT_MIN`, `b < a` is false, so the result is `1 - INT_MIN`, which overflows to `INT_MIN + 1` (a negative number). Wait, if `b = 1` and `a = INT_MIN`, `b < a` is false. The result is `b - a`. `1 - INT_MIN` overflows to a negative number. The function assumes `b - a >= 0` when `b >= a`. But if `b - a` overflows, it can be negative!
- **Expected Issue**: The compiler might incorrectly assume the result is non-negative, leading to incorrect optimizations.

### Strategy 4: Overflow in Subtraction

- **Target**: A test case with a select pattern `b < a ? 0 : b - a` where `b - a` can overflow (e.g., `b = 1`, `a = INT_MIN`).
- **Rationale**: The `isSelectKnownNonNegative` function assumes that if `b >= a`, then `b - a >= 0`. However, in signed integer arithmetic, `b - a` can overflow and become negative. For example, if `b = 1` and `a = INT_MIN`, `b >= a` is true. The result is `b - a = 1 - INT_MIN`, which overflows to `INT_MIN + 1` (a negative number). The function does not check for the `nsw` (no signed wrap) flag on the subtraction. Therefore, it incorrectly assumes the result is non-negative even when overflow occurs.
- **Expected Issue**: The compiler might incorrectly assume the result is non-negative, leading to incorrect optimizations.

### Strategy 5: Overflow in smin Subtraction

- **Target**: A test case with a subtraction pattern `LHS - smin(LHS, X)` where the subtraction can overflow (e.g., `LHS = 1`, `X = INT_MIN`).
- **Rationale**: The `isSubtractionKnownNonNegative` function assumes that `LHS - smin(LHS, X) >= 0`. However, if `LHS - smin(LHS, X)` overflows, the result can be negative. For example, if `LHS = 1` and `X = INT_MIN`, `smin(LHS, X) = INT_MIN`. Then `LHS - smin(LHS, X) = 1 - INT_MIN`, which overflows to `INT_MIN + 1` (a negative number). The function does not check for the `nsw` flag on the subtraction. Therefore, it incorrectly assumes the result is non-negative even when overflow occurs.
- **Expected Issue**: The compiler might incorrectly assume the result is non-negative, leading to incorrect optimizations.
