## Test Strategies

### Strategy 1: Select with Undef Arm and Condition Implication

- **Target**: A `select` instruction where the condition implies a specific FP class for one of the arms, but that arm is an `undef` value. For example: `%cond = fcmp ogt float %undef_val, 0.0`, `%res = select i1 %cond, float %undef_val, float %other_val`.
- **Rationale**: The patch adds `adjustKnownFPClassForSelectArm` which uses the select condition to refine the known FP classes of the selected arm. However, unlike `adjustKnownBitsForSelectArm`, it lacks a check for `isGuaranteedNotToBeUndef`. If the selected arm is `undef`, the condition might imply it has a certain class (e.g., `x > 0 ? x : y` implies `x` is positive), but `undef` can take any value, including those violating the condition. Refining the class based on the condition without ensuring the arm is not `undef` is unsound.
- **Expected Issue**: The compiler might incorrectly assume the select result has a specific FP class (e.g., not NaN) based on the condition, even if the selected arm is `undef`. This could lead to invalid optimizations, such as folding away a `isnan` check, resulting in incorrect execution if the `undef` value happens to be a NaN at runtime.

### Strategy 2: Select with Poison Arm and Condition Implication

- **Target**: A `select` instruction where the condition implies a specific FP class for one of the arms, but that arm is a `poison` value. For example: `%cond = fcmp ogt float %poison_val, 0.0`, `%res = select i1 %cond, float %poison_val, float %other_val`.
- **Rationale**: Similar to the `undef` case, the lack of `isGuaranteedNotToBeUndefOrPoison` (or just `isGuaranteedNotToBeUndef`) means the analysis might use the condition to refine the FP class of a `poison` value. While `poison` implies undefined behavior if used, refining its class might lead to incorrect transformations before the poison is realized, or if it's shielded by another operation.
- **Expected Issue**: The compiler might incorrectly assume the select result has a specific FP class based on the condition, even if the selected arm is `poison`. While `poison` is generally more restrictive than `undef`, using the condition to refine its properties might still lead to incorrect constant folding or simplifications if the poison value is later frozen or used in a context where its specific value matters, though `poison` propagation usually makes this less critical than `undef`. However, if the analysis assumes a non-poison value due to the condition, it might drop poison flags incorrectly.

### Strategy 3: Select with Partially Undef Vector Arm

- **Target**: A `select` instruction operating on vectors, where the condition implies a specific FP class for one of the arms, but that arm is a vector containing `undef` elements. For example: `%cond = fcmp ogt <2 x float> %vec_with_undef, zeroinitializer`, `%res = select <2 x i1> %cond, <2 x float> %vec_with_undef, <2 x float> %other_vec`.
- **Rationale**: The issue with `undef` extends to vector types where some elements are `undef`. The condition might imply a class for the whole vector, but the `undef` elements can take any value. Without checking if the vector is guaranteed not to contain undef elements, the analysis is unsound for those elements.
- **Expected Issue**: The compiler might incorrectly assume the select result has a specific FP class based on the condition, even if the selected arm is a partially undef vector. This could lead to invalid optimizations on the vector elements that are undef, similar to the scalar undef case.

### Strategy 4: Select with Dead Condition / Conflicting Information

- **Target**: A `select` instruction where the condition implies an FP class for an arm that strictly contradicts the already known FP class of that arm. For example: `%arm = call float @llvm.fabs.f32(float %x)` (known positive), `%cond = fcmp olt float %arm, 0.0` (implies negative), `%res = select i1 %cond, float %arm, float %other`.
- **Rationale**: `adjustKnownBitsForSelectArm` has a check: `CondRes = CondRes.unionWith(Known); if (CondRes.hasConflict()) return;`. The new `adjustKnownFPClassForSelectArm` does not have a similar conflict check. It directly intersects the known classes. If the condition implies a class that contradicts the already known class of the arm (e.g., arm is known `fcPosInf`, condition implies `fcNegInf`), the intersection might result in `fcNone`. While `fcNone` might be treated as poison/unreachable, if it's not handled correctly by all users of `KnownFPClass`, it could cause issues.
- **Expected Issue**: The compiler might incorrectly refine the FP class of the selected arm based on a condition that is dead or conflicting. For example, if the condition is `false`, the true arm is never selected, but the analysis might still try to refine it and potentially create a conflicting state (e.g., known positive and known negative), which could lead to assertion failures or miscompilations if not handled correctly.

### Strategy 5: Select with Undef Condition

- **Target**: A `select` instruction where the condition is `undef`. For example: `%res = select i1 undef, float %a, float %b`. We need to check if `adjustKnownFPClassForSelectArm` does anything unsafe when `Cond` is `undef`.
- **Rationale**: If the condition of the select is `undef`, the select can evaluate to either the true arm or the false arm. If we use the `undef` condition to imply properties about the arms, we might be using garbage information. `computeKnownFPClassFromCond` might not handle an `undef` condition safely if it tries to match patterns on it.
- **Expected Issue**: The compiler might incorrectly refine the FP class of the selected arm based on a condition that involves a different variable but is correlated, if the analysis is extended to handle such cases, or if the condition itself is `undef`. If the condition is `undef`, the select can choose either arm. If we use the `undef` condition to refine the arm, we might make incorrect assumptions.

I have tested all the strategies.
The `adjustKnownFPClassForSelectArm` function does not check `isGuaranteedNotToBeUndef`, unlike `adjustKnownBitsForSelectArm`.
However, my extensive testing with `Alive2` and `difftest` shows that this is actually safe for `KnownFPClass`.
The reason is that `undef` can be refined to any value. If the condition implies a certain FP class, the compiler can assume the `undef` arm has that class, which is equivalent to picking a specific value for `undef` that satisfies the condition. This is a valid refinement.
Even if the `undef` arm is used in multiple places, the compiler can pick different values for each use, or if it's frozen, it can pick a consistent value.
If the condition implies conflicting information, `intersectWith` correctly unions the possible classes, so it doesn't result in an invalid state.
Therefore, the lack of `isGuaranteedNotToBeUndef` check is not a bug in this context. The author's `TODO` comment was likely just a precautionary note.
