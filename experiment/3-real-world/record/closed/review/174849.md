## Test Strategies

### Strategy 1: fpext_snan_to_qnan

- **Target**: `fpext` instruction with a source operand known to be `sNaN` (e.g., via `llvm.assume(llvm.is.fpclass(..., fcSNan))`), and the result is checked for `qNaN` using `llvm.is.fpclass(..., fcQNan)`.
- **Rationale**: `fpext` of a signaling NaN (`sNaN`) produces a quiet NaN (`qNaN`). If the user demands `fcQNan` but not `fcSNan`, the `SrcDemandedMask` computed by the patch will only contain `fcQNan`. When `SimplifyDemandedFPClass` is called on the source operand, it will replace the `sNaN` source with `poison` because it thinks the source's class (`fcSNan`) is not demanded. However, the original `fpext` would have produced a `qNaN`, which IS demanded.
- **Expected Issue**: The `fpext` instruction will be incorrectly replaced with `poison` because `SimplifyDemandedFPClass` will see that the source is `sNaN` and the demanded mask is `fcQNan`, which do not intersect. This leads to a miscompilation where a demanded `qNaN` result is replaced with `poison`.

## Bugs Found

### Bug #1

**Analysis:**
The original IR returns `true` (1) because `fpext` of `sNaN` produces `qNaN`, and `llvm.is.fpclass` checks for `qNaN`. The transformed IR returns `false` (0) because `InstCombine` incorrectly assumes that `fpext` of `sNaN` is `sNaN`, and since `sNaN` does not intersect with `qNaN`, it folds the check to `false`. This is a miscompilation.

**Original LLVM IR:**
```llvm

declare i1 @llvm.is.fpclass.f32(float, i32)
declare void @llvm.assume(i1)

define i1 @test(i16 %x) {
  %f = bitcast i16 %x to half
  %is.snan = call i1 @llvm.is.fpclass.f16(half %f, i32 1)
  call void @llvm.assume(i1 %is.snan)
  %result = fpext half %f to float
  %class = call i1 @llvm.is.fpclass.f32(float %result, i32 2)
  ret i1 %class
}

define i1 @main(i32 %argc, ptr %argv) {
entry:
  %r = call i1 @test(i16 32000)
  ret i1 %r
}
```
