## Test Strategies

### Strategy 1: FDiv with Known Non-Zero, Non-Inf, Non-NaN Operand

- **Target**: `fdiv float %X, %X` where `%X` is constrained to be normal and non-zero.
- **Rationale**: When `X` is known to be a normal non-zero value (e.g., bounded by `fmax(Y, 1.0)`), `X / X` is exactly 1.0 and never NaN. This tests the core logic of the patch clearing `fcNan`.
- **Expected Issue**: The analysis should correctly identify that the result cannot be NaN, allowing optimizations like folding `X / X == X / X` to true or removing `fcmp uno`.

### Strategy 2: FDiv with Subnormal Operand and DAZ Denormal Mode

- **Target**: `fdiv float %X, %X` where `%X` is subnormal, compiled with a function attribute enabling DAZ (e.g., `"denormal-fp-math"="preserve-sign,preserve-sign"` or similar flush-to-zero mode).
- **Rationale**: The patch uses `isKnownNeverLogicalZero` with the function's denormal mode. If `X` can be subnormal and the mode flushes subnormals to zero, `X` is a logical zero, and `X / X` can be NaN.
- **Expected Issue**: If the denormal mode is DAZ (Denormals-Are-Zero), a subnormal `X` is treated as zero, making `X / X` evaluate to `0.0 / 0.0` which is NaN. The analysis must not clear `fcNan` in this case.

### Strategy 3: FRem with Known Non-Zero, Non-Inf, Non-NaN Operand

- **Target**: `frem float %X, %X` where `%X` is constrained to be normal and non-zero.
- **Rationale**: Similar to `fdiv`, `X % X` is exactly `[+-]0.0` and never NaN if `X` is not zero, infinity, or NaN. This tests the `FRem` path of the patch.
- **Expected Issue**: The analysis should correctly identify that the result cannot be NaN, allowing optimizations like folding `X % X == X % X` to true.

### Strategy 4: FDiv with Known Non-sNaN Operand

- **Target**: `fdiv float %X, %X` where `%X` is known to be a quiet NaN or a normal number (e.g., result of an operation that silences `sNaN`).
- **Rationale**: The patch explicitly clears `fcSNan` if the input is known never to be `sNaN`. This tests the `else if (KnownSrc.isKnownNever(fcSNan))` branch.
- **Expected Issue**: The analysis should clear `fcSNan` for the result, allowing optimizations that assume the result is not a signaling NaN.

### Strategy 5: FDiv with sNaN Operand

- **Target**: `fdiv float %X, %X` where `%X` is explicitly an `sNaN` (e.g., via `llvm.experimental.constrained.fdiv` or a constant `sNaN`).
- **Rationale**: If `X` is an `sNaN`, `X / X` evaluates to a `qNaN`. The patch only clears `fcSNan` if `X` is known *never* to be `sNaN`. This tests the boundary of the `sNaN` logic.
- **Expected Issue**: The analysis might conservatively retain `fcSNan` or incorrectly clear it depending on how `sNaN` is handled. In IEEE 754, `sNaN / sNaN` yields `qNaN`, so the result is never `sNaN`. The patch's conservative check might miss this, which is a potential gap.
