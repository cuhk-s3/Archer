## Test Strategies

### Strategy 1: Huge Scalar Integer Type

- **Target**: A function containing the shift-mask pattern `(x & ((1 << MaskShAmt) - 1)) << ShiftShAmt` using a huge scalar integer type like `i5000000`.
- **Rationale**: The patch adds a check to prevent extending an integer type if its doubled bit width exceeds `MAX_INT_BITS`. Testing with an integer type larger than `MAX_INT_BITS / 2` (e.g., `i5000000`) ensures the check correctly prevents the crash.
- **Expected Issue**: The compiler will crash due to an assertion failure in `IntegerType::get` because it attempts to create an integer type with a bit width greater than `MAX_INT_BITS` (8388608).

### Strategy 2: Huge Vector Integer Type

- **Target**: A function containing the shift-mask pattern using a vector type with huge integer elements, like `<2 x i5000000>`.
- **Rationale**: The optimization also applies to vector types. `WidestTy->getScalarSizeInBits()` gets the element's bit width. If the element type is huge (e.g., `<2 x i5000000>`), extending it will also exceed `MAX_INT_BITS`. This tests that the scalar size check correctly protects vector types as well.
- **Expected Issue**: The compiler will crash due to an assertion failure in `IntegerType::get` when extending the vector element type.

### Strategy 3: Boundary Scalar Integer Type (Exactly Half MAX_INT_BITS)

- **Target**: A function containing the shift-mask pattern using the exact boundary integer type `i4194304`.
- **Rationale**: Testing the exact boundary `i4194304` (which is `8388608 / 2`). The doubled width is exactly `8388608` (`MAX_INT_BITS`). The check `> MAX_INT_BITS` should evaluate to false, allowing the optimization to proceed without crashing.
- **Expected Issue**: The compiler might crash if the boundary condition is off-by-one, or it might incorrectly skip the optimization.

### Strategy 4: Boundary Scalar Integer Type + 1 (Half MAX_INT_BITS + 1)

- **Target**: A function containing the shift-mask pattern using the integer type `i4194305`.
- **Rationale**: Testing the boundary + 1 `i4194305`. The doubled width is `8388610`, which is strictly greater than `MAX_INT_BITS`. The check should evaluate to true, preventing the optimization and the crash.
- **Expected Issue**: The compiler will crash if the boundary condition is off-by-one and allows the extension.
