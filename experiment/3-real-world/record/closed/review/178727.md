## Test Strategies

### Strategy 1: Mismatched Type Width for PtrToAddr

- **Target**: A loop where the induction variable (and thus `S`) has an integer type that is different from the pointer index width of the pointer PHI node being reused (e.g., an `i32` induction variable and an `i64` pointer index width, or vice versa).
- **Rationale**: The patch replaces `Builder.CreatePtrToInt(BaseV, DiffV->getType())` with `Builder.CreatePtrToAddr(BaseV)`. `CreatePtrToAddr` creates a `ptrtoaddr` instruction, which converts a pointer to an integer type matching the pointer index width of the address space. However, the subsequent instruction is `Builder.CreateAdd(BaseV, DiffV)`. If the type of `BaseV` (which is now the index width of the pointer) does not match the type of `DiffV` (which is `STy`), the `CreateAdd` will fail an assertion or generate invalid IR. The original code used `CreatePtrToInt(BaseV, DiffV->getType())`, which explicitly cast the pointer to the correct integer type. The new code assumes `CreatePtrToAddr` will produce a type compatible with `DiffV`, which is only true if `STy` matches the pointer index width.
- **Expected Issue**: The compiler might crash or generate invalid IR because `ptrtoaddr` requires the destination type to match the pointer index width of the address space, but the code does not check if `STy` (the type of the SCEV expression `S`) matches the index width of `PhiTy` (the pointer type of the PHI node).

### Strategy 2: SCEV Type Mismatch in getMinusSCEV

- **Target**: A loop where the induction variable `S` has a different integer type than the pointer index width of the pointer PHI node.
- **Rationale**: In `tryToReuseLCSSAPhi`, when `STy->isIntegerTy() && PhiTy->isPointerTy()`, the code calls `SE.getPtrToAddrExpr(ExitSCEV)`. `getPtrToAddrExpr` returns a SCEV expression of the integer type corresponding to the pointer index width. Then it calls `CanReuse(AddrSCEV)`, which calls `SE.getMinusSCEV(S, AddrSCEV)`. `getMinusSCEV` requires both operands to have the same type (or compatible types). If `STy` (the type of `S`) is different from the type of `AddrSCEV` (the pointer index width), `getMinusSCEV` might assert or fail to compute the difference. The original code used `SE.getPtrToIntExpr(ExitSCEV, STy)`, which explicitly cast the pointer SCEV to `STy`, ensuring the types matched before calling `getMinusSCEV`.
- **Expected Issue**: The compiler might fail to reuse a valid PHI node because `SE.getPtrToAddrExpr(ExitSCEV)` might return a SCEV expression with a different type than `STy`, causing `SE.getMinusSCEV(S, AddrSCEV)` to fail or return `CouldNotCompute`.

### Strategy 3: Invalid Add Instruction due to Type Mismatch

- **Target**: A loop where `ptrtoaddr` fails to produce a reusable diff, but `ptrtoint` succeeds, and the type of `S` (`STy`) is different from the pointer index width.
- **Rationale**: The patch changes the fallback for pointer PHI nodes. If `STy` is an integer and `PhiTy` is a pointer, it tries `ptrtoaddr` first, then `ptrtoint`. If `ptrtoint` succeeds (because `ptrtoaddr` failed to produce a reusable diff), `Diff` is based on `IntSCEV`, which has type `STy`. Later, when expanding, if `PhiTy->isPointerTy()` and `STy` is not a pointer, it executes `BaseV = Builder.CreatePtrToAddr(BaseV);` unconditionally, even if the successful diff was found using `ptrtoint`! This means `BaseV` will have the pointer index width type, but `DiffV` will have type `STy`. If these types differ, `Builder.CreateAdd(BaseV, DiffV)` will crash. The original code used `Builder.CreatePtrToInt(BaseV, DiffV->getType())`, which always matched the types.
- **Expected Issue**: The compiler might generate invalid IR or crash because `DiffV` might have a different type than the result of `CreatePtrToAddr(BaseV)`.

### Strategy 4: Non-Default Address Space Pointer Size vs Index Width

- **Target**: A loop with a pointer PHI node in a non-default address space where the pointer size is different from the index width (e.g., `addrspace(1)` with 64-bit pointers and 32-bit index width).
- **Rationale**: `ptrtoaddr` extracts the index bits of a pointer, which might be smaller than the pointer size in some address spaces. If the SCEV expression `S` models the full pointer size or a different integer type, using `ptrtoaddr` might drop high bits or produce a type mismatch. The patch assumes `ptrtoaddr` is a drop-in replacement for `ptrtoint` in this context, but their semantics differ for non-default address spaces.
- **Expected Issue**: The compiler might incorrectly reuse a PHI node or generate invalid IR when dealing with non-default address spaces where the pointer size and index width differ.
