## Test Strategies

### Strategy 1: Atomic Store Forwarding

- **Target**: A test case with a non-atomic store followed by an atomic load to the same address, where the address is selected by a `select` instruction.
- **Rationale**: The fix checks if the store is simple (`SI->isSimple()`), which means it is not atomic and not volatile. However, the load being analyzed might be atomic. If the load is atomic, forwarding from a non-atomic store violates the memory model. The original code in `AnalyzeLoadAvailability` checks `S->isAtomic() < Load->isAtomic()` before forwarding, but `findDominatingValue` does not check the load's atomicity.
- **Expected Issue**: The load will be incorrectly replaced with the store's value, leading to a miscompilation where the loaded value is not atomic, violating the memory model.

### Strategy 2: Volatile Load Forwarding

- **Target**: A test case with a non-atomic store followed by an unordered atomic load to the same address, where the address is selected by a `select` instruction.
- **Rationale**: The fix checks if the store is simple (`SI->isSimple()`), which means it is not atomic and not volatile. However, the load being analyzed might be volatile. If the load is volatile, it cannot be replaced by a forwarded value. The original code in `AnalyzeLoadAvailability` checks `Load->isUnordered()` (which implies not volatile and not strongly ordered atomic), but `findDominatingValue` is called from `AnalyzeLoadAvailability` which already asserts `Load->isUnordered()`. Wait, `Load->isUnordered()` means it's not volatile and at most Unordered atomic. So the load is not volatile. But what if the load is Unordered atomic? A simple store is NotAtomic. Forwarding from NotAtomic to Unordered atomic is invalid (`S->isAtomic() < Load->isAtomic()`).
- **Expected Issue**: The load will be incorrectly replaced with the store's value, leading to a miscompilation where the loaded value is not volatile, violating the memory model.

### Strategy 3: Type Mismatch Forwarding

- **Target**: N/A
- **Rationale**: The fix checks `SI->getValueOperand()->getType() == LoadTy`. This ensures the types are exactly the same. However, what if the types are different but can be coerced? The fix is conservative here, so it won't miscompile. But what if the types are exactly the same, but the store is to a different pointer type? The fix checks `SI->getPointerOperand() == Loc.Ptr`. This ensures the pointers are exactly the same SSA value. This is safe.
- **Expected Issue**: The load will be incorrectly replaced with the store's value, leading to a miscompilation where the loaded value has a different type than the stored value, but the types have the same size.

### Strategy 4: Control Flow in Extended Basic Block

- **Target**: N/A
- **Rationale**: The fix iterates over the extended basic block (chain of basic blocks with single predecessors). It starts from `From` instruction and goes backwards. Since it only goes up through single predecessors, every instruction visited dominates the `From` instruction. This is safe.
- **Expected Issue**: The load will be incorrectly replaced with the store's value, leading to a miscompilation where the store does not actually dominate the load due to control flow within the extended basic block.

### Strategy 5: Clobbering Instruction Before Load

- **Target**: N/A
- **Rationale**: The fix checks `isModSet(BatchAA.getModRefInfo(Inst, Loc))`. If it is, it checks if it's a simple store to the exact location. If it is, it forwards the value. If it's not, it returns `nullptr` (stops searching). This correctly handles clobbers. However, what if the store itself only partially overwrites the location? The fix checks `SI->getPointerOperand() == Loc.Ptr` and `SI->getValueOperand()->getType() == LoadTy`. This ensures the store writes to the exact same pointer and with the exact same type as the load. This means it fully overwrites the location. This is safe.
- **Expected Issue**: The load will be incorrectly replaced with the store's value, leading to a miscompilation where the store is clobbered by another instruction before the load.

### Strategy 6: Forwarding from Non-Atomic Store to Atomic Load

- **Target**: A test case with a `select` instruction choosing between two pointers. One pointer has a non-atomic store dominating the `select`. The other pointer has a non-atomic store dominating the `select`. An `unordered` atomic load reads from the selected pointer.
- **Rationale**: The fix adds forwarding from a store in `findDominatingValue`. It checks `SI->isSimple()`, which means the store is not atomic and not volatile. However, the load that initiated the search (passed as `LoadTy` but we don't have the `LoadInst` itself in `findDominatingValue`) might be atomic (e.g., `unordered`). In `AnalyzeLoadAvailability`, there is a check: `if (S->isAtomic() < Load->isAtomic()) return std::nullopt;`. But `findDominatingValue` does not have access to the `LoadInst`'s atomicity, only its type. If `findDominatingValue` returns a value from a non-atomic store, `AnalyzeLoadAvailability` will use it to construct an `AvailableValue::getSelect`, bypassing the atomicity check. This allows forwarding a non-atomic store to an atomic load, violating the memory model.
- **Expected Issue**: The load will be incorrectly replaced with the store's value, leading to a miscompilation where the load is atomic and the store is not.
