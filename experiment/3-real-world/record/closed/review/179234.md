## Test Strategies

### Strategy 1: Test ldexp with large positive exponent and subnormal/zero input

- **Target**: `ldexp` with `exp` >= MantissaBits and `x` being subnormal.
- **Rationale**: The patch changes the logic to use `computeKnownBits` for the exponent. If `ExpBits.getSignedMinValue().sge(MantissaBits)`, it assumes the result cannot be subnormal. However, if the input `x` is `0.0` or `-0.0`, `ldexp(0.0, exp)` is `0.0`, which is not subnormal, but if `x` is a very small subnormal, multiplying it by `2^exp` might still result in a subnormal if `exp` is not large enough to overcome the subnormal range, or it might just be normal. Wait, the logic says `if (ExpBits.getSignedMinValue().sge(static_cast<int64_t>(MantissaBits))) Known.knownNot(fcSubnormal);`. If `x` is subnormal, say `2^(-149)` (for f32, MantissaBits=23). If `exp` is 23, `2^(-149) * 2^23 = 2^(-126)`, which is the smallest normal number. So if `exp >= 23`, the result is guaranteed to be at least normal (or zero/inf/nan). Wait, what if `x` is `0.0`? `ldexp(0.0, 23) = 0.0`, which is not subnormal. What if `x` is `NaN`? `ldexp(NaN, 23) = NaN`, not subnormal. So the logic `Known.knownNot(fcSubnormal)` seems correct. Let's test if there are any edge cases, e.g., `x` being a very small subnormal and `exp` being exactly `MantissaBits`.
- **Expected Issue**: The compiler might incorrectly assume the result of `ldexp(x, exp)` cannot be subnormal when `exp` is a large positive value, but `x` is a very small subnormal value or zero.

### Strategy 2: Test ldexp with negative exponent and infinity input

- **Target**: `ldexp` with `exp < 0` and `x` being `FLT_MAX`.
- **Rationale**: The patch adds `if (ExpBits.isNegative()) { if (KnownSrc.isKnownNeverPosInfinity()) Known.knownNot(fcPosInf); ... }`. This means if `exp < 0`, `ldexp` cannot introduce new infinities. If the input is already infinity, `ldexp(inf, -5) = inf`. The logic says "if KnownSrc is known never pos infinity, then result is known never pos infinity". This is correct because `ldexp(x, <0)` decreases the magnitude, so it can't overflow to infinity. Let's test if this holds for all edge cases, e.g., `x` being the maximum normal number and `exp` being `-1`.
- **Expected Issue**: The compiler might incorrectly propagate `fcPosInf` or `fcNegInf` when `exp` is known to be `<= 0`.

### Strategy 3: Test ldexp with non-negative exponent and zero/subnormal input

- **Target**: `ldexp` with `exp >= 0` and `x` being `FLT_MIN`.
- **Rationale**: The patch adds `if (ExpBits.isNonNegative()) { if (KnownSrc.isKnownNeverPosSubnormal()) Known.knownNot(fcPosSubnormal); ... }`. This means if `exp >= 0`, `ldexp` cannot introduce new subnormals or zeros. If the input is already subnormal, `ldexp(subnormal, 1)` might still be subnormal. The logic says "if KnownSrc is known never pos subnormal, then result is known never pos subnormal". This is correct because `ldexp(x, >=0)` increases the magnitude, so a normal number cannot become subnormal or zero. Let's test if this holds for all edge cases, e.g., `x` being the smallest normal number and `exp` being `0` or `1`.
- **Expected Issue**: The compiler might incorrectly propagate zero or subnormal properties when `exp` is known to be `>= 0`.

### Strategy 4: Test ldexp with zero exponent and denormal input under DAZ/FTZ

- **Target**: `ldexp` with `exp == 0` and `x` being a denormal number, under `denormal-fp-math=ieee,ieee` and `denormal-fp-math=preserve-sign,preserve-sign`.
- **Rationale**: The patch adds `if (ExpBits.isConstant() && ExpBits.getConstant().isZero()) { Known.propagateCanonicalizingSrc(KnownSrc, F->getDenormalMode(FltSem)); }`. This correctly handles the canonicalization of the source when `exp == 0`. We should test if `ldexp(x, 0)` correctly flushes denormals to zero when DAZ/FTZ is enabled.
- **Expected Issue**: The compiler might incorrectly fold `ldexp(x, 0)` to `x` without preserving the denormal flushing semantics.

### Strategy 5: Test ldexp with vector exponent containing zeros and non-zeros

- **Target**: Vector `ldexp` with various exponent vectors (splat 0, splat negative, splat positive, mixed).
- **Rationale**: The patch uses `ExpBits.isConstant() && ExpBits.getConstant().isZero()`. For vectors, `isConstant()` might return true if all elements are the same constant, or it might handle splats. If `exp` is a vector like `<0, 1>`, `isConstant()` will be false. We should test vector `ldexp` with splat zero, splat negative, splat positive, and mixed exponents to ensure `computeKnownBits` handles them correctly.
- **Expected Issue**: The compiler might miscompile when `exp` is a vector with some elements being zero and others being non-zero, or when `exp` is poison/undef.
