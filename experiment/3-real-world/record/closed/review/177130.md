## Test Strategies

### Strategy 1: InstCombine: Preserve trivially empty lifetime ranges with sanitize_memtag

- **Target**: An LLVM IR test case for InstCombine containing a function with the `sanitize_memtag` attribute and a trivially empty lifetime range (e.g., `llvm.lifetime.start` immediately followed by `llvm.lifetime.end`).
- **Rationale**: InstCombine normally removes trivially empty lifetime ranges (where `lifetime.start` is immediately followed by `lifetime.end` or has no uses in between). The patch explicitly prevents this removal when the `sanitize_memtag` attribute is present on the function. Testing this ensures the InstCombine part of the patch works correctly.
- **Expected Issue**: The trivially empty lifetime markers (`llvm.lifetime.start` and `llvm.lifetime.end`) would be incorrectly optimized away by InstCombine, which would prevent the memtag-stack sanitizer from detecting use-after-scope bugs for this range.

### Strategy 2: Clang: Emit lifetime markers for local variables at -O0

- **Target**: A C/C++ source file with a local variable, compiled with `-fsanitize=memtag-stack -O0 -emit-llvm`.
- **Rationale**: The patch modifies `shouldEmitLifetimeMarkers` in Clang to return true when `SanitizerKind::MemtagStack` is enabled, even at `-O0`. Testing a simple local variable ensures that Clang correctly emits `llvm.lifetime.start` and `llvm.lifetime.end` for it.
- **Expected Issue**: The lifetime markers for the local variable would not be emitted in the unoptimized IR, leading to missed use-after-scope detection by the memtag-stack sanitizer.

### Strategy 3: Clang: Emit lifetime markers for materialized temporary expressions at -O0

- **Target**: A C++ source file with a materialized temporary expression (e.g., `const int &x = condition ? 1 : 2;`), compiled with `-fsanitize=memtag-stack -O0 -emit-llvm`.
- **Rationale**: The patch modifies `EmitMaterializeTemporaryExpr` in Clang to not skip lifetime markers for conditional branches when `SanitizerKind::MemtagStack` is enabled. Testing a materialized temporary expression (e.g., binding a temporary to a reference in a conditional) ensures these markers are emitted.
- **Expected Issue**: The lifetime markers for the temporary expression would be skipped, meaning the sanitizer would not track the scope of the temporary, potentially missing use-after-scope bugs.

### Strategy 4: InstCombine: Preserve lifetime markers with sanitize_memtag and memory accesses

- **Target**: An LLVM IR test case for InstCombine containing a function with the `sanitize_memtag` attribute, a `lifetime.start`, some memory accesses (loads/stores) to the alloca, and a `lifetime.end`.
- **Rationale**: While the patch specifically targets trivially empty ranges, it's important to ensure that non-empty lifetime ranges (with memory accesses in between) are also preserved and not incorrectly optimized away due to any unintended side effects of the InstCombine changes.
- **Expected Issue**: The lifetime markers would be incorrectly removed by InstCombine, as it might not recognize the `sanitize_memtag` attribute if it's not properly propagated or checked in all relevant optimization paths.
