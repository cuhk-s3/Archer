## Test Strategies

### Strategy 1: Source Alloca Smaller Than Destination Alloca

- **Target**: MemCpyOptimizer stack move optimization
- **Rationale**: The fix checks if `SrcSize != DestSize` and if so, it checks if both are fixed size. If they are, it proceeds. Later, it sets the allocated type of `SrcAlloca` to `DestAlloca`'s type if `DestSize > SrcSize`. However, if `SrcPtr` is offset into `SrcAlloca`, `SrcSize` might be larger than `DestSize`, but the actual copied region is smaller. Wait, the code checks `Size != *DestSize`, so the copied size must be exactly `DestSize`. If `SrcSize` is smaller than `DestSize`, `Size` (which is `DestSize`) would be larger than `SrcSize`, which means the memcpy reads out of bounds of `SrcAlloca`. This is UB, but if it happens, the compiler might still perform the optimization. If `DestSize > SrcSize`, it changes `SrcAlloca`'s type to `DestAlloca`'s type, which increases its size. This seems safe. But what if `SrcSize` is larger than `DestSize`, and `SrcOffset` is non-zero? Then `SrcAlloca` is large enough. The issue might be with `SrcLoc` size.
- **Expected Issue**: The compiler will incorrectly optimize the memcpy, leading to a miscompilation where the destination alloca is replaced by the source alloca, but the source alloca is smaller than the destination alloca, causing out-of-bounds accesses.

### Strategy 2: ModRef Check with Offset Source Pointer

- **Target**: MemCpyOptimizer stack move optimization ModRef checks
- **Rationale**: The fix changes `SrcLoc` to use `SrcPtr` instead of `SrcAlloca` with `Size`. The `SrcModRefCallback` checks if any instruction modifies `SrcLoc`. However, if `SrcPtr` is an offset into `SrcAlloca`, `SrcLoc` correctly represents the copied region. But what about `DestModRefCallback`? It checks `DestLoc` which is `DestAlloca` with `*DestSize`. If `DestPtr` is an offset into `DestAlloca`, `DestLoc` still covers the whole `DestAlloca`. The issue might be that `SrcLoc` is `MemoryLocation(SrcPtr, LocationSize::precise(Size))`. If there is a write to `SrcAlloca` outside of `[SrcPtr, SrcPtr + Size)`, it won't be detected by `SrcModRefCallback`. Is this safe? Yes, because the memcpy only reads `[SrcPtr, SrcPtr + Size)`. But wait, `DestAlloca` is replaced by `SrcAlloca` (with an offset). Any uses of `DestAlloca` will now access `SrcAlloca` at the offset. If there was a write to `SrcAlloca` outside the copied region, it might alias with the new uses of `DestAlloca`? No, `DestAlloca` uses are within `[0, DestSize)`. After replacement, they access `SrcAlloca` at `[SrcOffset, SrcOffset + DestSize)`. This is exactly `[SrcPtr, SrcPtr + Size)`. So writes outside this region in `SrcAlloca` don't alias with `DestAlloca` uses. What if `DestAlloca` uses access out of bounds? That's UB.
- **Expected Issue**: The compiler will incorrectly optimize the memcpy, leading to a miscompilation where a store to the destination alloca is not detected as a clobber, causing incorrect elimination of the memcpy.

### Strategy 3: Capture Tracking with Offset Source Pointer

- **Target**: MemCpyOptimizer stack move optimization capture tracking
- **Rationale**: The fix uses `CaptureTrackingWithModRef` on `SrcAlloca`. It checks if `SrcAlloca` is captured. If `SrcPtr` is an offset into `SrcAlloca`, `SrcAlloca` might not be captured, but what if `SrcPtr` is captured? `CaptureTrackingWithModRef` tracks uses of `SrcAlloca`. Since `SrcPtr` is derived from `SrcAlloca`, uses of `SrcPtr` will be visited. So this seems correct. What if `SrcPtr` is passed to a function that captures it? `DetermineUseCaptureKind` will see it.
- **Expected Issue**: The compiler will incorrectly optimize the memcpy, leading to a miscompilation where the destination alloca is replaced by the source alloca, but the source alloca is captured.

### Strategy 4: Alignment with Offset Source Pointer

- **Target**: MemCpyOptimizer stack move optimization alignment check
- **Rationale**: The fix checks `(*SrcOffset - *DestOffset) % DestAlloca->getAlign().value() != 0`. If this is true, it bails out. This ensures that the offset difference preserves the alignment of `DestAlloca`. However, `SrcAlloca`'s alignment is updated to `std::max(SrcAlloca->getAlign(), DestAlloca->getAlign())`. If `SrcAlloca`'s alignment is increased, does the offset `SrcOffset` still preserve the alignment? For example, if `DestAlloca` has alignment 8, `DestOffset` is 0. `SrcOffset` is 8. `(8 - 0) % 8 == 0`. `SrcAlloca` has alignment 4. It is increased to 8. The new pointer is `SrcAlloca + 8`. Since `SrcAlloca` is aligned to 8, `SrcAlloca + 8` is aligned to 8. This matches `DestAlloca`'s alignment. What if `DestAlloca` has alignment 8, `DestOffset` is 0. `SrcOffset` is 16. `(16 - 0) % 8 == 0`. `SrcAlloca` has alignment 16. It is increased to 16 (max of 16 and 8). `SrcAlloca + 16` is aligned to 16, which is >= 8. This seems correct. What if `DestOffset` is non-zero? The code checks `*DestOffset != 0` and bails out. So `DestOffset` is always 0.
- **Expected Issue**: The compiler will incorrectly optimize the memcpy, leading to a miscompilation where the destination alloca is replaced by the source alloca, but the alignment of the new uses is incorrect.

### Strategy 5: Lifetime Markers with Offset Source Pointer

- **Target**: MemCpyOptimizer stack move optimization lifetime markers
- **Rationale**: The fix removes lifetime markers for `DestAlloca`. It does not modify lifetime markers for `SrcAlloca`. If `SrcAlloca` has lifetime markers, they apply to the whole `SrcAlloca`. Since `DestAlloca` is replaced by `SrcAlloca + SrcOffset`, the uses of `DestAlloca` will now be uses of `SrcAlloca`. If `SrcAlloca`'s lifetime ends before the uses of `DestAlloca`, this would be a use-after-lifetime. The optimization checks if `SrcAlloca` is valid for the whole function? No, it checks `PDT->dominates(Load, UI)` for ModRef. But it doesn't check if `SrcAlloca`'s lifetime covers `DestAlloca`'s uses. Wait, `SrcAlloca` and `DestAlloca` are allocas. Their lifetime is the whole function unless there are lifetime intrinsics. The code collects `LifetimeMarkers` for `DestAlloca` and deletes them. It does not check `SrcAlloca`'s lifetime markers! If `SrcAlloca` has a `lifetime.end` before a use of `DestAlloca`, the use will be invalid. Wait, `SrcModRefCallback` checks `BAA.getModRefInfo(UI, SrcLoc)`. A `lifetime.end` is a Mod. So if there is a `lifetime.end` for `SrcAlloca`, it will be seen as a Mod. If `DestAlloca` has a Ref after it, `isModSet(Res) && isRefSet(DestModRef)` will be true, and it will return false! So it is safe.
- **Expected Issue**: The compiler will incorrectly optimize the memcpy, leading to a miscompilation where the destination alloca is replaced by the source alloca, but the source alloca's lifetime is ended prematurely.
