## Test Strategies

### Strategy 1: Test no-nan inference from result to source

- **Target**: Instructions where a no-nan result is demanded, such as `fadd`, `fmul`, or calls with `nnan` return attributes.
- **Rationale**: The patch description mentions "fixes missed flag inference in some cases, due to not inferring no-nan result implies no-nan source." This suggests that the previous implementation might have missed this inference, or the new implementation might incorrectly apply it in cases where it shouldn't. We should test cases where a no-nan result is demanded, and check if the source is correctly inferred as no-nan, and also test cases where this inference might be invalid.
- **Expected Issue**: The optimization might incorrectly assume that the source of a no-nan result is also no-nan, leading to incorrect simplifications or miscompilations when the source can actually be NaN.

### Strategy 2: Test explicit nofpclass attribute handling

- **Target**: Instructions with explicit `nofpclass` attributes, such as calls or arguments.
- **Rationale**: The patch description mentions "Also start treating explicit nofpclass attributes as a leaf value, like a constant or argument." This means that when an instruction has a `nofpclass` attribute, the analysis might stop recursing and just use the attribute's information. We should test cases where an instruction has a `nofpclass` attribute and check if the analysis correctly uses this information without recursing further, and also test cases where this might lead to incorrect results if the attribute is not handled correctly.
- **Expected Issue**: The optimization might incorrectly treat an explicit `nofpclass` attribute as a leaf value, leading to missed simplifications or incorrect assumptions about the value's floating-point class.

### Strategy 3: Test demanded mask application at recursion limit

- **Target**: Deeply nested floating-point instructions that reach the `MaxAnalysisRecursionDepth`.
- **Rationale**: The patch moves the check for `Depth == MaxAnalysisRecursionDepth` after the handling of `CallBase` and `nofpclass` attributes, and adds `Known.knownNot(~DemandedMask);` before returning `false`. This means that even if the recursion limit is reached, the analysis will still apply the demanded mask to the known floating-point classes. We should test cases where the recursion limit is reached and check if the demanded mask is correctly applied, and also test cases where this might lead to incorrect results if the mask is not applied correctly.
- **Expected Issue**: The optimization might incorrectly apply the demanded mask at the recursion limit, leading to incorrect simplifications or miscompilations when the recursion depth is reached.

### Strategy 4: Test invalid no-nan inference

- **Target**: Instructions like `select` or `phi` where a no-nan result does not necessarily imply that all sources are no-nan.
- **Rationale**: While the patch aims to fix missed flag inference, it's important to ensure that the inference is only applied when it's valid. For example, a `select` instruction might have a no-nan result even if one of its sources is NaN, depending on the condition. We should test cases where a no-nan result does not necessarily imply a no-nan source, and check if the optimization correctly avoids making this invalid inference.
- **Expected Issue**: The optimization might incorrectly infer no-nan source from a no-nan result for instructions where this implication does not hold, leading to incorrect simplifications.

### Strategy 5: Test interaction between nofpclass and recursion limit

- **Target**: Deeply nested instructions with `nofpclass` attributes that reach the `MaxAnalysisRecursionDepth`.
- **Rationale**: The patch changes the order of handling `nofpclass` attributes and the recursion limit. We should test cases where an instruction has a `nofpclass` attribute and is at the recursion limit, and check if the optimization correctly handles this interaction.
- **Expected Issue**: The optimization might incorrectly handle the interaction between `nofpclass` attributes and the recursion limit, leading to missed simplifications or incorrect assumptions.
