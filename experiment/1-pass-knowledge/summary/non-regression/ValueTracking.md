# Subsystem Knowledge for ValueTracking
## Elements Frequently Missed

*   **Poison-Generating Flags (PGFs) on Specific Instructions**: The analysis frequently misses or improperly evaluates PGFs (like `samesign`, `nsw`, `nuw`) when they are attached to instructions like `phi` nodes or when comparing two seemingly inverse `icmp` instructions. It often assumes safety based on operands or incoming values while ignoring the flags on the instruction itself.
*   **Fast-Math Flags (FMFs) Mismatches**: The presence or absence of FMFs (like `nsz`, `nnan`, `ninf`) is often overlooked when combining instructions. The analysis frequently extracts relaxed semantics from one instruction (e.g., an `fcmp` with `nsz`) and incorrectly applies them to a dependent instruction (e.g., a `select` without `nsz`).
*   **Floating-Point Edge Cases (Signed Zeros and NaNs)**: The strict IEEE-754 semantics regarding signed zeros (`-0.0` vs `+0.0`) and NaN generation are frequently missed. This includes failing to account for `-0.0` yielding negative infinity during division, mismatched zero signs in min/max matching, and the cross-operand `0.0 * Infinity = NaN` case.
*   **Poison Elements in Vector Constants**: When performing algebraic simplifications (such as negations represented as subtraction from zero), the analysis often misses that the zero vector might contain `poison` elements (e.g., `<0, poison>`).
*   **Type-Specific Overflow in Range Calculations**: The analysis misses the unique overflow behavior of 1-bit (`i1`) integers. When calculating exclusive upper bounds by adding 1 to the maximum value, it fails to account for the immediate wrap-around to the minimum value in 1-bit types.
*   **Conditional Shielding of Undefined Behavior/Poison**: The analysis frequently misses the fact that `select` instructions act as conditional shields, preventing the evaluation or propagation of `poison` or Undefined Behavior (UB) from one of their branches.

## Patterns Not Well Handled

### Pattern 1: Unconditional Propagation of Poison via Select Transformations
The optimization pass struggles with transformations that replace a conditional `select` instruction with unconditional operations or intrinsics. When a `select` guards against a poison value (e.g., from an overflowing `sub nsw`, a flagged `icmp samesign`, or a vector containing poison elements), transforming it into an unconditional `smin`/`smax` intrinsic, a logical `xor`, or swapping its operands exposes the poison value unconditionally. The analysis fails to recognize that the original `select` safely shielded the poison, resulting in optimized code that is "more poisonous" than the original.

### Pattern 2: Over-Aggressive Floating-Point Canonicalization and Deduction
The pass does not well handle the deduction of floating-point properties when attempting to canonicalize patterns into min/max operations or fold comparisons. It frequently assumes that if individual operands meet certain criteria (e.g., "never infinity" and "never zero"), their combined operation (like multiplication) also adheres to safe bounds, ignoring cross-operand edge cases like `0 * Inf = NaN`. Furthermore, it aggressively matches `fcmp` and `select` pairs into `minnum`/`maxnum` intrinsics or non-strict inequalities (`ole`/`oge`) without strictly verifying that both instructions share the necessary Fast-Math Flags (like `nsz`), leading to incorrect sign flips for zero or incorrect infinity signs.

### Pattern 3: Unsafe Speculative Operand Substitution
The analysis mishandles operand substitution in speculatively (unconditionally) executed instructions based on downstream conditional equivalence. When a variable is checked against a constant (e.g., a pointer against `null`, or a divisor against `0`) in a conditional branch or `select`, the optimizer attempts to replace the variable with the constant in the preceding unconditional instruction. The pass fails to verify if the instruction's safety relies on the original variable's properties, inadvertently introducing immediate Undefined Behavior (like null pointer dereferences or division by zero) into the unconditional execution path.

### Pattern 4: Incomplete Flag Analysis on Non-Standard Instructions
The pass exhibits a flawed pattern of analyzing the safety of values by looking at their sources (operands or incoming edges) while ignoring the attributes of the instruction producing the value. For example, when analyzing `phi` nodes for poison safety, it checks if the incoming values are poison-free but completely ignores poison-generating Fast-Math Flags attached to the `phi` node itself. Similarly, it assumes two comparisons are exact logical inverses based purely on their predicates and operands, failing to check if one possesses a poison-generating flag that the other lacks.

### Pattern 5: Boundary Wrap-Arounds in Constant Range Calculations
The pass does not well handle constant range calculations for extreme boundary types, specifically 1-bit integers (`i1`). The high-level pattern of computing an exclusive upper bound by incrementing the maximum possible value breaks down for `i1`, where `1 + 1` overflows to `0`. The analysis fails to catch this type-specific wrap-around, leading to the creation of empty ranges `[0, 0)`. This causes the compiler to erroneously assume the result is unreachable or `poison`, aggressively folding away valid intrinsic calls like `abs`, `ctlz`, or `cttz`.
