# Subsystem Knowledge for DeadStoreElimination
## Elements Frequently Missed

*   **Pointer Attributes on Offset Pointers**: Attributes that guarantee memory safety or alignment (such as `dereferenceable`, `dereferenceable_or_null`, and `align`) are frequently mishandled when a pointer is modified. DSE misses the fact that these attributes are tied to the specific original pointer and cannot be safely preserved or blindly copied when the pointer is advanced by an offset (e.g., via `getelementptr`).
*   **Attribute Interactions and Overrides**: The interaction between conflicting or modifying parameter attributes, specifically `byval` and `initializes`. DSE frequently misses that `byval` creates a private local copy for the callee, which fundamentally changes the target memory of the `initializes` attribute from the caller's original memory to the callee's copy.
*   **Ephemeral Values and `llvm.assume` Uses**: Instructions that compute values exclusively used by assumption intrinsics (like `llvm.assume`). DSE and its underlying capture tracking analysis frequently miss that these ephemeral instructions still exist in the IR during the optimization phase and can perform memory accesses or pointer captures that must be respected until they are actually deleted.

## Patterns Not Well Handled

### Pattern 1: Partial Overwrite and Memory Intrinsic Shortening
When a memory intrinsic (`llvm.memset`, `llvm.memcpy`, `llvm.memmove`) is immediately followed by a store that overwrites its prefix, DSE attempts to optimize the intrinsic by shortening its length and advancing its base pointer. This pattern is not well handled because the transformation logic fails to strip or recalculate the pointer attributes for the newly created offset pointer. By incorrectly preserving attributes like `dereferenceable(N)` or `align M` on the advanced pointer, the compiler over-promises memory bounds and alignment guarantees, leading to undefined behavior and miscompilations.

### Pattern 2: Cross-Function Memory Initialization with Value Copies
This pattern involves a caller storing a value to a local allocation and then passing that memory to a callee function whose parameter is marked with both `byval` and `initializes`. DSE is designed to eliminate stores in the caller if it proves the callee will overwrite the same memory. However, this pattern is mishandled because DSE fails to account for the copy semantics of `byval`. It erroneously applies the `initializes` guarantee to the caller's original memory rather than the callee's private copy, resulting in the incorrect elimination of the caller's store and leaving the original memory uninitialized.

### Pattern 3: Pointer Capture Tracking with Ephemeral Instructions
When a pointer is captured or accessed by instructions whose results are only used by `llvm.assume` (ephemeral instructions), DSE relies on pointer capture tracking to determine if the memory is strictly local. This pattern is not well handled because the analysis prematurely ignores these ephemeral instructions, assuming they do not affect observable behavior and will be removed. Consequently, DSE is falsely informed that the pointer does not escape and aggressively deletes stores to that memory. Because the ephemeral instructions are still present in the IR during DSE, they end up reading uninitialized or stale memory, causing incorrect runtime behavior.
