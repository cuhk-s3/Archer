# Subsystem Knowledge for AliasAnalysis
## Elements Frequently Missed

* **`nsw` (No-Signed-Wrap) Flags in Transformed Expressions**: When decomposing and simplifying `getelementptr` (GEP) offsets, the analysis frequently misses the need to strip or recalculate `nsw` flags. Negating a scale or merging multiple index expressions can introduce signed overflow (e.g., multiplying `INT_MIN` by `-1`), making the original `nsw` assumptions invalid.
* **Integer Overflow in Scaled Indices**: The analysis frequently misses the possibility of wrapping math when evaluating variable indices. It incorrectly assumes that a strictly non-zero index multiplied by an element size (scale) will always produce a non-zero offset, ignoring that the multiplication can wrap around to exactly zero if the GEP is not marked `inbounds`.
* **Directionality of Partial Alias Offsets**: The analysis misses the asymmetric, direction-dependent nature of partial alias offsets. When swapping pointer arguments for recursive queries (such as evaluating `phi` or `select` incoming values), it fails to negate or adjust the resulting offset, returning an inverted value to downstream passes.
* **Pointer Captures in Ephemeral Instructions**: Capture tracking frequently misses pointer escapes that occur within "ephemeral" instructions (instructions whose results are only used by assumption intrinsics like `llvm.assume`). Because these instructions are slated for eventual removal, the analysis prematurely ignores them, missing the fact that they can still capture pointers while present in the IR.
* **Stale Dominator Tree State**: Alias Analysis frequently misses the fact that auxiliary data structures, specifically the Dominator Tree, may be out of sync with the actual Control Flow Graph (CFG). When queried by passes that actively modify the CFG, AA relies on this stale dominance information to make incorrect contextual aliasing deductions.

## Patterns Not Well Handled

### Pattern 1: Unsafe Algebraic Simplifications in GEP Offset Decomposition
When Alias Analysis evaluates `getelementptr` (GEP) instructions, it often decomposes their indices into linear expressions (e.g., `Scale * Variable + Constant`). To compare two pointers, the analysis attempts to simplify these expressions by merging terms (e.g., combining `A * x` and `B * x` into `(A + B) * x`) or subtracting offsets. However, this pattern is not well handled regarding the semantics of `nsw` (no signed wrap) flags. The analysis incorrectly preserves `nsw` flags on negated scales or merged expressions. Because the new combined or negated expressions can undergo signed overflow even if the original components did not, preserving the flag introduces false assumptions about value ranges. This leads to incorrect `NoAlias` deductions when overflow actually occurs.

### Pattern 2: Ignoring Integer Wrapping in Non-Zero Index Deductions
Alias Analysis sometimes relies on value tracking, loop bounds, or range metadata to prove that a variable index used in a GEP is strictly non-zero. The high-level pattern assumes that a non-zero index guarantees a non-zero final memory offset. This logic is flawed and poorly handled because it does not account for integer wrapping. If the GEP lacks the `inbounds` keyword, multiplying a large non-zero index by the type's scale (element size) can wrap around the integer bitwidth to produce a final offset of exactly zero. Ignoring this wrapping behavior causes the analysis to incorrectly conclude that the base pointer and the GEP pointer cannot overlap, resulting in invalid memory optimizations.

### Pattern 3: Asymmetric Result Handling in Recursive Queries
To determine aliasing for pointers derived from `phi` or `select` instructions, Alias Analysis performs recursive queries on the incoming values or operands. A common pattern is to swap the order of the pointer arguments during these recursive calls to simplify the internal implementation. However, this pattern fails to account for asymmetric alias results, specifically Partial Alias offsets, which represent the distance from the first pointer to the second. When the arguments are swapped, the returned offset is directionally incorrect (e.g., negated). Because the analysis does not correct the offset after the recursive call returns, downstream passes like Global Value Numbering (GVN) use the wrong offset to perform invalid memory forwarding.

### Pattern 4: Premature Elision of Ephemeral Values in Capture Tracking
Pointer capture tracking determines whether a pointer escapes the current function, which is critical for passes like Dead Store Elimination (DSE). The analysis employs a pattern of ignoring "ephemeral" values—instructions whose results are only used by intrinsics like `llvm.assume`—under the premise that they do not affect observable program semantics and will eventually be removed. However, this pattern is not well handled when the ephemeral instructions themselves capture the pointer (e.g., by storing it or passing it to a function). Ignoring them prematurely leads to a false conclusion that the pointer does not escape, prompting DSE to aggressively remove necessary stores while the capturing instructions still exist and execute in the IR.

### Pattern 5: Querying Alias Analysis with Stale Auxiliary Data
Memory optimization passes (such as redundant load elimination) often query Alias Analysis while simultaneously modifying the Control Flow Graph (CFG) through transformations like jump threading. Alias Analysis relies heavily on auxiliary data structures, particularly the Dominator Tree, to make advanced contextual deductions about memory clobbers. The pattern of querying AA without ensuring the Dominator Tree is fully updated is poorly handled. It leads to flawed dominance evaluations where AA mistakenly deduces that an intervening memory write does not alias or clobber a load's target address. Trusting this incorrect result, the compiler eliminates valid loads and replaces them with stale values.
