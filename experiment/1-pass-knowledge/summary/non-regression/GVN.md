# Subsystem Knowledge for GVN
## Elements Frequently Missed

*   **Call Return Attributes (`nonnull`, `range`, `align`, `dereferenceable`, etc.)**: During equivalence checking and value replacement, GVN frequently misses the differences in return attributes between identical function calls. It fails to recognize that these attributes dictate the validity of the returned value and can introduce `poison` if violated.
*   **Poison Semantics in Dominating Instructions**: GVN misses the context-specific safety of `poison` values. A dominating instruction might safely evaluate to `poison` because the control flow masks it, but propagating this instruction to replace a dominated, less-restrictive instruction introduces undefined behavior or `poison` into new, unsafe paths.
*   **Instruction Lifecycle Notifications**: GVN misses tracking the deletion or modification of instructions performed by underlying utility functions (e.g., PHI deduplication). It assumes the IR remains static unless modified directly by the GVN pass itself.
*   **Analysis Cache Invalidation**: Internal caches (such as Value Numbering or Memory Dependence Analysis) keyed by instruction memory addresses are frequently missed during cleanup. GVN fails to invalidate these cache entries when the corresponding instructions are removed.

## High-Level Patterns Not Well Handled

### Pattern 1: Improper Attribute Reconciliation During Value Replacement
When GVN or CSE identifies two identical function calls (or intrinsics) where one dominates the other, it routinely replaces the dominated call with the dominating one to eliminate redundancy. However, this pattern is poorly handled when the dominating call possesses more restrictive return attributes (such as `nonnull` or `range`) than the dominated call. GVN fails to intersect or strip these incompatible attributes during the replacement. Consequently, if the inputs violate the restrictive attributes of the dominating call, it evaluates to `poison`. While this `poison` might be safely ignored in the dominating context, replacing the relaxed, dominated call forces the `poison` value into paths where a well-defined value was expected, leading to miscompilation and poison propagation.

### Pattern 2: Stale Analysis Caches and the ABA Problem via Silent Instruction Deletion
GVN relies heavily on internal analysis caches to store value numbers and memory dependencies, often using the raw memory addresses of `Instruction` pointers as keys. A high-level pattern that is not well handled occurs when auxiliary simplification utilities (like PHI node deduplication) silently delete instructions and free their memory without notifying the GVN pass to update its caches. This creates an ABA problem: if a newly created instruction is allocated at the exact same memory address as a previously deleted instruction, GVN will look up the new instruction's address, find the stale cache entry of the deleted instruction, and incorrectly apply it. This leads to bizarre and invalid transformations, such as replacing a newly formed `load` instruction with an unrelated `phi` value simply because they shared the same memory address in the compiler's heap.
