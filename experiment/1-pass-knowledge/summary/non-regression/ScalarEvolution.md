# Subsystem Knowledge for ScalarEvolution
## Elements Frequently Missed

*   **Poison-Generating and Relational Flags (`nsw`, `nuw`, `samesign`)**: These flags are frequently left intact when instructions are hoisted, reused in new contexts, or mathematically transformed (e.g., truncated or shifted). The optimization pass misses the fact that these flags are context-dependent and value-dependent, leading to the generation of poison values or incorrect logical deductions when the new context violates the original flag's guarantees.
*   **Control-Flow Guards for Faulting Operations**: The presence of conditional branches or loop checks that protect potentially faulting operations (like integer division) is often missed during SCEV expansion. The analysis models the math but forgets the execution constraints, leading to unconditional execution of unsafe operations.
*   **Hidden Poison in Operand Trees**: When identifying existing instructions to reuse for SCEV expansion, the pass frequently misses poison values or poison-generating operations buried deeper in the instruction's operand tree or within mathematically folded operations (e.g., `mul %poison, 0`).
*   **Sign Bits and Arithmetic Properties During Bitwise Manipulations**: The pass misses the distinction between logical and arithmetic shifts when factoring negative constants, and ignores the alteration of sign bits when truncating integers. This results in unintended large positive constants or inverted sign semantics.
*   **Start Values in Zero-Stride Recurrences**: When analyzing loop exit limits for zero-stride induction variables, the pass frequently misses checking whether the initial value already satisfies the exit condition, prematurely assuming the loop is infinite and invokes Undefined Behavior (UB).
*   **Internal SCEV Cache References**: During cache invalidation triggered by CFG modifications, the pass misses deep, internal references to SCEV expressions (such as those used in Backedge Taken Counts), leaving stale data attached to newly non-trivial LCSSA phi nodes.

## Patterns Not Well Handled

### Pattern 1: Unsafe Instruction Reuse and Hoisting
The optimization pass frequently assumes that if an existing instruction is mathematically equivalent to a desired SCEV expression and dominates the target insertion point, it is safe to reuse. This pattern is poorly handled because mathematical equivalence in SCEV does not equate to execution safety in LLVM IR. The pass fails to strip context-specific flags (like `nsw`/`nuw`) from the reused instruction and fails to detect hidden poison in its operand tree or folded operations. Consequently, reusing these instructions introduces poison into previously safe code paths, which can propagate to memory accesses or branches and cause miscompilations.

### Pattern 2: Invalid Preservation of Flags During Value and Type Transformations
When the compiler transforms values to simplify expressions or match types—such as truncating operands for condition implication or shifting constants to factor out powers of two—it routinely carries over the original optimization flags (`samesign`, `nsw`, `nuw`). This pattern is not well handled because these transformations fundamentally alter the bitwise properties of the values. Truncation can change a positive integer to a negative one (invalidating `samesign`), and logical right shifts on negative numbers create massive positive numbers (invalidating `nsw`). Preserving these flags on altered values leads to aggressive, incorrect deductions and integer overflows.

### Pattern 3: Loss of Control-Flow Guards During SCEV Expansion
SCEV excels at modeling complex, conditionally executed loop logic into sequential mathematical expressions (like sequential min/max). However, the expansion of these models back into LLVM IR is poorly handled when the original logic contains potentially faulting operations (e.g., division by zero). The expander evaluates the operands unconditionally, stripping away the original protective control-flow guards. This hoists faulting operations into unconditional execution paths, introducing Undefined Behavior into programs that were originally safe.

### Pattern 4: Premature UB Assumptions in Degenerate Loops
The loop analysis framework struggles with degenerate loop conditions, specifically zero-stride recurrences. When the pass observes an induction variable with a step of zero, it correctly deduces that the value will not change. However, it handles this pattern poorly by immediately assuming that the loop must be infinite (and therefore UB in C/C++) without first evaluating the recurrence's start value against the exit condition. If the start value already satisfies the exit condition, the loop terminates immediately. The false UB assumption leads the compiler to aggressively and incorrectly eliminate the loop or subsequent control flow.

### Pattern 5: Shallow Cache Invalidation on CFG Mutations
Loop transformations like peeling, unrolling, or vectorization structurally alter the Control Flow Graph, often adding new predecessors to loop exit blocks. This changes trivial LCSSA phi nodes into non-trivial ones. The pattern of invalidating SCEV caches after such CFG mutations is not well handled because the invalidation mechanism is too shallow; it only traverses standard IR use-def chains. It fails to purge internal SCEV references (like loop-defined `AddRec` expressions used in trip counts), causing the compiler to retain and rely on stale, invalid expressions for subsequent optimizations.
