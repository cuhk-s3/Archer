# Subsystem Knowledge for SimplifyCFG
## Elements Frequently Missed

* **Context-Dependent Metadata and Attributes**: Metadata such as `!dereferenceable` or `!align` attached to instructions within conditionally executed blocks. The optimization pass frequently misses that these guarantees are only valid under the specific branch condition and cannot be universally applied or preserved when instructions are hoisted to an unconditional context.
* **Alignment Discrepancies Between Memory Accesses**: The specific alignment values of memory operations. When using a dominating memory access to prove the safety of a conditionally executed memory access, the pass frequently misses checking whether the conditional access has a stricter (higher) alignment requirement than the dominating access.
* **Target Insertion Point Context for Safety Checks**: The pass frequently evaluates the safety of an instruction (e.g., whether it is safe to speculatively execute) based on its *original* basic block rather than the *new* dominating block where it will be inserted. This causes the pass to miss the fact that local safety properties are lost upon moving the instruction.

## Patterns Not Well Handled

### Pattern 1: Speculative Hoisting of Memory Accesses Relying on Local Metadata
This pattern occurs when SimplifyCFG attempts to flatten control flow (e.g., folding a PHI node into a `select`) by hoisting instructions from a conditional block into a dominating predecessor block. Inside the conditional block, a dependent instruction (like a memory load) relies on local safety guarantees established by a preceding instruction's metadata (such as `!dereferenceable`). The optimization pass evaluates the safety of the dependent instruction in its original context and incorrectly concludes it is safe to hoist. However, when the instructions are moved to the unconditional block, the local metadata is either discarded or no longer guarantees safety for all execution paths. This results in the dependent instruction being executed unconditionally without the required safety guarantees, potentially leading to undefined behavior such as segmentation faults.

### Pattern 2: Store Speculation with Asymmetric Alignment Requirements
This pattern involves the speculation of a conditionally executed store instruction into a preceding, unconditionally executed basic block. The optimization pass justifies this speculation by identifying a prior memory access (load or store) to the same pointer in the dominating block, which proves the pointer is dereferenceable. However, the pass fails to properly handle cases where the conditional store has a higher alignment requirement than the unconditional memory access. By hoisting the store and retaining its strict alignment in an unconditional context, the compiler introduces a high risk of alignment faults. The pointer is only guaranteed to meet the lower alignment of the preceding access, making the unmodified hoisted store unsafe.
