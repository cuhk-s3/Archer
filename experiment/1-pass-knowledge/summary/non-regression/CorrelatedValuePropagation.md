# Subsystem Knowledge for CorrelatedValuePropagation
## Elements Frequently Missed

* **`undef` Values in Operands (e.g., PHI nodes)**: The optimization pass frequently misses the deferred evaluation nature of `undef`. While value tracking optimistically assumes `undef` can take a specific value to satisfy a condition (like being non-negative) during analysis, replacing an operation with the `undef`-tainted operand allows later passes to evaluate `undef` to any arbitrary value, breaking the assumed condition.
* **The `is_int_min_poison` Flag in `llvm.abs`**: The pass misses the semantic implications of this flag being set to `false`. When `false`, taking the absolute value of `INT_MIN` is well-defined. Failing to account for this when `undef` is present leads to invalid simplifications that violate the intrinsic's strict output guarantees.
* **Strict Output Range Guarantees of Intrinsics**: The pass misses the fact that intrinsics like `llvm.abs` enforce a hard constraint on their output range (e.g., strictly non-negative or `INT_MIN`). Removing the intrinsic based on optimistic input analysis removes this hard guarantee, illegally expanding the set of possible values if the input contains `undef`.

## Patterns Not Well Handled

### Pattern 1: Optimistic Range Analysis Interacting with `undef` and Constraining Intrinsics
The optimization pass relies on value tracking and range analysis to determine if an intrinsic operation is redundant. When an operand (such as a PHI node) contains both a known constrained value (e.g., a positive constant) and an `undef` value, the analysis optimistically assumes the `undef` value will conform to the constrained value's properties. Based on this optimistic assumption, the pass concludes the intrinsic (e.g., `llvm.abs`) is a no-op and replaces it with the operand itself. This pattern is poorly handled because removing the intrinsic strips away the operation's inherent output constraints. Subsequent compiler passes are then free to evaluate the `undef` portion of the operand to a value outside the optimistically assumed range (e.g., an arbitrary negative number), resulting in a miscompilation.

### Pattern 2: Invalid Elision of `llvm.abs` with Well-Defined `INT_MIN`
When the `llvm.abs` intrinsic is configured with `is_int_min_poison` set to `false`, the operation guarantees that the result is either non-negative or exactly `INT_MIN`. The pass fails to handle the interaction between this specific configuration and `undef` operands. If the pass replaces the intrinsic with the operand itself, it inadvertently allows the `undef` value to be materialized later as *any* negative integer, not just `INT_MIN`. This is not well handled because the pass does not recognize that when `is_int_min_poison` is `false`, `undef` must be treated as a full, unconstrained range rather than being optimistically constrained. Replacing the intrinsic under these conditions illegally expands the possible result space.
