# Subsystem Knowledge for IR
## Elements Frequently Missed

*   **1-Bit Integer Arithmetic Boundaries**: The unique properties of 1-bit integers (where the signed range is strictly `[-1, 0]` and the bit pattern for `1` represents `-1`) are frequently missed. Optimizations often incorrectly apply general integer arithmetic rules, failing to account for the fact that operations like adding 1 to the maximum absolute value or multiplying by 1 will cause unexpected signed overflows in a 1-bit context.
*   **Poison-Generating Instruction Flags**: Flags that introduce poison under specific semantic violations (such as the `samesign` flag on `icmp`) are often overlooked during instruction hoisting or control-flow simplification. The compiler treats the instruction as safe to execute unconditionally, missing the fact that the flag can generate poison if the original guarding condition is bypassed.
*   **Inactive Lanes in Vector Selects**: When threading operations over `select` instructions, the compiler frequently misses the dynamic masking effect of the `select` condition. It evaluates inactive lanes (such as constant vectors containing zero or `undef`) independently, failing to recognize that undefined behavior (like division by zero) in an inactive lane should not pollute the entire operation with poison.
*   **Internal Representations of Same-Width Floating-Point Types**: The distinct internal bit layouts (exponent vs. mantissa bits) of different floating-point types sharing the same bit width (e.g., `bfloat` and `half`) are missed. The compiler incorrectly assumes that a `bitcast` between them is a semantic no-op that preserves the numerical value, rather than an operation that fundamentally alters the interpreted value.

## Patterns Not Well Handled

### Pattern 1: Eager Evaluation of Guarded Poison or Undefined Behavior
The optimization pass struggles with operations that can generate poison or trigger undefined behavior (UB) when they are nested inside conditionally executed contexts, such as `select` instructions or conditional branches. When attempting to simplify control flow or thread operations across a `select`, the compiler evaluates the operations unconditionally. If the operation involves a poison-generating flag (like `icmp samesign`) or an invalid operand in a masked vector lane (like division by zero), the compiler eagerly folds the result to `poison`. This pattern is poorly handled because the simplification logic fails to respect the original control dependencies, allowing poison to leak into the optimized program even when the dynamic execution path would have safely avoided the UB.

### Pattern 2: Generalization of Arithmetic and Range Rules to 1-Bit Integers
The compiler frequently mishandles 1-bit integers by applying generalized arithmetic optimizations and constant range calculations that are only safe for wider integer types. For example, it assumes that multiplying a value by `1` cannot cause a signed overflow, or that adding `1` to a maximum absolute value safely computes an upper bound. In the context of 1-bit integers, these assumptions break down entirely due to the restricted `[-1, 0]` signed range, leading to immediate wrap-arounds. This pattern is not well handled because the optimization passes lack specialized boundary checks for 1-bit types, resulting in incorrect flag inference (like erroneously attaching `nsw`) or the generation of empty, invalid constant ranges.

### Pattern 3: Bypassing Bitcasts in Floating-Point Conversion Sequences
The optimization pass does not well handle sequences involving a `bitcast` between two different floating-point types of the same width, followed by a floating-point extension or truncation. The compiler's logic is built on the flawed heuristic that a same-width `bitcast` does not change the underlying data, prompting it to eliminate the `bitcast` and apply the subsequent cast directly to the original source type. This pattern causes miscompilations because it ignores the fact that types like `bfloat` and `half` have different binary semantics. Bypassing the `bitcast` changes the numerical value that the subsequent `fpext` or `fptrunc` operates on, leading to incorrect downstream calculations and constant folding.
