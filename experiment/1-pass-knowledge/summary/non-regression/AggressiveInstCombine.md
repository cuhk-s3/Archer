# Subsystem Knowledge for AggressiveInstCombine
## Elements Frequently Missed

* **Native Pointer Index Types**: The optimization pass frequently misses the preservation of the target's native pointer index type (e.g., 64-bit integers) during offset arithmetic. By incorrectly instantiating accumulated offsets as smaller fixed-width integers (e.g., 32-bit), it inadvertently triggers sign-extension issues for large offsets, altering the target memory address.
* **Byte Size vs. Bit Width Distinction**: The pass fails to correctly distinguish between bit width and byte size when constructing memory location queries for Alias Analysis. Passing the bit width (e.g., 16) instead of the byte size (e.g., 2) results in an inaccurate memory footprint being analyzed.
* **True Base Pointers for Combined Operations**: When merging multiple memory operations, the pass frequently misses calculating the true, lowest base address of the newly combined operation. Instead, it mistakenly relies on the base pointer of a single constituent load (typically the one at the insertion point), which skews alias checks.
* **Intervening Memory Modifications (Stores)**: Due to malformed Alias Analysis queries (caused by incorrect base pointers and size units), the pass frequently misses intervening `store` instructions that modify a portion of the memory region being accessed by the combined loads.

## Patterns Not Well Handled

### Pattern 1: Consecutive Load Folding with Large Constant Offsets
This pattern involves a sequence of consecutive memory loads derived from a base pointer using large constant offsets (e.g., offsets large enough to have the sign bit set if represented as a 32-bit integer). The optimization pass attempts to fold these loads into a single, wider load by stripping and accumulating the offsets. However, this pattern is not well handled because the pass truncates the accumulated offset into a smaller fixed-width integer type during pointer reconstruction. When this truncated offset is applied to the new pointer arithmetic, it gets incorrectly sign-extended (e.g., turning a large positive offset into a negative offset). This causes the combined wide load to read from an entirely incorrect memory address, leading to a miscompilation.

### Pattern 2: Load Combining Across Intervening Stores
This pattern consists of multiple contiguous or overlapping loads that the compiler attempts to merge into a single wider load, with an intervening store instruction placed between them that modifies a portion of the loaded memory. This pattern is poorly handled because the pass constructs an invalid memory location query for the Alias Analysis framework to check for overlaps. Specifically, it uses the bit width of the combined load instead of its byte size, and it uses the memory address of the load at the insertion point rather than the true base address of the combined load. Consequently, the Alias Analysis incorrectly returns `NoAlias` for the intervening store. This flawed legality check allows the compiler to illegally hoist the combined load above the store, causing the program to read stale data instead of the newly stored value.
