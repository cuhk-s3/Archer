# Subsystem Knowledge for SLPVectorizer
## Elements Frequently Missed

*   **Poison vs. Undef Semantics**: The vectorizer frequently misses the strict semantic differences between `undef` and `poison`. It erroneously substitutes `undef` with `poison` when reusing vectors, or reuses splat masks containing `poison` in lanes that strictly require well-defined values, leading to invalid poison propagation.
*   **Multi-Use Scalar Constraints**: When demoting or truncating scalars, the vectorizer misses checking all external uses of the scalar. It optimizes based only on the immediate vectorized chain, breaking other instructions that rely on the original, wider bitwidth.
*   **Dynamic/Non-Constant Vector Indices**: The vectorizer misses conservative tracking for `insertelement` instructions with dynamic indices. Instead of assuming the insertion could affect any lane, it ignores the dynamic index, leading to incorrect poison/undef markings and dropped elements.
*   **Instruction Optimization Flags (e.g., `samesign`)**: The vectorizer misses dropping context-sensitive optimization flags when transforming instructions. For example, it retains the `samesign` flag on `icmp` instructions even after truncating the operands, which can alter sign bits and violate the flag's condition.
*   **64-bit Pointer Offsets**: During memory access analysis, the vectorizer misses the upper 32 bits of pointer offsets. It incorrectly truncates 64-bit pointer differences to 32 bits, masking large non-contiguous jumps and falsely identifying them as contiguous memory accesses.
*   **Bitwidth-Sensitive Opcodes**: The vectorizer misses the strict operand constraints of bitwidth-sensitive operations (like shifts) when performing type demotion on alternate opcode sequences. It shrinks the type based on one operation, causing out-of-bounds errors for the alternate operation.
*   **Short-Circuiting Semantics**: The vectorizer misses the poison-suppressing nature of scalar short-circuiting logic (e.g., `select` chains). It blindly converts these into eager vector reductions, which evaluate all lanes and illegally introduce `poison` into the final result.
*   **Identity Operation Constraints**: The vectorizer misses cross-compatibility checks when unifying different scalar identity operations (e.g., `xor X, 0` and `and Y, -1`). It assumes they are interchangeable and applies a single opcode across mixed constants, breaking the identity semantics for some lanes.
*   **Result vs. Operand Bitwidth Constraints**: The vectorizer misses the distinction between an instruction's output size and its required input size. For example, it incorrectly uses the 1-bit boolean result size of an `icmp` instruction as a valid constraint to aggressively truncate the `icmp`'s operands.

## Patterns Not Well Handled

### Pattern 1: Shuffle Mask Generation and Composition
The SLP vectorizer struggles significantly with generating and composing `shufflevector` masks, especially when dealing with complex or irregular vectorization trees. When combining vectors of different sizes or mixed-length input vectors, the vectorizer incorrectly calculates index offsets based on the size of the resulting mask or the first source vector, rather than the actual maximum vector factors. Furthermore, when reusing gathered scalars, the mask composition logic fails to correctly map indices across multiple lanes. The vectorizer also erroneously applies temporary submasks—intended strictly for cost-model estimation—as actual IR masks, and miscalculates slice sizes based on original scalar counts rather than the actual deduplicated gathered scalars. These flaws consistently result in malformed `shufflevector` instructions that extract the wrong elements or leave lanes poisoned.

### Pattern 2: Minimum Bitwidth Analysis and Type Demotion
The vectorizer's minimum bitwidth analysis frequently applies overly aggressive and unsafe type demotions. It fails to isolate the demotion safely within the vectorized tree, often truncating multi-use scalars without verifying if external uses can tolerate the narrower type. In alternate opcode vectorization, it applies a uniform type demotion to paired operations without recognizing that bitwidth-sensitive operations (like logical shifts) will break if their operands (like shift amounts) exceed the new, narrower bounds. Additionally, it conflates the bitwidth of an instruction's users (e.g., a 1-bit `icmp` result) with the permissible bitwidth of its operands, leading to invalid truncations that destroy data before comparisons.

### Pattern 3: Instruction Reuse and Poison Propagation
The optimization pass aggressively attempts to reuse existing vector instructions (like splats or extract bundles) to minimize instruction count, but it handles `poison` lanes poorly during this process. When matching existing `shufflevector` instructions for reuse, it ignores whether the existing mask contains `poison` in lanes that the new operation requires to be well-defined. Similarly, when processing bundles of `extractelement` instructions with unused/padded lanes, it incorrectly assigns `poison` as the source operand, breaking the logic needed to identify common source vectors. This pattern of unsafe reuse consistently injects `poison` into otherwise well-defined computation chains.

### Pattern 4: Boolean Logic and Reduction Tree Vectorization
The vectorizer does not well handle the transition from scalar boolean control flow to horizontal vector reductions. Scalar boolean chains (often represented by `select` instructions) inherently possess short-circuiting semantics that safely bypass `poison` or `undef` operands. The vectorizer groups these into eager vector reduction intrinsics (e.g., `llvm.vector.reduce.or`), which evaluate all lanes simultaneously. Because the vectorizer fails to insert necessary `freeze` instructions or account for mixed-position operands (where a value acts as both a poison-propagating condition and a poison-suppressed value), this pattern routinely breaks original program semantics by allowing suppressed `poison` to propagate to the final result.

### Pattern 5: Memory Access Widening and Offset Analysis
The vectorizer exhibits flawed logic when analyzing memory access patterns to form vector loads and stores. When calculating the distance between pointers to detect contiguous accesses, it performs unsafe 32-bit truncations on 64-bit offsets, deceiving itself into vectorizing widely separated memory locations. Additionally, when handling strided loads that require reordering, the vectorizer attempts to optimize by emitting widened, contiguous vector loads without generating the correct compensating shuffle masks. This not only places data in the wrong vector lanes but also risks unsafe memory reads that cross valid memory boundaries.
