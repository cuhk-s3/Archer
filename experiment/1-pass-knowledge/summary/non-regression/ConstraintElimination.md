# Subsystem Knowledge for ConstraintElimination
## Elements Frequently Missed

* **Poison-Generating Flags on Logical Operations**: The optimization pass frequently misses the implications of poison-generating flags (such as `disjoint` or `exact`) when simplifying operands. When an operand is replaced based on contextual or short-circuiting logic, the pass fails to drop these flags, inadvertently creating invalid operand combinations that produce `poison`.
* **Internal Coefficient Overflow Checks**: During the decomposition of arithmetic operations into linear expressions, the pass misses bounds checking for its internal integer representation (typically 64-bit signed integers). This allows internal calculations of coefficients and offsets to silently overflow or wrap around when dealing with large constants.
* **Bit-Width Discrepancies (Wide Integer Types)**: The pass misses the semantic differences between wide LLVM IR integer types (e.g., `i128`) and its internal fixed-width constraint representation. It fails to properly handle or bail out when decomposing types that exceed the internal bit-width, leading to silent truncation.
* **Poison-Safety Verification (`isGuaranteedNotToBePoison`)**: The pass frequently misses checking whether a value is guaranteed not to be poison before adding it to the constraint system. It assumes that the mathematical properties of intrinsics (like `umin`/`smax`) always hold, ignoring the fact that their inputs might be unobserved poison values generated by upstream `nuw`/`nsw` violations.

## Patterns Not Well Handled

### Pattern 1: Linear Decomposition of Wide Types and Large Constants
The constraint elimination pass attempts to model arithmetic instructions (additions, subtractions, multiplications) as a linear system of variables, coefficients, and constant offsets. However, this decomposition pattern is poorly handled when the original IR involves wide integer types (like `i128`) or extremely large constants. Because the pass relies on a fixed-width internal representation (usually 64-bit integers) to store these coefficients, the internal math can silently overflow, wrap around, or truncate. This results in a corrupted mathematical model that does not accurately reflect the original program's semantics, causing the compiler to deduce false relationships and incorrectly eliminate necessary bounds checks or branches.

### Pattern 2: In-Place Operand Replacement Violating Instruction Flags
The pass uses implied context (such as branch conditions or short-circuiting logic) to simplify individual operands of binary instructions. A high-level pattern that is not well handled is performing this operand replacement in-place while leaving the original instruction and its strict optimization flags intact. For example, replacing an operand of an `or disjoint` instruction with `true` based on the assumption that the other operand is `false` works logically, but if the other operand evaluates to `true` at runtime, the instruction evaluates `true OR true`. This violates the `disjoint` flag's requirement, turning a valid short-circuited path into one that generates `poison`. The pass fails to realize that context-driven operand simplification requires either dropping the strict flags or simplifying the entire instruction.

### Pattern 3: Unconditional Constraint Extraction from Potentially Poisonous Operations
When the pass encounters certain intrinsics, such as min/max operations (`umin`, `umax`, `smin`, `smax`), it eagerly and unconditionally extracts mathematical constraints (e.g., `result <= operand`) and adds them to its internal system. This pattern is flawed because it does not account for the propagation of `poison` values. If an operand to the intrinsic was generated by an instruction that violated a fast-math or wrap flag (like `add nuw`), it may be `poison`. If this poison is never observed in a way that triggers Undefined Behavior (UB), the program is still valid. However, by unconditionally adding the intrinsic's constraints, the pass implicitly assumes the inputs are well-defined, allowing it to deduce false properties about the original inputs (e.g., assuming a value must be non-negative). These false deductions are then used to incorrectly fold independent, valid conditions elsewhere in the function.
