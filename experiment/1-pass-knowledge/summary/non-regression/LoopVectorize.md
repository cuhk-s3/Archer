# Subsystem Knowledge for LoopVectorize
## Elements Frequently Missed

* **Outside-Loop Users (LCSSA)**: Users of loop-defined values that reside outside the loop are frequently missed or mishandled by uniformity analysis. The analysis often only checks inside-loop users, leading to incorrect lane extraction (e.g., extracting lane 0 instead of the last active lane for an exit value).
* **Poison-Generating Operations in Inactive Lanes**: Operations that produce `poison` (e.g., out-of-bounds shifts or invalid memory accesses) in lanes that would normally be unreachable or skipped in scalar execution are frequently missed. This oversight allows `poison` to silently infect vector lanes.
* **Dead Control Flow Paths in PHI Nodes**: Values originating from dead or unreachable control flow blocks within PHI nodes are often overlooked. When vector code generation falls back to extracting specific lanes (like the last lane), it may inadvertently extract and use garbage values from these dead paths.
* **Internal Cached Analysis References (e.g., SCEV)**: Internal or indirect references to cached analysis expressions (such as loop-defined expressions used in Backedge Taken Counts) are missed during cache invalidation when loop transformations alter the Control Flow Graph (CFG).
* **Analysis Discrepancies Between Compilation Phases**: The mismatch in strictness between advanced analyses (e.g., SCEV) used during the vectorization planning phase and weaker structural analyses used during the code generation phase is frequently missed, leading to conflicting and unsafe optimization decisions.

## Patterns Not Well Handled

### Pattern 1: Uniformity Misclassification for Outside Users
When an induction variable or loop-defined instruction has only uniform users inside the loop but is also used outside the loop (e.g., as an exit value via an LCSSA Phi node), the vectorizer's uniformity analysis incorrectly classifies the instruction as strictly uniform. Consequently, the vectorizer optimizes the value by extracting only the first lane (lane 0). However, the outside user semantically requires the value from the final loop iteration (the last active lane). This pattern is not well handled because the analysis fails to account for the distinct lane requirements of cross-iteration or outside-loop users, leading to miscompiled exit values.

### Pattern 2: Inconsistent Loop-Invariant Analysis Across Phases
The loop vectorizer relies on multiple phases (cost modeling/planning and code generation) that sometimes use different levels of analysis. During planning, advanced analyses like SCEV might prove a value (e.g., a stored value from a PHI node with dead paths) is loop-invariant, allowing the vectorizer to bypass predication and plan a uniform, unconditional store. However, during code generation, a stricter structural analysis might fail to recognize the invariance, treating the value as a varying vector. The code generator then attempts to extract the last lane to perform the store. If the last lane corresponds to a dead control flow path, it extracts and stores a garbage value. This pattern is poorly handled because the lack of synchronization between analysis phases leads to divergent and unsafe code generation.

### Pattern 3: Poison Propagation in Speculative Early-Exit Reductions
Vectorizing loops with early exits requires evaluating the exit condition across multiple lanes simultaneously and reducing the results into a single scalar boolean (e.g., via a vector OR reduction). If the condition involves operations that naturally produce `poison` for certain out-of-bounds or unreachable iterations (which would be safely skipped in scalar execution), the speculative parallel evaluation injects `poison` into the vector lanes. Because bitwise reductions propagate `poison`, the entire reduced scalar branch condition becomes `poison`. This pattern is not well handled because the vectorizer does not adequately mask or suppress `poison`-generating instructions in inactive or speculative lanes before reduction, leading to undefined behavior and corrupted control flow.

### Pattern 4: Incomplete Cache Invalidation After CFG Modifications
Loop transformations such as vectorization, unrolling, or peeling frequently modify the Control Flow Graph (CFG), for example, by adding new predecessors to loop exit blocks. This changes the structural properties of instructions like trivial LCSSA Phi nodes, making them non-trivial. However, the invalidation mechanisms for cached analyses (like SCEV) often rely solely on traversing direct IR use-def chains. They fail to identify and invalidate internal or indirect references (such as loop-defined expressions used in Backedge Taken Counts). This pattern is poorly handled because the incomplete invalidation leaves stale, incorrect expressions in the analysis cache, causing subsequent queries to return wrong information and resulting in miscompilations like incorrect trip counts or infinite loops.
