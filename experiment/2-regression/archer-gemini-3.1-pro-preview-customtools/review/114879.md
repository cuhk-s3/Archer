## Test Strategies

### Strategy 1: Unconditional Expansion of Trapping Instruction from Select-Like PHI

- **Target**: A test case with a select-like PHI node where one incoming value is the result of a trapping instruction (e.g., `udiv`) executed conditionally. The operands of the `udiv` should dominate the PHI node.
- **Rationale**: The patch replaces `IsAvailableOnEntry` with `properlyDominates` in `createNodeFromSelectLikePHI`. `IsAvailableOnEntry` explicitly checked if a SCEV could be materialized without introducing a fault (e.g., it rejected `scUDivExpr`). `properlyDominates` only checks if the operands of the SCEV dominate the block. If a select-like PHI node has an incoming value that is a trapping instruction (like `udiv`) defined in one of the conditional branches, its SCEV might be a `scUDivExpr`. With the patch, `properlyDominates` might return true if the operands of the `udiv` dominate the PHI node's block. This allows SCEV to model the PHI node using the `udiv` SCEV. Later, if SCEVExpander expands this SCEV at the PHI node's block, it might emit the `udiv` unconditionally, bypassing the original control-flow guard and potentially causing a fault (e.g., division by zero).
- **Expected Issue**: The SCEV expander will unconditionally emit a potentially trapping instruction (like `udiv`) at the merge block, leading to undefined behavior or a fault if the condition that guarded the instruction in the original code is false.

### Strategy 2: Hoisting Trapping Instruction via SCEV Unknown

- **Target**: A test case with a select-like PHI node where one incoming value is a complex expression involving a trapping instruction, defined conditionally, but whose SCEV components dominate the PHI node.
- **Rationale**: Similar to the first strategy, but focusing on `scUnknown`. `IsAvailableOnEntry` rejected `scUnknown` if it was an instruction that did not dominate the block. `properlyDominates` checks if the instruction properly dominates the block. If an instruction is defined in a conditional branch and used in a PHI node, it does not dominate the PHI node's block. However, if the instruction is simplified or folded into a SCEV whose components *do* dominate the block, `properlyDominates` might return true. This could lead to the same issue of unconditional expansion of a conditionally executed trapping instruction.
- **Expected Issue**: The SCEV expander will emit a trapping instruction (like `udiv`) unconditionally, leading to a fault if the original guard condition is false.

### Strategy 3: Select-Like PHI with CouldNotCompute

- **Target**: A test case with a select-like PHI node where one incoming value is an expression that SCEV cannot compute (e.g., an inline assembly call or a highly complex unsupported operation), aiming to produce a `scCouldNotCompute` SCEV.
- **Rationale**: `IsAvailableOnEntry` explicitly rejected `scCouldNotCompute`. `properlyDominates` will crash if called on `scCouldNotCompute` (as seen in `computeBlockDisposition`). If `getSCEV(LHS)` or `getSCEV(RHS)` returns `scCouldNotCompute`, the new code will crash. While `getSCEV` usually doesn't return `scCouldNotCompute` directly for valid IR, there might be edge cases where it does, or where a sub-expression is `scCouldNotCompute`. This strategy aims to trigger the `llvm_unreachable` in `computeBlockDisposition`.
- **Expected Issue**: The SCEV expander will emit a trapping instruction unconditionally, leading to a fault.

### Strategy 4: Select-Like PHI with AddRecExpr from Inner Loop

- **Target**: A test case with a select-like PHI node in an outer loop, where one incoming value is an induction variable (AddRec) from an inner loop that has already finished executing.
- **Rationale**: `IsAvailableOnEntry` had specific logic for `scAddRecExpr`: it allowed them only if the loop of the AddRec was the same as or contained the loop of the basic block. `properlyDominates` checks if the loop header dominates the block. If a select-like PHI is in an outer loop, and one of its incoming values is an AddRec from an inner loop, `properlyDominates` might return true (since the inner loop header dominates the outer loop block after the inner loop), whereas `IsAvailableOnEntry` might have rejected it. This could lead to incorrect expansion of the AddRec outside its valid loop context.
- **Expected Issue**: The SCEV expander will emit a trapping instruction unconditionally, leading to a fault.

### Strategy 5: Select-Like PHI with Trapping Constant Expression

- **Target**: A test case with a select-like PHI node where one incoming value is a potentially trapping constant expression (e.g., division by a constant that might be zero in some contexts, or an out-of-bounds GEP).
- **Rationale**: Constant expressions can trap (e.g., a constant division by zero, though usually caught earlier). If a select-like PHI has a trapping constant expression as an incoming value, `IsAvailableOnEntry` might have rejected it (if it modeled it as a trapping operation), while `properlyDominates` will always return true for `scConstant`. This could lead to the unconditional evaluation of a trapping constant expression.
- **Expected Issue**: The SCEV expander will emit a trapping instruction unconditionally, leading to a fault.

### Analysis

The patch replaces the custom `IsAvailableOnEntry` function with the standard `properlyDominates` function in `ScalarEvolution::createNodeFromSelectLikePHI`. `IsAvailableOnEntry` was designed to check if a SCEV could be materialized at a basic block without introducing a fault. It explicitly rejected `scUDivExpr` and `scCouldNotCompute`, and had custom logic for `scUnknown` and `scAddRecExpr`.

The patch author realized that `SCEVExpander` already has a robust `isSafeToExpand` function that checks if a SCEV is safe to expand at a given insertion point. For example, `isSafeToExpand` explicitly checks if the denominator of a `scUDivExpr` is known to be non-zero, and if not, it refuses to expand it. Therefore, the custom checks in `IsAvailableOnEntry` were redundant and overly conservative. By replacing it with `properlyDominates`, SCEV can model more expressions (like conditionally executed `udiv`s or `AddRecExpr`s from other loops), and relies on `SCEVExpander` to ensure they are only expanded when safe.

The patch was initially reverted because it exposed an issue in `SCEVExpander/LCSSA`, which was subsequently fixed in D150681. After the fix, the patch was reapplied. My analysis confirms that `SCEVExpander` correctly handles the cases that `IsAvailableOnEntry` used to reject, such as `udiv` (by refusing to expand if unsafe) and `AddRecExpr` outside its loop (by inserting LCSSA PHI nodes). Thus, the patch is correct and does not introduce new bugs.
