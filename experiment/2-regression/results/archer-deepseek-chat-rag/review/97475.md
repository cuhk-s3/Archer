## Test Strategies

### Strategy 1: Complex condition patterns

- **Target**: Test findValuesAffectedByCondition coverage
- **Rationale**: The optimization relies on findValuesAffectedByCondition to detect affected values. Missing patterns could cause missed optimizations or incorrect constant folding.
- **Expected Issue**: Select arm should be constant but isn't folded due to missed affected value detection.

### Strategy 2: Recursion depth exhaustion

- **Target**: Deeply nested expressions in select arms
- **Rationale**: hasAffectedValue and computeKnownBits have recursion depth limits. Exceeding limits may cause incorrect constant folding or missed optimizations.
- **Expected Issue**: Incorrect constant folding when recursion depth is exceeded, or missed optimization due to early termination.

### Strategy 3: PHI node complexity

- **Target**: Select arms containing PHI nodes with complex dependencies
- **Rationale**: hasAffectedValue has special handling for PHI nodes (Depth = MaxAnalysisRecursionDepth-2). This may be insufficient for complex PHI networks.
- **Expected Issue**: Incorrect analysis of PHI nodes leading to wrong constant folding or missed optimizations.

### Strategy 4: Condition inversion edge cases

- **Target**: Complex logical conditions (AND/OR) with inversion
- **Rationale**: The optimization inverts condition for false arm. Complex logical conditions may not invert correctly, especially with nested operations.
- **Expected Issue**: Incorrect known bits for false arm due to wrong condition inversion.

### Strategy 5: Vector type mismatches

- **Target**: Vector selects with different element counts or types between condition and arms
- **Rationale**: The optimization checks SelType->isIntOrIntVectorTy() but may not handle all vector type combinations correctly.
- **Expected Issue**: Incorrect constant folding for vector selects or compiler crashes.

### Strategy 6: Undef/poison propagation

- **Target**: Conditions or select arms containing undef/poison values
- **Rationale**: Undef/poison in conditions may lead to incorrect known bits. The optimization may incorrectly fold to constant when undef is involved.
- **Expected Issue**: Incorrect constant folding when undef/poison values are present.

### Strategy 7: Conflicting known bits resolution

- **Target**: Conditions that imply conflicting known bits for a value
- **Rationale**: computeKnownBitsFromContext resets Known when hasConflict(). Need to ensure this handles all conflict cases correctly.
- **Expected Issue**: Incorrect constant folding when conflicting known bits should cancel but don't.

### Analysis

## Bug Triggering Analysis

The test case `vec_select_no_equivalence` exposes a real bug in the optimization. When `x = <0, 2>`:
- Original code: `x10 = shufflevector <0, 2>` = `<2, 0>`, `cond = icmp eq <0, 2>, <0, 0>` = `<true, false>`, `select <true, false>, <2, 0>, <0, 2>` = `<2, 2>`
- Optimized code: Returns `<0, 2>` (just returns `x`)

The bug occurs because the optimization incorrectly assumes that when the condition is true (x == 0), we can replace `x10` with `x`. However, `x10` is a `shufflevector` of `x`, not `x` itself. The optimization doesn't properly handle vector operations like `shufflevector` that transform the value.

## Fix Weakness Analysis

The weakness in the fix is that `findValuesAffectedByCondition` and `hasAffectedValue` don't properly track how values are transformed through operations. The optimization correctly identifies that `x` is affected by the condition `x == 0`, but fails to recognize that `x10` (a shuffle of `x`) is also affected in a way that doesn't allow simple substitution.

The optimization assumes that if we know `x == 0`, then any expression involving `x` can be simplified by substituting `0` for `x`. This is incorrect for operations like `shufflevector` where the result depends on the arrangement of elements, not just their values.

The fix needs to either:
1. Not apply the optimization when the select arm contains operations that transform the value in non-trivial ways
2. Or properly compute the result of substituting the known value through all operations

## Summary of Testing

I have tested 5 out of 14 test cases and found one confirmed bug. The other test cases I tested (n4, n5, src_or_eq_0_and_xor) work correctly. The bug is specific to operations like `shufflevector` that transform values in ways that aren't captured by simple constant substitution.
