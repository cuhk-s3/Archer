## Test Strategies

### Strategy 1: Imprecise ConstantRange triggering false unreachable

- **Target**: Test cases where LVI returns over-approximate ranges that include impossible values
- **Rationale**: If LVI's constant range is too conservative (includes values not actually possible), the optimization might incorrectly mark default as unreachable when it's actually reachable.
- **Expected Issue**: Default case incorrectly marked unreachable, leading to miscompilation when the impossible value occurs at runtime.

### Strategy 2: Wrapped range miscalculation

- **Target**: Switch conditions with signed/unsigned wrap-around ranges
- **Rationale**: ConstantRange::isSizeLargerThan might not correctly handle wrapped ranges, leading to incorrect coverage determination.
- **Expected Issue**: Incorrect unreachable default for wrapped ranges, or missed optimization for fully-covered wrapped ranges.

### Strategy 3: Single case full coverage missed

- **Target**: Switch with single case that covers entire value range
- **Rationale**: The optimization requires ReachableCaseCount > 1, but a single case could cover the entire range (e.g., switch on i1 with case 0, default unreachable).
- **Expected Issue**: Missed optimization opportunity, default remains reachable when it should be unreachable.

### Strategy 4: Case values outside constant range

- **Target**: Switch cases with values outside LVI's reported constant range
- **Rationale**: ReachableCaseCount includes cases even if their values are outside CR, but they shouldn't count toward coverage of CR.
- **Expected Issue**: Incorrectly marking default unreachable when cases outside CR don't actually cover values within CR.

### Strategy 5: PHI nodes hiding unreachable

- **Target**: Default block with PHI nodes followed by unreachable
- **Rationale**: The check uses getFirstNonPHIOrDbg() which might skip over PHI nodes, missing an existing unreachable instruction after PHIs.
- **Expected Issue**: Redundant transformation creating duplicate unreachable blocks or incorrect dominator updates.

### Strategy 6: Undef/poison handling

- **Target**: Switch conditions that could be undef/poison
- **Rationale**: UndefAllowed=false assumes no undef, but LVI might not guarantee this, leading to incorrect range analysis.
- **Expected Issue**: Incorrect optimization when switch on undef is possible (UB but still needs correct analysis).
