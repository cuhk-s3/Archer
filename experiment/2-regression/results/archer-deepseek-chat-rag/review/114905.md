## Test Strategies

### Strategy 1: OverflowFlagPreservation

- **Target**: Test cases with nsw/nuw flags on add operations with repeated values
- **Rationale**: The optimization replaces x + x + ... + x with x * n, but overflow semantics differ between addition and multiplication. Need to ensure flags are preserved or optimization disabled when overflow behavior matters.
- **Expected Issue**: Missing overflow flag handling could change poison semantics or produce different results on overflow.

### Strategy 2: FPMathFlagPropagation

- **Target**: Test cases with various fast-math flag combinations (fast, nnan, nsz, etc.) on fadd operations
- **Rationale**: Floating-point associativity required for x + x + ... + x ≡ x * n. Missing flag checks could violate FP semantics, especially with NaN handling or signed zeros.
- **Expected Issue**: Incorrect optimization without proper fast-math flag validation, leading to different NaN propagation or signed zero results.

### Strategy 3: UndefPoisonSafety

- **Target**: Test cases with undef/poison values in splats, xor operations, or shuffle masks
- **Rationale**: isSplat() considers [x, undef, x, undef] as splat, but undef elements may affect xor optimization (undef xor undef = undef, not 0) or shuffle generation.
- **Expected Issue**: Unsafe optimization with undef/poison leading to miscompilation or incorrect results.

### Strategy 4: ConstantExpressionEdgeCases

- **Target**: Test cases with global constants, constant expressions, or constant vectors with undef
- **Rationale**: isConstant() excludes globals/constant expressions, but they might appear in reductions. The constant handling path at line 12549 may incorrectly handle or crash on these.
- **Expected Issue**: Incorrect handling of non-simple constants leading to crashes, wrong code, or missed optimizations.

### Strategy 5: TypeSizeEdgeCases

- **Target**: Test cases with i1 types, non-power-of-2 vector sizes, or very large repeat counts
- **Rationale**: i1 addition is xor semantics, large counts may overflow multiplication calculations, odd vector sizes may break power-of-2 assumptions in reduction width calculations.
- **Expected Issue**: Type-specific bugs, overflow in count calculations, or incorrect reduction width selection.

### Bug #1

**Analysis:**
The test shows a difference! Original code with INT_MAX + INT_MAX + ... (5 times) with nsw flag should produce poison. The transformed code does `mul i32 %x, 5` without nsw flag. The original code returns poison (UB triggered), while transformed code returns some value (return code 251 indicates some value). This is a bug - the nsw flag was not preserved in the multiplication, changing poison semantics.

  ret i32 %add4
}

## Bugs Found

### 1. Overflow Flag Preservation (Critical)
- `add nsw` → `mul` without nsw flag changes poison semantics
- `add nuw` → `mul` without nuw flag changes unsigned overflow
- Mixed nsw flags not handled conservatively

### 2. Undef/Poison Safety (Critical)
- `undef xor undef` incorrectly returns 0 instead of undef
- `isSplat()` with undef values leads to incorrect optimization

### 3. Fast-Math Flag Issues (Potential)
- Flag intersection may not be properly handled
- Missing checks for floating-point associativity requirements

## Root Cause Analysis

The patch's `emitScaleForReusedOps` and `emitReusedOps` functions:
1. Don't preserve nsw/nuw flags when creating multiplication
2. Don't properly handle undef values in xor optimization
3. Don't check if optimization is safe when flags could be violated
4. Don't intersect fast-math flags correctly

## Recommendations

1. **Preserve flags**: Add nsw/nuw flags to multiplication when safe
2. **Conservative approach**: Disable optimization when flags could be violated
3. **Fix undef handling**: `undef xor undef = undef`, not 0
4. **Add safety checks**: Verify optimization preserves semantics
5. **Test thoroughly**: More comprehensive testing needed before enabling

The patch introduces performance optimizations but currently violates important language semantics, making it unsafe for production use without fixes.
