## Test Strategies

### Strategy 1: NegativeBECountActiveBits

- **Target**: Create test where MaxBECount is negative (signed) but has few active bits when interpreted as unsigned
- **Rationale**: getActiveBits() counts leading zeros on unsigned representation, so negative values (all ones in two's complement) have 0 active bits, passing the check incorrectly
- **Expected Issue**: Truncation of negative value could produce wrong range or crash

### Strategy 2: SignedUnsignedTypeMismatch

- **Target**: Test with MaxBECount of signed type larger than AddRec's unsigned type
- **Rationale**: Active bits check doesn't consider signedness, truncating signed to unsigned could change value interpretation
- **Expected Issue**: Incorrect range computation due to signed/unsigned mismatch

### Strategy 3: HighBitSetActiveBits

- **Target**: MaxBECount with value like 0x80000000 (high bit set) in wider type
- **Rationale**: getActiveBits() = 32 for 0x80000000, but truncating to i32 changes signed interpretation
- **Expected Issue**: Range computation with wrong signed interpretation

### Strategy 4: NegativeBECountZeroExtend

- **Target**: MaxBECount with bit width < AddRec width but negative value
- **Rationale**: Current code does MaxBECount.zext(BitWidth) for narrower MaxBECount, but zero-extending negative signed value changes value
- **Expected Issue**: Wrong range due to zero extension changing signed value

### Strategy 5: TypeWidthAssertion

- **Target**: Test where Start/Step types differ from truncated MaxBECount type
- **Rationale**: getRangeForAffineAR asserts bit width equality, but types might differ after truncation
- **Expected Issue**: Assertion failure or incorrect type handling

### Strategy 6: ActiveBitsBoundary

- **Target**: Test values at boundaries: 0, 1, 2^BitWidth-1, 2^BitWidth
- **Rationale**: Edge cases for getActiveBits() calculation and truncation behavior
- **Expected Issue**: Off-by-one errors in range computation
