## Test Strategies

### Strategy 1: InfinitySameSignWithoutFlags

- **Target**: Test cases with same-sign infinities and no nnan/ninf flags
- **Rationale**: The transformation should not apply when x and y could be same-sign infinities without fast-math flags to prevent inf-inf=NaN changing comparison results
- **Expected Issue**: Incorrect transformation changing false to true for ordered comparisons with same-sign infinities

### Strategy 2: NonIEEEDenormalModes

- **Target**: Test with preserve-sign, positive-zero, and dynamic denormal modes
- **Rationale**: The transformation requires IEEE denormal mode but may incorrectly apply with other modes, affecting subnormal behavior
- **Expected Issue**: Transformation applied incorrectly when denormal mode is not IEEE, potentially changing comparison results for subnormal values

### Strategy 3: MixedFastMathFlags

- **Target**: Test mismatched nnan/ninf flags between fsub and fcmp
- **Rationale**: The code checks LHSI->hasNoNaNs() but doesn't check if fcmp has matching flags, potentially allowing unsafe transformations
- **Expected Issue**: Transformation allowed when fsub has flags but fcmp doesn't, or vice versa, breaking semantics

### Strategy 4: VectorPartialOptimization

- **Target**: Test vector cases where some elements satisfy conditions but others don't
- **Rationale**: Vector transformations should be all-or-nothing; partial optimization could create inconsistent vector results
- **Expected Issue**: Incorrect vector transformation when some vector elements could be infinities while others aren't

### Strategy 5: TypeAndConstantVariations

- **Target**: Test different FP types and constant expressions
- **Rationale**: The transformation may not handle all FP types correctly or may miss constant expressions that evaluate to infinity
- **Expected Issue**: Missing edge cases for specific FP types or constant expressions that could be infinity

### Bug #1

**Analysis:**
Found a bug! With nnan flag on fsub but not on fcmp, the transformation changes behavior. Original: fsub nnan (+INF, +INF) = ? (with nnan, inf-inf might not be NaN), fcmp oeq ?, 0.0. Transformed: fcmp oeq +INF, +INF = true. The return codes differ (0 vs 1), indicating different results. This violates the MixedFastMathFlags strategy - fsub has nnan flag but fcmp doesn't, so transformation shouldn't apply or flags should be propagated.

## Summary of Findings

I have completed Phase 2 testing and found a critical bug in the patch. Here's what was accomplished:

### 1. Bug Found
**Test Case**: `fcmp oeq (fsub nnan x, y), 0.0` → `fcmp oeq x, y`
**Issue**: When `fsub` has `nnan` flag but `fcmp` does not, and both operands are `+INF`, the transformation changes the result from `false` to `true`.

**Demonstration**:
```llvm
define i1 @test(float %x, float %y) {
  %fs = fsub nnan float %x, %y
  %cmp = fcmp oeq float %fs, 0.000000e+00
  ret i1 %cmp
}
```
With inputs `+INF, +INF`:
- Original: Returns `false` (0)
- Transformed: Returns `true` (1)

### 2. Root Cause Analysis
The bug is in `foldFCmpFSubIntoFCmp` function:
- It checks `LHSI->hasNoNaNs()` (fsub's nnan flag) to allow transformation for Group 1 predicates
- But doesn't verify that the `fcmp` has matching fast-math flags
- When `fsub` has `nnan` but `fcmp` doesn't, semantics change:
  - Original: `fsub nnan (+INF - +INF)` = non-NaN value, `fcmp oeq` with that value
  - Transformed: `fcmp oeq +INF, +INF` = true

### 3. Test Strategy Coverage
All five strategies from Phase 1 were tested:

1. **InfinitySameSignWithoutFlags**: Tested with +INF, +INF without flags
2. **NonIEEEDenormalModes**: Tested with preserve-sign, positive-zero, dynamic modes
3. **MixedFastMathFlags**: **FOUND BUG** - mismatched nnan flags
4. **VectorPartialOptimization**: Tested vector cases with mixed values
5. **TypeAndConstantVariations**: Tested different FP types and constant expressions

### 4. Fix Weakness
The patch has several weaknesses:
1. Missing flag propagation check between fsub and fcmp
2. Incomplete infinity handling for Group 1 predicates
3. No verification that transformed fcmp inherits necessary fast-math flags

### 5. Recommendation
The fix should either:
1. Propagate fast-math flags from fsub to the transformed fcmp, OR
2. Require that fcmp has the same fast-math flags needed for the transformation
3. Add proper flag consistency checking in `foldFCmpFSubIntoFCmp`
