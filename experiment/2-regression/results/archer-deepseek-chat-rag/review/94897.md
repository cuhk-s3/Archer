## Test Strategies

### Strategy 1: Boundary Constant Values

- **Target**: Test constants at min/max boundaries for each predicate type
- **Rationale**: The getFlippedStrictnessPredicateAndConstant function must handle boundary cases correctly. Testing with constants at min/max values (0, UINT_MAX, INT_MIN, INT_MAX) for each predicate type (ult, ugt, slt, sgt) can expose issues where the transformation incorrectly attempts to flip strictness when it shouldn't.
- **Expected Issue**: Incorrect transformation or assertion failure when trying to flip strictness at boundary values.

### Strategy 2: Vector Edge Cases

- **Target**: Test vectors with mixed undef elements and boundary constants
- **Rationale**: Vectors with some undef elements and some boundary constant elements may cause issues in getFlippedStrictnessPredicateAndConstant. The function replaces undefs with a safe constant, but if all elements are boundary values, there may be no safe replacement.
- **Expected Issue**: Incorrect handling of vector undef elements leading to wrong constant adjustment or predicate flip.

### Strategy 3: Shift Amount Edge Cases

- **Target**: Test shift amounts near type bit width boundaries
- **Rationale**: When Amt = TypeBits - 1 or Amt = 0, edge cases in the transformation logic. Also when TypeBits - Amt results in very small types (i1, i2) or illegal types according to shouldChangeType.
- **Expected Issue**: Incorrect type calculation or illegal type transformation.

### Strategy 4: Signed/Unsigned Semantics

- **Target**: Test signed comparisons with unsigned-like constants and vice versa
- **Rationale**: The transformation uses ashr for all comparisons, but signed and unsigned semantics differ. Testing edge cases where sign extension affects the result, especially with high-bit constants.
- **Expected Issue**: Incorrect transformation due to sign extension issues with ashr for unsigned comparisons.

### Strategy 5: Transformation Chain Interactions

- **Target**: Test cases where the transformed icmp gets further canonicalized
- **Rationale**: The patch converts strict predicates to non-strict ones, which may then be canonicalized back to strict predicates by other passes. Testing chains of transformations can expose issues with idempotency or convergence.
- **Expected Issue**: Infinite loop or non-convergence in instcombine due to predicate flipping cycles.
