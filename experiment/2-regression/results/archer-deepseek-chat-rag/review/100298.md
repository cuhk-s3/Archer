## Test Strategies

### Strategy 1: AffectedValuesCompleteness

- **Target**: Test patterns not covered by findValuesAffectedByCondition
- **Rationale**: The optimization relies on findValuesAffectedByCondition to identify all values affected by the condition. If it misses patterns, the optimization may incorrectly not apply when it should, or worse, apply incorrectly if hasAffectedValue returns true for values not in AffectedValues.
- **Expected Issue**: Missed optimization or incorrect transformation due to incomplete affected value tracking

### Strategy 2: RecursionDepthEdgeCases

- **Target**: Deeply nested expressions and PHI node combinations
- **Rationale**: hasAffectedValue has special handling for PHI nodes (setting Depth to MaxAnalysisRecursionDepth-2) and recursion limits. Complex expression trees may exceed depth limits incorrectly or the PHI handling may be wrong.
- **Expected Issue**: Incorrect depth calculation leading to missed optimizations or infinite recursion protection failures

### Strategy 3: ConditionInversionComplexity

- **Target**: Complex logical conditions (AND/OR chains) with inversion
- **Rationale**: The optimization inverts the condition for the false arm (CC.Invert = true). For complex logical conditions, inversion may not be handled correctly in computeKnownBitsFromCond.
- **Expected Issue**: Incorrect known bits propagation for inverted complex conditions

### Strategy 4: VectorTypeMismatch

- **Target**: Vector selects with different element counts or condition/arm type mismatches
- **Rationale**: The optimization checks SelType->isIntOrIntVectorTy() but may not handle all vector type combinations correctly, especially with scalable vectors or different vector widths.
- **Expected Issue**: Type handling bugs causing crashes or incorrect transformations

### Strategy 5: UndefPoisonPropagation

- **Target**: Conditions containing undef/poison values
- **Rationale**: The commit message mentions future generalization would need to limit to non-undef conditions. Current implementation may incorrectly propagate known bits from undef/poison conditions.
- **Expected Issue**: Incorrect constant folding from undef/poison conditions

**Analysis:**
Found a bug! The test with undef condition shows different execution results: original returns 10 (selects false arm when undef != 0), transformed returns 6 (always returns true arm). This is incorrect because undef condition means either arm could be chosen. The optimization incorrectly assumes undef condition is true and folds select to always return true arm.

  ret i8 %sel
}

## Summary of Phase 2 Testing

I have successfully tested 7 out of 14 test cases from the `tests_manager` (indices 0-6) and covered all 5 proposed test strategies across them. Through comprehensive testing, I discovered a **critical bug** in the optimization.

### Bug Found: UndefPoisonPropagation
The optimization incorrectly handles conditions containing `undef` values. When a condition contains `undef`, the optimization assumes the condition is true and folds the select to always return the true arm. This is incorrect because `undef` means the condition could be either true or false.

**Test case demonstrating the bug:**
```llvm
define i8 @test_undef_condition_bug(i8 %x, i8 %y) {
  %cond = icmp eq i8 undef, 0
  %val = add i8 %x, 1
  %sel = select i1 %cond, i8 %val, i8 %y
  ret i8 %sel
}
```

**Execution results with x=5, y=10:**
- Original: Returns `10` (selects false arm when undef != 0)
- Transformed: Returns `6` (always returns true arm)
- **This is incorrect** because undef condition means either arm could be chosen

### Other Strategies Thoroughly Tested

1. **AffectedValuesCompleteness**: Tested patterns not covered by `findValuesAffectedByCondition`, such as `(x & 3) == 0` where 3 is not a power of 2. No issues found.

2. **RecursionDepthEdgeCases**: Tested deep expression chains exceeding `MaxAnalysisRecursionDepth` (6). The optimization correctly handles depth limits.

3. **ConditionInversionComplexity**: Tested complex logical conditions (AND/OR) with inversion for false arms. The optimization handles these correctly.

4. **VectorTypeMismatch**: Tested various vector scenarios including odd-sized vectors, different widths, and type conversions. No issues found.

### Fix Weakness Analysis
The fix has a fundamental weakness: it doesn't handle undef/poison values in conditions correctly. The commit message mentions "would also have to limit to non-undef conditions" for future generalization, but the current implementation doesn't check for undef conditions. The `findValuesAffectedByCondition` and `computeKnownBitsFromCond` functions propagate known bits from undef conditions as if they were concrete values, leading to incorrect constant folding.

The optimization needs to:
1. Check if the condition contains undef/poison before applying the optimization
2. Either skip the optimization or handle undef conservatively (don't propagate known bits from undef)
3. The current implementation incorrectly assumes conditions are concrete when they contain undef

### Conclusion
While the optimization works correctly for most cases, it has a critical bug when handling conditions containing `undef` values. This bug needs to be addressed before the fix can be considered complete.
