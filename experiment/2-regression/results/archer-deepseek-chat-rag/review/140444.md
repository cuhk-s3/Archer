## Test Strategies

### Strategy 1: BTC edge cases

- **Target**: Backedge taken count analysis
- **Rationale**: The patch assumes BTC > 0 and can evaluate at BTC and BTC-1. Edge cases: BTC=1 (loop executes exactly once), BTC=unknown, BTC with overflow, BTC computed incorrectly due to SCEV limitations.
- **Expected Issue**: Peeling last iteration when loop executes only once, causing peeled iteration to not execute or incorrect exit condition adjustment.

### Strategy 2: Non-NE/EQ exit conditions

- **Target**: Exit condition predicate checking
- **Rationale**: canPeelLastIteration only checks for NE/EQ predicates. Other predicates like ULT, ULE, SLT, SLE might also be adjustable by subtracting 1 from RHS, but aren't handled.
- **Expected Issue**: Missed optimization opportunity or incorrect transformation for other comparable predicates.

### Strategy 3: Step size variations

- **Target**: Induction step validation
- **Rationale**: Code assumes step is exactly 1. What about step 2, -1, or variable step? shouldPeelLastIteration uses evaluateAtIteration which might work for other steps, but exit condition adjustment subtracts 1.
- **Expected Issue**: Incorrect exit condition adjustment for non-unit steps or mis-evaluation of induction values.

### Strategy 4: Multiple exit edges

- **Target**: Exit edge handling in cloneLoopBlocks
- **Rationale**: When peeling last iteration, exit values are collected from latch->exit edge. What if there are multiple exit edges with different PHI incoming values?
- **Expected Issue**: Incorrect exit value replacement or missed PHI updates.

### Strategy 5: PHI node with multiple latch predecessors

- **Target**: PHI resolution in cloneLoopBlocks for PeelLast
- **Rationale**: For PeelLast, PHI nodes use getIncomingValueForBlock(Latch). If PHI has multiple incoming values from latch (different predecessors), this might pick wrong value.
- **Expected Issue**: Incorrect PHI value resolution leading to wrong computation.

### Strategy 6: RHS side effects or non-constant

- **Target**: Exit condition adjustment in peelLoop
- **Rationale**: The code subtracts 1 from Cmp->getOperand(1). If this operand has side effects or is not a constant, the transformation might be invalid.
- **Expected Issue**: Illegal IR transformation or changed semantics.

### Strategy 7: Convergent operations

- **Target**: Operation duplication in peeled iteration
- **Rationale**: Peeling duplicates loop body. Convergent operations (barriers, atomics) should not be duplicated. Existing canPeel checks might not catch all cases.
- **Expected Issue**: Incorrect semantics for convergent operations.

### Strategy 8: LCSSA form violations

- **Target**: LCSSA form maintenance after peeling
- **Rationale**: Peeling last iteration modifies exit edges and PHI nodes. formLCSSA is called, but complex CFG might not be properly updated.
- **Expected Issue**: Broken LCSSA form causing subsequent passes to fail.

### Strategy 9: Nested loop interactions

- **Target**: Outer loop PHI updates
- **Rationale**: Peeling inner loop's last iteration affects values flowing to outer loop. The patch might not properly update outer loop PHIs.
- **Expected Issue**: Incorrect values in outer loop after inner loop peeling.

### Strategy 10: Profile metadata handling

- **Target**: Branch weight updates for peeled last iteration
- **Rationale**: Branch weights are updated for peeled iterations, but the logic for last iteration peeling might differ from beginning peeling.
- **Expected Issue**: Incorrect branch probabilities affecting later optimizations.
