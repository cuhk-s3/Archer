## Test Strategies

### Strategy 1: ScalableVFWithInterleaving

- **Target**: Test with scalable VF (e.g., vscale x 4) and interleave count > 1
- **Rationale**: RuntimeVF for scalable vectors is not constant; index calculations with CreateMul(RuntimeVF, ConstantInt) might have type/overflow issues. The patch assumes fixed-width VF for RuntimeVF calculations.
- **Expected Issue**: Type mismatch, overflow, or incorrect lane extraction for scalable vectors with interleaving.

### Strategy 2: MixedScalarVectorInterleaving

- **Target**: Test with VF=1 (scalar) but interleave count > 1
- **Rationale**: When VF.isScalar(), the code returns State.get(getOperand(Idx)) directly without index adjustment. This might not handle interleaved scalars correctly across multiple parts.
- **Expected Issue**: Incorrect value extraction when VF=1 but IC>1, extracting wrong scalar from interleaved groups.

### Strategy 3: UnrollWithExtractLane

- **Target**: Test unrolling (UF>1) with ExtractLane and interleaving
- **Rationale**: During unrolling, ExtractLane gets additional operands for each part. If the original recipe has wrong operand count or type, unrolling could produce malformed IR or crash.
- **Expected Issue**: Unrolling failure, wrong operand count after unrolling, or type mismatch in unrolled ExtractLane.

### Strategy 4: CostModelUnderestimation

- **Target**: Test cost model for ExtractLane with large interleave counts
- **Rationale**: ExtractLane cost is modeled same as ExtractElement, but ExtractLane with IC>1 generates chain of ICmp+Select ops. This could lead to wrong vectorization decisions.
- **Expected Issue**: Incorrect vectorization decision due to underestimated cost of ExtractLane with high IC.

### Strategy 5: PatternMatchingEdgeCases

- **Target**: Test optimizeEarlyExitInductionUser with various induction patterns
- **Rationale**: The pattern match changed from ExtractElement(Incoming, FirstActiveLane(Mask)) to ExtractLane(FirstActiveLane(Mask), Incoming). Edge cases with multiple operands or nested patterns might not match.
- **Expected Issue**: Missed optimization opportunities for induction variable exit values in early-exit loops.

### Strategy 6: PoisonUndefPropagation

- **Target**: Test with poison/undef values in lane index or vector parts
- **Rationale**: The select chain in ExtractLane.generate() might propagate poison incorrectly if lane index is poison or out of bounds. ICmpUGE with poison could produce unexpected results.
- **Expected Issue**: Incorrect poison propagation, miscompilation with undef/poison inputs.

### Strategy 7: TypeSizeMismatch

- **Target**: Test with lane index type different from RuntimeVF type
- **Rationale**: IdxTy from inferScalarType(getOperand(0)) might be i32 while RuntimeVF is i64 (or vice versa). CreateMul with mismatched types could truncate or extend incorrectly.
- **Expected Issue**: Type mismatch in index calculations, truncation/extension bugs.
