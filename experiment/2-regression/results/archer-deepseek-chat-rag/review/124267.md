- **Rationale**: Test cases where samesign flag is incorrectly set or where operands can have different signs despite the flag. This could expose issues where the optimization incorrectly assumes same sign semantics.

### Analysis

## Bug Triggering Analysis

The bug is in the `isImpliedCondICmps` function in ValueTracking.cpp. The patch teaches the optimization about `samesign` flags, but incorrectly handles poison semantics. Specifically:

1. **Pattern**: The optimization recognizes that `x >s y` (or `x >=s y`) implies `(x -nsw y) < (-C)` is false when `C >= 0`. With the patch, it also recognizes `samesign ugt` and `samesign uge` as equivalent to signed comparisons when operands have same sign.

2. **Bug**: The optimization simplifies the entire `abs` pattern (select with icmp samesign ult) to just return `%add`. However, the icmp `samesign ult i32 %sub, -1` can be **poison** when `%sub` and `-1` have different signs.

3. **Example**: When `x = -3, y = -5` (both negative, same sign):
   - `icmp samesign ugt i32 -3, -5` = true (not poison, same sign)
   - `%sub = sub nsw i32 -3, -5` = 2 (positive)
   - `icmp samesign ult i32 2, -1` = **poison** (2 is positive, -1 is negative → different signs)
   - `select i1 poison, i32 %neg, i32 %add` = poison
   - **Original**: returns poison
   - **Transformed**: returns `%add` (3)

4. **Root Cause**: The optimization assumes that when `x >s y` (or equivalent with samesign), then `(x - y)` is non-negative, so the `abs` pattern always selects `%add`. But it fails to consider that `icmp samesign ult i32 %sub, -1` can be poison when comparing a non-negative `%sub` with `-1` (which is negative).

## Fix Weakness Analysis

The fix weakness is that it doesn't properly account for poison semantics of `samesign` icmp when the RHS is `-1` (or any constant with different sign than possible `%sub` values). The optimization should:

1. **Check sign compatibility**: Before removing the `icmp samesign`, verify that `%sub` and the constant (`-1`) can have the same sign for all possible values where the implication holds.

2. **Alternative approach**: Only apply the optimization when the constant RHS has the same sign as `%sub` for all valid `%sub` values (i.e., when `%sub` is non-negative and RHS is non-negative, or both non-positive).

3. **Poison awareness**: The optimization transforms poison-producing code into non-poison code, which is incorrect. Transformations must preserve poison semantics.

The test case reveals this weakness by using `-1` as the RHS constant, which is negative, while `%sub` (when `x > y`) is non-negative, guaranteeing different signs and thus poison in the `samesign` icmp.
