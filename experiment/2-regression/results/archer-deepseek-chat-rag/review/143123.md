## Test Strategies

### Strategy 1: MismatchedFlagsTest

- **Target**: Create tests with mismatched FMF between select and fcmp instructions
- **Rationale**: The fix assumes select's FMF are correct, but fcmp's nnan is preserved. Mismatched flags could lead to incorrect pattern matching decisions about NaN behavior or signed zeros.
- **Expected Issue**: Incorrect pattern recognition (e.g., matching as SPF_FMINNUM when it shouldn't, or vice versa) due to conflicting FMF from select and fcmp.

### Strategy 2: DefaultParameterCallerTest

- **Target**: Test callers that use default FMF parameter (like InstructionSimplify)
- **Rationale**: Some callers don't pass FMF, using the default FastMathFlags(). This might miss important flags from the actual select instruction.
- **Expected Issue**: Pattern matching fails to recognize valid patterns because missing FMF causes conservative rejection.

### Strategy 3: VectorFMFTest

- **Target**: Test vector select/compare patterns with FMF
- **Rationale**: Vector operations might have different FMF handling. The patch might not correctly propagate FMF for vector types.
- **Expected Issue**: Incorrect pattern matching for vector min/max operations due to improper FMF handling.

### Strategy 4: CastPropagationTest

- **Target**: Test select patterns involving casts (FPToSI/FPToUI) with FMF
- **Rationale**: The code sets NoSignedZeros for certain casts. This might interact incorrectly with the new FMF source (select vs fcmp).
- **Expected Issue**: Incorrect handling of -0.0 in cast-to-integer min/max patterns due to wrong FMF propagation.

### Strategy 5: RecursivePatternTest

- **Target**: Test nested select patterns with FMF
- **Rationale**: Recursive pattern matching might propagate FMF incorrectly through multiple levels of select instructions.
- **Expected Issue**: Incorrect pattern recognition in complex nested select expressions due to FMF propagation errors.

### Strategy 6: OtherFMFFlagsTest

- **Target**: Test with FMF flags other than nnan/nsz (ninf, arcp, contract, afn, reassoc)
- **Rationale**: The patch changes which instruction provides these flags (select instead of fcmp). Some optimizations might depend on these flags.
- **Expected Issue**: Incorrect optimization decisions due to missing or incorrect FMF flags from unexpected sources.

## Bug Triggering Analysis

This comprehensive test case demonstrates the complete behavior of the FMF fix. The fcmp has multiple FMF flags (nnan, ninf, arcp, contract, afn, reassoc) while the select only has nsz. According to the fix:

1. The select's nsz flag should be used
2. Only the nnan flag from fcmp should be preserved (via `FMF.setNoNaNs()`)
3. Other fcmp flags (ninf, arcp, contract, afn, reassoc) should NOT affect the pattern matching

The transformation shows that the resulting minnum intrinsic has both nnan and nsz flags, correctly combining the select's nsz with the preserved nnan from fcmp.

## Fix Weakness Analysis

The fix successfully addresses the core issue identified in the patch. The key observations are:

1. **Correct FMF Source**: The fix properly uses FMF from the select instruction instead of the fcmp instruction, which was the original bug.

2. **Selective Preservation**: Only the `nnan` flag from fcmp is preserved (via `if (isa<FPMathOperator>(CmpI) && CmpI->hasNoNaNs()) FMF.setNoNaNs()`), which is semantically correct since NaN behavior affects comparison results.

3. **Other FMF Flags**: Flags like ninf, arcp, contract, afn, reassoc from fcmp are correctly ignored, as they don't affect the semantics of the comparison itself.

4. **Vector Support**: The fix works correctly for vector types as demonstrated in the test cases.

5. **Recursive Patterns**: Nested select patterns with FMF propagate correctly through the recursion.

6. **Default Parameters**: Callers that don't pass FMF (like InstructionSimplify) use default FastMathFlags(), which is conservative but correct.

The fix appears robust and handles all the edge cases identified in Phase 1:
