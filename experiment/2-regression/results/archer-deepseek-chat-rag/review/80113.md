## Test Strategies

### Strategy 1: Variable Mask Position

- **Target**: Binary operator constant operand position
- **Rationale**: The fix only checks operand(1) for constant mask via match(BO->getOperand(1), m_APInt(Mask)). For commutative operations like and/or/xor, the constant could be in operand(0). This could miss optimization opportunities or require handling both operand positions.
- **Expected Issue**: Missed optimization when constant is in operand(0) instead of operand(1)

### Strategy 2: Multi-Use with Divergent Demands

- **Target**: Binary operator with multiple users having different demanded bit sets
- **Rationale**: DB.getDemandedBits(BO) returns union of demanded bits across all users. If some users need mask bits and others don't, the aggregated demanded bits may incorrectly allow simplification when it shouldn't.
- **Expected Issue**: Over-aggressive simplification breaking users that need mask bits

### Strategy 3: Vector with Partial Element Mask

- **Target**: Vector binary operations with element-wise constant masks
- **Rationale**: APInt represents the entire vector mask as a single integer. For vectors, different elements may have different mask values or some elements may be undef. The fix may incorrectly simplify based on aggregated mask bits.
- **Expected Issue**: Incorrect simplification for vector elements with different mask requirements

### Strategy 4: Poison-Generating Flag Interactions

- **Target**: Binary operators with NSW/NUW/exact flags
- **Rationale**: The fix calls clearAssumptionsOfUsers() which drops poison-generating flags. However, the simplification itself may affect poison semantics, especially for xor with constant that could create poison.
- **Expected Issue**: Lost optimization flags or incorrect poison semantics after simplification

### Strategy 5: Disjoint OR Semantics

- **Target**: or disjoint instructions with overlapping bits
- **Rationale**: or disjoint has special poison semantics when both inputs have 1 in same bit. Simplifying or disjoint %x, C to %x when Demanded ∩ Mask = ∅ may not preserve poison behavior if C has 1s in non-demanded bits.
- **Expected Issue**: Incorrect poison value generation when simplifying or disjoint

## Bugs Found

### Bug #1

**Analysis:**
This is a real bug! The transformation incorrectly simplifies `or disjoint %a, 3` to `%a` when demanded bits don't intersect with mask (3). However, when %a=3, the original `or disjoint` produces poison because both inputs have 1s in bits 0 and 1. The simplified version (`%a`) doesn't produce poison. The original execution returns poison (UB), while transformed execution returns 0 (and %a, 8 = and 3, 8 = 0). This violates poison semantics preservation.

**Original LLVM IR:**
```llvm

define i64 @or_disjoint_poison2(i64 %a) {
entry:
  %or = or disjoint i64 %a, 3
  %ret1 = and i64 %or, 8
  %ret2 = and i64 %or, 16
  ret i64 %ret1
}

define i64 @main(i32 %argc, ptr %argv) {
entry:
  %r = call i64 @or_disjoint_poison2(i64 3)
  ret i64 %r
}

```

### Bug #3

**Analysis:**
Another instance of the or disjoint bug! or disjoint with overlapping bits creates poison. The fix simplifies or disjoint to regular or (dropping disjoint flag) when demanded bits don't intersect with mask. But this loses poison semantics. Original returns poison, transformed returns 8 (or 56, 25 = 57, and 57, 8 = 0). Actually wait, 56|25=57, and 57&8=0. So transformed returns 0, original returns poison. Bug confirmed.

**Original LLVM IR:**
```llvm

define i64 @or_disjoint_of_and_poison(i64 %a, i64 %b) {
entry:

  %or = or disjoint i64 %and1, %and2
  %ret1 = and i64 %or, 8
  %ret2 = and i64 %or, 16
  ret i64 %ret1
}

define i64 @main(i32 %argc, ptr %argv) {
```
