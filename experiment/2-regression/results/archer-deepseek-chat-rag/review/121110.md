## Test Strategies

### Strategy 1: NonCommutativeCompareSwapping

- **Target**: Test operand swapping for non-commutative compare predicates (sgt, slt, ugt, ult, ogt, olt, ugt, ult)
- **Rationale**: The code uses isCommutative() to decide operand swapping, but many compare predicates are not commutative. Swapping operands for non-commutative predicates changes semantics.
- **Expected Issue**: Incorrect transformation changes comparison results for relational predicates

### Strategy 2: FCmpNaNHandling

- **Target**: Test FCmp with unordered predicates (UEQ, UNE, UGT, etc.) and NaN values
- **Rationale**: FCmp commutative property might have different behavior with NaN values. The transformation might not preserve NaN semantics correctly.
- **Expected Issue**: Different NaN behavior between original and transformed code

### Strategy 3: FastMathFlagsPreservation

- **Target**: Test FCmp with various fast-math flags (nnan, ninf, nsz, etc.)
- **Rationale**: Fast-math flags affect FCmp semantics. The transformation must preserve these flags when creating new compare.
- **Expected Issue**: Missing or incorrect fast-math flags on transformed compare

### Strategy 4: PoisonPropagationDivRem

- **Target**: Test compare operations with poison in shuffle mask (should not have div/rem restriction)
- **Rationale**: The poison safety check only applies to integer div/rem ops, but compares might have different poison propagation semantics.
- **Expected Issue**: Incorrect poison propagation for compares with poison mask elements

### Strategy 5: MixedVectorSizes

- **Target**: Test with different vector sizes between shuffle result and compare operands
- **Rationale**: The transformation creates ShuffleCmpTy with BinOpTy element type but ShuffleDstTy size. Need to ensure type construction is correct.
- **Expected Issue**: Type mismatch or incorrect type construction leading to compiler crash or miscompilation

### Strategy 6: PointerTypeICmp

- **Target**: Test ICmp with pointer types (both operands are pointers)
- **Rationale**: Pointer comparisons have different semantics than integer comparisons. The type check X->getType() != Z->getType() should handle pointer types.
- **Expected Issue**: Type checking fails for pointer comparisons or incorrect transformation

### Strategy 7: IRFlagsIntersection

- **Target**: Test compare instructions with different IR flags (exact, nsw, nuw for ICmp; fast-math for FCmp)
- **Rationale**: The code copies flags from LHS and ANDs with RHS flags, but compare flags might need special handling.
- **Expected Issue**: Incorrect flag preservation leading to missed optimizations or wrong semantics
