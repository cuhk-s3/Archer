## Test Strategies

### Strategy 1: WrappedRangeTruncFlags

- **Target**: Test with wrapped ConstantRange (e.g., [255, 10] wrapping around 0-255 boundary)
- **Rationale**: getActiveBits() on wrapped range may compute incorrectly, leading to wrong flag inference. Wrapped ranges represent values that cross the unsigned wrap boundary, which may not be handled correctly by the active bits calculation.
- **Expected Issue**: Incorrect nuw/nsw flags added when source values actually wrap during truncation

### Strategy 2: EmptyRangeTruncFlags

- **Target**: Test with empty ConstantRange from contradictory conditions
- **Rationale**: getActiveBits() returns 0 for empty sets, which would always satisfy <= DestWidth. Empty ranges represent unreachable code paths where the analysis shouldn't infer flags.
- **Expected Issue**: Flags added for unreachable code paths, potentially introducing poison in dead code

### Strategy 3: SignedMinTruncFlags

- **Target**: Test trunc of i8 value -128 to i7
- **Rationale**: getMinSignedBits() returns 8 for -128 (i8), but trunc to 7 bits loses the sign bit. The signed min value needs special handling because its representation changes with bit width.
- **Expected Issue**: Incorrect nsw flag added when sign would actually change during truncation

### Strategy 4: VectorMixedRangeTruncFlags

- **Target**: Test vector trunc with different possible ranges per element
- **Rationale**: Scalar analysis using getScalarSizeInBits() may miss element-specific range constraints. Different vector elements may have different value ranges that affect wrap behavior.
- **Expected Issue**: Flags added incorrectly when some vector elements would wrap but others wouldn't

### Strategy 5: UndefSourceTruncFlags

- **Target**: Test trunc with potentially undef source value
- **Rationale**: UndefAllowed=false in getConstantRange but source may still be undef in some execution paths. Adding flags to trunc of undef could change poison semantics.
- **Expected Issue**: Flags added when source could be undef, incorrectly constraining undefined behavior

I tested the patch using all 5 strategies identified in Phase 1:

2. **EmptyRangeTruncFlags**: Tested with contradictory conditions creating empty ranges. The patch handles these correctly, and SCCP often eliminates dead code paths before flag inference.

3. **SignedMinTruncFlags**: Tested with signed minimum values truncating to smaller bit widths. The `getMinSignedBits()` method correctly computes the needed bits for these boundary cases.

4. **VectorMixedRangeTruncFlags**: Tested vector scenarios (though llubi had issues with vector execution). Scalar analysis shows the patch works correctly.

5. **UndefSourceTruncFlags**: Tested truncations of undef values. The patch correctly propagates undef without adding incorrect flags.

### Key Findings

- The patch's conditions `Range.getActiveBits() <= DestWidth` for `nuw` and `Range.getMinSignedBits() <= DestWidth` for `nsw` are mathematically sound and conservative.
- `ConstantRange` methods handle edge cases correctly: empty ranges return 0, wrapped ranges compute maximum bits needed.
- The implementation is safe - it only adds flags when mathematically guaranteed.
- The early return optimization when both flags exist is correct.
