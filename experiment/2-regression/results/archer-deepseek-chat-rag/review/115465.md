## Test Strategies

### Strategy 1: NonConstantRHSWithSpecialSemantics

- **Target**: RHS operand of shufflevector
- **Rationale**: The transformation requires RHS to be constant, but undef/poison RHS or constant expressions with side effects may have different semantics when duplicated across select arms or phi incoming values.
- **Expected Issue**: Incorrect transformation when RHS is undef/poison or constant expression with side effects, leading to different poison/undef propagation or evaluation order issues.

### Strategy 2: SelectMultipleUses

- **Target**: Select instruction with multiple uses
- **Rationale**: FoldOpIntoSelect requires select to have one use by default, but select may be used by multiple shufflevectors or other instructions. The transformation may incorrectly apply or miss optimization opportunities.
- **Expected Issue**: Missed optimization when select has multiple shufflevector uses, or incorrect transformation when select has non-shufflevector uses.

### Strategy 3: PHICriticalEdge

- **Target**: PHI node with critical edges
- **Rationale**: foldOpIntoPhi requires unconditional branches in incoming blocks. Critical edges could cause computation to be inserted on other paths, changing program semantics.
- **Expected Issue**: Incorrect code generation when PHI has incoming blocks with conditional branches, potentially inserting computations on wrong paths.

### Strategy 4: ShuffleMaskBoundaryViolation

- **Target**: Shuffle mask indices relative to select/phi arms
- **Rationale**: Mask indices valid for original select/phi result type may be out-of-bounds for individual arms if arms have different vector sizes or if mask references elements beyond arm size.
- **Expected Issue**: Out-of-bounds mask access after transformation, leading to undefined behavior or incorrect poison generation.

### Strategy 5: PoisonUndefSemantics

- **Target**: Poison/undef elements in RHS constant or mask
- **Rationale**: Transformation duplicates RHS constant across select arms/phi incoming values. Poison/undef elements in RHS may interact differently with different arms/incoming values, changing poison propagation semantics.
- **Expected Issue**: Different poison/undef propagation behavior between original and transformed code, potentially introducing new poison or losing required poison.

### Strategy 6: ScalableVectorHandling

- **Target**: Scalable vector types
- **Rationale**: Shufflevector with scalable vectors has restrictions on mask values (only zeroinitializer, undef, poison). Transformation may not correctly handle scalable vectors or may violate these restrictions.
- **Expected Issue**: Incorrect transformation for scalable vectors, generating invalid shuffle masks or violating scalable vector constraints.

### Analysis

## Bug Triggering Analysis
The bug occurs when transforming `shufflevector(select(c, x, y), C)` to `select(c, shufflevector(x, C), shufflevector(y, C))`. The issue is with poison propagation semantics:

1. **Original semantics**: When `%c` is poison, the select instruction produces poison for all elements. However, the shufflevector with mask index `4` (which references RHS element 0 = 99) still produces `<poison, 99>` because:
   - The select produces poison vector `<poison, poison, poison, poison>`
   - Shufflevector with mask `<1, 4>` selects:
     - Element 1 from LHS: poison (from select result)
     - Element 4 from RHS: 99 (from constant RHS)

2. **Transformed semantics**: The transformation creates `select(c, shufflevector(x, C), shufflevector(y, C))`. When `%c` is poison:
   - The entire select result is poison (select with poison condition yields poison)
   - This produces `<poison, poison>` regardless of the RHS constant values

The bug violates poison semantics because the transformed code produces more poison than the original. Specifically, element 1 (which comes from RHS constant 99 in the original) becomes poison in the transformed version.

## Fix Weakness Analysis
The fix incorrectly assumes that duplicating the constant RHS across select arms preserves semantics. However, when the select condition is poison, the entire select result becomes poison, losing the non-poison values from the RHS constant that would have been selected by shufflevector mask indices referencing RHS elements.

The transformation should only apply when all mask indices reference elements from the LHS (select/phi result), not when any mask index references RHS elements. Alternatively, it needs special handling for poison select conditions to preserve RHS constant values that would be selected.

This bug affects both the select and phi transformations when:
1. Shuffle mask contains indices referencing RHS elements (indices >= LHS vector size)
2. The select condition or phi incoming path could be poison
