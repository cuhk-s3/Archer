## Test Strategies

### Strategy 1: Non-commutative predicate with swapped common operand position

- **Target**: Test case with non-commutative icmp/fcmp predicate where common operand is in different positions (op0 in one, op1 in other)
- **Rationale**: The code only swaps operands when IsCommutative is true (line 1668). For non-commutative predicates like sgt/ult/ogt, if we have 'icmp sgt X, Y' and 'icmp sgt Z, X', the transformation may incorrectly handle the operand ordering or fail to recognize the pattern.
- **Expected Issue**: The transformation might either incorrectly transform the pattern (swapping operands without adjusting predicate) or miss a valid optimization opportunity

### Strategy 2: FCmp with different fast-math flags

- **Target**: Test case with two fcmp instructions having different fast-math flags (e.g., one with nnan, one without)
- **Rationale**: The code intersects IR flags at line 1727-1730. For fcmp instructions with different fast-math flags, the intersection might produce incorrect semantics if flags are not properly handled.
- **Expected Issue**: The transformed fcmp might have incorrect fast-math flags, potentially changing the semantics (e.g., allowing NaN optimizations that weren't valid in the original)

### Strategy 3: Shuffle mask with poison elements for compares

- **Target**: Test case with shuffle mask containing poison elements feeding compare instructions
- **Rationale**: The original code had a poison check for div/rem (line 1650-1651) but no equivalent for compares. While compares don't trap on poison divisors, the transformation semantics with poison mask elements need verification.
- **Expected Issue**: The transformation might incorrectly handle poison elements in the shuffle mask, potentially introducing undefined behavior or incorrect results

### Strategy 4: Mixed ordered/unordered fcmp predicates

- **Target**: Test case with fcmp using ordered (oeq, olt) vs unordered (ueq, ult) predicates with potential NaN inputs
- **Rationale**: FCmp predicates have different semantics regarding NaN handling. The transformation should preserve these semantics. Testing with NaN-producing inputs can verify correctness.
- **Expected Issue**: The transformation might not correctly preserve the ordered/unordered semantics, especially when NaN values are involved

### Strategy 5: ICmp with pointer operands

- **Target**: Test case with icmp comparing pointer values (not just integers)
- **Rationale**: ICmp can compare pointers as well as integers. The transformation should handle pointer-typed vector elements correctly. This tests the type handling at lines 1659-1663.
- **Expected Issue**: The transformation might incorrectly handle pointer-typed comparisons, especially with address space considerations

## Summary of Fix Analysis and Testing

### Fix Understanding
The patch extends `VectorCombine::foldShuffleOfBinops` to handle compare instructions (icmp/fcmp) in addition to binary operators. The transformation converts patterns like:
- `shuffle (cmp X, Y), (cmp Z, W)` → `cmp (shuffle X, Z), (shuffle Y, W)`

### Key Changes
1. Pattern matching now uses `m_Cmp`/`m_SpecificCmp` in addition to `m_BinOp`
2. Commutativity is determined via `CmpInst::isCommutative()` which returns true only for equality predicates (eq/ne) for icmp, and equality/ord/uno/false/true for fcmp
3. Cost calculation uses `TTI.getCmpSelInstrCost()` for compares
4. Instruction creation uses `Builder.CreateCmp()` for compares

### Testing Results
All 6 test cases were tested with all 5 strategies:
1. **shuf_icmp_sgt_v4i32_swap**: Tests non-commutative predicate (sgt) with common operand in same position
2. **shuf_icmp_eq_v4i32**: Tests commutative predicate (eq) with all different operands
3. **shuf_icmp_eq_v2i64**: Tests commutative predicate with 64-bit integers
4. **shuf_icmp_ugt_v4i32**: Tests non-commutative predicate (ugt) with all different operands
5. **shuf_fcmp_oeq_v4i32**: Tests fcmp with ordered-equal predicate and poison mask elements
6. **shuf_fcmp_one_v4f32_swap**: Tests fcmp with ordered-not-equal predicate and swapped common operand

All transformations were verified to be correct:
- Alive2 verification showed all transformations are syntactically equal or semantically equivalent
- Differential testing with lli showed no execution differences between original and transformed code

### No Bugs Found
The fix correctly handles:
- Non-commutative predicates (sgt, ugt) - no incorrect operand swapping
- Commutative predicates (eq, one) - correct operand matching and swapping when beneficial
- Poison elements in shuffle masks - correctly propagated
- Different compare types (icmp, fcmp) - proper type handling
- Fast-math flags intersection for fcmp

The transformation is conservative and only applies when cost-beneficial, ensuring correctness.
