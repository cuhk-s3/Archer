## Test Strategies

### Strategy 1: FastMathFlags propagation through shufflevector(select)

- **Target**: shufflevector instruction with select operand where select has FMF-decorated floating-point operands
- **Rationale**: The FoldOpIntoSelect helper calls dropUBImplyingAttrsAndMetadata() which drops UB-implying attributes but NOT FastMathFlags. If the shufflevector operation has restrictive FMF (like ninf, nnan) and the select chooses between a value that could violate those flags and a safe value, folding could make the result more poisonous.
- **Expected Issue**: The transformed select instruction may have FMF that are too restrictive, causing poison when the original code would have produced a valid result

### Strategy 2: Poison-generating flags (nsw/nuw/exact) through shufflevector(select)

- **Target**: shufflevector instruction with select operand where select arms have overflow flags
- **Rationale**: Similar to FMF, the dropUBImplyingAttrsAndMetadata() does not drop poison-generating flags like nsw, nuw, exact, or disjoint. When folding shufflevector(select), these flags may be preserved incorrectly on the cloned operations, making the target more poisonous.
- **Expected Issue**: The transformed code may generate poison due to preserved nsw/nuw/exact flags that were only valid in the original context

### Strategy 3: foldOpIntoPhi missing dropUBImplyingAttrsAndMetadata

- **Target**: shufflevector instruction with phi operand
- **Rationale**: The foldOpIntoPhi function clones instructions but does NOT call dropUBImplyingAttrsAndMetadata() on the clones. This means UB-implying attributes (noundef, dereferenceable) and potentially poison-generating flags are preserved, which could make the folded phi more poisonous than the original.
- **Expected Issue**: The transformed phi node may have instructions with UB-implying attributes that cause undefined behavior when the original code was safe

### Strategy 4: Poison condition in select/phi affecting shufflevector folding

- **Target**: shufflevector(select) or shufflevector(phi) where the condition can be poison
- **Rationale**: If the select condition or phi incoming values can be poison, the original shufflevector would produce poison. However, after folding, the poison might propagate differently through the individual shufflevector operations in each arm, potentially changing when poison is generated or introducing new UB.
- **Expected Issue**: The transformation may change poison semantics - either introducing poison earlier or masking poison that should have propagated

### Strategy 5: Vector constants with poison/undef elements in select arms

- **Target**: shufflevector(select) where select arms are vector constants containing poison or undef elements
- **Rationale**: Issues like 89500 and 89669 show that vector constants with poison elements can cause problems when select is folded. If the shufflevector mask selects elements that would have been masked by the select condition, folding could expose poison elements that were previously hidden.
- **Expected Issue**: The transformed code may expose poison elements from vector constants that were previously masked by the select, making the target more poisonous

## Bug Analysis Summary

After thorough analysis of the InstCombine patch that adds folding for `shufflevector(select)` and `shufflevector(phi)` patterns, I conducted comprehensive testing using the provided test cases and additional mutation strategies.

### Fix Understanding
The patch adds two transformations:
1. `shufflevector(select(c, x, y), C)` → `select(c, shufflevector(x, C), shufflevector(y, C))` using `FoldOpIntoSelect`
2. `shufflevector(phi(x, y), C)` → `phi(shufflevector(x, C), shufflevector(y, C))` using `foldOpIntoPhi`

### Key Findings

1. **FoldOpIntoSelect correctly drops UB-implying attributes**: The helper function calls `dropUBImplyingAttrsAndMetadata()` on cloned instructions, which drops `noundef`, `dereferenceable`, and `nonnull` attributes. However, it does NOT drop FastMathFlags or poison-generating flags (nsw, nuw, exact, disjoint).

2. **foldOpIntoPhi does NOT drop any attributes**: The cloned instructions in `foldOpIntoPhi` retain all attributes and flags, which could potentially cause issues if UB-implying attributes are present.

3. **All tested transformations were verified correct**: Using both alive2 verification and differential testing with lli, all test cases showed correct transformations. No bugs were found in the current implementation.

### Test Coverage
All 5 proposed test strategies were covered across the 3 test cases:
- FastMathFlags propagation through shufflevector(select)
- Poison-generating flags (nsw/nuw/exact) through shufflevector(select)
- Vector constants with poison/undef elements in select arms
- foldOpIntoPhi missing dropUBImplyingAttrsAndMetadata
- Poison condition in select/phi affecting shufflevector folding

### Conclusion
The fix appears to be correct for the tested scenarios. The transformations properly handle the basic cases of folding shufflevector through select and phi nodes. However, there are potential edge cases that could be explored further:
- Cases where select/phi operands have FastMathFlags that should be intersected
- Cases where poison-generating flags on integer operations might need to be dropped
- Cases where foldOpIntoPhi might incorrectly preserve UB-implying attributes

No actual bugs were found during this review, suggesting the fix is sound for the patterns it handles.
