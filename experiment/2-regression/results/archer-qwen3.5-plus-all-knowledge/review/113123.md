## Test Strategies

### Strategy 1: NaN Sign Bit Preservation Test

- **Target**: Test with NaN inputs to verify fabs preserves NaN sign bit correctly
- **Rationale**: The fabs intrinsic preserves NaN sign bits according to LLVM langref. However, the original fcmp comparisons may have different behavior. For ordered predicates (olt, ole, ogt, oge), NaN inputs return false. For unordered predicates (ult, ule, ugt, uge), NaN inputs return true. The transformation must preserve this semantics. Test with signaling NaN, quiet NaN, and negative NaN to ensure the transformation is correct.
- **Expected Issue**: If the transformation incorrectly handles NaN sign bits, the fabs result may have a different sign bit than expected, causing the fcmp to produce different results for NaN inputs, especially with unordered predicates.

### Strategy 2: FastMathFlags Union vs Intersection Test

- **Target**: Test with different FMF combinations on the two fcmps
- **Rationale**: The code unions FMF from both fcmps (LHS->getFastMathFlags() | RHS->getFastMathFlags()). This is potentially unsound because if one fcmp has nnan and the other doesn't, the union will have nnan, but the original code without nnan on one side could handle NaN differently. The fabs intrinsic created will have the union of flags, which may be too aggressive.
- **Expected Issue**: If one fcmp has nnan flag and the other doesn't, the union may incorrectly allow the fabs to assume no NaN, leading to different behavior when NaN is actually present.

### Strategy 3: Signed Zero Handling Test

- **Target**: Test with -0.0 and +0.0 inputs
- **Rationale**: fabs(-0.0) = +0.0. The original pattern compares x with C and -C. If C is positive and x is -0.0, the original comparisons are: fcmp olt -0.0, C (true if C > 0), fcmp ogt -0.0, -C (depends on -C). After transformation: fabs(-0.0) = +0.0, fcmp olt +0.0, C. Need to verify these are equivalent for all cases including signed zeros.
- **Expected Issue**: The fabs transformation may not correctly preserve the semantics when dealing with signed zeros, especially with predicates that distinguish between -0.0 and +0.0.

### Strategy 4: Vector Type with Mixed NaN Lanes Test

- **Target**: Test vector types where some lanes are NaN and others are normal values
- **Rationale**: For vector types, each lane is processed independently. If some lanes have NaN values and others have normal values, the fabs operation will preserve NaN in some lanes while computing absolute value in others. The fcmp comparison must produce the correct result for each lane. This tests whether the transformation handles per-lane semantics correctly.
- **Expected Issue**: The transformation may incorrectly handle vector types with mixed NaN/normal lanes, causing some lanes to produce incorrect comparison results.

### Strategy 5: Predicate Swap Logic Edge Case Test

- **Target**: Test all combinations of predicates to verify the swap logic is correct
- **Rationale**: The code has complex logic for swapping predicates based on whether PredL or PredR is a 'less than' predicate. For AND: if PredR is less-than, swap. For OR: if PredL is less-than, swap. Then it checks if the resulting PredL (for AND) or PredR (for OR) is less-than. This logic needs to be verified for all 16 combinations of (olt, ole, ult, ule) x (ogt, oge, ugt, uge) swapped pairs.
- **Expected Issue**: The predicate swap logic may have bugs for certain predicate combinations, causing the wrong predicate to be used in the final fcmp, leading to incorrect comparison results.

### Analysis

## Summary of Analysis

The patch canonicalizes floating-point range check idioms into `fabs + fcmp` form:
- `and (fcmp olt/ole/ult/ule x, C), (fcmp ogt/oge/ugt/uge x, -C)` → `fabs(x) olt/ole/ult/ule C`
- `or (fcmp ogt/oge/ugt/uge x, C), (fcmp olt/ole/ult/ule x, -C)` → `fabs(x) ogt/oge/ugt/uge C`

### Key Findings:

1. **FMF Union Issue**: The code unions FastMathFlags from both fcmps (`LHS->getFastMathFlags() | RHS->getFastMathFlags()`). This could be problematic if one fcmp has `nnan` and the other doesn't - the union would have `nnan`, potentially making the fabs/fcmp more restrictive than the original.

2. **NaN Handling**: The transformation correctly handles NaN for both ordered and unordered predicates:
   - Ordered predicates (olt, ole, ogt, oge): NaN returns false for both original fcmps and the transformed fcmp
   - Unordered predicates (ult, ule, ugt, uge): NaN returns true for both original fcmps and the transformed fcmp

3. **Signed Zero**: fabs(-0.0) = +0.0, and the transformation correctly preserves semantics for signed zeros.

4. **Vector Types**: The transformation handles vector types with undef elements correctly.
