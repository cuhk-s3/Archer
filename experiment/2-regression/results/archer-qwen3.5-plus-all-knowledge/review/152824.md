## Test Strategies

### Strategy 1: Test NaN sign bit inference with knownNot

- **Target**: ValueTracking computeKnownFPClass with NaN values
- **Rationale**: The knownNot() method infers SignBit when isKnownNever(fcNan) is true and SignBit is not yet set. However, if NaN values with specific sign bits are possible, the SignBit inference could be incorrect. Test with vector constants containing NaN elements with different sign bits.
- **Expected Issue**: Incorrect sign bit inference when NaN values with different sign bits are possible, leading to wrong optimization decisions in InstCombine

### Strategy 2: Test vector constant sign bit with mixed NaN signs

- **Target**: computeKnownFPClass vector constant handling (lines 4315-4346)
- **Rationale**: The vector constant handling tracks SignBitAllZero and SignBitAllOne separately, but only sets Known.SignBit if they differ. If some elements are NaN with different sign bits, the logic may incorrectly conclude a known sign bit.
- **Expected Issue**: Vector constants with NaN elements having different sign bits may incorrectly report a known sign bit

### Strategy 3: Test min/max intrinsic sign bit inference with NaN operands

- **Target**: min/max intrinsic handling in computeKnownFPClass (lines 4598-4617)
- **Rationale**: The sign bit inference for min/max intrinsics at lines 4598-4617 only checks isKnownNeverNaN() but the SignBit inference logic may not properly handle cases where one operand could be NaN with a specific sign. The condition checks KnownLHS.SignBit and KnownRHS.SignBit but doesn't verify both are known.
- **Expected Issue**: Incorrect sign bit inference for min/max when operands could be NaN with specific signs

### Strategy 4: Test FMul sign bit inference with NaN operands

- **Target**: FMul handling in computeKnownFPClass (lines 4967-4972)
- **Rationale**: The FMul sign bit inference at lines 4967-4972 checks if KnownLHS.SignBit and KnownRHS.SignBit are set, but doesn't verify that NaN is ruled out before making sign bit conclusions. This could lead to incorrect sign bit inference when NaN values are possible.
- **Expected Issue**: FMul may incorrectly infer sign bit when NaN operands with specific signs are possible

### Strategy 5: Test copysign with NaN sign operand

- **Target**: copysign intrinsic handling (lines 4457-4465)
- **Rationale**: The copysign handling copies SignBit from the sign operand. If the sign operand could be NaN, the sign bit behavior for NaN is implementation-defined. The copysign method preserves SignBit even for NaN, which may not be correct for all targets.
- **Expected Issue**: copysign may incorrectly preserve sign bit information when the sign operand could be NaN
