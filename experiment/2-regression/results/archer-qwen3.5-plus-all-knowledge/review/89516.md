## Test Strategies

### Strategy 1: Poison condition in select

- **Target**: foldSelectWithSRem function
- **Rationale**: The transformation removes the conditional select and replaces it with an unconditional bitwise AND. If the condition of the original select can be poison, the original code would return poison when the condition is poison, but the transformed code unconditionally executes the AND operation. This could make the target less poisonous than the source in some cases, or could change behavior when the condition is poison.
- **Expected Issue**: The transformation may incorrectly proceed when the select condition can be poison, leading to different poison propagation behavior between source and target.

### Strategy 2: Vector divisor with poison elements

- **Target**: isKnownToBeAPowerOfTwo check for vector types
- **Rationale**: When the divisor is a vector type, isKnownToBeAPowerOfTwo checks if all elements are powers of two. However, if some elements are poison, the analysis might still return true. The original srem with poison divisor elements would produce poison for those lanes, but the transformed x & (n-1) might produce different results depending on how poison propagates through the constant creation and AND operation.
- **Expected Issue**: For vector types with poison elements in the divisor, the transformation may produce different poison behavior than the original code.

### Strategy 3: Divisor value of zero (OrZero=true)

- **Target**: isKnownToBeAPowerOfTwo with OrZero parameter
- **Rationale**: The isKnownToBeAPowerOfTwo check uses OrZero=true, meaning the divisor could potentially be zero. When n=0, the original srem is undefined behavior. The transformation creates x & (0-1) = x & (-1) = x. While both are technically UB, the specific behavior might differ. More importantly, if the divisor is known to be zero in some lanes of a vector, the behavior could differ.
- **Expected Issue**: When the divisor can be zero, the transformation might produce well-defined results (x) instead of UB, potentially changing program behavior in edge cases.

### Strategy 4: Non-power-of-two divisor that appears as power-of-two due to known bits

- **Target**: isKnownToBeAPowerOfTwo analysis precision
- **Rationale**: The isKnownToBeAPowerOfTwo function uses known bits analysis. If the divisor's known bits indicate it's a power of two, but the actual runtime value might not be (due to undef/poison bits), the transformation could be incorrect. The transformation assumes the divisor is always a power of two, but if there are undef bits that could make it non-power-of-two at runtime, the equivalence breaks.
- **Expected Issue**: If the divisor has undef bits that could make it non-power-of-two at runtime, the transformation to bitwise AND would be incorrect for those runtime values.

### Strategy 5: Overflow in n-1 calculation for n=INT_MIN

- **Target**: CreateAdd for computing n-1
- **Rationale**: When computing n-1 using CreateAdd(Remainder, getAllOnesValue()), if Remainder is INT_MIN (which is a power of two: 2^(N-1)), then n-1 would overflow. For example, for i8, if n=-128 (0x80), then n-1 = -129 which wraps to 127 (0x7F). The transformation x & 127 is not equivalent to the Euclidean remainder pattern when the divisor is INT_MIN. However, INT_MIN is technically a power of two, so isKnownToBeAPowerOfTwo would return true.
- **Expected Issue**: When the divisor is INT_MIN (the sign bit), the n-1 calculation overflows, producing an incorrect mask for the AND operation.
