## Test Strategies

### Strategy 1: Test trunc nsw + zext with signed comparison

- **Target**: icmp-of-trunc-ext.ll
- **Rationale**: The patch claims trunc nsw + zext/sext works for all predicates including signed. The test file has trunc_signed_nsw_zext which should fold. Need to verify the YIsSExt logic correctly identifies zext vs sext to use the right cast.
- **Expected Issue**: If YIsSExt is incorrectly computed, the fold might use zext instead of sext (or vice versa), producing wrong results for negative values.

### Strategy 2: Test trunc nsw + sext with signed comparison where extension operand is negative

- **Target**: icmp-of-trunc-ext.ll
- **Rationale**: For trunc nsw + sext with signed comparison, the sext must be preserved. If YIsSExt is incorrectly false when it should be true, the code will use zext instead of sext, changing the sign extension behavior.
- **Expected Issue**: Negative values in the extension operand would be incorrectly zero-extended instead of sign-extended, producing wrong comparison results.

### Strategy 3: Test trunc nsw + zext with signed comparison where zext operand could be negative after trunc

- **Target**: icmp-of-trunc-ext.ll
- **Rationale**: The patch allows trunc nsw + zext for signed comparisons. But zext always produces non-negative values. If the original trunc nsw value is negative, comparing it with a zext value using signed comparison might have different semantics than the folded version.
- **Expected Issue**: The transformation might be invalid for certain value ranges where trunc nsw produces negative values but zext produces non-negative values.

### Strategy 4: Test mixed nuw/nsw trunc operands with unsigned comparison

- **Target**: icmp-of-trunc-ext.ll
- **Rationale**: The test file has trunc_unsigned_either and trunc_signed_either which have one nuw and one nsw trunc. According to the patch, for unsigned comparisons, NoWrapFlags = nuw & nsw = 0, so these should NOT fold. But the test shows they don't fold (CHECK shows the original IR). Need to verify this is correct behavior.
- **Expected Issue**: If the intersection logic is wrong, mixed nuw/nsw might incorrectly fold or not fold.

### Strategy 5: Test trunc nsw + sext pattern where the operand order is swapped

- **Target**: icmp-of-trunc-ext.ll
- **Rationale**: The m_c_ICmp matcher is commutative, so the trunc could be on either side. The YIsSExt check uses isa<SExtInst>(Cmp.getOperand(0)) || isa<SExtInst>(Cmp.getOperand(1)), but this checks the wrong operands - it checks the trunc results, not the extension instruction Y.
- **Expected Issue**: When the extension operand is on the left side of the comparison, YIsSExt might be incorrectly computed, leading to wrong cast type in the folded comparison.
