## Test Strategies

### Strategy 1: Test with non-integer function arguments

- **Target**: Function with pointer or vector arguments
- **Rationale**: The fix adds Var >= 0 constraints for all function arguments in the constructor without checking if they are integer types. This could cause issues if the constraint system doesn't handle non-integer types correctly.
- **Expected Issue**: Potential assertion failure or incorrect constraint handling when function has pointer/vector arguments that get Var >= 0 constraints added

### Strategy 2: Test variable introduced in signed context then used in unsigned

- **Target**: Code where a variable is first used in a signed comparison then in an unsigned comparison
- **Rationale**: The fix only adds Var >= 0 constraints for unsigned predicates. If a variable is introduced in a signed context, it won't get the Var >= 0 constraint in the unsigned system, potentially causing missed optimizations or incorrect reasoning.
- **Expected Issue**: Missed optimization or incorrect constraint propagation when variable transitions from signed to unsigned context

### Strategy 3: Test recursive addFact through transferToOtherSystem with new variables

- **Target**: Code that triggers transferToOtherSystem which may recursively call addFact
- **Rationale**: When transferToOtherSystem calls addFact recursively, it may add Var >= 0 constraints for new variables. The assertion at the end should account for these, but there could be edge cases where the stack/constraint synchronization is off.
- **Expected Issue**: Assertion failure at line 1734-1736 due to mismatch between constraint count and stack entry count after recursive addFact calls

### Strategy 4: Test with multiple new variables in single unsigned fact

- **Target**: Unsigned comparison involving multiple new variables simultaneously
- **Rationale**: When an unsigned fact introduces multiple new variables, each gets a Var >= 0 constraint and a stack entry. The order of adding constraints and entries, and the order of removal, must be correct to maintain synchronization.
- **Expected Issue**: Stack/constraint desynchronization when removing entries, potentially causing wrong variables to be released or assertion failures

### Strategy 5: Test min/max intrinsic with unsigned predicate introducing new variable

- **Target**: Code using umin/umax intrinsics whose result is used in subsequent comparisons
- **Rationale**: Min/max intrinsics add facts to the system (lines 1696-1700). The result variable is new and should get Var >= 0 constraint. Testing this ensures the fix handles intrinsic results correctly.
- **Expected Issue**: Missing Var >= 0 constraint for min/max intrinsic result, leading to missed optimization in subsequent unsigned comparisons
