## Test Strategies

### Strategy 1: Poison-generating samesign flag on icmp

- **Target**: select with icmp samesign condition
- **Rationale**: The fix uses computeKnownBitsFromCond to derive known bits from the select condition. However, if the condition is an icmp with samesign flag, the comparison can produce poison when operands have different signs. The fix doesn't check for poison-generating flags on the condition, so it might incorrectly simplify select arms based on a condition that could be poison.
- **Expected Issue**: The select arm is incorrectly folded to a constant when the condition has a samesign flag that could produce poison, making the optimized code more poisonous than the original.

### Strategy 2: Vector condition with poison elements

- **Target**: select with vector condition containing poison
- **Rationale**: The fix doesn't explicitly handle vector conditions where some lanes might be poison. When findValuesAffectedByCondition processes a vector condition with poison elements, the AffectedValues set might not correctly track which lanes are affected. The hasAffectedValue function also doesn't check for poison in vector conditions.
- **Expected Issue**: Select arms are incorrectly simplified for vector lanes where the condition is poison, causing poison to propagate where it shouldn't.

### Strategy 3: Select arm with poison-generating flags

- **Target**: select arm containing instructions with nsw/nuw flags
- **Rationale**: When computing known bits for a select arm, the fix doesn't account for poison-generating flags on instructions within the arm. Even if the condition implies certain known bits, if the select arm contains an instruction with nsw/nuw that could overflow, the result could be poison. The fix might fold such an arm to a constant incorrectly.
- **Expected Issue**: A select arm with nsw/nuw flags is folded to a constant, but the original arm could produce poison on overflow, making the optimized code less poisonous than it should be.

### Strategy 4: Condition with undef value

- **Target**: select with condition involving undef
- **Rationale**: The fix doesn't distinguish between undef and poison in the condition. If the condition involves an undef value, the known bits derived from the condition might be incorrect because undef can take different values at different uses. The fix might incorrectly assume the condition provides reliable known bits information.
- **Expected Issue**: Select arms are incorrectly simplified based on a condition involving undef, leading to incorrect constant folding when undef takes different values.

### Strategy 5: Nested select with conflicting conditions

- **Target**: nested select instructions with related conditions
- **Rationale**: When a select arm contains another select with a related condition, the fix might incorrectly compute known bits. The hasAffectedValue function recursively checks operands, but the interaction between multiple conditions could lead to incorrect known bits computation if the conditions have complex relationships.
- **Expected Issue**: Nested select instructions are incorrectly simplified because the known bits computation doesn't properly handle the interaction between multiple conditions.

### Bug #2

**Analysis:**
This is a real bug. The verify tool found that the transformation is incorrect. The original code with %x = <0, 2> produces:
- %x10 = shufflevector %x, undef, <1, 0> = <2, 0>
- %cond = icmp eq %x, <0, 0> = <true, false>
- %s = select <true, false>, <2, 0>, <0, 2> = <2, 2>

But the optimized code just returns %x = <0, 2>.

The bug is that the fix incorrectly simplifies the select by using known bits from the condition. When the condition is %x == 0, the fix assumes that in the true branch, %x is known to be 0. However, for vector types, the condition is element-wise, and the fix doesn't properly handle the case where only some lanes of the condition are true.

In this case, when lane 0 of the condition is true, %x[0] is known to be 0. But the fix incorrectly applies this knowledge to simplify %x10, which is a shuffle of %x. The shuffle swaps the lanes, so %x10[0] = %x[1], not %x[0]. The fix doesn't account for this lane reordering when computing known bits.

The llubi crash is due to the test harness trying to execute code with poison/undef values, but the key issue is the value mismatch found by verify.

### Analysis

### Bug Triggering Analysis

The bug is triggered by the fix in commit 77eb05683082dd3751ccfab9963f5160f1852058d which adds optimization to simplify select instructions using KnownBits from the condition.

**Test case:**
```llvm
define <2 x i32> @vec_select_no_equivalence(<2 x i32> %x) {
  %x10 = shufflevector <2 x i32> %x, <2 x i32> undef, <2 x i32> <i32 1, i32 0>
  %cond = icmp eq <2 x i32> %x, zeroinitializer
  %s = select <2 x i1> %cond, <2 x i32> %x10, <2 x i32> %x
  ret <2 x i32> %s
}
```

**Why it triggers the bug:**
When %x = <0, 2>:
1. %x10 = shufflevector %x, undef, <1, 0> = <2, 0> (swaps lanes)
2. %cond = icmp eq %x, <0, 0> = <true, false>
3. %s = select <true, false>, <2, 0>, <0, 2> = <2, 2>

The fix incorrectly simplifies this to just return %x = <0, 2>.

**Root cause:** The fix uses `computeKnownBitsFromCond` to derive known bits from the select condition. When the condition is `%x == 0`, it correctly determines that in the true branch, %x is known to be 0. However, for vector types with element-wise conditions, the fix doesn't properly handle lane reordering caused by shuffle operations.

When lane 0 of the condition is true, %x[0] is known to be 0. But %x10[0] = %x[1] (due to the shuffle), not %x[0]. The fix incorrectly applies the known bits from %x to %x10 without accounting for the lane permutation, leading to incorrect constant folding.

### Fix Weakness Analysis

The fix assumes that if a value V is in `AffectedValues` (meaning V's known bits are affected by the condition), then computing known bits for V with the condition context will correctly refine the known bits. However, this assumption breaks when:

1. **Vector types with element-wise conditions**: The condition applies per-lane, but the fix doesn't track which lanes are affected.

2. **Shuffle/reordering operations**: When V is derived from the condition's affected values through a shuffle or other lane-reordering operation, the known bits don't map correctly between lanes.

3. **The `hasAffectedValue` function** recursively checks if any operand is affected, but doesn't account for how the operation transforms the relationship between condition lanes and result lanes.

The fix needs to either:
- Not apply this optimization for vector types with non-trivial shuffles
- Track per-lane affected values and known bits
- Check that the value's known bits can be correctly derived from the condition without lane reordering
