## Test Strategies

### Strategy 1: Test FMF intersection when fsub has flags but fcmp doesn't

- **Target**: foldFCmpFSubIntoFCmp function
- **Rationale**: The current code preserves the fcmp's FastMathFlags but doesn't consider intersecting with fsub's flags. If fsub has nnan/ninf but fcmp doesn't, the transformed fcmp should potentially inherit these flags to enable further optimizations, or the transformation might be too conservative.
- **Expected Issue**: Missing optimization opportunities or incorrect FMF handling when fsub has flags that fcmp lacks

### Strategy 2: Test vector types with mixed infinity knowledge per lane

- **Target**: isKnownNeverInfinity check for vectors
- **Rationale**: The isKnownNeverInfinity function returns true only if ALL vector elements are known to be non-infinity. If some lanes are known non-infinity but others are unknown, the transformation is blocked entirely. This might be overly conservative for vectors where per-lane analysis could allow partial transformations.
- **Expected Issue**: Overly conservative blocking of valid transformations for vector types with partial infinity knowledge

### Strategy 3: Test denormal mode edge cases

- **Target**: DenormalMode::getIEEE() check
- **Rationale**: The transformation requires IEEE denormal mode. Need to test cases where the function has different denormal modes (positive-zero, dynamic, preserve-sign) to ensure the transformation is correctly blocked in non-IEEE modes.
- **Expected Issue**: Incorrect transformation in non-IEEE denormal modes leading to different behavior with denormal values

### Strategy 4: Test fcmp with ninf flag but fsub without ninf flag

- **Target**: FMF preservation in transformation
- **Rationale**: If fcmp has ninf flag (assuming result is not infinity) but fsub doesn't have ninf, the transformation might be unsafe. The original fcmp ninf assumes (x-y) is not infinity, but after transformation, fcmp ninf assumes x and y are not infinity - these are different conditions. Example: x=+inf, y=+inf gives x-y=NaN (not infinity), so original is valid, but transformed fcmp ninf would be invalid.
- **Expected Issue**: Miscompilation when fcmp has ninf but fsub doesn't, and x/y can be same-sign infinities

### Strategy 5: Test one-use restriction with side effects

- **Target**: hasOneUse() check before transformation
- **Rationale**: The transformation only proceeds if fsub has one use. Need to test edge cases where fsub might have apparent one use but the use chain has side effects or the fsub is used in ways that affect the transformation safety.
- **Expected Issue**: Potential issues with instruction ordering or side effects when fsub has complex use patterns
