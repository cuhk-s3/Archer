## Test Strategies

### Strategy 1: Test with poison-generating flags on select arm operands

- **Target**: Select instruction where one arm contains an instruction with poison-generating flags (nsw, nuw, exact, samesign)
- **Rationale**: The fix computes known bits from the condition context and may simplify a select arm to a constant. However, if the select arm contains instructions with poison-generating flags, simplifying based on known bits could incorrectly ignore the poison semantics. The condition might guard against poison in the original code, but the constant replacement could lose this protection.
- **Expected Issue**: The optimization may replace a select arm with a constant when the original arm had poison-generating flags that were conditionally masked. This could make the optimized code more poisonous than the original.

### Strategy 2: Test with vector types containing poison elements

- **Target**: Select instruction with vector type where the condition or arms contain vectors with poison elements
- **Rationale**: The hasAffectedValue function recursively checks operands but may not properly handle vector elements with poison. The findValuesAffectedByCondition function may not correctly identify all affected values in vector operations, especially when some lanes have poison while others don't.
- **Expected Issue**: The optimization might incorrectly simplify a select arm when only some vector lanes are affected by the condition, leading to incorrect poison propagation in unaffected lanes.

### Strategy 3: Test with undef values in select condition or arms

- **Target**: Select instruction where the condition involves comparisons with undef values, or arms contain undef
- **Rationale**: The fix assumes that known bits computed from the condition are valid for simplification. However, if the condition or arms involve undef values, the known bits analysis might produce incorrect results. Undef can take different values at each use, which could break the assumption that the condition implies specific known bits.
- **Expected Issue**: The optimization may incorrectly fold a select arm to a constant when undef values are involved, causing the result to differ from the original semantics where undef could take different values.

### Strategy 4: Test with non-constant conditions that have side effects

- **Target**: Select instruction where the condition is computed from values that could have side effects or are non-speculatable
- **Rationale**: The fix injects the condition into the SimplifyQuery for known bits computation. If the condition involves non-speculatable operations or has side effects, using it to simplify select arms could change the evaluation order or introduce speculative execution of side effects.
- **Expected Issue**: The optimization might cause non-speculatable operations to be evaluated speculatively, or change the order of side effects, leading to incorrect behavior.

### Strategy 5: Test with recursive/complex condition expressions

- **Target**: Select instruction with complex logical condition expressions (AND/OR chains) involving multiple comparisons
- **Rationale**: The computeKnownBitsFromCond function handles logical operations recursively, but there might be edge cases in how it combines known bits from multiple conditions. The intersection/union logic at lines 761-764 might not correctly handle all cases, especially with inverted conditions.
- **Expected Issue**: Complex conditions might produce incorrect known bits due to bugs in the recursive combination logic, leading to incorrect constant folding of select arms.
