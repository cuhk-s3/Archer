## Test Strategies

### Strategy 1: Multi-use operand with external user

- **Target**: Operands of deleted instructions that have multiple users
- **Rationale**: The fix checks OpI->hasOneUser() before marking an operand for deletion. If an operand has multiple users but one is in the deletion list and others are external, the operand should NOT be deleted. However, the fix may incorrectly handle cases where the operand is used both inside and outside the vectorization tree.
- **Expected Issue**: Instruction with external uses may be incorrectly deleted, causing use-after-free or missing operands in the IR

### Strategy 2: Operand is VectorizedValue of TreeEntry

- **Target**: Operands that are VectorizedValue of existing TreeEntries
- **Rationale**: The fix explicitly checks that an operand should not be deleted if it's the VectorizedValue of any TreeEntry (lines 2487-2489). However, this check uses Entries collected from getTreeEntry(I) and MultiNodeScalars. If an operand is a VectorizedValue but not captured in Entries, it might be incorrectly marked for deletion.
- **Expected Issue**: Vectorized values from other tree entries may be incorrectly deleted, breaking the vectorized code

### Strategy 3: Reduction operations with UserIgnoreList

- **Target**: Reduction tree operands when UserIgnoreList is active
- **Rationale**: In reduction vectorization (lines 14183-14191), uses in UserIgnoreList are replaced with PoisonValue before removeInstructionsAndOperands is called. However, the Entries check in removeInstructionsAndOperands may not properly account for reduction operands that have been replaced with poison, potentially causing incorrect operand deletion.
- **Expected Issue**: Reduction operands may be incorrectly deleted or poison values may propagate incorrectly

### Strategy 4: Instruction without parent during cleanup

- **Target**: Instructions that have already been removed from parent
- **Rationale**: The fix checks if (!I->getParent()) continue; at line 2496. However, in the destructor (lines 4077-4082), instructions without parent are re-inserted before the terminator. This inconsistency could cause issues if removeInstructionsAndOperands is called on instructions that were already removed.
- **Expected Issue**: Instructions may be double-processed or incorrectly re-inserted, causing IR corruption

### Strategy 5: Chain of trivially dead operands

- **Target**: Long chains of operands that become dead recursively
- **Rationale**: The while loop (lines 2508-2543) processes dead operands recursively. It asserts isInstructionTriviallyDead(VI, TLI) and VI->use_empty(). However, if an operand becomes dead but the isInstructionTriviallyDead check fails due to TLI state changes or side effects, the assertion will fail.
- **Expected Issue**: Assertion failure when an instruction in the dead chain is not recognized as trivially dead by TLI

## Summary of Fix Analysis and Testing

### Fix Understanding
The patch introduces `removeInstructionsAndOperands` function in SLPVectorizer that:
1. Marks instructions for deletion
2. Proactively collects and marks trivially dead operands (hasOneUser, wouldInstructionBeTriviallyDead, not a VectorizedValue)
3. Drops all references from dead instructions
4. Removes instructions from parent basic blocks
5. Recursively processes operands that become dead

This is an improvement over the previous `eraseInstruction` which only marked instructions for deletion without cleaning up operands.

### Test Strategies Covered
All 5 strategies from Phase 1 have been tested:

1. **Multi-use operand with external user**: Tested with `test_multi_use_external` - verifies that operands with external uses are not incorrectly deleted.

2. **Chain of trivially dead operands**: Tested with `test_chain_trivially_dead` - verifies recursive cleanup of instruction chains.

3. **Operand is VectorizedValue of TreeEntry**: Tested with `test_vectorized_value_tree_entry` - verifies that VectorizedValues are properly protected from deletion.

4. **Instruction without parent during cleanup**: Tested with `test_no_parent_cleanup` - verifies handling of instructions already removed from parent.

5. **Reduction operations with UserIgnoreList**: Tested with `test_reduction_userignorelist` - verifies proper handling of reduction operands.

### Verification Results
All tests passed verification with alive2 and difftest:
- The transformations are semantically correct
- No bugs were found in the fix
- The fix correctly handles all edge cases identified in Phase 1

### Conclusion
The fix is correct and properly handles:
- Operands with multiple users (external uses are preserved)
- Recursive cleanup of dead instruction chains
- Protection of VectorizedValues from premature deletion
- Instructions without parents during cleanup
- Reduction operations with UserIgnoreList

No bugs were found. The fix successfully enables more vectorizable patterns and generates fewer useless extractelement instructions as intended.
