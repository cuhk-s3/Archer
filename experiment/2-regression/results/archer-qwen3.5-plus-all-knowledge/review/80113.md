## Test Strategies

### Strategy 1: Test commutative binary ops with constant in operand 0

- **Target**: Binary operations where the constant mask is in operand 0 instead of operand 1
- **Rationale**: The fix only checks BO->getOperand(1) for a constant mask using m_APInt. Since and/or/xor are commutative, the constant could legally be in operand 0 (e.g., 'or i64 3, %a' instead of 'or i64 %a, 3'). The current code would miss these cases, failing to simplify them even when the demanded bits analysis shows they can be simplified.
- **Expected Issue**: Binary operations with constant in operand 0 position will not be simplified even when the mask does not affect demanded bits, leading to missed optimization opportunities.

### Strategy 2: Test vector types with splat constants in operand 0

- **Target**: Vector binary operations with splat constant masks in operand 0
- **Rationale**: The m_APInt matcher supports splatted ConstantVector constants. For vector operations like 'or <2 x i64> <i64 3, i64 3>, %vec', the constant is in operand 0. The fix would miss these cases since it only checks operand 1.
- **Expected Issue**: Vector binary operations with splat constant masks in operand 0 will not be simplified, missing optimization opportunities for vector code.

### Strategy 3: Test edge case where mask equals zero for or/xor

- **Target**: Or/xor operations with zero mask
- **Rationale**: For 'or %a, 0' or 'xor %a, 0', the mask is 0. For or/xor, the condition is !Demanded.intersects(0), which is always true (empty set doesn't intersect with anything). This should always simplify to %a. However, if the zero constant is in operand 0, it won't be caught.
- **Expected Issue**: Operations like 'or i64 0, %a' or 'xor i64 0, %a' won't be simplified to just %a when the constant is in operand 0.

### Strategy 4: Test edge case where mask equals all-ones for and

- **Target**: And operations with all-ones mask
- **Rationale**: For 'and %a, -1' (all ones), the condition Demanded.isSubsetOf(all-ones) is always true since any set of bits is a subset of all ones. This should always simplify to %a. But if the all-ones constant is in operand 0, it won't be caught.
- **Expected Issue**: Operations like 'and i64 -1, %a' won't be simplified to just %a when the constant is in operand 0.

### Strategy 5: Test partial overlap cases for or/xor with constant in operand 0

- **Target**: Or/xor operations where mask partially overlaps with demanded bits, constant in operand 0
- **Rationale**: Consider 'or i64 8, %a' where only bit 4 (value 16) is demanded. The mask (8 = bit 3) doesn't intersect with demanded (16 = bit 4), so it should simplify. But with constant in operand 0, the current code won't detect this. This tests the core logic with swapped operands.
- **Expected Issue**: Cases where the mask in operand 0 doesn't affect demanded bits will not be simplified, producing suboptimal code.
