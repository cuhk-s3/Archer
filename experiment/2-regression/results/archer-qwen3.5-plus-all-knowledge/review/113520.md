## Test Strategies

### Strategy 1: Mixed integer and non-integer types in alternate shuffle

- **Target**: alternate shuffle with cast operations where one operand is integer and another is non-integer (e.g., float)
- **Rationale**: The fix adds a check `if (SrcSclTy->isIntegerTy() && ScalarTy->isIntegerTy())` before applying MinBWs logic. If the source type is integer but ScalarTy is not (or vice versa), the MinBWs logic is skipped. However, the code then falls through to use `SrcTy` which was computed from the original operand type, not the MinBWs-adjusted type. This could lead to incorrect cost estimation or codegen when mixing integer and non-integer types in alternate shuffle patterns.
- **Expected Issue**: Incorrect cast cost estimation or wrong cast instruction generated when alternate shuffle involves mixed integer/non-integer types, potentially leading to miscompilation or suboptimal code

### Strategy 2: MinBWs entry exists but condition BWSz <= SrcBWSz is false

- **Target**: alternate shuffle where MinBWs contains an entry but target bitwidth is larger than source
- **Rationale**: The fix checks `if (BWSz <= SrcBWSz)` and only applies truncation logic when true. If MinBWs suggests a smaller bitwidth but BWSz > SrcBWSz (target is wider), the code falls through to the normal cast cost calculation. However, the SrcTy was already modified based on MinBWs (lines 9074-9076), which could lead to incorrect cost calculation for extension operations.
- **Expected Issue**: Incorrect cast cost when MinBWs suggests smaller bitwidth but actual target requires wider type, leading to wrong vectorization decisions

### Strategy 3: Vector element type mismatch in vectorizeTree

- **Target**: vectorizeTree codegen phase where LHS vector element type differs from ScalarTy
- **Rationale**: In the vectorizeTree fix (lines 12604-12617), the code checks `if (LHS->getType()->isIntOrIntVectorTy() && ScalarTy->isIntegerTy())`. It then computes SrcBWSz from the vector element type. If LHS is already a vector but its element type doesn't match ScalarTy in signedness or other properties (not just bitwidth), the truncation might be applied incorrectly. The fix only checks bitwidth, not signedness.
- **Expected Issue**: Incorrect truncation applied when signedness differs between LHS vector element type and ScalarTy, causing sign extension vs zero extension mismatch

### Strategy 4: Early return in cost estimation skips shuffle cost

- **Target**: alternate shuffle with truncation where early return occurs
- **Rationale**: In getEntryCost, when `BWSz <= SrcBWSz` is true, the code returns VecCost early (line 9088) without adding the shuffle cost (lines 9097-9105). This is correct only if no shuffle is needed. However, if the alternate op shuffle is still required (different opcodes in different lanes), the shuffle cost should be included. The early return assumes truncation replaces the need for alternate handling entirely.
- **Expected Issue**: Missing shuffle cost in cost estimation when truncation path is taken, leading to incorrect profitability analysis and potential miscompilation if vectorization is chosen based on underestimated cost

### Strategy 5: MinBWs lookup uses wrong operand entry

- **Target**: alternate shuffle where operand 0 and operand 1 have different MinBWs entries
- **Rationale**: The fix uses `MinBWs.find(getOperandEntry(E, 0))` to look up the minimum bitwidth. In alternate shuffle patterns, different operands might have different minimum bitwidth requirements. Using only operand 0's MinBWs entry might not correctly represent the required bitwidth for the alternate operation, especially when the alternate op has different semantics or uses different operands.
- **Expected Issue**: Incorrect bitwidth used for truncation when operands have different MinBWs requirements, potentially causing truncation to narrower type than required by alternate operation
