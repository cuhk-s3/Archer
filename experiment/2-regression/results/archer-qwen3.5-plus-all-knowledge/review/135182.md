## Test Strategies

### Strategy 1: Test cloneIVUser failure with unsupported opcode

- **Target**: Create a test where the IV user has an opcode not supported by cloneIVUser (e.g., SDiv, AShr with non-constant shift)
- **Rationale**: The cloneIVUser function only supports Add, Sub, Mul, UDiv for arithmetic and And, Or, Xor, Shl, LShr, AShr for bitwise. If getExtendedOperandRecurrence or getWideRecurrence returns a valid AddRec but cloneIVUser returns null (e.g., for SDiv), the old code might leave both IVs. The fix should ensure truncation is used as fallback.
- **Expected Issue**: Without the fix, both the narrow and wide IV might coexist, causing redundant computations. With the fix, truncation should be properly inserted.

### Strategy 2: Test SCEV expression mismatch after cloning

- **Target**: Create a test where SCEV analysis produces different expressions for the cloned wide use vs the expected WideAddRec
- **Rationale**: The code has a failsafe check at line 1834-1840 that discards the WideUse if SCEV expressions don't match. Before the fix, if this check failed, the code might not properly fallback. The fix ensures that when this check fails (returning nullptr), the code falls through to truncation.
- **Expected Issue**: Test case where SCEV mismatch occurs should result in proper truncation insertion after the fix, not leaving both IVs.

### Strategy 3: Test widenLoopCompare with mismatched signedness

- **Target**: Create a test where the IV is sign-extended but used in an unsigned comparison, or vice versa
- **Rationale**: The widenLoopCompare function has strict requirements about signedness matching (line 1504). If these conditions aren't met, it returns false. The fix ensures that when widenLoopCompare fails, we still try widenWithVariantUse and then truncateIVUse.
- **Expected Issue**: Test should verify that when widenLoopCompare fails due to signedness mismatch, truncation is properly inserted as fallback.

### Strategy 4: Test widenWithVariantUse with non-extend users

- **Target**: Create a test where the IV user's uses include instructions that are not extends (e.g., direct use in memory operation)
- **Rationale**: The widenWithVariantUse function requires all users of NarrowUse to be either extends, LCSSA phis, or comparisons (lines 1572-1612). If there's a user that doesn't match, it returns false. The fix ensures truncation is used as the final fallback.
- **Expected Issue**: When widenWithVariantUse fails due to unsupported user types, truncation should be properly inserted.

### Strategy 5: Test nested IV users where intermediate cloning fails

- **Target**: Create a test with a chain of IV users where an intermediate user cannot be cloned (e.g., not in whitelist)
- **Rationale**: The worklist processing in createWideIV (lines 1991-2005) processes IV users transitively. If an intermediate user's widening fails after partial expansion, the fix should ensure proper fallback to truncation at each level, preventing accumulation of both narrow and wide IVs.
- **Expected Issue**: Test should verify that even with nested users where some cloning fails, the final result has proper truncation and no redundant IVs.
