## Test Strategies

### Strategy 1: Integer overflow flags preservation test

- **Target**: Test that nuw/nsw flags on integer mul/add operations are properly handled during hoisting
- **Rationale**: The patch uses Builder.CreateMul() which doesn't preserve nuw/nsw flags from the original instructions. If the original mul/add had overflow flags, the transformed code may have different semantics (poison vs wrapped result).
- **Expected Issue**: The hoisted integer multiplication may lose nuw/nsw flags, causing different behavior when overflow occurs - the original would produce poison but the transformed code produces wrapped results, or vice versa.

### Strategy 2: Vector integer type test

- **Target**: Test hoisting with vector integer types (e.g., <4 x i32>)
- **Rationale**: The code uses isIntOrIntVectorTy() which includes vector types. The CreateMul call should work for vectors, but the flag preservation issue applies to vectors too. Additionally, vector-specific edge cases may exist.
- **Expected Issue**: Vector integer operations may not have their flags properly preserved, or the transformation may incorrectly handle vector element-wise operations.

### Strategy 3: Deep nesting with limit boundary test

- **Target**: Test with exactly IntAssociationUpperLimit (default 5) multiplications to verify limit handling
- **Rationale**: The code checks Changes.size() > Limit at line 2732. Testing at the boundary ensures the limit is correctly applied for integer types vs FP types.
- **Expected Issue**: The limit check might use the wrong limit (FP limit instead of int limit) or have off-by-one errors at the boundary.

### Strategy 4: Mixed reassociable and non-reassociable operations

- **Target**: Test patterns where some operations in the tree have multiple uses or are not reassociable
- **Rationale**: The code returns false if any BinaryOperator doesn't haveOneUse() (line 2709) or isn't reassociable. Complex patterns with mixed properties could expose issues in the traversal logic.
- **Expected Issue**: The worklist traversal might incorrectly handle cases where some branches are valid but others are not, potentially causing partial transformations or crashes.

### Strategy 5: Zero and one factor test for integers

- **Target**: Test cases where the invariant factor is 0 or 1
- **Rationale**: Multiplying by 0 or 1 has special semantics. For integers, these are identity/annihilation operations. The transformation should preserve these semantics correctly.
- **Expected Issue**: Edge cases with 0 or 1 factors might expose issues with the reassociation logic, especially regarding overflow flags or constant folding interactions.

After comprehensive testing of the LICM integer mul/add hoisting fix (commit 2dd52046816ac706a25d588efac77705793d8778), **no bugs were found**. The fix is correct.

**Testing performed:**
1. **Integer overflow flags preservation**: Verified with Alive2 that the transformation preserves semantics even when nsw/nuw flags are present. The hoisted multiplications don't need flags because the poison behavior is maintained through the chain of operations.

2. **Vector integer types**: Verified that the transformation correctly handles vector types (<2 x i64>, etc.) using `isIntOrIntVectorTy()`.

3. **Limit boundary**: Verified that the `IntAssociationUpperLimit` (default 5) is correctly applied. Tests with 5 multiplications are transformed, while 6 would exceed the limit.

4. **Mixed reassociable patterns**: Verified that the `hasOneUse()` check correctly prevents transformation when operations have multiple uses.

5. **Special constants (0, 1)**: Verified that zero and one factors are handled correctly.

**Why the fix is correct:**
- The `isReassociableOp()` helper correctly identifies integer operations (always reassociable) vs FP operations (require fast-math flags)
- `CreateMul()` is appropriate for integers (no FMF flags to preserve)
- Separate limits and statistics allow independent tuning for integer vs FP
- The transformation maintains semantic equivalence as verified by Alive2

The fix successfully extends the existing FP reassociation hoisting to integer types without introducing bugs.
