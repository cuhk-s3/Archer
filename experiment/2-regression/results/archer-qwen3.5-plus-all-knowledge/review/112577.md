## Test Strategies

### Strategy 1: Test abs intrinsic with INT_MIN value and reduced bitwidth

- **Target**: llvm.abs intrinsic with is_int_min_poison flag
- **Rationale**: The fix sets the second argument of abs to false when bitwidth is reduced. However, if the original code had is_int_min_poison=true and the value could be INT_MIN in the original bitwidth, reducing bitwidth and setting flag to false could change semantics. For example, abs(i8 -128) with poison=true returns poison, but with the fix it would return -128.
- **Expected Issue**: Miscompilation when abs(INT_MIN) should return poison but instead returns INT_MIN due to forced false flag

### Strategy 2: Test smin/smax/umin/umax with operands having different sign bit counts

- **Target**: smin/smax/umin/umax intrinsics with asymmetric sign bits
- **Rationale**: The CompChecker checks that both operands have enough sign bits (for signed) or zero bits (for unsigned). However, if one operand has sufficient bits but the other doesn't, the truncation could be unsafe. The fix uses ComputeNumSignBits which may not account for all runtime cases.
- **Expected Issue**: Incorrect truncation when one operand's sign bits are insufficient, leading to wrong min/max results after truncation and extension

### Strategy 3: Test intrinsic with multi-use operands where only some uses allow truncation

- **Target**: Intrinsic operands that are used outside the vectorization tree
- **Rationale**: Similar to Issue 129059, if an operand to abs/smin/smax/umin/umax is used elsewhere in the code with the original bitwidth requirements, truncating it for the intrinsic could break those other uses. The collectValuesToDemote function may not properly check for multi-use scenarios specific to call instructions.
- **Expected Issue**: Truncated operand causes incorrect results when the original value is needed elsewhere, similar to the multi-use demotion bug

### Strategy 4: Test cost-based bitwidth selection with edge case costs

- **Target**: Cost estimation for different bitwidths in buildIntrinsicArgTypes
- **Rationale**: The fix chooses the best bitwidth based on cost estimation in the Checker lambda. If costs are equal or the cost model is inaccurate for certain target architectures, it might select a suboptimal or incorrect bitwidth. The PowerOf2Ceil rounding could also cause issues.
- **Expected Issue**: Suboptimal or incorrect bitwidth selection when costs are tied or cost model doesn't accurately reflect target capabilities for specific bitwidths

### Strategy 5: Test vectorization with mixed intrinsic types in same tree

- **Target**: SLP tree containing both reducible and non-reducible intrinsics
- **Rationale**: If a tree contains abs/smin/smax/umin/umax intrinsics alongside other operations that don't support bitwidth reduction, the MinBWs map might have inconsistent entries. The code checks It != MinBWs.end() but doesn't verify consistency across all operations in the tree.
- **Expected Issue**: Inconsistent bitwidth handling when some operations in the tree support reduction and others don't, leading to type mismatches or incorrect casts
