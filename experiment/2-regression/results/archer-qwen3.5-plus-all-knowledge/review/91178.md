## Test Strategies

### Strategy 1: AnyReplaced Accumulation Bug

- **Target**: Multi-operand instruction with mixed replacement results
- **Rationale**: The fix has a bug where AnyReplaced is assigned (=) instead of OR-assigned (|=) in the operand replacement loop. If the first operand is replaced (AnyReplaced=true) but a later operand returns a non-null value equal to the original (AnyReplaced=false), the final AnyReplaced will be false even though replacements occurred. This causes the function to incorrectly return nullptr, missing optimization opportunities.
- **Expected Issue**: The function returns nullptr when it should return a simplified value, causing missed optimizations in cases where some operands are replaced and others return unchanged non-null values.

### Strategy 2: Cross-Lane Bitcast Not Detected

- **Target**: Vector select with bitcast changing lane count in def-use chain
- **Rationale**: The fix checks for ShuffleVectorInst and CallBase as cross-lane operations, but doesn't check for bitcast instructions that change vector lane counts (e.g., <4 x i32> to <2 x i64>). As noted in Issue 77320, such bitcasts mix data across original lanes, making per-lane equivalence assumptions invalid. The recursive replacement might incorrectly propagate through such bitcasts.
- **Expected Issue**: Incorrect simplification of vector select where per-lane equivalence is assumed through a lane-count-changing bitcast, leading to miscompilation (e.g., folding subtraction to zero when condition is only true for subset of lanes).

### Strategy 3: Absorber Check with Recursive Poison Semantics

- **Target**: Binary operation with absorber constant after recursive replacement
- **Rationale**: The absorber case now checks if NewOps[0] or NewOps[1] equals the absorber after recursive replacement. However, the impliesPoison(BO, Op) check still uses the original Op. With recursive replacement through multiple levels, the poison relationship between the final operands and the original Op might be different, potentially allowing unsafe simplifications.
- **Expected Issue**: The simplification returns the absorber constant when it shouldn't, because impliesPoison doesn't account for the recursive replacement chain, potentially introducing poison where none existed.

### Strategy 4: Recursion Depth Off-By-One

- **Target**: Deeply nested instruction chains requiring maximum recursion
- **Rationale**: The fix changed from 'else if (MaxRecurse)' with 'MaxRecurse - 1' to 'else' with 'MaxRecurse'. Since MaxRecurse is decremented at the top, the new code effectively allows one more level of recursion in the AllowRefinement branch. This could lead to deeper recursion than intended, potentially causing stack overflow or inconsistent behavior at recursion limits.
- **Expected Issue**: Either stack overflow on deeply nested chains, or inconsistent simplification results at boundary recursion depths compared to expected behavior.

### Strategy 5: Vector-to-Scalar Cross-Lane via ExtractElement

- **Target**: Vector Op used in extractelement producing scalar result
- **Rationale**: The fix checks if Op is a vector and the instruction doesn't return a vector, returning nullptr for cross-lane operations. However, extractelement is not explicitly checked. If Op is a vector and used in extractelement (which returns scalar), the recursive replacement might incorrectly assume per-lane semantics apply when extracting a specific lane could have different behavior.
- **Expected Issue**: Incorrect simplification when a vector Op is recursively replaced through an extractelement instruction, assuming per-lane equivalence when the extraction might select a lane where the equivalence doesn't hold.

## Summary of Testing for InstructionSimplify Fix (PR63104)

The fix makes `simplifyWithOpReplaced()` recursive, allowing it to replace operands not only in the immediate instruction but also in instructions that the operands use. Key changes include:
1. Recursive operand replacement through def-use chains
2. Proper vector type checking to prevent cross-lane optimizations
3. Updated x^x and absorber patterns to check both operands after replacement
4. Proper recursion depth management

3. **Recursion Depth Off-By-One**: Tested with deeply nested instruction chains (5+ levels). The optimization correctly handles recursion depth.

4. **Absorber Check with Recursive Poison Semantics**: Tested with multiplication absorber patterns. The optimization correctly handles poison semantics.
