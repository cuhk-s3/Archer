## Test Strategies

### Strategy 1: VectorUndefPoisonEdgeCases

- **Target**: Test vector constants with undef/poison elements in getFlippedStrictnessPredicateAndConstant
- **Rationale**: The function replaces undefs with first safe constant, but edge cases exist: all undef elements, mixed undef/poison, scalable vectors with undef splats. These could lead to incorrect constant adjustments or predicate flips.
- **Expected Issue**: Incorrect transformation or assertion failure when handling vector constants with special elements

### Strategy 2: ShiftFlagInteraction

- **Target**: Test interaction between shift flags (nuw/nsw) and the always-ashr approach
- **Rationale**: The fix always uses ashr even for unsigned comparisons. For nuw shifts, lshr might be more appropriate. This could lead to incorrect transformations when the high bits of the shifted value matter.
- **Expected Issue**: Incorrect comparison result due to using arithmetic instead of logical shift for unsigned operations

### Strategy 3: TypeLegalityBoundary

- **Target**: Test edge cases of shouldChangeType with unusual type sizes
- **Rationale**: shouldChangeType has complex logic for legal/illegal types. Edge cases include: non-power-of-two types, very large types, target-specific legal types, or transitions to/from illegal types.
- **Expected Issue**: Transformation applied when it shouldn't be (or not applied when it should) due to type legality misjudgment

### Strategy 4: ConstantBoundaryOverflow

- **Target**: Test constant adjustment at min/max boundaries with vector mixed lanes
- **Rationale**: getFlippedStrictnessPredicateAndConstant checks min/max values but vector constants with mixed boundary/non-boundary lanes could have edge cases. Also signed/unsigned boundary differences.
- **Expected Issue**: Incorrect constant adjustment or missed transformation opportunity at value boundaries

### Strategy 5: MultiUseShiftElimination

- **Target**: Test scenarios where shift has multiple uses but could be eliminated
- **Rationale**: Transformation requires hasOneUse(), but other optimizations might eliminate extra uses. Need to test interaction with other passes that could enable the transformation.
- **Expected Issue**: Missed optimization opportunity or incorrect ordering of transformations
