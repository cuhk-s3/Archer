## Test Strategies

### Strategy 1: UndefPoisonSwitchCondition

- **Target**: Switch condition with undef/poison possibilities
- **Rationale**: The patch uses UndefAllowed=false but doesn't consider poison. A switch condition derived from operations that could produce poison might lead to incorrect range analysis and wrong unreachable default transformation.
- **Expected Issue**: Incorrectly making default unreachable when switch condition could be poison, leading to miscompilation.

### Strategy 2: SingleReachableCaseWithFullCoverage

- **Target**: Switch with exactly 1 reachable case that covers entire range
- **Rationale**: The patch requires ReachableCaseCount > 1, but a switch with single case covering entire range (e.g., switch on bool) should also have unreachable default. This misses optimization opportunities.
- **Expected Issue**: Missed optimization: default remains reachable when it could be unreachable.

### Strategy 3: ConservativeRangeAnalysis

- **Target**: LVI returning overly conservative ranges
- **Rationale**: If LVI returns a range larger than actual possible values (e.g., due to analysis limitations), the transformation might incorrectly think default is reachable when it's actually unreachable.
- **Expected Issue**: Missed optimization opportunity due to conservative analysis.

### Strategy 4: ProfileMetadataPreservation

- **Target**: Switch with branch weight metadata
- **Rationale**: The patch creates new unreachable block but doesn't update branch weight metadata. This could lead to incorrect profile-guided optimization decisions.
- **Expected Issue**: Lost or incorrect profile metadata affecting later optimizations.

### Strategy 5: MultiplePredecessorsToDefault

- **Target**: Default destination with multiple predecessors
- **Rationale**: When removing BB as predecessor from DefaultDest, PHI nodes in DefaultDest might have incorrect incoming values if other predecessors exist. The transformation doesn't account for PHI node updates beyond removing the predecessor.
- **Expected Issue**: Broken PHI nodes leading to miscompilation or verification failure.

## Bugs Found

### Bug #1

**Analysis:**
Found a bug! The original code triggers UB due to switch on poison condition (add nsw overflow), but the transformed code doesn't trigger UB. This is a real bug: the optimization incorrectly assumes the switch condition cannot be poison when it actually can be. The patch uses UndefAllowed=false in getConstantRangeAtUse, but poison is different from undef. When the switch condition could be poison, the optimization shouldn't make the default unreachable because switch on poison is UB.

**Original LLVM IR:**
```llvm

entry:
  %s = urem i32 %cond, 3
  %s1 = add nsw i32 %s, 2147483647  ; Could overflow to poison
  switch i32 %s1, label %unreachable [
    i32 2147483647, label %exit1
    i32 2147483648, label %exit2
    i32 2147483649, label %exit1
  ]

exit1:
  ret i32 1

exit2:
  ret i32 2

unreachable:
  ret i32 0
}

define i32 @main(i32 %argc, ptr %argv) {
entry:
  %r = call i32 @switch_range_poison(i32 5)
  ret i32 %r
}

```

### Bug #2

**Analysis:**

This confirms the bug we found earlier: the optimization incorrectly assumes the switch condition cannot be poison when it actually can be. The patch uses UndefAllowed=false in getConstantRangeAtUse, but poison is different from undef. When the switch condition could be poison, the optimization shouldn't transform the switch because switch on poison is UB.

**Original LLVM IR:**

  %poison_cond = add nsw i32 %cond, 2147483647  ; Could overflow to poison
