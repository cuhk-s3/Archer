## Test Strategies

### Strategy 1: VectorConstantMixedSigns

- **Target**: Vector constants with mixed positive and negative elements
- **Rationale**: Current code assumes uniform sign if SignBitAllOne != SignBitAllZero, but this condition is also true when some elements are positive and some negative (SignBitAllOne=false, SignBitAllZero=false). The code would incorrectly set SignBit = SignBitAllOne (false) implying all elements are positive.
- **Expected Issue**: Incorrect sign bit inference for vector constants with mixed signs

### Strategy 2: NaNSignBitPropagation

- **Target**: Operations like FPExt, FPTrunc that don't guarantee NaN sign preservation
- **Rationale**: The code clears SignBit if not known never NaN for FPExt, but other operations might propagate sign bit incorrectly when NaN is possible. For example, in FMul sign inference, if either operand could be NaN, the XOR sign logic might be wrong.
- **Expected Issue**: Incorrect sign bit inference when NaN inputs are possible but sign isn't preserved

### Strategy 3: MinMaxNaNSignInference

- **Target**: Min/max intrinsics (maxnum, minnum, maximum, minimum) with possible NaN inputs
- **Rationale**: The sign bit inference for min/max has complex logic that checks Known.isKnownNeverNaN() before applying sign rules. However, the conditions like (KnownLHS.SignBit == false || KnownRHS.SignBit == false) might give wrong answer if NaN is possible but sign bits are known for non-NaN cases.
- **Expected Issue**: Incorrect sign bit for min/max when NaN inputs are possible

### Strategy 4: ZeroSignBitInference

- **Target**: knownNot() logic when value could be zero
- **Rationale**: In knownNot(), if isKnownNever(fcNan) && !SignBit, and isKnownNever(fcNegative), it sets SignBit = false. But if value could be zero (which is neither negative nor positive), this inference is incorrect. Zero has sign bit 0, but the logic assumes 'never negative' implies 'must be positive'.
- **Expected Issue**: Incorrect sign bit inference when value could be zero

### Strategy 5: DenormalFlushSignPreservation

- **Target**: Operations with denormal flushing modes that preserve sign
- **Rationale**: When denormals flush to zero, the sign might be preserved (negative denormal -> -0). The sign bit inference in various places might not account for this, especially in operations like sqrt that check isKnownNeverLogicalNegZero().
- **Expected Issue**: Incorrect sign bit inference when denormals flush to signed zero
