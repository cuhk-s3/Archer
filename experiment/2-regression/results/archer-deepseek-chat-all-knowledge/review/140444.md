- **Target**: Test loops where subtracting 1 from RHS causes overflow/wrap
- **Rationale**: The patch subtracts 1 from the RHS of exit condition. If RHS is at type limits (e.g., 0 for unsigned, INT_MIN for signed), this could cause wrap/overflow issues.
- **Expected Issue**: Incorrect loop bounds or undefined behavior due to overflow

### Strategy 2: SCEV Precision Limitations

- **Target**: Test loops with complex expressions where SCEV analysis may be imprecise
- **Rationale**: shouldPeelLastIteration() relies on SCEV to compute values at specific iterations. Imprecise SCEV could lead to incorrect peeling decisions.

- **Rationale**: canPeelLastIteration() assumes latch is exiting block with NE/EQ compare. Real-world loops may violate these assumptions.

### Analysis

## Bug Triggering Analysis

The test case `peel_loop_memory_deps_crash` triggers a bug in the loop peeling transformation. The bug occurs when:

1. **Complex Control Flow**: The loop contains multiple basic blocks (`loop`, `load`, `skip`) with conditional branching based on `%cmp2 = icmp uge i64 %iv, 1`.

2. **Memory Dependencies**: There's a load from the previous array element (`%prev = sub i64 %iv, 1`), creating a dependency chain between iterations.

3. **PHI Node Issues**: The transformation fails with error: "PHI node entries do not match predecessors!" and "Instruction does not dominate all uses!"

**Why this triggers the bug**: The peeling transformation doesn't correctly handle PHI nodes when there are multiple basic blocks within the loop. The `%iv` PHI node expects incoming values from `%entry` and `%skip`, but during transformation, the control flow is modified incorrectly, breaking SSA form and dominance properties.

## Fix Weakness Analysis

The patch has several weaknesses exposed by this test:

1. **Control Flow Assumptions**: The transformation assumes simple loop structures and doesn't handle complex control flow with multiple basic blocks correctly.

2. **PHI Resolution**: The `cloneLoopBlocks` function's PHI resolution logic (`cloneLoopBlocks` lines 913-939) doesn't account for PHI nodes with multiple incoming blocks when peeling from the end.

3. **Dominator Tree Updates**: The transformation modifies the dominator tree but doesn't properly update it for complex control flow patterns.

4. **Memory Dependency Preservation**: While not directly causing the crash, the memory dependency chain highlights that the transformation may not preserve memory semantics correctly for complex loops.

The bug is serious because it breaks fundamental LLVM invariants (SSA form, dominance properties), which can lead to miscompilation or compiler crashes.
