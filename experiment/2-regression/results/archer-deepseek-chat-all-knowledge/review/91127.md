## Test Strategies

### Strategy 1: Incorrect Nowrap Flag Retention

- **Target**: Test cases where trunc instructions get nuw/nsw flags from previous optimizations that should have dropped them
- **Rationale**: If flags are incorrectly retained from previous transformations, this fold will incorrectly apply, potentially introducing miscompilation by folding comparisons that shouldn't be folded
- **Expected Issue**: The fold transforms comparisons that change program semantics due to incorrectly retained nowrap flags

### Strategy 2: Mixed Nowrap Flag Combinations

- **Target**: Test icmp ult (trunc nuw X), (trunc nsw Y) and similar combinations for unsigned comparisons
- **Rationale**: The code checks !NoWrapFlags which passes for mixed flags (nuw & nsw = 0), but the semantic validity of folding comparisons with mixed flag types needs verification
- **Expected Issue**: The fold might incorrectly transform comparisons with mixed nuw/nsw flag types for unsigned predicates

### Strategy 3: Vector Element-Wise Poison Semantics

- **Target**: Vector trunc instructions where some lanes would cause wrap and others wouldn't
- **Rationale**: Nowrap flags apply to entire vectors; if any element wraps, the whole vector is poison. The fold might incorrectly assume all elements are safe
- **Expected Issue**: The transformation changes poison semantics for vector comparisons when only some elements would wrap

### Strategy 4: Type Conversion with Nowrap Flags

- **Target**: Tests with non-power-of-two or very large integer types combined with nowrap flags
- **Rationale**: isDesirableIntType() heuristic might not correctly handle all type conversions when nowrap flags guarantee safety
- **Expected Issue**: The fold either incorrectly applies or misses opportunities due to type desirability checks

### Strategy 5: Interaction with Source Value Poison Flags

- **Target**: Trunc instructions where source values have nsw/nuw flags from arithmetic operations
- **Rationale**: The fold only checks trunc flags, not flags on source values. Source value flags might affect overall poison semantics
- **Expected Issue**: Transformation changes poison semantics when source values have wrap flags that interact with trunc semantics

# Summary of Found Bugs

Through comprehensive testing, I've identified multiple critical bugs in the `foldICmpTruncWithTruncOrExt` transformation:

## 1. Vector Element-Wise Poison Semantics Bug
**Test**: `test_vector_elementwise_poison` with `<i32 65536, i32 100>`
- Element 0 violates `nuw` (bit 16 set)
- Original returns poison (any vector element violation → entire vector poison)
- Transformed uses bitcast+extractelement, returns false instead of poison

## 2. Type Conversion with Nowrap Flags Bug
**Test**: `test_type_conversion_nowrap` with `(i33 131072, i33 65536)`
- 131072 has bit 17 set, violates `nuw` for trunc i33→i17
- Original returns poison
- Transformed folds to `icmp ult i33`, returns false

## 3. Incorrect Nowrap Flag Retention Bug
**Test**: `test_incorrect_flag_retention_scenario` with `(2147483647, 1)`
- `add nsw` overflows, produces poison
- Original returns poison
- Transformed folds using `nsw` flags, returns false

## 4. Mixed Flag Combinations Bug
**Test**: Various tests with mixed `nuw`/`nsw` flags
- The transformation doesn't properly handle mixed flag semantics
- `!NoWrapFlags` check passes for mixed flags but may not be valid

## Root Cause Analysis

The transformation has fundamental flaws:

1. **Blind trust in nowrap flags**: Assumes `nuw`/`nsw` flags are always semantically valid
2. **Poison semantics violation**: Changes poison-producing computations to non-poisonous ones
3. **Vector handling incorrect**: Doesn't preserve element-wise poison semantics
4. **Type safety issues**: `isDesirableIntType()` heuristic insufficient
5. **Source value interaction**: Ignores poison from source operations

The fix needs to either:
- Use more conservative checks (e.g., verify flags with known bits)
- Preserve poison semantics by not folding when flags could be violated
- Add runtime checks (not feasible)
- Limit transformation to provably safe cases only
