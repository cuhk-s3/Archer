## Test Strategies

### Strategy 1: PoisonConditionTest

- **Target**: Select conditions containing poison/undef values
- **Rationale**: The optimization injects the select condition as an assumption without checking if it's poison-free. If the condition contains poison, the injected assumption could be invalid, leading to incorrect constant folding.
- **Expected Issue**: Select arm incorrectly folded to constant when condition is poison, changing program semantics from poison to constant value.

### Strategy 2: DeepExpressionTreeTest

- **Target**: Select arms with deep expression trees exceeding recursion limits
- **Rationale**: hasAffectedValue has recursion depth limits and special handling for PHI nodes. Deep trees or specific PHI patterns could cause the function to return false incorrectly, missing optimization opportunities or incorrectly claiming no affected values.
- **Expected Issue**: Missed optimization where known bits should become constant but doesn't due to recursion depth limits.

### Strategy 3: VectorPoisonLaneTest

- **Target**: Vector select conditions with poison lanes
- **Rationale**: Vector conditions with some poison lanes could cause incorrect affected values identification. The optimization might assume all lanes of the condition are valid when computing known bits.
- **Expected Issue**: Incorrect constant folding for vector select arms due to mishandling of poison lanes in condition.

### Strategy 4: ComplexConditionInversionTest

- **Target**: Complex conditions (logical AND/OR) with inversion
- **Rationale**: The Invert flag in CondContext might not correctly handle inversion of complex logical conditions for known bits analysis. computeKnownBitsFromCond uses pattern matching that might not preserve semantics correctly when inverted.
- **Expected Issue**: Incorrect known bits computation for false arm due to improper condition inversion.

### Strategy 5: AffectedValuesMissTest

- **Target**: Values affected through indirect relationships not captured by findValuesAffectedByCondition
- **Rationale**: findValuesAffectedByCondition has specific pattern matching for affected values. It might miss values affected through complex expressions or non-standard patterns, causing missed optimizations.
- **Expected Issue**: Select arm not simplified even though known bits should be constant given the condition.

## Summary of Phase 2 Testing

I have completed comprehensive testing of the patch that adds optimization to simplify select arms based on KnownBits implied by the condition. Here are the key findings:

### 1. Bug Found (Unrelated to Patch)
**Test Case**: `vec_select_no_equivalence`
**Bug**: Existing InstCombine bug where `select (icmp eq %x, 0), (shufflevector %x, ...), %x` is incorrectly transformed to just return `%x`. When `x = <0, 2>`, the condition is `<true, false>`, `x10 = <2, 0>`, and the select should return `<2, 2>`, but optimization returns `<0, 2>`.

**This bug is unrelated to the new patch** - it's an existing issue in InstCombine.

### 2. Patch Robustness Analysis
All 5 proposed test strategies from Phase 1 were thoroughly tested across multiple test cases:

1. **PoisonConditionTest**: ✅ Correctly handled - poison conditions result in poison values
2. **DeepExpressionTreeTest**: ✅ Correctly handled - deep expression trees are folded appropriately
3. **VectorPoisonLaneTest**: ✅ Correctly handled - vector operations with poison lanes preserve semantics
4. **ComplexConditionInversionTest**: ✅ Correctly handled - logical AND/OR conditions and inversions work correctly
5. **AffectedValuesMissTest**: ⚠️ Partially addressed - some patterns may be missed for optimization but no miscompilation

### 3. Patch Strengths
- **Safe poison handling**: Never incorrectly uses poison conditions as assumptions
- **Correct constant folding**: Only folds to constants when known bits become constant
- **Proper recursion limits**: Handles deep expression trees within limits
- **Vector support**: Works correctly with vector types and poison lanes

### 4. Minor Weakness
The only minor issue is that `findValuesAffectedByCondition` has limited pattern matching and may miss some affected value relationships (e.g., through add operations). However, this only affects optimization opportunities, not correctness.

### Conclusion
The patch is **robust and correct**. It adds a useful optimization to simplify select arms based on condition-implied KnownBits without introducing new bugs. The existing bug found (`vec_select_no_equivalence`) is unrelated to this patch.
