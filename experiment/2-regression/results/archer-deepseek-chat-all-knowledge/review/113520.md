- **Target**: Test cases with sext and zext alternate ops where destination is narrower than source
- **Rationale**: The fix uses a single signedness from MinBWs, but alternate ops might require different signedness handling for truncation

# Bug Analysis Summary

## Bug Triggering Analysis
The test case `test_mixed_signedness_trunc` targets the **MixedSignednessAlternateCasts** strategy identified in Phase 1. The fix has a critical weakness: when handling alternate cast operations (like `sext` and `zext`) where the destination type is narrower than the source type, the fix uses a single signedness value from `MinBWs` (`It->second.first`) for truncation. However, when alternate ops have different signedness requirements (one `sext` and one `zext`), using a single signedness for truncation can lead to incorrect value truncation or poison generation.

Specifically:
- `sext i8 127 to i32` produces `127` (positive)
- `zext i8 255 to i32` produces `255` (positive in unsigned context)
- Truncating both with the same signedness (likely signed from `MinBWs`) could incorrectly handle the unsigned value

## Fix Weakness Analysis
The fix assumes that when `BWSz <= SrcBWSz` (destination bitwidth ≤ source bitwidth), we can safely truncate using a single signedness from `MinBWs`. This assumption breaks when:
1. Alternate ops have different signedness (`sext` vs `zext`)
2. The `MinBWs` analysis picks one signedness (likely the most conservative)
3. Truncation with wrong signedness can produce poison or incorrect values

The weakness is in the code at lines 12609-12610 in `vectorizeTree`:
```cpp
if (BWSz < SrcBWSz)
  LHS = Builder.CreateIntCast(LHS, VecTy, It->second.first);
```

And similarly in `getEntryCost` at lines 9079-9082. The fix doesn't account for mixed signedness in alternate ops when truncating.
