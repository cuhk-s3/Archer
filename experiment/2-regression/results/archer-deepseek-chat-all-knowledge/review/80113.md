- **Expected Issue**: Optimization incorrectly applied or missed due to undef handling in vector constants

### Strategy 2: Non-Splat Vector Constants

- **Target**: Test non-splat vector constants as mask operand
- **Rationale**: m_APInt only matches splat vectors, non-splat vectors won't match, potentially missing cases where optimization could apply element-wise

## Bugs Found

### Bug #1

**Analysis:**
Found a bug! The 'or disjoint' instruction has poison semantics: it produces poison if both inputs have 1 in the same bit position. When BDCE simplifies 'or disjoint %a, 3' to just '%a', it changes the poison semantics. If %a has bit 0 or 1 set (overlapping with mask 3), the original 'or disjoint' would produce poison, but after simplification it doesn't. This is a semantic change that should be prevented. The patch should check for poison-generating flags (like 'disjoint') and not apply the optimization in such cases, or ensure poison semantics are preserved.

**Original LLVM IR:**
```llvm

define i64 @test_or_disjoint_poison_flags(i64 %a) {
entry:
  ; 'or disjoint' already has poison semantics
  %or = or disjoint i64 %a, 3
  %ret1 = and i64 %or, 8
  %ret2 = and i64 %or, 16
  %add = add i64 %ret1, %ret2
  ret i64 %add
}

define i64 @main(i32 %argc, ptr %argv) {
entry:
  %r = call i64 @test_or_disjoint_poison_flags(i64 255)
  ret i64 %r
}

```
