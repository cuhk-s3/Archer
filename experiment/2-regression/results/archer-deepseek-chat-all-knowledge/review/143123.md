## Test Strategies

### Strategy 1: Mismatched nnan propagation

- **Target**: Create test where comparison has nnan but select doesn't, and NaN values appear in select arms
- **Rationale**: The fix propagates nnan from comparison to FMF. If select arms contain NaN values and select lacks nnan, pattern matching might incorrectly assume no NaN handling needed, leading to miscompilation.
- **Expected Issue**: Incorrect min/max pattern recognition or optimization that assumes no NaN when NaN values are actually possible in select arms.

### Strategy 2: Direct API misuse

- **Target**: Test direct calls to matchDecomposedSelectPattern with incorrect FMF parameter
- **Rationale**: The API now has default FMF parameter. Callers might not pass correct FMF, especially in recursive or complex pattern matching scenarios.
- **Expected Issue**: Pattern matching uses incorrect FMF leading to wrong optimization decisions.

### Strategy 3: Missing nsz from comparison

- **Target**: Create test where comparison has nsz but select doesn't, involving signed zero semantics
- **Rationale**: The fix ignores nsz from comparison. If comparison ignores signed zeros but select must preserve them, pattern matching could be incorrect for min/max with zeros.
- **Expected Issue**: Incorrect handling of signed zeros in min/max pattern recognition, potentially changing -0.0 to +0.0 or vice versa.

### Strategy 4: Vector FMF complexity

- **Target**: Test with vector types where FMF might have lane-specific implications
- **Rationale**: Vector operations might have different FMF semantics. The fix treats FMF uniformly across all lanes, which might not be correct for partial vector operations.
- **Expected Issue**: Incorrect pattern matching for vector min/max operations with mixed NaN/zero patterns across lanes.

### Strategy 5: Recursive pattern FMF propagation

- **Target**: Test nested select patterns where FMF should not propagate through all levels
- **Rationale**: When matching min/max of min/max patterns, FMF might need to be reconsidered at each level. The fix's FMF parameter might propagate incorrectly in recursion.
- **Expected Issue**: Incorrect pattern recognition in complex nested select structures due to over-propagated FMF.

## Fix Analysis

The patch successfully addresses issues #137998 and #141017 by:

1. **Adding FMF parameter to `matchDecomposedSelectPattern`**: Defaults to `FastMathFlags()`
2. **Passing select's FMF from `matchSelectPattern`**: Uses `SI->getFastMathFlags()` if SI is FPMathOperator
3. **Only propagating `nnan` from comparison**: `if (isa<FPMathOperator>(CmpI) && CmpI->hasNoNaNs()) FMF.setNoNaNs()`
4. **Ignoring other FMF from comparison**: `nsz`, `ninf`, etc. come only from select
