## Test Strategies

### Strategy 1: Vector with mixed zero/non-zero lanes

- **Target**: Test vector ctlz/cttz where some lanes are zero and others are not
- **Rationale**: The transformation might incorrectly propagate poison to non-zero lanes in vector types when only some lanes have zero input
- **Expected Issue**: Poison propagation to lanes that should have well-defined values

### Strategy 2: Multiple indirect uses through PHI

- **Target**: Create pattern where ctlz/cttz appears to have single use but has multiple indirect uses through PHI nodes
- **Rationale**: The hasOneUse() check might miss indirect uses through PHI nodes or other control flow
- **Expected Issue**: Transformation applied when ctlz/cttz actually has multiple uses

### Strategy 3: Shift with nsw/nuw flags

- **Target**: Test shift operations with nsw/nuw flags using ctlz/cttz as shift amount
- **Rationale**: Changing shift amount to poison might violate nsw/nuw semantics differently than shift amount >= bitwidth
- **Expected Issue**: Incorrect poison propagation with nsw/nuw flags

### Strategy 4: Non-power-of-two bitwidth types

- **Target**: Test with i33, i15, etc. non-power-of-two types
- **Rationale**: Shift semantics for amount >= bitwidth might be different for non-power-of-two types
- **Expected Issue**: Incorrect transformation for non-power-of-two bitwidths

### Strategy 5: Metadata and attribute preservation

- **Target**: Test ctlz/cttz with noundef, range metadata, or other attributes
- **Rationale**: Transformation might fail to drop metadata/attributes that become invalid when returning poison
- **Expected Issue**: Invalid metadata/attributes retained after transformation

### Bug #2

**Analysis:**
The difference shows that with noundef attribute, the transformation introduces UB that wasn't present in the original. Original: cttz returns 32 (well-defined value), ashr with shift amount 32 produces poison. Transformed: cttz returns poison, violating noundef, causing immediate UB. This is a real bug - the transformation fails to drop the noundef attribute when changing is_zero_poison to true.

### Bug #4

**Analysis:**
This shows another bug: when cttz has multiple uses but one is dead (add instruction), the dead code elimination removes the add first, then the transformation applies, changing is_zero_poison to true while keeping noundef. This introduces UB when y=0. The transformation should check if cttz has multiple uses BEFORE any dead code elimination, or should drop noundef when changing is_zero_poison.

**Original LLVM IR:**
```llvm

define i32 @shl_cttz_false_multiuse_noundef_simple(i32 %x, i32 %y) {
entry:
  %cttz = call noundef i32 @llvm.cttz.i32(i32 %y, i1 false)
  %add = add i32 %cttz, 1
  %res = shl i32 %x, %cttz
  ret i32 %res
}

declare i32 @llvm.cttz.i32(i32, i1 immarg)
```
