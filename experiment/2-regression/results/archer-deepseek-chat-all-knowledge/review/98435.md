## Test Strategies

### Strategy 1: Mixed Undef/Poison Vector Pattern Matching

- **Target**: Test cases using m_AllOnes(), m_Zero(), m_One() with vectors containing both undef and poison elements
- **Rationale**: The patch assumes uniform treatment of special elements, but mixed vectors could expose inconsistent behavior between different parts of the compiler. Pattern matching might incorrectly reject valid patterns or accept invalid ones when vectors contain both undef and poison.
- **Expected Issue**: Pattern matching might incorrectly reject valid patterns or accept invalid ones when vectors contain both undef and poison, leading to missed optimizations or incorrect transformations.

### Strategy 2: Legacy Undef Generation in Optimizations

- **Target**: Optimization passes that still generate undef values (e.g., demanded elements, constant folding)
- **Rationale**: If any pass hasn't been updated to generate poison instead of undef, pattern matching will fail where it should succeed, causing missed optimization opportunities.
- **Expected Issue**: Missed optimization opportunities or incorrect transformations due to failed pattern matches when undef values are generated by other passes.

### Strategy 3: Third-Party IR with Undef Elements

- **Target**: IR generated by external tools or older compiler versions containing undef elements
- **Rationale**: The stricter matching breaks backward compatibility with IR containing undef, potentially causing performance regressions.
- **Expected Issue**: Performance regression or missed optimizations for code using undef from external sources or older compiler versions.

### Strategy 4: Inconsistent APInt Matcher Usage

- **Target**: Code using both m_APIntAllowUndef and m_AllOnes() on the same values
- **Rationale**: Different matchers have different semantics (allow vs forbid undef), leading to inconsistent behavior in transformations.
- **Expected Issue**: Transformations that work with one matcher but not another, causing optimization inconsistencies or incorrect simplifications.

### Strategy 5: Scalable Vector Edge Cases

- **Target**: Scalable vector constants with undef/poison elements
- **Rationale**: The fix's logic only explicitly handles fixed vectors; scalable vectors might have different behavior or edge cases.
- **Expected Issue**: Incorrect pattern matching or crashes with scalable vectors containing special elements, particularly in the getSplatValue() path.
