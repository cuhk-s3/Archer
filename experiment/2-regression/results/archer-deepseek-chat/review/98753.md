## Test Strategies

### Strategy 1: Pointer Provenance Violation

- **Target**: Replace scalar pointers with same address but different provenance in load/store operations
- **Rationale**: The optimization assumes pointer equality implies full substitutability, but pointer provenance restricts which pointers can access which memory regions. Substituting based solely on address equality can violate memory safety.
- **Expected Issue**: Miscompilation where load/store uses wrong provenance, potentially accessing restricted memory or causing alias analysis violations

### Strategy 2: Vector Cross-Lane Operation

- **Target**: Use equality comparison on vectors followed by shufflevector or bitcast that mixes lanes
- **Rationale**: The optimization assumes per-lane substitutability, but operations like shufflevector mix elements across lanes. Substituting vec1 for vec2 in shufflevector could change which elements are selected.
- **Expected Issue**: Incorrect vector element selection after simplification, producing wrong results

### Strategy 3: Poison Refinement via Arithmetic

- **Target**: Use equality condition that makes arithmetic operations produce poison (e.g., division by zero, shift by too-large amount)
- **Rationale**: AllowRefinement=true may convert undef to poison when substituting. If a==b makes (a-b) = 0, then udiv 1/(a-b) becomes poison, potentially refining undef to poison.
- **Expected Issue**: Undefined behavior introduced where none existed before, or poison propagated incorrectly

### Strategy 4: llvm.is.constant Misapplication

- **Target**: Combine equality comparison with llvm.is.constant on the compared value
- **Rationale**: The patch guards against folding llvm.is.constant, but there may be indirect ways to misuse it. Equality assumption doesn't make a value constant at compile time.
- **Expected Issue**: Incorrect constant folding leading to wrong control flow or missed optimizations

### Strategy 5: Atomic/Memory Ordering Violation

- **Target**: Use equality on pointers to atomic variables with different memory ordering
- **Rationale**: Atomic operations have strict ordering constraints. Substituting atomic_load(p) with atomic_load(q) when p==q might reorder or combine atomic operations incorrectly.
- **Expected Issue**: Memory ordering violations, lost atomicity, or data races introduced
