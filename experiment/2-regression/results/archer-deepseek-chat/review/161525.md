## Test Strategies

### Strategy 1: Non-IEEE Denormal Mode Testing

- **Target**: Function attributes for denormal-fp-math
- **Rationale**: The transformation requires IEEE denormal mode but may incorrectly apply with other modes like positive-zero, preserve-sign, or dynamic
- **Expected Issue**: Incorrect transformation when denormal handling differs from IEEE

### Strategy 2: Fast-Math Flag Mismatch Testing

- **Target**: Different nnan/ninf flags on fsub vs fcmp
- **Rationale**: The check uses LHSI->hasNoNaNs()/hasNoInfs() but fcmp may have different flags, leading to incorrect assumptions about NaN/Inf behavior
- **Expected Issue**: Transformation applied when fsub has flags but fcmp doesn't, or vice versa

### Strategy 3: Vector Element Infinity Mix Testing

- **Target**: Vector types with some infinity elements
- **Rationale**: isKnownNeverInfinity on vectors may return false if any element could be infinity, but transformation might still be valid for non-infinity elements
- **Expected Issue**: Missed optimization opportunity or incorrect transformation for mixed vector lanes

### Strategy 4: Context-Sensitive Infinity Analysis Testing

- **Target**: isKnownNeverInfinity with limited depth
- **Rationale**: Depth=0 may miss context from surrounding instructions that prove values cannot be infinity
- **Expected Issue**: Missed optimization when values are provably non-infinity through context

### Strategy 5: Signed Zero Edge Case Testing

- **Target**: -0.0 vs +0.0 in subtraction
- **Rationale**: -0.0 - 0.0 = -0.0 while 0.0 - 0.0 = 0.0, but fcmp ignores sign of zero, so transformation should still be valid
- **Expected Issue**: Incorrect handling of signed zero cases in the transformation

### Bug #2

**Analysis:**
Found a bug! When fsub doesn't have ninf flag but fcmp does, the transformation incorrectly assumes that inf-inf=NaN case is prevented. However, ninf flag on fcmp doesn't prevent the fsub from producing NaN when both operands are infinity. The transformation should check that the fsub has ninf flag, not just that the fcmp has it.

### Analysis

## Bug Triggering Analysis
Found a bug in the patch! The transformation incorrectly applies when the fsub instruction doesn't have the `ninf` flag but the fcmp instruction does. The `ninf` flag on fcmp only affects comparison semantics, not the fsub operation. When both operands are infinity, `fsub` produces NaN regardless of the fcmp's flags.

The counterexample from alive2 shows:
- `%x = <+inf, +0.0>`
- `%y = <SNaN, SNaN>`
- Source: `%fs = <QNaN, QNaN>` (fsub converts SNaN to QNaN), `%cmp = <1, 1>` (fcmp ninf uge QNaN, 0.0 = true)
- Target: `%cmp = <poison, 1>` (fcmp ninf uge +inf, SNaN = poison because ninf flag assumes no infinities)

## Fix Weakness Analysis
The fix checks `LHSI->hasNoNaNs() && !LHSI->hasNoInfs()` (the fsub's flags), but when the fcmp has ninf flag but fsub doesn't, the transformation still applies. This is incorrect because the `ninf` flag on fcmp doesn't prevent `inf - inf = NaN` in the fsub. The transformation should be more conservative when fast-math flags differ between the fsub and fcmp instructions.

The bug reveals that the fix doesn't properly handle cases where:
1. fsub lacks ninf/nnan flags but fcmp has them
2. The infinity check should consider both instructions' flags
3. Different fast-math flags between fsub and fcmp create unsafe assumptions
