## Test Strategies

### Strategy 1: PoisonConditionPropagation

- **Target**: Select conditions containing poison/undef values
- **Rationale**: The optimization assumes the condition value (true/false) when analyzing select arms, but if the condition itself contains poison/undef, this assumption could propagate poison incorrectly or fold to wrong constants.
- **Expected Issue**: Incorrect constant folding or poison propagation when condition contains poison/undef elements

### Strategy 2: DeepDependencyChainExceedsDepth

- **Target**: Complex expressions with deep dependency chains in select arms
- **Rationale**: hasAffectedValue has recursion depth limits (MaxAnalysisRecursionDepth). Deep chains might be truncated, causing missed optimizations or incorrect analysis of whether values are affected.
- **Expected Issue**: Missed optimization opportunities or incorrect analysis for deep expression trees

### Strategy 3: PHINodeCycleDetection

- **Target**: PHI nodes with cycles in select arms
- **Rationale**: Special handling for PHI nodes adjusts depth to MaxAnalysisRecursionDepth-2, which could interact poorly with cycle detection or cause premature termination.
- **Expected Issue**: Infinite recursion or incorrect depth handling for PHI nodes with cycles

### Strategy 4: VectorPoisonElementPropagation

- **Target**: Vector selects with poison elements in condition or arms
- **Rationale**: Vector conditions with poison elements might incorrectly propagate known bits lane-wise. The optimization might assume all lanes have the same condition value.
- **Expected Issue**: Incorrect lane-wise constant folding for vector selects with poison

### Strategy 5: ConflictingKnownBitsResolution

- **Target**: Conditions that imply conflicting known bits for affected values
- **Rationale**: If computeKnownBits returns hasConflict() due to contradictory information, it resets KnownBits. The optimization only checks isConstant(), not hasConflict(), potentially missing that the result is not reliable.
- **Expected Issue**: Incorrect constant folding when known bits are conflicting

### Bug #2

**Analysis:**
The original returns 0 (since poison & 0 = 0, icmp eq 0, 0 = true, t3=0, select returns 0), but optimization returns poison. This is a bug: poison & 0 = 0, not poison. The and with poison and 0 should be 0, not poison.

**Original LLVM IR:**
```llvm

define i32 @n4_poison(i32 %arg) {
  %t = and i32 %arg, poison
  %t1 = icmp eq i32 %t, 0
  %t2 = lshr i32 %arg, 2
  %t3 = and i32 %t2, 2
  %t4 = select i1 %t1, i32 %t3, i32 1
  ret i32 %t4
}

define i32 @main(i32 %argc, ptr %argv) {
entry:
  %r = call i32 @n4_poison(i32 0)
  ret i32 %r
}

```

### Analysis

## Summary of Findings

We have successfully tested the fix and found multiple bugs:

1. **Vector shuffle bug**: The optimization incorrectly simplifies `select` with shufflevector in the true arm. When condition implies `%x[0] = 0`, it assumes `shufflevector(%x)[0] = 0`, but shufflevector rearranges elements.

2. **Poison semantics bug**: The optimization incorrectly returns `poison` for `and i32 0, poison`, which should be `0` (poison & 0 = 0). This occurs in both `n4_poison` and `n5_poison` test cases.

The bugs were found using our Phase 1 strategies:
- **VectorPoisonElementPropagation**: Exposed the vector shuffle bug
- **PoisonConditionPropagation**: Exposed the poison semantics bug

Other strategies (DeepDependencyChainExceedsDepth, PHINodeCycleDetection, ConflictingKnownBitsResolution) didn't expose bugs but helped verify the optimization works correctly in those scenarios.

The fix needs improvements in:
1. Tracking value transformations through operations like shufflevector
2. Correct poison semantics for operations with absorbing values
3. Conservative handling of vector operations that rearrange elements
