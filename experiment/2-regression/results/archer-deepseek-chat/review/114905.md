## Test Strategies

### Strategy 1: FPNonAssocReduction

- **Target**: FAdd reductions without reassoc flag
- **Rationale**: Without reassoc flag, (a+a)+(b+b) and (a+b)*2 may produce different floating-point results due to rounding differences, violating mathematical equivalence assumption.
- **Expected Issue**: Different numerical results between optimized and unoptimized code, or miscompilation with fast-math flags.

### Strategy 2: IntOverflowScaling

- **Target**: Add reductions with nsw/nuw flags near type limits
- **Rationale**: a+a and a*2 have different overflow/poison semantics with wrapping flags. Scaling transformation may change poison propagation or violate no-signed-wrap assumptions.
- **Expected Issue**: Poison propagation differences, signed overflow UB violation, or incorrect results with overflow.

### Strategy 3: XorUndefShuffle

- **Target**: Xor reductions with undef values and even repetition counts
- **Rationale**: emitReusedOps creates shufflevector with undef mask elements that may incorrectly propagate poison. The shuffle may introduce new poison or suppress existing poison incorrectly.
- **Expected Issue**: Incorrect poison propagation through shufflevector, or miscompilation with undef/poison values.

### Strategy 4: ConstExprReduction

- **Target**: Reductions including ConstantExpr or GlobalValue
- **Rationale**: isConstant() excludes ConstantExpr and GlobalValue but they may appear in reductions. The optimization may misclassify them as non-constants and apply incorrect transformations.
- **Expected Issue**: Misclassification leading to incorrect optimization, crash, or type mismatch in scaling operations.

### Strategy 5: UndefSplatAmbiguity

- **Target**: Reductions with pattern [undef, a, undef, a, ...]
- **Rationale**: isSplat() returns true for this pattern (all non-undef are a), but undef lanes are not guaranteed equal to a. Optimization may incorrectly assume all lanes are a.
- **Expected Issue**: Incorrect optimization assuming undef lanes equal a, leading to wrong results when undef takes other values.

### Analysis

## Bug Triggering Analysis

The test case `test_int_overflow` with input `a = 2147483647` (max positive int32) triggers a bug in the SLPVectorizer's horizontal reduction optimization. The original code has `add nsw i32 %a, %a` where `nsw` means "no signed wrap". When `a = 2147483647`, `a + a` overflows in signed arithmetic, producing poison due to the `nsw` flag.

The optimization transforms `a + a + b + b` into `a*2 + b*2` using `mul` instructions. However, `mul i32 %a, 2` wraps without producing poison when `a = 2147483647`, while `add nsw i32 %a, %a` produces poison. This violates the `nsw` semantics.

Similarly, the `nuw` (no unsigned wrap) version with `a = 4294967295` (max uint32) has the same issue: `add nuw` produces poison on overflow, but `mul` wraps without poison.

## Fix Weakness Analysis

The fix has several weaknesses:

1. **Missing overflow flag propagation**: When converting repeated addition to multiplication, the optimization doesn't preserve `nsw`/`nuw` flags. It should add `nsw` to `mul` when the original `add` had `nsw`.

2. **Floating-point associativity assumption**: The optimization assumes `a + a = a*2` for floating-point without checking `reassoc` fast-math flag. Without `reassoc`, floating-point addition is not associative, so `(a+a)+(b+b) ≠ (a+b)*2` due to rounding differences.

3. **Undef splat ambiguity**: `isSplat([undef, x, undef, x])` returns true, but the optimization incorrectly assumes undef lanes equal `x`. This leads to incorrect transformations like converting `undef + x + undef + x` to `2*x + undef`.

4. **Missing fast-math flag checks**: For `FAdd`, the optimization should check for `reassoc` flag before applying scaling transformations.

The fix needs to:
- Preserve overflow flags when converting addition to multiplication
- Check fast-math flags for floating-point operations
- Handle undef values more carefully in splat detection
- Exclude operations with `nsw`/`nuw` flags from scaling optimization unless proper flag propagation is implemented
