## Test Strategies

### Strategy 1: SignedValueInUnsignedSystem

- **Target**: Create tests where signed values (with possible negative ranges) flow into unsigned comparisons
- **Rationale**: The fix assumes all unsigned variables are ≥ 0, but signed values cast to unsigned could be negative in signed interpretation, potentially leading to incorrect optimizations
- **Expected Issue**: Incorrect optimization where `%signed uge 0` is assumed true but `%signed` could be negative signed value, causing miscompilation

### Strategy 2: ContradictoryConstraintSystem

- **Target**: Create tests where existing constraints imply `Var < 0` but `Var >= 0` is added to unsigned system
- **Rationale**: Adding `Var >= 0` to an already unsatisfiable system shouldn't cause miscompilation or affect correctness proofs for other facts
- **Expected Issue**: System becomes unsatisfiable, potentially affecting ability to prove other facts correctly or causing incorrect conclusions

### Strategy 3: TypeBoundaryConstraints

- **Target**: Test with `i1`, `i2`, `i128` types and vector types in unsigned comparisons
- **Rationale**: `>= 0` constraint semantics differ for small types (e.g., i1 only has 0,1), and coefficient representation may overflow for large types
- **Expected Issue**: Incorrect reasoning about i1 values or overflow in coefficient calculations leading to wrong constraints

### Strategy 4: ConstraintOrdering

- **Target**: Create scenarios where multiple variables are added simultaneously or Value2Index changes during constraint addition
- **Rationale**: The fix adds constraints in a loop after updating Value2Index, but indices might shift if size changes during the loop
- **Expected Issue**: Wrong coefficient index if Value2Index.size() changes during constraint creation loop, leading to incorrect constraint

### Strategy 5: PoisonInConstraintSystem

- **Target**: Variables with `poison` or `undef` possible values used in unsigned comparisons
- **Rationale**: `Var >= 0` assumes defined value, but poison/undef could make constraint meaningless or incorrectly constrain poison values
- **Expected Issue**: Incorrect optimization assuming non-negativity for potentially poison values, violating poison semantics

The patch adds constraints that reflect the mathematical truth: all unsigned LLVM values are non-negative when interpreted as mathematical integers. This is semantically correct and doesn't over-constrain valid programs.

### 3. Patch Benefits
The patch successfully addresses the limitation described in the commit: the unsigned constraint system now knows all variables are non-negative, enabling:
- More effective reasoning about unsigned comparisons
- Better transfer between signed and unsigned systems
- Improved optimization power
