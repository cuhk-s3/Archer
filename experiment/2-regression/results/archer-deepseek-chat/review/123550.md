## Test Strategies

### Strategy 1: Signed large MaxBECount truncation

- **Target**: Create test with signed loop counter producing MaxBECount with high bit set in wider type
- **Rationale**: Tests the assumption that active bits <= BitWidth guarantees safe truncation for signed values. A signed MaxBECount with the high bit set (negative in two's complement) may have fewer active bits than needed to preserve signed value after truncation.
- **Expected Issue**: Truncation may change signed interpretation, leading to incorrect range computation or overflow detection

### Strategy 2: MaxBECount at power-of-two boundary

- **Target**: Test with MaxBECount = 2^31 in i64 for i32 AddRec
- **Rationale**: Active bits = 32, passes the check, but truncating 0x80000000 from i64 to i32 changes from positive (in i64) to negative (in i32 signed interpretation). This tests signedness preservation during truncation.
- **Expected Issue**: Range computation may misinterpret iteration count due to signed/unsigned confusion after truncation

### Strategy 3: Mixed signed/unsigned contexts

- **Target**: Test where MaxBECount derived from signed comparison but used for unsigned AddRec range
- **Rationale**: Tests assumption that truncation preserves value semantics across signedness contexts. The MaxBECount may come from signed SCEV analysis but be used for unsigned range computation.
- **Expected Issue**: Incorrect range due to signed/unsigned mismatch after truncation, potentially causing overflow misdetection

### Strategy 4: Zero MaxBECount edge case

- **Target**: Test with MaxBECount = 0 in wider type (e.g., i64 zero for i32 AddRec)
- **Rationale**: Tests edge case where active bits = 0. While truncation of zero is always safe, this tests the boundary condition and ensures no division by zero or other edge case issues.
- **Expected Issue**: Potential edge case bugs in range computation with zero iteration count

### Strategy 5: MaxBECount with padding bits

- **Target**: Test with MaxBECount having inactive high bits that aren't zero (e.g., 0xFFFF...F000 in i64 for value 0 in lower 32 bits)
- **Rationale**: Tests whether high bits beyond active bits affect truncation. getActiveBits() ignores high zeros, but non-zero bits beyond active bits range could indicate poisoned or undefined bits.
- **Expected Issue**: Truncation may discard non-zero high bits that could affect range analysis or indicate undefined behavior
