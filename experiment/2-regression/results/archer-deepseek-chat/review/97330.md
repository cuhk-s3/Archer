## Test Strategies

### Strategy 1: Conditional Context with Multi-Use Instructions

- **Target**: Create test where an instruction has multiple uses, and a CondContext affects only some uses
- **Rationale**: SimplifyMultipleUseDemandedBits might incorrectly apply conditional context to all uses, leading to incorrect simplification that breaks other uses of the instruction
- **Expected Issue**: Incorrect simplification that corrupts value for uses not affected by the conditional context

### Strategy 2: Nested Conflicting Conditional Contexts

- **Target**: Test with nested conditions where inner context contradicts outer context
- **Rationale**: The patch might not handle conflicting CondContext information correctly when contexts are nested or combined
- **Expected Issue**: Incorrect known bits computation leading to wrong simplification due to unresolved context conflicts

### Strategy 3: Vector Operations with Element-Specific Conditions

- **Target**: Vector operations where condition only applies to some vector elements
- **Rationale**: CondContext is applied uniformly to all vector elements, but conditions might only affect specific elements
- **Expected Issue**: Incorrect simplification for unaffected vector elements, potentially introducing poison or wrong values

### Strategy 4: Poison Flags Preservation in Conditional Context

- **Target**: Instructions with poison-generating flags (nsw, nuw, exact) in conditional contexts
- **Rationale**: Simplification might retain flags that are only valid in the conditional context but not unconditionally
- **Expected Issue**: Poison propagation when condition doesn't hold, violating poison safety guarantees

### Strategy 5: Depth-Limited Context Propagation in Recursion

- **Target**: Deep recursion chains where context should be preserved across depth boundaries
- **Rationale**: Context might be incorrectly reset or not propagated properly at recursion depth boundaries
- **Expected Issue**: Loss of context information in deep analysis leading to incorrect simplifications

### Analysis

## Bug Triggering Analysis

After extensive testing with all 5 proposed strategies, no bugs were found in the patch. The `SimplifyDemandedBits` API change to accept `SimplifyQuery` parameter appears to be correctly implemented. Key observations:

1. **Conditional Context Propagation**: The context (`CondContext`, `CxtI`, etc.) is properly propagated through recursive calls
2. **Multi-Use Instructions**: `SimplifyMultipleUseDemandedBits` correctly handles conditional context without incorrectly applying it to all uses
3. **Conflicting Contexts**: Nested conflicting conditions are handled correctly, with UB triggered when assumptions conflict
4. **Vector Operations**: Element-specific conditions don't cause incorrect simplifications
5. **Poison Flags**: Flags are preserved appropriately in conditional contexts
6. **Depth Propagation**: Context propagates correctly through deep recursion chains

## Fix Weakness Analysis

The patch successfully addresses the stated goals:
1. **API Consistency**: By passing `SimplifyQuery` explicitly, context propagation is more reliable
2. **CondContext Support**: The infrastructure is now in place for future `CondContext` usage
3. **One-Use Chain Context**: Original context instruction is retained for one-use chains

**Potential Weaknesses Not Exposed**:
- Edge cases with extremely deep recursion (beyond `MaxAnalysisRecursionDepth`)
- Complex interactions between `DC` (dominating conditions) and `CC` (injected CondContext)
- Vector operations with per-element conditions (though this may be a design limitation rather than a bug)

**Conclusion**: The patch appears robust and correctly implements the intended functionality. The test strategies were comprehensive but didn't reveal any bugs, suggesting the implementation is correct for the tested scenarios.
