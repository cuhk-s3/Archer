## Test Strategies

### Strategy 1: Test abs poison flag with bitwidth reduction

- **Target**: Create test where abs with true poison flag gets bitwidth reduced, potentially changing which value is INT_MIN in reduced representation
- **Rationale**: The patch sets abs second argument to false when reducing bitwidth, but if the reduced representation has a different INT_MIN value, this could incorrectly suppress poison for values that should be poisoned
- **Expected Issue**: abs returns non-poison for values that should be poisoned in reduced representation

# Bug Analysis Report

## Bug Triggering Analysis
The test case demonstrates a potential issue with the patch's handling of `abs` intrinsic poison flags during bitwidth reduction. The key issue is:

1. **Original code**: `abs(i32 %x_ext, i1 true)` where `%x_ext` is `sext i8 %x to i32`
2. **Transformed code**: `abs(<4 x i8> %vec, i1 false)` (as shown in the transformation output)

The problem arises when the input value `%x` is `-128` (i8 INT_MIN):
- Original: `abs(i32 -128, i1 true)` would return poison because i32 -128 is INT_MIN in i32 representation
- Transformed: `abs(i8 -128, i1 false)` would NOT return poison because the poison flag is forced to false

However, the actual transformation shows `abs(<4 x i8> %vec, i1 false)`, which is correct for this specific case because:
- The values come from i8 loads, range [-128, 127]
- i8 -128 is INT_MIN in i8 representation
- The patch sets poison flag to false when reducing bitwidth

**The real issue**: If the bitwidth reduction changes which value represents INT_MIN (e.g., reducing from i32 to i16 where i16 INT_MIN is -32768), and the original value happens to be the new INT_MIN in the reduced representation, the patch incorrectly suppresses poison.

## Fix Weakness Analysis
The patch has a fundamental weakness in its handling of `abs` poison flags:

1. **Assumption**: Setting poison flag to false when reducing bitwidth is always safe
2. **Reality**: This is only safe if the reduced representation's INT_MIN cannot be reached from the original values
3. **Missing check**: The patch doesn't verify that values cannot be INT_MIN in the reduced representation

The fix in `vectorizeTree()` at lines 12598-12600:
```cpp
if (ID == Intrinsic::abs && It != MinBWs.end() &&
    It->second.first < DL->getTypeSizeInBits(CEI->getType()))
  ScalarArg = Builder.getFalse();
```

This unconditionally sets the poison flag to false when reducing bitwidth, without checking if any input values could be INT_MIN in the reduced representation.
