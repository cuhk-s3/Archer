- **Target**: Alternate casts involving non-integer types (fptosi/fptoui, ptrtoint/inttoptr)

- **Expected Issue**: Missing truncation/extension when needed for non-integer casts or incorrect cost model.

## Bug Triggering Analysis

After extensive testing with multiple strategies targeting the fix for PR87477, I have not found any bugs in the current implementation. The fix appears to be robust and handles the edge cases correctly. Here's my analysis:

3. **NonIntegerAlternateCasts**: Tested alternate casts involving non-integer types (fptosi/fptoui). The fix's integer type checks correctly skip the bitwidth comparison logic for non-integer casts, which is appropriate since the fix specifically addresses integer type handling.

4. **SignVsZeroExtensionAlternate**: Tested alternate sext/zext casts where destination bitwidth > source bitwidth. The fix's condition `BWSz <= SrcBWSz` correctly doesn't trigger for extension cases, and the SLP vectorizer handles extensions separately through its normal vectorization logic.

5. **VectorIntegerAlternateCasts**: Tested alternate casts on vector integer types. The transformation shows correct handling with shufflevector operations.

### Fix Weakness Analysis

The fix appears to be well-designed and addresses the specific issue mentioned in PR87477:

1. **Correct Bitwidth Comparison**: The fix uses `DL->getTypeSizeInBits()` to compare actual type sizes, which is more accurate than the previous approach.

2. **Proper Integer Type Guarding**: The integer type checks (`SrcSclTy->isIntegerTy() && ScalarTy->isIntegerTy()` in cost calculation and `LHS->getType()->isIntOrIntVectorTy() && ScalarTy->isIntegerTy()` in code generation) appropriately limit the fix to integer types where the bitwidth analysis is meaningful.

3. **No Unnecessary Truncation**: The condition `BWSz < SrcBWSz` (not `<=`) ensures truncation is only applied when actually needed, avoiding unnecessary instructions for equal bitwidth cases.

4. **Handles Extension Cases Appropriately**: For cases where `BWSz > SrcBWSz` (extensions), the fix correctly doesn't apply, allowing the normal SLP vectorization logic to handle sign/zero extensions.

### Why No Bugs Were Found:

The fix is conservative and targeted:
- It only modifies behavior for integer types where MinBWs analysis is involved
- It adds proper bitwidth comparison before deciding on truncation
- It maintains compatibility with existing SLP vectorization logic for other cases

The test cases I generated, including the comprehensive test that mimics potential issue scenarios, all transformed correctly without showing any execution differences between original and optimized code.

The fix for PR87477 appears to be correct and handles the edge cases properly.
