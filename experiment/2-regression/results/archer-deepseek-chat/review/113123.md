## Test Strategies

### Strategy 1: FastMathFlagUnionMismatch

- **Target**: Fast-math flags on fcmp instructions
- **Rationale**: Union of flags (LHS->getFastMathFlags() | RHS->getFastMathFlags()) might be incorrect when flags conflict (e.g., one has nnan, other doesn't), leading to incorrect NaN handling in transformed code.
- **Expected Issue**: Incorrect NaN handling when only one comparison has nnan flag

### Strategy 2: NegativeZeroSignBit

- **Target**: -0.0 constants and unordered predicates
- **Rationale**: fabs(-0.0) = +0.0 changes sign bit, which matters for unordered comparisons and sign-sensitive operations. Need to verify semantic equivalence for -0.0 cases.
- **Expected Issue**: Different results for -0.0 inputs with unordered predicates (ULT/ULE/UGT/UGE)

### Strategy 3: VectorUndefConstant

- **Target**: Vector constants with undef elements
- **Rationale**: m_APFloatAllowUndef allows undefs in vector constants, but bitwiseIsEqual and neg operations on such constants may have undefined behavior or produce incorrect results.
- **Expected Issue**: Incorrect transformation or crash when handling vectors with undef elements

### Strategy 4: PredicateSwappingEdgeCases

- **Target**: All predicate combinations with swapped predicate check
- **Rationale**: Verify getSwappedPredicate works correctly for all predicate pairs (OLT↔OGT, OLE↔OGE, ULT↔UGT, ULE↔UGE) and that the transformation handles all valid combinations.
- **Expected Issue**: Missing or incorrect predicate pairs causing missed optimizations or miscompilations

### Strategy 5: ZeroConstantEdgeCase

- **Target**: Test cases where C = 0 or C = -0.0
- **Rationale**: bitwiseIsEqual(neg(*RHSC)) for ±0.0 needs verification, and mathematical correctness when range collapses to point (x < 0 && x > 0 is always false).
- **Expected Issue**: Incorrect transformation or missed optimization for zero constants
