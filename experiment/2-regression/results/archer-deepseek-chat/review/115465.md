## Test Strategies

### Strategy 1: Poison in shuffle mask elements

- **Target**: Shuffle masks with poison/undef elements
- **Rationale**: Moving shuffle after select/phi could change which lanes become poison, potentially exposing poison that was previously guarded by the select condition
- **Expected Issue**: Different poison propagation leading to miscompilation or unexpected UB

### Strategy 2: Constant RHS with poison elements

- **Target**: Constant vectors containing poison/undef elements as RHS operand
- **Rationale**: Poison in constant could interact differently with select condition when shuffle is moved, potentially leaking poison into lanes that were previously safe
- **Expected Issue**: Poison propagation changes causing miscompilation

### Strategy 3: Fast-math flag propagation

- **Target**: Floating-point operations with fast-math flags in select/phi arms
- **Rationale**: Fast-math flags might not be preserved when shuffle is moved across control flow, leading to incorrect floating-point semantics
- **Expected Issue**: Loss of optimization flags or incorrect floating-point behavior

### Strategy 4: Multiple uses with different operations

- **Target**: Select/phi with multiple uses by different instruction types
- **Rationale**: Transformation might incorrectly apply when not all users are identical (for phi) or fail to apply when it should (for select with FoldWithMultiUse)
- **Expected Issue**: Either missed optimization opportunity or incorrect transformation

### Strategy 5: NSW/NUW flag preservation

- **Target**: Integer operations with wrap flags (nsw/nuw) in select arms
- **Rationale**: Moving shuffle could change poison generation behavior for wrapped operations, especially if shuffle mask selects specific elements
- **Expected Issue**: Incorrect wrap semantics or unexpected poison generation

### Analysis

## Bug Triggering Analysis

The test case `foldselect0_poison_mask` demonstrates a potential issue with the new transformation. The original code has:
- `shufflevector(select(c, <42,7>, <0,1>), poison, <1, poison>)`

After the transformation, it becomes:
- `select(c, shufflevector(<42,7>, poison, <1, poison>), shufflevector(<0,1>, poison, <1, poison>))`

The issue is that when the shuffle mask contains poison elements, moving the shuffle after the select could change poison propagation semantics. In the original code, the poison in the mask creates poison in the result lane regardless of the select condition. In the transformed code, the poison creation happens inside each select arm, which might interact differently with the select condition.

## Fix Weakness Analysis

The fix assumes that shuffling after select/phi is always equivalent to shuffling before, but this may not hold when:
1. Shuffle masks contain poison elements
2. Constant RHS operands contain poison elements
3. Operations in select arms have poison-generating flags (nsw/nuw, fast-math)

The `FoldOpIntoSelect` and `foldOpIntoPhi` helpers may not properly handle poison propagation when moving operations across control flow boundaries. The transformation needs to ensure that poison semantics are preserved, especially when:
- Poison in shuffle masks creates poison in result lanes
- Poison in constant vectors could leak into lanes that were previously safe
- Fast-math flags could change NaN/poison behavior

The verification tools showed the transformation as "correct" but the differential testing revealed execution issues (crashes in llubi), suggesting there may be subtle poison/UB issues that aren't caught by the current verification.
