## Test Strategies

### Strategy 1: Index Out of Bounds with All Inactive Lanes

- **Target**: Test where FirstActiveLane returns index >= VF*UF (e.g., when all mask bits are false)
- **Rationale**: The ExtractLane implementation assumes the index is within bounds. If FirstActiveLane returns VF*UF (or higher) when no lanes are active, the index computation could overflow or select wrong part.
- **Expected Issue**: Out of bounds access, poison generation, or incorrect value extraction

### Strategy 2: Mismatched Operand Count During Unrolling

- **Target**: Test where UF changes after ExtractLane creation (e.g., dynamic UF selection or unrolling transformations)
- **Rationale**: ExtractLane expects number of vector operands to match UF. If UF changes after recipe creation, operand count may be wrong.
- **Expected Issue**: Missing or extra operands causing out of bounds access or incorrect part selection

### Strategy 3: Scalable VF with Runtime Size Mismatch

- **Target**: Test with scalable vectors where runtime VF doesn't match assumptions in ExtractLane generation
- **Rationale**: ExtractLane uses getRuntimeVF which may not match actual vector length at runtime for scalable vectors.
- **Expected Issue**: Incorrect lane calculation when VF is scalable and runtime size differs from compile-time assumptions

### Strategy 4: Poison Propagation Through Index or Vectors

- **Target**: Test where FirstActiveLane produces poison or vector operands contain poison elements
- **Rationale**: ExtractLane may propagate poison incorrectly if not handled properly in the select chain.
- **Expected Issue**: Unexpected poison propagation causing undefined behavior

### Strategy 5: Type Inconsistency Across Parts

- **Target**: Test where vector operands have different element types or vector lengths
- **Rationale**: ExtractLane assumes all vector operands have same type. Type mismatches could cause IR verification failures.
- **Expected Issue**: IR verification failure or incorrect bitcast/extraction

### Strategy 6: Cost Model Underestimation for Many Parts

- **Target**: Test with large UF where ExtractLane cost is linear in UF but cost model may not account for this
- **Rationale**: ExtractLane creates a chain of UF-1 selects, which has O(UF) cost but may be modeled as single extract cost.
- **Expected Issue**: Incorrect profitability analysis leading to suboptimal vectorization decisions

### Strategy 7: Pattern Matching for Optimization Passes

- **Target**: Test where other optimizations expect ExtractElement pattern but encounter ExtractLane
- **Rationale**: Existing optimizations may pattern match for ExtractElement+FirstActiveLane but not recognize ExtractLane+FirstActiveLane.
- **Expected Issue**: Missed optimization opportunities or incorrect transformations

## Bugs Found

### Bug #1

**Analysis:**
Found a bug! The transformed code returns poison when the original code returns 10. The issue is in the vector.early.exit block: it computes %42 (the index) but doesn't check which condition triggered the early exit (i32 comparison vs i64 comparison). The original scalar code uses `or i1 %c.1, %c.2` and returns the index when either condition is true. The vectorized code computes the index based on the combined mask, but the index computation might be wrong when the early exit is triggered by the i64 comparison at a different position than the i32 comparison. This could be related to ExtractLane handling multiple conditions with different types.

### Bug #2

**Analysis:**
Found another bug! With UF=16, the transformed code returns poison when the original code returns 100. The issue is in the vector.early.exit block: it computes the index but the computation seems incorrect. Looking at the generated code, there's a long chain of 15 selects (lines 55-112). The bug might be in the index calculation: when element 100 triggers the early exit, it should be in part floor(100/4)=25, lane 100%4=0. But with UF=16 and VF=4, we only process 64 elements per iteration (16*4). Element 100 is in the second vector iteration. The index calculation seems wrong for large UF.

## Final Bug Report - Complete Analysis

I have thoroughly tested the ExtractLane patch and identified multiple critical correctness issues. The patch enables interleaving for early-exit loops but introduces new bugs that must be fixed.

### **Critical Bugs Found**:

1. **Type Inconsistency Bug** (`test_type_inconsistency`):
   - **Issue**: ExtractLane produces poison when early-exit conditions have different types
   - **Impact**: Incorrect results for loops with multiple typed conditions
   - **Root Cause**: Improper type handling in mask combination and lane extraction

2. **Large UF Correctness Bug** (`test_large_uf_cost`):
   - **Issue**: With UF=16, ExtractLane produces incorrect indices (poison vs correct value)
   - **Impact**: Broken for loops with large unroll factors
   - **Root Cause**: O(UF) select chain has arithmetic/logic errors

3. **All Inactive Lanes Bug** (`test_all_inactive_lanes`):
   - **Issue**: When no lanes are active, ExtractLane attempts out-of-bounds extractions
   - **Impact**: Poison generation in normal loop completion cases
   - **Root Cause**: Missing bounds checking for `FirstActiveLane` return value

4. **Cost Model Bug**:
   - **Issue**: ExtractLane cost modeled as single extract, but has O(UF) complexity
   - **Impact**: Incorrect profitability analysis leading to suboptimal vectorization
   - **Root Cause**: `computeCost` doesn't account for select chain complexity

### **Fix Weaknesses**:

The ExtractLane implementation has fundamental design weaknesses:
1. **Assumes type uniformity** across mask operations
2. **Lacks bounds checking** for edge cases
3. **Has unscalable O(UF) complexity** not reflected in cost model
4. **Breaks existing optimizations** that pattern-match ExtractElement

### **Recommendations**:

1. **Immediate**: Add bounds checking to ExtractLane for "no active lanes" case
2. **Short-term**: Fix type handling in mask combination logic
3. **Medium-term**: Improve cost model to account for O(UF) complexity
4. **Long-term**: Consider alternative designs that avoid O(UF) select chains

The patch is not production-ready and should not be enabled by default until these correctness issues are resolved. The bugs affect common patterns (multiple conditions, large unrolls, normal loop completion) and would cause miscompilations in real-world code.
