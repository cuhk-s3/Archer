## Test Strategies

### Strategy 1: UndefPoisonCondition

- **Target**: Select condition with undef/poison
- **Rationale**: The optimization assumes condition is well-defined. Undef/poison conditions could lead to incorrect constant folding if we assume one arm is always taken.
- **Expected Issue**: May fold select arm to constant when condition is undef/poison, changing program semantics from undef to defined value.

### Strategy 2: DeepDependencyChain

- **Target**: Select arms with deep instruction nesting (> MaxAnalysisRecursionDepth)
- **Rationale**: hasAffectedValue has recursion depth limit of 6. Deep dependency chains won't be recognized as affected, missing optimization opportunities or incorrectly not optimizing.
- **Expected Issue**: Fails to fold to constant when it should, or incorrectly folds due to incomplete affected values tracking.

### Strategy 3: ComplexLogicalInversion

- **Target**: Complex logical conditions (AND/OR combinations) with inversion for false arm
- **Rationale**: Inversion logic in computeKnownBitsFromCond must correctly handle all logical operator combinations. Complex conditions may be incorrectly inverted.
- **Expected Issue**: Incorrect KnownBits computed for false arm, leading to wrong constant folding.

### Strategy 4: VectorPoisonElements

- **Target**: Vector select with poison elements in condition mask
- **Rationale**: Vector select with poison in condition elements - optimization may assume all lanes follow same condition.
- **Expected Issue**: May fold vector lanes incorrectly when some condition elements are poison.

### Strategy 5: ConflictingAssumptions

- **Target**: Condition that implies contradictory KnownBits (e.g., from logical AND of impossible conditions)
- **Rationale**: If condition is logically impossible (A && !A), KnownBits may have conflict. Need to handle KnownBits.hasConflict() properly.
- **Expected Issue**: May incorrectly fold to constant or crash when KnownBits has conflict.

### Strategy 6: MultiUseAffectedValue

- **Target**: Value appears in both select arms with different context
- **Rationale**: Same value used in both arms but affected differently by condition (e.g., %x in true arm, %x+1 in false arm). Current logic uses same AffectedValues set for both.
- **Expected Issue**: May incorrectly track affected values or miss optimization opportunities.

### Strategy 7: PointerConditionAffectingInts

- **Target**: Pointer comparison condition affecting integer computations
- **Rationale**: Pointer comparisons (e.g., ptr != null) can affect integer values through ptrtoint. findValuesAffectedByCondition may miss these connections.
- **Expected Issue**: Fails to recognize integer values affected by pointer condition, missing optimization.

### Strategy 8: RecursivePHIDependency

- **Target**: PHI nodes with recursive dependencies through back edges
- **Rationale**: hasAffectedValue has special handling for PHI nodes with depth adjustments. May not correctly handle PHIs that depend on themselves.
- **Expected Issue**: Infinite recursion or incorrect affected value detection for cyclic PHI dependencies.

### Strategy 9: FastMathSelectWithIntArms

- **Target**: Floating-point select with fast-math flags containing integer computations in arms
- **Rationale**: Select may have fast-math flags but arms contain integer operations. Optimization may incorrectly apply or ignore fast-math semantics.
- **Expected Issue**: Incorrect constant folding due to mismatched type/semantic handling.

### Strategy 10: TimeOfCheckTimeOfUse

- **Target**: Condition or affected values modified between analysis and transformation
- **Rationale**: Affected values computed once but IR may be modified between analysis and transformation phases.
- **Expected Issue**: Stale affected values set leading to incorrect transformations.

### Bug #3

**Analysis:**
Found a bug! The optimization incorrectly folds the true arm (x10) to zeroinitializer when condition is true. However, x10 is shufflevector of x, not x itself. When x = <0, 1>, x10 = <1, 0>. When condition is true for first element (x[0]==0), we should take x10[0] = 1, not 0. The optimization incorrectly assumes x10 is affected by the condition in the same way as x, but shufflevector changes the values.

## Summary of Findings

Through comprehensive testing of the InstCombine select optimization using KnownBits of condition, we found two critical bugs:

### Bug 1: Undef/Poison Condition Handling
**Test**: `test_undef_condition_simple`
**Issue**: The optimization doesn't check for `undef`/`poison` conditions. When condition is `undef`, `select undef, X, Y` should return `undef` (or could be either X or Y). The optimization incorrectly assumes the condition holds and folds the true arm to a constant.

**Root Cause**: Missing check for `undef`/`poison` in condition before applying optimization. The comment in the code mentions "would also have to limit to non-undef conditions" but this isn't implemented.

### Bug 2: Incorrect Affected Value Tracking Through Vector Operations
**Test**: `test_vec_select_multiuse_affected`
**Issue**: `hasAffectedValue` incorrectly tracks values through operations like `shufflevector`. When condition constrains `x[i]`, it assumes `shufflevector(x)[i]` is similarly constrained, but `shufflevector` changes element mapping.

**Root Cause**: `hasAffectedValue` only checks if a value depends on an affected value, not how the operation transforms the relationship. Operations like `shufflevector`, `extractelement`, `insertelement` can break the element-wise relationship between values and conditions.

### Fix Weaknesses Identified:
1. **No undef/poison guard**: Optimization should skip when condition is `undef` or `poison`
2. **Overly simplistic dependency tracking**: `hasAffectedValue` needs to understand how operations transform the semantic relationship
3. **Missing vector operation analysis**: Need special handling for vector operations that rearrange elements
4. **Incomplete type checking**: Should verify operations preserve the needed relationship

### Recommendations for Fix:
1. Add check: `if (isa<UndefValue>(CondVal) || isa<PoisonValue>(CondVal)) return nullptr;`
2. Enhance `hasAffectedValue` to track how operations transform element indices
3. Limit optimization to operations that preserve element-wise relationships
4. Add special cases for vector operations or disable optimization for complex vector patterns

The optimization has the right idea but needs stronger safeguards to handle edge cases properly.
