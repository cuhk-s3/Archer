## Test Strategies

### Strategy 1: Poison Flag Propagation

- **Target**: IR flags on zext/trunc operations
- **Rationale**: The transformation copies IR flags but may not handle poison-generating flags correctly. zext with nneg flag or trunc with nuw/nsw flags could create poison when moving operations.
- **Expected Issue**: Transformation creates poison values from previously defined values, causing miscompilation.

### Strategy 2: Shift Amount Truncation Semantics

- **Target**: LSHR with large shift amounts
- **Rationale**: For LSHR, truncating the shift amount could change semantics: large shift amount that would be poison (≥ bitwidth) becomes defined after truncation if it fits in smaller type.
- **Expected Issue**: Poison value becomes defined value, changing program semantics.

### Strategy 3: KnownBits Imprecision

- **Target**: Complex bitwise expressions with imprecise KnownBits
- **Rationale**: KnownBits analysis may be imprecise for complex expressions, leading to incorrect countMaxActiveBits estimation. Could allow transformation when result actually uses >X bits.
- **Expected Issue**: Transformation loses high bits that are actually used, causing incorrect results.

### Strategy 4: Non-Binary Op Users

- **Target**: ZExt operand used by non-binary operations
- **Rationale**: The transformation requires all users of ZExt operand to be binary ops. If there are other users (select, phi, store), transformation aborts but cost calculation may have already considered them.
- **Expected Issue**: Inconsistent behavior or missed optimization opportunities.

### Strategy 5: Insertion Point Safety

- **Target**: OtherOperand with complex dominance relationships
- **Rationale**: Inserting trunc after OtherOperand may not be safe if OtherOperand has multiple uses in different basic blocks or if insertion point dominates uses.
- **Expected Issue**: Trunc inserted in wrong location causing dominance violation or incorrect program semantics.

### Analysis

## Bug Triggering Analysis\n\nThe transformation `shrinkType` in VectorCombine has a critical bug when handling `lshr` operations with large shift amounts. The transformation converts:\n\n`lshr(zext(<N x iX> %x to <N x iY>), <N x iY> %y)` → `zext(lshr(<N x iX> %x, trunc(<N x iY> %y) to <N x iX>))`\n\nThis is incorrect when `%y` contains values ≥ Y (the larger bitwidth). According to LLVM semantics:\n- For `lshr`, if the shift amount is ≥ the bitwidth of the first operand, the result is poison\n- When truncating `%y` from iY to iX, large values (≥ Y) become smaller values that fit in iX\n- This changes poison values into defined values, violating semantics\n\n**Example**:\n- `x = <i8 255>`, `zext(x) = <i32 0x000000FF>`\n- `y = <i32 40>` (≥ 32)\n- Original: `lshr(0x000000FF, 40)` = poison (shift amount ≥ 32)\n- After transformation: `trunc(40) = 8`, `lshr(255, 8) = 0`, `zext(0) = 0`\n- Poison → defined value 0\n\n## Fix Weakness Analysis\n\nThe fix has several weaknesses:\n\n1. **Missing shift amount validation**: The transformation doesn't check if truncating the shift amount preserves poison semantics. For `lshr`, it should ensure that truncation doesn't convert poison-generating shift amounts into valid ones.\n\n2. **Incomplete KnownBits analysis**: The `countMaxActiveBits()` check only ensures the result fits in X bits, but doesn't consider shift amount semantics.\n\n3. **No poison flag handling**: The transformation copies IR flags but may not properly handle poison-generating flags like `nneg` on `zext`.\n\n4. **Cost model issues**: The transformation uses TTI cost estimates but doesn't account for semantic correctness issues that could lead to miscompilation.\n\n5. **Non-binary user handling**: The transformation aborts if zext has non-binary users, but the cost calculation may have already considered them.\n\n6. **Insertion point safety**: The trunc insertion logic (`Builder.SetInsertPoint(OI->getNextNode())`) may not handle cross-block dominance correctly.\n\nThe test case demonstrates this by using shift amount 40 (≥ 32) which becomes 8 after truncation from i32 to i8, changing poison to a defined value.
