## Test Strategies

### Strategy 1: MaxBECount at boundary value

- **Target**: Create a test where MaxBECount equals 2^N - 1 (maximum value fitting in N bits) stored in a wider type (e.g., 2^32 - 1 in i64), and AddRec is N-bit (e.g., i32)
- **Rationale**: This tests the boundary case where getActiveBits() equals BitWidth exactly. The fix should allow truncation in this case, and the range computation should produce correct results.
- **Expected Issue**: If the condition uses < instead of <=, or if there's an off-by-one error in getActiveBits(), the truncation might not happen when it should, leading to missed range optimization or incorrect range computation.

### Strategy 2: MaxBECount just over boundary

- **Target**: Create a test where MaxBECount equals 2^N (minimum value not fitting in N bits) stored in a wider type, and AddRec is N-bit
- **Rationale**: This tests that truncation does NOT happen when the value doesn't fit. The condition getActiveBits() <= BitWidth should be false, and the code should fall through without using MaxBECount for range computation.
- **Expected Issue**: If the condition is incorrectly implemented, truncation might happen when it shouldn't, leading to incorrect range computation (truncated value would be 0 instead of 2^N).

### Strategy 3: MaxBECount with large type but small value

- **Target**: Create a test where MaxBECount is stored in a very wide type (e.g., i128) but has a small value (e.g., 100), and AddRec is a narrower type (e.g., i32)
- **Rationale**: This tests the main scenario the fix is addressing: MaxBECount has a larger bit width than AddRec, but the value fits. The fix should allow truncation and enable range computation.
- **Expected Issue**: Without the fix, this case would not use MaxBECount for range computation, resulting in less precise ranges. With the fix, if there's a bug, the range might be incorrectly computed.

### Strategy 4: MaxBECount with sign bit set after truncation

- **Target**: Create a test where MaxBECount equals 2^(N-1) (e.g., 2^31 for i32), which has the sign bit set when truncated to N bits
- **Rationale**: This tests that the range computation correctly handles the case where the truncated MaxBECount has the sign bit set. Since MaxBECount is a count (unsigned), it should be treated as 2^(N-1), not as a negative number.
- **Expected Issue**: If the range computation incorrectly treats the truncated value as signed, it might compute incorrect ranges or trigger assertions.

### Strategy 5: MaxBECount = 0 edge case

- **Target**: Create a test where MaxBECount is 0 (loop doesn't iterate) stored in a wider type, and AddRec is a narrower type
- **Rationale**: This tests the edge case where getActiveBits() returns 0. The condition 0 <= BitWidth should be true for any BitWidth >= 1, and truncation should result in 0.
- **Expected Issue**: If there's a bug in handling the zero case, the range computation might produce incorrect results for loops that don't iterate.
