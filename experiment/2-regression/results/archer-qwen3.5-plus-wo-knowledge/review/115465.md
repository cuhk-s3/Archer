## Test Strategies

### Strategy 1: Test shufflevector(select) with non-constant RHS

- **Target**: shuffle_select.ll
- **Rationale**: The patch only applies when RHS is a constant (match(RHS, m_Constant())). Test cases where RHS is not a constant (e.g., undef, poison, or another vector) should NOT trigger the transformation, but we should verify the code doesn't incorrectly apply the transform.
- **Expected Issue**: If the transformation is incorrectly applied when RHS is not constant, it could produce incorrect code or crash.

### Strategy 2: Test shufflevector(select) with multi-use select

- **Target**: shuffle_select.ll
- **Rationale**: FoldOpIntoSelect requires the select to have one use (unless FoldWithMultiUse is true, which is not passed in the patch). Test cases where the select has multiple uses should NOT trigger the transformation.
- **Expected Issue**: If the transformation is applied to a multi-use select, it could incorrectly duplicate the select or produce wrong results.

### Strategy 3: Test shufflevector(select) where neither arm is constant

- **Target**: shuffle_select.ll
- **Rationale**: FoldOpIntoSelect requires at least one of TV or FV to be a constant (line 1695-1696). Test cases where both arms of the select are non-constant values should NOT trigger the transformation.
- **Expected Issue**: If the transformation is applied when neither arm is constant, FoldOpIntoSelect will return nullptr, but we should verify no incorrect behavior occurs.

### Strategy 4: Test shufflevector(phi) with multi-use phi

- **Target**: Create new test or modify existing phi test
- **Rationale**: foldOpIntoPhi requires the phi to have one use (or all uses are identical). Test cases where the phi has multiple different uses should NOT trigger the transformation.
- **Expected Issue**: If the transformation is incorrectly applied to a multi-use phi with different operations, it could produce incorrect code.

### Strategy 5: Test shufflevector(phi) with non-phi-translatable operands

- **Target**: Create new test or modify existing phi test
- **Rationale**: foldOpIntoPhi requires all operands to be phi-translatable (lines 1789-1810). Test cases where the shufflevector has operands that cannot be phi-translated should NOT trigger the transformation.
- **Expected Issue**: If the transformation is applied when operands are not phi-translatable, it could produce incorrect code or crash during phi-translation.

### Strategy 6: Test shufflevector(phi) with multiple non-simplified incoming values

- **Target**: Create new test or modify existing phi test
- **Rationale**: foldOpIntoPhi allows at most one non-simplified incoming value (lines 1837-1839). Test cases where multiple incoming values cannot be simplified should NOT trigger the transformation.
- **Expected Issue**: If the transformation is applied when multiple incoming values cannot be simplified, it could produce incorrect code.

### Strategy 7: Test shufflevector(phi) with conditional branch predecessor

- **Target**: Create new test or modify existing phi test
- **Rationale**: foldOpIntoPhi requires the predecessor block to have an unconditional branch (line 1846-1848). Test cases where the phi has a predecessor with a conditional branch should NOT trigger the transformation for that incoming value.
- **Expected Issue**: If the transformation is applied across a conditional branch, it could incorrectly move computations to wrong paths.

### Strategy 8: Test shufflevector(phi) with loop backedge

- **Target**: Create new test or modify existing phi test
- **Rationale**: foldOpIntoPhi explicitly rejects pushing operations across loop backedges (lines 1859-1863). Test cases where the phi is at a loop header with a backedge should NOT trigger the transformation.
- **Expected Issue**: If the transformation is applied across a backedge, it could cause infinite combine loops or incorrect code motion.

### Strategy 9: Test shufflevector(select) with bool select type

- **Target**: shuffle_select.ll
- **Rationale**: FoldOpIntoSelect skips bool selects with constant operands (lines 1698-1700). Test cases where the select returns i1 or <N x i1> should NOT trigger the transformation.
- **Expected Issue**: If the transformation is applied to bool selects, it could interfere with folding to logical ops.

### Strategy 10: Test shufflevector(select) min/max idiom preservation

- **Target**: shuffle_select.ll
- **Rationale**: FoldOpIntoSelect has special handling to preserve min/max idioms (lines 1702-1715). Test cases where the select condition is an fcmp used exclusively by the select in a min/max pattern should NOT trigger the transformation.
- **Expected Issue**: If the transformation breaks the min/max idiom, it could hinder other analyses that recognize these patterns.
