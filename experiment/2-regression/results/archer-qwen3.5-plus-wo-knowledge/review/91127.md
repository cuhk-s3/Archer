## Test Strategies

### Strategy 1: Test trunc nuw + sext combination

- **Target**: trunc nuw combined with sext in icmp
- **Rationale**: The patch explicitly handles trunc nuw + zext (for unsigned/equality) and trunc nsw + sext/zext (for all predicates), but does not handle trunc nuw + sext. This combination should NOT be folded for signed comparisons as it would be incorrect. Test to ensure this case is not incorrectly optimized.
- **Expected Issue**: Incorrect transformation of icmp with trunc nuw and sext, potentially changing signed comparison semantics when the truncated value's sign differs from the original

### Strategy 2: Test mixed nowrap flags in trunc+trunc for unsigned comparison

- **Target**: icmp with two truncs having different nowrap flags (one nuw, one nsw)
- **Rationale**: The code computes NoWrapFlags as the AND of both trunc's flags. For unsigned comparisons, it requires NoWrapFlags to be non-zero. When one trunc has nuw and the other has nsw, the AND is 0, correctly rejecting the fold. Test to verify this rejection happens and doesn't incorrectly fold.
- **Expected Issue**: Incorrect folding of icmp ult/ugt with mixed nowrap flags (nuw on one trunc, nsw on the other), which could produce wrong results when values have different sign extension patterns

### Strategy 3: Test trunc nsw + zext with negative zext operand in signed comparison

- **Target**: icmp slt/sgt with trunc nsw and zext where zext operand can be negative when interpreted as signed
- **Rationale**: The patch claims trunc nsw + zext/sext works for all predicates. However, zext always produces non-negative values. When comparing with trunc nsw (which can produce negative values), the semantics depend on the nsw guarantee. Test edge cases where the zext operand's original type has a sign bit that could affect the comparison.
- **Expected Issue**: Incorrect comparison result when zext operand's original value has sign bit set, causing mismatch between truncated signed comparison and extended comparison

### Strategy 4: Test trunc+trunc with different source types and multi-use operands

- **Target**: icmp with two truncs having different source types where truncs have multiple uses
- **Rationale**: The code requires both trunc operands to have exactly one use when X and Y have different types. This is to avoid creating casts when the truncs are used elsewhere. Test cases where one trunc has one use and the other has multiple uses to verify the fold is correctly rejected.
- **Expected Issue**: Incorrect optimization when trunc operands have asymmetric use counts, potentially creating invalid IR or changing semantics

### Strategy 5: Test trunc nsw + zext with equality predicate and boundary values

- **Target**: icmp eq/ne with trunc nsw and zext at boundary values (e.g., 0, max signed, max unsigned)
- **Rationale**: Equality comparisons with trunc nsw + zext should work, but boundary values might expose issues with the nsw guarantee. Test values at the boundary of the truncation range to ensure the nsw condition is properly respected.
- **Expected Issue**: Incorrect equality comparison when values are at the boundary of the truncation range, where nsw guarantee might not hold or zext behavior differs
