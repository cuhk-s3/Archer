## Test Strategies

### Strategy 1: Mixed signedness in trunc subtree

- **Target**: Test with a trunc subtree where some values require sign-extension and others require zero-extension when restored
- **Rationale**: The patch computes IsSigned per TreeEntry based on SExt opcode or isKnownNonNegative check. If a trunc subtree has mixed signedness requirements across different lanes, the patch may incorrectly apply a single signedness flag to all values in the subtree, leading to incorrect sign/zero extension when restoring values.
- **Expected Issue**: Incorrect sign extension when restoring truncated values, causing wrong results for negative numbers in some lanes

### Strategy 2: Depth limit boundary condition

- **Target**: Test with trunc subtrees at exactly the depth limit boundary (Limit=2 or Limit=3)
- **Rationale**: The patch uses MaxDepthLevel and Limit to control how deep to traverse trunc subtrees. The condition 'MaxDepthLevel <= Limit' combined with complex nested conditions may have boundary issues where valid demotion opportunities are missed or invalid ones are accepted at the exact boundary.
- **Expected Issue**: Either missed optimization opportunities or incorrect truncation at depth boundary, potentially causing wrong code generation

### Strategy 3: CastMaxMinBWSizes initialization edge case

- **Target**: Test with trees containing only constants and casts (AllConstsOrCasts=true path)
- **Rationale**: When AllConstsOrCasts is true, CastMaxMinBWSizes is set to (max, 1). This initialization may interact incorrectly with the early exit condition 'CastMaxMinBWSizes->first / CastMaxMinBWSizes->second <= 2' causing premature exit from computeMinimumValueSizes for valid optimization cases.
- **Expected Issue**: Missed optimization opportunities when tree contains only constants and casts with ratio > 2

### Strategy 4: Reduction bitwidth mismatch with subtree minbitwidth

- **Target**: Test reduction operations where ReductionBitWidth differs from the minbitwidth computed for trunc subtrees
- **Rationale**: The patch adds cost for final resize if ReductionBitWidth != MinBWs.first. However, the interaction between ReductionBitWidth computation (from UserIgnoreList values) and per-subtree MinBWs may cause incorrect cost estimation or incorrect final cast type when both mechanisms are active.
- **Expected Issue**: Incorrect final vector type or missing cast instruction for reduction results, causing type mismatch or wrong values

### Strategy 5: TruncNodes iteration with modified VectorizableTree

- **Target**: Test with multiple trunc nodes where TruncNodes set is modified during iteration
- **Rationale**: The patch iterates through TruncNodes using 'TruncNodes.erase(TruncNodes.begin())' while also using NodeIdx to traverse VectorizableTree. If the tree structure changes or TruncNodes indices become stale during computeMinimumValueSizes, the iteration may skip valid subtrees or process invalid ones.
- **Expected Issue**: Missed optimization for some trunc subtrees or processing of incorrect tree entries, leading to suboptimal code or crashes
