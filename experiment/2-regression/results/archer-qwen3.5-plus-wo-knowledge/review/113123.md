## Test Strategies

### Strategy 1: Test with negative constant C

- **Target**: Create test where C is negative (e.g., and (fcmp olt x, -5.0), (fcmp ogt x, 5.0))
- **Rationale**: The patch handles C and -C as bitwise negations. When C is negative, -C becomes positive. The transformation fabs(x) < negative_C should always be false since fabs(x) >= 0. This edge case may not be explicitly tested.
- **Expected Issue**: The transformation might incorrectly produce a non-constant result instead of folding to false, or might produce incorrect comparison predicate.

### Strategy 2: Test with mixed ordered/unordered predicates

- **Target**: Create test with mixed predicate types (e.g., and (fcmp olt x, C), (fcmp ugt x, -C))
- **Rationale**: The condition FCmpInst::getSwappedPredicate(PredL) == PredR requires predicates to be exact swapped versions. Mixed ordered/unordered predicates (olt vs ugt) should NOT match since getSwappedPredicate(olt) = ogt, not ugt. This ensures the transformation doesn't incorrectly apply to invalid patterns.
- **Expected Issue**: The transformation might incorrectly apply to mixed predicate types, producing semantically incorrect results for NaN inputs.

### Strategy 3: Test fast math flag union edge cases

- **Target**: Test with conflicting fast math flags (e.g., LHS has nnan only, RHS has ninf only)
- **Rationale**: The code unions fast math flags: LHS->getFastMathFlags() | RHS->getFastMathFlags(). This could be problematic if flags have conflicting semantics. For example, if one comparison assumes no NaN and the other doesn't, the union might incorrectly assume no NaN for the transformed comparison.
- **Expected Issue**: The transformed fcmp might have overly aggressive fast math flags that change semantics for edge cases like NaN or infinity inputs.

### Strategy 4: Test vector types with varying undef patterns

- **Target**: Test vector types with different undef element patterns (e.g., <4 x float> with undef in different positions)
- **Rationale**: The test file only covers <2 x float> with specific undef patterns. Larger vectors or different undef patterns might expose issues with the m_APFloatAllowUndef matcher or the constant propagation logic.
- **Expected Issue**: The transformation might incorrectly handle vectors with undef elements in certain positions, producing incorrect constant vectors or failing to transform when it should.

### Strategy 5: Test or-pattern with reversed operand order

- **Target**: Test or (fcmp olt x, C), (fcmp ogt x, -C) where the less-than predicate comes first
- **Rationale**: The code has logic to swap predicates based on IsLessThanOrLessEqual. For the or-pattern with reversed order, the swap logic should correctly identify and swap to get the greater-than predicate. This tests the swap logic branch that might be less exercised.
- **Expected Issue**: The swap logic might incorrectly handle the or-pattern with reversed order, producing fabs(x) with wrong predicate (e.g., olt instead of ogt).
