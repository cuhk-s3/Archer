## Test Strategies

### Strategy 1: Vector type with mixed zero inputs

- **Target**: Test ctlz/cttz on vector types where some elements are zero and others are non-zero, used as shift amounts
- **Rationale**: The optimization changes is_zero_poison from false to true. For vectors, if some elements are zero, ctlz/cttz would return the bit width for those elements. When used as shift amounts, this should produce poison for those elements. Need to verify the optimization handles vector semantics correctly.
- **Expected Issue**: Incorrect code generation for vector types where the optimization incorrectly assumes all elements behave the same, potentially causing wrong results for non-zero elements.

### Strategy 2: Shift with nuw/nsw/exact flags

- **Target**: Test ctlz/cttz used as shift amount in shifts with nuw, nsw, or exact flags
- **Rationale**: The patch doesn't consider shift flags. While the optimization should be safe (shift by >= bit width produces poison regardless of flags), we should verify that the combination of flags doesn't introduce unexpected behavior.
- **Expected Issue**: The optimization might interact unexpectedly with shift flags, potentially causing different poison propagation or missed optimization opportunities.

### Strategy 3: Multiple uses but all are shift amounts

- **Target**: Test ctlz/cttz with multiple uses where all uses are as shift amounts
- **Rationale**: The optimization only triggers when hasOneUse() is true. If there are multiple uses but all are shift amounts, the optimization doesn't apply. This might be a missed optimization opportunity, or there could be a reason for this restriction.
- **Expected Issue**: Missed optimization: the code could be more aggressive and apply the optimization even with multiple uses if all are shift amounts.

### Strategy 4: Shift amount used in other contexts after the shift

- **Target**: Test cases where the shift result (not the ctlz/cttz) is used elsewhere, potentially affecting the semantics
- **Rationale**: While the patch checks that ctlz/cttz has one use (the shift), the shift result might be used in ways that could expose differences in poison propagation.
- **Expected Issue**: If the shift result's poison value propagates differently based on whether the shift amount was poison vs. a valid large number, this could cause observable differences.

### Strategy 5: AShr (arithmetic shift right) with negative values

- **Target**: Test ctlz/cttz used as shift amount in ashr with negative input values
- **Rationale**: AShr has different semantics from LShr for negative values (sign extension). While the shift amount >= bit width still produces poison, we should verify the optimization doesn't break anything specific to ashr.
- **Expected Issue**: Potential incorrect handling of ashr with negative values when the shift amount comes from ctlz/cttz with zero input.

### Analysis

### Bug Triggering Analysis
The patch adds an optimization that sets `is_zero_poison` to `true` for `ctlz`/`cttz` intrinsics when they have exactly one use and that use is a shift instruction where the ctlz/cttz result is used as the shift amount.

The key insight is that when the input to ctlz/cttz is zero:
- With `is_zero_poison=false`: returns the bit width (e.g., 32 for i32)
- With `is_zero_poison=true`: returns poison

When used as a shift amount, shifting by >= bit width produces poison anyway, so both paths lead to the same result (poison). This transformation is semantically correct.

### Fix Weakness Analysis
After comprehensive testing of all 6 test cases with all 5 strategies:
1. **AShr with negative values**: Verified correct - both original and optimized produce poison
2. **Shift with nuw/nsw/exact flags**: Verified correct - flags don't affect the poison propagation
3. **Vector type with mixed zero inputs**: Verified correct - per-element semantics preserved
4. **Multiple uses but all are shift amounts**: Verified correct - optimization correctly does NOT apply when hasOneUse() is false
5. **Shift amount used in other contexts after the shift**: Verified correct - poison propagates correctly through shift result

All tests passed verification with Alive2 and differential testing with lli. The optimization is correct and no bugs were found.

### Summary
The fix correctly implements the optimization to set `zero_is_poison` for ctlz/cttz when they are only used as shift amounts. The transformation is semantically equivalent because:
1. For zero input: both paths produce poison (either directly from ctlz/cttz or from the shift)
2. For non-zero input: the `is_zero_poison` flag doesn't affect the result

The optimization is safe and improves codegen for targets that don't provide defined values for ctlz/cttz with zero input.
