## Test Strategies

### Strategy 1: Test shuffle with element count change

- **Target**: shuffle-of-cmps.ll
- **Rationale**: The patch creates ShuffleCmpTy using FixedVectorType::get(BinOpTy->getElementType(), ShuffleDstTy). If the shuffle changes the number of elements (e.g., from <4 x i1> to <8 x i1>), we need to verify the type construction and mask conversion work correctly for the new compare.
- **Expected Issue**: Incorrect type construction for ShuffleCmpTy when shuffle result has different element count than compare inputs, potentially causing type mismatch in the new compare instruction.

### Strategy 2: Test non-commutative icmp predicates with swapped common operand

- **Target**: shuffle-of-cmps.ll
- **Rationale**: For non-commutative predicates (sgt, ult, sge, ule, etc.), IsCommutative is false, so the operand swap at line 1668-1669 won't happen. Test cases where the common operand is in different positions (e.g., icmp sgt x,y and icmp sgt z,x) to verify the transformation handles this correctly without swapping.
- **Expected Issue**: The transformation might incorrectly handle cases where the common operand is in different positions for non-commutative predicates, potentially producing semantically incorrect results.

### Strategy 3: Test non-commutative fcmp predicates

- **Target**: shuffle-of-cmps.ll
- **Rationale**: FCmpInst::isCommutative() returns true only for equality predicates (OEQ, ONE, UEQ, UNE) and special predicates (FALSE, TRUE, ORD, UNO). Non-equality predicates like OGT, OLT, OGE, OLE are non-commutative. Test these predicates to ensure correct handling.
- **Expected Issue**: The transformation might incorrectly apply commutative optimization to non-commutative fcmp predicates, producing wrong comparison results.

### Strategy 4: Test shuffle mask with poison elements for compares

- **Target**: shuffle-of-cmps.ll
- **Rationale**: The original code has a check to prevent poison introduction for div/rem binops. For compares, there's no such check. Test with poison mask elements to verify the transformation is safe for compares (compares shouldn't have poison issues like div/rem).
- **Expected Issue**: Potential undefined behavior if poison mask elements interact incorrectly with compare operands, though compares are generally safer than div/rem.

### Strategy 5: Test FCmp with fast-math flags intersection

- **Target**: shuffle-of-cmps.ll
- **Rationale**: The code intersects IR flags from LHS and RHS at lines 1727-1729. For FCmp, this includes fast-math flags. Test with different fast-math flags on LHS and RHS to verify correct flag intersection.
- **Expected Issue**: Incorrect fast-math flag intersection could change the semantics of floating-point comparisons, especially for NaN handling or ordering guarantees.
