## Test Strategies

### Strategy 1: Test operand deletion with shared operands

- **Target**: Instructions that share operands with other live instructions
- **Rationale**: The removeInstructionsAndOperands function marks operands for deletion if they have one user and would be trivially dead. However, if an operand is shared between a deleted instruction and a live instruction (outside the vectorization tree), the check for hasOneUser() should prevent deletion. But there might be edge cases where the operand is used by another instruction that is also being deleted in the same batch, causing incorrect deletion order or use-after-free issues.
- **Expected Issue**: Use-after-free or incorrect deletion of operands that are still needed by live instructions, potentially causing crashes or incorrect code generation.

### Strategy 2: Test MultiNodeScalars entries during operand deletion

- **Target**: Instructions that appear in MultiNodeScalars map
- **Rationale**: The removeInstructionsAndOperands function checks if an operand's VectorizedValue matches any Entry in the MultiNodeScalars map to avoid deleting it. However, the logic checks Entries.empty() || none_of(Entries, ...). If Entries is empty (instruction not in tree), the condition passes and the operand could be deleted even if it's a VectorizedValue in another node. This could lead to deleting vectorized values that are still needed.
- **Expected Issue**: Vectorized values that are shared across multiple tree entries could be incorrectly deleted, causing use-after-free or incorrect code generation.

### Strategy 3: Test reduction with UserIgnoreList and operand deletion

- **Target**: Reduction operations with UserIgnoreList
- **Rationale**: The patch adds code to replace uses with PoisonValue for instructions in UserIgnoreList before calling removeInstructionsAndOperands. However, the removeInstructionsAndOperands function doesn't explicitly check UserIgnoreList when determining if an operand should be deleted. If an operand is in UserIgnoreList but gets marked for deletion because it appears to have one user (the instruction being deleted), it could be incorrectly removed.
- **Expected Issue**: Reduction operands that should be preserved (in UserIgnoreList) could be incorrectly deleted, breaking the reduction semantics.

### Strategy 4: Test deleted instruction check in store vectorization

- **Target**: Store instructions in vectorizeStores
- **Rationale**: The patch adds R.isDeleted(SI) check at the beginning of the store loop. However, PrevValTy is only set when encountering a non-deleted store. If all stores in a sequence are deleted before processing, PrevValTy remains nullptr, which could cause issues in subsequent iterations or when the loop continues with new stores.
- **Expected Issue**: Incorrect type comparison or null pointer dereference when processing stores after all previous stores have been deleted.

### Strategy 5: Test instruction re-insertion in destructor

- **Target**: Instructions without parent in BoUpSLP destructor
- **Rationale**: The destructor has special handling for instructions without a parent - it re-inserts them before the terminator. However, the new removeInstructionsAndOperands function calls removeFromParent() which removes the instruction from its parent. If an instruction is removed but then the destructor tries to re-insert it, there could be issues with the instruction's state (e.g., operands already dropped, debug info already salvaged). The destructor also doesn't check if the instruction was already processed by removeInstructionsAndOperands.
- **Expected Issue**: Double-processing of instructions in the destructor, potentially causing crashes or incorrect debug info handling when instructions that were already processed by removeInstructionsAndOperands are re-inserted and processed again.
