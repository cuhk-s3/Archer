## Test Strategies

### Strategy 1: Test with empty nofpclass attribute (fcNone)

- **Target**: Function with nofpclass attribute that is fcNone (no restrictions)
- **Rationale**: The code checks 'if (ReturnClass == fcNone) return nullptr;' but this check happens AFTER the type compatibility check. If a function has no nofpclass attribute, getRetNoFPClass() returns fcNone, and the optimization should exit early. However, we should test edge cases where the attribute exists but is effectively empty.
- **Expected Issue**: Potential incorrect optimization if fcNone check is bypassed or if the attribute is present but empty

### Strategy 2: Test with vector FP types with nofpclass

- **Target**: Functions returning vector FP types with nofpclass attributes
- **Rationale**: The isNoFPClassCompatibleType function handles vector types, but SimplifyDemandedUseFPClass may have different behavior for vectors vs scalars. The test file shows some vector tests but we should verify edge cases with mixed vector elements.
- **Expected Issue**: Incorrect simplification for vector types where some lanes satisfy the nofpclass constraint but others don't

### Strategy 3: Test with array of FP types with nofpclass

- **Target**: Functions returning array types containing FP values with nofpclass attributes
- **Rationale**: The isNoFPClassCompatibleType function recursively unwraps ArrayType to check the element type. However, SimplifyDemandedUseFPClass may not properly handle array types, as it primarily handles instructions, constants, select, call, and fneg.
- **Expected Issue**: Incorrect handling of array return types where the nofpclass attribute applies to array elements

### Strategy 4: Test with one-use requirement violation

- **Target**: Return values that have multiple uses before the return instruction
- **Rationale**: SimplifyDemandedUseFPClass has a check 'if (!I->hasOneUse()) return nullptr;' for instructions. If the return value is an instruction with multiple uses (not just the return), the optimization should not apply. We need to test cases where this assumption could be violated or edge cases around this check.
- **Expected Issue**: Incorrect optimization when the return value instruction has exactly one use but that use is not the return instruction

### Strategy 5: Test with fcAllFlags nofpclass (all classes forbidden)

- **Target**: Function with nofpclass(all) attribute meaning no FP value is valid
- **Rationale**: If nofpclass(all) is set, then ~ReturnClass would be fcNone, and SimplifyDemandedUseFPClass would return PoisonValue. The test file shows this case (ret_nofpclass_all_var), but we should test more complex scenarios where the return value computation involves multiple operations.
- **Expected Issue**: Incorrect poison propagation or missed optimization when all FP classes are forbidden

### Strategy 6: Test with recursive phi nodes and nofpclass

- **Target**: Functions with recursive phi nodes in loops returning values with nofpclass constraints
- **Rationale**: The SimplifyDemandedUseFPClass function has recursion depth limits (MaxAnalysisRecursionDepth). Complex phi node patterns with nofpclass attributes could hit these limits or cause incorrect analysis.
- **Expected Issue**: Incorrect simplification or infinite loop when analyzing recursive phi nodes with nofpclass constraints

### Strategy 7: Test with copysign intrinsic and sign-specific nofpclass

- **Target**: Return values involving copysign with nofpclass attributes that restrict sign
- **Rationale**: The SimplifyDemandedUseFPClass has special handling for copysign intrinsic. The code checks for fcPositive and fcNegative specifically. Edge cases with sign-specific nofpclass attributes (like nofpclass(nzero) or nofpclass(pzero)) could expose issues.
- **Expected Issue**: Incorrect copysign transformation when sign-specific nofpclass constraints are present
