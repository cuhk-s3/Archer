## Test Strategies

### Strategy 1: LShr with large shift amount

- **Target**: lshr(zext(x), shift_amt) where shift_amt has values >= small_type_bit_width
- **Rationale**: The shrinkType transformation truncates the shift amount operand. If the shift amount is >= the bit width of the small type (e.g., 8 for i8), truncating it will wrap around, changing the semantics. For example, lshr(zext(x), 256) should produce poison, but after transformation to zext(lshr(x, trunc(256))) = zext(lshr(x, 0)), it returns x instead.
- **Expected Issue**: Incorrect code generation: shift amounts that should produce poison or different results are incorrectly transformed, leading to wrong runtime behavior

### Strategy 2: LShr with variable shift amount that can exceed small type width

- **Target**: lshr(zext(x), %shift) where %shift is a variable that can take values >= small_type_bit_width at runtime
- **Rationale**: Even if KnownBits analysis shows the result fits in BW bits, the shift amount itself might have values that when truncated produce incorrect behavior. The KnownBits check on the result doesn't constrain the shift amount values.
- **Expected Issue**: Silent semantic change: the transformation is applied when it shouldn't be, causing incorrect results for certain runtime values of the shift amount

### Strategy 3: ZExt source type is not FixedVectorType

- **Target**: Pattern where I.getType() is FixedVectorType but ZExted->getType() is not a FixedVectorType
- **Rationale**: The code casts ZExted->getType() to FixedVectorType at line 2513 without verification. While the function is only called when I.getType() is FixedVectorType, there's no guarantee the ZExt source type is also a FixedVectorType. This could cause a crash or undefined behavior.
- **Expected Issue**: Potential crash or incorrect behavior due to invalid cast when ZExt source type is not FixedVectorType

### Strategy 4: OtherOperand with high bits set beyond small type width

- **Target**: binop(zext(x), y) where y has known high bits set that would be lost in truncation
- **Rationale**: The KnownBits check at line 2518-2520 only verifies the result of the instruction fits in BW bits. It doesn't check if OtherOperand itself can be safely truncated. For operations like OR or XOR, y might have high bits that are masked out by the operation but would be lost in truncation, potentially affecting correctness if the operation semantics change.
- **Expected Issue**: Incorrect transformation when OtherOperand has significant high bits that are masked in the original operation but lost in truncation

### Strategy 5: zext nneg not matched

- **Target**: binop(zext nneg(x), y) patterns
- **Rationale**: The pattern matcher m_ZExt only matches regular zext instructions, not zext nneg. This means valid optimization opportunities with nneg zext are missed. The transformation would be equally valid and profitable for nneg zext since nneg guarantees non-negative values.
- **Expected Issue**: Missed optimization: valid and profitable transformations are not performed for zext nneg patterns

### Analysis

### Bug Triggering Analysis

The bug is triggered by the `shrinkType` transformation in VectorCombine when applied to `lshr(zext(x), shift_amt)` patterns where the shift amount, when truncated to the smaller type, becomes >= the bit width of the smaller type.

**Concrete example:**
- Original: `lshr(zext(<16 x i8> %x), <i32 8, ...>)`
  - zext converts i8 to i32, so we have `<16 x i32>`
  - lshr by 8 on i32 is valid (8 < 32), produces result
- Transformed: `zext(lshr(%x, trunc(<i32 8>)))` = `zext(lshr(%x, <i8 8>))`
  - lshr by 8 on i8 is poison (8 >= 8, the bit width of i8)
  - Result is poison, not the same as original

**Alive2 verification confirms:**
```
ERROR: Target is more poisonous than source
Source value: <0, 0, ..., 0> (valid result)
Target value: <poison, poison, ..., poison>
```

### Fix Weakness Analysis

The fix has a critical weakness: it doesn't validate that the shift amount operand can be safely truncated for LShr operations. The code treats LShr the same as bitwise logic operations (and, or, xor), but LShr has asymmetric operand semantics:
- First operand: data value (can be truncated if result fits)
- Second operand: shift count (has special semantics - shift >= bit width produces poison)

The fix only checks:
1. KnownBits of the result fits in small type bit width
2. Cost model profitability

But it fails to check:
1. Whether the shift amount, when truncated, would be >= the small type's bit width
2. Whether truncating the shift amount changes the poison semantics

**The fix should:**
- Either exclude LShr from this transformation entirely
- Or add a check that `trunc(shift_amt) < SmallTy_bit_width` for all elements
- Or verify that the shift amount is a constant < SmallTy_bit_width before transforming
