## Test Strategies

### Strategy 1: Test ORD predicate with potential infinity operands

- **Target**: fcmp ord (x - y), 0
- **Rationale**: The FCMP_ORD predicate is missing from both predicate groups in the switch statement. It falls into the default case and the transformation never occurs. However, fcmp ord (inf - inf), 0 returns false while fcmp ord inf, inf returns true, so this predicate needs the infinity check like the first group.
- **Expected Issue**: The optimization is not applied for FCMP_ORD predicate even when it would be safe (when infinities are ruled out), representing a missed optimization opportunity.

### Strategy 2: Test UNO predicate with potential infinity operands

- **Target**: fcmp uno (x - y), 0
- **Rationale**: The FCMP_UNO predicate is missing from both predicate groups in the switch statement. It falls into the default case and the transformation never occurs. However, fcmp uno (inf - inf), 0 returns true while fcmp uno inf, inf returns false, so this predicate needs the infinity check like the first group.
- **Expected Issue**: The optimization is not applied for FCMP_UNO predicate even when it would be safe (when infinities are ruled out), representing a missed optimization opportunity.

### Strategy 3: Test FALSE predicate transformation

- **Target**: fcmp false (x - y), 0
- **Rationale**: The FCMP_FALSE predicate falls into the default case which breaks without attempting the transformation. Since fcmp false always returns false regardless of operands, this transformation should always be safe without any infinity checks.
- **Expected Issue**: Missed optimization: fcmp false (x - y), 0 is not transformed to fcmp false x, y even though it's always semantically equivalent.

### Strategy 4: Test TRUE predicate transformation

- **Target**: fcmp true (x - y), 0
- **Rationale**: The FCMP_TRUE predicate falls into the default case which breaks without attempting the transformation. Since fcmp true always returns true regardless of operands, this transformation should always be safe without any infinity checks.
- **Expected Issue**: Missed optimization: fcmp true (x - y), 0 is not transformed to fcmp true x, y even though it's always semantically equivalent.

### Strategy 5: Test denormal mode edge case with non-IEEE modes

- **Target**: fcmp with non-IEEE denormal mode
- **Rationale**: The code checks for DenormalMode::getIEEE() before applying the transformation. Need to verify that the transformation is correctly blocked for positive-zero, preserve-sign, and dynamic denormal modes, as these could affect the semantics of comparing with zero.
- **Expected Issue**: Potential incorrect transformation if denormal mode check doesn't properly handle all non-IEEE cases, or missed optimization if the check is too conservative.
