## Test Strategies

### Strategy 1: Test vector constant sign bit inference with mixed signs

- **Target**: computeKnownFPClass for vector constants
- **Rationale**: The patch adds new logic for tracking sign bits in vector constants (lines 4317-4344). The condition `if (SignBitAllOne != SignBitAllZero)` sets SignBit only when all elements have the same sign. However, if the vector contains a mix of positive and negative values, SignBit should remain nullopt. Test with vectors containing mixed sign constants to verify the logic handles all combinations correctly.
- **Expected Issue**: Incorrect sign bit inference for vectors with mixed signs could lead to wrong copysign optimizations or incorrect fabs simplifications

### Strategy 2: Test FMul sign bit inference with NaN operands

- **Target**: FMul sign bit inference in computeKnownFPClass (lines 4967-4972)
- **Rationale**: The new FMul sign bit inference at lines 4967-4972 checks `if (KnownLHS.SignBit && KnownRHS.SignBit)` before inferring the result sign. However, this code is only reached if both operands are known never NaN (lines 4959-4965). The sign inference logic assumes that same signs produce positive result and different signs produce negative. Test with cases where operands could be NaN to ensure the NaN check is correctly ordered.
- **Expected Issue**: If NaN handling is incorrect, FMul could incorrectly infer sign bits when one operand could be NaN, leading to wrong copysign or fabs optimizations

### Strategy 3: Test minnum/maxnum sign bit inference with -0.0/+0.0 edge cases

- **Target**: minnum/maxnum sign bit inference (lines 4598-4617)
- **Rationale**: The patch adds complex sign bit inference for minnum/maxnum/minimum/maximum intrinsics. The logic at lines 4605-4616 handles the case where signs differ or zeros are involved. The IEEE 754 standard has special rules for -0.0 vs +0.0 in min/max operations. Test with combinations of -0.0, +0.0, and known-sign values to verify the sign inference is correct.
- **Expected Issue**: Incorrect sign inference for minnum/maxnum with zero operands could cause copysign to be incorrectly optimized to fabs or fneg(fabs)

### Strategy 4: Test copysign optimization with known negative sign argument

- **Target**: InstCombine copysign optimization (lines 2414-2428)
- **Rationale**: The patch enables a new optimization path when the sign argument's sign bit is known to be 1 (negative). This transforms `copysign Mag, -Sign` into `fneg (fabs Mag)`. Test cases where the sign argument is known negative through various means (constants, UIToFP, fabs results, etc.) to ensure the optimization is correctly applied and the result has correct fast math flags.
- **Expected Issue**: The new optimization path might incorrectly apply when the sign bit knowledge comes from incomplete analysis, or fast math flags might not be properly propagated

### Strategy 5: Test knownNot() sign bit inference interaction with NaN

- **Target**: KnownFPClass::knownNot() method (lines 359-367)
- **Rationale**: The knownNot() method now infers SignBit when `isKnownNever(fcNan) && !SignBit` is true. It sets SignBit to false if all negative classes are ruled out, or true if all positive classes are ruled out. This logic assumes that if NaN is ruled out and only one sign category remains, the sign bit is known. Test scenarios where knownNot is called with various masks to ensure the SignBit inference doesn't conflict with existing SignBit values or produce incorrect results.
- **Expected Issue**: The knownNot() method might incorrectly set SignBit when called in certain sequences, potentially causing downstream optimizations to make incorrect assumptions
