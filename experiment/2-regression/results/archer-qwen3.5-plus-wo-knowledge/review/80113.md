## Test Strategies

### Strategy 1: Constant in Operand 0

- **Target**: Binary operator with constant mask in getOperand(0) instead of getOperand(1)
- **Rationale**: The fix only checks BO->getOperand(1) for a constant mask using match(BO->getOperand(1), m_APInt(Mask)). Since and/or/xor are commutative, the constant could be in either operand. If the constant is in getOperand(0), the current code won't match and won't apply the optimization, potentially missing valid simplifications.
- **Expected Issue**: Binary operations with constant masks in getOperand(0) will not be simplified even when the demanded bits analysis shows the mask doesn't affect demanded bits, leading to missed optimization opportunities.

### Strategy 2: Vector Non-Splat Constants

- **Target**: Binary operator with non-splat vector constant mask
- **Rationale**: The m_APInt matcher handles scalar constants and splat vector constants. For non-splat vector constants (where each element has a different value), the match will fail. The DemandedBits analysis for vectors returns demanded bits for the scalar type, so there could be cases where a non-splat vector mask could be simplified but isn't handled.
- **Expected Issue**: Binary operations with non-splat vector constant masks won't be simplified even when element-wise analysis would show the mask doesn't affect demanded bits for any element.

### Strategy 3: Partial Overlap Edge Case for And

- **Target**: And operation where Demanded.isSubsetOf(Mask) is on the boundary
- **Rationale**: For 'and' operations, the condition is Demanded.isSubsetOf(*Mask). This means all demanded bits must be within the mask. Edge cases where demanded bits exactly match mask bits, or where they're adjacent but not overlapping, need careful testing. If the implementation has off-by-one errors in bit position calculations, these cases could trigger incorrect simplifications.
- **Expected Issue**: Incorrect simplification when demanded bits are exactly at the boundary of the mask, potentially removing bits that are actually demanded or failing to remove bits that aren't demanded.

### Strategy 4: Disjoint Or with Overlapping Demanded Bits

- **Target**: Or disjoint operation where mask bits partially overlap with demanded bits
- **Rationale**: For 'or' (including 'or disjoint'), the condition is !Demanded.intersects(*Mask). This should prevent simplification when any demanded bit overlaps with the mask. However, the 'disjoint' flag adds poison semantics - if both inputs have 1 in the same position, the result is poison. The fix doesn't distinguish between regular 'or' and 'or disjoint', which could lead to different poison behavior after simplification.
- **Expected Issue**: Simplifying 'or disjoint' to just the first operand could change poison semantics. If the original 'or disjoint' would produce poison for certain inputs (when both operands have 1 in same position), but the simplified version (just operand 0) wouldn't, this changes program behavior for poison-sensitive code.

### Strategy 5: Xor with All-Ones Mask (Not)

- **Target**: Xor operation with all-ones mask (bitwise not operation)
- **Rationale**: Xor with all-ones is equivalent to bitwise not. If the demanded bits don't intersect with the all-ones mask (which is only possible if Demanded is zero, meaning no bits are demanded), the operation could be simplified. However, this is an edge case where the instruction should probably be eliminated entirely rather than simplified to operand 0. The current code would replace xor %x, -1 with %x when no bits are demanded, which is correct, but testing this boundary condition is important.
- **Expected Issue**: Edge case where Demanded.isZero() could lead to unexpected simplification behavior, or the instruction should be marked for complete removal rather than simplification to operand 0.

## Bugs Found

### Bug #1

**Analysis:**
This is a REAL BUG! The transformation from 'or disjoint' to 'or' changes the poison semantics.

In the original code:
- %and1 = 8 & 56 = 8
- %and2 = 8 & 25 = 8
- %or = or disjoint 8, 8 produces POISON because both operands have bit 3 set (the 'disjoint' flag requires that no bit position has 1 in both operands)
- The function returns poison, which is UB

In the optimized code:
- %or = or 8, 8 = 8 (regular or, no poison check)
- %ret1 = 8 & 8 = 8
- %ret2 = 8 & 16 = 0
- result = 8

The original produces poison (UB), while the optimized produces 8. This is a semantic difference!

The fix incorrectly simplifies 'or disjoint' when it should preserve the poison semantics. The condition `!Demanded.intersects(*Mask)` checks if the mask bits don't affect demanded bits, but it doesn't account for the fact that 'or disjoint' has different poison semantics than regular 'or'.

When we have `or disjoint %and1, %and2` where both %and1 and %and2 could have overlapping bits set, the 'disjoint' flag adds a poison check. Simplifying this to just one operand removes this poison check, which changes program behavior.

The fix should either:
1. Not simplify 'or disjoint' at all, or
2. Only simplify 'or disjoint' when it can prove the operands are truly disjoint (no overlapping 1 bits possible)
