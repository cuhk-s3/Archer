### Strategy 1: Test loop with exactly 2 iterations where last iteration peeling should trigger

- **Target**: Create a loop that executes exactly 2 iterations with a condition that becomes known only at the last iteration (e.g., i == N-1 where N=2)
- **Rationale**: The canPeelLastIteration check requires BTC > 0 (at least 2 iterations). Testing with exactly 2 iterations exercises the boundary condition where peeling the last iteration should be possible and beneficial.
- **Expected Issue**: If the BTC check or shouldPeelLastIteration logic is incorrect, the optimizer might either fail to peel when it should, or incorrectly peel causing wrong code generation.

### Strategy 2: Test loop with condition that is known false at last iteration but true at second-to-last

- **Target**: Create a loop with a condition like 'i < N-1' where at the last iteration i == N-1 makes the condition false, and at second-to-last i == N-2 makes it true
- **Rationale**: The shouldPeelLastIteration function checks that the inverse predicate is known at the last iteration and the original predicate is known at second-to-last. This tests the core logic of when last iteration peeling is beneficial.
- **Expected Issue**: If the predicate inversion logic in shouldPeelLastIteration is incorrect (especially considering Pred may have been inverted at line 473), the wrong decision could be made about peeling.

### Strategy 3: Test loop with multiple exit blocks when PeelLast is true

- **Target**: Create a loop with multiple exit paths (not just through the latch) and a condition that would benefit from last iteration peeling
- **Rationale**: The peelLoop function when PeelLast=true captures exit values from PHIs in the exit block (line 1080-1081). With multiple exits, this logic may not correctly handle all exit values, potentially causing LCSSA form violations or incorrect value propagation.
- **Expected Issue**: The code assumes a single exit block (L->getExitBlock()) but loops with multiple exits could cause incorrect exit value handling or crashes.

### Strategy 4: Test loop where DesiredPeelCountLast could exceed 1 due to line 498

- **Target**: Create a loop where countToEliminateCompares would set DesiredPeelCount to a value > 1 through the normal front-peeling path, which then gets propagated to DesiredPeelCountLast at line 498
- **Rationale**: Line 498 sets DesiredPeelCountLast = max(DesiredPeelCountLast, NewPeelCount) even when peeling from the front. This could incorrectly set DesiredPeelCountLast to values > 1, but the assertion at line 1012-1014 requires PeelCount == 1 for PeelLast.
- **Expected Issue**: The assertion 'PeelCount == 1' could fire if DesiredPeelCountLast > 1 is passed to peelLoop, or the code might incorrectly try to peel multiple iterations from the end.

### Strategy 5: Test loop with step != 1 induction variable

- **Target**: Create a loop where the induction variable has a step of 2 or more, with a condition that would appear to benefit from last iteration peeling
- **Rationale**: The canPeelLastIteration function checks that the induction step is 1 (line 355). The exit condition adjustment code (line 1180-1182) subtracts 1 from the bound, which assumes step=1. Testing with step != 1 verifies this check is working.
- **Expected Issue**: If the step=1 check fails or is bypassed, subtracting 1 from the bound with step != 1 would cause incorrect loop behavior (wrong number of iterations).

## Bug Analysis

### Bug Triggering Analysis
The bug is triggered when a loop has multiple exit blocks and the optimizer incorrectly attempts to peel the last iteration. The test case has:
1. A loop with two exit blocks: `early_exit` and `exit`
2. A condition `i == 63` that would normally benefit from last iteration peeling
3. An early exit condition based on a runtime value `%n`

When the loop-unroll pass runs with `-unroll-full-max-count=0`, it incorrectly decides to peel the last iteration even though the loop has multiple exits. This causes a crash with the error:
```
PHI node entries do not match predecessors!
  %iv = phi i64 [ 0, %entry ], [ %iv.next, %loop ]
label %loop
label %latch
Instruction does not dominate all uses!
```

### Fix Weakness Analysis
The root cause is in the `canPeelLastIteration()` function in `llvm/lib/Transforms/Utils/LoopPeel.cpp`. The function checks:
```cpp
BasicBlock *Latch = L.getLoopLatch();
return Latch && Latch == L.getExitingBlock() && ...
```

The check `Latch == L.getExitingBlock()` is supposed to ensure the latch is the only exiting block. However, this check is insufficient because:
1. `getExitingBlock()` returns a single block even when there are multiple exiting blocks
2. The code doesn't verify that there's only ONE exit block

When peeling the last iteration with multiple exits, the code at line 1079 uses `L->getExitBlock()` which assumes a single exit block. With multiple exits, this causes:
1. Incorrect handling of PHI nodes in exit blocks
2. Broken dominator relationships
3. Invalid transformed IR

The fix should add an explicit check in `canPeelLastIteration()` to verify that the loop has exactly one exit block before allowing last iteration peeling.
