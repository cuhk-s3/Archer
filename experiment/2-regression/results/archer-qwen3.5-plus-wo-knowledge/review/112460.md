## Test Strategies

### Strategy 1: Test freeze with mixed poison/undef operands

- **Target**: Create test with freeze instructions where some scalar lanes have poison values and others have undef values
- **Rationale**: The freeze instruction has special semantics for poison/undef - it returns an arbitrary but fixed value. When vectorizing, we need to ensure the element-wise freeze semantics are preserved correctly. The patch may not correctly handle cases where different lanes have different types of undefined values.
- **Expected Issue**: Incorrect vectorization where poison/undef semantics are not preserved element-wise, potentially causing different frozen values to be observed for the same freeze instruction

### Strategy 2: Test freeze with diamond merge pattern

- **Target**: Create test where freeze instruction result is used multiple times (diamond pattern)
- **Rationale**: The vectorizeTree code for freeze checks if E->VectorizedValue is already set (diamond merge detection). This pattern tests whether the diamond merge handling for freeze instructions correctly preserves the semantics that all uses of the same freeze observe the same value.
- **Expected Issue**: Multiple vectorized freeze instructions generated instead of reusing the same vectorized value, breaking the semantic guarantee that all uses observe the same frozen value

### Strategy 3: Test freeze in alternating opcode patterns (add-freeze-sub-freeze)

- **Target**: Create test with alternating patterns involving freeze and arithmetic operations like add/sub
- **Rationale**: The existing addsub_freeze test shows freeze at the end of add/sub patterns. Testing freeze interleaved with alternating opcodes could expose issues in how the SLP tree building handles freeze within the alternating pattern detection logic.
- **Expected Issue**: SLP fails to recognize valid vectorization opportunities or incorrectly vectorizes patterns where freeze breaks the alternating opcode sequence

### Strategy 4: Test freeze with vector type operands

- **Target**: Create test where freeze operates on vector values that are then extracted
- **Rationale**: The freeze instruction operates element-wise on vectors. Testing cases where freeze is applied to vectors and then elements are extracted could expose issues in how the vectorization handles nested vector operations with freeze.
- **Expected Issue**: Incorrect handling of freeze on vector types, potentially generating redundant freeze operations or failing to vectorize when it should

### Strategy 5: Test freeze with external uses across basic blocks

- **Target**: Create test where freeze result has uses in different basic blocks
- **Rationale**: The SLP vectorizer needs to track external uses correctly. Freeze instructions with uses across basic blocks could expose issues in how the vectorization state and external use tracking handles the unique semantics of freeze (same value guarantee).
- **Expected Issue**: Incorrect code generation where freeze semantics are violated due to improper handling of cross-block uses, or missed vectorization opportunities
