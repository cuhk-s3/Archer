## Test Strategies

### Strategy 1: Select with PHI in true/false value - context instruction not updated

- **Target**: Create a test where a PHI node has an incoming value that is a select instruction with the PHI as one operand. The other operand of the select should come from a different basic block than the PHI's incoming edge.
- **Rationale**: The breakSelfRecursivePHI function updates ValOut to the other select operand but does NOT update CtxIOut. This means the context instruction remains pointing to the terminator of the PHI's incoming block, not where the select's other operand is evaluated. This could lead to incorrect known bits analysis if the context matters for the analysis.
- **Expected Issue**: The known bits analysis may use an incorrect context instruction, potentially missing optimization opportunities or computing incorrect known bits due to wrong branch condition analysis.

### Strategy 2: Nested select patterns (select of select with PHI)

- **Target**: Create a test with nested select patterns where a PHI is used in an outer select, which is itself used in another select that feeds the PHI.
- **Rationale**: The breakSelfRecursivePHI function only handles one level of select recursion. If there's a pattern like `select(cond1, PHI, select(cond2, PHI, X))`, the function will only unwrap one level, leaving the inner select with PHI reference, potentially causing incomplete recursion breaking.
- **Expected Issue**: The analysis may not fully break the recursion, leading to either infinite recursion (if depth limits aren't hit) or incomplete known bits computation.

### Strategy 3: PHI with more than 2 incoming values referencing another PHI

- **Target**: Create a test where a PHI node has 3+ incoming values, and one of them is another PHI that references the original PHI. The other PHI should have more than 2 unique incoming values.
- **Rationale**: The breakSelfRecursivePHI function explicitly checks `IncPhi->getNumIncomingValues() == 2`. If the incoming PHI has more than 2 values, the recursion breaking won't happen, even if there are only 2 unique values (as noted in the TODO comment).
- **Expected Issue**: The recursion won't be broken for PHIs with more than 2 incoming edges, potentially missing optimization opportunities or causing the analysis to give up prematurely.

### Strategy 4: Select with PHI where both true and false values reference PHI indirectly

- **Target**: Create a test where a select has the PHI in one arm, and the other arm is an instruction that also eventually references the PHI (e.g., through another select or PHI).
- **Rationale**: The current code only checks if one of the select operands directly matches the PHI using m_Specific. If the other operand also transitively depends on the PHI, the recursion breaking may not be sufficient, and the analysis could still spin.
- **Expected Issue**: The analysis may not correctly identify the recursive dependency, leading to either incorrect known bits or excessive recursion.

### Strategy 5: Min/Max intrinsic with PHI recursion (unhandled case)

- **Target**: Create a test where a PHI node has an incoming value that is a umax/smax/umin/smin intrinsic with the PHI as one operand.
- **Rationale**: The code has a TODO comment indicating Min/Max handling is not implemented. Similar to select, if a PHI feeds into a min/max intrinsic which then feeds back to the PHI, the recursion breaking logic doesn't handle this case.
- **Expected Issue**: The recursion won't be broken for min/max patterns, potentially causing the analysis to either spin or give up prematurely, missing known bits information.
