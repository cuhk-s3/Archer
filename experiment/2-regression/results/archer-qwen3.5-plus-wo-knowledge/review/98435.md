## Test Strategies

### Strategy 1: Test m_AllOnes with vector containing undef elements

- **Target**: PatternMatch m_AllOnes() matcher with vector constants containing undef elements
- **Rationale**: The patch changes cstval_pred_ty to only ignore poison elements, not undef elements. Previously, m_AllOnes() would match a vector like <i32 -1, i32 undef, i32 -1, i32 undef>. After the fix, this should NOT match because undef elements are no longer ignored. This could cause transformations that rely on m_AllOnes() to fail to trigger when they previously would have.
- **Expected Issue**: Transformations that expect m_AllOnes() to match vectors with undef elements will no longer apply, potentially missing optimization opportunities or causing different code generation

### Strategy 2: Test m_Zero with vector containing undef elements

- **Target**: PatternMatch m_Zero() matcher with vector constants containing undef elements
- **Rationale**: Similar to m_AllOnes(), m_Zero() uses cst_pred_ty which now only ignores poison elements. A vector like <i32 0, i32 undef, i32 0> should no longer match m_Zero(). This affects many simplifications in InstructionSimplify and InstCombine that rely on matching zero vectors.
- **Expected Issue**: Simplifications like 'X | 0 -> X' or 'X ^ 0 -> X' may not apply when the zero constant has undef elements, potentially leaving redundant operations

### Strategy 3: Test m_Not with vector containing undef elements in the -1 mask

- **Target**: PatternMatch m_Not() matcher used in simplifyOrLogic and simplifyXorInst
- **Rationale**: The patch removed m_NotForbidUndef and changed m_Not to use m_AllOnes() which now rejects vectors with undef elements. Several logic simplifications in simplifyOrLogic (like '(~A ^ B) | (A & B) --> ~A ^ B') and simplifyXorInst rely on m_Not matching. If the -1 mask has undef elements, these transforms won't apply.
- **Expected Issue**: Logic simplifications that depend on recognizing ~A as 'xor A, -1' will fail when the -1 constant has undef elements, missing opportunities to simplify boolean logic

### Strategy 4: Test AShr simplification with -1 vector containing undef elements

- **Target**: simplifyAShrInst in InstructionSimplify.cpp
- **Rationale**: The simplification '-1 >>a X --> -1' uses m_AllOnes() to match the -1 operand. If the -1 is a vector with undef elements (like <i32 -1, i32 undef>), the match will now fail and the simplification won't apply. The comment even notes 'We could return the original -1 constant to preserve poison elements' suggesting awareness of this issue.
- **Expected Issue**: AShr operations with all-ones vectors containing undef elements won't be simplified to just the -1 constant, potentially leaving unnecessary shift operations

### Strategy 5: Test and(shl(zext(X), Y), SignMask) transform with undef in SignMask

- **Target**: InstCombineAndOrXor.cpp visitAnd transformation
- **Rationale**: The patch removed code that sanitized undef elements in the sign mask: 'SanitizedSignMask = Constant::replaceUndefsWith(...)'. Now the transform directly uses Op1 (the SignMask). If the SignMask has undef elements, the transformation may produce incorrect results because the undef elements won't be properly handled.
- **Expected Issue**: The transformation 'and(shl(zext(X), Y), SignMask) -> and(sext(X), SignMask)' may produce incorrect results when SignMask has undef elements, as the undef handling was removed but the semantics may require it

### Analysis

### Bug Triggering Analysis
The patch changes cstval_pred_ty-based pattern matchers (m_AllOnes, m_Zero, m_One, m_Not, etc.) to only ignore poison elements in vector constants, not undef elements. This is the intended behavior as stated in the patch description - LLVM now uses poison instead of undef for non-demanded vector elements.

The test case `test_or_zero_undef` demonstrates this: when trying to simplify `X | <0, undef, 0, undef>`, the m_Zero() matcher no longer matches because the vector contains undef elements. Before the fix, this would have matched and simplified to just X. After the fix, the optimization doesn't apply because undef elements are no longer ignored.

### Fix Weakness Analysis
This is NOT a bug - it's the intended behavior of the fix. The patch explicitly states:
- "Change all the cstval_pred_ty based PatternMatch helpers to only allow poison elements inside vector splats, not undef elements"
- "Historically, we used to represent non-demanded elements in vectors using undef. Nowadays, we use poison instead."
- "keeping support for undef elements no longer makes sense, and we should drop it"

The fix is working correctly. Tests with poison elements (like `test_or_zero_poison`) correctly simplify, while tests with undef elements don't trigger the optimization. This is the expected and desired behavior.

The verification shows:
1. With poison: `or <4 x i32> %x, <i32 0, i32 poison, i32 0, i32 poison>` simplifies to `%x`
2. With undef: `or <4 x i32> %x, <i32 0, i32 undef, i32 0, i32 undef>` does NOT simplify

This confirms the fix is working as intended - pattern matchers now correctly distinguish between poison (which can be ignored) and undef (which cannot be safely ignored in these contexts).
