## Test Strategies

### Strategy 1: Deep nested instruction chain with vector Op and scalar result

- **Target**: Test cross-lane operation handling when Op is a vector but instruction returns scalar
- **Rationale**: The fix added a check for `!I->getType()->isVectorTy()` when Op is a vector to handle cross-lane operations. A test with a vector Op feeding into a scalar-returning instruction (like extractelement) could expose issues if the recursion doesn't properly handle type mismatches.
- **Expected Issue**: Miscompilation when a vector value is recursively replaced through a chain that includes cross-lane operations like extractelement, potentially returning wrong scalar values

### Strategy 2: Recursive replacement with x^x pattern through multiple instruction levels

- **Target**: Test x^x simplification when both operands become RepOp through recursive replacement at different depths
- **Rationale**: The fix changed the x^x check from `NewOps[0] == NewOps[1]` with assertion to `NewOps[0] == RepOp && NewOps[1] == RepOp`. A test where operands become equal to RepOp through different recursion depths could expose issues if the recursion doesn't properly track RepOp equality.
- **Expected Issue**: Incorrect simplification of x^x to 0 when operands are not actually both RepOp, or failure to simplify when they should be

### Strategy 3: Absorber constant with recursive operand replacement

- **Target**: Test absorber pattern when absorber constant appears through recursive replacement rather than direct substitution
- **Rationale**: The fix changed the absorber check from `RepOp == Absorber` to `NewOps[0] == Absorber || NewOps[1] == Absorber`. A test where the absorber appears through recursive replacement (not direct) could expose issues if impliesPoison check doesn't work correctly with recursively replaced values.
- **Expected Issue**: Incorrect simplification to absorber constant when poison could leak, or failure to simplify when absorber pattern is valid

### Strategy 4: Recursion limit exhaustion with complex instruction chains

- **Target**: Test behavior when MaxRecurse limit is hit during deep recursive replacement
- **Rationale**: The fix added `if (!MaxRecurse--) return nullptr;` at the start. A test with deeply nested instruction chains could hit the recursion limit, and the behavior at the boundary needs to be correct to avoid partial simplifications.
- **Expected Issue**: Partial simplification or incorrect results when recursion limit is reached mid-traversal, potentially leaving instructions in inconsistent state

### Strategy 5: AnyReplaced flag with mixed replacement results

- **Target**: Test case where some operands are replaced but others are not, checking AnyReplaced logic
- **Rationale**: The fix added `AnyReplaced` flag to track if any replacement happened. A test where recursive calls return mixed results (some operands replaced, some not) could expose issues if the flag logic doesn't correctly handle all combinations.
- **Expected Issue**: Incorrect return of nullptr when replacements did occur, or incorrect continuation when no replacements actually happened

### Analysis

### Summary of Fix Analysis and Testing

The fix for InstructionSimplify's `simplifyWithOpReplaced()` function makes it recursive to support replacement of operands not only in the immediate instruction but also in instructions it uses. This re-applies PR63104 which was needed to fix a stage2 miscompile.

**Key changes in the fix:**
1. Added recursion limit check at function entry (`if (!MaxRecurse--) return nullptr;`)
2. Removed the check that required Op to be a direct operand of I
3. Added proper handling for vector Op with scalar-returning instructions (cross-lane operations)
4. Made operand replacement recursive by calling `simplifyWithOpReplaced` on each operand
5. Added `AnyReplaced` flag to track if any replacement occurred
6. Fixed x^x and x-x patterns to check `NewOps[0] == RepOp && NewOps[1] == RepOp` instead of relying on an assertion
7. Fixed absorber pattern to check if either operand equals the absorber and return the absorber constant

**Testing performed:**
- Tested 7 test cases thoroughly: select_eq_xor_recursive, select_eq_and_recursive_propagates_poison, select_eq_mul_absorber, select_eq_xor_recursive2, select_eq_and_recursive, select_eq_xor_recursive_allow_refinement, and custom test cases
- All transformations verified correct by alive2
- Differential testing with llubi/lli showed consistent behavior between original and optimized code
- All 5 Phase 1 strategies were covered:
  1. Deep nested instruction chain with vector Op and scalar result - tested with vector operations
  2. Recursive replacement with x^x pattern through multiple instruction levels - tested with xor patterns
  3. Absorber constant with recursive operand replacement - tested with mul/and absorber patterns
  4. Recursion limit exhaustion with complex instruction chains - tested with 30+ level deep chains
  5. AnyReplaced flag with mixed replacement results - tested with various conditional scenarios

**Conclusion:**
No bugs were found in the fix. The transformation is correct and the fix properly addresses the issues from PR63104. The recursive `simplifyWithOpReplaced()` function correctly handles:
- Deep instruction chains without hitting recursion limits prematurely
- x^x pattern simplification through multiple recursion levels
- Absorber constant patterns with recursive operand replacement
- Vector operations with proper cross-lane handling
- Mixed replacement scenarios with the AnyReplaced flag

The fix is sound and no additional issues were identified.
