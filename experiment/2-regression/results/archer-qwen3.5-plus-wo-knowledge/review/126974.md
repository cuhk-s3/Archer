## Test Strategies

### Strategy 1: Test samesign with mixed-sign constant ranges

- **Target**: isImpliedCondCommonOperandWithCR with constant ranges spanning both positive and negative values
- **Rationale**: The fix flips signedness when exactly one predicate has samesign, but if the constant range includes both positive and negative values, the samesign constraint may be violated, making the implication incorrect. For example, if LCR is [-10, 10] and we have 'icmp samesign ult X, LCR' implying 'icmp sgt X, RCR', the flip may produce wrong results when X can be negative.
- **Expected Issue**: Incorrect implication result (true/false instead of nullopt) when constant ranges span both signs, potentially causing miscompilations in optimizations that rely on this analysis.

### Strategy 2: Test both predicates having samesign

- **Target**: isImpliedCondCommonOperandWithCR where both LPred and RPred have samesign flag
- **Rationale**: The fix only handles the XOR case (exactly one has samesign). When both have samesign, the XOR is false and no special handling occurs. However, there may be valid implications that could be detected when both have samesign, especially when the signedness flip should apply to both.
- **Expected Issue**: Missed optimization opportunities where implications could be proven but aren't due to the fix not handling the both-samesign case.

### Strategy 3: Test samesign ult implying signed comparison with boundary values

- **Target**: Constant ranges at signed/unsigned boundary (e.g., INT_MIN, INT_MAX, 0)
- **Rationale**: The samesign ult comparison is equivalent to slt when operands have the same sign. Testing boundary values like 0, INT_MIN, INT_MAX can expose issues where the signedness flip produces incorrect results near these boundaries.
- **Expected Issue**: Incorrect implication at boundary values where signed and unsigned interpretations differ significantly, such as when comparing against 0 or INT_MIN.

### Strategy 4: Test samesign with empty or full constant ranges

- **Target**: Edge cases where LCR or RCR is empty set or full set
- **Rationale**: The CRImpliesPred lambda checks CR.icmp(Pred, RCR) which has special handling for empty sets (returns true). The interaction between samesign flipping and empty/full ranges may produce unexpected results.
- **Expected Issue**: Incorrect handling of empty or full constant ranges when combined with samesign predicates, potentially causing false implications.

### Strategy 5: Test samesign with equality predicates

- **Target**: Cases involving icmp samesign eq/ne comparisons
- **Rationale**: The getFlippedSignednessPredicate function returns equality predicates unchanged (since eq/ne are sign-independent). The fix may not correctly handle cases where one predicate is samesign eq/ne and the other is a relational predicate.
- **Expected Issue**: Incorrect implication results when mixing samesign equality predicates with relational predicates, as the signedness flip doesn't change equality predicates.

### Analysis

### Bug Triggering Analysis

The bug is triggered when:
1. The LHS comparison is a signed comparison (`sgt`) that can be true for values with different signs (e.g., positive > negative)
2. The RHS comparison uses `samesign` keyword with a constant of different sign than some values that satisfy the LHS

In the test case:
- LHS: `icmp sgt i32 %a, -10` is true for any a > -10, including positive values like a=5
- RHS: `icmp samesign uge i32 %a, -20` requires %a and -20 to have the same sign

When a=5:
- LHS: 5 > -10 is true (positive > negative in signed comparison)
- RHS: samesign uge 5, -20 → 5 is positive, -20 is negative → different signs → poison

The optimization incorrectly simplifies the select to always return 30, assuming the samesign uge is always true when the LHS is true. However, when a=5, the samesign constraint is violated, producing poison.

### Fix Weakness Analysis

The fix in `isImpliedCondCommonOperandWithCR` handles the case where exactly one predicate has `samesign` by flipping the signedness. However, it doesn't properly account for the fact that:

1. When LHS is `sgt` (no samesign) and RHS is `samesign uge`, the fix flips RHS to `sge` and checks implication
2. The implication `sgt X, -10` → `sge X, -20` is correct for signed comparison
3. BUT the `samesign` keyword adds an additional constraint: the result is poison if operands have different signs
4. The fix doesn't check whether the values satisfying LHS also satisfy the samesign constraint of RHS

The fix assumes that flipping signedness is sufficient, but it misses the samesign constraint validation. When LHS allows values with different signs than the RHS constant, the samesign comparison produces poison, not a boolean value.

This is a real bug because the optimization produces incorrect code that returns 30 when it should return poison (UB) for inputs like a=5.
