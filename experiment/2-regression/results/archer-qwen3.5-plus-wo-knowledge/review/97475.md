## Test Strategies

### Strategy 1: Undef/Poison Condition Test

- **Target**: select with potentially undef/poison condition
- **Rationale**: The fix doesn't explicitly check for undef/poison conditions. If the condition can be undef, the KnownBits inference would be incorrect since we can't determine which arm is selected.
- **Expected Issue**: Incorrect constant folding when condition is undef/poison, leading to wrong results

### Strategy 2: Vector Condition with Mixed Elements

- **Target**: select with vector condition where different elements imply different KnownBits
- **Rationale**: The current implementation treats the condition uniformly, but vector conditions may have different implications per element. This could lead to over-aggressive simplification.
- **Expected Issue**: Over-simplification of select arms when vector condition has mixed KnownBits implications across elements

### Strategy 3: FCmp Condition Mismatch

- **Target**: select with FCmp condition affecting integer values
- **Rationale**: findValuesAffectedByCondition handles FCmp but computeKnownBitsFromCond only handles ICmpInst. This mismatch could cause incorrect KnownBits computation for FCmp conditions.
- **Expected Issue**: Incorrect or missing simplification when select condition is an FCmp that should affect integer value KnownBits

### Strategy 4: Complex Logical Condition Test

- **Target**: select with complex logical AND/OR of multiple comparisons
- **Rationale**: The recursive handling of logical operations in computeKnownBitsFromCond may not correctly handle all combinations, especially with the Invert flag for the false arm.
- **Expected Issue**: Incorrect KnownBits computation for values affected by complex logical conditions, leading to wrong constant folding

### Strategy 5: Indirectly Affected Value Test

- **Target**: select arm with values indirectly affected through multiple instruction layers
- **Rationale**: The hasAffectedValue function traverses operands recursively but may miss some indirectly affected values, especially through complex instruction chains or when depth limits are hit.
- **Expected Issue**: Missed optimization opportunities or incorrect simplification when affected values are deeply nested or connected through complex instruction chains

### Bug #4

**Analysis:**
This is a confirmed real bug in the vector condition handling. The verify tool shows the transformation is incorrect with a concrete counterexample:

When %x = <0, 2>:
- Source: %x10 = <2, 0>, %cond = <true, false>, %s = <2, 2>
- Target: returns %x = <0, 2>

The bug is in the vector condition handling - the optimization incorrectly assumes uniform KnownBits across all vector elements. This relates to the "Vector Condition with Mixed Elements" strategy from Phase 1.

The test case vec_select_no_equivalence (test 0) has been verified with alive2 showing the bug, and the bug has been confirmed with difftest.
