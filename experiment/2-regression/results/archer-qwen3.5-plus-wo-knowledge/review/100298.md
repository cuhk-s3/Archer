## Test Strategies

### Strategy 1: FCmp condition with affected values

- **Target**: select with floating-point comparison condition
- **Rationale**: The findValuesAffectedByCondition function handles FCmp patterns (fneg, fabs) and adds affected values, but computeKnownBitsFromCond only handles ICmpInst. This mismatch could cause incorrect KnownBits computation or missed optimizations.
- **Expected Issue**: The optimization might incorrectly simplify select arms when the condition is an FCmp, or fail to simplify when it should.

### Strategy 2: Logical AND/OR condition with nested affected values

- **Target**: select with compound condition like (A && B) or (A || B)
- **Rationale**: The computeKnownBitsFromCond handles logical operations recursively with union/intersection logic, but findValuesAffectedByCondition just pushes operands to worklist. Complex logical conditions might not properly propagate affected values or known bits.
- **Expected Issue**: Incorrect KnownBits computation for select arms when condition involves logical operations, potentially leading to wrong constant folding.

### Strategy 3: Argument as affected value with non-instruction arm

- **Target**: select where arm is an argument that was marked as affected
- **Rationale**: findValuesAffectedByCondition can add arguments to the affected set (line 9570-9571), but hasAffectedValue returns false for non-Instruction values (line 3546). This could cause the optimization to miss cases where the arm itself is an affected argument.
- **Expected Issue**: Missed optimization opportunity when select arm is an argument that should be affected by the condition.

### Strategy 4: Inverted condition with different affected values

- **Target**: select where false arm simplification requires different affected values than true arm
- **Rationale**: The affected values are computed once with the original condition, but when CC.Invert=true for the false arm, the set of values affected by the inverted condition might be different. The current code uses the same affected set for both arms.
- **Expected Issue**: Incorrect simplification of false arm when the inverted condition affects different values than the original condition.

### Strategy 5: Vector select with mismatched element counts

- **Target**: select with vector condition and vector arms of different element counts
- **Rationale**: The code checks CondType->isVectorTy() == SelType->isVectorTy() but doesn't verify element count matching. Scalable vectors or mismatched counts could cause issues in KnownBits computation.
- **Expected Issue**: Incorrect KnownBits computation or crash when dealing with scalable vectors or mismatched vector element counts between condition and select arms.
