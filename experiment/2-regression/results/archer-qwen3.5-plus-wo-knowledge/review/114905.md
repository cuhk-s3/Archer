## Test Strategies

### Strategy 1: Overflow semantics test for Add-to-Mul conversion

- **Target**: Horizontal reduction with repeated values using add with nsw/nuw flags
- **Rationale**: The patch converts repeated additions (a + a + a) to multiplication (a * 3). However, this transformation changes overflow semantics. If the original additions had nsw (no signed wrap) or nuw (no unsigned wrap) flags, the multiplication may not preserve these semantics correctly, potentially causing undefined behavior or different results on overflow.
- **Expected Issue**: The optimized code may produce different results or trigger undefined behavior when overflow occurs, compared to the original scalar additions with nsw/nuw flags.

### Strategy 2: Floating-point precision test for FAdd-to-FMul conversion

- **Target**: Horizontal reduction with repeated floating-point values
- **Rationale**: The patch converts repeated floating-point additions (a + a + a) to multiplication (a * 3.0). This transformation is not semantically equivalent in IEEE 754 floating-point arithmetic due to different rounding behavior. (a + a) + a may round differently than a * 3.0, especially for values near rounding boundaries.
- **Expected Issue**: The optimized code may produce different floating-point results compared to the original scalar additions, violating IEEE 754 semantics when fast-math flags are not set.

### Strategy 3: Xor reduction with mixed even/odd counts

- **Target**: Horizontal reduction with xor operation where some values appear even times and others appear odd times
- **Rationale**: The emitReusedOps function for Xor creates a shuffle mask to zero out elements with even counts. The mask generation logic may have bugs when handling mixed even/odd counts, especially with non-power-of-2 vector widths or when the shuffle interacts with the reduction.
- **Expected Issue**: Incorrect shuffle mask generation leading to wrong results when xor reduction has values with mixed even/odd occurrence counts.

### Strategy 4: SameScaleFactor path with non-uniform vectorization

- **Target**: Horizontal reduction where all unique values have the same count but vectorization width doesn't divide evenly
- **Rationale**: When SameScaleFactor is true, the code takes a special path that emits emitScaleForReusedOps on the reduced subtree. If the vectorization width doesn't divide the total count evenly, or if the reduction tree structure doesn't match expectations, the scale factor may be applied incorrectly.
- **Expected Issue**: Incorrect scale factor applied to the reduced value, producing wrong results when the number of repetitions doesn't align with vectorization width.

### Strategy 5: OptReusedScalars without vectorization (AnyVectorized=false)

- **Target**: Horizontal reduction with repeated values where no vectorization occurs but OptReusedScalars is true
- **Rationale**: When OptReusedScalars is true but no vectorization happens (AnyVectorized=false), the code emits scalar ops for each unique value. This path may have issues with proper tracking of VectorizedVals, potentially causing the final reduction to include values that should have been excluded or vice versa.
- **Expected Issue**: Incorrect final reduction result due to improper tracking of which values were already processed, leading to double-counting or missing values in the final reduction.

## Bugs Found

### Bug #1

**Analysis:**
This is a real bug. The original code has `add nsw` instructions which produce poison on overflow. When 1073741824 is added 5 times:
- Step 1: 1073741824 + 1073741824 = 2147483648 (overflows i32 max of 2147483647, produces poison)
- The poison propagates through the chain, and the function returns poison, triggering UB.

The transformed code uses `mul i32 %a, 5` which does NOT have the nsw flag, so it wraps around normally without producing poison. This is a semantic mismatch - the transformation incorrectly removes the nsw semantics.

The fix incorrectly assumes that `add nsw` repeated N times is equivalent to `mul` without nsw, but they have different overflow behavior.

**Original LLVM IR:**
```llvm

; This tests the strategy: "Overflow semantics test for Add-to-Mul conversion"
; The nsw flag on add has different overflow semantics than mul

target datalayout = "e-m:e-i64:64-f80:128-n8:16:32:64-S128"
target triple = "x86_64--"

define i32 @test_add_overflow_nsw(i32 %a) {
entry:
  %t1 = add nsw i32 %a, %a
  %t2 = add nsw i32 %t1, %a
  %t3 = add nsw i32 %t2, %a
  %t4 = add nsw i32 %t3, %a
  ret i32 %t4
}

define i32 @main(i32 %argc, ptr %argv) {
entry:
  %r = call i32 @test_add_overflow_nsw(i32 1073741824)
  ret i32 %r
}
```

### Analysis

### Bug Summary

**Bug Found**: The SLPVectorizer patch incorrectly transforms repeated `add nsw` instructions to `mul` without preserving the `nsw` flag, changing overflow semantics.

**Bug Triggering Analysis**:
- Original code: `%t1 = add nsw i32 %a, %a` ... (chain of 4 add nsw)
- Transformed code: `%0 = mul i32 %a, 5` (mul without nsw)
- With input 1073741824:
  - Original: Produces poison on first overflow (2147483648 > INT32_MAX), triggers UB
  - Transformed: Wraps normally, returns a value (no UB)

**Fix Weakness Analysis**:
The `emitScaleForReusedOps` function in SLPVectorizer.cpp converts repeated additions to multiplication but does not:
1. Preserve `nsw`/`nuw` flags from the original additions
2. Check if the original operations had overflow semantics that must be preserved
3. Verify that the transformation is semantically equivalent when overflow flags are present

**Recommended Fix**:
The optimization should either:
1. Only apply when the original additions don't have `nsw`/`nuw` flags, OR
2. Add the corresponding `nsw`/`nuw` flag to the generated `mul` instruction (though this still may not be semantically equivalent due to different overflow conditions), OR
3. Perform more sophisticated analysis to ensure overflow semantics are preserved

**Test Coverage**:
- Tests 0 and 1 were fully tested covering all 5 strategies
- Bug confirmed via difftest with llubi
- Other strategies (xor, samescale, optreused, fadd) verified correct for non-overflow cases
