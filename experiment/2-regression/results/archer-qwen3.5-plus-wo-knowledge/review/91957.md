## Test Strategies

### Strategy 1: Integer overflow in reassociation

- **Target**: expr-reassociate-int.ll
- **Rationale**: The patch adds integer mul/add support but doesn't check for overflow. The transformation ((A*B)+(C*D))*E -> (A*E*B)+(C*E*D) may produce different results when overflow occurs. Test with values that overflow in the original form but not in the transformed form, or vice versa.
- **Expected Issue**: Incorrect transformation when integer overflow behavior differs between original and transformed expressions, leading to wrong computed values.

### Strategy 2: Vector type integer reassociation

- **Target**: expr-reassociate-int.ll
- **Rationale**: The code checks isIntOrIntVectorTy() but the existing tests only cover scalar integers. Vector types may have different semantics or the transformation might not handle element-wise operations correctly.
- **Expected Issue**: Crash or incorrect transformation when processing vector integer types due to untested code paths in hoistMulAddAssociation.

### Strategy 3: NSW/NUW flag preservation

- **Target**: expr-reassociate-int.ll
- **Rationale**: The patch creates new mul instructions with Builder.CreateMul() without preserving NSW/NUW flags from the original operations. This could change overflow semantics.
- **Expected Issue**: Generated code lacks NSW/NUW flags that were present in original, potentially missing optimization opportunities or changing undefined behavior boundaries.

### Strategy 4: Limit boundary conditions

- **Target**: expr-reassociate-int.ll
- **Rationale**: The IntAssociationUpperLimit defaults to 5. Test with exactly 5, 6, and 0 reassociations to verify limit checking works correctly for integers. The limit check happens after adding to Changes array.
- **Expected Issue**: Off-by-one errors in limit checking could allow more transformations than intended or block valid transformations.

### Strategy 5: Mixed integer and FP in same function

- **Target**: expr-reassociate-int.ll or expr-reassociate.ll
- **Rationale**: Test a function with both integer and FP reassociation opportunities to ensure the separate statistics and limits work correctly and don't interfere with each other.
- **Expected Issue**: Incorrect limit selection (using FP limit for int or vice versa) when both types are present, or statistics being incorrectly attributed.

### Analysis

### Bug Triggering Analysis
The test case demonstrates that the LICM transformation for integer mul/add reassociation does NOT preserve NSW/NUW flags on the hoisted multiplication instructions.

In the original IR:
- `%mul.1 = mul nuw nsw i64 %cell.1, %d1` - has nuw nsw flags
- `%mul.2 = mul nuw nsw i64 %cell.2, %d2` - has nuw nsw flags
- `%reass.mul = mul nuw nsw i64 %reass.add, %delta` - has nuw nsw flags

After transformation:
- `%factor.op.mul = mul i64 %d1, %delta` - NO nuw nsw flags!
- `%factor.op.mul1 = mul i64 %d2, %delta` - NO nuw nsw flags!
- `%mul.1 = mul nuw nsw i64 %cell.1, %factor.op.mul` - keeps flags on the variant multiply

The hoisted invariant multiplications (`%factor.op.mul`, `%factor.op.mul1`) lose their NSW/NUW flags because the code uses `Builder.CreateMul()` without preserving the flags from the original operation.

### Fix Weakness Analysis
The fix in `hoistMulAddAssociation` creates new multiplication instructions using:
```cpp
Mul = Builder.CreateMul(U->get(), Factor, "factor.op.mul");
```

This does not preserve the NSW/NUW flags that may have been present on the original operations. The FP version uses `CreateFMulFMF` which properly transfers fast-math flags, but the integer version doesn't have equivalent flag preservation.

This is a correctness issue because:
1. NSW/NUW flags indicate undefined behavior on overflow - losing them changes the semantics
2. The transformation may introduce undefined behavior that wasn't present before, or vice versa
3. Downstream optimizations may miss opportunities due to missing flags

The test verifies this by showing the transformed IR lacks the nuw nsw flags on the hoisted multiplications.
