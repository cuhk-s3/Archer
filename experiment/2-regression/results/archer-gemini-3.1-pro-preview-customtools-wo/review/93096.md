## Test Strategies

### Strategy 1: Truncate with negative values and nuw

- **Target**: TruncInst with nuw flag
- **Rationale**: The fix uses `Range.getActiveBits() <= DestWidth` to infer `nuw`. However, `getActiveBits()` on a negative number (e.g., -1) returns the bit width of the source type, which is greater than `DestWidth`. But wait, if `Range` is `[-1, 0]`, `getUnsignedMax()` is `-1` (all ones), so `getActiveBits()` is `BitWidth`. This means `nuw` won't be set. What if `Range` is `[-2, -1]`? `getUnsignedMax()` is `-1`, `getActiveBits()` is `BitWidth`. What if `Range` is `[0, 1]`? `getUnsignedMax()` is `1`, `getActiveBits()` is 1. If `DestWidth` is 1, `nuw` is set. This is correct. Wait, what if `Range` is empty? `getActiveBits()` is 0. `nuw` is set. This is correct. What if `Range` is full? `getActiveBits()` is `BitWidth`. `nuw` is not set. This is correct. What if `Range` is wrapped? `[2, 1]`. `getUnsignedMax()` is `BitWidth` all ones. `getActiveBits()` is `BitWidth`. `nuw` is not set. This is correct. Is there any case where `getActiveBits() <= DestWidth` but `nuw` is violated? `nuw` means the truncated bits are all zero. If `getActiveBits() <= DestWidth`, then all bits above `DestWidth` are zero. So the truncated bits are all zero. This implies `nuw` is correct.
- **Expected Issue**: The trunc instruction will be incorrectly marked with `nuw` flag, which could lead to incorrect optimizations later.

### Strategy 2: Truncate with nsw flag and getMinSignedBits

- **Target**: TruncInst with nsw flag
- **Rationale**: The fix uses `Range.getMinSignedBits() <= DestWidth` to infer `nsw`. `nsw` means the truncated bits are all equal to the sign bit of the truncated result. `getMinSignedBits()` returns the minimum number of bits needed to represent the signed values in the range. If `getMinSignedBits() <= DestWidth`, it means all values in the range can be represented in `DestWidth` bits as signed integers. This exactly means that the top bits (from `DestWidth` to `BitWidth-1`) are all equal to the sign bit of the `DestWidth`-bit value. Therefore, `nsw` is correct. Wait, what if `Range` is empty? `getMinSignedBits()` is 0. `nsw` is set. Correct. What if `Range` is full? `getMinSignedBits()` is `BitWidth`. `nsw` not set. Correct. What if `Range` is wrapped? `[2, -1]`. `getSignedMin()` is `-2` (if 3-bit, `110`), `getSignedMax()` is `1`. `getMinSignedBits()` is `max(getSignificantBits(-2), getSignificantBits(1))`. `-2` is `110`, significant bits is 2. `1` is `001`, significant bits is 2. So `getMinSignedBits()` is 2. If `DestWidth` is 2, `nsw` is set. But wait! The range `[2, -1]` in 3-bit is `{2, 3, 4, 5, 6, 7}`. Wait, `[2, -1]` means `{2, 3, 4, 5, 6}`. `getSignedMin()` for wrapped set is `APInt::getSignedMinValue(getBitWidth())` which is `-4`. `getSignedMax()` is `APInt::getSignedMaxValue(getBitWidth())` which is `3`. So `getMinSignedBits()` will be `max(3, 3) = 3`. So it won't be `<= 2`. Let's check `ConstantRange::getSignedMin()` and `getSignedMax()`. For wrapped sets, they return the absolute min and max possible values for the bit width. So `getMinSignedBits()` will be `BitWidth`. Thus `nsw` won't be set. This is safe.
- **Expected Issue**: The trunc instruction will be incorrectly marked with `nsw` flag, which could lead to incorrect optimizations later.

### Strategy 3: Truncate with nuw flag and poison/undef

- **Target**: TruncInst with undef operand
- **Rationale**: The fix uses `GetRange` which calls `getConstantRange(Solver.getLatticeValueFor(Op), Op->getType(), /*UndefAllowed=*/false)`. If the operand is undef, `getConstantRange` with `UndefAllowed=false` returns a full range. So `getActiveBits()` will be `BitWidth`, and `nuw` won't be set. If the operand is a constant range including undef, it returns the range. Wait, if `UndefAllowed=false`, `getConstantRange` returns full range if the lattice value is `constantrange_including_undef`. Let's check `getConstantRange` in `SCCPSolver.cpp`. `if (LV.isConstantRange(UndefAllowed)) return LV.getConstantRange(); return ConstantRange::getFull(Ty->getScalarSizeInBits());`. If `LV` is `constantrange_including_undef` and `UndefAllowed` is `false`, `isConstantRange(false)` returns `false`. So it returns full range. This is safe.
- **Expected Issue**: The trunc instruction will be incorrectly marked with `nuw` flag, which could lead to incorrect optimizations later.

### Strategy 4: Truncate with nuw flag and constant range including undef

- **Target**: TruncInst with constant undef operand
- **Rationale**: Wait, what if the operand is a constant? `GetRange` says: `if (auto *Const = dyn_cast<ConstantInt>(Op)) return ConstantRange(Const->getValue());`. If it's `undef`, `isa<Constant>(Op)` is true, so it returns `ConstantRange::getFull(Bitwidth)`. This is safe.
- **Expected Issue**: The trunc instruction will be incorrectly marked with `nuw` flag, which could lead to incorrect optimizations later.

### Strategy 5: Truncate with nuw flag and vector types

- **Target**: TruncInst with vector types
- **Rationale**: The fix uses `TI->getDestTy()->getScalarSizeInBits()`. If `TI` is a vector truncation, `DestWidth` is the scalar size. `Range` is obtained from `GetRange`. For vectors, `SCCPSolver` tracks the lattice value of the whole vector. If all elements are the same constant, it might be tracked as a single constant. Wait, `SCCPSolver` tracks vector elements? No, `SCCPSolver` tracks the whole vector as a single `ValueLatticeElement`. If it's a constant range, it represents the range of all elements. `GetRange` returns this range. `getActiveBits()` will be the max active bits of any element. `DestWidth` is the scalar size. This is correct for vectors as well. Wait, what if `Range` is empty? `getActiveBits()` is 0. `nuw` is set. Correct. What if `Range` is full? `getActiveBits()` is `BitWidth`. `nuw` not set. Correct. Is there any issue with vectors? `TI->getDestTy()->getScalarSizeInBits()` gets the scalar size. `Op->getType()->getScalarSizeInBits()` gets the scalar size. `ConstantRange` bitwidth is the scalar size. This is consistent.
- **Expected Issue**: The trunc instruction will be incorrectly marked with `nuw` flag, which could lead to incorrect optimizations later.
