## Test Strategies

### Strategy 1: Different Source Types for Trunc + Trunc

- **Target**: icmp eq (trunc nuw X), (trunc nuw Y) where X and Y have different types
- **Rationale**: The fold allows X and Y to have different types if both truncs have one use. However, if X and Y have different types, `Builder.CreateIntCast(Y, X->getType(), YIsSExt)` will cast Y to X's type. If X is larger than Y, Y will be extended. If the comparison is equality, and the truncated values are equal, the extended Y might not equal X because X could have non-zero upper bits that were truncated away, while Y's extended upper bits might be zero (or sign-extended). Wait, if it's `trunc nuw`, the upper bits of X and Y must be zero. If it's `trunc nsw`, the upper bits must be sign bits. If X and Y have different types, say X is i32 and Y is i16, and both are truncated to i8. If X is `trunc nuw i32 1 to i8` and Y is `trunc nuw i16 1 to i8`, then X is 1 and Y is 1. The comparison is `1 == 1` (true). The folded comparison is `X == zext(Y to i32)`. `1 == 1` (true). What if X is `trunc nsw i32 -1 to i8` and Y is `trunc nsw i16 -1 to i8`? X is -1, Y is -1. Comparison is true. Folded is `X == sext(Y to i32)`. `-1 == -1` (true). What if X is `trunc nuw i32 257 to i8`? That's poison. So the upper bits are guaranteed to be zero or sign bits. Thus, different types might actually be safe! Let me re-evaluate.
- **Expected Issue**: The comparison will be evaluated incorrectly because the upper bits of X and Y are not guaranteed to be equal when X and Y have different types, even if the truncated values are equal.

### Strategy 2: Different Source Types for Trunc + ZExt/SExt

- **Target**: icmp eq (trunc nsw X), (zext Y)
- **Rationale**: The fold allows X and Y to have different types. For `icmp (trunc nuw X), (zext Y)`, Y is extended to X's type. Since X is `trunc nuw`, its upper bits are zero. Y is `zext`, so its upper bits are zero. This seems safe. For `icmp (trunc nsw X), (sext Y)`, X's upper bits are sign bits. Y is `sext`, so its upper bits are sign bits. This also seems safe. Wait, what if Y is `zext` and X is `trunc nsw`? The code says: `else if (match(&Cmp, m_c_ICmp(Pred, m_NSWTrunc(m_Value(X)), m_OneUse(m_ZExtOrSExt(m_Value(Y)))))) { YIsSExt = isa<SExtInst>(Cmp.getOperand(0)) || isa<SExtInst>(Cmp.getOperand(1)); }`. If Y is `zext`, `YIsSExt` is false. So Y is zero-extended to X's type. But X is `trunc nsw`, so its upper bits are sign bits. If X is negative, its upper bits are 1s. Y is zero-extended, so its upper bits are 0s. The comparison `X == zext(Y)` will be false, even if the truncated X equals Y! Let's trace this: `icmp eq (trunc nsw X), (zext Y)`. Suppose X is i16, Y is i8. Truncate X to i8. Suppose X is -1 (0xFFFF). `trunc nsw i16 -1 to i8` is -1 (0xFF). This is valid because the truncated bits (all 1s) match the sign bit of the result (1). Y is i8, suppose Y is 255 (0xFF). `zext i8 255 to i8` is 255 (0xFF). The original comparison is `icmp eq i8 0xFF, 0xFF`, which is true. The folded comparison is `icmp eq i16 X, zext(Y to i16)`. X is 0xFFFF. `zext(Y to i16)` is 0x00FF. `0xFFFF == 0x00FF` is false! This is a bug!
- **Expected Issue**: The comparison will be evaluated incorrectly because the upper bits of X and Y are not guaranteed to be equal when X and Y have different types, even if the truncated values are equal.

### Strategy 3: Different Source Types for Trunc + ZExt/SExt with swapped operands
