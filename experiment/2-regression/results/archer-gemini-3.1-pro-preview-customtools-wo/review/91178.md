## Test Strategies

### Strategy 1: Vector Op with Scalar Instruction

- **Target**: A test case where `simplifyWithOpReplaced` is called with a vector `Op` and a scalar instruction `I` (like `extractelement` or a vector reduction intrinsic).
- **Rationale**: The previous code assumed that if the replaced operand `Op` is a vector, the instruction `I` must also return a vector. With recursive replacement, `Op` could be a vector while `I` is a scalar instruction (e.g., `extractelement`). This would trigger the assertion `I->getType()->isVectorTy()`.
- **Expected Issue**: Compiler crash due to assertion failure `assert(I->getType()->isVectorTy() && "Vector type mismatch");`.

### Strategy 2: x ^ x or x - x with Replacement Higher Up

- **Target**: A test case with a `sub` or `xor` instruction where both operands are identical instructions that contain `Op`. When `Op` is replaced by `RepOp`, both operands simplify to the same value `X` (not `RepOp`).
- **Rationale**: The previous code assumed that if `NewOps[0] == NewOps[1]` for `sub` or `xor`, then `NewOps[0]` must be exactly `RepOp`. With recursive replacement, `NewOps[0]` and `NewOps[1]` could both simplify to the same value `X` (where `X != RepOp`) after `Op` is replaced by `RepOp` deeper in the instruction chain. This would cause the assertion to fail.
- **Expected Issue**: Compiler crash due to assertion failure `assert(NewOps[0] == RepOp && "Precondition for non-poison assumption");`.

### Strategy 3: Absorber Constant with Replacement Higher Up

- **Target**: A test case with a binary operator (like `and`, `mul`, `or`) where one operand simplifies to the absorber constant (e.g., 0 for `and`/`mul`, -1 for `or`) after `Op` is replaced by `RepOp` deeper in the instruction chain.
- **Rationale**: The previous code checked if `RepOp` itself was the absorber constant. With recursive replacement, `RepOp` might not be the absorber, but replacing `Op` with `RepOp` deeper in the chain might cause one of the operands (`NewOps[0]` or `NewOps[1]`) to simplify to the absorber constant. The old code would fail to apply the absorber fold in this case.
- **Expected Issue**: Missed optimization or miscompile due to failing to recognize the absorber constant.

### Strategy 4: Recursive Replacement Depth Limit

- **Target**: A test case with a deeply nested chain of instructions (e.g., a long chain of `add` or `phi` nodes) that triggers `simplifyWithOpReplaced`.
- **Rationale**: The patch makes `simplifyWithOpReplaced` recursive and introduces `MaxRecurse`. If the recursion depth is not properly decremented or checked, deeply nested instruction chains could cause a stack overflow. The patch changes `MaxRecurse - 1` to `MaxRecurse` in the `simplifyInstructionWithOperands` call, relying on the `--MaxRecurse` at the beginning of the function. We should test deeply nested chains to ensure the limit works correctly.
- **Expected Issue**: Infinite recursion or stack overflow during compilation.
