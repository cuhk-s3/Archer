## Test Strategies

### Strategy 1: Test with ctlz/cttz used as shift amount where shift amount is out of bounds

- **Target**: A function where ctlz/cttz is used as a shift amount, and the input can be zero.
- **Rationale**: The patch assumes that if ctlz/cttz is used as a shift amount, it's safe to set `is_zero_poison` to true. However, if the input to ctlz/cttz is zero, the result is the bitwidth. If this result is used as a shift amount, the shift instruction will produce poison (since shifting by bitwidth or more is poison). By setting `is_zero_poison` to true, the ctlz/cttz instruction itself produces poison when the input is zero. This changes the semantics from "shift produces poison" to "ctlz/cttz produces poison". This is generally safe because poison propagates, but we need to ensure there are no edge cases where this difference matters.
- **Expected Issue**: The shift instruction will produce poison, but the ctlz/cttz instruction will produce a defined value (bitwidth). This mismatch can lead to incorrect optimizations if the ctlz/cttz result is used elsewhere or if the shift amount is expected to be within bounds.

### Strategy 2: Test with ctlz/cttz used as shift amount in a non-standard shift instruction

- **Target**: A function where ctlz/cttz is used as a shift amount in a shift instruction that might not produce poison for out-of-bounds shift amounts (if any exist and are matched by `m_Shift`).
- **Rationale**: The patch uses `m_Shift` to match the user of ctlz/cttz. `m_Shift` matches `shl`, `lshr`, and `ashr`. These instructions produce poison if the shift amount is out of bounds. However, if there are other instructions that `m_Shift` matches (or might match in the future) that do not produce poison for out-of-bounds shift amounts, this optimization would be incorrect. We should verify that `m_Shift` only matches instructions that produce poison for out-of-bounds shift amounts.
- **Expected Issue**: The shift instruction might not produce poison if it's a funnel shift or similar instruction that handles out-of-bounds shift amounts gracefully. In this case, setting `is_zero_poison` to true on the ctlz/cttz instruction would incorrectly introduce poison into the program.

### Strategy 3: Test with ctlz/cttz used as shift amount with multiple uses

- **Target**: A function where ctlz/cttz is used as a shift amount, but also has another use (e.g., returned or used in an arithmetic operation).
- **Rationale**: The patch checks `II.hasOneUse()`. If the ctlz/cttz instruction has multiple uses, the optimization is not applied. This is safe, but we should test this case to ensure the optimization is indeed not applied.
- **Expected Issue**: The optimization might not trigger if the ctlz/cttz instruction has multiple uses, even if all uses are shift amounts. This is a missed optimization opportunity, but not a correctness issue. However, if the `hasOneUse()` check is removed or modified in the future, it could lead to correctness issues if one of the uses is not a shift amount.

### Strategy 4: Test with ctlz/cttz used as shift amount in a conditionally executed block

- **Target**: A function where ctlz/cttz is executed unconditionally, but its result is used as a shift amount in a conditionally executed block.
- **Rationale**: If the shift instruction is in a conditionally executed block, and the condition ensures that the input to ctlz/cttz is non-zero, setting `is_zero_poison` to true is safe. However, if the condition does not ensure this, and the ctlz/cttz instruction is executed unconditionally, setting `is_zero_poison` to true might introduce poison if the input is zero. The `hasOneUse()` check ensures that the only use is the shift instruction, so if the shift is not executed, the poison is not observed. We should test this to confirm.
- **Expected Issue**: The optimization might incorrectly set `is_zero_poison` to true if the shift instruction is not actually executed when the input to ctlz/cttz is zero. For example, if the shift is guarded by a condition that checks if the input is non-zero. In this case, the ctlz/cttz instruction would produce poison, which might be observed if it's used before the branch. However, the `hasOneUse()` check mitigates this, as the only use is the shift instruction.

### Strategy 5: Test with ctlz/cttz used as shift amount with vector types

- **Target**: A function where ctlz/cttz is used as a shift amount with vector types, and some elements of the input can be zero.
- **Rationale**: The patch applies to vector types as well. We need to ensure that the optimization is correct for vectors, especially when only some elements are zero.
- **Expected Issue**: The optimization might interact incorrectly with vector types. If the input is a vector, and some elements are zero while others are non-zero, setting `is_zero_poison` to true will make the corresponding elements of the result poison. This is correct if the shift instruction also produces poison for those elements. We should test vector types to ensure the semantics are preserved.
