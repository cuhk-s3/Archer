## Test Strategies

### Strategy 1: SIToFP Operand Demotion

- **Target**: Vector of `sitofp` instructions where the operands are derived from smaller integer types (e.g., `sext i8 to i32`).
- **Rationale**: The patch adds support for demoting the operands of `sitofp` instructions if they have enough sign bits. Testing with operands that are explicitly sign-extended from a smaller type ensures this path is exercised.
- **Expected Issue**: The SLPVectorizer should successfully demote the operands of the `sitofp` instructions to a smaller bitwidth (e.g., `i8` or `i16`), reducing the cost of the vectorization tree. If it fails to do so, the patch's logic for `sitofp` is not working as intended.

### Strategy 2: UIToFP Operand Demotion

- **Target**: Vector of `uitofp` instructions where the operands are derived from smaller integer types (e.g., `zext i8 to i32`).
- **Rationale**: The patch also adds support for `uitofp`. Testing with operands that are zero-extended from a smaller type ensures the unsigned to floating-point conversion is correctly analyzed for minimum bitwidth.
- **Expected Issue**: The SLPVectorizer should successfully demote the operands of the `uitofp` instructions. Since `uitofp` treats its operands as unsigned, the demotion logic must correctly handle zero-extension or ensure that the sign bit is zero. Incorrect handling could lead to miscompilations where negative values are incorrectly produced.

### Strategy 3: ICmp Signed Operand Demotion

- **Target**: Vector of `icmp` instructions with signed predicates where the operands are derived from smaller integer types.
- **Rationale**: The patch adds support for demoting operands of `icmp` instructions. Testing with signed comparisons (`sgt`, `slt`, etc.) and operands that have enough sign bits exercises this new logic.
- **Expected Issue**: The SLPVectorizer should demote the operands of the `icmp` instructions to a smaller bitwidth, performing the comparison on the narrower type or extending it back efficiently. Failure to do so indicates the `icmp` logic is not triggering.

### Strategy 4: ICmp Unsigned Operand Demotion

- **Target**: Vector of `icmp` instructions with unsigned predicates where the operands are derived from smaller integer types.
- **Rationale**: Testing `icmp` with unsigned predicates (`ugt`, `ult`, etc.) ensures that the bitwidth reduction logic (which relies on sign bits) does not violate the semantics of unsigned comparisons.
- **Expected Issue**: The SLPVectorizer should correctly demote the operands. If the demotion incorrectly sign-extends values that should be zero-extended for an unsigned comparison, it could lead to incorrect comparison results (miscompilation).

### Strategy 5: Mixed ExtraBitWidthNodes Roots

- **Target**: A complex vectorization tree that includes `trunc`, `sitofp`, and `icmp` instructions sharing the same demotable operands.
- **Rationale**: The patch renames `TruncNodes` to `ExtraBitWidthNodes` and changes the `IsTruncRoot` logic to check if the user is actually a `Trunc` instruction. Testing a mix of `trunc`, `sitofp`, and `icmp` instructions in the same vectorization tree exercises this shared logic.
- **Expected Issue**: The SLPVectorizer might incorrectly identify the root as a `Trunc` root or fail to explore the tree deeply enough, leading to missed optimization opportunities or incorrect bitwidth calculations.
