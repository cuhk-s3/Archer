### Strategy 1: Truncated abs with poison flag

- **Target**: A vectorization tree ending in a `trunc` of an `llvm.abs` intrinsic, where the operand of `abs` has values that change sign when truncated.
- **Rationale**: The patch modifies `SLPVectorizer` to allow demoting (truncating) the operands of `llvm.abs` and other min/max intrinsics. For `llvm.abs`, the second argument is a boolean flag indicating whether the result is poison if the input is `INT_MIN`. The patch attempts to handle this by setting the flag to `false` if the bitwidth is reduced: `if (ID == Intrinsic::abs && It != MinBWs.end() && It->second.first < DL->getTypeSizeInBits(CEI->getType())) ScalarArg = Builder.getFalse();`. However, this check uses `CEI->getType()`, which is the return type of the original scalar call. The condition `It->second.first < DL->getTypeSizeInBits(CEI->getType())` checks if the new bitwidth is less than the original bitwidth. If it is, it sets the flag to `false`. But what if the original flag was `true`? If the original flag was `true`, it means the original scalar `abs` returns poison for `INT_MIN` of the *original* type. If we truncate the operation to a smaller type, say from `i32` to `i8`, a value like `-128` (which is `INT_MIN` for `i8`) was *not* `INT_MIN` for `i32`. So the original `abs` would return `128`. But if we truncate the operation to `i8` and use `llvm.abs.i8(..., true)`, it will return poison for `-128`. The patch correctly sets the flag to `false` to avoid this. Wait, is there a bug here? Let's re-read the patch.
```cpp
          if (ID == Intrinsic::abs && It != MinBWs.end() &&
              It->second.first < DL->getTypeSizeInBits(CEI->getType()))
            ScalarArg = Builder.getFalse();
```
This seems correct. It sets the flag to `false` if the bitwidth is reduced.
Wait, what if the bitwidth is *not* reduced, but the operands are truncated? No, `It->second.first` is the new bitwidth.
Let's look at the `collectValuesToDemote` logic for `abs`:

This means for `Intrinsic::abs`, it NEVER checks if the operands can actually be safely truncated! It just assumes they can be truncated to `BitWidth` (which was determined by the users of the `abs` instruction, e.g., a `trunc` instruction).
Is it always safe to truncate the operand of `abs`?
Suppose we have `trunc (abs i32 %x) to i8`.
Can we replace this with `abs i8 (trunc i32 %x to i8)`?
Let's check:
If `%x = 129` (0x00000081).
`abs(%x)` = 129.
`trunc(129) to i8` = -127 (0x81).
If we truncate first:
`trunc(%x) to i8` = -127 (0x81).
`abs(-127)` = 127 (0x7F).
These are NOT equal! `-127 != 127`.
So `trunc (abs %x)` is NOT equivalent to `abs (trunc %x)` in general!
It is only equivalent if the sign bit of the truncated type matches the sign bit of the original type, or something like that. Specifically, we need `SignBits <= ComputeNumSignBits(%x)`.
But the patch completely omits `CallChecker` for `Intrinsic::abs`!

This is a massive bug. `abs` cannot be arbitrarily truncated.
- **Expected Issue**: The vectorized `llvm.abs` intrinsic will return a poison value for `INT_MIN` of the truncated type, which is incorrect because the original scalar `llvm.abs` was operating on a larger type where the value was not `INT_MIN`. This leads to a miscompilation where a valid value becomes poison.

### Strategy 2: Unsafe truncation of llvm.abs operand

- **Target**: Create a test with `llvm.abs.i32` followed by `trunc i32 ... to i8`, where the input to `abs` is a value like `129` (or a variable with known bits that allow it to be `129`). The SLP vectorizer will incorrectly vectorize and truncate the `abs`.
- **Rationale**: As analyzed above, `CallChecker` is not set for `Intrinsic::abs`. This means `SLPVectorizer` will demote the `abs` instruction and its operands to a smaller bitwidth solely based on the fact that its result is truncated, without verifying if the operand's value fits in the smaller bitwidth (i.e., without checking `ComputeNumSignBits`). For example, `trunc i32 (abs i32 129) to i8` yields `129` (which is `-127` in `i8`), but `abs i8 (trunc i32 129 to i8)` yields `abs i8 (-127) = 127`. The results differ.
- **Expected Issue**: The vectorized `llvm.abs` will produce incorrect results because the operand was truncated without checking if it's safe to do so (e.g., if the value exceeds the range of the truncated type).

### Strategy 3: Incorrect bitwidth check for smin/smax

- **Target**: Focus on `llvm.abs`.
- **Rationale**: For `smin` and `smax`, the `CompChecker` checks if `SignBits <= ComputeNumSignBits(...)`. `SignBits = OrigBitWidth - BitWidth`. This ensures that the value can be safely truncated to `BitWidth` without losing its sign or magnitude. This seems correct for `smin`/`smax`. However, the issue with `abs` is the most prominent one. Let's also consider `umin`/`umax`. For `umin`/`umax`, it checks `MaskedValueIsZero(..., Mask)`. This ensures the upper bits are zero, so truncation is safe. This is also correct. The missing check for `abs` is the main flaw.
- **Expected Issue**: The vectorized `llvm.smin` or `llvm.smax` will produce incorrect results because the `CallChecker` uses `OrigBitWidth - BitWidth` which might be incorrect if `BitWidth` is not a power of 2, but `MinBW` is used for the actual vector type. Wait, `BitWidth` is the exact bitwidth needed.

### Strategy 4: Negative value truncation for llvm.abs

- **Target**: Test with negative values that don't fit in the truncated type.
- **Rationale**: If the input to `abs` is `-129` (0xFFFFFF7F). `abs(-129) = 129`. `trunc(129) to i8 = -127` (0x81). If truncated first: `trunc(-129) to i8 = -127` (0x81). `abs(-127) = 127` (0x7F). Again, mismatch. The lack of `CallChecker` for `abs` means any value that doesn't fit in the truncated signed range will be miscompiled.
- **Expected Issue**: The vectorized `llvm.abs` will produce incorrect results for negative values that become positive when truncated, or vice versa.

### Strategy 5: Arbitrary bitwidth for intrinsics

- **Target**: Test with non-power-of-2 truncation of `abs`.
- **Rationale**: The patch uses `MinBW = PowerOf2Ceil(BitWidth)` for the cost calculation, but the actual vectorization uses `It->second.first` which is `BitWidth` (not necessarily power of 2). Wait, `computeMinimumValueSizes` sets `MinBWs` to the exact `BitWidth`. Then `vectorizeTree` creates `OpVec = Builder.CreateIntCast(OpVec, VecTy, ...)`. `VecTy` is the vector type with the original element type or the truncated type? In `vectorizeTree`, `VecTy` is the type of the tree entry, which is the truncated type if `MinBWs` is set. The truncated type has element type `IntegerType::get(Ctx, MinBWs[...].first)`. This could be a non-power-of-2 type like `i7`. `llvm.abs.v4i7` is valid in LLVM IR, so this might not crash, but it's worth noting. The main bug is definitely the missing `CallChecker` for `abs`.
- **Expected Issue**: The compiler might crash or produce invalid IR if `MinBW` is not a valid type for the intrinsic, though `FixedVectorType::get(IntegerType::get(...))` handles arbitrary bitwidths. However, the backend might not support `abs` on arbitrary bitwidths.
