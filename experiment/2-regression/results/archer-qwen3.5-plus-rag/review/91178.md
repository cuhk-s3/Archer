## Test Strategies

### Strategy 1: Deep recursive chain with absorber

- **Target**: select.ll absorber tests
- **Rationale**: The fix changes how absorber constants are handled when replacements happen deeper in the instruction chain. The old code assumed one operand would be RepOp, but now with recursive replacement, the absorber check uses the actual absorber constant. Test with deeply nested instruction chains where the absorber pattern appears after multiple levels of replacement.
- **Expected Issue**: Incorrect simplification when absorber constant appears after recursive replacement but the impliesPoison check fails due to the deeper chain structure

### Strategy 2: Vector Op with scalar result (cross-lane)

- **Target**: select.ll vector tests
- **Rationale**: The fix now properly handles the case where Op is a vector but the instruction returns a scalar (cross-lane operation). Previously there was an assertion that both must be vectors. Test cases where a vector comparison result is used in a scalar context through extractelement or similar operations.
- **Expected Issue**: Miscompilation when vector Op is incorrectly allowed to simplify through scalar-returning instructions that should be blocked as cross-lane operations

### Strategy 3: x^x pattern with indirect replacement

- **Target**: select.ll xor recursive tests
- **Rationale**: The fix changes the x^x and x-x pattern matching from checking NewOps[0] == NewOps[1] with an assertion that they equal RepOp, to directly checking both operands equal RepOp. This handles cases where the replacement happens higher up the chain. Test with patterns where both operands simplify to RepOp through different paths.
- **Expected Issue**: Incorrect constant folding to zero when operands simplify to the same value but not necessarily RepOp, potentially losing poison semantics

### Strategy 4: No replacement made but recursion continues

- **Target**: select.ll recursive tests
- **Rationale**: The fix adds an AnyReplaced flag and returns nullptr if no replacement was made. This prevents unnecessary simplification attempts. Test cases where recursive calls return without making any actual replacements but the original code would have continued.
- **Expected Issue**: Infinite recursion or incorrect simplification when no actual replacement occurs but the function continues processing

### Strategy 5: Poison propagation through recursive replacement

- **Target**: select.ll poison tests with impliesPoison
- **Rationale**: The absorber case relies on impliesPoison(BO, Op) to ensure poison doesn't leak. With recursive replacement, the relationship between BO and Op may be more complex. Test cases where poison could leak through the recursive chain despite the impliesPoison check passing at the top level.
- **Expected Issue**: Poison value incorrectly eliminated when the impliesPoison check doesn't account for all paths through the recursive replacement chain
