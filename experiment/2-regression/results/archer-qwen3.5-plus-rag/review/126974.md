## Test Strategies

### Strategy 1: Both predicates with samesign but different signedness

- **Target**: isImpliedCondCommonOperandWithCR with both LPred and RPred having samesign but different base predicates
- **Rationale**: The fix only handles the XOR case (exactly one has samesign). When both have samesign but with different signedness (e.g., samesign ult vs samesign slt), the code doesn't flip them. This could miss valid implications or incorrectly report implications.
- **Expected Issue**: The function may return std::nullopt when it should return true/false, or vice versa, when both predicates have samesign but different signedness predicates

### Strategy 2: Constant range computation ignores samesign

- **Target**: computeConstantRange call in isImpliedCondICmps that passes ICmpInst::isSigned(LPred)
- **Rationale**: The computeConstantRange function is called with ICmpInst::isSigned() which only checks the predicate type, not the samesign attribute. For samesign ult, the constant range should be computed as if it were slt (signed), but the code passes false (unsigned). This could lead to incorrect constant ranges being used in the implication check.
- **Expected Issue**: Incorrect constant range computation for samesign predicates leads to wrong implication results. For example, samesign ult with a negative constant might be computed with unsigned range semantics instead of signed.

### Strategy 3: Equality predicates with samesign flipping

- **Target**: getFlippedSignednessPredicate for EQ/NE predicates
- **Rationale**: EQ and NE predicates return themselves when getFlippedSignednessPredicate is called. If one predicate is samesign eq and the other is ne, the XOR condition triggers but flipping eq->eq doesn't change anything. This could lead to infinite loops or incorrect results in edge cases.
- **Expected Issue**: When one predicate is samesign eq/ne and the other is a different predicate, the flip operation doesn't change eq/ne, potentially causing incorrect implication results

### Strategy 4: Mixed samesign with boundary values

- **Target**: Implication checks involving samesign predicates with boundary constant values (INT_MIN, INT_MAX, 0, -1)
- **Rationale**: The samesign attribute affects how comparisons work at boundary values. For example, samesign ult with INT_MIN behaves differently than regular ult. The fix might not correctly handle implications involving these boundary cases when flipping signedness.
- **Expected Issue**: Incorrect implication results when samesign predicates involve boundary values like INT_MIN, INT_MAX, 0, or -1, especially when the signedness flip changes the interpretation of these values

### Strategy 5: Samesign ult implying sge with overlapping ranges

- **Target**: Test case where samesign ult should imply sge (or vice versa) through the signedness flip mechanism
- **Rationale**: The fix enables samesign ult to be treated as slt for implication purposes. A test where samesign ult [0, 10) should imply sge [0, 5) would verify the fix works correctly. Conversely, testing cases where the implication should fail would verify no false positives occur.
- **Expected Issue**: The implication check might incorrectly return true when it should return false (or std::nullopt), or miss valid implications, when samesign predicates are involved with specific constant ranges

### Bug #1

**Analysis:**
This is a REAL BUG!

**Bug Triggering Analysis:**
When %a = 5:
- Original: sgt(5, -10) = true, so we branch to taken. Then samesign uge(5, -20) = poison because 5 is positive and -20 is negative (different signs). The select returns poison, and returning poison triggers UB.
- Transformed: The optimizer incorrectly simplifies the code to just return 30, ignoring that samesign uge(5, -20) would be poison.

**Fix Weakness Analysis:**
The fix in isImpliedCondCommonOperandWithCR is supposed to handle the case where exactly one predicate has samesign. In this case:
- LHS: sgt (no samesign)
- RHS: samesign uge (has samesign)

The XOR condition is true, so the code flips the signedness. However, the issue is that the optimization assumes sgt implies samesign uge, but this is only valid when the values actually have the same sign. The samesign attribute adds an additional constraint that the operands must have the same sign, otherwise the result is poison.

The fix incorrectly treats samesign uge as equivalent to sge for implication purposes without properly accounting for the poison semantics when signs differ. The constant range computation doesn't capture the "same sign" constraint, leading to incorrect simplification.

This reveals that the fix doesn't properly handle the poison semantics of samesign predicates - it only considers the comparison semantics when signs match, but ignores that samesign produces poison when signs don't match.

### Analysis

**Bug Summary:**

The patch "VT: teach implied-cond-cr about samesign (#122447)" introduces a bug in the `isImpliedCondCommonOperandWithCR` function in `llvm/lib/Analysis/ValueTracking.cpp`. The fix attempts to handle samesign predicates by flipping signedness when exactly one predicate has the samesign attribute, but it fails to account for the **poison semantics** of samesign predicates.

**Bug Triggering Test Case:**
```llvm
define i32 @sgt_implies_ge_dominating_cr(i32 %a, i32 %len) {
entry:
  %a.sgt.minus.10 = icmp sgt i32 %a, -10
  br i1 %a.sgt.minus.10, label %taken, label %end

taken:
  %a.ge.minus.20 = icmp samesign uge i32 %a, -20
  %res = select i1 %a.ge.minus.20, i32 30, i32 0
  ret i32 %res

end:
  ret i32 -1
}
```

When called with `%a = 5` (or any positive value):
1. `sgt(5, -10)` = true, branches to `taken`
2. `samesign uge(5, -20)` = **poison** (5 is positive, -20 is negative - different signs)
3. Original code returns poison → UB
4. Optimized code incorrectly simplifies to `ret i32 30` → no UB

**Root Cause:**

The fix in `isImpliedCondCommonOperandWithCR` treats `samesign uge` as equivalent to `sge` for implication purposes by flipping signedness. However, this is incorrect because:

1. **Samesign has a guard condition**: `samesign pred` produces poison if operands have different signs, regardless of the comparison result.

2. **Constant range analysis is insufficient**: The `computeConstantRange` function computes value ranges but doesn't capture the "same sign" constraint. When checking if `sgt %a, -10` implies `samesign uge %a, -20`, the analysis correctly determines that values > -10 satisfy >= -20, but misses that samesign uge additionally requires %a to be negative.

3. **Incorrect implication**: The transformation `sgt X, C1` → `samesign uge X, C2` is only valid when X is guaranteed to have the same sign as C2. Without this guarantee, the transformation is unsound.

**Fix Recommendation:**

The `isImpliedCondCommonOperandWithCR` function should not enable implications involving samesign predicates unless it can prove that the operands will have the same sign. This requires either:
1. Checking that the constant range of the common operand is entirely non-negative or entirely negative, OR
2. Not performing the samesign flip optimization when the sign constraint cannot be verified.
