## Test Strategies

### Strategy 1: Integer overflow with nsw flag

- **Target**: Test case with integer mul/add reassociation where original operations have nsw flags
- **Rationale**: The patch creates new mul instructions without preserving nsw/nuw flags. If the original mul operations had nsw (no signed wrap) flags, they would produce poison on signed overflow. The new mul created in the preheader doesn't have nsw, so it will silently wrap, changing program semantics when overflow occurs.
- **Expected Issue**: The transformed code will produce different results (wrapped values instead of poison) when signed overflow occurs in the hoisted multiplication, potentially leading to incorrect optimizations downstream that assume no overflow.

### Strategy 2: Integer overflow with nuw flag

- **Target**: Test case with integer mul/add reassociation where original operations have nuw flags
- **Rationale**: Similar to nsw, if the original mul operations had nuw (no unsigned wrap) flags, they would produce poison on unsigned overflow. The new mul created doesn't preserve nuw flags, changing semantics.
- **Expected Issue**: The transformed code will produce wrapped values instead of poison on unsigned overflow, potentially causing incorrect behavior.

### Strategy 3: Mixed nsw/nuw flags in expression tree

- **Target**: Test case where different mul operations in the expression tree have different overflow flags
- **Rationale**: The transformation processes multiple mul operations in the expression tree. If some have nsw/nuw and others don't, the transformation should handle this correctly. The current code doesn't check or preserve any flags.
- **Expected Issue**: Inconsistent handling of overflow semantics across the expression tree, potentially producing incorrect results or missing poison propagation.

### Strategy 4: Vector integer types with overflow flags

- **Target**: Test case using vector integer types (e.g., <4 x i32>) with nsw/nuw flags
- **Rationale**: The patch checks for IntOrIntVectorTy, so it handles vector types. However, the same flag preservation issue applies to vectors. Each element of the vector could have different overflow behavior.
- **Expected Issue**: Vector operations with overflow flags will lose those flags after transformation, changing semantics for vector code.

### Strategy 5: Deep expression tree exceeding limit boundary

- **Target**: Test case with expression tree size at the boundary of IntAssociationUpperLimit
- **Rationale**: The patch uses a separate limit (IntAssociationUpperLimit, default 5) for integer operations. Testing at the boundary (exactly 5, exactly 6 changes) can reveal off-by-one errors or incorrect limit checking.
- **Expected Issue**: The transformation might incorrectly apply or reject transformations at the limit boundary, or the limit check might not correctly count the number of changes.

## Bugs Found

### Bug #1

**Analysis:**
This is a REAL BUG! The differential test found a clear difference:

- **Original code**: With d1=INT64_MAX (9223372036854775807), delta=2, cell=1:
  - `%mul.1 = mul nsw i64 1, INT64_MAX` = INT64_MAX (no overflow)
  - `%result = mul nsw i64 INT64_MAX, 2` = **poison** (signed overflow: INT64_MAX * 2 exceeds i64 range)
  - `store poison` triggers UB: "store poison value is UB"

- **Transformed code**:
  - Preheader: `%factor.op.mul = mul i64 INT64_MAX, 2` = -2 (wrapped, NO nsw flag!)
  - Loop: `%mul.1 = mul nsw i64 1, -2` = -2 (no overflow, valid result)
  - `store -2` succeeds without UB

The bug is that the transformation creates a new `mul` instruction in the preheader **without preserving the nsw flag** from the original operations. This changes the semantics:
- Original: overflow produces poison → UB
- Transformed: overflow silently wraps → different result

This is exactly the issue identified in Phase 1: the patch uses `CreateMul` without preserving nsw/nuw flags, while for FP it correctly uses `CreateFMulFMF` to preserve fast-math flags.

The fix should preserve nsw/nuw flags when creating the new mul instruction for integer types, similar to how `CreateFMulFMF` preserves fast-math flags for FP.

## Bug Summary: LICM Integer Overflow Flag Preservation

### Bug Triggering Analysis

The bug is in the LICM `hoistMulAddAssociation` transformation which was added to support integer mul/add operations (commit 2dd52046816ac706a25d588efac77705793d8778). The transformation fails to preserve `nsw` (No Signed Wrap) and `nuw` (No Unsigned Wrap) flags when hoisting integer multiplications to the loop preheader.

**Test Case:**
- Pattern: `(cell * d1) * delta` with `nsw` flags
- Input: `d1 = INT64_MAX`, `delta = 2`, `cell = 1`

**Original Code:**
```
%mul.1 = mul nsw i64 1, INT64_MAX    ; = INT64_MAX (no overflow)
%result = mul nsw i64 INT64_MAX, 2   ; = poison (signed overflow!)
store i64 poison                     ; UB triggered
```

**Transformed Code:**
```
; Preheader (BUG: no nsw flag!)
%factor.op.mul = mul i64 INT64_MAX, 2  ; = -2 (silent wrap)

; Loop
%mul.1 = mul nsw i64 1, -2  ; = -2 (no overflow)
store i64 -2                ; No UB, wrong result!
```

### Fix Weakness Analysis

**Root Cause in `llvm/lib/Transforms/Scalar/LICM.cpp` (lines 2746-2749):**
```cpp
Value *Mul;
if (I.getType()->isIntOrIntVectorTy())
  Mul = Builder.CreateMul(U->get(), Factor, "factor.op.mul");  // BUG!
else
  Mul = Builder.CreateFMulFMF(U->get(), Factor, Ins, "factor.op.fmul");  // Correct
```

The code correctly uses `CreateFMulFMF` for floating-point (preserves fast-math flags), but uses `CreateMul` without flag parameters for integers, losing `nsw`/`nuw` semantics.

**Required Fix:**
Preserve `nsw`/`nuw` flags from the original multiplication operations when creating the preheader multiplication, similar to how `CreateFMulFMF` preserves fast-math flags.
