- **Rationale**: The optimization assumes x > y implies x - y > 0 when there's no overflow (nsw). With samesign, the icmp result could be poison even when the subtraction wouldn't overflow. Test cases where samesign causes poison but nsw sub would be well-defined.

### Strategy 4: Test mixed samesign predicates in implication chain

- **Target**: Chained implications with mixed samesign and non-samesign predicates

### Strategy 5: Test boundary values with samesign SGT/SLT implications

## Bugs Found

### Bug #1

**Analysis:**
This is a real bug. The original IR has undefined behavior because:
1. `icmp samesign ugt i32 2147483647, -1` produces poison (operands have different signs)
2. `sub nsw i32 2147483647, -1` also produces poison (signed overflow: INT_MAX - (-1) = INT_MAX + 1 = INT_MIN which overflows)

The original correctly triggers UB when executed. However, the optimized version simplifies to `ret i1 true`, which returns a concrete value instead of propagating the poison/UB. This is incorrect because the optimization is removing the UB behavior - the transformed code returns a well-defined value (true) when the original should have UB.

The fix incorrectly assumes that when LHS is samesign ugt and implies RHS is false, the result can be simplified. But when samesign produces poison, the implication logic breaks down. The optimization should not simplify away the poison behavior.

**Original LLVM IR:**
```llvm

; Test boundary explicit - x = INT_MAX, y = -1 (different signs, samesign ugt is poison)
define i1 @test_boundary_explicit() {
entry:
  %lhs = icmp samesign ugt i32 2147483647, -1
  %sub = sub nsw i32 2147483647, -1
  %rhs = icmp slt i32 %sub, 0
  %result = xor i1 %lhs, %rhs
  ret i1 %result
}

define i1 @main(i32 %argc, ptr %argv) {
entry:
  %r = call i1 @test_boundary_explicit()
  ret i1 %r
}
```

### Analysis

### Bug Triggering Analysis

The bug is triggered when:
1. An `icmp samesign ugt` instruction has operands with different signs (e.g., INT_MAX and -1)
2. This produces a poison value because the samesign flag requires both operands to have the same sign
3. The optimization in `isImpliedCondICmps` uses `CmpPredicate::getMatching()` to match `samesign ugt` with `sgt`
4. The optimization then applies the rule that "x >s y implies (x -nsw y) <s 0 is false"
5. This causes the code to simplify `xor i1 %lhs, %rhs` to a constant `true`, removing the poison behavior

The original IR correctly has UB (poison value returned), but the optimized IR returns a concrete value `true`, which is incorrect because it removes the undefined behavior.

### Fix Weakness Analysis

The fix in commit 66badf224ade6e78d5da005f6a9819092fd8767b changed the predicate matching from direct equality (`LPred == ICMP_SGT`) to using `CmpPredicate::getMatching()`. This allows samesign predicates to match their signed counterparts.

The weakness is that the optimization doesn't properly account for the poison semantics of samesign. When operands have different signs:
- `icmp samesign ugt x, y` produces poison
- The mathematical relationship "x > y implies x - y > 0" assumes well-defined comparison results
- But with poison, this implication chain is broken

The fix should either:
1. Check that both operands actually have the same sign before applying the optimization
2. Or ensure that the optimization preserves poison semantics when samesign is involved

The test case with constant operands (INT_MAX, -1) clearly demonstrates this: the samesign ugt is poison, but the optimization simplifies it away as if the comparison was well-defined.
