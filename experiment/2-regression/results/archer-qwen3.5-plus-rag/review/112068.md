## Test Strategies

### Strategy 1: Zero Input Poison Propagation

- **Target**: Test ctlz/cttz with zero input used as shift amount
- **Rationale**: When input is zero, ctlz/cttz with is_zero_poison=false returns bitwidth, and shift by bitwidth produces poison. With is_zero_poison=true, ctlz/cttz returns poison directly, and shift by poison also produces poison. This test verifies the transformation preserves poison semantics.
- **Expected Issue**: If the transformation is incorrect, the program might produce a defined value instead of poison/UB when input is zero, or vice versa.

### Strategy 2: Vector Mixed Zero/Non-Zero Elements

- **Target**: Test ctlz/cttz on vector types with some zero and some non-zero elements
- **Rationale**: For vectors, each element is processed independently. When some elements are zero and others are non-zero, the transformation must correctly handle poison propagation for zero elements while preserving correct results for non-zero elements.
- **Expected Issue**: If the transformation incorrectly handles vectors, non-zero elements might produce wrong results, or zero elements might not produce poison as expected.

### Strategy 3: All Shift Types Coverage

- **Target**: Test the transformation with shl, lshr, and ashr shift operations
- **Rationale**: The m_Shift pattern matches all shift types (shl, lshr, ashr). All shifts produce poison when shift amount >= bitwidth. The transformation should work correctly for all shift types.
- **Expected Issue**: If the transformation has a bug specific to certain shift types, some shifts might produce incorrect results or fail to produce poison when expected.

### Strategy 4: Multiple Uses Prevention

- **Target**: Test ctlz/cttz with multiple uses where only one is a shift
- **Rationale**: The transformation requires hasOneUse(). If ctlz/cttz has multiple uses (e.g., one shift and one other use), the transformation should not apply because the other use might depend on the specific value when input is zero.
- **Expected Issue**: If the transformation incorrectly applies when there are multiple uses, the non-shift use might get poison instead of the expected bitwidth value when input is zero.

### Strategy 5: Shift Operand Position Verification

- **Target**: Test ctlz/cttz used as the value being shifted (operand 0) vs shift amount (operand 1)
- **Rationale**: The m_Shift(m_Value(), m_Specific(&II)) pattern specifically matches when ctlz/cttz is the second operand (shift amount). If ctlz/cttz is the first operand (value being shifted), the transformation should not apply because the semantics would change (shift by 1 of bitwidth vs shift by 1 of poison).
- **Expected Issue**: If the pattern incorrectly matches when ctlz/cttz is the value being shifted, the transformation would incorrectly change semantics: shift(bitwidth, 1) = defined value vs shift(poison, 1) = poison.

### Analysis

### Bug Triggering Analysis
The patch correctly implements the optimization to set `is_zero_poison=true` for `ctlz`/`cttz` intrinsics when they are only used as shift amounts. The transformation is semantically correct because:

1. When input is zero:
   - Before: `cttz(0, false)` returns bitwidth (e.g., 32), and `shift(x, 32)` produces poison (shift amount >= bitwidth)
   - After: `cttz(0, true)` returns poison, and `shift(x, poison)` produces poison
   - Both produce poison, so the transformation is correct

2. When input is non-zero:
   - `cttz(x, false)` = `cttz(x, true)` for x != 0
   - Both produce the same result

3. The `hasOneUse()` check ensures the transformation only applies when cttz/ctlz has exactly one use (the shift), preventing incorrect transformations when there are multiple uses.

4. The `m_Shift(m_Value(), m_Specific(&II))` pattern ensures cttz/ctlz is used as the shift amount (operand 1), not the value being shifted (operand 0).

- **Shift Operand Position Verification**: Tested with cttz as LHS (value being shifted) - transformation correctly does NOT apply
