## Test Strategies

### Strategy 1: CondContext-Loss-Test

- **Target**: ComputeNumSignBits calls in SimplifyDemandedUseBits
- **Rationale**: The patch passes SimplifyQuery to SimplifyDemandedBits but ComputeNumSignBits calls only use Q.CxtI, losing CondContext, DomConditionCache, AssumptionCache, and DominatorTree information. This could cause missed optimizations when CondContext provides additional constraints.
- **Expected Issue**: When CondContext provides sign bit information through conditional constraints, ComputeNumSignBits will miss this information because it only receives CxtI instead of the full SimplifyQuery, potentially leading to suboptimal code generation or missed simplification opportunities.

### Strategy 2: OneUseChain-Context-Preservation

- **Target**: One-use chain handling in SimplifyDemandedUseBits
- **Rationale**: The commit message mentions 'retaining the original context instruction for one-use chains'. The patch changes from passing CxtI to passing Q, but ComputeNumSignBits calls still only extract Q.CxtI. For one-use chains where the context instruction matters for sign bit analysis, this could cause incorrect results.
- **Expected Issue**: In one-use chains where the context instruction provides critical sign bit information through assumptions or dominator conditions, the analysis may fail to propagate this information correctly, leading to missed optimizations.

### Strategy 3: DomConditionCache-Loss

- **Target**: DC field usage in ComputeNumSignBits
- **Rationale**: The SimplifyQuery has a DC (DomConditionCache) field that can provide additional information about dominating conditions. When ComputeNumSignBits is called with only Q.CxtI, the DC information is lost, potentially missing sign bit constraints from dominating comparisons.
- **Expected Issue**: When a dominating condition (e.g., icmp slt %x, 0) establishes sign bit information, ComputeNumSignBits will miss this because DC is not passed, leading to conservative (incorrect) sign bit counts.

### Strategy 4: AssumptionCache-Loss

- **Target**: AC field usage in ComputeNumSignBits
- **Rationale**: The SimplifyQuery has an AC (AssumptionCache) field that can provide sign bit information from assume intrinsics. When ComputeNumSignBits is called with only Q.CxtI, the AC information is lost.
- **Expected Issue**: When assume intrinsics or range metadata provide sign bit constraints, ComputeNumSignBits will miss this information, potentially leading to missed optimizations in code that uses assume or llvm.assume-like intrinsics.

### Strategy 5: Vector-SignBits-DemandedElts

- **Target**: Vector ComputeNumSignBits with DemandedElts
- **Rationale**: The internal ComputeNumSignBits function takes DemandedElts parameter for vectors, but the calls in InstCombineSimplifyDemanded.cpp use the public API that doesn't pass DemandedElts. For vector operations with partial element demands, this could lead to incorrect sign bit analysis.
- **Expected Issue**: For vector shift operations where only some elements are demanded, ComputeNumSignBits may return overly conservative results because it analyzes all elements instead of just the demanded ones, potentially blocking valid optimizations.

### Analysis

### Bug Analysis Summary

After thorough analysis of the patch "[InstCombine] Pass SimplifyQuery to SimplifyDemandedBits()", I found that while the patch correctly passes SimplifyQuery to SimplifyDemandedBits() and computeKnownBits() calls, there is an inconsistency in how ComputeNumSignBits() is called.

**Key Finding:**
The patch changes SimplifyDemandedBits() to accept a full SimplifyQuery parameter, but the three calls to ComputeNumSignBits() in InstCombineSimplifyDemanded.cpp (lines 658, 740, 779) still only pass `Q.CxtI` instead of the full query context. This means:
- CondContext (CC) information is lost
- DomConditionCache (DC) information is lost
- AssumptionCache (AC) information is lost
- DominatorTree (DT) information is lost

**Test Results:**
All verification tests passed (no bugs found). The transformations are correct, but the analysis may be missing optimization opportunities:

1. **AssumptionCache-Loss**: Tested with `test_ashr_signbit_extension` - the assume(x < 0) correctly simplifies `ashr x, 31` to -1, showing AC information IS being used through other paths (likely through computeKnownBits).

2. **DomConditionCache-Loss**: Tested with `test_domcond_signbits` - dominator conditions are handled correctly through other mechanisms.

3. **CondContext-Loss-Test**: Tested with `test_condcontext_shl_ashr` - conditional context is preserved.

4. **OneUseChain-Context-Preservation**: Tested with `test_oneuse_chain` - one-use chains work correctly.

5. **Vector-SignBits-DemandedElts**: Tested with `test_vector_signbits` - vector operations are optimized correctly.

**Conclusion:**
The patch is correct but potentially incomplete. While ComputeNumSignBits() calls only use Q.CxtI, the analysis shows that the existing code paths (through computeKnownBits and other mechanisms) already handle the context information adequately for the tested cases. The patch successfully enables future CondContext usage in SimplifyDemandedBits() as intended.

No actual bugs were found - the transformations are semantically correct. The identified issue is a potential missed optimization opportunity rather than a correctness bug.
