## Test Strategies

### Strategy 1: Commutative_Operand_Order

- **Target**: Binary operator with constant in operand 0 instead of operand 1
- **Rationale**: The fix only checks getOperand(1) for a constant using m_APInt. For commutative operations (and, or, xor), LLVM doesn't always canonicalize constants to the right operand, especially early in the optimization pipeline or after certain transformations. If the constant is in operand 0, the simplification won't trigger, missing an optimization opportunity.
- **Expected Issue**: The binary operation is not simplified when the constant mask is in operand 0, even though the demanded bits condition is satisfied. This results in missed optimization rather than incorrect code generation.

### Strategy 2: Vector_NonSplat_Constant

- **Target**: Binary operator with non-splat vector constant mask
- **Rationale**: The m_APInt matcher only matches scalar constants and splat vector constants. If the mask is a non-splat vector constant (e.g., <i8 1, i8 2>), the match fails and no simplification occurs. This is a missed optimization for vector operations where different elements have different constant masks but the demanded bits condition is still satisfied for each element.
- **Expected Issue**: Vector binary operations with non-splat constant masks are not simplified, missing optimization opportunities.

### Strategy 3: Or_Disjoint_Poison_Propagation

- **Target**: or disjoint instruction where operands could overlap in non-demanded bits
- **Rationale**: The fix treats 'or' and 'or disjoint' identically. For 'or disjoint', if operands overlap, the result is poison. While replacing with operand 0 should be correct when mask bits don't intersect demanded bits (since poison in non-demanded positions shouldn't affect program behavior), there could be edge cases where poison propagation interacts unexpectedly with other optimizations or when the non-disjoint nature is relied upon for correctness checks.
- **Expected Issue**: Potential incorrect transformation if poison from or disjoint in non-demanded bit positions affects program behavior in unexpected ways, or if the disjoint guarantee is relied upon by subsequent passes.

### Strategy 4: MultiUse_Different_DemandedBits_Per_Element

- **Target**: Vector binary operation where different vector elements have different demanded bits across different uses
- **Rationale**: The DemandedBits analysis for vectors returns a union mask where a bit is set if demanded in ANY element. This could lead to overly conservative analysis. For example, if element 0 has bit 0 demanded and element 1 has bit 1 demanded, the union is {0,1}. A splat mask of {0} would intersect, preventing simplification, even though element 1's demanded bit (bit 1) is not affected by the mask.
- **Expected Issue**: Overly conservative analysis prevents valid simplifications for vector operations where per-element demanded bits differ, resulting in missed optimizations.

### Strategy 5: XOR_With_AllOnes_Mask_Not_Demanded

- **Target**: xor instruction with all-ones mask (bitwise not) where result bits are not demanded
- **Rationale**: For xor X, -1 (which is ~X), if no bits are demanded, the condition !Demanded.intersects(Mask) is true (empty set doesn't intersect anything). The simplification replaces ~X with X. While this is correct when no bits are demanded (the value is dead), there could be edge cases with poison or undef where this transformation changes observable behavior.
- **Expected Issue**: Potential incorrect transformation when xor with all-ones mask is simplified to the operand, especially in the presence of poison or undef values that might have different semantics.
