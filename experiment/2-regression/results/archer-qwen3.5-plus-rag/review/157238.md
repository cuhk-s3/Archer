## Test Strategies

### Strategy 1: Test bitcast from integer with known negative sign to detect -Inf

- **Target**: Bitcast from i32/i64 with sign bit known to be 1, exponent all 1s, fraction all 0s (representing -Inf)
- **Rationale**: The patch creates InfKB from +Inf and clears the sign bit in Zero only, but not in One. When Bits represents -Inf (sign bit known to be 1), Bits == InfKB fails because Bits.One has sign=1 but InfKB.One has sign=0. This causes the code to miss detecting -Inf.
- **Expected Issue**: The fpclass analysis fails to recognize -Inf, returning fcAllFlags or incomplete fcInf instead of fcInf or fcNegInf

### Strategy 2: Test bitcast from integer with known negative sign to detect -0

- **Target**: Bitcast from i32/i64 with sign bit known to be 1, all other bits 0 (representing -0)
- **Rationale**: Same bug as -Inf: ZeroKB is created from +0 with sign bit cleared in Zero only. When Bits represents -0, Bits == ZeroKB fails because Bits.One has sign=1 but ZeroKB.One has sign=0.
- **Expected Issue**: The fpclass analysis fails to recognize -0, returning fcAllFlags or incomplete fcZero instead of fcZero or fcNegZero

### Strategy 3: Test bitcast where KnownBits::eq returns true for exact match

- **Target**: Bitcast from integer where Bits is exactly equal to InfKB (all bits known, matching +Inf)
- **Rationale**: The code has 'assert(!InfResult.value())' assuming KnownBits::eq never returns true. But if Bits is exactly +Inf (all bits known), eq returns true, causing assertion failure in debug builds or incorrect knownNot(fcInf) in release.
- **Expected Issue**: Assertion failure in debug builds, or incorrect inference that value is not Inf when it definitely is +Inf

### Strategy 4: Test bitcast with partially known bits that should prove NaN

- **Target**: Bitcast where Bits.One (assuming unknown=0) produces NaN, testing the NaN detection logic
- **Rationale**: The NaN detection uses 'APFloat(Ty->getFltSemantics(), Bits.One).isNaN()' to check if value is always NaN. Need to verify this works correctly when some bits are unknown but the known bits guarantee NaN.
- **Expected Issue**: Incorrect NaN classification - either missing NaN detection or false positive

### Strategy 5: Test bitcast with sign bit known combined with fpclass detection

- **Target**: Bitcast where sign bit is known (e.g., non-negative) and the value is detected as Inf/Zero/NaN
- **Rationale**: The code first sets sign bit info (lines 5935-5938), then overwrites KnownFPClasses with fcInf/fcZero/fcNan (lines 5961/5973/5948). This loses the sign information in KnownFPClasses, though SignBit field is preserved. Need to verify the combination is handled correctly by consumers.
- **Expected Issue**: Inconsistent state where KnownFPClasses says fcInf but SignBit says positive, potentially causing incorrect optimizations that only check KnownFPClasses
