## Test Strategies

### Strategy 1: Test SCEV constant replacement for non-live-in operands in Mul

- **Target**: Binary multiplication where first operand is SCEV-constant but not a live-in (e.g., computed in loop)
- **Rationale**: The fix replaces operands with SCEV-derived constants without checking if the operand is a live-in. The follow-up fix (commit 30f3752e) shows this can be incorrect for non-live-in values, especially those involving header-phi recipes like reductions.
- **Expected Issue**: Incorrect code generation where a loop-varying value that SCEV simplifies to a constant is incorrectly replaced with a live-in constant, breaking the data flow for reduction or recurrence patterns

### Strategy 2: Test SCEV constant replacement for reduction operands

- **Target**: Reduction patterns where one operand of Mul is part of a reduction PHI
- **Rationale**: The follow-up fix specifically mentions that replacing recipes involving header-phi recipes like reductions may not be correct. A reduction PHI's operand might be SCEV-analyzable as constant but replacing it with a live-in breaks the reduction chain.
- **Expected Issue**: Reduction produces incorrect results because the reduction PHI operand is incorrectly replaced with a constant live-in instead of maintaining the reduction chain

### Strategy 3: Test first-order recurrence with SCEV-constant operand

- **Target**: First-order recurrence pattern where an operand is SCEV-determined constant
- **Rationale**: Similar to reductions, first-order recurrences use header-phi recipes. The fix doesn't distinguish between live-ins and values defined by header-phi recipes, potentially breaking recurrence semantics.
- **Expected Issue**: First-order recurrence produces wrong values because SCEV constant replacement breaks the recurrence dependency chain

### Strategy 4: Test binary ops other than Mul with SCEV-constant second operand that is loop-computed

- **Target**: Non-Mul binary operations (e.g., Shl, LShr, And) where second operand is computed in loop but SCEV says it's constant
- **Rationale**: The fix applies SCEV constant replacement to the second operand of all binary ops. If the second operand is computed within the loop (not a true live-in) but SCEV determines it's constant, replacing it with a live-in could be incorrect.
- **Expected Issue**: Incorrect vectorization where a loop-computed value that SCEV simplifies to constant is replaced with a live-in, potentially causing use-before-def or incorrect semantics

### Strategy 5: Test SCEV constant replacement with predicated instructions

- **Target**: Predicated binary operations where operand is SCEV-constant
- **Rationale**: Predicated instructions have special handling in vectorization. Replacing a predicated instruction's operand with a live-in constant via SCEV might not account for the predication mask, leading to incorrect behavior when the instruction should only execute conditionally.
- **Expected Issue**: Predicated binary operation executes with wrong operand value in masked lanes, or the constant replacement ignores the predication semantics

### Analysis

### Bug Triggering Analysis

The fix in commit 6d6eea92e36c301e34a7ec11b2a40e3080f79f53 modifies `VPRecipeBuilder::tryToWiden` to use SCEV to simplify operands of binary operations to constants during VPlan construction. This was done to align VPlan construction with the legacy cost model behavior.

The key change is the `GetConstantViaSCEV` lambda that:
1. Gets the underlying value of a VPValue operand
2. If it's already a constant or not SCEVable, returns the original operand
3. If SCEV determines it's a constant, replaces it with a live-in constant via `Plan.getOrAddLiveIn(C->getValue())`

For `Mul` instructions, both operands are checked. For other binary ops, only the second operand is checked.

### Fix Weakness Analysis

The fix has a critical weakness: it doesn't check if the operand is actually a live-in before replacing it with a live-in constant. The follow-up fix (commit 30f3752e) adds `if (!Op->isLiveIn()) return Op;` to address this issue.

The problem is that SCEV can determine a value is constant even if it's computed within the loop (not a true live-in). For example:
- `%ext.0 = sext i8 0 to i32` - computed in entry but SCEV knows it's 0
- Values computed in loop body that SCEV simplifies to constants

The fix would incorrectly replace such operands with live-in constants, which could break:
1. Reduction patterns where the operand is part of a reduction PHI
2. First-order recurrences with header-phi recipes
3. Non-live-in values that SCEV determines are constant

### Test Coverage

The test case `second_lshr_operand_zero_via_scev` covers the strategy "Test binary ops other than Mul with SCEV-constant second operand that is loop-computed". The `%ext.0` value is computed via `sext i8 0 to i32` in the entry block, and SCEV determines it's constant 0. The `lshr` instruction uses this as the second operand.

The verification and differential testing show that the transformation is correct - the fix properly replaces the SCEV-determined constant with a live-in, and the vectorized code produces the same results as the original.

However, the follow-up fix (commit 30f3752e) shows that the original fix had issues with non-live-in operands, particularly for reductions and recurrences. The test cases I created for those strategies didn't trigger vectorization, so the SCEV replacement didn't occur, but they demonstrate the scenarios where the fix could have issues.

### Summary

The fix correctly addresses the cost model discrepancy by using SCEV to identify constant operands. However, it has a weakness: it doesn't verify that operands are actually live-ins before replacing them. The follow-up fix adds this check. The provided test case demonstrates correct behavior for a valid scenario (loop-invariant constant computed via sext), while the analysis identifies potential issues with reductions, recurrences, and non-live-in SCEV constants that the follow-up fix addresses.
