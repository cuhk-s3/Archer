## Test Strategies

### Strategy 1: Poison Operand Test

- **Target**: ucmp/scmp with poison operands
- **Rationale**: The patch does not check if operands of ucmp/scmp could be poison. If operands are poison, the result is poison, and replacing with a constant would be incorrect. Similar to the min/max poison issue fixed in commit 3d91d96.
- **Expected Issue**: Incorrect constant folding when ucmp/scmp operands may be poison, leading to miscompilation by removing poison semantics

### Strategy 2: Vector Type Test

- **Target**: ucmp/scmp with vector types
- **Rationale**: The langref states ucmp/scmp support vector types, but the test file only tests scalar types. The constant creation uses ConstantInt::get which should handle vectors, but this is untested.
- **Expected Issue**: Potential incorrect handling of vector ucmp/scmp, either failing to fold when possible or incorrectly folding element-wise comparisons

### Strategy 3: Signedness Mismatch Edge Case

- **Target**: ucmp/scmp with cross-signedness constraints
- **Rationale**: The constraint system maintains separate signed and unsigned systems. The patch relies on checkCondition to handle signedness correctly via getGTPredicate/getLTPredicate. Edge cases where signed and unsigned comparisons differ need testing.
- **Expected Issue**: Incorrect folding when signed and unsigned orderings differ (e.g., negative signed values are large unsigned values)

### Strategy 4: Return Type Boundary Test

- **Target**: ucmp/scmp with minimal i2 return type
- **Rationale**: The langref requires return type to be at least i2. Testing with exactly i2 return type ensures the constant creation (1, -1, 0) works correctly for the minimal valid type.
- **Expected Issue**: Potential issues with constant creation for minimal i2 return type, especially for -1 which requires proper sign extension

### Strategy 5: Dominance Boundary Test

- **Target**: ucmp/scmp where constraint dominance is borderline
- **Rationale**: The checkCondition function checks dominance via NumIn/NumOut. The patch doesn't add special handling for dominance boundaries. Testing cases where the constraint is at the edge of dominance scope could reveal issues.
- **Expected Issue**: Incorrect folding when the establishing constraint is at the dominance boundary or in a different control flow path

### Bug #2

**Analysis:**
This is a real bug. The test case demonstrates incorrect folding of scmp intrinsic when the establishing constraint uses unsigned comparison but scmp uses signed comparison.

Bug Triggering Analysis:
- Input: %x = -1 (0xFFFFFFFF), %y = 1
- Constraint: icmp ugt -1, 1 evaluates to true (unsigned: 4294967295 > 1)
- scmp(-1, 1) should return -1 because signed comparison: -1 < 1
- But the optimizer incorrectly folds scmp to 0 (equality case)

The bug is in the checkAndReplaceCmp function which uses getGTPredicate/getLTPredicate to get the correct signed/unsigned predicate, but the constraint system may have facts from the wrong signedness system. When the constraint is established via unsigned comparison (icmp ugt), the fact is added to the unsigned system. But when checking scmp (signed), the code queries using signed predicates (ICMP_SGT, ICMP_SLT), which may not find the fact or may find incorrect implications.

Actually, looking at the alive2 output more carefully, it folded to 0 (equality), not 1. This suggests the EQ check matched incorrectly. The issue is that checkCondition(ICMP_EQ, LHS, RHS, I, Info) returned true when it shouldn't have.

Fix Weakness Analysis:
The patch assumes that the constraint system correctly handles signedness separation, but there's a gap: when checking conditions for scmp/ucmp, the code uses the appropriate signed/unsigned predicates via getGTPredicate/getLTPredicate, but the EQ check always uses ICMP_EQ regardless of the intrinsic's signedness. The constraint system maintains separate signed and unsigned systems, and facts from one shouldn't imply results in the other without proper conversion.

### Bug #4

**Analysis:**
This is another instance of the same signedness mismatch bug. The test case demonstrates incorrect folding of scmp intrinsic when the establishing constraint uses unsigned comparison but scmp uses signed comparison.

Bug Triggering Analysis:
- Input: %x = 0, %y = 1
- Constraint: icmp uge 0, 1 evaluates to false (unsigned: 0 is not >= 1)
- In the false branch, we know x < y (unsigned)
- scmp(0, 1) signed comparison: 0 < 1, so scmp should return -1
- But the optimizer incorrectly folds scmp to 0 (equality case)

The bug is the same as previously identified: the EQ check in checkAndReplaceCmp uses ICMP_EQ regardless of signedness, and the constraint system may incorrectly conclude equality from unsigned constraints when checking signed intrinsics.

Fix Weakness Analysis:
The patch's checkAndReplaceCmp function has three checks:
1. GT predicate (correctly uses getGTPredicate for signed/unsigned)
2. LT predicate (correctly uses getLTPredicate for signed/unsigned)
3. EQ predicate (always uses ICMP_EQ)

The issue is that the EQ check doesn't account for signedness separation in the constraint system. When an unsigned constraint establishes x != y (from uge being false, we know x < y unsigned, which implies x != y), the signed EQ check may incorrectly match.

Actually, looking more carefully: when uge is false, we know x < y (unsigned). This doesn't imply x == y, so the EQ check shouldn't match. The bug must be in how the constraint system handles the negation of uge.

When icmp uge x, y is false, we have x < y (unsigned). The constraint system adds this fact to the unsigned system. But when checking ICMP_EQ for scmp (signed), it queries the signed system, which may not have the correct information.

The root cause is that the constraint system maintains separate signed and unsigned systems, but the EQ check doesn't properly query both systems or convert between them.
