## Test Strategies

### Strategy 1: Nested is_constant intrinsic

- **Target**: Test case where llvm.is.constant appears as an operand within a larger expression that gets substituted
- **Rationale**: The fix only checks if the top-level instruction is is_constant, but doesn't prevent folding when is_constant appears deeper in the operand tree. This could cause incorrect simplification when the equality assumption incorrectly affects the is_constant check.
- **Expected Issue**: The optimizer may incorrectly simplify expressions containing nested is_constant checks, potentially changing program semantics when a value assumed constant under equality is not actually constant.

### Strategy 2: Vector is_constant with lane-specific behavior

- **Target**: Test case with vector types where is_constant behaves differently per lane
- **Rationale**: The fix has special handling for vector types (lines 4398-4404) but the is_constant check doesn't account for vector-specific behaviors. Some lanes might be constant while others are not.
- **Expected Issue**: Incorrect simplification of vector operations where is_constant results vary across lanes, leading to wrong results for some vector elements.

### Strategy 3: Poison value interaction with equality assumption

- **Target**: Test case where the substituted value could introduce poison under the equality assumption
- **Rationale**: The fix mentions avoiding poison issues by not calling simplification recursively, but the interaction between equality assumptions and poison values in complex expressions needs testing. If Op1 contains operations that could produce poison when A is replaced with B, the simplification might be incorrect.
- **Expected Issue**: The optimizer might incorrectly simplify expressions where the equality assumption (a==b) hides potential poison values that would be exposed after substitution.

### Strategy 4: Wrapping is_constant with other instructions

- **Target**: Test case where is_constant result is used through a bitcast, zext, or other wrapping instruction
- **Rationale**: The match pattern m_Intrinsic<Intrinsic::is_constant>() only matches direct is_constant instructions. If the result is wrapped in another instruction (like a bitcast or zero-ext), the check won't trigger and the folding might incorrectly proceed.
- **Expected Issue**: Expressions like 'zext(llvm.is.constant(x))' or 'bitcast(llvm.is.constant(x))' might be incorrectly simplified when they should be preserved.

### Strategy 5: Self-referential simplification with is_constant

- **Target**: Test case where the PreventSelfSimplify guard interacts with is_constant in a cycle
- **Rationale**: The AllowRefinement=true path uses PreventSelfSimplify to avoid cycles, but this might not properly handle cases where is_constant is involved in a self-referential pattern. The comment at lines 4473-4481 describes a specific cycle case but doesn't mention is_constant.
- **Expected Issue**: Infinite recursion or incorrect simplification when is_constant is part of a self-referential pattern that the PreventSelfSimplify guard doesn't properly detect.
