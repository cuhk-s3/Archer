## Test Strategies

### Strategy 1: FCmp_Cond_Bug

- **Target**: select with floating-point comparison condition
- **Rationale**: Line 9675 in findValuesAffectedByCondition uses 'match(Cond, m_FCmp(...))' instead of 'match(V, m_FCmp(...))'. This means when processing FCmp conditions in the worklist, the code will never match because it's checking the original Cond instead of the current worklist item V. This causes affected values from FCmp conditions to not be tracked, leading to missed optimization opportunities or incorrect KnownBits computation.
- **Expected Issue**: Select with FCmp condition where the arms should be simplifiable based on the condition will not be optimized, or worse, incorrect constant folding may occur if other analysis paths provide incomplete information.

### Strategy 2: Poison_Condition_Test

- **Target**: select with poison/undef condition
- **Rationale**: The patch doesn't explicitly handle poison or undef in the select condition. When the condition is poison/undef, the KnownBits analysis might incorrectly assume certain bits are known when they're actually undefined. The computeKnownBitsFromCond function doesn't check if the condition itself could be poison.
- **Expected Issue**: Incorrect constant folding when the select condition contains poison/undef values, potentially changing program semantics by replacing non-constant expressions with constants when the result should remain non-deterministic.

### Strategy 3: Vector_Condition_Elementwise

- **Target**: select with vector condition and vector arms
- **Rationale**: The code checks SelType->isIntOrIntVectorTy() but the KnownBits analysis from conditions may not properly handle element-wise semantics for vector conditions. When condition is a vector, different elements may have different known bits implications for corresponding elements in the arms.
- **Expected Issue**: Incorrect KnownBits computation for vector selects where per-element condition information should be tracked separately but is instead merged incorrectly.

### Strategy 4: Depth_Zero_Affected_Value

- **Target**: select arm that is directly in AffectedValues set
- **Rationale**: The hasAffectedValue function has 'if (Depth != 0 && Affected.contains(V))' which means at Depth=0 (the initial call), even if the select arm itself is in AffectedValues, it won't be detected. The function only returns true for affected values when Depth > 0.
- **Expected Issue**: Missed optimization opportunities when the select arm itself (not just its operands) is directly affected by the condition, because hasAffectedValue returns false at depth 0.

### Strategy 5: LogicalOp_Condition_Invert

- **Target**: select with logical and/or condition where invert flag matters
- **Rationale**: The computeKnownBitsFromCond function handles logical operations with invert logic, but the interaction between CC.Invert flag and logical operations in the condition may not be correctly handled. When CC.Invert is true (for the false arm), the union/intersection logic in computeKnownBitsFromCond may produce incorrect results.
- **Expected Issue**: Incorrect KnownBits for the false arm of select when condition contains logical operations, because the invert handling in computeKnownBitsFromCond may not correctly propagate through nested logical operations.
