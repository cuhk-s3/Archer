## Test Strategies

### Strategy 1: Vector with mixed poison and non-matching elements

- **Target**: Test m_AllOnes/m_Zero matching with vectors containing poison elements alongside non-matching constant values
- **Rationale**: The fix changes cstval_pred_ty to skip poison elements but still require all non-poison elements to match the predicate. A vector like <poison, 0, 1> should NOT match m_AllOnes or m_Zero since it has non-poison elements that don't match. Need to verify the matcher correctly rejects such cases.
- **Expected Issue**: Incorrect match when vector has poison elements mixed with non-matching constant values, leading to wrong optimization

### Strategy 2: All-poison vector matching

- **Target**: Test m_AllOnes/m_Zero matching with vectors where ALL elements are poison
- **Rationale**: The fix requires HasNonPoisonElements to be true for a match. A vector with all poison elements should NOT match m_AllOnes or m_Zero because there are no non-poison elements to verify. This edge case needs testing.
- **Expected Issue**: All-poison vector incorrectly matching m_AllOnes or m_Zero, causing incorrect constant folding

### Strategy 3: Poison propagation in Not pattern with vector select

- **Target**: Test xor with all-ones pattern in select contexts where poison elements in the -1 constant could affect result
- **Rationale**: Similar to the follow-up fix in commit 7339f7ba, using m_Not (which allows poison) instead of a forbid-poison variant in select folds can cause incorrect poison propagation. The false value may be less poisonous than m_Not result.
- **Expected Issue**: Select instruction incorrectly folded when m_Not contains poison elements that should have been forbidden, resulting in less-poisonous output than semantically correct

### Strategy 4: Vector splat with poison from shuffle/insertelement

- **Target**: Test pattern matching on vectors constructed via shufflevector or insertelement that produce poison splats
- **Rationale**: Vectors created via shufflevector with poison mask or insertelement with poison can produce splats with poison elements. The fix should correctly handle these when matching patterns like m_AllOnes.
- **Expected Issue**: Incorrect pattern match on poison-containing splats from shuffle/insertelement operations

### Strategy 5: AShr of all-ones with poison elements

- **Target**: Test simplifyAShrInst with -1 constant containing poison elements being shifted
- **Rationale**: The fix updated simplifyAShrInst to return Constant::getAllOnesValue instead of Op0 to preserve poison elements. Need to verify this correctly handles cases where the input -1 has poison elements that should propagate.
- **Expected Issue**: AShr optimization incorrectly removes poison information or fails to fold when it should

### Analysis

### Bug Found: AShr Optimization Loses Poison Information

**Bug Triggering Analysis:**

The test case triggers a bug in the `simplifyAShrInst` function in `llvm/lib/Analysis/InstructionSimplify.cpp`. The original code:
1. Creates `<i32 0, i32 poison>` XORed with `<i32 -1, i32 -1>`
2. This produces `<i32 -1, i32 poison>` (0 XOR -1 = -1, but poison XOR anything = poison)
3. The `ashr` of this value by any shift amount should preserve the poison in element 1

However, after the fix, `m_AllOnes()` matches `<-1, poison>` because it now ignores poison elements. The optimization then folds `ashr <-1, poison>, %shift` to `Constant::getAllOnesValue()`, which returns `<-1, -1>` WITHOUT the poison element.

**Fix Weakness Analysis:**

The patch changed `simplifyAShrInst` to return `Constant::getAllOnesValue(Op0->getType())` instead of `Op0`. The comment says "We could return the original -1 constant to preserve poison elements" but the implementation doesn't do this.

The problem:
1. `Constant::getAllOnesValue()` creates a clean all-ones constant without poison
2. The original `Op0` may have poison elements that should propagate
3. By replacing with a clean constant, poison information is lost

The fix should either:
1. Return `Op0` directly (preserving poison elements), or
2. Use a method that preserves the poison pattern from input

This is a regression introduced by the fix - while correctly changing pattern matching to ignore poison instead of undef, it broke poison propagation in the AShr optimization.
