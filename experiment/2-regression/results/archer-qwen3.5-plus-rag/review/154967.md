## Test Strategies

### Strategy 1: Poison propagation in ExtractLane

- **Target**: Early-exit loop with interleaving where lanes after early exit contain poison values
- **Rationale**: The ExtractLane instruction does not freeze its inputs like AnyOf does (see similar issue #154156). When used with early-exit, lanes after the early exit could contain poison values. The comparison (ICmpUGE) and subtraction operations in ExtractLane could propagate poison, leading to incorrect results or undefined behavior.
- **Expected Issue**: Poison from inactive lanes propagates through the ExtractLane computation, causing incorrect extracted values or undefined behavior in the exit block

### Strategy 2: ExtractLane with multiple vector operands during unrolling

- **Target**: Early-exit loop with interleave count > 1 where ExtractLane has more than 2 operands after unrolling
- **Rationale**: The unrolling code in VPlanUnroll.cpp only matches ExtractLane with exactly 2 operands (Op0 for lane index, Op1 for first vector). After unrolling, additional vector operands are added, but the pattern match may not correctly handle cases where the original ExtractLane already has multiple vector operands or when the unrolling adds operands incorrectly.
- **Expected Issue**: Incorrect operand handling during unrolling leads to wrong vector being selected or out-of-bounds extraction

### Strategy 3: ExtractLane with lane index at part boundaries

- **Target**: Early-exit loop where the first active lane is exactly at a part boundary (e.g., lane 4, 8, 12 with VF=4)
- **Rationale**: The ExtractLane implementation uses comparisons (ICmpUGE) to determine which part contains the target lane. At exact part boundaries, the comparison logic may have edge case issues, especially with the select chain that determines the final result.
- **Expected Issue**: Incorrect lane extracted when the first active lane falls exactly at a part boundary, causing wrong values in the exit block

### Strategy 4: ExtractLane with scalar VF in mixed VF range

- **Target**: Loop vectorized with a VF range that includes both scalar and vector VFs, testing ExtractLane when scalar VF is selected
- **Rationale**: The ExtractLane generation code has a special case for scalar VF (State.VF.isScalar()). The comment in VPlanTransforms.cpp mentions limiting the range to scalar VF to prevent mis-compilation. Testing with mixed VF ranges could expose issues in the scalar path.
- **Expected Issue**: Mis-compilation when scalar VF is selected in a range that also includes vector VFs, with ExtractLane producing incorrect results

### Strategy 5: ExtractLane with out-of-bounds lane index

- **Target**: Early-exit loop where FirstActiveLane could return a value >= total lanes (VF * num_parts)
- **Rationale**: The ExtractLane code assumes LaneToExtract is within valid bounds. If FirstActiveLane returns a value >= total number of lanes (e.g., when no lane is active, returning VF), the ExtractElement could be out-of-bounds or the comparison logic could produce incorrect results.
- **Expected Issue**: Out-of-bounds extraction or incorrect result when lane index exceeds total number of lanes across all parts

## Bugs Found

### Bug #1

**Analysis:**
The differential test found a difference between original and transformed code. The original returns 253 (0xFD = -3) while transformed returns 255 (0xFF = -1). This indicates a potential bug in the ExtractLane implementation for reverse iteration loops. The issue could be related to how the lane index is computed when iterating backwards - the FirstActiveLane calculation or the ExtractLane extraction may not correctly handle the reverse iteration pattern.

Looking at the transformed code, the vector.early.exit block computes:
- %42 = add i64 %index, %41 (where %41 is the first active lane within the vector)
- %43 = sub i64 1023, %42

For reverse iteration, the offset calculation may be incorrect when the early exit occurs at certain lane positions. The difference of 2 (255-253=2) suggests an off-by-two error in the lane extraction or offset computation.

This could be related to the ExtractLane implementation not correctly handling the reverse iteration pattern where the lane indices need to be inverted.
