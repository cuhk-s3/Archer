## Test Strategies

### Strategy 1: Test with mismatched operand bitwidths in alternate cast nodes

- **Target**: alternate-cast.ll test with mixed source types
- **Rationale**: The fix compares actual type sizes using MinBWs analysis. If the main and alternate operands have different original bitwidths but the same MinBW-reduced bitwidth, the fix should handle this correctly. Testing with operands that have different original bitwidths (e.g., i16 and i32) but get reduced to the same bitwidth by MinBWs analysis could expose issues in how SrcTy is computed.
- **Expected Issue**: Incorrect cast opcode selection or type mismatch when main and alternate operands have different original bitwidths but same MinBW-reduced bitwidth

### Strategy 2: Test with BWSz > SrcBWSz case (extension needed)

- **Target**: alternate-cast.ll with extension scenarios
- **Rationale**: The fix only handles the case when BWSz <= SrcBWSz (truncation or no-op). However, if BWSz > SrcBWSz, the code falls through to use E->getOpcode() and E->getAltOpcode() for the cast. This path may not properly account for MinBWs when extension is needed instead of truncation.
- **Expected Issue**: Missing extension cast or wrong extension type when target bitwidth is larger than source bitwidth after MinBWs reduction

### Strategy 3: Test with floating-point to integer cast alternates

- **Target**: alternate-cast.ll fptosi/fptoui patterns
- **Rationale**: The fix has a condition `if (SrcSclTy->isIntegerTy() && ScalarTy->isIntegerTy())` which means it only applies to integer-to-integer casts. For floating-point to integer casts (fptosi/fptoui), this condition is false, so the old code path is used. Testing mixed float/int alternate casts could reveal issues.
- **Expected Issue**: Incorrect cost calculation or codegen for alternate nodes involving floating-point to integer casts

### Strategy 4: Test with MinBWs entry missing for operand

- **Target**: alternate-cast.ll without MinBWs reduction
- **Rationale**: The fix checks `if (SrcIt != MinBWs.end())` to get the reduced bitwidth. If the operand is not in MinBWs, it uses the original type size. Testing cases where one operand is in MinBWs but the other is not could expose asymmetry issues in how SrcTy is computed for the cast cost.
- **Expected Issue**: Inconsistent cast cost calculation when only one operand has MinBWs entry, leading to incorrect vectorization decisions

### Strategy 5: Test with same bitwidth but different signedness

- **Target**: alternate-cast.ll with sext/zext patterns
- **Rationale**: The fix uses `It->second.first` for the bitwidth but the signedness information (`It->second.second`) is used in CreateIntCast. Testing cases where the bitwidths are equal but signedness differs between main and alternate operations could expose issues in how the signedness is propagated.
- **Expected Issue**: Wrong signedness used in CreateIntCast when bitwidths match but signedness requirements differ between main and alternate operations
