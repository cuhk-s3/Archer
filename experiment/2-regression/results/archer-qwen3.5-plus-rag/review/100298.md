## Test Strategies

### Strategy 1: Poison propagation in select arm

- **Target**: select with arm that can produce poison
- **Rationale**: The optimization replaces select arms with constants based on known bits. If the original arm could produce poison but the constant cannot (or vice versa), this changes semantics. Test case: select where one arm is an operation that can produce poison (like division by zero that's guarded by the condition), but the condition makes the poison path unreachable.
- **Expected Issue**: Incorrect constant folding that removes poison, changing program semantics

### Strategy 2: Condition with undef/poison value

- **Target**: select with undef/poison in condition
- **Rationale**: The commit message mentions limiting to non-undef conditions for generalization. If the condition itself contains undef/poison, the known bits analysis might incorrectly assume the condition is well-defined. Test case: select where condition is computed from a value that could be undef.
- **Expected Issue**: Incorrect simplification when condition is poison/undef, leading to wrong constant

### Strategy 3: Vector type with mixed element types

- **Target**: select with vector types where hasAffectedValue might not recurse correctly
- **Rationale**: The hasAffectedValue function only recurses into operands with isIntOrIntVectorTy(). For vector operations with non-integer element types or mixed types, this might miss affected values. Test case: select with vector result where arms involve type conversions or non-integer operations.
- **Expected Issue**: Missed optimization or incorrect folding for vector types

### Strategy 4: PHI node in select arm with condition dependency

- **Target**: select arm containing PHI node that depends on condition
- **Rationale**: The hasAffectedValue function has special handling for PHI nodes (reduces depth). This could cause early termination of analysis for PHI nodes that are affected by the condition. Test case: select where arm is a PHI node whose incoming values are affected by the select condition through control flow.
- **Expected Issue**: Incorrect known bits computation due to depth limit with PHI nodes

### Strategy 5: findValuesAffectedByCondition misses affected values with IsAssume=false

- **Target**: select condition with non-constant RHS in comparison
- **Rationale**: When IsAssume=false, findValuesAffectedByCondition only adds LHS of comparisons when RHS is constant (line 9594-9595). If the condition is something like 'icmp eq X, Y' where both X and Y are non-constant variables, neither might be added to AffectedValues, causing the optimization to miss cases or incorrectly fold.
- **Expected Issue**: Missed affected values leading to either missed optimization or incorrect constant folding

## Bugs Found

### Bug #1

**Analysis:**
This is a real bug. The original code has `sdiv i32 %y, %x` which causes division by zero UB when x=0. The select statement chooses the division result when cmp is true (x==0). The optimization incorrectly folds the true arm to a constant (0) based on the condition, but this removes the division operation entirely, eliminating the UB that should occur. The transformed code returns 5 directly without any UB, which is a semantic change. This violates poison/UB preservation - the optimization should not remove operations that can cause UB even if the result is "guarded" by a select.

**Original LLVM IR:**
```llvm

define i32 @test_poison_select2(i32 %x, i32 %y) {
  %cmp = icmp eq i32 %x, 0
  %div = sdiv i32 %y, %x
  %result = select i1 %cmp, i32 %div, i32 5
  ret i32 %result
}

define i32 @main(i32 %argc, ptr %argv) {
entry:
  %r = call i32 @test_poison_select2(i32 0, i32 10)
  ret i32 %r
}

```

## Summary of Bug Analysis

### Bug 1: Poison/UB Preservation Bug (test_poison_select2)

**Bug Triggering Analysis:**
The bug is triggered when:
1. A select statement has a condition that compares a value to a constant (e.g., `icmp eq i32 %x, 0`)
2. One arm of the select contains an operation that causes UB when the condition is true (e.g., `sdiv i32 %y, %x` which divides by zero when x==0)
3. The optimization uses the condition to infer known bits for the select arm

When x=0:
- The condition `%cmp = icmp eq i32 %x, 0` is true
- The true arm `%div = sdiv i32 %y, %x` causes division by zero UB
- The optimization incorrectly infers that when cmp is true, %div must be 0, and folds the select to just return 5
- This removes the division operation entirely, eliminating the UB

**Fix Weakness Analysis:**
The fix in commit 77eb05683082dd3751ccfab9963f5160f1852058d fails to:
1. Check if the select arm (or sub-expressions) could cause UB when evaluated
2. Preserve poison/UB semantics when folding arms to constants
3. Use `isSafeToSpeculativelyExecute` or similar before performing the optimization

### Bug 2: Vector Select Bug (vec_select_no_equivalence)

**Bug Triggering Analysis:**
The bug occurs with vector select where:
- The condition is a vector of i1 (element-wise comparison)
- The select arm involves a shufflevector operation
- The optimization incorrectly assumes element-wise known bits apply

Counter-example: x=<0,2>
- cond = <true, false> (first element is 0, second is not)
- x10 = <2, 0> (swapped via shufflevector)
- Result should be: select <true, false>, <2, 0>, <0, 2> = <2, 2>
- But optimized code returns x = <0, 2>, which is wrong!

**Fix Weakness Analysis:**
The optimization's `computeKnownBitsFromCond` doesn't properly handle:
1. Vector select with element-wise conditions
2. Shufflevector operations that mix elements across the vector
3. Per-element known bits tracking through shuffle operations

Both bugs demonstrate that the KnownBits-based select arm simplification is too aggressive and doesn't properly account for:
- Poison/UB preservation
- Vector element-wise semantics
- Operations that mix data across elements
