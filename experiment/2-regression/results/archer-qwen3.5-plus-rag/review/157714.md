## Test Strategies

### Strategy 1: Test GEP chain with mixed nowrap flags

- **Target**: foldGEPICmp with GEP chain
- **Rationale**: The fix accumulates NoWrap flags using bitwise AND across all GEPs in the chain. If one GEP in the chain lacks necessary nowrap flags (nuw/nsw/inbounds), the combined flags may be insufficient for the transformation, potentially leading to incorrect folding for non-equality comparisons.
- **Expected Issue**: Incorrect transformation when a GEP chain has mixed nowrap flags - some GEPs have nuw/nsw but others don't, causing the CanFold check to incorrectly pass or fail

### Strategy 2: Test GEP chain with vector types

- **Target**: EmitGEPOffsets with vector GEP chains
- **Rationale**: The EmitGEPOffsets function handles vector type conversion by splatting offsets. A chain of GEPs with vector types may have complex interactions where the vector splatting logic doesn't correctly handle all elements in the chain, especially if different GEPs have different vector element counts.
- **Expected Issue**: Incorrect offset computation for vector GEP chains where different GEPs in the chain have mismatched vector types or element counts

### Strategy 3: Test GEP chain with zero-length chain edge case

- **Target**: CommonPointerBase::compute when LHS equals RHS directly
- **Rationale**: When LHS equals RHS directly (no GEPs in between), the LHSGEPs vector would be empty. The EmitGEPOffsets function returns a null constant in this case, but the interaction with NewICmp and the NoWrap flags needs verification.
- **Expected Issue**: Edge case where GEP chain is empty (LHS == RHS) may produce incorrect comparison result or incorrect NoWrap flag handling

### Strategy 4: Test GEP chain with different index types

- **Target**: EmitGEPOffsets with mixed index types in GEP chain
- **Rationale**: Different GEPs in a chain may have different index types. The EmitGEPOffsets function uses a single IdxTy for the result, but intermediate GEPs might have different index types that need truncation or extension. The fix may not correctly handle type conversions across the chain.
- **Expected Issue**: Incorrect offset type or value when GEP chain has GEPs with different index types, leading to type mismatch or incorrect comparison

### Strategy 5: Test GEP chain with inbounds and non-inbounds mix

- **Target**: foldGEPICmp with mixed inbounds flags in GEP chain
- **Rationale**: The inbounds flag has stricter semantics than nuw/nsw. A chain mixing inbounds and non-inbounds GEPs may have different overflow/poison semantics. The fix accumulates flags with bitwise AND, which may lose important inbounds information needed for correct transformation.
- **Expected Issue**: Incorrect transformation when GEP chain mixes inbounds and non-inbounds GEPs, potentially changing poison semantics or allowing invalid address comparisons

### Bug #2

**Analysis:**
The bug found in test index 3 (gep_mugtiple_ugt_inbounds_nusw) is a real bug. The transformation incorrectly folds a GEP chain with mixed inbounds/nusw flags.

The issue is:
- Original: %gep1 = inbounds, %gep2 = nusw, comparing %gep2 > %base with ugt
- Transformed: (idx+idx2)*4 > 0 with sgt

The counterexample from Alive2 shows:
- idx = 2305843009213693946, idx2 = 13
- Source returns 1 (true), Target returns 0 (false)

The problem is that the fix combines flags using bitwise AND:
- inbounds implies nusw, so combined = nusw & nusw = nusw
- But the transformation to signed comparison (sgt) is incorrect because:
  1. The original comparison is ugt (unsigned)
  2. With nusw flags, the transformation should preserve unsigned semantics
  3. The transformed code uses sgt which gives different results for negative values

The difftest confirmed this: source triggers UB (return poison) while target returns 0, showing different behavior.

This is a real bug in the fix - the transformation incorrectly handles GEP chains with mixed inbounds/nusw flags when the combined offset can wrap around.
