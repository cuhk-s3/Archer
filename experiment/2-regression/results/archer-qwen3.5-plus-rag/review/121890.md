## Test Strategies

### Strategy 1: Same indices without nowrap flags

- **Target**: icmp-gep.ll test comparing GEPs with same indices but different base pointers, where at least one GEP lacks nowrap flags
- **Rationale**: The current patch adds CanFold checks to most code paths but misses the 'indices are the same' code path at line 767. If two GEPs have identical indices but different base pointers, and at least one lacks nowrap flags, the comparison should NOT be folded to base pointer comparison for unsigned predicates. Without the CanFold check, this could incorrectly fold the comparison.
- **Expected Issue**: Incorrect optimization where (gep nuw p, i) ult (gep q, i) is incorrectly folded to p ult q, even though the second GEP has no nowrap flags and the addition might wrap, changing the comparison result.

### Strategy 2: Mixed nuw and no-flags GEPs with same indices

- **Target**: icmp-gep.ll test with one GEP having nuw flag and another having no flags, same indices, unsigned comparison
- **Rationale**: When one GEP has nuw and the other has no flags, the intersection NW is none. For unsigned comparisons, CanFold(none) should return false, preventing the fold. Without the CanFold check in the same-indices path, this fold would incorrectly proceed.
- **Expected Issue**: The optimizer incorrectly folds the comparison, potentially producing wrong results when the non-nuw GEP's offset calculation wraps.

### Strategy 3: nusw-only GEPs with same indices and unsigned comparison

- **Target**: icmp-gep.ll test with both GEPs having only nusw flag (no nuw), same indices, unsigned comparison
- **Rationale**: With nusw-only flags, the comparison should be converted to signed, not folded to base pointer comparison. The same-indices path doesn't use NewICmp and doesn't check CanFold, so it might incorrectly keep the unsigned predicate or fold when it shouldn't.
- **Expected Issue**: Incorrect predicate preservation or incorrect folding when nusw-only GEPs are compared with same indices.

### Strategy 4: Vector GEP comparison with mixed nowrap flags

- **Target**: icmp-gep.ll test with vector GEPs having different nowrap flags
- **Rationale**: The code has special handling for vectors (e.g., line 742-744, 779). The CanFold check might not be properly applied to vector cases in all code paths, especially the same-indices path which doesn't check for vector types before folding.
- **Expected Issue**: Vector comparisons might be incorrectly folded when nowrap flags don't justify the transformation.

### Strategy 5: Inbounds GEPs with nuw and same indices

- **Target**: icmp-gep.ll test with inbounds GEPs (which imply nusw but not nuw) that also have explicit nuw, same indices
- **Rationale**: Inbounds implies nusw. If we have inbounds+nuw GEPs with same indices, the intersection would have both flags. The same-indices path should handle this correctly, but without the CanFold check, edge cases might be missed. Additionally, the separate code path at lines 774-798 for inbounds GEPs with different base pointers uses signed predicate unconditionally, which might conflict with the new nuw-aware logic.
- **Expected Issue**: Inconsistent handling of inbounds+nuw GEPs between different code paths, potentially leading to suboptimal or incorrect code generation.
