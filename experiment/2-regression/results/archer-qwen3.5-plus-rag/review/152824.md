## Test Strategies

### Strategy 1: Test copysign with known negative sign from min/max intrinsics

- **Target**: llvm/lib/Transforms/InstCombine/InstCombineCalls.cpp copysign handling
- **Rationale**: The patch adds sign bit inference for minnum/maxnum/minimum/maximum intrinsics. If copysign receives a sign argument from these intrinsics where the sign bit is inferred to be 1 (negative), the new code path should transform copysign to fneg(fabs Mag). Test case should verify this transformation is correct when the min/max result is known to be negative.
- **Expected Issue**: Incorrect transformation if sign bit inference for min/max is wrong, leading to copysign producing wrong sign result

### Strategy 2: Test FMul sign bit inference with mixed sign operands

- **Target**: llvm/lib/Analysis/ValueTracking.cpp FMul case
- **Rationale**: The patch adds sign bit inference for FMul: if both operands have known sign bits and they are the same, result sign bit is 0; if different, result sign bit is 1. Test should verify this logic handles all combinations correctly, especially when one or both operands could be NaN (which have sign bits but the result NaN sign is implementation-defined).
- **Expected Issue**: Incorrect sign bit inference when NaN is involved, since NaN sign bits may not propagate correctly through multiplication

### Strategy 3: Test vector constant sign bit inference with mixed signs

- **Target**: llvm/lib/Analysis/ValueTracking.cpp vector constant handling
- **Rationale**: The patch modifies vector constant handling to track whether all sign bits are zero or all are one. If SignBitAllOne != SignBitAllZero, it sets Known.SignBit. Test should verify this handles vectors with mixed sign elements correctly (should result in nullopt, not a specific value).
- **Expected Issue**: Incorrect sign bit inference for vectors with mixed sign elements, potentially causing wrong optimizations

### Strategy 4: Test knownNot() sign bit inference edge case with NaN

- **Target**: llvm/include/llvm/Analysis/ValueTracking.h KnownFPClass::knownNot()
- **Rationale**: The knownNot() method now infers sign bit when isKnownNever(fcNan) is true and either fcNegative or fcPositive is ruled out. Test should verify this doesn't incorrectly infer sign bit when NaN is still possible, or when both negative and positive are still possible.
- **Expected Issue**: Sign bit incorrectly inferred when NaN is still possible, or when the logic for determining sign bit from remaining classes is flawed

### Strategy 5: Test maxnum/minnum with -0.0/+0.0 edge cases

- **Target**: llvm/lib/Analysis/ValueTracking.cpp maxnum/minnum handling
- **Rationale**: The patch adds sign bit inference for maxnum/minnum when one operand's sign bit is known. However, maxnum(+0.0, -0.0) result is unspecified per IEEE 754. Test should verify the sign bit inference doesn't assume a specific result when both +0.0 and -0.0 are possible inputs.
- **Expected Issue**: Incorrect sign bit inference for maxnum/minnum when operands could be +0.0/-0.0, leading to wrong copysign or fabs optimizations
