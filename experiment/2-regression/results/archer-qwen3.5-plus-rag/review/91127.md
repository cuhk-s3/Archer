## Test Strategies

### Strategy 1: Mixed nuw/nsw flags for unsigned comparison

- **Target**: icmp-of-trunc-ext.ll test cases like trunc_unsigned_either
- **Rationale**: The patch requires that for unsigned comparisons, the intersection of nowrap flags must be non-empty. When one trunc has nuw and the other has nsw, the intersection is empty, and the fold should NOT proceed. This tests the flag intersection logic.
- **Expected Issue**: If the fold incorrectly proceeds with mixed nuw/nsw flags, the comparison result could be wrong because nuw and nsw have different semantic guarantees (nuw requires upper bits to be zero, nsw requires upper bits to match the sign bit).

### Strategy 2: trunc nsw with unsigned comparison (non-obvious case)

- **Target**: icmp-of-trunc-ext.ll test cases like trunc_unsigned_nsw
- **Rationale**: The commit message specifically mentions that 'for unsigned predicates we can do the transform for two trunc nsw as well (rather than only trunc nuw)'. This is a non-obvious case that should be tested thoroughly with various unsigned predicates (ult, ule, ugt, uge) and values that exercise the sign bit behavior.
- **Expected Issue**: If the transformation is incorrect for trunc nsw with unsigned comparison, values with different sign bits could produce wrong results because nsw allows upper bits to be all 1s (for negative values), which affects unsigned comparison differently than nuw.

### Strategy 3: trunc nuw with signed comparison (should NOT fold)

- **Target**: icmp-of-trunc-ext.ll test cases like trunc_signed_nuw
- **Rationale**: For signed comparisons, the patch requires both truncs to have nsw flag. If both truncs only have nuw (not nsw), the fold should NOT proceed. This tests the signed comparison flag requirement.
- **Expected Issue**: If the fold incorrectly proceeds with only nuw flags for signed comparison, negative values could produce wrong results because nuw only guarantees upper bits are zero, not that they match the sign bit.

### Strategy 4: trunc nsw + zext with signed comparison

- **Target**: icmp-of-trunc-ext.ll test cases like trunc_signed_nsw_zext
- **Rationale**: The patch allows trunc nsw + zext for all predicates including signed. This is a subtle case because zext zero-extends while nsw allows sign-extended upper bits. The transformation must correctly handle the cast direction.
- **Expected Issue**: If the YIsSExt flag is computed incorrectly or the cast is wrong, comparing values with different sign extensions could produce incorrect results for signed predicates.

### Strategy 5: Poison propagation edge case

- **Target**: Create test where trunc nowrap conditions are violated
- **Rationale**: When trunc nowrap conditions are violated, the original code produces poison. The transformation removes the truncs and compares original values. While this is technically UB territory, we should verify the transformation doesn't introduce observable differences in non-UB cases and that Alive2 proofs cover these edge cases.
- **Expected Issue**: If the transformation doesn't properly account for poison semantics, cases where one trunc produces poison but the other doesn't could lead to inconsistent behavior between original and transformed code.
