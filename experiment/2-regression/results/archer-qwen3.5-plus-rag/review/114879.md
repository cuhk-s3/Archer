## Test Strategies

### Strategy 1: UDivExpr in select-like phi

- **Target**: Test case with select-like PHI where one or both incoming values contain a udiv expression
- **Rationale**: The old IsAvailableOnEntry function would always mark any SCEV containing scUDivExpr as unavailable, preventing the select-like PHI from being converted to a select node. The new properlyDominates check will allow the conversion if the udiv operands properly dominate the PHI block. This could lead to different SCEV representations.
- **Expected Issue**: SCEV may now create a select node for patterns involving udiv that it previously rejected, potentially leading to incorrect SCEV analysis or crashes during SCEV expansion if the udiv could trap

### Strategy 2: Argument values in select-like phi

- **Target**: Test case with select-like PHI where incoming values are function arguments
- **Rationale**: In the old IsAvailableOnEntry, arguments were handled specially - they would cause setUnavailable() to be called. In the new properlyDominates, arguments return ProperlyDominatesBlock (since they're not Instructions). This changes the availability semantics for arguments.
- **Expected Issue**: SCEV may incorrectly create select nodes for PHI nodes with argument incoming values, which could lead to incorrect loop analysis or optimization decisions

### Strategy 3: AddRecExpr in outer loop used in inner loop select

- **Target**: Nested loop test where a select-like PHI in an inner loop uses values from an outer loop's AddRecExpr
- **Rationale**: The old IsAvailableOnEntry had special handling for AddRecExpr - it would only allow add recurrences on the loop BB is in or an outer loop. The new properlyDominates checks if the AddRecExpr's loop header dominates the BB. These conditions may differ in edge cases.
- **Expected Issue**: Different SCEV expressions may be generated for nested loop patterns, potentially affecting loop optimization passes that depend on accurate SCEV analysis

### Strategy 4: Instruction defined in same block as PHI

- **Target**: Test case where the incoming values of a select-like PHI are instructions defined in the same block as the PHI
- **Rationale**: In computeBlockDisposition, if an instruction's parent block equals the BB being checked, it returns DominatesBlock (not ProperlyDominatesBlock). The old IsAvailableOnEntry would check DT.dominates() which returns true for same-block instructions. This affects the properlyDominates check which requires ProperlyDominatesBlock.
- **Expected Issue**: Select-like PHI patterns where incoming values are defined in the same block may now be rejected (properlyDominates returns false when instruction is in same block), whereas before they might have been accepted

### Strategy 5: Complex SCEV expressions with mixed dominance

- **Target**: Test case with select-like PHI where incoming SCEV expressions have operands with mixed dominance properties (some properly dominate, some only dominate)
- **Rationale**: The new properlyDominates implementation propagates DominatesBlock status through n-ary expressions - if any operand only DominatesBlock (not ProperlyDominatesBlock), the whole expression returns DominatesBlock. This is different from the old IsAvailableOnEntry which had a simpler traversal model.
- **Expected Issue**: SCEV may reject select-like PHI conversions that it previously accepted, or vice versa, leading to different optimization opportunities being missed or incorrectly applied

### Analysis

### Bug Analysis Summary

After thorough testing of the ScalarEvolution fix that replaces `IsAvailableOnEntry` with `properlyDominates`, no bugs were found. All test cases passed verification and differential testing.

### Fix Understanding

The patch replaces the custom `IsAvailableOnEntry` function with the standard `properlyDominates` function in `createNodeFromSelectLikePHI`. The key changes are:
1. Removed the 84-line `IsAvailableOnEntry` function that implemented custom availability checking
2. Changed two calls from `IsAvailableOnEntry(L, DT, getSCEV(LHS), PN->getParent())` to `properlyDominates(getSCEV(LHS), PN->getParent())`

### Test Coverage

All five proposed test strategies from Phase 1 were covered:

1. **UDivExpr in select-like phi**: Tested with `test_udiv_select` - The old code would reject any SCEV containing scUDivExpr, but the new code allows it if operands properly dominate. The transformation was verified correct.

2. **Argument values in select-like phi**: Tested with `tautological_select_like_phi` and `test_arg_select` - The old code marked arguments as unavailable, but the new code treats them as ProperlyDominatesBlock. The transformation was verified correct.

3. **Instruction defined in same block as PHI**: Tested with `test_same_block_select` - The new code returns DominatesBlock (not ProperlyDominatesBlock) for same-block instructions, causing properlyDominates to return false. The transformation was verified correct.

4. **AddRecExpr in outer loop used in inner loop select**: Tested with `test_nested_loop_select` - The old code had special handling for AddRecExpr (same loop or outer loop), while the new code checks if the loop header dominates. The transformation was verified correct.

5. **Complex SCEV expressions with mixed dominance**: Tested with `test_mixed_dominance` - The new code propagates DominatesBlock status through n-ary expressions. The transformation was verified correct.

### Conclusion

The fix appears to be correct. The replacement of `IsAvailableOnEntry` with `properlyDominates` is semantically sound because:
1. The old `IsAvailableOnEntry` was overly conservative (rejecting valid patterns like udiv expressions)
2. The new `properlyDominates` correctly captures the dominance requirements for select-like PHI conversion
3. Safety concerns about trapping operations (like udiv) are properly handled by SCEVExpander's `isSafeToExpand()`, not by the availability check

All differential tests showed matching results between original and transformed code, confirming the fix does not introduce any regressions.
