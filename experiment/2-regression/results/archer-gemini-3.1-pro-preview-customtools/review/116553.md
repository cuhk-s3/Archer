## Test Strategies

### Strategy 1: Non-dedicated exit block

- **Target**: Create a loop with an exit block that also has a predecessor outside the loop. Add a condition in the exit block that would be simplified if the loop header condition is assumed to hold.
- **Rationale**: The patch propagates the loop header condition to all exit blocks. However, if an exit block is not dedicated (i.e., it has predecessors outside the loop), the condition from the loop header might not hold when the exit block is reached from outside the loop. The patch does not check if the loop header dominates the exit block or if the exit block is dedicated.
- **Expected Issue**: The pass will incorrectly assume the induction variable is bounded by the loop header condition in the exit block, even though the exit block can be reached from outside the loop. This could lead to incorrect simplification of conditions in the exit block.

### Strategy 2: Multiple exiting blocks

- **Target**: Create a loop with multiple exiting blocks. One exit is the header, another is inside the loop. Check if the condition is correctly propagated to the exit block of the inner exiting block.
- **Rationale**: If there are multiple exiting blocks, the patch assumes that exiting from any of them implies the induction variable is bounded by the header's condition. This is generally true because the header dominates all other blocks in the loop. However, we should test this to ensure no edge cases are missed.
- **Expected Issue**: The pass will incorrectly assume the induction variable is bounded by the loop header condition in the exit block, even though the loop header condition might not have been executed if the loop is exited from a different block before the header condition is checked (e.g., if the header condition is not the first thing in the header, or if there are multiple exiting blocks and one exits before the header condition is checked). Wait, the header condition is the branch at the end of the header. If the header is the only exiting block, it's fine. If there are other exiting blocks, they must be dominated by the header. Since it's a loop, the header dominates all blocks in the loop. So any other exit must be reached after the header. But wait, what if the loop header condition is `i != N`, and there is another exit condition `i == M` inside the loop. If the loop exits from the `i == M` condition, does `i <= N` hold? Yes, because the header condition `i != N` was checked and passed (meaning `i < N` if it's increasing) before reaching the `i == M` exit. Wait, the patch adds the condition `i <= N` to ALL exit blocks. If the loop exits from the header, `i == N`, so `i <= N` holds. If the loop exits from another block, it means the header condition evaluated to true (e.g., `i != N`), so `i < N`, which implies `i <= N`. So this seems correct, AS LONG AS the header condition is actually checked before the other exit. Since it's the header, it's always checked first in each iteration. But what if the loop is exited on the FIRST iteration from a block other than the header? Wait, the header is the entry point of the loop. So the header is always executed first. The header's terminator is the branch. So the header's condition is checked at the end of the header. If there is an exit BEFORE the header's terminator? The header is a single basic block. You can't exit from the middle of a basic block. So the header's terminator is always executed if the header finishes. Wait, what if the header can throw an exception? Then it doesn't reach the exit block. What if the header is NOT the exiting block? The code checks `L->isLoopExiting(&BB)`. So `BB` (the header) is an exiting block. The condition is at the terminator of `BB`. So the condition is always evaluated.

### Strategy 3: Poison in loop header condition

- **Target**: Create a loop where the header condition involves a potentially poison value.
- **Rationale**: If the loop header condition involves poison, propagating it might lead to incorrect optimizations.
- **Expected Issue**: The pass will incorrectly assume the induction variable is bounded by the loop header condition in the exit block, even though the loop header condition might be based on a poison value. If the loop header condition is poison, the branch is undefined behavior. But if we propagate this to the exit block, we might optimize based on poison. However, this is a general issue with ConstraintElimination and poison, not specific to this patch.

### Strategy 4: Negative step

- **Target**: Create a loop with a negative step and check if it crashes or is handled correctly.
- **Rationale**: The patch asserts that the step is not negative. We should ensure that negative steps are correctly filtered out before reaching this code.
- **Expected Issue**: The pass will incorrectly assume the induction variable is bounded by the loop header condition in the exit block, even though the step is negative. The patch asserts `!StepOffset.isNegative()`. If we can create a loop with a negative step that somehow bypasses the earlier checks, it would crash the compiler. Wait, the earlier check `if (StepOffset.isNegative()) return;` handles negative steps. So the assert is safe.

### Strategy 5: Step greater than 1

- **Target**: Create a loop with a step greater than 1, where `B - Start` is not a multiple of the step. Check if the optimization is correctly suppressed. Create another loop where `B - Start` is a multiple of the step, and check if the optimization is applied.
- **Rationale**: The patch handles steps greater than 1 by checking if `B - Start` is a multiple of the step. We should test this logic to ensure it correctly filters out cases where the induction variable might step over the bound.
- **Expected Issue**: The pass will incorrectly assume the induction variable is bounded by the loop header condition in the exit block, even though the loop header condition is `EQ` or `NE`, and the step is greater than 1. If the step is greater than 1, the induction variable might step over the bound `B`. For example, if `i` starts at 0, steps by 2, and the condition is `i != 3`. The loop will exit when `i == 4` (if it doesn't wrap). In this case, `i <= 3` is FALSE in the exit block. The patch adds `CmpInst::ICMP_ULE, A, B` (i.e., `i <= 3`). This would be incorrect! Let's check if step > 1 is allowed. The code has: `if (!StepOffset.isOne()) { const SCEV *BMinusStart = SE.getMinusSCEV(SE.getSCEV(B), StartSCEV); if (isa<SCEVCouldNotCompute>(BMinusStart) || !SE.getConstantMultiple(BMinusStart).urem(StepOffset).isZero()) return; }`. This checks that `B - Start` is a multiple of `StepOffset`. So in the example `Start = 0, B = 3, Step = 2`, `B - Start = 3`, which is not a multiple of 2. So it returns early. What if `Start = 0, B = 4, Step = 2`? Then `B - Start = 4`, which is a multiple of 2. The condition is `i != 4`. The loop will exit when `i == 4`. In the exit block, `i <= 4` is TRUE. So this is correct. What if the condition is `i == 4`? The loop exits when `i == 4`. `i <= 4` is TRUE. So this is also correct.
