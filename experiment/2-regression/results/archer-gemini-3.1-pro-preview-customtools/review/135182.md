The bug is triggered when `widenLoopCompare` succeeds after `tryAddRecExpansion` fails.

Let me summarize the fix.
The fix addresses an issue in `SimplifyIndVar` where `widenIVUse` would sometimes leave both the original unwidened IV and the partially widened IV in the code.
This happened when the user evaluated to a recurrence after widening (`WideAddRec.first` is true), but either `cloneIVUser` failed (e.g., because the opcode is not in the whitelist) or the SCEV expressions of the cloned user and the expected recurrence did not match.
In these cases, the old code would simply return `nullptr`, leaving the narrow IV use intact.
The new code wraps the recurrence expansion logic in a lambda `tryAddRecExpansion`. If this lambda fails (returns `nullptr`), the code now falls through to try `widenLoopCompare`, `widenWithVariantUse`, and finally `truncateIVUse`.
This ensures that even if the recurrence expansion fails, the wide IV is still truncated to the narrow type for this specific use, allowing the original narrow IV to be isolated and removed by `DeleteDeadPHIs`.
This makes the code more robust and less tightly coupled to SCEV's internal capabilities.
