## Test Strategies

### Strategy 1: Bitcast to Float with Known Sign Bit and NaN

- **Target**: A bitcast from an integer to a float where the integer has a known sign bit and the known one bits form a NaN.
- **Rationale**: When `computeKnownFPClass` is called on a `bitcast` instruction, it first computes the `KnownBits` of the source integer. If the sign bit is known, it calls `Known.signBitMustBeZero()` or `Known.signBitMustBeOne()`. These functions update `Known.KnownFPClasses` by clearing the opposite sign classes, but they leave `fcNan` set (e.g., `KnownFPClasses &= (fcPositive | fcNan)`). Later, if the value is determined to be NaN based on the exponent and fraction bits (`APFloat(Ty->getFltSemantics(), Bits.One).isNaN()`), the code does `Known.KnownFPClasses = fcNan;`. This assignment completely overwrites the previous `KnownFPClasses` value, discarding the sign information that was just carefully preserved by `signBitMustBeZero()` or `signBitMustBeOne()`. The correct operation should be `Known.KnownFPClasses &= fcNan;` or `Known.KnownFPClasses = fcNan | (Known.KnownFPClasses & (fcPositive | fcNegative));` to preserve the sign information, or simply `Known.KnownFPClasses &= fcNan;` since `fcNan` doesn't have sign in `FPClassTest` but `Known.SignBit` is tracked separately. Wait, `Known.SignBit` is tracked separately, but `Known.KnownFPClasses = fcNan` overwrites the classes. Actually, `Known.KnownFPClasses` is a bitmask. If it was `fcPositive | fcNan` and we set it to `fcNan`, we are just clearing `fcPositive`, which is correct if it's definitely NaN. But wait, what if `Bits.One` is NaN, does it mean it's *always* NaN? Yes, if the known one bits form a NaN, then no matter what the unknown bits are, it's a NaN. So setting `Known.KnownFPClasses = fcNan` is correct in that it can only be NaN. However, does `fcNan` include sign information? No, `fcNan` is `fcSNan | fcQNan`. The sign bit is tracked in `Known.SignBit`. So this might actually be correct. Let's re-evaluate.
- **Expected Issue**: The `KnownFPClasses` will be incorrectly set to `fcNan` (which is `fcSNan | fcQNan`), losing the previously known information about the sign bit or other classes (e.g., `fcPositive`). This can lead to incorrect optimizations later on, as the compiler might think the value can only be NaN, while it could actually be something else if the `KnownBits` analysis was incomplete, or it might lose the sign information of the NaN.

### Strategy 2: Bitcast to Float with Known Sign Bit and Inf/Zero

- **Target**: A bitcast from an integer to a float where the integer is exactly a constant `+Inf`, `-Inf`, `+0.0`, or `-0.0`.
- **Rationale**: The code attempts to check if the value is exactly Infinity or Zero by comparing `Bits` with `InfKB` or `ZeroKB`. It does this by clearing the sign bit of `InfKB.Zero` and `ZeroKB.Zero`, effectively making the sign bit "unknown" in these reference `KnownBits`. Then it checks `if (Bits == InfKB)`. The `operator==` for `KnownBits` requires exact equality of both `Zero` and `One` masks. If the actual `Bits` from the source integer has a *known* sign bit (either 0 or 1), its `Zero` or `One` mask will have the sign bit set. However, `InfKB` and `ZeroKB` have the sign bit cleared in both `Zero` and `One`. Therefore, `Bits == InfKB` will *always evaluate to false* if `Bits` has a known sign bit, even if the rest of the bits exactly match Infinity or Zero. This means the analysis will fail to identify `fcInf` or `fcZero` when the sign bit is known, missing optimization opportunities. Furthermore, `KnownBits::eq(Bits, InfKB)` returns `std::nullopt` if they don't conflict but aren't exactly equal constants. If `Bits` has a known sign bit and matches Inf otherwise, `eq` will return `std::nullopt`, so it won't even trigger the `knownNot(fcInf)` path. Wait, if `Bits` is exactly `+Inf`, `Bits.Zero` has the sign bit set. `InfKB.Zero` has the sign bit cleared. `Bits.One` matches `InfKB.One`. `eq(Bits, InfKB)` will see `Bits.One` (no sign bit) intersects `InfKB.Zero` (no sign bit)? No. `Bits.Zero` (sign bit set) intersects `InfKB.One` (no sign bit)? No. So `eq` returns `std::nullopt`. Thus, `+Inf` is not recognized as `fcInf`, nor is it recognized as `knownNot(fcInf)`. It just falls through.
- **Expected Issue**: The `KnownFPClasses` will be incorrectly set to `fcInf` or `fcZero`, losing the sign information. For example, if `Bits == InfKB`, it sets `Known.KnownFPClasses = fcInf;`. But `fcInf` is `fcPosInf | fcNegInf`. If the sign bit was known to be 0, `Known.SignBit` is false, but `Known.KnownFPClasses` now contains both `fcPosInf` and `fcNegInf`. Wait, `InfKB.Zero.clearSignBit();` means `InfKB` has an unknown sign bit. So `Bits == InfKB` means `Bits` exactly matches `InfKB`, which means `Bits` has an unknown sign bit! If `Bits` had a known sign bit, `Bits == InfKB` would be false, because `InfKB` has an unknown sign bit. Let's check `KnownBits::eq`. `KnownBits::eq` returns true if they are exactly equal constants. Wait, `Bits == InfKB` is `Bits.Zero == InfKB.Zero && Bits.One == InfKB.One`. Since `InfKB.Zero` has the sign bit cleared, `InfKB` does not know the sign bit. If `Bits` knows the sign bit, `Bits.Zero` or `Bits.One` will have the sign bit set, so `Bits == InfKB` will be FALSE! Thus, it will never match `fcInf` or `fcZero` if the sign bit is known! This is a huge missed optimization and a logical flaw.

### Strategy 3: Missed Identification of Exact Inf/Zero due to KnownBits Equality

- **Target**: A bitcast from an integer to a float where the integer is known to be exactly `+Inf` (e.g., `0x7F800000` for `float`).
- **Rationale**: The patch uses `Bits == InfKB` to check if the value is definitely Infinity. As analyzed above, if `Bits` has a known sign bit, this equality will fail because `InfKB` has an unknown sign bit. To fix this, the patch should have checked if the non-sign bits of `Bits` exactly match the non-sign bits of `InfKB`. Because of this bug, if we have a bitcast of an integer that is known to be exactly `+Inf` (e.g., a constant, or a value with all bits known), the patch will fail to set `Known.KnownFPClasses = fcInf` (or `fcPosInf`). It will just leave it as whatever it was. This is a missed deduction.
- **Expected Issue**: The analysis will fail to deduce that the value is not Infinity or not Zero. `KnownBits::eq` returns `false` only if there is a conflict (i.e., `LHS.One & RHS.Zero` or `LHS.Zero & RHS.One`). If `Bits` has some unknown bits, but they don't conflict with `InfKB`, `eq` returns `std::nullopt`. If `Bits` has a known sign bit, it doesn't conflict with `InfKB` (since `InfKB` has no sign bit in `Zero` or `One`). So `eq` returns `std::nullopt`. The code only calls `Known.knownNot(fcInf)` if `InfResult` is `false`. This is correct, but the `Bits == InfKB` check is too strict.

### Strategy 4: Bitcast of Exact +Inf or -Inf Fails to Deduce fcInf

- **Target**: A bitcast from an integer with known bits corresponding to exactly `+Inf` (e.g., `0x7F800000`) or `-Inf` (e.g., `0xFF800000`).
- **Rationale**: The condition `Bits == InfKB` requires the `KnownBits` of the source to exactly match `InfKB`. Since `InfKB.Zero.clearSignBit()` was called, `InfKB` represents a state where all bits except the sign bit are exactly known (exponent all 1s, mantissa all 0s), and the sign bit is completely unknown. Therefore, `Bits == InfKB` will only be true if the source integer also has a completely unknown sign bit and exactly known exponent/mantissa. If the source integer is a constant like `0x7F800000` (+Inf), its sign bit is known to be 0. Thus, `Bits.Zero` will have the sign bit set, while `InfKB.Zero` will not. The equality `Bits == InfKB` will evaluate to false. Consequently, the analysis will fail to deduce that the value is `fcInf` (specifically `fcPosInf`), missing a clear optimization opportunity for exact constants or values with known sign bits.
- **Expected Issue**: The analysis will incorrectly set `Known.KnownFPClasses = fcInf` when `Bits == InfKB`. But `Bits == InfKB` means `Bits` has an UNKNOWN sign bit (since `InfKB` has an unknown sign bit). So `Known.KnownFPClasses` becomes `fcInf` (which is `fcPosInf | fcNegInf`). This is actually correct: it's either `+Inf` or `-Inf`. But wait, what if `Bits` is exactly `+Inf`? Then `Bits` has a known sign bit (0). `Bits == InfKB` is FALSE. So it does NOT set `fcInf`. What if `Bits` is exactly `-Inf`? `Bits == InfKB` is FALSE. So it does NOT set `fcInf`. The ONLY way `Bits == InfKB` is TRUE is if `Bits` has exactly the exponent bits set to 1, fraction bits set to 0, and the sign bit is UNKNOWN. This is a very specific and rare state (e.g., `x & 0x7FFFFFFF == 0x7F800000`). If this rare state happens, it correctly sets `fcInf`. But for the common cases of exact `+Inf` or `-Inf`, it fails.

### Strategy 5: Bitcast of Exact +0.0 or -0.0 Fails to Deduce fcZero

- **Target**: A bitcast from an integer with known bits corresponding to exactly `+0.0` (e.g., `0x00000000`) or `-0.0` (e.g., `0x80000000`).
- **Rationale**: Similar to the `fcInf` case, `ZeroKB` is constructed with an unknown sign bit. The check `Bits == ZeroKB` will only succeed if the source integer has an unknown sign bit and all other bits known to be 0. If the source is exactly `0` (+0.0) or `0x80000000` (-0.0), its sign bit is known, so `Bits == ZeroKB` will be false. The analysis will fail to deduce `fcZero` (or `fcPosZero`/`fcNegZero`).
- **Expected Issue**: The analysis will fail to deduce `fcZero` for exact `+0.0` or `-0.0` constants or values with known sign bits, for the exact same reason as `fcInf`. `Bits == ZeroKB` will be false if the sign bit is known.
