## Test Strategies

### Strategy 1: Vector Cross-Lane Shuffle

- **Target**: `and (icmp eq <2 x i32> %a, <i32 1, i32 3>), (icmp eq <2 x i32> (shufflevector %a, poison, <1, 0>), <i32 3, i32 1>)`
- **Rationale**: `simplifyAndOrWithICmpEq` replaces the whole vector `%a` with `<i32 1, i32 3>` in the `shufflevector`, assuming that if `%a == <i32 1, i32 3>` is true for a lane, the whole vector can be replaced. But `shufflevector` moves elements across lanes, so the value at lane 0 depends on `%a[1]`, which might not be equal to 3 even if `%a[0] == 1`.
- **Expected Issue**: The `and` is incorrectly simplified to `icmp eq <2 x i32> %a, <i32 1, i32 3>`, which evaluates to `<true, false>` when `%a = <1, 2>`, while the original expression evaluates to `<false, false>`.

### Strategy 2: Vector Cross-Lane Bitcast

- **Target**: `and (icmp eq <4 x i32> %a, <i32 1, i32 2, i32 3, i32 4>), (icmp eq <2 x i64> (bitcast %a to <2 x i64>), <i64 0x0000000200000001, i64 0x0000000400000003>)`
- **Rationale**: Similar to `shufflevector`, `bitcast` from `<4 x i32>` to `<2 x i64>` combines elements across lanes. Replacing the whole vector `%a` with the constant vector is invalid because the equality might only hold for some lanes.
- **Expected Issue**: The `and` is incorrectly simplified, leading to a miscompilation when `%a` partially matches the constant vector.

### Strategy 3: ExtractElement with Undef

- **Target**: `or (icmp ne i32 %a, %b), (extractelement <2 x i1> <i1 true, i1 undef>, i32 (zext i1 (icmp ne i32 %a, %b) to i32))`
- **Rationale**: When `%a == %b`, the `extractelement` simplifies to `true`, so `simplifyAndOrWithICmpEq` returns the `extractelement` instruction. However, when `%a != %b`, the `extractelement` evaluates to `undef`, while the original `or` expression evaluates to `true`. This transforms `true` into `undef`, which is an invalid refinement.
- **Expected Issue**: The `or` is incorrectly simplified to the `extractelement` instruction, which evaluates to `undef` when `%a != %b`, causing a miscompilation.

### Strategy 4: ObjectSize Intrinsic

- **Target**: `and (icmp eq ptr %a, @global), (icmp eq i64 (call i64 @llvm.objectsize.i64.p0(ptr %a, i1 false, i1 false, i1 false)), 100)`
- **Rationale**: `simplifyWithOpReplaced` replaces `%a` with `@global` in the `llvm.objectsize` call, which simplifies to the size of `@global` (e.g., 100). This causes the `and` to be simplified to `icmp eq ptr %a, @global`. However, `llvm.objectsize` is evaluated at compile time, and for an unknown pointer `%a`, it evaluates to `-1`. If `%a == @global` at runtime, the original expression evaluates to `false`, but the simplified expression evaluates to `true`.
- **Expected Issue**: The `and` is incorrectly simplified to `icmp eq ptr %a, @global`, causing a miscompilation when `%a` is equal to `@global` at runtime but its size is unknown at compile time.

### Analysis

### Summary of the Fix and Analysis

The patch introduces `simplifyAndOrWithICmpEq` to simplify `and`/`or` instructions where one operand is an `icmp eq` or `icmp ne`. It uses `simplifyWithOpReplaced` to replace one operand of the `icmp` with the other inside the other operand of the `and`/`or`. The patch also modifies `simplifyWithOpReplaced` to not fold away `llvm.is.constant` checks based on assumptions, as `llvm.is.constant` is evaluated at compile time and replacing a runtime value with a constant can lead to miscompilations.

During Phase 1, several potential issues were identified:
1. **Cross-Lane Vector Operations**: If `Op1` involves cross-lane operations like `shufflevector` or `bitcast`, replacing the whole vector `A` with `B` is invalid because the value at lane `i` might depend on `A[j]` (where `j != i`), which might not be equal to `B[j]`.
2. **Other Compile-Time Intrinsics**: `llvm.objectsize` is also evaluated at compile time. Replacing a runtime pointer with a global variable can cause `llvm.objectsize` to evaluate to a constant, leading to the same miscompilation as `llvm.is.constant`.
3. **Undef Refinement**: If `Op1` evaluates to `undef` when `A != B`, but simplifies to `true` when `A == B`, returning `Op1` for an `or` expression transforms `true` into `undef`, which is an invalid refinement.

However, during Phase 2 testing, it was discovered that `simplifyWithOpReplaced` does NOT recurse into operands of operands. It only replaces DIRECT operands of the instruction `I`.
- For cross-lane vector operations (like `shufflevector` or `bitcast`), the variable `%a` is an operand of the cross-lane instruction, which is in turn an operand of the `icmp` (or other instruction returning `i1`). Since `%a` is not a direct operand of the `icmp`, `simplifyWithOpReplaced` will not replace it, and the simplification will not trigger.
- For `llvm.objectsize`, the variable `%a` is an operand of the `llvm.objectsize` call, which is an operand of the `icmp`. Again, `%a` is not a direct operand of the `icmp`, so it will not be replaced.
- For `extractelement` with `undef`, `InstCombine` simplifies the instruction before `simplifyAndOrWithICmpEq` is called, preventing the invalid refinement.

Therefore, the patch is robust because `simplifyWithOpReplaced`'s lack of recursion inherently protects against these complex cases where replacing a variable with a constant would change the semantics in a non-trivial way. The only instruction that takes a direct operand, returns `i1`, and evaluates a compile-time property is `llvm.is.constant`, which was explicitly fixed in the patch. No bugs were found.
