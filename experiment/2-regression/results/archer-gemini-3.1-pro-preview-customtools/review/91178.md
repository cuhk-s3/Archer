## Test Strategies

### Strategy 1: Vector to Scalar Cross-Lane Operation

- **Target**: `simplifyWithOpReplaced` with a vector `Op` and a scalar instruction `I` (e.g., `extractelement`).
- **Rationale**: The old code asserted `I->getType()->isVectorTy()` when `Op` is a vector. If `Op` is a vector and `I` is a scalar (e.g., `extractelement`), the assertion would fail. The patch fixes this by checking `!I->getType()->isVectorTy()` and returning `nullptr`.
- **Expected Issue**: Compiler crash due to assertion failure `Vector type mismatch` or incorrect simplification of cross-lane operations.

### Strategy 2: Recursive Replacement in Xor/Sub

- **Target**: `simplifyWithOpReplaced` where `Op` is replaced by `RepOp` in a chain, leading to `Xor` or `Sub` where `NewOps[0] == NewOps[1]` but they are NOT `RepOp`.
- **Rationale**: The old code asserted `NewOps[0] == RepOp` when `NewOps[0] == NewOps[1]` for `Sub` and `Xor`. With recursion, `NewOps[0]` could be some other value derived from `RepOp`. The patch fixes this by checking `NewOps[0] == RepOp && NewOps[1] == RepOp`.
- **Expected Issue**: Assertion failure `Precondition for non-poison assumption` or incorrect simplification ignoring poison flags.

### Strategy 3: Recursive Replacement in Absorber

- **Target**: `simplifyWithOpReplaced` where `Op` is replaced by `RepOp` in a chain, leading to a binop with an absorber constant, but `RepOp` itself is NOT the absorber constant.
- **Rationale**: The old code checked `RepOp == Absorber`. With recursion, `RepOp` might be something else, but `NewOps[0]` or `NewOps[1]` becomes the absorber constant after recursive replacement. The patch fixes this by checking `NewOps[0] == Absorber || NewOps[1] == Absorber`.
- **Expected Issue**: Missed optimization or miscompile if the assumption is violated.

### Strategy 4: MaxRecurse Limit

- **Target**: Deeply nested instruction chain that exceeds `MaxRecurse`.
- **Rationale**: The patch introduces `MaxRecurse` to limit recursion depth. If the chain is too deep, it should bail out. Testing with a deeply nested instruction chain ensures the recursion limit is respected.
- **Expected Issue**: Infinite recursion or stack overflow if `MaxRecurse` is not properly decremented or checked.

### Strategy 5: Poison Implication with Recursive Absorber

- **Target**: `simplifyWithOpReplaced` with an absorber constant where `impliesPoison(BO, Op)` is checked.
- **Rationale**: Ensure that the poison implication is correctly evaluated when the absorber constant is introduced via recursive replacement. The patch relies on `impliesPoison(BO, Op)` to be correct even when the absorber is introduced recursively.
- **Expected Issue**: Incorrect simplification if poison implication is not met.
