## Test Strategies

### Strategy 1: Truncate Sign-Wrapped Range

- **Target**: A function with a `trunc` instruction where the operand has a known range that fits exactly in the destination type's signed range.
- **Rationale**: If the range is sign-wrapped (e.g., `[127, -128]`), `getMinSignedBits()` might return a value smaller than `DestWidth` if the logic in `ConstantRange` doesn't handle sign-wrapped ranges correctly for this specific use case. However, `getMinSignedBits()` returns `BitWidth` for sign-wrapped ranges, so this is safe. But what if the range is `[255, 2]` (i.e., `[-1, 1]`)? `getMinSignedBits()` returns 2. If `DestWidth` is 1, `2 <= 1` is false. If `DestWidth` is 2, `2 <= 2` is true, so `nsw` is set. Truncating `[-1, 1]` to `i2` with `nsw` is valid because `-1` -> `11` -> `-1`, `0` -> `00` -> `0`, `1` -> `01` -> `1`. Wait, what if the range is `[254, 2]` (`[-2, 1]`)? `getMinSignedBits()` is 2. `DestWidth` is 2. `nsw` is set. `-2` -> `10` -> `-2`. Valid. What if the range is `[254, 3]` (`[-2, 2]`)? `getMinSignedBits()` is 3. `DestWidth` is 2. `nsw` is not set. Valid. The logic seems perfectly sound. Let's test edge cases like `[-8, 7]` truncated to `i4`.
- **Expected Issue**: The `trunc` instruction will be incorrectly marked with `nsw` flag, which can lead to poison values and miscompilations when the truncated value is sign-extended back.

### Strategy 2: Truncate Upper-Wrapped Range

- **Target**: A function with a `trunc` instruction where the operand has a known range that fits exactly in the destination type's unsigned range.
- **Rationale**: If the range is upper-wrapped (e.g., `[255, 2]`), `getActiveBits()` returns `BitWidth`. So `nuw` is not set. This is correct. What if the range is `[0, 3]`? `getActiveBits()` is 2. `DestWidth` is 2. `nuw` is set. Valid. What if the range is `[0, 4]`? `getActiveBits()` is 3. `DestWidth` is 2. `nuw` is not set. Valid. Let's test edge cases like `[0, 15]` truncated to `i4`.
- **Expected Issue**: The `trunc` instruction will be incorrectly marked with `nuw` flag, leading to poison values.

### Strategy 3: Truncate Negative Range to i1

- **Target**: A function with a `trunc` instruction to `i1` where the operand has a known range of `[-1, 0]`.
- **Rationale**: If the range is `[-1, 0]`, `getMinSignedBits()` is 1. If `DestWidth` is 1, `nsw` is set. Truncating `-1` to `i1` gives `1`. Sign-extending `1` from `i1` gives `-1`. So `nsw` is valid. What if the range is `[-2, 0]`? `getMinSignedBits()` is 2. `DestWidth` is 1. `nsw` is not set. Valid. Let's test truncating `[-1, 0]` to `i1`.
- **Expected Issue**: The `trunc` instruction will be incorrectly marked with `nsw` flag.

### Strategy 4: Truncate Empty Range

- **Target**: A function with a `trunc` instruction where the operand has an empty range (e.g., after a branch on a false condition).
- **Rationale**: If the range is empty, `getActiveBits()` and `getMinSignedBits()` return 0. Both `nuw` and `nsw` will be set. This is valid because the code is unreachable or the value is poison. Let's test this edge case.
- **Expected Issue**: The `trunc` instruction will be incorrectly marked with `nuw` flag.

### Strategy 5: Truncate Vector Type

- **Target**: A function with a vector `trunc` instruction where the operand has a known bounding range.
- **Rationale**: If the `trunc` instruction operates on a vector type, `getScalarSizeInBits()` correctly returns the scalar size. The range returned by `GetRange` for a vector might be a single range bounding all elements. If this bounding range satisfies the condition, `nuw` or `nsw` is set for the whole vector. This is valid. Let's test vector truncation.
- **Expected Issue**: The `trunc` instruction will be incorrectly marked with `nsw` or `nuw` flag.
