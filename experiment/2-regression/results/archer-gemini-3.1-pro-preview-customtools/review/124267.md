Bug Triggering Analysis:
The test case triggers a miscompilation in `isImpliedCondICmps` when handling `samesign` flags. The optimization attempts to deduce that if `x > y` (with `samesign ugt`) and `C >= 0`, then `(x -nsw y) <u -C` is false. However, it incorrectly uses `isImpliedCondMatchingOperands(samesign ugt, ult)`, which drops the `samesign` flag and checks if `ugt` implies `ult` is false. Since `ugt` does imply `ult` is false, it returns `false`, leading the optimization to incorrectly deduce that `(x -nsw y) <u -C` is false. But `(x -nsw y) <u -C` can be true! For example, if `x = 2147483647` and `y = 0`, `x >u y` is true and they have the same sign. `x - y = 2147483647`. `-1` as unsigned is `4294967295`. `2147483647 <u 4294967295` is TRUE. The optimization incorrectly replaces the `select` with `%add`, which overflows `nsw` and becomes `poison`, while the original code correctly returns `-2147483648`.

Fix Weakness Analysis:
The weakness in the fix is that it introduces `CmpPredicate::getMatching` to handle `samesign` for `SGT` and `SGE` predicates, but it relies on `isImpliedCondMatchingOperands` to check the implication between the LHS and RHS predicates. `isImpliedCondMatchingOperands` drops the `samesign` flag by calling `.getPredicate()` on the `CmpPredicate` objects. This causes it to incorrectly deduce implications based purely on the base predicates (e.g., `ugt` implies `ult` is false), ignoring the fact that the RHS condition might be an unsigned comparison where the constant `-C` is treated as a large unsigned number, making the implication invalid.
