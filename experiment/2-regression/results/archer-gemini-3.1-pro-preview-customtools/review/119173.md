## Test Strategies

### Strategy 1: Shift with SCEV-constant second operand

- **Target**: A loop with a shift instruction (`shl`, `lshr`, `ashr`) where the shift amount is loop-invariant and can be proven constant by SCEV, but is not an IR constant.
- **Rationale**: This directly tests the new logic for non-`Mul` binops where the second operand is folded to a constant. Shifts are particularly sensitive to constant vs. variable second operands in cost models.
- **Expected Issue**: Without the fix, VPlan would treat the shift amount as a variable (vector), while the legacy cost model would treat it as a constant, leading to a cost mismatch assertion failure.

### Strategy 2: Mul with SCEV-constant first operand

- **Target**: A `mul` instruction in a loop where the first operand is a SCEV constant (e.g., 1) and the second is variable.
- **Rationale**: Tests the `Mul`-specific logic that explicitly checks and folds both the first and second operands.
- **Expected Issue**: Without the fix, VPlan might not fold the first operand, leading to a cost mismatch since the legacy cost model checks both operands for `Mul` (specifically for multiplication by 1).

### Strategy 3: Commutative Binop with SCEV-constant first operand

- **Target**: An `add` or `and` instruction where the *first* operand is a SCEV constant, but the second is not.
- **Rationale**: The patch explicitly only folds the second operand for non-`Mul` binops. This test ensures that the first operand of an `add` or `and` is NOT folded, matching the legacy cost model's exact behavior.
- **Expected Issue**: If the fix incorrectly folded the first operand, it would diverge from the legacy cost model, potentially causing a cost mismatch.

### Strategy 4: Div/Rem with SCEV-constant second operand

- **Target**: A `udiv` or `srem` instruction where the divisor (second operand) is a SCEV constant.
- **Rationale**: Div/Rem instructions have special speculation cost logic. Ensuring the SCEV constant folding works correctly for the divisor (second operand) is important for accurate cost modeling.
- **Expected Issue**: Cost mismatch or incorrect VPlan construction if the divisor is not recognized as a constant by VPlan but is by the legacy cost model.

### Strategy 5: Non-Binop instruction with SCEV-constant operand

- **Target**: An `icmp` or `select` instruction where one operand is a SCEV constant.
- **Rationale**: The patch restricts the SCEV folding logic to `Instruction::isBinaryOp`. This ensures we don't apply it to compares or other instructions, matching the legacy cost model.
- **Expected Issue**: If the logic was incorrectly applied to non-binops, it could lead to unexpected constant folding in VPlan that the legacy cost model doesn't perform, causing a mismatch.
