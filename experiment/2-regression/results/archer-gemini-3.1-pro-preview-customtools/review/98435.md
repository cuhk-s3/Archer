### Analysis

The patch changes `m_AllOnes`, `m_Zero`, `m_SignMask`, and `m_Not` to only allow `poison` elements inside vector splats, not `undef` elements. It also changes `simplifyAShrInst` to return the original `-1` constant to preserve `poison` elements instead of returning a new `-1` constant.

I have tested these changes by mutating existing test cases to use `undef` instead of `poison` in vector splats. The tests show that the optimizations no longer trigger when `undef` is used, which is the expected behavior after the patch.

For example, in `icmp_eq_m1_or_eq_m1_fail`, the original code with `poison` was optimized, but with `undef`, it is not optimized. This confirms that `m_AllOnes` no longer matches `undef` elements.

Similarly, `dec_mask_neg_v2i32` with `undef` is not optimized, confirming `m_SignMask` no longer matches `undef`. `not_or_or_not_2i6` with `undef` is not optimized, confirming `m_Not` no longer matches `undef`. `shl_lshr_pow2_const_case1_poison3_vec` with `undef` is not optimized, confirming `m_Zero` no longer matches `undef`. `lowmask_add_splat_poison` with `undef` is not optimized, confirming `simplifyAShrInst` no longer matches `undef`.

The patch correctly implements the intended behavior of not accepting `undef` elements in `m_AllOnes` and friends. I did not find any bugs or miscompiles introduced by this patch. The tests verify that the optimizations are correctly disabled when `undef` is present, preventing potential miscompiles that could occur if `undef` was incorrectly treated as a specific value.
