## Test Strategies

### Strategy 1: Integer Overflow Semantics Mutation

- **Target**: Create test cases with integer overflow where evaluation order matters
- **Rationale**: The transformation changes evaluation order from ((A1 * B1) + (A2 * B2)) * C to (A1 * C * B1) + (A2 * C * B2). For integers, this can change overflow behavior when intermediate results overflow but final results don't (or vice versa).
- **Expected Issue**: Different execution results between original and transformed code due to integer overflow differences

### Strategy 2: Fast-Math Flag Preservation Mutation

- **Target**: Test integer operations with nuw/nsw flags and floating-point operations with various fast-math flag combinations
- **Rationale**: The patch preserves fast-math flags for FP via CreateFMulFMF but doesn't preserve nuw/nsw flags for integers via CreateMul. This could lose optimization constraints or enable unsafe transformations.
- **Expected Issue**: Loss of nuw/nsw flags for integers or incorrect fast-math flag propagation for FP

### Strategy 3: Type Consistency Mutation

- **Target**: Create expressions with mixed types or vector types of different widths
- **Rationale**: The code checks I.getType()->isIntOrIntVectorTy() at the root but doesn't verify all operations in the tree have consistent types. Mixed types could cause incorrect limit selection or code generation.
- **Expected Issue**: Incorrect transformation, crashes, or wrong limit application

### Strategy 4: Non-BinaryOperator Operand Mutation

- **Target**: Test add/fadd operations with non-binary-operator operands (constants, loads, phis)
- **Rationale**: The code assumes add/fadd operands are BinaryOperator via isa<BinaryOperator>(). This could reject valid patterns or crash if the assumption fails.
- **Expected Issue**: Incorrect rejection of valid patterns or assertion failures

### Strategy 5: Vector Type Semantics Mutation

- **Target**: Test vector operations with poison/undef lanes and different vectorization factors
- **Rationale**: Vector reassociation may have different semantics for poison propagation. The transformation could change which lanes become poison.
- **Expected Issue**: Different poison/undef behavior between original and transformed code

### Strategy 6: Instruction Deletion Safety Mutation

- **Target**: Create complex expression trees where instructions get deleted during transformation
- **Rationale**: The commit mentions a fix for accessing type of deleted instruction. Complex transformations could still have similar issues.
- **Expected Issue**: Use-after-free, crashes, or incorrect type queries during transformation

### Strategy 7: Signed Integer Edge Cases Mutation

- **Target**: Test signed integer multiplication with overflow at different points in evaluation
- **Rationale**: Signed integer overflow is undefined behavior in C/C++ but defined as two's complement wrap in LLVM. The transformation could change which multiplications overflow.
- **Expected Issue**: Different signed overflow behavior or missed optimization due to overflow concerns

### Strategy 8: One-Use Constraint Edge Mutation

## Bugs Found

### Bug #1

**Analysis:**
The test shows a potential issue with integer overflow semantics. The transformation changes evaluation order from ((inv1 * val1) + (inv2 * val2)) * inv_factor to (inv1 * inv_factor * val1) + (inv2 * inv_factor * val2). For integer arithmetic with overflow (wrap-around), this can produce different results. Example: inv1 = 2^63-1 (max i64), inv_factor = 2, val1 = 2. Original: (max_i64 * 2) overflows to -2, then + ... Transformed: (max_i64 * 2 * 2) = (max_i64 * 4) which overflows differently. This is a real bug because integer multiplication is not associative when overflow is considered.

## Agent Review

## Bug Triggering Analysis\n\nThis test case exposes a critical bug in the `hoistMulAddAssociation` transformation for integer operations. The bug is related to **integer overflow semantics**.\n\n**Why the test triggers the bug:**\n1. The transformation changes evaluation order from `((inv1 * val1) + (inv2 * val2)) * inv_factor` to `(inv1 * inv_factor * val1) + (inv2 * inv_factor * val2)`.\n2. For integer arithmetic with wrap-around overflow (two's complement), this transformation is **not valid** because integer multiplication is not associative when overflow is considered.\n3. Concrete example: `inv1 = 9223372036854775807` (max i64), `inv_factor = 2`, `val1 = 2`\n   - Original: `(max_i64 * 2)` overflows to `-2`, then `+ ...`, `* 2`\n   - Transformed: `(max_i64 * 2 * 2)` = `(max_i64 * 4)` which overflows differently\n4. The transformation assumes integer multiplication is associative, which is only true without overflow. With overflow (wrap-around), the evaluation order matters.\n\n## Fix Weakness Analysis\n\nThe fix has several weaknesses:\n\n1. **Missing overflow consideration**: The transformation doesn't check for `nuw`/`nsw` flags that would guarantee no overflow. Integer operations without these flags can overflow, making the transformation unsafe.\n\n2. **Flag preservation issue**: Even when the original multiplication has `nuw`/`nsw` flags, the hoisted multiplications (`factor.op.mul`) don't preserve these flags (uses `CreateMul` without flags instead of `CreateMul` with appropriate flags).\n\n3. **Type-based limit selection**: The limit is selected based on root instruction type (`I.getType()->isIntOrIntVectorTy()`), but the safety of the transformation depends on overflow semantics, not just type.\n\n4. **No overflow safety check**: The code should check if integer operations can overflow (lack of `nuw`/`nsw` flags) and either:\n   - Skip the transformation\n   - Preserve evaluation order constraints\n   - Add overflow checks\n\n**Expected impact**: Different execution results between original and transformed code when integer overflow occurs at different points in the evaluation.
