## Test Strategies

### Strategy 1: Mixed undef/poison vectors

- **Target**: Vector constants with both undef and poison elements
- **Rationale**: Test that cstval_pred_ty correctly distinguishes between undef and poison elements (should ignore poison but not undef)
- **Expected Issue**: Patterns might incorrectly match vectors with undef elements or fail to match vectors with only poison elements

### Strategy 2: Previously correct m_Not usage

- **Target**: Existing code using m_Not with vector constants containing undef elements
- **Rationale**: The stricter m_Not semantics might break previously valid optimizations that relied on undef elements being ignored
- **Expected Issue**: Optimizations no longer firing when they should, potentially missing performance opportunities

### Strategy 3: Scalable vector edge cases

- **Target**: Scalable vectors and zero-element vectors
- **Rationale**: The code has special handling for scalable vectors (returns false) and asserts non-zero elements
- **Expected Issue**: Incorrect behavior with scalable vectors or crashes with zero-element vectors

### Strategy 4: isElementWiseEqual semantics

- **Target**: Constant::isElementWiseEqual with vectors containing undef elements
- **Rationale**: Changed from checking UndefValue to PoisonValue might change equality semantics for vectors with undef elements
- **Expected Issue**: Different equality results for vectors with undef vs poison elements

### Strategy 5: Sign mask with undef elements

- **Target**: Sign mask constants with undef elements created via constant expressions
- **Rationale**: Test if m_SignMask() correctly rejects vectors with undef elements in InstCombine transformations
- **Expected Issue**: Transformation might incorrectly apply to vectors with undef elements or fail to apply to vectors with only poison elements

### Strategy 6: API consistency

- **Target**: All m_APInt* variants (m_APInt, m_APIntAllowUndef, m_APIntForbidUndef)
- **Rationale**: Ensure consistency between different APInt matchers after the semantic change
- **Expected Issue**: Inconsistent behavior between m_APInt (default forbids undef) and cstval_pred_ty matchers (now also forbid undef)
