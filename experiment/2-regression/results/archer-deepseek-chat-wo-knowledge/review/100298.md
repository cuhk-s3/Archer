### Bug #2

**Analysis:**
This is a real bug! The optimization incorrectly simplified the select. The issue is: when the condition is icmp eq %x, 0, and the true value is shufflevector(%x, undef, <1,0>), the optimization thinks that when the condition is true (element-wise), %x equals 0, so shufflevector(0, undef, <1,0>) = 0. Therefore, both select arms equal 0 when condition is true, and when condition is false, the false arm is %x. So it thinks select(cond, x10, x) = x. But this is wrong because the condition is element-wise, and shufflevector swaps elements. When x = <0,2>, cond = <true, false>, x10 = <2,0>. The select chooses element 0 from x10 (2) and element 1 from x (2), giving <2,2>. The optimization incorrectly assumed the relationship holds element-wise independently, but shufflevector creates cross-element dependencies.

## Agent Review

## Bug Triggering Analysis

The test case `vec_select_no_equivalence` exposes a real bug in the optimization. The bug occurs when:

1. **Condition**: `icmp eq <2 x i32> %x, zeroinitializer` (element-wise comparison)
2. **True value**: `shufflevector <2 x i32> %x, <2 x i32> undef, <2 x i32> <i32 1, i32 0>` (swaps elements)
3. **False value**: `%x`

The optimization incorrectly transforms `select(cond, shufflevector(x, undef, <1,0>), x)` to just `x`. This is wrong because:

- When `cond` is true for an element (i.e., `x[i] == 0`), the true arm for that element is `shufflevector(x, undef, <1,0>)[i]`.
- For element 0: if `x[0] == 0`, true arm = `x[1]` (from shuffle)
- For element 1: if `x[1] == 0`, true arm = `x[0]` (from shuffle)

The optimization assumes that when `cond[i]` is true, both select arms for element `i` are equal (both 0). But shufflevector creates cross-element dependencies: the true arm for element `i` depends on `x[j]` where `j` is the shuffle mask index.

**Counterexample**: `x = <0, 2>`
- `cond = <true, false>` (element 0 is 0, element 1 is not 0)
- `x10 = shuffle(<0,2>, undef, <1,0>) = <2,0>`
- `select(cond, x10, x) = select(<true, false>, <2,0>, <0,2>) = <2,2>`
- But optimization returns `x = <0,2>`

## Fix Weakness Analysis

The weakness in the fix is that `hasAffectedValue` and `findValuesAffectedByCondition` don't properly handle **cross-element dependencies** in vector operations like `shufflevector`. The optimization correctly identifies that `x` is affected by the condition, and that `x10` depends on `x`. When the condition is true (element-wise), it knows `x == 0` (element-wise). But it incorrectly assumes this implies `shufflevector(x, ...) == 0` (element-wise), which is false for shuffles that mix elements.

The fix needs to account for:
1. **Vector operations with cross-element dependencies**: shufflevector, extractelement, insertelement
2. **Element-wise vs. whole-vector relationships**: The condition provides element-wise information, but shufflevector creates cross-element relationships
3. **Mask-aware analysis**: Need to track which elements of the result depend on which elements of the input

This bug demonstrates the **Vector Edge Cases** strategy: the optimization fails for vector operations that don't preserve element-wise relationships.
