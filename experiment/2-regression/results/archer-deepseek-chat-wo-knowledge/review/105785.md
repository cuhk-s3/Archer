## Test Strategies

### Strategy 1: Type Compatibility Testing

- **Target**: Return type of ucmp/scmp intrinsics
- **Rationale**: The fix assumes i8 return type but ucmp/scmp could have different return types. Using ConstantInt::getSigned() with -1 for non-i8 types needs verification.
- **Expected Issue**: Incorrect constant creation or type mismatch errors

### Strategy 2: Signed/Unsigned Confusion Testing

- **Target**: Mixed signed/unsigned comparisons in constraint context
- **Rationale**: The constraint system has signed and unsigned subsystems. scmp should use signed predicates but if values are known non-negative, they might be treated as unsigned.
- **Expected Issue**: Wrong folding due to signedness confusion

### Strategy 3: Complex Operand Testing

- **Target**: Operands of ucmp/scmp that can't be decomposed
- **Rationale**: The constraint system can't decompose all expressions (e.g., >64-bit values, complex GEPs). The fix should handle these gracefully.
- **Expected Issue**: Crash or incorrect optimization when decomposition fails

### Strategy 4: Dominance and Execution Testing

- **Target**: ucmp/scmp in complex control flow
- **Rationale**: The fix adds ucmp/scmp to worklist without checking if they're guaranteed to execute (unlike assume). This could lead to incorrect optimizations.
- **Expected Issue**: Incorrect folding when intrinsic might not execute

### Strategy 5: Constraint System Saturation Testing

- **Target**: Multiple ucmp/scmp with many variables
- **Rationale**: The constraint system has row limits. Adding many ucmp/scmp checks could exceed limits or cause performance issues.
- **Expected Issue**: Performance degradation or missed optimizations

### Strategy 6: Poison Propagation Testing

- **Target**: ucmp/scmp with poison/undef operands
- **Rationale**: The fix should handle poison/undef values correctly according to LLVM semantics.
- **Expected Issue**: Incorrect poison propagation or optimization

### Strategy 7: Vector Type Testing

- **Target**: Vector versions of ucmp/scmp
- **Rationale**: The fix might not handle vector types correctly (though ucmp/scmp are typically scalar).
- **Expected Issue**: Crash or incorrect optimization with vector types

## Fix Weakness Analysis

The fix appears robust for the tested scenarios. However, potential weaknesses remain:

1. **Edge case with i1 return type**: I tried to test scmp with i1 return type but got a compilation error. The fix uses `ConstantInt::getSigned(I->getType(), -1)` which would fail for i1 since -1 can't be represented. This could be a compile-time error rather than a runtime bug.

2. **Order of checks in checkAndReplaceCmp()**: The function checks GT, LT, EQ in that order. If multiple could be true (e.g., with poison), it returns the first match. This is correct since GT, LT, EQ are mutually exclusive for non-poison values.

3. **Missing execution guarantee check**: Unlike `assume` intrinsics, `ucmp`/`scmp` are added to worklist without checking if they're guaranteed to execute. This is correct because they're checks, not facts - we only need to check them if they execute.

The fix successfully adds support for `ucmp`/`scmp` intrinsics in ConstraintElimination without introducing obvious bugs in the tested scenarios.
