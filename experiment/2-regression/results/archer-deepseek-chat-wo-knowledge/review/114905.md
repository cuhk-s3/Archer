## Test Strategies

### Strategy 1: FPPrecisionDifference

- **Target**: Transform fadd repeated additions to fmul with scale
- **Rationale**: Repeated FP addition may round differently than multiplication due to different rounding steps
- **Expected Issue**: Different numerical results due to rounding differences between a+a+a+a and a*4.0

### Strategy 2: IntegerOverflowSemantics

- **Target**: Transform signed add repeated additions to mul with scale
- **Rationale**: nsw/nuw flags may not be preserved, and overflow behavior differs between add (UB on overflow) and mul (defined wrap)
- **Expected Issue**: Different overflow behavior or missed optimization opportunities with overflow flags

### Strategy 3: FMFPropagation

- **Target**: Ensure fast-math flags are correctly propagated through transformation
- **Rationale**: nnan, nsz, reassoc flags affect legality of transformations from repeated adds to mul
- **Expected Issue**: Incorrect FMF on transformed operations or missed optimizations due to flag mismatches

### Strategy 4: XorPoisonSemantics

- **Target**: Xor transformation with poison/undef operands
- **Rationale**: poison xor poison is poison, not 0, and undef xor undef is undef, not 0
- **Expected Issue**: Incorrect poison/undef propagation leading to miscompilation

### Strategy 5: FPMinMaxNaN

- **Target**: Repeated FP min/max operations with NaN operands
- **Rationale**: min(NaN, x) and max(NaN, x) may not be identity operations depending on FP semantics and fast-math flags
- **Expected Issue**: Different NaN propagation behavior in optimized vs unoptimized code

### Strategy 6: ConstantExprFolding

- **Target**: Constant folding path for repeated constants
- **Rationale**: Constant folding may create different constant expressions with different semantics or performance characteristics
- **Expected Issue**: Different constant values or expression forms that affect downstream optimizations

### Strategy 7: VectorTypeMismatch

- **Target**: emitReusedOps with heterogeneous types or type conversions
- **Rationale**: Creating scale vector assumes uniform types, but values may have different types due to casts or promotions
- **Expected Issue**: Type mismatch or compilation failure when constructing scale vectors

### Strategy 8: MixedRepetitionEdgeCases

- **Target**: Combinations of values with different repetition counts and SameScaleFactor logic
- **Rationale**: Complex cases with varying repetition counts may expose bugs in SameValuesCounter and emitReusedOps logic
- **Expected Issue**: Incorrect scaling factors applied or missed optimization opportunities

### Strategy 9: TrackedValueCorruption

- **Target**: TrackedVals/TrackedToOrig mapping validity
- **Rationale**: Other transformations during vectorization may invalidate these mappings leading to incorrect scaling
- **Expected Issue**: Incorrect scaling applied to wrong values due to stale tracking information

### Strategy 10: ExclusionBoundaryCases

- **Target**: Operations near exclusion boundaries (Mul/FMul/FMulAdd edge cases)
- **Rationale**: May incorrectly include/exclude certain operation kinds or have subtle interactions with other optimizations
- **Expected Issue**: Misapplied optimization for excluded operations or missed opportunities for similar transformations

## Bugs Found

### Bug #1

**Analysis:**
This is a real bug! The original code with add nsw has undefined behavior on overflow (1073741824 + 1073741824 overflows). The transformed code with mul (without nsw) produces a wrapped value. The optimization incorrectly transforms add nsw to mul without preserving the nsw flag, changing the semantics from UB to defined wrap behavior.

**Original LLVM IR:**
```llvm
target datalayout = "e-m:e-i64:64-f80:128-n8:16:32:64-S128"
target triple = "x86_64--"

define i32 @test_integer_overflow_nsw(i32 %a) {
  %1 = add nsw i32 %a, %a
  %2 = add nsw i32 %1, %a
  %3 = add nsw i32 %2, %a
  %4 = add nsw i32 %3, %a
  %5 = add nsw i32 %4, %a
  %6 = add nsw i32 %5, %a
  %7 = add nsw i32 %6, %a
  ret i32 %7
}
```
