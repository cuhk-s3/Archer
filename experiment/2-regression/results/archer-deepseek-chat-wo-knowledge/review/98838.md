## Test Strategies

### Strategy 1: Circular dependency chain

- **Target**: Create circular use-def chains between instructions being vectorized
- **Rationale**: The recursive deletion logic might not handle circular dependencies correctly, potentially deleting instructions that are still needed or causing infinite loops.
- **Expected Issue**: Incorrect deletion of live instructions or compiler crash due to infinite recursion.

### Strategy 2: Side-effect instruction operand

- **Target**: Include instructions with side effects (volatile, atomic, calls) as operands to vectorizable instructions
- **Rationale**: wouldInstructionBeTriviallyDead might incorrectly mark side-effecting instructions as dead when they have single users.
- **Expected Issue**: Premature deletion of side-effecting instructions, changing program semantics.

### Strategy 3: Multi-tree-entry instruction

- **Target**: Create instructions that appear in multiple tree entries through MultiNodeScalars
- **Rationale**: The check Entry->VectorizedValue == OpI might miss cases where an operand appears in multiple entries but not as VectorizedValue.
- **Expected Issue**: Incorrect deletion of instructions still needed by other parts of the vectorization tree.

### Strategy 4: Poison propagation in reductions

- **Target**: Create reduction patterns where not all users are in UserIgnoreList
- **Rationale**: Replacing uses with poison might affect users outside the reduction tree, changing poison propagation semantics.
- **Expected Issue**: Incorrect poison propagation or miscompilation due to changed semantics.

### Strategy 5: Exception handling region

- **Target**: Place vectorizable instructions within exception handling regions
- **Rationale**: Instructions in EH regions might have special constraints on deletion that the algorithm doesn't consider.
- **Expected Issue**: Invalid IR after deletion or compiler crash.

### Strategy 6: Lifetime intrinsic users

- **Target**: Add llvm.lifetime.start/end intrinsics as users of vectorizable instructions
- **Rationale**: The deletion logic might not properly handle lifetime intrinsics, which have special semantics.
- **Expected Issue**: Incorrect memory lifetime analysis or compiler crash.

### Strategy 7: Cross-block dependencies

- **Target**: Create dependencies between instructions in different basic blocks
- **Rationale**: The algorithm assumes local processing but might miss cross-block dependencies when determining deadness.
- **Expected Issue**: Incorrect deletion of instructions with cross-block uses.

### Strategy 8: Metadata/dbg.value users

- **Target**: Add debug info metadata users to vectorizable instructions
- **Rationale**: salvageDebugInfo might not handle all debug info scenarios when operands are nulled out.
- **Expected Issue**: Debug info corruption or compiler crash.

### Strategy 9: Non-instruction users

- **Target**: Create uses of vectorizable instructions by non-instructions (constants, metadata, etc.)
- **Rationale**: The code assumes all users are instructions for deletion checks.
- **Expected Issue**: Assertion failure or incorrect deletion due to unhandled user types.

### Strategy 10: Instruction without parent during processing

- **Target**: Create scenario where instruction loses parent during vectorization process
- **Rationale**: The destructor handles missing parents, but other code might assume instructions have parents.
- **Expected Issue**: Null pointer dereference or incorrect behavior when removing instructions.
