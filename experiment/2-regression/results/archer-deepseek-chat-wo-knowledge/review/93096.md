## Test Strategies

### Strategy 1: EmptyRangeTrunc

- **Target**: Create trunc instructions where source has empty constant range
- **Rationale**: ConstantRange::getActiveBits() returns 0 for empty sets, which could incorrectly infer flags when range is empty but not impossible
- **Expected Issue**: Incorrectly inferring nsw/nuw flags for impossible values

### Strategy 2: WrappedRangeTrunc

- **Target**: Create trunc instructions where source has wrapped constant ranges
- **Rationale**: getActiveBits() on wrapped ranges might not capture the actual value constraints
- **Expected Issue**: Overly optimistic flag inference leading to miscompilation

### Strategy 3: ZeroValueTrunc

- **Target**: Create trunc of zero value to various widths
- **Rationale**: getActiveBits() returns 0 for zero, which is ≤ any DestWidth, potentially inferring nuw incorrectly
- **Expected Issue**: Incorrect nuw flag for zero when destination width is 0 (invalid) or other edge cases

### Strategy 4: VectorElementMismatch

- **Target**: Create trunc between vectors with different element counts
- **Rationale**: getScalarSizeInBits() might not handle element count mismatches
- **Expected Issue**: Crash or incorrect flag inference for malformed trunc

### Strategy 5: UndefSourceTrunc

- **Target**: Create trunc with undef source operand
- **Rationale**: Undef handling with UndefAllowed=false returns full range, which might be too conservative
- **Expected Issue**: Missing optimization opportunities or incorrect flag inference

### Strategy 6: BoundaryValueTrunc

- **Target**: Create trunc with source values at exact power-of-two boundaries
- **Rationale**: getActiveBits() returns exact bit width for values like 255 (i8 max), which equals DestWidth for i8→i8 trunc
- **Expected Issue**: Incorrect flag inference for boundary cases

### Strategy 7: NegativeValueActiveBits

- **Target**: Create trunc of negative values with nuw flag inference
- **Rationale**: getActiveBits() on negative values returns bits for unsigned representation, not relevant for signed semantics
- **Expected Issue**: Incorrect nuw flag inference for negative values

### Strategy 8: PartialFlagInference

- **Target**: Create trunc where only one flag exists but the other could be inferred
- **Rationale**: Early return checks both flags, might miss opportunities to add missing flag
- **Expected Issue**: Missing optimization when only one flag is present

## Bug Triggering Analysis\n\nThe test case `test_empty_range_bug` exposes a potential issue with the fix for inferring nsw/nuw flags on trunc instructions. The issue is related to the **EmptyRangeTrunc** strategy identified in Phase 1.\n\n**Key Issue**: When the source value has an empty constant range (contradictory constraints: x == 256 && x < 256), `ConstantRange::getActiveBits()` returns 0 for empty sets. The fix checks `Range.getActiveBits() <= DestWidth`, which evaluates to `0 <= 8` (true), potentially inferring nuw flag incorrectly for impossible values.\n\n**Why this is problematic**:\n1. Empty range means no valid value satisfies the constraints\n2. The path is unreachable (dead code)\n3. Inferring flags on instructions in dead code could affect other optimizations that assume the flags are valid\n4. While the path is unreachable, incorrect flag inference could still cause issues in subsequent passes\n\n## Fix Weakness Analysis\n\nThe fix's weakness is in not handling empty ranges specially. The methods `getActiveBits()` and `getMinSignedBits()` return 0 for empty ranges, which satisfies the `<= DestWidth` condition. The fix should either:\n1. Check for empty ranges early and return false\n2. Or ensure that flag inference is safe for empty ranges (flags on dead code might be harmless but could confuse other passes)\n\n**The test demonstrates**: Even though the transformation is verified as correct by alive2 (because the path is unreachable), the flag inference logic is making decisions based on mathematically incorrect premises (treating empty range as having 0 active bits). This could lead to issues in more complex scenarios where the empty range analysis is less obvious.
