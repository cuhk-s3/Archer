## Test Strategies

### Strategy 1: Denormal Mode Violation

- **Target**: Function denormal mode attribute
- **Rationale**: The fix requires IEEE denormal mode for correctness. Non-IEEE denormal modes (positive-zero, preserve-sign, dynamic) could cause incorrect transformations when dealing with subnormal values.
- **Expected Issue**: Incorrect transformation when denormal mode is not IEEE, leading to wrong comparison results for subnormal values.

### Strategy 2: Infinity Analysis Gap

- **Target**: isKnownNeverInfinity analysis
- **Rationale**: The fix relies on isKnownNeverInfinity to prove X or Y is not infinity. If this analysis is incomplete or returns false positives, the transformation could incorrectly apply when X and Y are both infinities of the same sign.
- **Expected Issue**: Incorrect transformation when both X and Y are infinities of same sign, producing NaN from fsub but transformed comparison doesn't account for NaN.

### Strategy 3: Fast Math Flag Interaction

- **Target**: FSub fast math flags
- **Rationale**: The fix checks LHSI->hasNoNaNs() and LHSI->hasNoInfs() but doesn't check the FCmp's fast math flags. If FCmp has different flags than FSub, the transformation might be invalid.
- **Expected Issue**: Incorrect transformation when FCmp and FSub have mismatched fast math flags, particularly regarding NaN handling.

### Strategy 4: Vector Element Analysis

- **Target**: Vector types with mixed infinity patterns
- **Rationale**: For vector types, isKnownNeverInfinity might not correctly handle cases where some elements are infinity and others aren't. The transformation assumes uniform behavior across all vector elements.
- **Expected Issue**: Incorrect transformation for vector types with mixed infinity patterns across elements.

### Strategy 5: Depth Limitation

- **Target**: isKnownNeverInfinity depth parameter
- **Rationale**: The fix uses Depth=0 for isKnownNeverInfinity, which might not be sufficient to prove non-infinity for complex expressions. This could cause false negatives where transformation should apply but doesn't.
- **Expected Issue**: Missed optimization opportunity for complex expressions that are provably non-infinity with deeper analysis.

### Strategy 6: Predicate Grouping Logic

- **Target**: Switch statement predicate grouping
- **Rationale**: The fix groups predicates differently: some require infinity checks, others don't. If the logic for which predicates need infinity checks is incorrect, transformations could be incorrectly allowed or blocked.
- **Expected Issue**: Incorrect transformation for predicates that should/shouldn't have infinity checks.

### Strategy 7: Zero Pattern Matching

- **Target**: m_AnyZeroFP pattern
- **Rationale**: The fix uses match(RHSC, m_AnyZeroFP()) but doesn't check for negative zero. While fcmp ignores sign of zero, the transformation might have subtle issues with signed zeros in certain contexts.
- **Expected Issue**: Incorrect handling of signed zeros in edge cases.

### Bug #1

Found a bug! The original code with nnan fsub and infinity inputs triggers undefined behavior (poison value), but the transformed code returns false without UB. This is a real bug - the transformation changes poison/UB semantics. The fix doesn't properly handle the interaction between FSub's nnan flag and FCmp's lack of nnan flag when infinity values are involved.

   - When FSub has `nnan`/`ninf` flags but FCmp doesn't
   - With infinity inputs (`inf - inf = NaN` with nnan flag = poison)
   - Transformation incorrectly changes undefined behavior (poison) to defined behavior
   - Affects multiple predicates (UGT, OEQ, etc.) in the first group requiring infinity checks

3. **Verified Other Aspects of the Fix**:
   - Denormal mode handling is correct (requires IEEE denormal mode)
   - Vector element analysis is conservative but correct
   - Predicate grouping logic appears mathematically sound
   - Zero pattern matching correctly handles both positive and negative zero
   - Depth limitation (Depth=0) might miss optimization opportunities but is safe

### Key Findings

**Confirmed Bug**: The fix doesn't check that FCmp has compatible fast math flags with FSub. When FSub has `nnan`/`ninf` flags but FCmp doesn't, and inputs could produce NaN (like `inf - inf`), the transformation changes poison/UB semantics to defined values.

**Fix Weakness**: The transformation should either:
1. Require FCmp to have at least the same fast math flags as FSub
2. Handle flag mismatches more carefully to preserve poison semantics
3. Not apply when there's a flag mismatch that could affect semantics

**Other Issues**: While not bugs, the fix has conservative aspects that might miss optimization opportunities:
- Depth=0 for `isKnownNeverInfinity` might not prove non-infinity for complex expressions
- Conservative vector handling (no transformation if any lane could have infinity)
- Denormal mode check applies to all predicates, might be overly conservative
