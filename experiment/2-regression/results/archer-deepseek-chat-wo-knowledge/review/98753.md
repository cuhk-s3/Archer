## Test Strategies

### Strategy 1: PoisonSubstitution

- **Target**: Test substitution that introduces or propagates poison
- **Rationale**: If a or b is poison, or substitution creates poison (e.g., division by zero), the optimization might incorrectly assume the expression simplifies
- **Expected Issue**: Optimization incorrectly simplifies expression that should remain unchanged due to poison semantics

### Strategy 2: SideEffectPreservation

- **Target**: Test with expressions containing side effects (calls, volatile loads, etc.)
- **Rationale**: Substitution might eliminate or duplicate side effects
- **Expected Issue**: Optimization changes observable behavior by removing/duplicating side effects

### Strategy 3: TypeSafety

- **Target**: Test type mismatches in equality comparisons
- **Rationale**: icmp eq allows some type conversions; substitution might not be type-safe
- **Expected Issue**: Type mismatch causing incorrect code generation

### Strategy 4: VectorLaneIsolation

- **Target**: Test vector equality with per-element substitution
- **Rationale**: Vector icmp eq returns a vector of booleans; substitution should respect lane independence
- **Expected Issue**: Cross-lane contamination in vector operations

### Strategy 5: RecursionDepth

- **Target**: Create deeply nested expressions that exceed recursion limits
- **Rationale**: Insufficient recursion handling might cause incomplete simplification or crashes
- **Expected Issue**: Incorrect simplification or compiler crash

### Strategy 6: FPEquality

- **Target**: Test floating-point equality with NaN values
- **Rationale**: FP icmp eq has special NaN semantics; substitution assumptions might be invalid
- **Expected Issue**: Incorrect optimization with NaN values

### Strategy 7: PointerProvenance

- **Target**: Test pointer equality with different provenance
- **Rationale**: Equal pointers don't guarantee interchangeable use in all contexts
- **Expected Issue**: Optimization that violates pointer provenance rules

### Strategy 8: IntrinsicSafety

- **Target**: Test other intrinsics besides llvm.is.constant that shouldn't be folded
- **Rationale**: Other intrinsics might have similar issues with assumption-based folding
- **Expected Issue**: Incorrect folding of other intrinsics based on equality assumptions

## Bugs Found

### Bug #1

**Analysis:**
This is a real bug. The original code triggers UB (division by zero) when a = 0, b = 0, but the optimized code doesn't. The optimization incorrectly simplified sdiv i32 %a, %a to 1, which is only valid when a != 0. When a == b == 0, the expression should be poison/UB, but the optimization eliminated the division entirely. This violates poison semantics.

**Original LLVM IR:**
```llvm
define i1 @test_poison_substitution(i32 %a, i32 %b) {
  %cmp = icmp eq i32 %a, %b
  %div = sdiv i32 %a, %a
  %sub = sub i32 %div, %b
  %cmp2 = icmp eq i32 %sub, 0
  %and = and i1 %cmp, %cmp2
  ret i1 %and
}
```
