## Test Strategies

### Strategy 1: NaN Sign Bit Propagation

- **Target**: Test sign bit inference when value could be NaN
- **Rationale**: The knownNot() method infers sign bit only when isKnownNever(fcNan). Need to test cases where NaN determination is incorrect or borderline, especially through operations that might preserve/propagate NaN.
- **Expected Issue**: Incorrect sign bit inference when value is actually NaN or when NaN handling interacts with sign inference.

### Strategy 2: Zero Multiplication Sign

- **Target**: Test FMul sign inference with zeros
- **Rationale**: 0 * x = 0 with sign being XOR of operand signs, but special cases with NaN/infinity. The FMul sign inference assumes same sign -> positive, different -> negative, but needs to handle zeros correctly.
- **Expected Issue**: Incorrect sign bit for -0 * +0, -0 * positive, or cases where 0 * inf = NaN.

### Strategy 3: Denormal Flushing to Signed Zero

- **Target**: Test sign inference with denormal flushing modes
- **Rationale**: When denormals flush to zero with sign preservation (PreserveSign mode), negative denormals become -0. This affects sign bit inference for operations that produce denormals.
- **Expected Issue**: Incorrect sign bit inference for operations that produce denormals in PreserveSign mode.

### Strategy 4: Vector Mixed Signs with Operations

- **Target**: Test vector operations with mixed signs
- **Rationale**: Current vector constant handling doesn't set SignBit for mixed signs, but operations on such vectors might have deterministic sign (e.g., fabs on vector with mixed signs produces all positive).
- **Expected Issue**: Missing sign bit inference for vector operations where result sign is deterministic despite mixed input signs.

### Strategy 5: Min/Max Intrinsic Edge Cases

- **Target**: Test complex sign inference logic for min/max intrinsics
- **Rationale**: The logic has multiple conditions involving zero handling and NaN. Need to test maxnum/minnum/maximum/minimum with zeros, NaN, and mixed sign combinations.
- **Expected Issue**: Incorrect sign inference for maxnum/minnum/maximum/minimum with zeros, NaN, or mixed signs, particularly around the complex zero-handling condition.

### Strategy 6: Copysign Constant vs Known Sign

- **Target**: Test InstCombine copysign optimization with non-constant known signs
- **Rationale**: Patch changes from matching constant negative to using computeKnownFPSignBit(). Need to ensure optimization works for non-constant but known-negative values and doesn't incorrectly transform.
- **Expected Issue**: Missing optimization for non-constant but known-negative values, or incorrect transformation when sign bit is unknown.

## Agent Review

# Bug Triggering Analysis

This test case triggers the "NaN Sign Bit Propagation" strategy. The test checks if `fabs(x) < 0.0` can be simplified. `fabs` always returns a non-negative value or NaN (with preserved sign bit). For ordered comparison `olt`, if the result is NaN, the comparison returns false. Therefore, the result should always be false. The optimization correctly simplifies this to `ret i1 false`.

# Fix Weakness Analysis

The fix correctly handles NaN sign bit propagation for `fabs`. The `computeKnownFPClass` function for `fabs` calls `Known.fabs()` which calls `signBitMustBeZero()`. This method sets `SignBit = false` and filters `KnownFPClasses` to `(fcPositive | fcNan)`. This correctly preserves NaN in the class set while setting sign bit to false. The `knownNot()` method also has logic to infer sign bit when not NaN, which works correctly here.

However, the weakness is that `signBitMustBeZero()` preserves `fcNan` in the class set, which is correct because NaN sign bits are preserved but not guaranteed. Callers must check `isKnownNeverNaN()` before relying on sign bit. In this case, `fcmp olt` with ordered comparison handles NaN correctly (returns false), so the optimization is safe.
