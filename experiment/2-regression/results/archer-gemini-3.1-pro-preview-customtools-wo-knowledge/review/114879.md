## Test Strategies

### Strategy 1: Select-like PHI with udiv in dominating block

- **Target**: A select-like PHI node merging a udiv result (defined before the branch) and another value.
- **Rationale**: IsAvailableOnEntry explicitly rejected scUDivExpr. Replacing it with properlyDominates allows udiv to be modeled in select-like PHIs. This tests the basic positive case of the patch.
- **Expected Issue**: The PHI node should be successfully modeled as a SCEV expression (e.g., umax/smax or select) instead of SCEVUnknown, demonstrating the expanded capability.

### Strategy 2: Select-like PHI with udiv inside conditional branch

- **Target**: A PHI node where one incoming value is a udiv instruction located inside the conditional branch, with its operands defined before the branch.
- **Rationale**: The udiv SCEV's dominance is determined by its operands. If operands dominate the PHI, the udiv SCEV dominates the PHI, even if the instruction itself is inside a conditional branch and doesn't properly dominate the PHI block. This tests if SCEVExpander correctly handles potentially trapping instructions.
- **Expected Issue**: SCEV should model the PHI, but SCEVExpander must not hoist the udiv instruction to a place where it might trap (e.g., division by zero) if the condition is false.

### Strategy 3: Select-like PHI with potential division by zero in untaken branch

- **Target**: A select-like PHI where the udiv denominator can be zero, but the branch containing the udiv is dynamically dead when the denominator is zero.
- **Rationale**: SCEV might model the udiv and SCEVExpander might try to evaluate it. If SCEVExpander doesn't check isSafeToExpand, it might trap. This specifically targets the trapping behavior mentioned in the commit message.
- **Expected Issue**: Exposes whether SCEVExpander correctly avoids expanding trapping udivs, preventing runtime crashes.

### Strategy 4: Loop with select-like PHI containing udiv used in loop exit (LCSSA)

- **Target**: A loop containing a select-like PHI with a udiv, whose result is used outside the loop, requiring LCSSA form.
- **Rationale**: The commit message mentions an issue in SCEVExpander/LCSSA (D150681) that was exposed by this patch. Testing LCSSA with udiv in select-like PHI ensures the fix and its dependencies are robust.
- **Expected Issue**: LCSSA formation or SCEV expansion might fail, crash, or generate invalid code if the udiv is not handled correctly during expansion outside the loop.

### Strategy 5: Select-like PHI with scCouldNotCompute

- **Target**: A select-like PHI where one of the incoming values might somehow evaluate to a SCEVCouldNotCompute object (if possible).
- **Rationale**: IsAvailableOnEntry safely returned false for scCouldNotCompute. However, properlyDominates calls getBlockDisposition, which has an llvm_unreachable for scCouldNotCompute. If getSCEV(LHS) or getSCEV(RHS) can ever return scCouldNotCompute, this will crash.
- **Expected Issue**: If getSCEV returns scCouldNotCompute, properlyDominates will hit an llvm_unreachable assertion, causing a compiler crash.
