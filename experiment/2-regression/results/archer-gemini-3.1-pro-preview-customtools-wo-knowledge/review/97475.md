### Strategy 1: Poison Condition with Affected Value

- **Target**: A select instruction where the condition is `undef` or `poison`, and the true/false arms contain values affected by the condition.
- **Rationale**: The optimization assumes that if the condition is true/false, the affected values have certain known bits. However, if the condition is poison, the select itself evaluates to poison. If we simplify the select arm to a constant based on the poison condition, we might replace a potentially non-poison value with a constant, or we might change the behavior if the select is later used in a way where poison is propagated differently. Wait, if the condition is poison, the select result is poison. Replacing an arm with a constant doesn't change the fact that the result is poison. But what if the condition is a vector and some elements are poison? Or what if the condition is undef? If the condition is undef, the select can choose either arm. If we simplify an arm assuming the condition is true/false, we might be wrong if the condition is undef. The commit message mentions: "but this can be generalized to handle other patterns by using SimplifyDemandedBits instead (in that case we would also have to limit to non-undef conditions)." This implies that for the current implementation (folding to a constant), undef conditions are safe. Why? Because if the condition is undef, the select can evaluate to either TrueVal or FalseVal. If we replace TrueVal with a constant C because TrueVal == C when Cond == true, and Cond is undef, then if the select chooses TrueVal, it's as if Cond == true, so TrueVal == C is valid. Wait, is it? If Cond is undef, it can take any value. If it takes 'true', then TrueVal evaluates to C. So replacing TrueVal with C is correct. What if TrueVal's value depends on Cond? No, TrueVal's value is independent of Cond, but we know that *if* Cond is true, TrueVal must be C. If Cond is undef, does TrueVal have to be C? Not necessarily. For example, Cond = (X == 10). If X is undef, Cond is undef. TrueVal is X. If Cond is true, X is 10. So we replace TrueVal with 10. The select becomes `select undef, 10, FalseVal`. If undef is chosen as true, the result is 10. But originally, if undef is chosen as true, X could be anything, so the result could be anything. Replacing it with 10 is a refinement, which is valid. So undef condition seems fine.
- **Expected Issue**: The select instruction will be incorrectly simplified to a constant, leading to incorrect execution results when the condition is poison.

### Strategy 2: Vector Select with Vector Condition and Vector Arms

- **Target**: A vector select instruction where the condition is a vector comparison, and the true/false arms contain cross-lane operations (like `shufflevector`) on the affected values.
- **Rationale**: The optimization uses `computeKnownBits` to find known bits of the select arms based on the condition. For vector types, `computeKnownBits` computes the known bits that are common to all elements of the vector. If the condition is a vector condition, `findValuesAffectedByCondition` and `computeKnownBitsFromCond` might extract known bits. Wait, `computeKnownBitsFromCond` handles vector conditions? Let's check `computeKnownBitsFromCmp`. It handles vector comparisons by computing known bits for the whole vector. But for a vector select, the condition is an element-wise selection. If `Cond` is `<2 x i1> <true, false>`, then `TrueVal` is selected for element 0, and `FalseVal` for element 1. The optimization injects `Cond` into the context for `TrueVal`. `computeKnownBitsFromCond` will assume `Cond` is true for ALL elements when evaluating `TrueVal`. But `Cond` is only true for the elements where `TrueVal` is selected! Wait. If `Cond` is a vector, `CondContext` sets `Cond` as the condition. `computeKnownBitsFromCond` will assume `Cond` is all-ones? No, `computeKnownBitsFromCond` assumes `Cond` is true. For a vector, what does it mean for `Cond` to be true? `computeKnownBitsFromContext` calls `computeKnownBitsFromCond(V, Q.CC->Cond, Known, Depth, Q, Q.CC->Invert)`. If `Cond` is a vector comparison, e.g., `X == <1, 1>`, `computeKnownBitsFromCmp` will assume `X == <1, 1>` is true. But for a vector select, `Cond` being true only applies to the elements where the condition is true. `computeKnownBitsFromCmp` does not know about element-wise truth. It assumes the entire condition is true (i.e., all elements are true). This is a major bug! If we have `select <2 x i1> %cmp, <2 x i32> %X, <2 x i32> %Y`, and `%cmp` is `%X == <1, 2>`, the optimization will assume `%X == <1, 2>` is true for ALL elements when simplifying `%X`. So it will conclude `%X` is `<1, 2>`, and replace `%X` with `<1, 2>`. But `%X` might be `<1, 3>`, so `%cmp` is `<true, false>`. The select should return `<1, 3>`'s first element (1) and `%Y`'s second element. If we replace `%X` with `<1, 2>`, the select returns `<1, 2>`'s first element (1) and `%Y`'s second element. This happens to be correct because we only care about the elements where `%cmp` is true! Wait. If we replace `%X` with `<1, 2>`, we are changing the false elements of `%X` to 2. But the select will pick `%Y` for the false elements. So the false elements of `%X` don't matter! Is this always safe? What if `computeKnownBits` uses the assumption that `%X == <1, 2>` to simplify something that cross-pollinates between lanes? For example, a horizontal add, or a shufflevector. If `%X` is used in a shufflevector, and we assume `%X == <1, 2>`, we might simplify the shufflevector incorrectly. But `hasAffectedValue` only looks through instructions that operate element-wise? Let's check `hasAffectedValue`. It looks through `any_of(I->operands(), ...)`. It doesn't restrict the opcode! It just checks if the operand is an integer or integer vector type. So it could look through a `shufflevector` or `bitcast`! If `TrueVal` is `shufflevector %X, %X, <1, 0>`, and `%cmp` is `%X == <1, 2>`. `hasAffectedValue` will return true. `computeKnownBits` will assume `%X == <1, 2>` for ALL elements. Then it will compute known bits for `shufflevector %X, %X, <1, 0>`, which will be `<2, 1>`. If it's a constant, it will replace `TrueVal` with `<2, 1>`. But wait, `%cmp` is `<true, false>`. So `%X` is `<1, 3>`. `TrueVal` is `shufflevector <1, 3>, <1, 3>, <1, 0>` which is `<3, 1>`. The select will choose element 0 of `TrueVal`, which is 3. But we replaced `TrueVal` with `<2, 1>`, so the select will choose element 0 of `<2, 1>`, which is 2! This is a miscompilation! The assumption that `Cond` is true for all elements is invalid when the operation involves cross-lane operations like `shufflevector`.
- **Expected Issue**: The select instruction will be incorrectly simplified, leading to a miscompilation where the simplified constant does not match the actual value of the arm when the condition is true/false.

### Strategy 3: Vector Select with Bitcast from Integer to Vector

- **Target**: A vector select instruction where the condition is a vector comparison, and the true/false arms contain operations that cast the vector to a scalar and back, mixing lanes.
- **Rationale**: Similar to the shufflevector case, if the condition is a scalar condition, but the select is a vector select (wait, if the condition is scalar, the select is scalar or the select is vector but condition is scalar? If condition is scalar, it applies to all elements, so it's safe). What if the condition is a vector condition, and the affected value is bitcasted to a scalar, operated on, and bitcasted back? `hasAffectedValue` only looks through integer or integer vector types. `bitcast <2 x i16> %X to i32` has an integer type. If `%cmp` is a vector comparison `<2 x i1> %X == <1, 2>`, `computeKnownBits` will assume `%X == <1, 2>` for all elements. Then it will compute known bits for the `i32` bitcast, which will be a constant. Then it will replace the `i32` value with a constant. But the vector condition is only true for some elements. Wait, if the select is a vector select, the condition must be a vector. If the arm is a vector, and it contains a bitcast from a scalar, the scalar must have been computed from the vector. If `hasAffectedValue` traverses through a bitcast from vector to scalar, it means the vector was cast to scalar, then back to vector. The scalar operation mixes lanes. `computeKnownBits` will assume the vector condition is true for all lanes, which is wrong.
- **Expected Issue**: The select instruction will be incorrectly simplified, leading to a miscompilation where the simplified constant does not match the actual value of the arm when the condition is true/false.

### Strategy 4: Vector Select with Vector Condition and Non-Element-Wise Instructions

- **Target**: A vector select instruction where the condition is a vector comparison, and the true/false arms contain vector reduction intrinsics or other non-element-wise operations on the affected values.
- **Rationale**: `hasAffectedValue` does not check if the instructions it traverses are element-wise. It just checks if the operands are integer or integer vector types. If an instruction is not element-wise (e.g., `udiv` where the divisor is a vector, or `shl` where the shift amount is a vector, wait, those are element-wise). What about `call` to an intrinsic like `llvm.vector.reduce.add`? The intrinsic returns a scalar, so it's an integer type. If the select is a scalar select, the condition is scalar, so it's safe. What if the select is a vector select? Then the condition is a vector. The arm is a vector. The arm could be a vector made by `insertelement` from a scalar, which was computed by `vector.reduce.add` on a vector `%X`. If the condition is `%X == <1, 2>`, `computeKnownBits` assumes `%X == <1, 2>` for all elements. It computes the reduction as 3. It replaces the arm with a constant vector `<3, 3>`. But the condition might only be true for element 0. So `%X` is `<1, 5>`. The reduction is 6. The select should choose element 0 of the arm, which should be 6. But it chooses element 0 of `<3, 3>`, which is 3. Miscompilation!
- **Expected Issue**: The select instruction will be incorrectly simplified, leading to a miscompilation.
