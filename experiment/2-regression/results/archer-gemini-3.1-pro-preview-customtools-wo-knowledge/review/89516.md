## Test Strategies

### Strategy 1: Commutativity of m_Add matching wrong operands

- **Target**: Create a test where the condition checks one value, but the `add` and `select` use different values that happen to match the `srem` pattern.
- **Rationale**: The `m_Add` matcher is commutative. If the `TrueVal` is `add i32 %n, %rem`, `m_Value(RemRes)` might bind to `%n` and `m_Value(Remainder)` might bind to `%rem`. Then `match(RemRes, m_SRem(m_Value(Op), m_Specific(Remainder)))` will try to match `%n` as `srem i32 %x, %rem`. If `%n` happens to be an `srem` instruction itself, this could lead to an incorrect match. However, a more direct issue is that `RemRes` is already bound in the `m_ICmp` match! Wait, `m_Value(RemRes)` in `m_Add` will overwrite the previous binding of `RemRes` if it's not `m_Specific(RemRes)`. Let's check `PatternMatch.h`: `m_Value(X)` assigns to `X`. So `match(TrueVal, m_Add(m_Value(RemRes), m_Value(Remainder)))` will overwrite `RemRes` with one of the operands of `add`. If the `add` is `add i32 %other, %rem`, `RemRes` becomes `%other` and `Remainder` becomes `%rem`. Then `match(RemRes, m_SRem(m_Value(Op), m_Specific(Remainder)))` checks if `%other` is `srem %x, %rem`. If it is, and `%rem` is a power of two, and `FalseVal == RemRes` (which is `%other`), the fold triggers. But the condition was `icmp slt %original_remres, 0`. Since `RemRes` was overwritten, the condition checked `%original_remres`, but the fold uses `%other` and `%rem`. This completely breaks the relationship between the condition and the `add`/`srem`!
- **Expected Issue**: The optimization will incorrectly fold the select to `x & (n - 1)`, which evaluates to `3` instead of `-1`.

### Strategy 2: Overwriting RemRes in m_Add

- **Target**: A select instruction where the condition is `icmp slt %unrelated, 0`, the true value is `add %srem_res, %pow2`, and the false value is `%srem_res`.
- **Rationale**: The variable `RemRes` is first bound in `match(CondVal, m_ICmp(Pred, m_Value(RemRes), m_APInt(C)))`. Then it is used in `match(TrueVal, m_Add(m_Value(RemRes), m_Value(Remainder)))`. Because `m_Value` assigns to the variable rather than checking for equality (unlike `m_Specific` or `m_Deferred`), the second `m_Value(RemRes)` overwrites the value bound in the first match. This means the `RemRes` checked in the `icmp` does not have to be the same as the `RemRes` used in the `add` and `srem`. If we construct a case where `icmp` checks an unrelated value, the fold will still apply, changing the semantics.
- **Expected Issue**: The optimization will incorrectly fold the select to `x & (n - 1)`, which evaluates to `3` instead of `-1`.
