## Test Strategies

### Strategy 1: Incorrect Disjoint Flag on Or Instruction

- **Target**: Create a test case with an `or` instruction that has multiple uses or is used in a context where it is not disjoint, but one of its uses is in a context (e.g., guarded by a branch or an assume) where the operands are disjoint. The `or` instruction should be simplified via `SimplifyDemandedBits` from the context-sensitive use.
- **Rationale**: The patch passes `SimplifyQuery` with context instruction `Q.CxtI` to `SimplifyDemandedUseBits` and `SimplifyMultipleUseDemandedBits`. When simplifying an `Or` instruction, it checks if the operands have no common bits set using `haveNoCommonBitsSet(LHSCache, RHSCache, Q)`. If true, it sets the `disjoint` flag on the `Or` instruction. However, `Q.CxtI` might be a user of the `Or` instruction (e.g., an `icmp` or a branch condition), and the `Or` instruction might have multiple uses or be executed in a context where the condition implied by `Q.CxtI` does not hold. Setting the `disjoint` flag modifies the `Or` instruction globally, which is incorrect if the disjointness only holds in the context of `Q.CxtI`.
- **Expected Issue**: The `or` instruction will be incorrectly marked as `disjoint`, leading to poison values when executed, because the disjointness was proven using context-sensitive known bits from a specific use, but the `or` instruction itself is executed unconditionally or in a broader context where the bits are not disjoint.

### Strategy 2: Incorrect Global Simplification Based on Context-Sensitive Known Bits

- **Target**: Create a test case where an instruction (e.g., `add`, `xor`, `and`) is modified in-place by `SimplifyDemandedUseBits` based on context-sensitive known bits (e.g., from an `assume` or a dominating branch), but the instruction has another use where the context does not apply.
- **Rationale**: Similar to the `disjoint` flag issue, `SimplifyDemandedUseBits` might perform other in-place modifications to instructions based on context-sensitive known bits. For example, it might convert an `add` to an `or` if it determines there are no common bits set, or it might drop `nsw`/`nuw` flags, or it might replace an operand with a constant. If these modifications are based on `Q.CxtI` but the instruction has other uses where the context does not hold, the global modification will be incorrect.
- **Expected Issue**: The `add` instruction will be incorrectly simplified to an `or` instruction, or an `or` instruction will be incorrectly simplified to an `add` instruction, or other similar simplifications will be applied globally based on context-sensitive known bits, leading to incorrect results for other uses of the instruction.

### Strategy 3: Incorrect Operand Replacement in SimplifyDemandedUseBits

- **Target**: Create a test case where an `or` instruction has a single use in a branch condition or an `assume`. The context of the branch/assume implies that the operands of the `or` are disjoint. `SimplifyDemandedBits` sets the `disjoint` flag on the `or`. If the operands are not actually disjoint at runtime, the `or` produces poison, and the branch on poison causes UB.
- **Rationale**: `SimplifyDemandedUseBits` can replace operands of an instruction if it determines that certain bits are not demanded or are known. For example, `ShrinkDemandedConstant` modifies a constant operand in-place. If this modification is based on context-sensitive known bits derived from `Q.CxtI`, it might be incorrect for other uses of the instruction. Wait, `SimplifyDemandedUseBits` only modifies the instruction if it has one use, or if it's the root. If it's the root, it's only modifying the specific use. But if it modifies an instruction deeper in the tree that has one use *from the perspective of the current traversal*, but that instruction is actually used elsewhere? No, `hasOneUse()` checks global uses. However, if an instruction has one use, its context is the same as its user's context. So context-sensitive simplification is safe if the instruction has one use. But wait, what if the instruction has one use, but it's hoisted above the context condition? The context condition only applies *at the use*. If we modify the instruction in-place, it's still executed at its original location. If the modification makes it produce poison (e.g., setting `disjoint`), and it's executed before the condition, it might trigger UB or poison if the poison is used elsewhere? No, it has one use, so the poison is only used at the use, where the condition holds. So it's safe. Wait, what if the instruction has multiple uses, but we are in `SimplifyMultipleUseDemandedBits`? `SimplifyMultipleUseDemandedBits` does NOT modify the instruction in-place. It only returns a simplified value or computes known bits. So in-place modifications only happen when `hasOneUse()` is true. If `hasOneUse()` is true, the instruction is only used at `Q.CxtI` (or its ancestors). Is it safe to modify it in-place? Yes, because its only use is under the context. Wait, what if the instruction is an `or` with one use, and we set `disjoint` on it. The `or` is executed before the branch. It produces poison. The poison is only used in the branch condition. Branch on poison is UB! So setting `disjoint` on an `or` based on context-sensitive known bits can cause UB if the `or` is executed before the context condition and its result is used in the branch condition itself.
- **Expected Issue**: The `SimplifyDemandedBits` pass will incorrectly replace an operand of an instruction with a constant or another value based on context-sensitive known bits, but the replacement will be invalid for other uses of the instruction if the instruction is not cloned.

### Strategy 4: Context-Sensitive Known Bits Leakage in SimplifyMultipleUseDemandedBits

- **Target**: Create a test case where an `or` instruction has multiple uses. One use is in a context (e.g., an `icmp` used in a branch) where `SimplifyDemandedBits` is called with `Q.CxtI` set to the `icmp`. The context implies the operands of the `or` are disjoint. The `or` is the root of the `SimplifyDemandedBits` query. The `disjoint` flag is set on the `or`, poisoning the other uses of the `or` which are not in that context.
- **Rationale**: `SimplifyMultipleUseDemandedBits` computes known bits for an instruction with multiple uses. It uses `computeKnownBitsFromContext(I, Known, Depth, Q)`. This adds context-sensitive known bits from `Q.CxtI` to the `Known` bits of the instruction `I`. Then, it might return a constant if `DemandedMask.isSubsetOf(Known.Zero | Known.One)`. This constant will be used to replace the *specific use* of `I` at `Q.CxtI`. This is actually correct, because it only replaces the use at `Q.CxtI`. Wait, is there any case where `SimplifyMultipleUseDemandedBits` modifies the instruction `I` itself? No, it only returns a `Value*` which replaces the use. What if it returns `I->getOperand(0)`? That replaces the use with `I->getOperand(0)`. That is also correct for the specific use. So `SimplifyMultipleUseDemandedBits` seems safe. Let's re-evaluate the `disjoint` flag issue. In `SimplifyDemandedUseBits`, for `Instruction::Or`: `if (!cast<PossiblyDisjointInst>(I)->isDisjoint()) { ... if (haveNoCommonBitsSet(LHSCache, RHSCache, Q)) { cast<PossiblyDisjointInst>(I)->setIsDisjoint(true); return I; } }`. Here, `I` is modified in-place. This happens even if `I` has multiple uses if `Depth == 0`! At `Depth == 0`, `DemandedMask.setAllBits()` is called, and we proceed to the `switch`. So if `I` is the root of the `SimplifyDemandedBits` query, it can have multiple uses, and we still modify it in-place by setting `disjoint`! This is a critical bug. If `I` has multiple uses, setting `disjoint` based on `Q.CxtI` (which might be just one of the uses) will poison the other uses.
- **Expected Issue**: The `SimplifyDemandedBits` pass will incorrectly simplify an instruction based on context-sensitive known bits, leading to a miscompilation where the instruction's result is used in a context where the known bits do not hold.

### Strategy 5: Incorrect In-Place Modification of Root Instruction with Multiple Uses

- **Target**: Create a test case where an instruction (e.g., `and`, `or`, `xor`) has multiple uses. One use is an `icmp`. The `icmp` is in a basic block dominated by a condition that provides known bits for the operands of the instruction. `foldICmpUsingKnownBits` calls `SimplifyDemandedBits` on the instruction with `Q.CxtI` set to the `icmp`. `SimplifyDemandedUseBits` modifies the instruction in-place (e.g., by shrinking a constant or setting `disjoint`) based on the context-sensitive known bits. This corrupts the value for the other uses of the instruction.
- **Rationale**: In `SimplifyDemandedUseBits`, if `Depth == 0`, it allows the instruction to have multiple uses (`if (Depth == 0 && !V->hasOneUse()) DemandedMask.setAllBits();`). It then proceeds to simplify the instruction. For example, in `Instruction::And`, it might do `ShrinkDemandedConstant(I, 1, DemandedMask & ~LHSKnown.Zero)`. Since `DemandedMask` is all ones, this might not do much, but what if `LHSKnown.Zero` has bits set from context-sensitive known bits? Then `ShrinkDemandedConstant` will modify the constant operand of the `And` instruction in-place! Since the `And` instruction has multiple uses, this modifies the semantics for all uses, which is incorrect because `LHSKnown.Zero` was computed using `Q.CxtI` which only applies to one use. Wait, `LHSKnown` is computed using `Q.CxtI`. If `Depth == 0`, `Q.CxtI` is the root instruction itself, or the user of the root instruction? In `InstCombinerImpl::SimplifyDemandedBits(Instruction *I, unsigned OpNo, ...)`, `Q` is passed from the caller. The caller is `InstCombinerImpl::foldICmpUsingKnownBits`, which does `SimplifyQuery Q = SQ.getWithoutDomCondCache().getWithInstruction(&I);` where `I` is the `icmp`. Then it calls `SimplifyDemandedBits(&I, 0, ..., Q)`. Inside `SimplifyDemandedBits`, it calls `SimplifyDemandedUseBits(U.get(), ..., Q)`. So `V` is `U.get()`, which is the operand of the `icmp`. `Q.CxtI` is the `icmp`. If `V` has multiple uses, `Depth == 0`, so we proceed. We compute `LHSKnown` of `V`'s operands using `Q.CxtI` (the `icmp`). These known bits might be context-sensitive (e.g., from a dominating branch). Then we use these known bits to modify `V` in-place (e.g., `ShrinkDemandedConstant`, setting `disjoint`, etc.). This modifies `V` for all its users, but the known bits only hold under the context of the `icmp`! This is a huge bug.
- **Expected Issue**: The `SimplifyDemandedBits` pass will incorrectly drop `nsw`/`nuw` flags or modify constants in an instruction with multiple uses, because it modifies the instruction in-place when `Depth == 0` even if it has multiple uses.
