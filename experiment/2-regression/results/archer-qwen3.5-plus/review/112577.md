## Test Strategies

### Strategy 1: Test abs intrinsic with INT_MIN value and reduced bitwidth

- **Target**: llvm.abs intrinsic with is_int_min_poison=true
- **Rationale**: The patch forces is_int_min_poison to false when bitwidth is reduced (lines 12598-12600). However, if the original scalar call had is_int_min_poison=true and the input can be INT_MIN at the reduced bitwidth, this change could silently suppress poison that should be generated, changing program semantics.
- **Expected Issue**: When vectorizing abs calls with is_int_min_poison=true and the minbitwidth analysis determines a smaller bitwidth is safe, the vectorized code will use is_int_min_poison=false. If any lane contains the minimum value for the REDUCED bitwidth (not the original), poison should be generated but won't be, potentially producing incorrect results instead of poison.

### Strategy 2: Test smin/smax/umin/umax with mixed sign-bit patterns

- **Target**: smin/smax intrinsics with operands having different sign bit counts
- **Rationale**: The CompChecker for smin/smax (lines 14381-14387) checks that SignBits <= ComputeNumSignBits for both operands. However, ComputeNumSignBits returns the number of sign bits including the sign bit itself. If one operand has exactly SignBits and another has more, the check passes but truncation could still be unsafe if the values differ in the truncated region.
- **Expected Issue**: When operands have borderline sign bit counts (exactly equal to required SignBits), truncation may incorrectly succeed. For example, if OrigBitWidth=32, BitWidth=16, SignBits=16, and one operand has exactly 16 sign bits while having non-sign-extending bits in positions 16-31, the truncation would lose information.

### Strategy 3: Test umin/umax with partial zero masks

- **Target**: umin/umax intrinsics where only one operand has zero upper bits
- **Rationale**: The CompChecker for umin/umax (lines 14376-14379) requires BOTH operands to have zero upper bits (MaskedValueIsZero). However, for umin/umax semantics, if only one operand is bounded, the result is still bounded by that operand. The current check is overly conservative and may miss optimization opportunities, but more critically, if the check logic has bugs, it could allow unsafe truncations.
- **Expected Issue**: If the MaskedValueIsZero check has edge cases (e.g., with symbolic values or values derived from select instructions), the truncation might be applied when upper bits are not provably zero, leading to incorrect results after truncation and extension.

### Strategy 4: Test multi-use scalar with abs intrinsic bitwidth reduction

- **Target**: abs intrinsic whose result is used by both vectorized and non-vectorized code
- **Rationale**: The minbitwidth analysis in collectValuesToDemote checks if values can be demoted, but the patch doesn't explicitly verify that all uses of the scalar can tolerate the reduced bitwidth. The existing code has a known issue pattern about 'Multi-Use Scalar Constraints' where external uses are not checked.
- **Expected Issue**: If an abs intrinsic's result is used both in the vectorized tree and externally (e.g., stored to memory, used in a comparison outside the tree), the bitwidth reduction might break the external uses that expect the original bitwidth. The truncation would lose upper bits that external users depend on.

### Strategy 5: Test abs with is_int_min_poison=false originally but INT_MIN at reduced bitwidth

- **Target**: abs intrinsic with is_int_min_poison=false where input equals INT_MIN of reduced bitwidth
- **Rationale**: When the patch forces is_int_min_poison=false for reduced bitwidth (lines 12598-12600), it assumes this is safe. However, if the original call had is_int_min_poison=false and the input happens to be INT_MIN of the REDUCED bitwidth (but not the original), the semantics should return INT_MIN (not poison). The forced false is correct, but the truncation itself might change the value if the original was NOT INT_MIN of original bitwidth but becomes INT_MIN after truncation.
- **Expected Issue**: Consider i32 abs with value 0xFFFF8000 (-32768 in lower 16 bits, but a large negative in i32). If truncated to i16, it becomes 0x8000 (INT_MIN for i16). The abs of i32 0xFFFF8000 with is_int_min_poison=false should return 0xFFFF8000. But if truncated to i16 first, abs(i16 0x8000) with is_int_min_poison=false returns 0x8000, which when extended back is 0x00008000, not 0xFFFF8000. The sign extension after abs is wrong.

### Analysis

### Bug Triggering Analysis

The test case triggers a bug in the SLPVectorizer's handling of the `llvm.abs` intrinsic when:
1. The original scalar code uses `is_int_min_poison=true` (second argument to llvm.abs)
2. The minbitwidth analysis determines that a smaller bitwidth (i8) is sufficient
3. The vectorizer forces `is_int_min_poison=false` in the vectorized code (lines 12598-12600 of the patch)

In the test case:
- When `%is_min=true`, all values are `i8 -128` (INT_MIN for i8)
- These are sign-extended to i32: `sext i8 -128 to i32` = `i32 -128` (0xFFFFFF80)
- The scalar code calls `llvm.abs.i32(i32 -128, i1 true)` which should return **poison** because -128 is INT_MIN for i8 (the original domain of the value)
- However, the vectorizer reduces to i8 and generates `llvm.abs.v8i8(<8 x i8> -128, i1 false)` which returns **128** (not poison)

The transformation is incorrect because:
- Original semantics: abs(INT_MIN, true) = poison
- Transformed semantics: abs(INT_MIN, false) = INT_MIN (128 for i8)

### Fix Weakness Analysis

The fix assumes that when bitwidth is reduced for abs, forcing `is_int_min_poison=false` is always safe. The comment at lines 12596-12597 says "if decided to reduce bitwidth of abs intrinsic, it second argument must be set false (do not return poison, if value is signed min)."

However, this assumption is incorrect because:
1. The minbitwidth analysis determines the reduced bitwidth based on the **result** bitwidth needed, not on whether the input can be INT_MIN for the reduced type
2. Even if the original i32 value is not INT_MIN for i32, it could be INT_MIN for the reduced i8 type after truncation
3. The original `is_int_min_poison` flag expresses important semantic intent: whether INT_MIN should produce poison or the absolute value
4. Forcing it to false changes the semantics when the input IS INT_MIN for the reduced type

The fix should either:
- Not reduce bitwidth when `is_int_min_poison=true` and the input could be INT_MIN for the reduced type
- Preserve the original `is_int_min_poison` flag value instead of forcing it to false
- Add additional checks to ensure no lane contains INT_MIN for the reduced bitwidth before forcing the flag to false
