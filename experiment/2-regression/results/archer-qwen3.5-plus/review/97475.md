## Test Strategies

### Strategy 1: Poison condition test

- **Target**: select with poison condition
- **Rationale**: If the select condition is poison, the result should be poison regardless of the arms. The optimization might incorrectly simplify arms assuming the condition is true/false, potentially changing semantics.
- **Expected Issue**: The optimizer might replace a select arm with a constant computed under the assumption that the condition is true/false, but if the condition is poison, the select should remain poison.

### Strategy 2: Poison-generating operation in select arm

- **Target**: select arm containing operations that can produce poison (e.g., division, shift with variable amount)
- **Rationale**: If one arm contains an operation that produces poison (e.g., division by zero), but that arm is guarded by the condition, the original code is safe. The optimization might incorrectly compute KnownBits for the arm assuming the opposite condition, potentially masking the poison or changing semantics.
- **Expected Issue**: The optimizer might incorrectly simplify an arm that would produce poison, changing the program's behavior when the poison would have been triggered.

### Strategy 3: Vector select with mixed poison elements

- **Target**: vector select where some lanes have poison conditions
- **Rationale**: For vector selects, some lanes might have poison conditions while others don't. The optimization needs to handle per-lane poison correctly. If the optimization treats the entire vector condition as known, it might incorrectly simplify lanes that should remain poison.
- **Expected Issue**: The optimizer might incorrectly simplify vector select arms, producing non-poison results for lanes that should be poison.

### Strategy 4: Multi-use value affected by condition

- **Target**: value affected by condition that has uses outside the select
- **Rationale**: If a value affected by the condition is used both inside and outside the select, the KnownBits from the condition should only apply within the select context. The optimization might incorrectly propagate KnownBits to other uses.
- **Expected Issue**: The optimizer might incorrectly compute KnownBits for a multi-use value, affecting other uses of the value outside the select context.

### Strategy 5: FCmp condition with special floating-point values

- **Target**: select with floating-point comparison condition involving NaN, infinity, or signed zero
- **Rationale**: The findValuesAffectedByCondition function has limited support for floating-point comparisons. It might not correctly identify affected values for FCmp conditions, or the KnownBits computation might not handle floating-point edge cases correctly.
- **Expected Issue**: The optimizer might miss optimization opportunities or incorrectly simplify selects with FCmp conditions involving special floating-point values.

### Analysis

## Bug Summary

Found a real bug in the InstCombine optimization pass introduced by commit 77eb05683082dd3751ccfab963f5160f1852058d ([InstCombine] Simplify select using KnownBits of condition).

### Bug Description

The optimization incorrectly simplifies vector select instructions where the select arm is a shuffle of the condition operand. The KnownBits from the condition are applied to the wrong vector lanes after shuffling.

### Test Case

```llvm
define <2 x i32> @vec_select_no_equivalence(<2 x i32> %x) {
  %x10 = shufflevector <2 x i32> %x, <2 x i32> undef, <2 x i32> <i32 1, i32 0>
  %cond = icmp eq <2 x i32> %x, zeroinitializer
  %s = select <2 x i1> %cond, <2 x i32> %x10, <2 x i32> %x
  ret <2 x i32> %s
}
```

For input `<0, 2>`:
- **Original (correct)**: Returns `<2, 2>`
  - `%x10 = shufflevector <0, 2>, undef, <1, 0> = <2, 0>`
  - `%cond = icmp eq <0, 2>, <0, 0> = <true, false>`
  - `%s = select <true, false>, <2, 0>, <0, 2> = <2, 2>`

- **Optimized (incorrect)**: Returns `<0, 2>`
  - The optimizer incorrectly replaces the entire select with just `%x`

### Root Cause

The new optimization in `visitSelectInst` computes KnownBits for select arms based on the condition. When the condition is `icmp eq %x, <0, 0>`, it knows that when the condition is true for lane 0, `%x[0] == 0`. However, when the select arm is `shufflevector %x, undef, <1, 0>`, lane 0 of the arm is actually `%x[1]`, not `%x[0]`. The optimization fails to account for how shuffle masks permute the KnownBits information across vector lanes.

### Fix Weakness

The `findValuesAffectedByCondition` function and the KnownBits computation in `computeKnownBitsFromContext` don't handle shuffle/permutation operations correctly. They track which values are affected by a condition but don't track how the KnownBits should be permuted when those values are shuffled.
