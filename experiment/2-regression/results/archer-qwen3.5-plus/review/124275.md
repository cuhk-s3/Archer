## Test Strategies

### Strategy 1: Direct self-reference without context instruction initialization

- **Target**: breakSelfRecursivePHI function
- **Rationale**: When ValOut == PHI (direct self-reference), the function returns early without setting CtxIOut. This leaves CtxIOut uninitialized, and getWithInstruction(CxtI) is called with an uninitialized value, potentially causing undefined behavior or incorrect analysis.
- **Expected Issue**: Uninitialized memory access or incorrect known bits analysis when a PHI node directly references itself as an incoming value

### Strategy 2: PHI with more than 2 incoming values referencing another PHI

- **Target**: breakSelfRecursivePHI function - PHI handling branch
- **Rationale**: The code only handles 2-operand PHI nodes (IncPhi->getNumIncomingValues() == 2). If a PHI has 3+ incoming values where one references the original PHI, the recursion breaking logic won't trigger, potentially causing infinite recursion or missed optimization opportunities.
- **Expected Issue**: Infinite recursion or incorrect known bits when analyzing a PHI with 3+ incoming values where one incoming value is a PHI that references the original PHI

### Strategy 3: Select with poison/undef in the non-recursive arm

- **Target**: breakSelfRecursivePHI function - select handling
- **Rationale**: When breaking recursion through a select's non-recursive arm, the fix assumes the other arm provides valid known bits. However, if the other arm contains poison or undef, the analysis could incorrectly propagate known bits that aren't actually guaranteed.
- **Expected Issue**: Incorrect known bits propagation when the non-recursive select arm contains poison or undef values

### Strategy 4: Context instruction from block without proper terminator

- **Target**: CtxIOut initialization in breakSelfRecursivePHI
- **Rationale**: The code sets CtxIOut to PHI->getIncomingBlock(*U)->getTerminator(). In malformed IR or during certain transformations, a block might not have a proper terminator, leading to null pointer issues or incorrect context.
- **Expected Issue**: Null pointer dereference or incorrect context when the incoming block's terminator is null or invalid

### Strategy 5: Floating-point specific edge cases in computeKnownFPClass

- **Target**: computeKnownFPClass usage of breakSelfRecursivePHI
- **Rationale**: The same breakSelfRecursivePHI helper is used for both integer KnownBits and floating-point FPClass analysis. However, FP semantics (NaN propagation, signed zeros, infinity) might require different handling when breaking recursion through select/PHI patterns.
- **Expected Issue**: Incorrect FP class analysis (e.g., missing NaN, incorrect sign bit) when recursive select/PHI patterns involve floating-point values with special semantics

## Summary of Fix Analysis and Testing

### Fix Understanding
The patch introduces a helper function `breakSelfRecursivePHI` to handle recursive select/PHI patterns in `ComputeKnownBits` and `computeKnownFPClass`. The function:
1. Detects when a PHI's incoming value is a select that references the PHI itself
2. Detects when a PHI's incoming value is another 2-operand PHI that references the original PHI
3. Uses the "other" value to break recursion cycles

### Key Issues Identified

1. **Uninitialized CtxIOut**: When `ValOut == PHI` (direct self-reference), the function returns early without initializing `CtxIOut`. However, the caller checks `if (IncValue == P) continue;` before using `CxtI`, so this doesn't cause actual issues in practice.

2. **Limited PHI handling**: The code only handles 2-operand PHI nodes. PHI nodes with 3+ incoming values won't have their recursion broken properly.

3. **Poison/undef in non-recursive arm**: When breaking recursion through a select's non-recursive arm containing poison/undef, the analysis might incorrectly propagate known bits.

4. **FP semantics**: The same helper is used for both integer KnownBits and floating-point FPClass analysis, but FP has special semantics (NaN, signed zeros) that might need different handling.

### Testing Results
All 7 test cases were tested with all 5 strategies:
- Direct self-reference without context instruction initialization
- PHI with more than 2 incoming values referencing another PHI
- Floating-point specific edge cases in computeKnownFPClass
- Select with poison/undef in the non-recursive arm
- Context instruction from block without proper terminator

No bugs were found. The transformations were verified correct by alive2 and difftest. The fix appears to be correct, though there are some limitations (like the 2-operand PHI restriction) that are acknowledged in TODO comments.

### Test Case Provided
The test case `test_select_poison_arm` demonstrates a select/PHI recursion pattern where the non-recursive arm contains poison. This tests whether the analysis correctly handles poison values when breaking recursion.
