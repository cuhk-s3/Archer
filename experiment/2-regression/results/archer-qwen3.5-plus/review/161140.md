## Test Strategies

### Strategy 1: Mixed Interchangeable Opcodes with Different Constants

- **Target**: Test mixing interchangeable instructions with different constant values that should NOT be convertible
- **Rationale**: The BinOpSameOpcodeHelper assumes certain constant values allow interchangeability (e.g., mul by 1, shl by 0). If we mix instructions with constants that are NOT interchangeable (e.g., mul x, 2 and shl y, 3), the converter may incorrectly try to unify them or produce wrong operand values.
- **Expected Issue**: The vectorizer may incorrectly unify non-interchangeable instructions, producing wrong constant operands in the vectorized code, leading to incorrect computation results.

### Strategy 2: Interchangeable Instructions with Overflow Flags

- **Target**: Test interchangeable instructions that have different overflow/nuw/nsw flags
- **Rationale**: When converting between interchangeable instructions (e.g., add x, 0 to sub x, 0), the overflow flags (nuw, nsw) may have different semantics. The fix doesn't appear to handle flag propagation during conversion.
- **Expected Issue**: The vectorized instruction may have incorrect or missing overflow flags, potentially changing program semantics for overflow-sensitive code or missing optimization opportunities.

### Strategy 3: Non-Commutative Operation Operand Order

- **Target**: Test interchangeable conversions for non-commutative operations (Sub, Shl, AShr) with constants on different operand positions
- **Rationale**: The isBinOpWithConstantInt function checks RHS first, then LHS for non-commutative ops. The getOperand function must correctly handle operand position when converting. If the constant is on the 'wrong' side for a non-commutative operation, the conversion may produce incorrect operand order.
- **Expected Issue**: For operations like sub (0 - x vs x - 0) or shl, the converted instruction may have operands in wrong order, producing incorrect results (e.g., 0-x instead of x-0).

### Strategy 4: Interchangeable Instructions with Poison/undef in VL

- **Target**: Test VL containing mix of interchangeable instructions and PoisonValue
- **Rationale**: The allSameOpcode function treats PoisonValue as matching. However, when BinOpSameOpcodeHelper processes a VL with poison values, it may not correctly handle the interaction between poison semantics and interchangeable instruction conversion.
- **Expected Issue**: Poison values in VL may cause incorrect opcode selection or operand conversion, leading to improper poison propagation in the vectorized result.

### Strategy 5: Bitwidth Mismatch in Interchangeable Conversion

- **Target**: Test interchangeable instructions with constants of different bitwidths or shift amounts that exceed demoted types
- **Rationale**: The getOperand function converts constant values between opcodes (e.g., shl amount to mul constant). If the VL undergoes type demotion via MinBWs analysis, the converted constant may not fit the demoted type, especially for shift operations where the shift amount must be less than the bitwidth.
- **Expected Issue**: After type demotion, converted shift amounts or constants may be out of bounds for the demoted type, causing undefined behavior or incorrect results.

### Analysis

Bug found: SLPVectorizer crashes when converting interchangeable instructions where the shift amount exceeds the type bitwidth. The BinOpSameOpcodeHelper::InterchangeableInfo::getOperand() calls APInt::getOneBitSet(8, 10) for mul i8 X, 1024 -> shl i8 X, 10 conversion, triggering assertion failure. Fix needs to validate shift amount < bitwidth before considering mul/shl interchangeable.
