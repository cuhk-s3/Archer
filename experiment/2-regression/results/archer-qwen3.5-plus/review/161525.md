## Test Strategies

### Strategy 1: Test with infinity inputs for unordered predicates

- **Target**: fcmp with UGT/ULT/UNE predicates and fsub without nnan/ninf flags
- **Rationale**: The optimization checks for infinity cases for UGT/ULT/UNE/OEQ/OGE/OLE predicates. When X and Y are both +inf or both -inf, (X - Y) produces NaN. For unordered predicates like UGT, NaN causes the comparison to return true, but comparing X > Y when both are +inf should return false. This semantic difference could cause miscompilation.
- **Expected Issue**: Incorrect boolean result when comparing inf - inf vs inf > inf for unordered predicates

### Strategy 2: Test with vector types having partial infinity knowledge

- **Target**: vector fcmp fsub where only some lanes are known non-infinity
- **Rationale**: The isKnownNeverInfinity function for vectors returns true only if ALL elements are known non-infinity. If the analysis cannot prove all lanes are non-infinity, but the transformation still applies, some lanes could produce incorrect results when inf-inf=nan occurs in those lanes.
- **Expected Issue**: Incorrect results in vector lanes where infinity subtraction produces NaN

### Strategy 3: Test with non-IEEE denormal modes

- **Target**: fcmp fsub with denormal-fp-math set to positive-zero, preserve-sign, or dynamic
- **Rationale**: The transformation explicitly checks for DenormalMode::getIEEE(). Tests with other denormal modes should verify the transformation is correctly blocked. If the check is incomplete, transformations might apply incorrectly under different denormal handling modes.
- **Expected Issue**: Transformation incorrectly applied when denormal mode is not IEEE, potentially changing behavior for denormal results

### Strategy 4: Test fast-math flag preservation

- **Target**: fcmp with various fast-math flags (nnan, ninf, nsz, arcp, contract, afn, reassoc)
- **Rationale**: When transforming fcmp pred (x-y), 0 to fcmp pred x, y, the fast-math flags on the original fcmp should be preserved. The patch code doesn't explicitly show flag handling, which could lead to flags being lost or incorrectly applied to the new comparison.
- **Expected Issue**: Fast-math flags lost or incorrectly modified during transformation, changing floating-point semantics

### Strategy 5: Test with NaN inputs for ordered predicates

- **Target**: fcmp with OGT/OLT/ONE predicates where X or Y could be NaN
- **Rationale**: For ordered predicates, if X or Y is NaN, fcmp pred X, Y returns false. But fcmp pred (X-Y), 0 with NaN inputs also involves NaN arithmetic. The transformation assumes equivalence, but the NaN propagation through subtraction might differ from direct comparison in edge cases, especially when combined with fast-math flags.
- **Expected Issue**: Different boolean results when NaN is involved in subtraction vs direct comparison

## Bug Analysis Summary

After extensive testing of the InstCombine fix for folding `fcmp pred (x - y), 0` into `fcmp pred x, y`, no bugs were found. The fix correctly handles:

### 1. Infinity Checks for Unordered Predicates
The code correctly blocks transformation for UGT, ULT, UNE, OEQ, OGE, OLE predicates when:
- fsub doesn't have nnan or ninf flags
- Neither operand is known to be non-infinity

Tested: `fcmp_ugt_fsub_const`, `fcmp_ult_fsub_const`, `fcmp_oeq_fsub_const`, `fcmp_ole_fsub_const`, `fcmp_une_fsub_const` - all correctly NOT transformed.

### 2. Fallthrough Predicates
For OGT, OLT, ONE, UEQ, UGE, ULE predicates, the transformation correctly applies without infinity checks since these predicates have consistent semantics for the inf-inf=nan case.

Tested: `fcmp_uge_fsub_const`, `fcmp_ule_fsub_const` - correctly transformed.

### 3. Denormal Mode Check
The transformation correctly blocks when denormal-fp-math is not IEEE:
- Tested with "positive-zero", "preserve-sign", "dynamic" modes - all correctly NOT transformed.

### 4. Fast-Math Flag Preservation
Fast-math flags (nnan, ninf, nsz) are correctly preserved on the resulting fcmp instruction.

Tested: Various combinations with nnan, ninf, nsz flags - all correctly preserved.

### 5. Vector Type Handling
Vector types are correctly handled:
- Without nnan/ninf flags, transformation is blocked for predicates requiring infinity checks
- With nnan/ninf flags, transformation correctly applies

### Conclusion
The fix is correct and handles all edge cases properly. The key insight is that the code correctly distinguishes between:
1. Predicates where inf-inf=nan matters (UGT, ULT, UNE, OEQ, OGE, OLE) - requires infinity checks
2. Predicates where the transformation is always valid (OGT, OLT, ONE, UEQ, UGE, ULE) - no infinity check needed

The denormal mode check and fast-math flag preservation are also correctly implemented.
