## Test Strategies

### Strategy 1: Test with single reachable case and matching range

- **Target**: switch.ll - test_unreachable_default
- **Rationale**: The fix has condition 'ReachableCaseCount > 1' which skips optimization when there's exactly 1 reachable case. If the range size is 1 and there's 1 case covering that value, the default should be unreachable but the optimization won't run. This tests whether the '> 1' condition is too restrictive.
- **Expected Issue**: Missed optimization: default block not marked unreachable when there's exactly 1 reachable case and range size is 1

### Strategy 2: Test with non-contiguous case values in range

- **Target**: switch.ll - create new test
- **Rationale**: The fix checks if range size <= ReachableCaseCount, assuming all values in range are covered. But if cases are for non-contiguous values (e.g., 0, 2, 4) and range is [0, 5) (size 5), with 3 reachable cases, the check '5 > 3' is true so default is not marked unreachable (correct). But if range is [0, 3) (size 3) with cases 0, 2, 4, LVI should mark case 4 as False. If LVI is imprecise and marks it Unknown, ReachableCaseCount=3, range size=3, '3 > 3' is false, default incorrectly marked unreachable.
- **Expected Issue**: Incorrect optimization: default marked unreachable when LVI imprecisely reports cases outside range as Unknown

### Strategy 3: Test with PHI node producing condition with undef from one path

- **Target**: switch.ll - test_unreachable_default_cond_may_be_undef variant
- **Rationale**: When condition comes from a PHI node where one incoming value is undef, LVI's predicate analysis and range analysis might have different precision. The range check uses UndefAllowed=false which returns full range for undef, but predicate analysis might not correctly identify all cases as False. This tests consistency between predicate and range analysis.
- **Expected Issue**: Incorrect optimization: default marked unreachable when PHI produces undef and LVI analyses are inconsistent

### Strategy 4: Test with range exactly matching case count but gaps in coverage

- **Target**: switch.ll - create new test with specific values
- **Rationale**: Create a switch where range size equals ReachableCaseCount but there are gaps. E.g., range [0, 4) has size 4, cases for 0, 1, 2, 3 would cover all. But if cases are 0, 1, 2, 5 and LVI incorrectly says case 5 is Unknown (should be False since 5 not in [0,4)), then ReachableCaseCount=4, range size=4, '4 > 4' is false, default incorrectly marked unreachable. Value 3 would fall through to unreachable.
- **Expected Issue**: Miscompilation: value 3 reaches unreachable instead of default block due to incorrect case coverage analysis

### Strategy 5: Test with empty range (unknown value) and cases

- **Target**: switch.ll - create new test
- **Rationale**: If the condition value is completely unknown (not even a range), getConstantRangeAtUse returns empty range. Empty range has size 0. isSizeLargerThan(ReachableCaseCount) for empty range: (Upper-Lower).ugt(MaxSize) = 0.ugt(n) = false for n>=0. So !false = true, default would be marked unreachable. But if value is unknown, we shouldn't assume default is unreachable. This tests handling of empty/unknown ranges.
- **Expected Issue**: Incorrect optimization: default marked unreachable when condition range is empty/unknown

## Summary of Fix Analysis and Testing

### Fix Understanding
The patch adds logic to check whether the default case of a switch statement is unreachable by:
1. Counting "reachable" cases (cases where LVI returns Unknown, meaning they might fire)
2. Getting the constant range of the switch condition with UndefAllowed=false
3. If range size <= ReachableCaseCount and ReachableCaseCount > 1, marking the default as unreachable

### Key Findings from Testing

All tests passed verification with alive2 and differential testing. The fix correctly:

1. **Marks default as unreachable when all values are covered**: When the switch condition has a bounded range (e.g., [0, 3)) and all values in that range are covered by explicit cases, the default is correctly marked as unreachable.

2. **Preserves default when values may be undef**: When the condition doesn't have `noundef` attribute, the range analysis returns a full range (due to `UndefAllowed=false`), so `isSizeLargerThan` returns true and the default is NOT marked unreachable.

3. **Handles already unreachable defaults**: When the default block already starts with an `UnreachableInst`, the optimization doesn't modify it.

4. **Correctly handles reachable defaults**: When the select produces a value outside the case values (e.g., 4 instead of 0, 1, 2), the default remains reachable.
