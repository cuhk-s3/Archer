## Test Strategies

### Strategy 1: Constant in First Operand

- **Target**: BinaryOperator with constant as first operand
- **Rationale**: The fix only checks BO->getOperand(1) for a constant using m_APInt matcher. For commutative operations (and, or, xor), the constant could be in getOperand(0) instead. This would cause the optimization to miss valid simplification opportunities.
- **Expected Issue**: Binary operations like 'and 24, %x' or 'or 3, %x' where the constant is the first operand would not be simplified even when the demanded bits condition is satisfied, missing optimization opportunities.

### Strategy 2: Vector Non-Splat Constants

- **Target**: BinaryOperator with non-splat vector constant mask
- **Rationale**: The m_APInt matcher only matches scalar constants or splat vector constants. Non-splat vector constants (e.g., '<i8 4, i8 8>') would not be matched, potentially missing simplification opportunities for vector operations where different elements have different mask values but the demanded bits condition is still satisfied.
- **Expected Issue**: Vector binary operations with non-splat constant masks would not be simplified even when the demanded bits for each element don't intersect with the corresponding mask bits.

### Strategy 3: Zero Demanded Bits Edge Case

- **Target**: BinaryOperator with zero demanded bits but non-trivially-dead usage
- **Rationale**: When Demanded.isZero() is true, Demanded.isSubsetOf(Mask) returns true for any Mask (empty set is subset of any set). If the instruction has uses but zero demanded bits (e.g., used only in debug info or in a way that doesn't demand bits), the simplification would replace the binary op with operand 0, which might change observable behavior if the original result was used in ways not captured by DemandedBits.
- **Expected Issue**: A binary operation like 'and %x, 0' with zero demanded bits might be incorrectly simplified to '%x' instead of being handled by the dead code elimination path, potentially changing program behavior if the result is used in ways not tracked by DemandedBits.

### Strategy 4: Mixed Operand Types in Commutative Operations

- **Target**: BinaryOperator with constant operand in varying positions across different uses
- **Rationale**: In complex code patterns, the same binary operation pattern might appear with the constant in different operand positions. The fix only handles constant-in-second-operand cases, creating inconsistent optimization behavior.
- **Expected Issue**: Inconsistent optimization where '%or1 = or %x, 3' gets simplified but '%or2 = or 3, %x' does not, even though both have the same semantics and demanded bits patterns.

### Strategy 5: Poison/Undef in Vector Constants

- **Target**: BinaryOperator with vector constant containing undef elements
- **Rationale**: The m_APInt matcher by default forbids undefs (AllowUndef=false). Vector constants with undef elements would not match, potentially missing simplification opportunities. The fix doesn't use m_APIntAllowUndef to handle such cases.
- **Expected Issue**: Vector binary operations with constants containing undef elements (e.g., '<i8 4, undef>') would not be simplified even when the defined elements satisfy the demanded bits condition.
