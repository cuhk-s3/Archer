## Test Strategies

### Strategy 1: Test vector types with multiple elements

- **Target**: Modify existing scalar tests to use vector types
- **Rationale**: The fix uses hasOneUse() which checks if the instruction has exactly one use. For vector types, even if the vector ctlz/cttz has one use, individual lanes might have different semantics. The m_Shift matcher may not correctly handle all vector shift scenarios.
- **Expected Issue**: The optimization might incorrectly set is_zero_poison=true for vector ctlz/cttz when some lanes could have defined behavior that differs from poison

### Strategy 2: Test with shift using ctlz/cttz as the value operand instead of shift amount

- **Target**: Create test where ctlz/cttz result is the first operand (value) of a shift, not the shift amount
- **Rationale**: The m_Shift matcher matches any shift instruction. The pattern m_Shift(m_Value(), m_Specific(&II)) matches shifts where the second operand is the ctlz/cttz. But shifts have two operands: the value being shifted and the shift amount. The fix assumes the ctlz/cttz is used as the shift amount (second operand), but the matcher might match incorrectly.
- **Expected Issue**: If ctlz/cttz is used as the value operand (first operand) of a shift, setting is_zero_poison=true would be incorrect because the result would be used directly, not as a shift amount

### Strategy 3: Test with multiple uses through intermediate instructions

- **Target**: Create test where ctlz/cttz has one direct use but that use is consumed by multiple other instructions
- **Rationale**: The fix checks hasOneUse() which only checks direct uses. If the single use is an instruction that is itself used multiple times in different contexts (some as shift amount, some not), the optimization could be incorrect.
- **Expected Issue**: The ctlz/cttz result might be used in a context where poison would propagate incorrectly to non-shift uses

### Strategy 4: Test with variable shift amounts that could equal bitwidth

- **Target**: Test cases where the shift amount (ctlz/cttz result) could equal or exceed the bitwidth when input is zero
- **Rationale**: When ctlz/cttz input is zero, the result equals the bitwidth. Shifting by >= bitwidth is undefined behavior. The fix assumes this is acceptable, but if the shift has nuw/nsw/exact flags or is part of a larger expression, the UB might manifest differently.
- **Expected Issue**: The optimization might introduce UB in cases where the original code with is_zero_poison=false would have produced a defined result

### Strategy 5: Test with select or other control flow that guards zero input

- **Target**: Create test where ctlz/cttz is conditionally used as shift amount based on whether input is zero
- **Rationale**: If there's control flow or a select that guards against zero input to the shift, changing is_zero_poison from false to true might change the semantics. The original code would produce bitwidth for zero input, but with poison, the entire computation becomes poison.
- **Expected Issue**: The optimization removes the defined behavior for zero input cases that were previously handled correctly

### Analysis

## Bug Analysis Summary

After comprehensive testing of the InstCombine fix for setting `zero_is_poison` for ctlz/cttz when used as shift amounts, no bugs were found. The fix is correct.

### Fix Understanding
The patch adds an optimization in `foldCttzCtlz` that sets `is_zero_poison=true` for ctlz/cttz intrinsics when:
1. The intrinsic has exactly one use (`hasOneUse()`)
2. The current `is_zero_poison` flag is `false`
3. The single use is a shift instruction where the ctlz/cttz result is used as the shift amount (second operand)

### Test Coverage
All 6 test cases were thoroughly tested, covering all 5 proposed strategies:

1. **Test with variable shift amounts that could equal bitwidth**: Tested with y=0 input where cttz/ctlz returns bitwidth (32 for i32). Both original and transformed code have UB for shift by >= bitwidth, so no difference.

2. **Test with multiple uses through intermediate instructions**: Tested with multi-use cases (cttz used by both a call and shift). The fix correctly does NOT apply when hasOneUse() returns false.

3. **Test vector types with multiple elements**: Tested vector types (<2 x i32>) with various inputs. The optimization correctly applies to vector types.

4. **Test with shift using ctlz/cttz as the value operand instead of shift amount**: Tested cases where cttz/ctlz is the first operand (value being shifted) rather than the second operand (shift amount). The fix correctly does NOT apply because `m_Shift(m_Value(), m_Specific(&II))` only matches when ctlz/cttz is the second operand.

5. **Test with select or other control flow that guards zero input**: Tested cases with select instructions guarding zero input. The transformation is correct because even with poison, the select chooses the safe value.

### Verification Results
- All Alive2 verifications passed (transformation seems correct)
- All difftest executions showed no differences between original and transformed code
- The fix correctly handles edge cases:
  - Multi-use instructions are not optimized
  - ctlz/cttz used as value operand (not shift amount) is not optimized
  - Vector types are handled correctly
  - Guarded zero cases work correctly

### Conclusion
The fix is correct and does not introduce any bugs. The optimization safely sets `is_zero_poison=true` only when the ctlz/cttz result is exclusively used as a shift amount, where the difference between returning bitwidth (UB for shift) and poison is semantically equivalent.
