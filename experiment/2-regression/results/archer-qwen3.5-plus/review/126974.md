## Test Strategies

### Strategy 1: Both samesign predicates with different signedness

- **Target**: isImpliedCondCommonOperandWithCR
- **Rationale**: The fix only handles the case when exactly one predicate has samesign (XOR condition). When both predicates have samesign but different signedness (e.g., samesign ult vs samesign slt), the XOR is false and the new logic is skipped. This could miss valid implications that should be detected.
- **Expected Issue**: The optimizer fails to recognize that 'icmp samesign ult X, C1' implies 'icmp samesign slt X, C2' (or vice versa) when the constant ranges would allow such inference if signedness was flipped.

### Strategy 2: Samesign with equality predicate

- **Target**: isImpliedCondCommonOperandWithCR
- **Rationale**: When one predicate is samesign ult/ugt/slt/sgt and the other is samesign eq/ne, the getFlippedSignednessPredicate returns the same predicate for eq/ne. This could lead to incorrect implications being inferred or valid implications being missed.
- **Expected Issue**: Incorrect optimization when comparing samesign relational predicates against samesign equality predicates, potentially folding conditions incorrectly.

### Strategy 3: Constant range computation ignores samesign flag

- **Target**: computeConstantRange call in isImpliedCondICmps
- **Rationale**: The code calls computeConstantRange with ICmpInst::isSigned(LPred) which only checks if the predicate is signed (SLT/SLE/SGT/SGE), not whether samesign is set. When samesign is set on an unsigned predicate, the constant range should potentially be computed differently since samesign implies operands have the same sign.
- **Expected Issue**: Incorrect constant range computation leads to wrong implication results. For example, 'icmp samesign ult X, -1' should have different range semantics than 'icmp ult X, -1' because samesign ult with a negative constant has specific constraints.

### Strategy 4: Cross-signedness implication with samesign on wrong predicate

- **Target**: isImpliedCondCommonOperandWithCR
- **Rationale**: The fix flips the predicate that has samesign. But consider: 'icmp samesign ult X, 5' implies 'icmp slt X, 5'. The samesign is on ult (unsigned), and we're checking against slt (signed). The fix flips ult to slt, then checks. But what if the constant range spans negative values? The samesign constraint means X and 5 must have same sign, so X must be non-negative.
- **Expected Issue**: Missing the constraint that samesign imposes on the operand values themselves. When samesign ult X, C is true and C is positive, X must also be non-negative. This additional constraint is not captured in the constant range computation.

### Strategy 5: Vector type with mixed samesign elements

- **Target**: isImpliedCondCommonOperandWithCR with vector types
- **Rationale**: For vector icmp instructions, samesign applies element-wise. The fix doesn't appear to handle the case where different elements might have different samesign implications, especially when the constant is a vector with mixed sign elements.
- **Expected Issue**: Incorrect optimization for vector icmp with samesign when constants have mixed signs across elements, leading to wrong implication results for some vector lanes.

### Analysis

### Bug Triggering Analysis

The test case `sgt_implies_ge_dominating_cr` triggers a bug in the fix for teaching `isImpliedCondCommonOperandWithCR` about samesign.

**The Issue:**

1. **Original code semantics**:
   - `icmp sgt %a, -10` is true when `%a > -10` (signed comparison)
   - This includes positive values (like 100) AND negative values greater than -10 (like -5, -9)

2. **In the taken block**:
   - `icmp samesign uge %a, -20` requires `%a` and `-20` to have the same sign
   - Since `-20` is negative, `%a` must also be negative for well-defined behavior
   - If `%a` is positive, `samesign uge` produces **poison** (undefined behavior)

3. **The bug manifestation**:
   - When `%a = 100` (positive):
     - `sgt 100, -10` is TRUE → we enter the taken block
     - `samesign uge 100, -20` produces **poison** (100 is positive, -20 is negative)
     - Original code: returns poison (UB)
     - Optimized code: incorrectly simplifies to constant 30

4. **Root cause in the fix**:
   The fix in `isImpliedCondCommonOperandWithCR` handles the case when exactly one predicate has `samesign` by flipping the signedness of that predicate. It checks if `sgt %a, -10` implies `sge %a, -20` (after flipping `samesign uge` to `sge`).

   The implication `sgt %a, -10` → `sge %a, -20` is mathematically true for all values. However, the fix ignores that `samesign uge` has an additional constraint: it produces poison when operands have different signs. This poison-generating behavior is not captured by simply flipping the predicate.

### Fix Weakness Analysis

**The fix's fundamental weakness**: It treats `samesign` predicates as semantically equivalent to their flipped non-samesign counterparts for implication checking, but completely ignores the poison-generating nature of `samesign` when operands have different signs.

**What the fix assumes**:
- `samesign uge %a, -20` ≡ `sge %a, -20` for implication purposes

**Why this is wrong**:
- `samesign uge %a, -20` produces poison when `%a` has different sign from `-20`
- `sge %a, -20` produces a valid boolean for all `%a`
- The fix's predicate flipping loses the poison constraint

**The correct behavior**: The optimizer should NOT simplify the select to a constant because the `samesign` comparison can produce poison for values that satisfy the dominating condition but have the wrong sign.

This bug demonstrates that the fix needs to account for the poison-generating semantics of `samesign`, not just the comparison semantics after flipping signedness.
