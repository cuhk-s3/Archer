## Test Strategies

### Strategy 1: Test trunc nsw + zext with signed comparison and negative source value

- **Target**: Modify trunc_signed_nsw_zext test to use a source value that could be negative before zext
- **Rationale**: The transformation icmp slt (trunc nsw X), (zext Y) -> icmp sgt (zext Y), X assumes zext produces correct signed values. If Y's source type can represent negative values (e.g., i8 with sign bit set), zext will zero-extend (making it positive), which may differ from sext behavior in signed comparisons.
- **Expected Issue**: Incorrect comparison result when Y's original value has sign bit set (e.g., i8 value 0xFF = 255 as unsigned, but -1 as signed). The zext produces 255, but if the semantics expected -1, the comparison would be wrong.

### Strategy 2: Test trunc nuw + sext with signed comparison (should NOT transform)

- **Target**: Add test case for icmp slt (trunc nuw X), (sext Y) pattern
- **Rationale**: The code only allows trunc nuw + zext for unsigned comparisons (line 1515-1518) and trunc nsw + zext/sext for all predicates (line 1520-1526). The combination trunc nuw + sext for signed comparisons is not explicitly handled and should not transform because nuw doesn't guarantee signed semantics.
- **Expected Issue**: If the transformation incorrectly applies to trunc nuw + sext with signed comparison, it could produce wrong results because nuw only guarantees no unsigned overflow, not that the high bits match the sign.

### Strategy 3: Test vector types with trunc nuw/nsw and icmp

- **Target**: Create vector versions of existing scalar tests (e.g., <4 x i16> trunc to <4 x i8>)
- **Rationale**: The code uses getScalarSizeInBits() suggesting vector support, but there are no vector test cases in the test file. Vector semantics might differ, especially for poison propagation across lanes.
- **Expected Issue**: Transformation might incorrectly handle vector types, especially if some lanes have poison values or if the nowrap flags don't apply uniformly across all lanes.

### Strategy 4: Test mixed nowrap flags (nuw on one side, nsw on other) with equality comparison

- **Target**: Add test for icmp eq (trunc nuw X), (trunc nsw Y) pattern
- **Rationale**: The code computes NoWrapFlags as intersection (line 1492-1493). For nuw & nsw, the result is 0, which should skip the transformation for unsigned/equality (line 1501-1502). However, equality comparisons don't care about signedness, so this transformation might actually be legal.
- **Expected Issue**: The transformation might be overly conservative, missing a legal optimization for equality comparisons with mixed nowrap flags. Or if it incorrectly transforms, it could produce wrong results.

### Strategy 5: Test zext nneg (not just zext) with trunc nsw and signed comparison

- **Target**: Create test with zext nneg instead of plain zext in trunc_signed_nsw_zext pattern
- **Rationale**: The code checks isa<SExtInst> to determine YIsSExt (line 1524-1525). For zext nneg, this returns false, so the transformation uses zext instead of sext. While zext nneg is semantically equivalent to sext for non-negative inputs, the transformation drops the nneg flag, which could affect correctness if the input can actually be negative.
- **Expected Issue**: If zext nneg is used but the input can be negative (violating the nneg contract), the transformation would produce zext (not sext), leading to incorrect signed comparison results. The original zext nneg would be poison for negative inputs, but the transformed zext would produce a defined (but wrong) value.

## Bugs Found

### Bug #1

**Analysis:**
This is a real bug. The original code has `zext nneg i8 %y to i16` which produces poison when %y is negative (violating the nneg contract). The transformation drops the nneg flag and produces `zext i8 %y to i32`, which gives a defined result (255 for input -1).

The semantic difference:
- Original: `zext nneg i8 -1 to i16` = poison → icmp with poison = poison → return poison = UB
- Transformed: `zext i8 -1 to i32` = 255 → `icmp sgt 255 -1000` = true → returns true (defined)

The transformation is incorrect because it changes undefined behavior (poison) into defined behavior. The fix should either:
1. Not transform when the zext has nneg flag and the comparison is signed, or
2. Preserve the nneg flag in the transformed zext, or
3. Use sext instead of zext when the original was zext nneg (since nneg implies the value is non-negative, sext and zext would give the same result for valid inputs)

The root cause is that the code at line 1524-1525 checks `isa<SExtInst>` to determine YIsSExt, but doesn't handle `zext nneg` specially. For `zext nneg`, the transformation should either use sext (to preserve signed semantics) or preserve the nneg flag.

### Analysis

### Bug Triggering Analysis
The bug is triggered when the input has `trunc nsw` + `zext nneg` with a signed comparison. The transformation drops the `nneg` flag from the zext and produces a plain `zext`. When the input violates the `nneg` contract (e.g., i8 value -1 = 0xFF), the original code produces poison (UB), but the transformed code produces a defined result (255).

Specifically:
- Original: `zext nneg i8 -1 to i16` = poison (because -1 violates nneg) → `icmp slt` with poison = poison → return poison = UB
- Transformed: `zext i8 -1 to i32` = 255 → `icmp sgt 255 X` = defined result

The difftest confirmed this: original returns UB (return_code 1 with "UB triggered: Return a poison value"), while transformed returns a defined value (return_code 255).

### Fix Weakness Analysis
The fix weakness is in the `foldICmpTruncWithTruncOrExt` function at lines 1520-1527. The code matches `trunc nsw` + `m_ZExtOrSExt` for all predicates, but the `YIsSExt` determination (line 1524-1525) only checks `isa<SExtInst>`, which returns false for `zext nneg`. This causes the transformation to use `zext` instead of `sext`, dropping the `nneg` flag in the process.

The fix should either:
1. Not transform when the extension is `zext nneg` and the comparison is signed, or
2. Preserve the `nneg` flag in the transformed `zext`, or
3. Use `sext` instead of `zext` when the original was `zext nneg` (since nneg implies non-negative, sext and zext give the same result for valid inputs, but sext preserves the signed semantics)

The root cause is that the code doesn't handle `zext nneg` specially - it treats it as plain `zext` and drops the `nneg` flag, changing the semantics from "poison if negative" to "defined result".
