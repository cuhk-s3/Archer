## Test Strategies

### Strategy 1: Mixed NoWrap Flags in GEP Chain

- **Target**: Create a chain of GEPs where different GEPs have different NoWrap flags (e.g., first GEP has nuw, second has nusw, third has inbounds)
- **Rationale**: The CommonPointerBase::compute function intersects all NoWrap flags. When EmitGEPOffsets sums the offsets, it uses the intersected flags for all additions. If the intersection loses important flags, the resulting comparison may be incorrect or miss optimization opportunities.
- **Expected Issue**: The optimization may incorrectly apply NoWrap flags to offset additions that could overflow, producing poison values, or may fail to fold when it should because the intersection is too conservative.

### Strategy 2: Vector Pointer GEP Chain Comparison

- **Target**: Create a chain of GEPs with vector pointer types compared to the base vector pointer
- **Rationale**: The EmitGEPOffsets function has special handling for vector types with CreateVectorSplat when offset types don't match IdxTy. With GEP chains, the interaction between vector splatting and offset summation needs verification.
- **Expected Issue**: Incorrect vector splatting or type mismatches when summing offsets from multiple GEPs in a chain with vector pointer types.

### Strategy 3: GEP Chain with Multiple Uses

- **Target**: Create a GEP chain where intermediate GEPs have multiple uses beyond the icmp comparison
- **Rationale**: When RewriteGEPs=true, EmitGEPOffset may rewrite GEP instructions to avoid duplicating arithmetic. With a chain, rewriting one GEP could affect other uses of intermediate GEPs in the chain.
- **Expected Issue**: Use-after-free or incorrect code generation when intermediate GEPs in the chain are rewritten/destroyed but still have other uses.

### Strategy 4: Empty or Single-Element GEP Chain Edge Case

- **Target**: Test cases where the GEP chain degenerates (GEPLHS equals RHS directly, or chain has only one GEP)
- **Rationale**: When Base.LHSGEPs is empty, EmitGEPOffsets returns a null constant. The code path should handle this correctly. Also need to verify the single-GEP case still works as before the patch.
- **Expected Issue**: Incorrect comparison result when the GEP chain is empty (should compare 0 cmp 0) or regression in the original single-GEP case.

### Strategy 5: Signed vs Unsigned Predicate with Mixed Sign Flags

- **Target**: Test icmp with signed predicates (slt, sgt) on GEP chains where some GEPs have nusw but not nuw flags
- **Rationale**: The NewICmp function converts signed to unsigned comparison if !NW.hasNoUnsignedWrap(). With a chain, the intersection of flags might lose nuw even if individual GEPs have it, affecting predicate selection.
- **Expected Issue**: Incorrect predicate selection (signed vs unsigned) leading to wrong comparison results when offsets can be negative.
