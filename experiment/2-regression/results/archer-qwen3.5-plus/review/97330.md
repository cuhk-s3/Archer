## Test Strategies

### Strategy 1: Test ComputeNumSignBits with incomplete SimplifyQuery context

- **Target**: Calls to ComputeNumSignBits in SimplifyDemandedUseBits for Shl, LShr, and AShr instructions
- **Rationale**: The patch changes SimplifyDemandedBits to accept a SimplifyQuery parameter, but three calls to ComputeNumSignBits (lines 658, 740, 779 in InstCombineSimplifyDemanded.cpp) only pass Q.CxtI instead of using a SimplifyQuery-aware overload. This loses important context like DominatorTree, AssumptionCache, DomConditionCache, and CondContext that could affect the sign bit analysis.
- **Expected Issue**: When ComputeNumSignBits is called with only CxtI instead of the full SimplifyQuery, it may miss optimization opportunities that rely on dominator information, assumptions, or conditional context. This could result in suboptimal code generation or missed simplifications in cases where sign bit information could be derived from control flow or assumptions.

### Strategy 2: Test sign bit analysis with DomConditionCache disabled

- **Target**: Shl/LShr/AShr instructions where sign bits depend on dominating conditions
- **Rationale**: The patch mentions retaining original context instruction for one-use chains, but the ComputeNumSignBits calls lose the DC (DomConditionCache) from the SimplifyQuery. If sign bit analysis depends on conditions from dominating blocks, this information is lost.
- **Expected Issue**: Sign bit count may be underestimated when the analysis could use dominating condition information, potentially preventing valid simplifications like eliminating unnecessary shifts.

### Strategy 3: Test sign bit analysis with CondContext

- **Target**: Shift instructions inside select or conditional contexts
- **Rationale**: The SimplifyQuery can carry CondContext (CC) which provides information about conditions that hold in the current context. The ComputeNumSignBits calls that only pass Q.CxtI lose this CondContext information.
- **Expected Issue**: When analyzing sign bits in a conditional context (e.g., inside a select where a condition implies certain bit values), the analysis may return fewer sign bits than actually known, preventing optimizations.

### Strategy 4: Test sign bit analysis with AssumptionCache

- **Target**: Shift instructions where sign bits can be derived from assume intrinsics
- **Rationale**: The SimplifyQuery carries AssumptionCache (AC) which can provide information from assume intrinsics. The ComputeNumSignBits calls that only pass Q.CxtI lose this AC information.
- **Expected Issue**: Sign bit analysis may miss information from assume intrinsics, leading to conservative results that prevent valid optimizations.

### Strategy 5: Test one-use chain context retention for sign bit analysis

- **Target**: One-use chains involving shift operations
- **Rationale**: The commit message mentions 'marginally strengthens the analysis by retaining the original context instruction for one-use chains'. However, the ComputeNumSignBits calls may not properly retain this context through the SimplifyQuery, potentially breaking the intended strengthening.
- **Expected Issue**: For one-use chains, the context instruction may not be properly propagated to ComputeNumSignBits, causing the analysis to use incorrect or less precise context, potentially leading to wrong sign bit counts.

### Analysis

### Bug Analysis Summary

The patch modifies `SimplifyDemandedBits` to accept a `SimplifyQuery` parameter, enabling future use of `CondContext` and strengthening analysis by retaining context for one-use chains. However, there's a potential issue:

**Issue Identified**: Three calls to `ComputeNumSignBits` in `InstCombineSimplifyDemanded.cpp` (lines 658, 740, 779) only pass `Q.CxtI` instead of using a full `SimplifyQuery`-aware overload. This loses important context:
- `Q.DT` (DominatorTree)
- `Q.AC` (AssumptionCache)
- `Q.DC` (DomConditionCache)
- `Q.CC` (CondContext)
- `Q.IIQ` (InstrInfoQuery)
- `Q.CanUseUndef`

**Test Results**: All verification tests passed (no bugs found). The transformations were proven correct by alive2, and differential testing showed no execution differences. This suggests that either:
1. The current code paths don't critically depend on the lost context for correctness
2. The `ComputeNumSignBits` function internally reconstructs sufficient context from `CxtI`
3. The potential issue is a missed optimization opportunity rather than a correctness bug

**Conclusion**: The patch appears to be correct but may have missed an opportunity to fully leverage the `SimplifyQuery` context in `ComputeNumSignBits` calls. The fix is a preparatory change for future enhancements (CondContext support) and the marginal strengthening mentioned in the commit message. No actual bugs were found - the transformations preserve semantics correctly.
