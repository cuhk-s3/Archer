## Test Strategies

### Strategy 1: Multi-use scalar with external users requiring original bitwidth

- **Target**: Test case with a scalar value that is used both within the vectorized tree (where it can be truncated) and by an external user that requires the original bitwidth
- **Rationale**: The patch modifies collectValuesToDemote to check MultiNodeScalars but the logic for determining if a value can be safely demoted may not properly account for all external uses. The IsPotentiallyTruncated function checks if upper bits are zero, but external users may need the full original bitwidth even if the value could theoretically fit in a smaller type.
- **Expected Issue**: The vectorizer may incorrectly truncate a multi-use scalar, causing the external user to receive a truncated value that loses important bits, leading to incorrect computation results.

### Strategy 2: Mixed sign extension patterns in trunc/ext subtrees

- **Target**: Test case with interleaved zext/sext/trunc operations where different subtrees have different signedness requirements
- **Rationale**: The patch introduces per-subtree minbitwidth detection with GetOperandSignedness lambda. However, the logic for determining signedness (checking SExt opcode or isKnownNonNegative) may not correctly handle cases where operands have mixed signedness or where the signedness depends on runtime values. The patch assumes consistent signedness within a subtree.
- **Expected Issue**: Incorrect sign extension/truncation when operands have mixed signedness, causing sign bit corruption or incorrect zero-extension behavior.

### Strategy 3: Reduction bitwidth mismatch with final cast

- **Target**: Test case with reduction operations where the computed ReductionBitWidth differs from the MinBWs entry for the root
- **Rationale**: The patch adds code at the end of vectorizeTree to perform a final resize cast if ReductionBitWidth != 0 and differs from MinBWs. The cast uses IsSigned from MinBWs lookup, but if ReductionBitWidth was computed from different values than those in MinBWs, the signedness may be incorrect. Also, the cast is inserted at ReductionRoot's position which may not be the correct insertion point.
- **Expected Issue**: Incorrect final cast for reduction results - either wrong signedness (zext vs sext) or incorrect insertion point causing use-def order violations.

### Strategy 4: CastMaxMinBWSizes early exit condition too aggressive

- **Target**: Test case with store/insertelement root where CastMaxMinBWSizes ratio is <= 2 but demotion would still be profitable
- **Rationale**: The patch has an early exit condition: if (IsStoreOrInsertElt || UserIgnoreList) && TruncNodes.size() <= 1 && (!CastMaxMinBWSizes || CastMaxMinBWSizes->second == 0 || CastMaxMinBWSizes->first / CastMaxMinBWSizes->second <= 2) return. This ratio check of 2 may be too conservative, preventing beneficial demotions when the max/min bitwidth ratio is small but demotion would still reduce register pressure.
- **Expected Issue**: Missed optimization opportunities where demotion would be beneficial but is skipped due to the ratio check, resulting in suboptimal code with higher register pressure.

### Strategy 5: Depth level limit causing incomplete subtree analysis

- **Target**: Test case with deep trunc/ext chains where MaxDepthLevel limit prevents full analysis of profitable demotion candidates
- **Rationale**: The patch introduces MaxDepthLevel tracking with limits (Limit = 2 or 3). The collectValuesToDemote function returns false if MaxDepthLevel <= Limit and certain conditions aren't met. For deep expression trees with multiple levels of casts, this limit may prevent the analysis from reaching profitable demotion points deeper in the tree.
- **Expected Issue**: Incomplete minbitwidth analysis for deep expression trees, causing missed demotion opportunities or inconsistent bitwidth assignments across the tree.
