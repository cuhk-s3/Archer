## Test Strategies

### Strategy 1: Test new variable non-negativity constraint stack tracking

- **Target**: Test case with multiple new variables introduced in unsigned constraints
- **Rationale**: The fix adds Var >= 0 constraints for new variables in the unsigned system, and each such constraint creates a separate DFSInStack entry. The assertion was only adjusted for function arguments, but the new variable constraints also affect the count. Need to verify the assertion correctly accounts for all non-negativity constraints.
- **Expected Issue**: Assertion failure 'updates to CS and DFSInStack are out of sync' when multiple new variables are introduced in unsigned constraints, because the assertion subtracts only FunctionArgs.size() but doesn't account for the relationship between new variable constraints and their stack entries.

### Strategy 2: Test signed predicate with variables only known non-negative in unsigned system

- **Target**: Test case using signed predicates on variables that are only marked non-negative in the unsigned system
- **Rationale**: The fix only adds Var >= 0 to the UNSIGNED system. If the signed/unsigned transfer logic incorrectly uses unsigned system information when processing signed predicates, it could make incorrect deductions. The transfer logic at lines 819-824 checks IsKnownNonNegative which queries the signed system, but the new constraints are only in the unsigned system.
- **Expected Issue**: Incorrect optimization where a signed comparison is simplified based on non-negativity information that only exists in the unsigned system, potentially leading to wrong code generation for negative inputs.

### Strategy 3: Test function with no arguments but new variables

- **Target**: Test case with a function that has no arguments but introduces new variables through constraints
- **Rationale**: The constructor fix adds Var >= 0 for function arguments, but if there are no arguments, this loop does nothing. The addFact fix should still add Var >= 0 for new variables. Need to verify this works correctly when FunctionArgs is empty.
- **Expected Issue**: Missing non-negativity constraints for new variables when the function has no arguments, causing missed optimizations or incorrect behavior in the unsigned system.

### Strategy 4: Test constraint removal when new variable goes out of scope

- **Target**: Test case where a new variable's constraint goes out of scope
- **Rationale**: When a new variable is added, two stack entries are created: one for the main constraint (with ValuesToRelease) and one for Var >= 0 (with empty ValuesToRelease). When the scope ends, both entries should be popped correctly. The Var >= 0 entry has empty ValuesToRelease, so no variable mapping is removed for it, but the constraint should still be popped.
- **Expected Issue**: Constraint system corruption when variables go out of scope - either the Var >= 0 constraint is not properly removed, or the variable mapping is incorrectly modified, leading to stale constraints or incorrect variable indices.

### Strategy 5: Test getConstraintForSolving interaction with new Var >= 0 constraints

- **Target**: Test case that triggers getConstraintForSolving with predicates that might interact with the new Var >= 0 constraints
- **Rationale**: The getConstraintForSolving function has special handling at lines 716-724 to return trivially true for V UGE 0 patterns. With the new Var >= 0 constraints in the system, this special case might interact unexpectedly, either causing redundant constraints or preventing legitimate constraint propagation.
- **Expected Issue**: Redundant constraint processing or missed optimization opportunities when the trivial V UGE 0 handling interacts with the explicitly added Var >= 0 constraints in the unsigned system.

### Bug #1

**Analysis:**
This is a real bug! The original IR has UB because the add nuw instruction with n=3 and -2 produces poison (3 + 254 = 257 which overflows 8-bit). The constraint elimination pass incorrectly optimizes away the branch condition, assuming it's always false, but this changes the semantics - the original would have UB while the transformed version doesn't trigger UB and returns a concrete value. The fix incorrectly assumes variables are non-negative without considering that nuw operations can produce poison when they overflow.

**Original LLVM IR:**
```llvm

define i1 @test_n_must_ule_2_due_to_nuw(i8 %n, i8 %i) {
entry:
  %sub.n.1 = add nuw i8 %n, -2
  %add = add nuw i8 %i, %sub.n.1
  %c.1 = icmp uge i8 %i, %add
  br i1 %c.1, label %if.then, label %if.end

  %t = icmp ule i8 %n, 2
  ret i1 %t
```

### Analysis

### Bug Triggering Analysis

The bug is triggered when:
1. A function has `add nuw` operations that can overflow and produce poison
2. The constraint elimination pass analyzes the unsigned comparison involving the result
3. The fix adds Var >= 0 constraints for all variables in the unsigned system
4. The pass incorrectly deduces that the branch condition is always false

With inputs n=3, i=1:
- Original: `sub.n.1 = add nuw i8 3, -2` = `add nuw i8 3, 254` = 257 which overflows 8-bit, producing poison
- The branch `icmp uge i8 1, poison` produces poison
- Branching on poison is UB

After optimization:
- The pass incorrectly determines the branch condition is always false (based on constraint analysis that assumes non-negative values)
- The branch is replaced with `br i1 false`, eliminating the UB
- This changes the program semantics (UB vs defined behavior)

### Fix Weakness Analysis

The fix adds Var >= 0 constraints for all variables in the unsigned system, but it fails to account for:
1. **Poison propagation**: Variables derived from `nuw`/`nsw` operations that overflow produce poison, not valid non-negative values
2. **Constraint validity**: The constraint system assumes all variables represent valid integer values, but poison values don't satisfy Var >= 0
3. **UB preservation**: The optimization must preserve UB semantics - if the original has UB, the optimized version must also have UB

The fix incorrectly treats poison-producing operations as producing valid non-negative values, leading to incorrect constraint deductions and UB elimination.
