## Test Strategies

### Strategy 1: Known sign bit with exact Inf pattern

- **Target**: BitCast from integer with known sign bit and exact infinity bit pattern
- **Rationale**: The patch clears the sign bit from InfKB.Zero to represent both +Inf and -Inf, but when comparing Bits == InfKB, if Bits has a known sign bit, the comparison fails because InfKB has unknown sign. This causes the code to miss recognizing known ±Inf when the sign is known.
- **Expected Issue**: The analysis fails to deduce fcInf for a bitcast where the integer source has a known sign bit and exact infinity pattern, missing optimization opportunities.

### Strategy 2: Known sign bit with exact Zero pattern

- **Target**: BitCast from integer with known sign bit and exact zero bit pattern
- **Rationale**: Similar to the Inf case, ZeroKB has its sign bit cleared, causing comparison failure when Bits has a known sign bit. The code misses recognizing known ±Zero when the sign is known.
- **Expected Issue**: The analysis fails to deduce fcZero for a bitcast where the integer source has a known sign bit and exact zero pattern.

### Strategy 3: Partial NaN pattern with unknown fraction bits

- **Target**: BitCast where exponent bits are all known-1 but fraction bits are partially unknown
- **Rationale**: The NaN detection checks if Bits.One (unknown=0) is NaN and if ~Bits.Zero (unknown=1) is not NaN. When exponent is all 1s but fraction has unknown bits, the logic might not correctly determine if the value must be NaN, cannot be NaN, or is uncertain.
- **Expected Issue**: Incorrect fpclass deduction for values with all exponent bits set but partially unknown fraction bits - may incorrectly rule out NaN or incorrectly assert must-be-NaN.

### Strategy 4: Non-IEEE float bitcast

- **Target**: BitCast to non-IEEE float types like x86_fp80 or ppc_fp128
- **Rationale**: The patch only applies NaN/Inf/Zero detection for Ty->isIEEE(). For non-IEEE floats, only sign bit transfer occurs. This might miss optimization opportunities or the isIEEE() check might have edge cases.
- **Expected Issue**: Missing fpclass deductions for non-IEEE float bitcasts, or incorrect behavior if isIEEE() returns unexpected values for certain float types.

### Strategy 5: Vector bitcast with per-element known bits variation

- **Target**: BitCast from integer vector where different elements have different known bit patterns
- **Rationale**: For vector types, the code uses getScalarType() and computeKnownBits with DemandedElts. Different vector elements might have different known bits, and the analysis might not correctly handle per-element fpclass deduction.
- **Expected Issue**: Incorrect or overly conservative fpclass deduction for vector bitcasts where elements have varying known bit patterns.

### Analysis

### Bug Analysis

**Bug Triggering Analysis:**
The patch in ValueTracking.cpp attempts to deduce floating-point class (fpclass) information when bitcasting from integer to float. The bug is in the Inf/Zero detection logic:

```cpp
auto InfKB = KnownBits::makeConstant(
    APFloat::getInf(Ty->getFltSemantics()).bitcastToAPInt());
InfKB.Zero.clearSignBit();  // Makes sign bit "unknown"
if (const auto InfResult = KnownBits::eq(Bits, InfKB)) {
  assert(!InfResult.value());
  Known.knownNot(fcInf);
} else if (Bits == InfKB) {  // BUG: This comparison fails when Bits has known sign
  Known.KnownFPClasses = fcInf;
}
```

The issue is that `InfKB.Zero.clearSignBit()` makes the sign bit unknown in the template, but when comparing `Bits == InfKB`, if `Bits` has a known sign bit, the comparison fails because:
- `Bits.Zero` has sign bit = 1 (known to be 0 for positive)
- `InfKB.Zero` has sign bit = 0 (unknown after clearSignBit)
- They don't match, so fcInf is not deduced

**Test Case:**
The test `test_const_plus_inf_bug` bitcasts the constant 0x7F800000 (+Inf) to float. The KnownBits analysis knows all bits exactly:
- Bits.One = 0x7F800000
- Bits.Zero = 0x807FFFFF (sign bit known-0, fraction bits known-0)

But InfKB after clearSignBit has:
- InfKB.One = 0x7F800000
- InfKB.Zero = 0x007FFFFF (sign bit unknown)

The comparison `Bits == InfKB` fails because Bits.Zero != InfKB.Zero, so fcInf is not deduced. The output shows `nofpclass(nan ninf zero nsub nnorm)` which means the analysis knows it's NOT -Inf, but doesn't know it IS +Inf. It should show `nofpclass(nan zero sub norm)` with fcInf deduced.

**Fix Weakness Analysis:**
The fix's weakness is in the sign bit handling for Inf/Zero detection. The code clears the sign bit from the template to represent "either +Inf or -Inf", but then uses exact equality comparison which fails when the actual value has a known sign. The fix should either:
1. Not clear the sign bit and check both +Inf and -Inf separately, or
2. Use a compatibility check instead of exact equality (check if Bits is compatible with InfKB, not exactly equal)
