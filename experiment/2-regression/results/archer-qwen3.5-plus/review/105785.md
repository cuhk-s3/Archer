## Test Strategies

### Strategy 1: Vector type ucmp/scmp folding

- **Target**: Test vector type ucmp/scmp intrinsics where constraints imply a specific result
- **Rationale**: The checkAndReplaceCmp function uses ConstantInt::get(I->getType(), 1) and ConstantInt::getSigned(I->getType(), -1) to create replacement values. For vector types, this creates scalar constants which would cause type mismatches when calling replaceAllUsesWith.
- **Expected Issue**: Type mismatch assertion failure or incorrect IR generation when folding vector ucmp/scmp intrinsics

### Strategy 2: Poison operand handling for ucmp/scmp

- **Target**: Test ucmp/scmp with operands that may be poison due to nuw/nsw violations
- **Rationale**: Unlike min/max intrinsics which check isGuaranteedNotToBePoison before adding facts, the ucmp/scmp handling has no such check. If operands are poison but not observed, the pass may incorrectly fold the intrinsic based on constraints that assume well-defined values.
- **Expected Issue**: Incorrect folding of ucmp/scmp when operands are poison, leading to observable behavior changes in valid programs

### Strategy 3: i2 return type edge case

- **Target**: Test ucmp/scmp with i2 return type where result should be -1
- **Rationale**: The code uses ConstantInt::getSigned(I->getType(), -1) for the LT case. For i2 type, -1 in 2-bit signed representation is 3 (binary 11), but the expected semantic value is -1. Need to verify if this creates the correct bit pattern.
- **Expected Issue**: Incorrect constant value when folding ucmp/scmp to -1 with i2 return type

### Strategy 4: Signed/unsigned constraint system mismatch

- **Target**: Test cases where signed and unsigned constraint systems give different results for ucmp/scmp
- **Rationale**: The checkCondition function uses getGTPredicate/getLTPredicate which return appropriate signed/unsigned predicates. However, the constraint decomposition and solving must use the correct constraint system (signed vs unsigned). A mismatch could cause incorrect folding.
- **Expected Issue**: Incorrect folding when signed predicate is checked against unsigned constraints or vice versa

### Strategy 5: Wide integer type handling (i128+)

- **Target**: Test ucmp/scmp with i128 or wider integer types
- **Rationale**: The ConstraintElimination pass has known issues with wide integer types due to 64-bit internal coefficient representation. The ucmp/scmp folding may incorrectly handle comparisons involving i128 values where internal decomposition overflows.
- **Expected Issue**: Incorrect folding of ucmp/scmp with wide integer types due to internal coefficient overflow

### Analysis

### Bug Analysis Summary

After comprehensive testing of the ConstraintElimination patch for ucmp/scmp intrinsics, no bugs were found. The fix correctly:

1. **Folds ucmp/scmp based on established constraints**: When the constraint system establishes GT, LT, or EQ relationships between operands, the intrinsic is correctly folded to 1, -1, or 0 respectively.

2. **Handles signed/unsigned correctly**: The pass correctly uses signed predicates for scmp and unsigned predicates for ucmp, and does not incorrectly fold when there's a signedness mismatch (as shown in scmp_4 and ucmp_4 negative tests).

3. **Handles wide integer types (i128)**: The constraint system correctly handles i128 comparisons without internal overflow issues.

4. **Handles i2 return type**: The ConstantInt::getSigned correctly produces -1 (binary 11) for i2 type.

5. **Handles vector types**: The pass does not crash on vector type ucmp/scmp intrinsics.

6. **Does not fold when operands may be poison**: The pass correctly does not fold ucmp/scmp when the constraint system cannot establish the relationship due to potential poison values.
