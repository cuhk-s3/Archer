## Test Strategies

### Strategy 1: Test srem with zero divisor (OrZero case)

- **Target**: foldSelectWithSRem pattern matching
- **Rationale**: The code uses isKnownToBeAPowerOfTwo(Remainder, OrZero=true), which allows Remainder to be zero. However, srem x, 0 is undefined behavior. The transformation would incorrectly fold this to x & -1 = x, losing the UB semantics.
- **Expected Issue**: The optimization incorrectly folds a select pattern where the divisor could be zero, transforming UB (srem by zero) into defined behavior (bitwise and), causing miscompilation.

### Strategy 2: Test srem overflow case (INT_MIN % -1)

- **Target**: foldSelectWithSRem with negative power-of-two divisor
- **Rationale**: The isKnownToBeAPowerOfTwo check with OrZero=true could potentially allow -1 (which is 2^N-1, not a power of 2, but edge cases in analysis might miss this). More importantly, srem INT_MIN, -1 is UB due to overflow. The transformation should not apply when the divisor could be negative.
- **Expected Issue**: If the power-of-two analysis incorrectly identifies certain negative values, or if the divisor can be -1 in some lane of a vector, the transformation could fold UB into defined behavior.

### Strategy 3: Test vector with mixed zero/non-zero lanes

- **Target**: foldSelectWithSRem with vector types
- **Rationale**: For vector types, isKnownToBeAPowerOfTwo with OrZero=true could return true even if some lanes are zero and others are power-of-two. This would cause the transformation to apply when some lanes have UB (srem by zero) while others don't.
- **Expected Issue**: Vector lanes with zero divisors would have their UB incorrectly transformed to defined bitwise-and operations, while other lanes are correctly transformed, causing inconsistent behavior across lanes.

### Strategy 4: Test with noundef attribute on divisor

- **Target**: foldSelectWithSRem pattern with noundef divisor
- **Rationale**: If the divisor has noundef attribute and is known to be non-zero power-of-two, the transformation should be safe. Testing this ensures the optimization works correctly when preconditions are explicitly guaranteed.
- **Expected Issue**: This is a positive test case - should verify the transformation correctly applies when the divisor is guaranteed non-zero via noundef or range attributes.

### Strategy 5: Test with poison-producing srem operand

- **Target**: foldSelectWithSRem when Op or Remainder could be poison
- **Rationale**: The original pattern has srem which could produce poison if the divisor is poison or zero. The transformation to x & (n-1) removes the srem entirely. If the srem was supposed to produce poison (e.g., from a poison divisor), the transformation would incorrectly produce defined results.
- **Expected Issue**: The transformation eliminates the srem instruction, potentially losing poison propagation that should occur when the divisor operand is poison or produces poison.

## Bugs Found

### Bug #1

**Analysis:**
This is a real bug. The original code has `srem i32 %x, %divisor` where divisor can be 0 (when %b is false). Division by zero is undefined behavior in LLVM. The original code correctly triggers UB when executed with divisor=0. However, the optimization transforms this to `and i32 %x, 7` which produces a defined result (5 & 7 = 5) instead of triggering UB. This is a miscompilation because the optimization incorrectly folds a pattern that contains UB into a pattern with defined behavior. The root cause is the `OrZero=true` parameter in `isKnownToBeAPowerOfTwo(Remainder, /*OrZero*/ true)` which allows the divisor to be zero, but srem by zero is UB and should not be optimized away.

**Original LLVM IR:**
```llvm

define i32 @test_zero_divisor(i32 %x, i1 %b) {
  %divisor = select i1 %b, i32 8, i32 0
  %rem = srem i32 %x, %divisor
  %cond = icmp slt i32 %rem, 0
  %add = add i32 %rem, %divisor
  %sel = select i1 %cond, i32 %add, i32 %rem
  ret i32 %sel
}

define i32 @main(i32 %argc, ptr %argv) {
entry:
  %r = call i32 @test_zero_divisor(i32 5, i1 false)
  ret i32 %r
}

```

### Bug #5

**Analysis:**
This is a different bug! The original code has urem x, 0 which is UB, but since %b=true, the urem x, 0 result (%r1) is never used - only %r2 is selected. However, the optimization folds this to `urem x, (select b, z, y)` which evaluates the select first, then does urem. But the original code evaluates BOTH urem operations before the select, so urem x, 0 is evaluated and triggers UB. The transformation changes the evaluation order, causing UB to be triggered in the original but not in the transformed. This is a miscompilation - the optimization incorrectly hoists the select before the urem operations, changing when UB is triggered. Note: This is a urem issue, not the srem foldSelectWithSRem bug we identified earlier, but it's still a real bug in InstCombine's handling of remainder operations with selects.
