## Test Strategies

### Strategy 1: Test integer mul/add with vector types

- **Target**: expr-reassociate-int.ll
- **Rationale**: The patch uses I.getType()->isIntOrIntVectorTy() to determine the type. Test with vector integer types to ensure the type checking works correctly for vectors and that the correct multiplication instruction is generated.
- **Expected Issue**: Incorrect handling of vector integer types could lead to wrong multiplication instructions being generated or incorrect limit checking.

### Strategy 2: Test mixed integer and FP operations in expression tree

- **Target**: expr-reassociate-int.ll
- **Rationale**: The code uses I.getType() consistently to determine the operation type. Test scenarios where the expression tree might have mixed types (though this should be invalid IR) or edge cases where type inference could be ambiguous.
- **Expected Issue**: Type mismatch issues if the expression tree has inconsistent types, potentially causing incorrect transformations or crashes.

### Strategy 3: Test integer overflow flags (nuw/nsw) preservation

- **Target**: expr-reassociate-int.ll
- **Rationale**: The patch uses Builder.CreateMul() for integer types but doesn't explicitly handle overflow flags (nuw/nsw). The original integer mul instruction may have nuw/nsw flags that should be preserved in the hoisted multiplication. Test cases with nuw/nsw flags to verify they are properly handled.
- **Expected Issue**: Loss of nuw/nsw flags in the hoisted multiplication could change program semantics when overflow occurs, leading to poison values not being generated when expected.

### Strategy 4: Test limit boundary conditions for integer reassociation

- **Target**: expr-reassociate-int.ll
- **Rationale**: The patch introduces IntAssociationUpperLimit separate from FPAssociationUpperLimit. Test boundary conditions where the number of reassociations equals, exceeds, or is one less than the limit to verify the limit checking works correctly.
- **Expected Issue**: Off-by-one errors in limit checking could cause either missed optimizations or too many transformations being performed.

### Strategy 5: Test instruction insertion point for integer vs FP

- **Target**: expr-reassociate-int.ll
- **Rationale**: For FP types, the code passes 'Ins' (the user instruction) to CreateFMulFMF as an insert point. For integer types, CreateMul doesn't use this insert point. This could lead to different insertion order which might affect debugging or instruction scheduling. Test to verify the insertion point behavior is correct.
- **Expected Issue**: Different insertion points for integer vs FP could lead to suboptimal code placement or debugging issues, though likely not a correctness issue.

## Bug Analysis Summary

After comprehensive testing of the LICM patch that extends `hoistFPAssociation` to support integer mul/add operations (renamed to `hoistMulAddAssociation`), no bugs were found. All test cases passed both Alive2 verification and differential execution testing.

### Test Coverage

The following test strategies were thoroughly tested across all 4 test cases:

1. **Test limit boundary conditions for integer reassociation**: Tested with multiple terms (2, 3, 5 terms) and verified that the `IntAssociationUpperLimit` correctly controls when transformations are applied. When the limit is exceeded, the transformation is correctly skipped.

2. **Test integer overflow flags (nuw/nsw) preservation**: Tested with `nuw nsw` flags on integer operations. The transformation correctly preserves these flags in the hoisted multiplication instructions.

3. **Test integer mul/add with vector types**: Tested with `<2 x i64>` vector types. The transformation correctly handles vector integer types using `isIntOrIntVectorTy()` check.

4. **Test mixed integer and FP operations in expression tree**: Verified that the code correctly distinguishes between integer and FP operations using the `isReassociableOp()` helper function.

5. **Test instruction insertion point for integer vs FP**: Verified that the transformation produces correct results regardless of the insertion point differences between `CreateMul` and `CreateFMulFMF`.

### Fix Analysis

The patch correctly:
- Uses `I.getType()->isIntOrIntVectorTy()` to determine the operation type before accessing the type (fixing the use-after-free bug mentioned in the commit message)
- Applies separate limits (`IntAssociationUpperLimit` vs `FPAssociationUpperLimit`) for integer and FP operations
- Creates the correct multiplication instruction (`CreateMul` for integers, `CreateFMulFMF` for FP)
- Properly handles the `isReassociableOp()` check for both integer (always reassociable) and FP (requires `allowReassoc` and `noSignedZeros` flags)

All transformations were verified to be semantically correct by Alive2 and produced identical execution results in differential testing.
