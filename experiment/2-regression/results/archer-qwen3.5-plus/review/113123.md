## Test Strategies

### Strategy 1: FMF Union vs Intersection Bug

- **Target**: FastMathFlags propagation in the range check transformation
- **Rationale**: The fix uses union (|) of FastMathFlags from both fcmp instructions when creating the new fabs and fcmp. However, other similar transformations in the same file (lines 1327-1329, 1368-1370) use intersection (&). Using union can incorrectly propagate flags like 'nnan' when only one of the original fcmps had that flag, potentially changing semantics for NaN inputs.
- **Expected Issue**: When one fcmp has 'nnan' flag and the other doesn't, the transformed code will have 'nnan' on fabs/fcmp, causing undefined behavior or incorrect results when the input is NaN, whereas the original code would correctly return false for ordered comparisons with NaN.

### Strategy 2: NaN Input with Mixed FMF Flags

- **Target**: Test case with NaN input where LHS and RHS have different FastMathFlags
- **Rationale**: Create a test where LHS fcmp has 'nnan' flag but RHS doesn't (or vice versa), then pass a NaN value. The original code should handle NaN correctly (returning false for ordered comparisons), but the transformed code with union FMF may have 'nnan' on fabs, causing incorrect behavior.
- **Expected Issue**: The transformed code may produce poison or incorrect boolean result when given NaN input, while the original code would correctly return false.

### Strategy 3: Negative Zero Edge Case

- **Target**: Test with negative zero input to verify fabs sign bit handling
- **Rationale**: IEEE-754 specifies fabs(-0.0) = +0.0. Test the transformation with -0.0 input to ensure the semantics are preserved. While fabs correctly handles this, we should verify the comparison predicates work correctly with the sign change.
- **Expected Issue**: Potential mismatch in behavior if the original comparisons relied on signed zero semantics that fabs changes.

### Strategy 4: Vector Type with Poison Elements

- **Target**: Vector fcmp with poison/undef elements in the constants
- **Rationale**: The code uses m_APFloatAllowUndef for matching constants. Test with vector types where some lanes have poison/undef in the constant operands to ensure the transformation doesn't propagate poison to lanes that were previously protected by the logical operation.
- **Expected Issue**: Poison may be propagated to vector lanes that were previously safe due to short-circuit evaluation or the structure of the original comparisons.

### Strategy 5: Predicate Swap Logic Verification

- **Target**: The predicate swapping logic in lines 1445-1448
- **Rationale**: The code swaps predicates and constants when IsLessThanOrLessEqual(IsAnd ? PredR : PredL) is true. Test edge cases where the predicate ordering might cause incorrect predicate to be used in the final fcmp, especially for the 'or' case with different predicate combinations.
- **Expected Issue**: Incorrect predicate may be used in the transformed fcmp, leading to wrong boolean results for certain input ranges.

## Bug Analysis

### Bug Triggering Analysis
The bug is triggered when the InstCombine pass transforms a floating-point range check idiom with asymmetric FastMathFlags (FMF).

**Test case**: `and (fcmp nnan olt x, C), (fcmp ogt x, -C)` where only the first comparison has the `nnan` flag.

**Input**: NaN value (0x7FF8000000000000)

**Original behavior**:
- `fcmp nnan olt NaN, C` returns false (ordered comparison with NaN returns false, nnan is just a hint)
- `fcmp ogt NaN, -C` returns false
- Result: `and(false, false) = false` - no UB

**Transformed behavior**:
- The transformation uses union of FMF flags: `nnan | (no flags) = nnan`
- `fabs nnan NaN` causes undefined behavior because `nnan` promises the input is not NaN
- Result: UB/crash

### Fix Weakness Analysis
The fix in `InstCombineAndOrXor.cpp` lines 1451-1452 uses:
```cpp
Builder.setFastMathFlags(LHS->getFastMathFlags() | RHS->getFastMathFlags());
```

This **union** of FMF flags is incorrect. Other similar transformations in the same file (lines 1327-1329, 1368-1370) correctly use **intersection** (&):
```cpp
FastMathFlags FMF = LHS->getFastMathFlags();
FMF &= RHS->getFastMathFlags();
Builder.setFastMathFlags(FMF);
```

The union operation incorrectly propagates restrictive flags like `nnan` to the new `fabs` intrinsic when only one of the original fcmps had that flag.

### Verified Bug Cases
Multiple tests confirmed this bug:
- Test index 2 (`test_and_olt_fmf_propagation_union`): Original returns 0, transformed returns 1 (UB)
- Test index 9 (`test_and_olt_fmf_propagation`): Original returns 0, transformed returns 1 (UB)
- Test index 24 (`test_and_olt_mixed_fmf`): Original returns 0, transformed returns 1 (UB)
- Test index 1 (`test_and_olt_commuted_mixed_fmf`): Original returns 0, transformed returns 1 (UB)

All tests confirm the bug with NaN input when FMF flags are asymmetric between the two fcmp instructions.
