## Test Strategies

### Strategy 1: Test AnyReplaced bug with multiple operands

- **Target**: Instruction with multiple operands where only some are replaced
- **Rationale**: The AnyReplaced logic uses assignment (=) instead of OR (|=), so if the last operand isn't replaced but earlier ones are, AnyReplaced will be false and the function will incorrectly return nullptr.
- **Expected Issue**: Missed optimization opportunity where simplification should occur but doesn't due to incorrect AnyReplaced tracking

### Strategy 2: Test absorber return value semantics

- **Target**: Select with absorber pattern where RepOp differs from Absorber
- **Rationale**: The fix changes the absorber case to return Absorber instead of RepOp. If RepOp and Absorber have different semantics in certain contexts, this could cause incorrect code generation.
- **Expected Issue**: Incorrect constant folding where the returned absorber constant doesn't match the expected semantic value

### Strategy 3: Test x^x pattern with indirect replacement

- **Target**: Xor or Sub instruction where both operands simplify to RepOp through recursive replacement
- **Rationale**: The new check requires both NewOps to equal RepOp exactly. If the recursive replacement produces equivalent but not identical values, the simplification might be missed.
- **Expected Issue**: Missed x^x or x-x simplification when operands are recursively replaced to equivalent but not pointer-equal values

### Strategy 4: Test cross-lane vector operations with bitcast

- **Target**: Vector operation where Op is vector but instruction involves bitcast to different vector type
- **Rationale**: The fix checks for ShuffleVectorInst and CallBase but may miss bitcast operations that change vector lane structure, leading to incorrect per-lane assumptions.
- **Expected Issue**: Incorrect simplification of vector operations that mix data across lanes through bitcast

### Strategy 5: Test recursion depth exhaustion

- **Target**: Deeply nested instruction chain with replacement opportunity
- **Rationale**: The recursive calls share the same MaxRecurse counter. With deep nesting, the recursion limit might be hit prematurely, causing missed optimizations or inconsistent behavior.
- **Expected Issue**: Inconsistent optimization results based on instruction chain depth, with deeper chains failing to simplify
