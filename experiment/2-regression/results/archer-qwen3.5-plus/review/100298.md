## Test Strategies

### Strategy 1: Poison propagation through select arm simplification

- **Target**: select with poison-generating operations in arms
- **Rationale**: The optimization simplifies select arms to constants based on KnownBits from the condition. However, if the original arm contains poison-generating operations (like overflow flags, exact division, etc.) that would produce poison for certain inputs, replacing it with a constant removes the poison-generating behavior. This changes semantics when the condition is false and the arm would have produced poison.
- **Expected Issue**: The optimized code produces a defined constant value instead of poison, changing program semantics when the original arm would have produced poison.

### Strategy 2: Multi-use instruction corruption

- **Target**: select arms with multi-use instructions
- **Rationale**: The hasAffectedValue function checks if values in the select arm are affected by the condition, but it doesn't verify that the affected instructions are single-use. If an instruction in the select arm has multiple uses outside the select, computing KnownBits with the injected condition and replacing the arm with a constant could be incorrect for other uses that don't have the same condition context.
- **Expected Issue**: Instructions with multiple uses get incorrectly simplified based on condition context that only applies to the select, corrupting values for other uses.

### Strategy 3: Vector type element-wise poison handling

- **Target**: select with vector types containing poison elements
- **Rationale**: The optimization handles vector types but may not correctly account for poison elements in vectors. When computing KnownBits for a vector arm, if some lanes contain poison that would be masked by the select condition, replacing the entire vector with a constant could propagate defined values to lanes that should remain poison.
- **Expected Issue**: Vector select arms with poison elements in some lanes get replaced with fully-defined constants, incorrectly defining values that should be poison.

### Strategy 4: FCmp condition handling gap

- **Target**: select with floating-point comparison conditions
- **Rationale**: Looking at findValuesAffectedByCondition, there's a bug at line 9675: it checks `match(Cond, m_FCmp(...))` but should check `match(V, m_FCmp(...))` since V is the value being processed in the worklist, not Cond. This means FCmp conditions may not properly populate AffectedValues, leading to missed optimizations or incorrect behavior when FCmp is the condition.
- **Expected Issue**: Select statements with floating-point comparison conditions may have incorrect AffectedValues sets, causing either missed optimizations or incorrect KnownBits computation.

### Strategy 5: Overflow flag (nsw/nuw) preservation

- **Target**: select arms with overflow-flagged operations
- **Rationale**: When a select arm contains operations with nsw/nuw flags that could produce poison on overflow, and the condition implies the operation won't overflow, the optimization replaces the arm with a constant. However, this removes the overflow check semantics - if the condition analysis is wrong or the input changes, the original code would produce poison but the optimized code produces a defined value.
- **Expected Issue**: Operations with nsw/nuw flags in select arms get replaced with constants, removing overflow detection that was present in the original code.

## Bugs Found

### Bug #1

**Analysis:**
This is a real bug. The original code has `add nsw i32 %a, 1` which produces poison when %a = 2147483647 (INT_MAX) due to signed overflow. The select instruction then selects this poison value when the condition is true.

The optimization incorrectly simplifies the select arm by computing KnownBits under the assumption that the condition is true. It determines that when %a = 2147483647, the result of `add nsw i32 %a, 1` would be -2147483648 (the wrapped value). However, this ignores the fact that `add nsw` produces poison on overflow, not the wrapped value.

The fix fails to account for poison-generating flags (nsw/nuw) on operations in the select arm. When the condition implies an overflow scenario, the original operation would produce poison, but the optimization replaces it with a defined constant, changing the program semantics.

This directly relates to the "Poison propagation through select arm simplification" and "Overflow flag (nsw/nuw) preservation" strategies identified in Phase 1.

**Original LLVM IR:**
```llvm

define i32 @test_nsw_overflow(i32 %a, i32 %b) {
  %cond = icmp eq i32 %a, 2147483647
  %x = add nsw i32 %a, 1
  %result = select i1 %cond, i32 %x, i32 %b
  ret i32 %result
}

define i32 @main(i32 %argc, ptr %argv) {
entry:
  %r = call i32 @test_nsw_overflow(i32 2147483647, i32 100)
  ret i32 %r
}

```
