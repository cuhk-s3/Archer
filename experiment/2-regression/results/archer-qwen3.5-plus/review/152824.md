## Test Strategies

### Strategy 1: Test vector constants with mixed sign bits including NaN

- **Target**: Vector constant handling in computeKnownFPClass
- **Rationale**: The vector constant handling code (lines 4317-4344) tracks SignBitAllZero and SignBitAllOne separately, but only sets Known.SignBit when they differ. If a vector contains NaN elements with different sign bits, the logic may incorrectly infer the sign bit. NaN sign bits are implementation-defined and should not be relied upon.
- **Expected Issue**: Incorrect sign bit inference for vectors containing NaN elements with varying sign bits, leading to incorrect copysign optimizations

### Strategy 2: Test FMul with NaN operands having known sign bits

- **Target**: FMul sign bit inference at lines 4967-4972
- **Rationale**: The code infers sign bit for FMul when both operands have known sign bits, but only checks if operands are never NaN at lines 4959-4965. However, the sign bit inference at 4967-4972 doesn't verify that the result won't be NaN (e.g., 0 * Inf = NaN). If the result is NaN, the sign bit is undefined.
- **Expected Issue**: Incorrect sign bit inference when FMul produces NaN (0 * Inf), causing wrong copysign simplification

### Strategy 3: Test min/max with NaN operands and signed zeros

- **Target**: Min/Max sign bit inference at lines 4598-4617
- **Rationale**: The sign bit inference for min/max operations at lines 4598-4617 is guarded by Known.isKnownNeverNaN(), but the condition at line 4605-4609 for maximum/minimum checks zero handling. The interaction between signed zeros and the sign bit inference may be incorrect when one operand is +0 and another is -0.
- **Expected Issue**: Incorrect sign bit inference for min/max operations with mixed signed zeros, leading to wrong fabs/copysign optimizations

### Strategy 4: Test copysign with sign argument that is known negative NaN

- **Target**: InstCombine copysign optimization at lines 2415-2428
- **Rationale**: The new code uses computeKnownFPSignBit which can return true (negative), false (positive), or nullopt. When the sign argument is a NaN with known negative sign bit, the optimization reduces copysign to fneg(fabs). However, copysign with NaN sign argument should preserve NaN payload semantics, and the transformation may not preserve this correctly.
- **Expected Issue**: Incorrect transformation of copysign when sign argument is a negative NaN, potentially losing NaN payload or changing semantics

### Strategy 5: Test knownNot method interaction with existing sign bit

- **Target**: KnownFPClass::knownNot method at lines 359-367
- **Rationale**: The knownNot method infers sign bit when NaN is ruled out and SignBit is nullopt (!SignBit). However, if SignBit was already set to a value, this inference is skipped. The interaction between knownNot being called after sign bit is already set could lead to inconsistent state if KnownFPClasses is updated but SignBit isn't re-verified.
- **Expected Issue**: Inconsistent KnownFPClass state when knownNot is called after sign bit is already determined, potentially causing incorrect optimizations in edge cases

### Analysis

### Bug Triggering Analysis

The test case `test_vector_all_negative_nan` triggers a bug in the sign bit inference for vector constants containing NaN values. The patch merges `cannotBeOrderedLessThanZeroImpl` into `computeKnownFPClass` to improve sign bit inference. However, the vector constant handling code (lines 4317-4344 in ValueTracking.cpp) incorrectly infers the sign bit for vectors containing NaN elements.

In the test case:
1. A vector of two negative NaN constants is created via bitcast
2. This vector is used as the sign argument to `copysign`
3. The optimization incorrectly infers that the sign bit is known to be 1 (negative) because all elements have the same sign bit pattern
4. It then transforms `copysign(x, neg_nan_vec)` to `fneg(fabs(x))`

The problem is that NaN sign bits are implementation-defined and should not be relied upon for optimization. According to IEEE 754, the sign bit of a NaN has no semantic meaning for most operations. The transformation changes the semantics:
- Original: `copysign(x, NaN)` should return a NaN with the sign from the NaN argument
- Optimized: `fneg(fabs(x))` returns a negative non-NaN value for non-NaN input

### Fix Weakness Analysis

The fix weakness is in the vector constant handling code in `computeKnownFPClass`:

```cpp
if (VFVTy && CV) {
  Known.KnownFPClasses = fcNone;
  bool SignBitAllZero = true;
  bool SignBitAllOne = true;

  for (unsigned i = 0; i != NumElts; ++i) {
    const APFloat &C = CElt->getValueAPF();
    Known.KnownFPClasses |= C.classify();
    if (C.isNegative())
      SignBitAllZero = false;
    else
      SignBitAllOne = false;
  }
  if (SignBitAllOne != SignBitAllZero)
    Known.SignBit = SignBitAllOne;
  return;
}
```

The code tracks `SignBitAllZero` and `SignBitAllOne` but doesn't exclude NaN elements from this analysis. When all elements are negative NaNs, `SignBitAllZero` becomes false and `SignBitAllOne` stays true, so `Known.SignBit` is set to true. However, NaN sign bits should not be used for sign bit inference because:
1. NaN sign bits are implementation-defined
2. copysign with a NaN sign argument has special semantics that differ from copysign with a non-NaN sign argument

The fix should exclude NaN elements from sign bit inference for vector constants, or at least not infer a known sign bit when the value is known to be NaN.
