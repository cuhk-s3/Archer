# Subsystem Knowledge for Local

## Elements Frequently Missed

* **Poison-Generating Flags (`nsw`, `nuw`, `exact`)**: When replacing an instruction that does not support poison-generating flags (such as an `extractvalue` from an overflow intrinsic) with a standard binary operator that does, the optimization pass frequently misses stripping these flags. Because the original instruction lacks a flag representation, default intersection logic fails, leaving the replacement instruction overly restrictive.
* **Call Return Attributes (`nonnull`, `range`, `align`)**: During Common Subexpression Elimination (CSE) or Global Value Numbering (GVN), the optimization pass frequently misses reconciling return attributes when replacing a dominated call with a dominating call. Failing to intersect or drop incompatible attributes causes the replacement call to produce `poison` under conditions where the original call would have produced a well-defined value.
* **Analysis Cache Invalidation (Instruction Pointers)**: Local utility transformations (like PHI deduplication) frequently miss notifying the parent optimization pass when instructions are deleted and their memory is freed. This leaves stale instruction pointers in analysis caches (e.g., MemoryDependenceAnalysis), which corrupts compiler assumptions if a new instruction is allocated at the same memory address.

## Patterns Not Well Handled

### Pattern 1: Value Replacement with Asymmetric Poison Semantics
Optimization passes like CSE, GVN, and InstCombine often identify two instructions as computing the same mathematical or logical value and attempt to replace one with the other. However, they do not well handle the asymmetry in poison semantics between the two instructions. If the dominating or replacement instruction carries more restrictive metadata, attributes (e.g., `nonnull` on a call), or flags (e.g., `nsw` on an `add`) than the replaced instruction, it can evaluate to `poison` for certain inputs. While this `poison` might have been safely masked by control flow in the replacement instruction's original context, propagating it to the replaced instruction's uses introduces miscompilations. The passes fail to properly intersect, drop, or strip these restrictive properties when the instruction types differ (e.g., `extractvalue` vs. `add`) or when handling call instructions.

### Pattern 2: Uncoordinated IR Mutation and Stale Analysis Caches
Local optimization utilities often perform localized IR cleanups, such as identifying and deleting duplicate PHI nodes to simplify the control flow graph. This pattern is not well handled because these utilities directly mutate the IR and free instruction memory without coordinating with the higher-level optimization pass that invoked them. Because passes maintain internal analysis caches keyed by raw instruction memory addresses, the silent deletion leaves dangling pointers in the cache. When the memory allocator subsequently reuses the exact same address for a newly created instruction, the optimization pass erroneously retrieves and applies the stale analysis data to the new instruction, leading to invalid transformations based on incorrect data flow or memory dependence assumptions.
