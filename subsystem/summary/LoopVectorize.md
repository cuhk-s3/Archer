# Subsystem Knowledge for LoopVectorize

## Elements Frequently Missed

*   **Outside Users of Loop-Defined Values (LCSSA Phi Nodes)**: The uniformity analysis frequently misses or ignores users outside the loop when determining if an instruction is uniform. This leads to incorrect scalarization where the vectorizer extracts from the first lane (lane 0) instead of the last active lane required by the outside user.
*   **Poison Generation in Speculated Lanes**: The vectorizer misses the implications of evaluating potentially undefined operations (like out-of-bounds shifts) in inactive or speculated vector lanes. When these `poison` values feed into vector boolean reductions (e.g., `llvm.vector.reduce.or`) for early exits, the `poison` propagates and corrupts the entire control flow.
*   **Internal SCEV Cache References**: Invalidation mechanisms frequently miss internal Scalar Evolution (SCEV) cache references (such as Backedge Taken Counts) when loop transformations (like peeling or vectorization) modify the Control Flow Graph. The invalidation only traverses IR use-def chains, leaving stale, invalid expressions attached to LCSSA Phi nodes.
*   **Scalar Vectorization Factors (VF=1) with Interleaving**: The internal logic for identifying the first active lane and extracting live-out values misses support for cases where the vectorization factor is 1 but the interleave count is greater than 1. This causes the compiler to bypass necessary extraction steps for early exits.
*   **Dead or Unreachable Control Flow Paths in Structural Checks**: During the VPlan code generation phase, structural invariance checks miss the broader context of dead or unreachable control flow paths that advanced analyses (like SCEV) successfully navigate. This causes loop-invariant values to be incorrectly treated as varying vectors.

## Patterns Not Well Handled

### Pattern 1: Live-Out Value Extraction and Lane Selection
The loop vectorizer struggles significantly with extracting the correct scalar value from a vector for uses outside the loop (live-outs). This pattern manifests in several ways depending on the analysis applied. If uniformity analysis incorrectly flags an induction update as uniform because it ignores outside users, the vectorizer extracts from lane 0 instead of the final active lane. If the cost model and code generation phases disagree on whether a value is loop-invariant (advanced SCEV analysis vs. structural VPlan checks), the vectorizer may attempt to extract the last lane of a varying vector that actually contains garbage data from a dead control flow path. Furthermore, when interleaving is applied without vectorization (VF=1, IC>1), the vectorizer completely fails to generate the logic required to identify which interleaved part triggered an early exit, leading to the extraction of an incorrect live-out value.

### Pattern 2: Speculative Evaluation of Uncountable Early Exits
Loops with uncountable early exits (e.g., data-dependent breaks) require the vectorizer to evaluate exit conditions concurrently across multiple lanes and reduce the results. This high-level pattern is poorly handled because it inherently involves speculative execution of iterations that would normally be skipped in scalar execution. If the exit condition involves operations that can produce `poison` (such as an oversized bit shift on data from a later, unreachable iteration), this `poison` is injected into the vector lanes. Because the vectorizer uses bitwise reductions (like `llvm.vector.reduce.or`) to collapse the lane conditions into a single scalar branch condition, a single `poison` lane contaminates the entire reduction. This results in undefined behavior and miscompilation of the loop's control flow.

### Pattern 3: State and Analysis Inconsistencies Across Compiler Phases
A recurring pattern of failure arises from discrepancies between different analysis and transformation phases within the compiler pipeline. The optimization pass often relies on powerful, advanced analyses (like SCEV) to make initial cost and predication decisions, but falls back on weaker, structural checks during actual code generation (VPlan). For example, SCEV might prove a stored value is loop-invariant despite complex PHI nodes, prompting the vectorizer to plan a uniform unconditional store. However, VPlan's structural checks reject the invariance, forcing a varying vector extraction that yields incorrect data. Similarly, when loop transformations alter the CFG by adding predecessors to exit blocks, they fail to thoroughly invalidate the advanced SCEV state. This leaves a dangerous inconsistency where subsequent passes query SCEV and receive stale, mathematically incorrect expressions for LCSSA Phi nodes, leading to infinite loops or incorrect trip counts.
