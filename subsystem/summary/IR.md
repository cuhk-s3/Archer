# Subsystem Knowledge for IR

## Elements Frequently Missed

*   **Poison-Generating Attributes on Instructions**: The analysis logic frequently overlooks specific flags and attributes that imply poison generation or undefined behavior (UB) upon execution. Specifically, the `samesign` flag on `icmp` instructions and the `noundef` attribute on function calls are often missed. While the optimizer checks for general side effects (like memory writes), it fails to account for these attributes, leading to incorrect assumptions that the instructions are unconditionally safe to speculate or hoist.
*   **Signed Arithmetic Properties of 1-bit Integers (`i1`)**: The optimizer frequently misinterprets the mathematical properties of 1-bit integers. It often treats the constant `1` as a positive identity or a standard increment, missing the fact that in a 1-bit signed context, the bit pattern `1` represents `-1`. This leads to errors in `nsw` (No Signed Wrap) inference and range calculations.
*   **Semantic Distinctions in Cast Operations**: The optimizer misses the semantic difference between bit-reinterpretation instructions (`bitcast`) and value-conversion instructions (`fpext`, `fptrunc`). It tends to focus solely on bit-width equality, incorrectly assuming that a `bitcast` between two types of the same size can be elided when followed by a conversion, ignoring that `bitcast` changes the interpretation of the bits (e.g., bfloat to half) while conversions preserve the numeric value.
*   **Overflow in Compiler-Side Range Calculation**: When calculating value ranges for optimization (e.g., for `llvm.abs`), the internal logic used by the compiler to compute bounds can overflow when dealing with small bit-widths (like `i1`). This results in wrapped values (e.g., `1 + 1 = 0`) being used as valid bounds, creating erroneous "empty" or incorrect ranges.

## Patterns Not Well Handled

### Pattern 1: Safety Analysis for Speculative Execution
The optimization pass struggles to correctly identify instructions that are safe to speculate (execute eagerly or hoist). The pattern involves instructions that appear safe regarding memory and exceptions (e.g., marked `readnone` or `nounwind`) but contain conditional UB triggers dependent on operand values (such as `noundef` or `samesign`). The optimizer incorrectly assumes that absence of side effects equates to safety from UB. Consequently, it hoists these instructions to control flow paths where the operands might be `poison` or `undef`, triggering undefined behavior that would not have occurred in the original program flow.

### Pattern 2: Arithmetic and Range Analysis on Extremal Bit-Widths
The optimizer handles standard integer widths (i32, i64) correctly but fails to handle the corner cases presented by 1-bit integers (`i1`). This pattern appears when the compiler attempts to infer flags (like `nsw`) or calculate value ranges (for `abs` or multiplication). The logic often applies standard arithmetic rules—such as "multiplying by 1 is safe" or "max value + 1 expands the range"—which break down in 1-bit arithmetic where the representable range is only `[0, -1]`. This leads to the inference of impossible constraints or the addition of overflow flags that are immediately violated by valid inputs.

### Pattern 3: Folding of Heterogeneous Cast Sequences
The optimizer poorly handles sequences of cast instructions that involve both bit-reinterpretation (`bitcast`) and floating-point conversion (`fpext`, `fptrunc`). The pattern arises when the optimizer attempts to fold a two-step conversion (Type A -> Type B -> Type C) into a single step (Type A -> Type C). If Type A and Type B share the same bit width but different internal formats (e.g., `bfloat` and `half`), the optimizer incorrectly assumes the intermediate `bitcast` is redundant. By removing it, the optimizer forces the final conversion to interpret the raw bits of Type A using the semantics of Type A, rather than the reinterpreted semantics of Type B, resulting in numeric miscalculations.
