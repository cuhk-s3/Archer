# Subsystem Knowledge for ScalarEvolution

## Elements Frequently Missed

*   **Poison-Generating Flags (`nsw`, `nuw`, `samesign`) in New Contexts**: The optimization pass frequently misses the need to strip or recompute flags when an instruction's scope is expanded (e.g., hoisted or reused) or when its operands are modified (e.g., truncated). Flags that were valid in the original context often become invalid and generate poison in the new context.
*   **Undefined Behavior (UB) in Synthesized/Shifted Expressions**: While the pass checks the original IR for safety, it frequently misses performing the same UB checks (such as division by zero) on newly constructed mathematical models, like shifted backedge expressions or sequential min/max expansions.
*   **Control-Flow Guards and Short-Circuiting Semantics**: When expanding SCEV models back into LLVM IR, the pass frequently misses the original conditional guards that protected potentially faulting operations, leading to unconditional execution of unsafe instructions.
*   **Deep Poison Sources in Reused Instructions**: When reusing instructions that mathematically match a SCEV expression, the pass frequently misses poison that originates deeper in the operand tree or from operations that mathematically fold away (e.g., `mul x, 0`) but still propagate poison during actual execution.
*   **Sign Semantics During Bitwise Factorization**: The pass frequently misses the distinction between logical and arithmetic right shifts when factoring out powers of two from negative constant multipliers, inadvertently transforming small negative constants into large positive ones that cause overflows.
*   **Start Values in Zero-Stride Recurrences**: When analyzing loop exit limits for zero-stride recurrences, the pass frequently misses evaluating the initial start value before concluding that the loop is infinite and invoking UB assumptions.
*   **Internal SCEV Cache References**: During cache invalidation after Control Flow Graph (CFG) modifications, the pass frequently misses internal SCEV references (such as Backedge Taken Counts) that rely on altered LCSSA phi nodes, leading to stale data retention.

## Patterns Not Well Handled

### Pattern 1: Instruction Reuse and Hoisting without Contextual Safety Validation
When the SCEV expander or induction variable simplification passes attempt to reduce redundancy, they often reuse or hoist existing instructions that evaluate to the desired mathematical expression. However, this pattern relies heavily on mathematical equivalence while ignoring execution-context differences. The pass fails to handle the fact that the reused instruction might carry poison-generating flags (`nsw`, `nuw`) or implicit poison from its operands/folded operations that are strictly tied to its original, narrower scope. Consequently, reusing these instructions in a broader or different control-flow context introduces poison into previously safe execution paths, leading to miscompilations.

### Pattern 2: Mathematical Modeling and Expansion Introducing Unconditional UB
SCEV models complex loop behaviors using abstract mathematical expressions, such as shifting backedge values to represent PHI nodes or creating sequential expressions to model conditionally executed logic. This pattern is poorly handled when these abstract models are evaluated or expanded back into IR. The transformation process strips away the original control-flow guards or iteration-specific states that made the operations safe. As a result, potentially faulting operations (like integer division) are emitted unconditionally or evaluated at unsafe points (like loop entry), introducing Undefined Behavior that did not exist in the original program.

### Pattern 3: Type Conversions and Factorizations Breaking Flag and Sign Assumptions
To simplify expressions or match types, SCEV frequently applies transformations like truncating operands or factoring out common powers of two (trailing zeros) from constants. This pattern is not well handled because the pass assumes that high-level properties, such as the `samesign` flag or `nsw`/`nuw` flags, inherently survive these bitwise alterations. Truncation can change the sign bit of an integer, and using logical right shifts on negative numbers destroys their sign and magnitude. By failing to drop these flags or use sign-aware operations (like arithmetic shifts), the compiler makes invalid logical deductions or forces overflows, severely corrupting predicate logic and arithmetic results.

### Pattern 4: Edge-Case Loop Recurrence and Exit Limit Assumptions
SCEV attempts to deduce loop finiteness and compute trip counts based on the stride of loop recurrences. A specific pattern that is poorly handled is the zero-stride recurrence. When the pass observes a step of zero, it prematurely assumes that the recurrence value will never change, concluding that the loop must be infinite if the exit condition relies on it. In C/C++, infinite loops without side effects are considered UB, so the compiler aggressively optimizes based on this UB assumption. However, it fails to handle the edge case where the start value of the recurrence already satisfies the exit condition on the first iteration, meaning the loop would terminate normally. This leads to the erroneous elimination of valid loops and subsequent instructions.

### Pattern 5: CFG Modifications Desyncing the SCEV Cache
Loop transformations such as vectorization, unrolling, or peeling structurally alter the CFG, often adding new predecessors to loop exit blocks. This pattern changes the nature of LCSSA phi nodes from trivial to non-trivial. The issue arises because the SCEV invalidation mechanism is not well-equipped to handle these structural shifts deeply. It typically only traverses direct IR use-def chains to invalidate caches. It fails to track down and invalidate internal, derived SCEV expressions (like cached loop trip counts) that were built upon the previously trivial phi nodes. This leaves stale, invalid expressions in the SCEV cache, poisoning subsequent compiler analyses that query these values.
