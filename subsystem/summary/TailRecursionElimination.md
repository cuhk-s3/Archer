# Subsystem Knowledge for TailRecursionElimination

## Elements Frequently Missed

*   **Parameter Memory Attributes (`readonly`, `readnone`)**: The optimization pass frequently misses checking, updating, or stripping memory-related attributes attached to function parameters. When the pass alters the memory contents of a parameter to set up the next loop iteration, it fails to remove attributes that promise immutability.
*   **Attribute Invalidation Logic during Memory Reuse**: The pass misses the necessary logic to invalidate function-level guarantees when it decides to reuse and mutate memory allocations (like `byval` arguments) that were originally guaranteed not to be mutated during the function's execution.
*   **Semantic Conflicts in `byval` Argument Mutation**: The pass misses the semantic implications of modifying `byval` arguments. While `byval` implies a local copy is made for the callee, modifying this local copy violates other attached attributes (like `readonly`) that downstream passes rely on.

## Patterns Not Well Handled

### Pattern 1: Reusing `byval` Memory with Conflicting Immutability Attributes
When transforming a tail-recursive call into a loop, Tail Recursion Elimination (TRE) must update the arguments for the next iteration. For `byval` arguments, the pass optimizes this by inserting memory copy instructions (e.g., `llvm.memcpy`) to overwrite the original parameter's memory with the new values. However, this pattern is not well handled when the original parameter is decorated with immutability attributes such as `readonly`. The pass fails to strip these attributes from the function signature. This creates a severe semantic contradiction in the IR: memory explicitly marked as read-only is being written to. Downstream optimization passes rely on the `readonly` attribute, falsely assuming the memory is never modified, which leads to miscompilations like incorrectly eliding memory loads or optimizing terminating conditions into infinite loops.

### Pattern 2: Function-Level Attribute Invalidation via Loop Transformation
Tail recursion elimination inherently changes the lifecycle and mutability of function parameters. A parameter that is constant and read-only for the duration of a single recursive call frame becomes a mutable state variable when the recursion is flattened into a loop. The pass does not well handle the reconciliation of function-level guarantees (attributes) with the newly introduced intra-function mutations. It assumes that reusing the parameter's memory is a safe, localized optimization without considering that the function signature's global promises are violated by the loop's backedge updates. This pattern of transforming inter-frame parameter passing into intra-frame memory mutation without updating the function's contract is a fundamental oversight in the pass.
