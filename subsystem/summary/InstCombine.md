## Elements Frequently Missed

*   **Poison-Generating Metadata and Attributes**: The optimizer frequently fails to strip or invalidate attributes like `noundef`, `!range`, and `samesign` when transforming instructions. When a conditional operation (which guarded against invalid inputs) is converted to an unconditional one, or when operands are simplified, these attributes often remain, escalating benign `poison` values into immediate Undefined Behavior.
*   **Fast Math Flags (FMF) Propagation**: Flags such as `ninf` (no infinity), `nnan` (no NaN), and `nsz` (no signed zeros) are often incorrectly propagated. The optimizer frequently applies flags from a result to its operands, or from a specific branch of a `select` to a hoisted common instruction, ignoring that the flags may not be valid for the unselected path or the raw operands.
*   **Wrap Flags (`nsw`, `nuw`, `inbounds`)**: These flags are often incorrectly preserved during reassociation, operand swapping, or instruction merging (e.g., PHI of GEPs). The optimizer misses that changing the order of operations or merging paths can violate the strict no-wrap requirements that held for the original sequence.
*   **`undef` vs. `poison` Semantics**: The distinction between `undef` (arbitrary bit pattern) and `poison` (deferred UB) is frequently missed. Issues arise when `undef` is refined to `poison` (making code more undefined) or when `undef` operands are duplicated (allowing them to resolve to different values in different uses), breaking logical equivalences.
*   **Vector Constants with Poison Elements**: Pattern matching logic often treats vector constants containing `poison` elements (e.g., `<i8 -1, i8 poison>`) as if they were fully defined constants (e.g., all-ones). This leads to incorrect folding where `poison` lanes propagate to the result.
*   **Pointer Provenance and Bounds**: The optimizer frequently treats pointer equality (`icmp eq`) as license to substitute one pointer for another. It misses that pointers with the same address may have different provenance or validity (e.g., one is in-bounds, the other is not). Additionally, `inbounds` is often preserved on GEPs without verifying that intermediate offsets in a transformed sequence remain within bounds.
*   **Lane-Crossing in Vectors**: When analyzing vector instructions, the optimizer often assumes that constraints on lane *i* apply to the data in lane *i*. It misses that instructions like `shufflevector` move data across lanes, invalidating these assumptions (e.g., applying a condition derived from lane 0 to data moved from lane 1).

## Patterns Not Well Handled

### Pattern 1: Linearization of Conditional Logic
The optimizer struggles when transforming conditional logic (like `select` instructions or control flow) into unconditional linear sequences (such as intrinsic calls, bitwise arithmetic, or `uadd.sat`). In the original code, the condition often acts as a guard, preventing the execution of operations on invalid inputs (e.g., shifting by a large amount, inputs violating `!range`). When the optimizer linearizes this code, it often fails to remove the strict constraints (metadata or flags) associated with the operation. Consequently, the new unconditional instruction executes on inputs that were previously filtered out, triggering Undefined Behavior or poison where the original code was safe.

### Pattern 2: Speculative Hoisting and Sinking
A recurring failure pattern involves hoisting instructions above a `select` (speculation) or sinking instructions into `select` operands. The optimizer incorrectly assumes that if an instruction is safe to execute on the selected path, it is safe to execute on *both* paths. This leads to the propagation of UB-implying attributes (like `noundef`, `ninf`, or `inbounds`) to the unselected operand. If the unselected value violates these attributes (e.g., it is `poison`, infinity, or out-of-bounds), the transformed code introduces Undefined Behavior on a valid execution path.

### Pattern 3: Context-Sensitive Analysis Applied Globally
The optimizer frequently misapplies context-sensitive information. It uses facts that are true only at a specific use-site (e.g., due to a dominating branch or `assume`) to simplify the definition of an instruction. If that instruction has multiple users, this simplification is invalid for users on other execution paths where the context does not hold. Similarly, it uses future context (like a subsequent truncation) to narrow an instruction (like division) without realizing that the instruction might receive large inputs on paths where the truncation does not occur, leading to spurious traps (e.g., division by zero after truncation).

### Pattern 4: Equivalence-Based Substitution
The optimizer often replaces a value with a logically "equivalent" value found elsewhere in the code, but fails to account for the "meta" properties of the replacement. For example, it might replace a computed value with a source value used in a comparison, or reuse an existing instruction for a funnel shift. However, the replacement value might carry poison-generating flags (like `nsw`) or be an unstable `undef` value. By substituting a "clean" value with a "constrained" or "unstable" one, the optimizer introduces poison or inconsistency that did not exist in the original data flow.

### Pattern 5: Complex Operand Canonicalization
When simplifying chains of operations—such as reassociating shifts, converting `shl` to `mul`, or swapping comparison operands—the optimizer often fails to update the auxiliary properties of the instruction. It might preserve `nsw` when converting a shift to a multiplication (ignoring `INT_MIN` edge cases), fail to switch from zero-extension to sign-extension when swapping comparison operands, or keep `inbounds` when merging GEPs with opposing offsets. The transformation logic correctly handles the mathematical equivalence but neglects the strict validity constraints attached to the original form.
