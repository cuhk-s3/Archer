# Subsystem Knowledge for CorrelatedValuePropagation

## Elements Frequently Missed

* **Out-of-Range Switch Cases**: The optimization pass frequently misses the proper identification and elimination of explicit `switch` cases that fall outside the known, restricted range of the switch condition. This happens due to inconsistencies or limitations in the analysis queries used to prove a case is dead.
* **`undef` Value Semantics**: The pass misses the deferred materialization property of `undef` values. When an operand contains `undef` (e.g., via a PHI node), the analysis optimistically assigns it a constrained value to satisfy a predicate, missing the fact that replacing an operation directly with the `undef`-tainted operand allows it to take arbitrary, potentially invalid values later in the pipeline.
* **Intrinsic-Specific Output Constraints**: The pass misses the strict output guarantees of certain intrinsics (like `llvm.abs` with `is_int_min_poison` set to `false`). It fails to recognize that stripping the intrinsic removes the structural guarantee of the output (e.g., being strictly non-negative), which cannot be safely replaced by an operand that might evaluate to an unconstrained `undef`.

## Patterns Not Well Handled

### Pattern 1: Flawed Switch Coverage and Default Reachability Analysis
This pattern occurs when the optimization pass attempts to determine if the `default` branch of a `switch` instruction is unreachable. The pass relies on comparing the total number of surviving explicit cases against the total number of possible values in the condition's known range. If the analysis fails to eliminate out-of-range cases, these invalid cases are incorrectly counted as "reachable." Consequently, the total count of cases artificially inflates and meets or exceeds the size of the valid range. The pass erroneously concludes that all possible valid runtime values are explicitly covered and marks the `default` branch as `unreachable`. This is not well handled because the pass uses a naive counting mechanism rather than strictly intersecting the actual values of the surviving cases with the known valid range of the condition.

### Pattern 2: Invalid Simplification of Constraining Intrinsics due to Optimistic `undef` Handling
This pattern involves the simplification of intrinsics that enforce specific value constraints, such as `llvm.abs` guaranteeing a non-negative result. When the operand of the intrinsic is a PHI node containing both a known constrained value (e.g., a positive integer) and an `undef` value, the underlying value tracking analysis optimistically assumes the `undef` can take a value that matches the constrained condition. Relying on this optimistic assumption, the pass completely removes the intrinsic and replaces it with the raw operand. This is poorly handled because it violates the semantics of the intrinsic: while the intrinsic guarantees a constrained output, the raw `undef` operand can later be materialized by the compiler as any arbitrary value (including negative values). The pass fails to treat `undef` as a full unconstrained range when the intrinsic's structural guarantees (like handling `INT_MIN` without yielding poison) are required.
