# Subsystem Knowledge for VectorCombine

## Elements Frequently Missed

* **Poison-Induced Undefined Behavior (UB) in Specific Operands**: The optimization pass frequently misses the severe consequences of introducing `poison` values into UB-sensitive operands. Specifically, feeding `poison` into the divisor of division/remainder instructions (`sdiv`, `udiv`, `srem`, `urem`) or the shift amount of shift instructions (`shl`, `lshr`, `ashr`) elevates a safely propagating `poison` value into immediate Undefined Behavior.
* **Operand Ordering for Non-Commutative Operations**: When vectorizing scalar operations, the pass misses the strict requirement to preserve Left-Hand Side (LHS) and Right-Hand Side (RHS) operand ordering for non-commutative binary operations (like shifts), leading to swapped arguments.
* **Target Endianness and Data Layout**: The pass misses querying the target's `datalayout` to determine endianness. It hardcodes little-endian assumptions when calculating bit offsets for vector-to-scalar packing, causing incorrect bit extractions on big-endian targets.
* **Scalar Element Byte-Size Validation**: When scalarizing memory operations, the pass misses checking whether the individual scalar element (e.g., `i1`) is byte-sized. It incorrectly relies only on the total vector size being a multiple of bytes, which leads to clobbering adjacent bits during scalar stores.
* **Strict Predicate Matching for Comparisons**: The pass misses enforcing exact base predicate equality when folding multiple `icmp` instructions. It incorrectly allows conditionally equivalent predicates (e.g., relying on `samesign` flags) to merge, applying assumptions from one instruction to operands that do not guarantee those conditions.
* **Out-of-Bounds Vector Indices**: The pass misses the fact that out-of-bounds indices in `insertelement` or `extractelement` evaluate to `poison`. When scalarizing these operations, it fails to account for this `poison` generation, which corrupts downstream scalar operations.
* **Shift Amount Bounds During Type Shrinking**: When narrowing the type of shift instructions, the pass misses validating that the shift amount remains strictly less than the new, smaller bitwidth, resulting in `poison` values when the shift amount exceeds the narrowed type's capacity.

## Patterns Not Well Handled

### Pattern 1: Hoisting or Scalarizing Poison-Generating Operations into UB-Sensitive Contexts
The pass frequently attempts to optimize code by hoisting structural operations (such as `shufflevector` with `poison` masks) above arithmetic operations, or by scalarizing vector operations involving out-of-bounds indices or narrowed types. While these transformations are structurally valid, they are not well handled because they often expose or generate `poison` values in new contexts. When these newly generated `poison` values are fed into the operands of UB-sensitive instructions—such as the divisor of an integer division/remainder or the shift amount of a shift instruction—the transformation incorrectly elevates a previously safe `poison` propagation into immediate Undefined Behavior.

### Pattern 2: Vectorizing or Folding Instructions with Incompatible Lane Properties
The pass attempts to combine multiple scalar operations into a single vector operation, or fold multiple vector operations (like shuffled compares) into a unified instruction. This pattern is not well handled because the pass relies on overly permissive matching logic. It fails to strictly validate the compatibility of the combined lanes. For example, it ignores operand ordering for non-commutative operations, leading to swapped arguments in the resulting vector instruction. Similarly, it blindly applies conditional flags (like `samesign`) from one lane's predicate to other lanes where the condition does not hold. This silently corrupts the logic of the program, altering control flow and producing incorrect boolean results.

### Pattern 3: Unsafe Lowering of Vector Layouts to Scalar Memory or Bitwise Arithmetic
To optimize vector element access, the pass often converts vector extractions/insertions into scalar memory accesses or packs the entire vector into a large integer for bitwise arithmetic. This pattern is not well handled because it makes unsafe, hardcoded assumptions about underlying memory and register layouts without respecting target or type constraints. It assumes little-endian bit offsets without querying the target's data layout, causing the wrong bits to be extracted on big-endian architectures. Furthermore, it assumes that if a whole vector is byte-sized, its individual elements are safely and independently addressable in memory. This leads to the emission of scalar memory operations for non-byte-sized elements (like `i1`), which the backend expands into full byte accesses, incorrectly clobbering adjacent vector elements during stores.
