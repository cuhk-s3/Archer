# Subsystem Knowledge for AliasAnalysis

## Elements Frequently Missed

*   **`nsw` (No Signed Wrap) Flags in Derived Computations**: The optimization pass frequently misses the fact that algebraic transformations (like negating a scale or adding two scales together) invalidate the `nsw` guarantee of the original instructions. Preserving these flags on newly formed expressions leads to incorrect assumptions about overflow (e.g., ignoring `INT_MIN` edge cases).
*   **Integer Overflow and Wraparound**: The analysis often misses the possibility of integer wraparound when evaluating pointer offsets. Specifically, it assumes that multiplying a strictly non-zero variable index by a constant scale will result in a non-zero offset, failing to account for cases where the multiplication wraps exactly to zero (especially in the absence of `inbounds` markers).
*   **Directionality of Partial Alias Offsets**: When performing recursive alias queries (e.g., looking through `phi` or `select` nodes), the analysis frequently misses the direction-dependent nature of partial alias offsets. Swapping the order of arguments in the recursive query without negating the returned offset results in inverted and incorrect overlap calculations.
*   **Ephemeral Values and Intrinsics (e.g., `llvm.assume`)**: The analysis misses the active presence of ephemeral instructions during pointer capture tracking. By prematurely ignoring these values under the assumption they will eventually be deleted, the analysis fails to recognize that the pointers they use are technically captured as long as the instruction remains in the IR.
*   **Stale Analysis Dependencies (Dominator Tree)**: The alias analysis misses the synchronization state of underlying analysis structures, such as the Dominator Tree. When queried by passes that actively modify the Control Flow Graph (CFG), it blindly trusts stale dominance data to make context-sensitive aliasing deductions.

## Patterns Not Well Handled

### Pattern 1: Unsafe Algebraic Simplification of Pointer Offsets
When Alias Analysis decomposes `getelementptr` (GEP) instructions to compare their offsets, it often performs algebraic simplifications such as merging shared variable indices (e.g., combining `A * x` and `B * x` into `(A + B) * x`) or subtracting offsets to find a relative difference (which negates the scale). This pattern is not well handled because the analysis incorrectly preserves poison-generating flags like `nsw` from the original instructions onto the new, simplified mathematical expressions. Because the new expressions (like a negated scale applied to `INT_MIN`, or a merged scale that exceeds bitwidth) can overflow even if the original ones did not, retaining the `nsw` flag causes the compiler to illegally deduce that certain overlapping memory locations can never alias, leading to false `NoAlias` results.

### Pattern 2: Blind Assumptions on Non-Zero Indices and Wrapping Math
The analysis struggles with handling variable GEP indices that are proven to be non-zero (via loop bounds, dominating conditions, etc.). The high-level pattern involves the compiler assuming that a non-zero index, when multiplied by a type's element size (the scale), guarantees a non-zero final offset. This is not well handled because it completely ignores integer modulo arithmetic. If the GEP lacks an `inbounds` keyword, a large non-zero index multiplied by a scale can wrap around the integer bitwidth and result in an exact offset of zero. By failing to account for this wrapping behavior, the analysis incorrectly concludes that the base pointer and the GEP pointer do not alias, allowing downstream passes to illegally eliminate or reorder memory accesses.

### Pattern 3: Directionality Loss in Recursive Alias Queries
To resolve aliasing for pointers derived from `phi` or `select` instructions, Alias Analysis recursively queries the incoming values of these instructions against the target memory location. This pattern is poorly handled because the analysis often swaps the left-hand and right-hand arguments during the recursive call to simplify the internal logic. However, partial alias results return a specific offset that represents the distance of one pointer relative to the other. Because the analysis forgets to invert (negate) this offset when the arguments are swapped back, it returns a mathematically incorrect overlap distance. Passes like Global Value Numbering (GVN) then use this corrupted offset to perform invalid load forwarding.

### Pattern 4: Premature Discarding of Ephemeral Uses in Capture Tracking
Pointer capture tracking attempts to determine if a pointer escapes the current function. A common pattern is to filter out "ephemeral" instructions—such as `llvm.assume` or instructions only used by it—because they do not contribute to the final executable code. This is not well handled because the filtering happens too early. If an ephemeral instruction captures a pointer, ignoring it causes the analysis to report that the pointer does not escape. Memory optimization passes (like Dead Store Elimination) rely on this report and may delete stores to that memory, assuming it is purely local. However, because the ephemeral instructions still exist in the IR at that moment, the deleted stores cause the program to read uninitialized or stale memory.

### Pattern 5: Contextual Alias Queries with Out-of-Sync Analysis Data
Optimization passes (like jump threading or load elimination) frequently query Alias Analysis while simultaneously mutating the Control Flow Graph (CFG). This pattern is not well handled because Alias Analysis relies heavily on the Dominator Tree to make advanced, context-sensitive deductions (e.g., proving that a specific store cannot clobber a load based on execution paths). When the CFG is modified, the Dominator Tree becomes temporarily stale. If Alias Analysis is queried before the tree is updated, it evaluates dominance relationships based on an outdated view of the program. This leads to incorrect `NoAlias` conclusions, causing the optimization pass to unsafely eliminate loads or bypass valid memory writes.
