# Subsystem Knowledge for SLPVectorizer

## Elements Frequently Missed

*   **Short-Circuiting and Poison-Masking Semantics**: The vectorizer frequently misses the implicit poison-masking behavior of scalar short-circuiting instructions (like `select`, logical AND/OR). When converting these to eager vector operations or reordering them, it fails to realize that unevaluated `poison` values in the scalar code will become evaluated and propagate in the vectorized code.
*   **Multi-Use and Shared Node Constraints**: When a scalar value or intermediate tree node is used in multiple different contexts, the vectorizer often only checks the constraints of one context. It misses the strict requirements of other external uses, leading to unsafe type demotions or truncations.
*   **Strict Poison vs. Undef Differentiation**: The optimization pass frequently treats `undef` and `poison` interchangeably or fails to check if a reused vector lane contains `poison`. It misses the fact that replacing `undef` with `poison` (or reusing a mask with `poison` lanes for a fully defined splat) violates IR semantics by making the program more poisonous.
*   **Instruction Flags on Mutated Operations**: Fast-math or optimization flags (such as `samesign` on `icmp`) are frequently missed or incorrectly retained when the underlying operands are mutated (e.g., truncated). The pass forgets to drop these flags when the transformation invalidates the assumptions they represent.
*   **Signedness Information Across Opaque Instructions**: The vectorizer frequently loses track of individual scalar signedness requirements (whether to `sext` or `zext`) when scalars are grouped into a uniform vector node, or when the values pass through instructions like `freeze` that obscure the original extension semantics.
*   **Dynamic/Non-Constant Indices**: The pass misses the conservative tracking required for `insertelement` operations with dynamic (non-constant) indices. Instead of assuming the insertion could affect any lane, it ignores the dynamic index, leading to incorrect element tracking and dropped instructions.
*   **64-bit Pointer Offsets**: During memory access analysis, the vectorizer misses the upper 32 bits of 64-bit pointer offsets. By truncating pointer differences to 32 bits, it incorrectly identifies widely separated, non-contiguous memory accesses as contiguous.
*   **Bitwidth-Sensitive Operands**: When demoting the bitwidth of alternate opcode sequences, the pass misses the specific operand constraints of bitwidth-sensitive instructions (like `shl`). It applies the demotion uniformly, causing shift amounts to exceed the new, narrower bitwidth.
*   **Cost-Model Artifacts**: The pass occasionally misses the distinction between temporary IR constructs (like subregister masks generated purely for cost estimation) and actual executable IR, erroneously leaking the cost-model masks into the final `shufflevector` instructions.

## Patterns Not Well Handled

### Pattern 1: Bitwidth Reduction and Type Demotion
The SLP vectorizer struggles significantly with safely reducing the bitwidth of vectorized integer operations. The core issue is that the pass applies demotion too aggressively without maintaining a global view of all constraints. It fails to handle shared nodes with conflicting strict/permissive contexts, incorrectly applies "ignore lists" to intermediate tree nodes instead of just roots, and loses track of mixed signedness requirements when extracting narrowed scalars for external uses. Furthermore, it blindly applies demotion to alternate opcode sequences without respecting bitwidth-sensitive operations (like shifts), and forgets to strip optimization flags (like `samesign`) that become invalid in the narrower type.

### Pattern 2: Shuffle Mask Generation for Gathers and Mixed-Size Vectors
Generating correct `shufflevector` masks when combining multiple source vectors into a single gather or build-vector operation is a highly error-prone pattern. The vectorizer frequently miscalculates index offsets when the source vectors have different lengths (often basing the offset solely on the first vector's length or the resulting mask's size). It also struggles with calculating the correct slice sizes when the number of gathered scalars diverges from the original tree entry count, and fails to correctly compose masks when gathered scalars are reused across multiple lanes. This results in malformed shuffles that extract the wrong elements or pad the vector with poison.

### Pattern 3: Vectorization of Short-Circuiting Boolean Reductions
The pass does not well handle the conversion of scalar boolean reduction chains (e.g., `select` chains representing logical AND/OR) into horizontal vector reductions. Scalar chains inherently short-circuit, safely masking `poison` or `undef` values if an earlier condition dictates the result. Vector reductions, however, evaluate all lanes eagerly. The vectorizer fails to insert necessary `freeze` instructions or preserve the evaluation order, causing `poison` values from out-of-bounds extractions or undefined lanes to propagate into the final reduction result, fundamentally breaking the original program semantics.

### Pattern 4: Unification of Alternate Opcodes and Identity Operations
When the vectorizer encounters isomorphic chains with different opcodes (e.g., `xor X, 0` and `mul X, 1`), it attempts to unify them into a single vector instruction by treating identity operations as universally interchangeable. This pattern is poorly handled because the pass assumes it can simply pick one opcode and blend the constants. It fails to realize that there is no valid semantic translation between fundamentally different operations (bitwise vs. arithmetic). Consequently, it generates flawed constant conversions (e.g., `mul X, 0`), destroying the identity semantics and zeroing out valid data.

### Pattern 5: Vector and Instruction Reuse with Poison/Undef
To optimize code generation, the vectorizer attempts to reuse existing vectors or `shufflevector` instructions (such as splats or gathers) that match the required operands. However, the matching logic is flawed because it does not strictly verify the compatibility of `poison` and `undef` lanes. It will replace an `undef` operand with a reused vector lane containing `poison`, or reuse a splat mask containing `poison` lanes for an operation that strictly requires valid data. This pattern makes the resulting IR more poisonous than the original scalar code.

### Pattern 6: Memory Access Analysis and Reordering
The vectorizer mishandles the analysis and transformation of strided or non-contiguous memory loads. When scalar loads require reordering to match the vector lane semantics, the pass sometimes generates widened, contiguous vector loads without emitting the necessary compensating `shufflevector` masks, leading to incorrect lane assignments and potentially unsafe out-of-bounds memory accesses. Additionally, its pointer distance analysis is flawed for 64-bit architectures, as it truncates offsets to 32 bits, erroneously grouping distant memory accesses into single contiguous vector operations.
